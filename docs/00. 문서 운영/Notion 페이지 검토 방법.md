# Notion 페이지 검토 방법

이 문서는 최초 Notion 반영이 끝난 강의 페이지를 검토할 때 사용하는 방법입니다. 목적은 이미 업로드된 Notion 페이지에서 제목 스타일, 섹션 배치, 개념 본문과 강사용 스크립트의 순서, 실습과 퀴즈의 연결성을 확인하는 것입니다.

검토 단계에서는 수정하지 않습니다. 문제 후보를 먼저 모아 사용자에게 보고한 뒤, 사용자가 요청한 항목만 Notion MCP/API로 부분 수정합니다.

## 적용 시점

- 최초 Notion 업로드가 끝난 뒤 페이지를 최종 검토할 때
- 사용자가 특정 `##` 제목2 단원 전체를 검토해 달라고 요청할 때
- 제목 스타일뿐 아니라 페이지 전체의 섹션 흐름, 번호, 개념/스크립트/실습/퀴즈 연결성을 점검할 때
- 여러 섹션을 한꺼번에 바꾸기 전에 수정 후보만 먼저 보고해야 할 때

## 검토 단위

기본 검토 단위는 Notion의 `heading_2`입니다.

예:

- `02. HTML 시작하기`
- `03. CSS 시작하기`
- `04. 완성하기`

`heading_2` 아래의 모든 `heading_3` 섹션을 같은 검토 범위로 묶습니다. 단, 실제 비교는 제목3 하나씩 독립적으로 하지 않고 같은 주제의 묶음으로 봅니다.

묶음 예:

- 개념 본문: `08. 하이퍼 링크 (<a> 태그)`
- 강사용 스크립트: `08. #스크립트 하이퍼 링크 (<a> 태그)`
- 실습: `08. #실습 구독페이지: 링크 추가하기`
- 같은 주제의 요약 또는 퀴즈가 있는 경우 같은 번호나 단원 마지막 번호 흐름 안에서 연결성을 확인합니다.

## 검토 전 원칙

- Notion 페이지를 검토하는 동안 로컬 `courses/*/강의자료.md`와 `_notion.md`는 수정하지 않습니다.
- 검토 중에는 Notion도 수정하지 않습니다.
- 먼저 스냅샷을 만들고, 스냅샷을 기준으로 문제 후보를 찾습니다.
- 문제 후보와 실제 수정 여부를 분리합니다.
- 최신 라이브러리, 외부 서비스 UI, 설치 명령처럼 현재성이 필요한 내용은 직접 고치지 않고 확인 필요 항목으로 보고합니다.

## 스냅샷 만들기

Notion MCP/API로 대상 페이지의 top-level block을 읽고, 사용자가 지정한 `heading_2` 범위를 찾습니다.

1. 대상 페이지의 자식 블록을 읽습니다.
2. 지정한 `heading_2`를 찾습니다.
3. 다음 `heading_2`가 나오기 전까지의 모든 `heading_3`를 수집합니다.
4. 각 `heading_3`의 자식 블록을 재귀적으로 읽습니다.
5. 아래 정보를 사람이 읽을 수 있는 텍스트 스냅샷으로 만듭니다.

스냅샷에 포함할 정보:

- block id
- block type
- 제목 텍스트
- `heading_3`의 toggle 여부
- `heading_4` 소제목
- paragraph, quote, list, toggle 텍스트
- code block 언어와 코드 내용
- image, file 존재 여부
- `#스크립트`, `#실습`, `#퀴즈`, `#요약` 태그

스냅샷은 검토용 임시 파일로 둘 수 있습니다. 예:

```text
/tmp/codeit-fs-15-02-html-start-review.md
```

## 멀티 에이전트 검토 방식

멀티 에이전트를 사용할 때는 모든 에이전트가 같은 스냅샷을 읽어야 합니다. 그래야 개념 본문과 스크립트, 실습, 퀴즈를 같은 기준으로 비교할 수 있습니다.

권장 분담:

1. 초반 섹션 검토
   - 범위: 단원 앞부분의 개념 본문, 스크립트, 실습
   - 초점: 개념 설명과 스크립트 순서 일치, 누락된 실습, 코드 흐름

2. 후반 섹션 검토
   - 범위: 단원 후반의 개념 본문, 스크립트, 실습
   - 초점: 누적 코드 흐름, 이전 섹션과 이후 섹션의 코드 일치, 파일/이미지/링크 흐름

3. 구조 검토
   - 범위: `heading_2` 전체
   - 초점: 제목 번호, 제목 스타일, 별표 기준, `#스크립트`/`#실습`/`#퀴즈` 배치, 요약/퀴즈 정합성

각 에이전트에게 반드시 지시할 내용:

- 파일이나 Notion을 수정하지 않습니다.
- 문제 후보만 보고합니다.
- 섹션 제목과 이유를 함께 적습니다.
- 확실한 오류와 확인 필요 항목을 구분합니다.
- 같은 주제의 개념 본문과 `#스크립트`를 같은 입력으로 비교합니다.

## 검토 항목

### 제목 스타일

- `heading_2`는 단원의 큰 제목인지 확인합니다.
- `heading_3`는 하나의 강의 항목인지 확인합니다.
- `heading_3` 내부 소제목은 `heading_4`인지 확인합니다.
- `heading_3` 내부에 다시 `heading_3`가 들어가 있으면 문제 후보로 봅니다.
- 제목에는 백틱이나 인라인 코드 스타일을 쓰지 않습니다.
- 제목의 `#스크립트`, `#실습`, `#퀴즈` 태그 색상이 규칙에 맞는지 확인합니다.
- 학생용 개념 본문의 `heading_4`가 핵심 개념, 적용 방식, 주의점처럼 개념 흐름을 나누는지 확인합니다.
- 학생용 개념 본문에 `확인하기`, `저장하고 확인하기`, `사용해보기`, `넣기1`, `Live Server로 확인하기`, `마무리`처럼 영상 진행형 제목이 있으면 문제 후보로 봅니다.
- 단원에 `#요약` 섹션이 별도로 있는데 개별 개념 섹션마다 `정리` heading_4가 반복되면 통일성 문제 후보로 봅니다.

### 번호와 페어링

- 같은 주제를 다루는 개념 본문, `#스크립트`, `#실습`, `#퀴즈`의 섹션 번호가 같은지 확인합니다.
- 번호는 새 개념이 시작될 때만 증가합니다. 예를 들어 `02. 배경색`, `02. #스크립트 배경색`, `02. #실습 ... 배경색 ...`처럼 같은 주제 묶음은 모두 `02`를 사용합니다.
- 같은 번호가 반복될 때는 같은 주제 묶음인지, 아니면 학생이 순서를 오해할 수 있는 중복인지 확인합니다.
- 하나의 실습이 두 개 이상의 개념을 함께 다루면 단일 번호 페어링이 불명확하므로, 기존 번호를 유지하고 필요 시 확인 필요 항목으로 보고합니다.
- 중간 번호가 비어 있으면 원본 번호 유지인지 누락인지 확인 필요 항목으로 보고합니다.
- `#스크립트`, `#실습`, `#퀴즈`, `#요약`에 별표가 붙어 있으면 문제 후보로 봅니다.

### 개념과 스크립트 일치

- 학생용 개념 본문과 강사용 스크립트가 같은 코드 흐름을 설명하는지 확인합니다.
- 개념 본문에서 설명한 순서와 스크립트의 시연 순서가 맞는지 확인합니다.
- 개념 본문의 `heading_4` 흐름과 `#스크립트`의 `heading_4` 흐름이 비교 가능한지 확인합니다. 제목이 완전히 같을 필요는 없지만, 같은 개념과 코드 시연이 같은 순서로 이어져야 합니다.
- 개념 본문에는 있으나 스크립트에는 없는 핵심 시연, 또는 스크립트에는 있으나 개념 본문에는 없는 핵심 설명을 찾습니다.
- 스크립트는 구어체를 허용하지만, 개념 자체가 틀린 경우 문제 후보로 봅니다.
- 스크립트 안에 교재 규칙과 충돌하는 소제목 단계나 별표가 있는지 확인합니다.
- `#스크립트` 안의 진행형 제목은 허용하지만, 같은 제목이 학생용 개념 본문에 그대로 남아 있으면 문제 후보로 봅니다.

### 실습과 누적 코드 흐름

- 실습 링크가 앞선 개념과 같은 주제를 다루는지 확인합니다.
- 이전 섹션에서 만든 결과 코드가 이후 섹션의 코드와 이어지는지 확인합니다.
- 같은 프로젝트 안의 URL, 파일명, 이미지명, 코드 조각이 서로 맞는지 확인합니다.
- 같은 단원에서 반복되는 예시 값, 파일명, URL, 코드 결과가 개념 본문, 스크립트, 실습, 퀴즈 사이에서 일관되는지 확인합니다.
- 코드잇 실습 페이지 내부 문제를 확인하지 못한 경우, 실습 목표나 성공 기준을 추정하지 않습니다.

### 코드와 코드블록

- 코드블록 언어가 실제 내용과 맞는지 확인합니다.
- 코드블록이 단순 텍스트, URL, 명령어, HTML, CSS, JavaScript 등 실제 내용에 맞는 언어로 설정되어 있는지 확인합니다.
- 코드블록 안에 해당 언어의 문법이 아닌 설명 조각이 들어가면 문제 후보로 봅니다.
- 코드 예시가 문법적으로 닫히지 않았거나 이전/다음 섹션의 누적 코드와 충돌하면 문제 후보로 봅니다.
- 코드 일부를 생략해야 한다면 해당 언어에서 유효한 주석이나 설명 방식으로 표시되어 있는지 확인합니다.
- 초급자용 자료에서는 같은 개념이나 문법 표기를 한 단원 안에서 일관되게 유지합니다.

### 요약과 퀴즈

- 요약 섹션이 실제 요약인지, 단순 링크만 있는지 확인합니다.
- 퀴즈가 앞선 개념 범위를 벗어나지 않는지 확인합니다.
- 중요도 높은 학생용 개념 본문이 퀴즈에서 최소한 한 번은 확인되는지 검토합니다.
- 퀴즈 난이도와 시간은 [퀴즈 작성 기준](./퀴즈%20작성%20기준.md)을 따릅니다.

## 문제 후보 분류

검토 결과는 아래 네 가지로 분류합니다.

### 확실한 수정 후보

명확하게 잘못되었거나 규칙과 충돌하는 항목입니다.

예:

- 필요한 도구나 기능 이름을 다른 도구나 기능 이름으로 잘못 설명함
- `#스크립트` 제목에 별표가 붙음
- `heading_3` 내부 세부 소제목이 다시 `heading_3`로 들어감
- 코드블록 언어가 실제 내용과 맞지 않음

### 내용 연결성 후보

개념 본문, 스크립트, 실습, 이후 누적 코드가 서로 다르게 이어지는 항목입니다.

예:

- 개념 본문의 예시 값과 스크립트의 예시 값이 다름
- 이전 섹션의 결과 코드와 이후 섹션의 예시 코드가 다름
- 설명한 순서와 코드 예시의 순서가 다름

### 확인 필요 항목

원본 의도나 실제 실습 페이지를 확인해야 판단할 수 있는 항목입니다.

예:

- 특정 번호가 빠진 것이 원본 번호 유지인지 누락인지 불명확함
- 특정 섹션에 스크립트나 실습이 없는 것이 의도인지 누락인지 불명확함
- 외부 서비스 UI나 설치 방식이 최신 상태와 맞는지 확인 필요함

### 수정 보류 항목

기술적으로 틀리지는 않지만 초급자 혼란을 줄이기 위해 통일하면 좋은 항목입니다.

예:

- 같은 단원 안에서 동일한 문법을 여러 표기 방식으로 설명함
- 본문에서는 권장 방식을 설명했지만 일부 예시에는 그 방식이 반영되지 않음

## 보고 형식

검토 결과는 수정하지 않고 아래 형식으로 보고합니다.

```markdown
## Notion 페이지 검토 결과

- 대상:
- 검토 범위:
- 검토 방식:
- Notion 수정 여부: 없음
- 로컬 Markdown 수정 여부: 없음

### 우선 수정 후보

1. 섹션:
   - 문제:
   - 이유:
   - 권장 처리:

### 구조/규칙 후보

1. 섹션:
   - 문제:
   - 이유:

### 확인 필요 항목

1. 섹션:
   - 확인할 내용:
   - 이유:

### 수정 보류 가능 항목

1. 섹션:
   - 내용:
   - 판단:
```

## 수정 단계

사용자가 수정하라고 요청한 뒤에만 수정합니다.

- 최초 업로드 이후에는 로컬 Markdown을 갱신하지 않습니다.
- Notion MCP/API로 요청받은 섹션 또는 블록만 수정합니다.
- 여러 섹션을 수정해야 하면 우선순위를 정해 작은 단위로 처리합니다.
- 수정 후에는 수정한 블록과 검증 결과만 보고합니다.
