# 퀴즈 작성 기준

이 문서는 HTML, CSS, JavaScript, React, Express, Jest, AWS 등 전체 과정에서 퀴즈를 검토하거나 개선할 때 사용하는 기준입니다. 목적은 단순 암기 여부보다 개념 이해, 코드 해석, 상황 판단, 오류 수정 능력을 확인하는 것입니다.

## 적용 범위

- 기존 `#퀴즈` 섹션을 검토할 때 적용합니다.
- 사용자가 퀴즈 개선, 추가, 난이도 조정을 명시적으로 요청한 경우에만 새 문제를 추가하거나 교체합니다.
- 최초 Notion 업로드 전에는 `courses/*/강의자료.md`에서 반영할 수 있습니다.
- 최초 Notion 업로드 이후에는 로컬 Markdown을 갱신하지 않고 Notion MCP/API로 요청받은 퀴즈 섹션만 수정합니다.
- 강의에서 다루지 않은 개념을 문제의 핵심 판단 기준으로 사용하지 않습니다.

## 기본 원칙

- 퀴즈는 수업 내용을 외웠는지보다 수업 내용을 실제 상황에 적용할 수 있는지 확인해야 합니다.
- AI가 있는 학습 환경을 전제로, 검색 가능한 정의만 묻는 문제는 최소화합니다.
- 문제는 학습 범위 안에서 출제하되, 선택지만 보고 답을 고를 수 없도록 상황, 코드, 조건, 결과를 함께 제시합니다.
- 정답은 하나로 명확해야 합니다. 복수 정답 문제는 사용자가 명시적으로 요청한 경우에만 사용합니다.
- 오답 선택지는 실제 초급자가 자주 하는 착각을 반영하되, 억지 함정이나 말장난으로 만들지 않습니다.
- 최신 라이브러리, 패키지, AWS 서비스, 외부 서비스 정책처럼 현재 문서 확인이 필요한 내용은 직접 단정하지 않고 확인 필요 항목으로 보고합니다.

## 문제 유형 비율

10문항, 15분 퀴즈를 기준으로 아래 비율을 기본값으로 사용합니다.

| 유형 | 비율 | 목적 |
| --- | --- | --- |
| 기본 개념 확인 | 1-2문항 | 핵심 용어와 역할을 최소한으로 확인 |
| 개념 구분 | 2문항 | 비슷한 개념의 차이를 구분하는지 확인 |
| 코드 해석 | 2-3문항 | 코드 실행 결과, 렌더링 결과, 에러 원인을 판단 |
| 상황 판단 | 2-3문항 | 요구사항과 제약 조건에 맞는 해결책을 선택 |
| 오류 수정 | 1-2문항 | 잘못된 코드나 설정을 찾아 가장 적절하게 고침 |

단순 암기형 문제는 전체의 20%를 넘기지 않는 것을 원칙으로 합니다. 15분을 유지하는 퀴즈라면 최소 6문항 이상은 코드 해석, 상황 판단, 오류 수정 유형으로 구성합니다.

## 난이도 기준

### 하

- 한 개념의 이름, 역할, 기본 사용 위치를 확인합니다.
- 선택지만 보고 바로 답이 보이는 문제는 전체 퀴즈의 도입용으로만 사용합니다.
- 예: `<title>`의 역할, `charset`의 의미, `npm test`의 용도

### 중

- 두 개 이상의 개념을 구분하거나, 짧은 코드의 결과를 예측합니다.
- 수업 내용을 이해했다면 풀 수 있지만, 단순 정의 암기만으로는 어렵게 만듭니다.
- 예: `<p>`와 `<br>`의 차이, `useState` 업데이트 결과, Express 미들웨어 실행 순서

### 상

- 상황, 제약 조건, 코드 조각을 함께 읽고 가장 적절한 해결책을 선택합니다.
- 여러 선택지가 부분적으로 맞아 보이더라도, 요구사항을 가장 잘 만족하는 답이 하나여야 합니다.
- 예: React 상태가 갱신되지 않는 원인 판단, Jest 테스트 실패 원인 분석, AWS 구성에서 비용과 가용성 요구를 함께 만족하는 선택

## AWS SAA형 문제 작성 방식

AWS SAA 스타일은 특정 AWS 문제에만 한정하지 않고, 모든 기술 과정의 상황 판단형 문제에 적용할 수 있습니다.

1. 먼저 상황을 제시합니다.
   - 무엇을 만들고 있는지
   - 어떤 문제가 발생했는지
   - 어떤 제약 조건이 있는지
   - 어떤 결과가 필요한지
2. 선택지는 모두 그럴듯하게 작성합니다.
   - 하나는 요구사항을 가장 정확히 만족해야 합니다.
   - 나머지는 일부만 맞거나, 상황의 핵심 제약을 놓쳐야 합니다.
3. 문제는 "무엇인가요?"보다 "가장 적절한 방법은 무엇인가요?"에 가깝게 작성합니다.
4. 해설은 정답 이유와 주요 오답이 부적절한 이유를 함께 설명합니다.
5. 아직 배우지 않은 기술이나 외부 정책 변화가 큰 내용을 정답 판단의 핵심으로 삼지 않습니다.

## 기술별 출제 방향

### HTML/CSS

- 태그 이름만 묻기보다 브라우저가 코드를 어떻게 해석하는지 묻습니다.
- 화면에 표시되는 결과, 태그의 의미, 경로, 선택자, 박스 모델, 레이아웃 결과를 확인합니다.
- 예: 이미지가 보이지 않는 이유, `<title>`과 `<h1>`의 차이, `<div>`와 `<span>`의 기본 동작 차이

### JavaScript

- 문법 이름보다 값의 흐름, 타입, 조건, 반복, 함수 실행 결과를 확인합니다.
- 코드 실행 결과, 에러 원인, 비동기 처리 순서, DOM 조작 결과를 묻습니다.
- 예: 이벤트 리스너가 실행되는 시점, 배열 메서드 결과, 비동기 함수의 반환값

### React

- 컴포넌트 구조, props, state, 렌더링 흐름, 이벤트 처리, effect 실행 조건을 확인합니다.
- 단순 API 이름보다 "왜 화면이 갱신되지 않는지", "어떤 상태를 어디에 두어야 하는지"를 묻습니다.
- 예: controlled input, key의 역할, state 직접 변경 문제, effect 의존성 배열

### Express/API

- 요청과 응답의 흐름, 라우팅, 미들웨어, 상태 코드, 에러 처리, 인증 흐름을 확인합니다.
- 단순 메서드 이름보다 요구사항에 맞는 라우트와 응답 설계를 묻습니다.
- 예: `GET`과 `POST` 선택, `404`와 `500` 구분, 미들웨어 실행 순서

### Jest/테스트

- matcher 이름만 묻기보다 어떤 동작을 검증해야 하는지 확인합니다.
- 테스트 실패 원인, arrange-act-assert 구조, mock이 필요한 상황을 묻습니다.
- 예: 비동기 테스트에서 `await` 누락, 호출 횟수 검증, 예외 발생 검증

### AWS

- 서비스 이름 암기보다 요구사항에 맞는 아키텍처 판단을 확인합니다.
- 가용성, 비용, 보안, 운영 부담, 확장성 중 문제에서 제시한 우선순위를 기준으로 답을 고르게 합니다.
- AWS 서비스 사양, 요금, 정책은 변경될 수 있으므로 현재 문서 확인이 필요한 경우 직접 수정하지 않고 보고합니다.

## 시간 기준

- 5분 퀴즈: 3-4문항, 대부분 하-중 난이도
- 10분 퀴즈: 6-8문항, 중 난이도 중심
- 15분 퀴즈: 8-10문항, 중 난이도 중심에 상 난이도 1-2문항 포함
- 코드 해석형이나 상황 판단형 문제는 한 문항당 1.5-2분을 기준으로 계산합니다.
- 긴 코드가 필요한 문제는 한 퀴즈 안에서 2문항 이하로 제한합니다.

## 해설 기준

- 모든 문제에는 `정답 및 해설` 토글을 둡니다.
- 해설은 정답만 쓰지 않고, 왜 그 답이 맞는지 설명합니다.
- 오답이 실제로 헷갈릴 만한 선택지라면 왜 틀렸는지도 짧게 설명합니다.
- 설명은 학생용 본문과 동일하게 `합니다/입니다`체로 작성합니다.
- 코드나 태그는 본문 인라인 코드 규칙을 따릅니다. 제목에는 백틱을 사용하지 않습니다.

## 검수 체크리스트

퀴즈를 수정하거나 새로 만들 때 아래 항목을 확인합니다.

- 단순 암기형 문제가 전체의 20%를 넘지 않습니다.
- 15분 퀴즈라면 최소 6문항 이상이 코드 해석, 상황 판단, 오류 수정 유형입니다.
- 각 문제의 정답이 하나로 명확합니다.
- 선택지가 모두 같은 범주 안에서 비교됩니다.
- 오답은 말장난이 아니라 실제 오개념을 반영합니다.
- 수업에서 다루지 않은 개념이 정답 판단의 핵심으로 쓰이지 않습니다.
- 최신 문서 확인이 필요한 내용은 직접 단정하지 않고 보고합니다.
- `정답 및 해설` 토글이 있고, 해설이 충분히 구체적입니다.
- 퀴즈 시간과 문항 수, 난이도가 서로 맞습니다.
