# codeit-fs-15 목차

이 저장소는 코드잇 풀스택 부트캠프 15기 강의 자료를 만들기 위한 작업 공간입니다.

## 01. docs

경로: [docs](./docs)

용도:

- 이 레포의 운영 문서
- 강의자료 제작 기준
- 커리큘럼 진행 상태 기록
- 참고자료 정리 원칙
- Notion 변환 원칙

작성 기준:

- [15기 강의자료 개선 기준](./docs/00.%20문서%20운영/15기%20강의자료%20개선%20기준.md)을 먼저 확인합니다.
- [01 기준 샘플](./docs/00.%20문서%20운영/01%20기준%20샘플.md)은 Notion 동기화, formatter 생성, 품질 점검의 기준 예시로 사용합니다.
- [강의자료 최종 검수 절차](./docs/00.%20문서%20운영/강의자료%20최종%20검수%20절차.md)는 자동 검사와 사람이 읽는 최종 검수를 분리하는 기준으로 사용합니다.
- [강의자료 검토 및 업로드 절차](./docs/00.%20문서%20운영/강의자료%20검토%20및%20업로드%20절차.md)는 현재 작업 기준입니다. 원문 보존, 제한적 수정, 최초 업로드 이후 Notion MCP 섹션 단위 교정 원칙은 이 문서를 따릅니다.
- [Notion 페이지 검토 방법](./docs/00.%20문서%20운영/Notion%20페이지%20검토%20방법.md)은 최초 업로드 이후 Notion 페이지나 `heading_2` 단원 전체를 검토할 때 사용합니다. 이때 먼저 스냅샷을 만들고, 제목 스타일과 개념/스크립트/실습/요약/퀴즈 연결성을 확인한 뒤 문제 후보만 보고합니다.
- 학생용 본문의 개념 설명, 문체, 교육적 적절성 판단은 `docs` 기준 문서를 따릅니다. `$notion-formatter`는 이런 콘텐츠 판단 기준이 아니라 최초 Notion 업로드 또는 명시적 전체 교체를 위한 변환/업로드 도구입니다.
- `docs`에는 실제 강의자료 본문을 두지 않습니다.
- `docs`에는 이 레포의 구조, 작성 기준, 진행 상태, 운영 기록만 둡니다.

## 02. courses

경로: [courses](./courses)

용도:

- 15기에서 실제로 사용할 강의 자료
- 과정별 수업 문서
- 실습 가이드
- 퀴즈와 해설

작성 기준:

- 15기 강의 순서와 과정 제목은 `references/raw/notion`의 `NN.제목` 폴더명을 그대로 따릅니다.
- 내부 내용은 원문을 보존한 상태에서 내용, 말투, 억양이 교과서적으로 적절한지 검토하고, 교과서적으로 적절하지 않은 학생용 본문은 수정합니다.
- 명확하게 잘못된 내용이 있으면 최초 Notion 업로드 전에는 `courses/*/강의자료.md`에서 수정하고, 최초 업로드 이후에는 `강의자료.md`를 갱신하지 않고 Notion MCP/API로 해당 페이지의 해당 섹션만 수정합니다.
- 최신 라이브러리, 패키지, 외부 서비스 정책처럼 현재 문서 확인이 필요한 내용은 직접 수정하지 않고 사용자에게 확인 필요 항목으로 보고합니다.
- `#스크립트`는 학생용 읽기 자료가 아니라 강사용 수업 진행 대본입니다. 구어체를 허용하며 교과서형 문장으로 전면 변환하지 않습니다.
- Notion 화면에서 보이는 왼쪽 세로 바(`|`, quote 블록)는 `#스크립트` 말하기 흐름에만 사용합니다. 스크립트가 아닌 학생용 개념 설명은 quote 블록을 쓰지 않고 일반 문단, 목록, 소제목으로 작성합니다.
- 원문에 있던 코드, 이미지, zip, 스크립트 흐름은 삭제하지 않습니다.
- 원문 구조를 크게 바꾸거나 섹션을 새로 나누지 않습니다. 구조 변경은 사용자가 명시적으로 요청한 경우에만 진행합니다.
- 같은 주제를 다루는 개념 본문, `#스크립트`, `#실습`, `#퀴즈`는 같은 섹션 번호를 유지합니다.
- 번호는 새 개념이 시작될 때만 증가합니다. 같은 주제의 실습이나 퀴즈는 별도 번호를 새로 받지 않습니다.
- 같은 강의 항목 안에서 자료 유형이 섞여 있으면 `본문 → #스크립트 → #실습 → #퀴즈` 순서로 정리합니다.
- 같은 단원에서 반복되는 예시 값, 파일명, URL, 코드 결과는 개념 본문, 스크립트, 실습, 퀴즈 사이에서 일관되게 유지합니다.
- `####` 제목4는 [제목 계층 기준](./docs/00.%20문서%20운영/제목%20계층%20기준.md)에 따라 하나의 `###` 강의 항목 안에서 핵심 개념, 설명 흐름 전환, 코드 예시, 주의점을 구분할 때만 사용합니다.
- 학생용 개념 본문의 제목4는 `확인하기`, `사용해보기`, `넣기1`, `Live Server로 확인하기`, `마무리`처럼 영상 진행을 표시하는 제목으로 쓰지 않습니다. 이런 진행형 제목은 필요한 경우 `#스크립트` 안에서만 허용합니다.
- `#요약` 섹션이 별도로 있으면 개별 개념 섹션마다 `정리` 제목4를 반복하지 않습니다. 짧은 마무리는 제목4 없이 일반 문단이나 목록으로 처리합니다.
- 같은 주제의 개념 본문과 `#스크립트`는 제목4의 설명 흐름이 서로 비교 가능해야 합니다. 제목이 완전히 같을 필요는 없지만, 개념 본문은 개념 순서로, `#스크립트`는 같은 시연 순서로 이어져야 합니다.
- `##`, `###`, `####` 제목에는 백틱을 추가하지 않습니다. 예: `01. 사이트에 이름 붙이기 (<head>, <body>, <title> 태그)`처럼 제목에 들어간 태그명은 일반 제목 텍스트로 두고, 인라인 코드 스타일은 본문 설명에만 사용합니다.
- 자동 검사는 보조 도구로만 사용하고, 최종 통과 여부는 [강의자료 검토 및 업로드 절차](./docs/00.%20문서%20운영/강의자료%20검토%20및%20업로드%20절차.md)와 [강의자료 최종 검수 절차](./docs/00.%20문서%20운영/강의자료%20최종%20검수%20절차.md)에 따라 직접 읽고 판단합니다.
- 퀴즈를 검토하거나 사용자의 요청으로 개선할 때는 [퀴즈 작성 기준](./docs/00.%20문서%20운영/퀴즈%20작성%20기준.md)을 따릅니다. 단순 암기형 문제를 줄이고, 코드 해석, 상황 판단, 오류 수정형 문제를 중심으로 구성합니다.
- 중요도 별표는 [중요도 별표 기준](./docs/00.%20문서%20운영/중요도%20별표%20기준.md)에 따라 학생용 개념 본문 제목에만 붙입니다. `#스크립트`, `#실습`, `#퀴즈`, `#요약`, `#Video`, 설치, 링크, 참고 자료에는 기본적으로 붙이지 않습니다.
- `⭐⭐⭐`는 이후 진도를 따라가기 어려워지는 핵심 구조나 필수 문법에만 사용합니다. `⭐⭐`는 자주 틀리는 보조 개념, `⭐`는 특수 문법이나 보충 팁에 사용합니다. 한 `##` 단원 안에서 `⭐⭐⭐`는 2개를 넘기지 않는 것을 원칙으로 합니다.
- 코드잇 실습 페이지의 내부 문제를 직접 확인하지 못한 경우, 파일명이나 주변 코드만 보고 목표, 진행 단계, 성공 확인을 추정해 쓰지 않습니다.
- 퀴즈, 실습, 문제은행, 숙제, 추가 설명은 사용자가 명시적으로 요청하지 않으면 새로 만들지 않습니다.
- 실습 코드는 `references/code/*`를 기준으로 확인할 수 있지만, 최신 라이브러리나 패키지 업데이트가 필요해 보이면 직접 수정하지 않고 보고합니다. 11기 참고 코드는 `references/code/codeit-fs-11/`에 둡니다.
- 코드 일부를 생략해야 한다면 해당 언어에서 유효한 주석이나 설명 방식으로 표시합니다. 코드블록은 학생이 복사했을 때 문법적으로 깨지지 않는 것을 원칙으로 합니다.
- 과정별 문서는 `courses/NN.제목/강의자료.md`처럼 과정 폴더 바로 아래에 둡니다. 별도 `v1`, `v2` 폴더는 만들지 않고 변경 이력은 Git 커밋과 필요 시 운영 문서로 관리합니다.
- Notion 업로드용 Markdown은 최초 업로드 전 원본 문서 작성이 끝난 뒤에만 만듭니다.
- Notion 변환이 필요하면 이 저장소 전용 로컬 스킬 `$notion-formatter`를 사용합니다. 이 스킬은 최초 Notion 업로드 또는 사용자가 명시한 전체 교체 때만 사용하는 변환/업로드 도구이며, 학생용 본문 문체나 개념 설명을 판단하는 기준으로 사용하지 않습니다. 스킬 파일은 Codex가 레포 로컬 스킬로 읽는 `.agents/skills/notion-formatter`에만 두고, 전역 스킬 위치인 `~/.codex/skills`나 `~/.agents/skills`에는 두지 않습니다. 최초 업로드 전에는 원본 `courses` 문서를 기준 초안으로 두고, `_notion.md`는 업로드용 파생본으로만 취급합니다.
- 새로운 과정의 최초 정리 단계에서는 과정 전체를 규칙에 맞게 정리할 수 있습니다. 이때 교과서형 문체, 제목 계층, 스크립트와 본문 분리, 별표 기준, 코드블록과 인라인 코드 규칙을 전체적으로 점검합니다.
- 최초 정리 후 Notion에 처음 반영할 때는 사용자의 요청에 따라 전체 업로드 또는 전체 교체를 진행할 수 있습니다.
- 최초 Notion 반영이 끝난 뒤에는 사용자가 특정 섹션을 보며 교정하는 단계로 봅니다. 이 단계에서는 `courses/*/강의자료.md`와 `_notion.md`를 다시 수정하지 않고, Notion MCP/API로 업로드된 페이지에서 요청받은 섹션이나 블록만 찾아 수정합니다.
- 최초 Notion 반영 이후 사용자가 페이지 전체나 `heading_2` 단원 전체 검토를 요청하면 [Notion 페이지 검토 방법](./docs/00.%20문서%20운영/Notion%20페이지%20검토%20방법.md)에 따라 수정 없이 문제 후보를 먼저 보고합니다.
- Markdown에서 백틱으로 감싼 인라인 코드는 Notion 업로드 시 인라인 코드로 유지하고 텍스트 색상을 `blue`로 지정합니다. 굵게는 적용하지 않으며, 코드블록에는 이 규칙을 적용하지 않습니다.

## 03. references

경로: [references](./references)

용도:

- 15기 강의 자료를 만들기 위해 참고할 원본 자료
- 이전 기수, 특히 11기에서 사용한 PDF와 Markdown 자료
- 기존 강의 자료와 임시 메모
- 원본 자료를 바탕으로 15기용 문서를 만들기 위한 기준 자료

하위 구성:

- [raw](./references/raw): 사용자가 넣어주는 원본 zip, PDF, Markdown 자료
- [raw/notion](./references/raw/notion): Notion export zip을 해제한 자료
- [raw/zip](./references/raw/zip): 원본 zip과 첨부 zip을 모아둔 보관 폴더
- `raw/pdf`: 별도 PDF 원본이 생기면 추가할 위치
- [code](./references/code): 이전 기수 코드와 사용자가 추가한 실습 코드. 11기 코드는 `references/code/codeit-fs-11/`

## 정리 원칙

- 사용자는 정리 전 자료를 `references/raw`에 넣습니다.
- `references/raw`의 원본 파일은 가능한 한 수정하지 않습니다.
- Notion export zip은 `references/raw/notion/<페이지 제목>/` 아래에 해제합니다.
- `references/raw/notion`의 과정 폴더명은 `NN.제목` 형식으로 정리합니다. 예: `01.프로그래밍 시작하기`
- Notion export 안의 첨부 zip은 원본 zip 옆에 같은 이름의 폴더로 해제합니다.
- 해제 후 zip 파일은 `references/raw/zip` 아래로 모읍니다.
- PDF 원본이 별도로 필요하면 `references/raw/pdf` 아래에 둡니다.
- 제가 `references/raw`의 PDF와 Markdown, `references/code/*`의 실습 코드를 읽고 15기용 강의 자료의 오류와 교과서형 문체를 검토합니다.
- 이전 기수 자료는 원문을 보존하되, 최초 업로드 전에는 명확한 오류와 교과서형 문체에 맞지 않는 부분만 `courses`에서 제한적으로 수정합니다.
- Notion 업로드가 필요해지면 `courses` 초안을 먼저 확정한 뒤 `$notion-formatter`로 `_notion.md`를 생성합니다.
- Notion 반영은 사용자의 명시 요청이 있을 때만 진행합니다. 새로운 과정의 최초 반영 단계에서는 전체 업로드 또는 전체 교체를 사용할 수 있지만, 최초 반영 이후 교정 단계에서는 `courses/*/강의자료.md`를 업데이트하지 않고 Notion MCP/API로 필요한 섹션 또는 블록만 부분 수정합니다. 기존 페이지 내용을 archive 처리하고 `_notion.md`를 통째로 다시 올리는 `replace` 방식은 최초 반영이거나 사용자가 전체 교체를 명시적으로 요청한 경우에만 사용합니다.
- 버전이나 정책이 바뀔 수 있는 내용은 최종 문서 작성 전에 다시 확인합니다.
