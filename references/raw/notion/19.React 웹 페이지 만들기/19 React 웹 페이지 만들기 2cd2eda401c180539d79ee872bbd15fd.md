# 1. 웹 애플리케이션의 기본 구조 이해

## 1-01. URL vs URI ⭐⭐

웹 개발을 하다 보면 'URL'과 'URI'라는 용어를 자주 접하게 됩니다. 비슷해 보이지만 사실 조금 다른 의미를 가지고 있습니다. 이 둘의 차이를 명확히 이해하면, 웹 리소스를 다루는 방식을 더 정확하게 파악할 수 있습니다.

---

### URL과 URI의 관계

먼저 결론부터 말하자면:

- **모든 URL은 URI**입니다.
- **하지만 모든 URI가 URL인 것은 아닙니다.**
- 즉, `URL ⊂ URI` (URL은 URI의 부분집합)

URI는 더 넓은 개념이고, URL은 그 안에 포함되는 특수한 형태라고 생각하면 됩니다.

### 1. 범위

### URI (Uniform Resource Identifier)

인터넷상의 모든 자원을 **식별**하는 일반적 식별자입니다. **"이것이 무엇인지"** 알려주는 역할을 합니다.

### URL (Uniform Resource Locator)

URI의 한 종류로, 자원을 찾기 위한 **위치**를 함께 제공하는 식별자입니다. **"이것이 어디에 있는지, 어떻게 접근하는지"**까지 알려줍니다.

---

### 2. 구성 요소

### URI의 예시

자원의 **식별자만** 명시합니다.

```
mailto:alice@example.com
```

- 이메일 주소 자원을 식별하는 URI

```
urn:isbn:9780134190440
```

- 특정 책(ISBN 9780134190440)을 식별하는 URN

```
/products/42
```

- "제품 ID 42" 자원을 식별하는 URI 경로

```
tag:john.doe@example.com,2025-09-17:project-alpha
```

- 태그 식별자

### URL의 예시

프로토콜, 호스트, 경로 등 자원에 **접근할 수 있는 위치 정보**를 포함합니다.

```
https://example.com/index.html
```

이 URL은 다음 정보를 모두 포함합니다:

- **프로토콜**: `https://` (어떻게 접근할 것인가)
- **호스트**: `example.com` (어디에 있는가)
- **경로**: `/index.html` (무엇을 가져올 것인가)

---

### 3. 용도

### URI

자원을 **"무엇인지(Who or What)"** 식별합니다.

예를 들어, `/products/42`는 "42번 상품"이라는 자원을 식별하지만, 이것이 어느 서버에 있는지는 알 수 없습니다.

### URL

자원의 **"어디에 있고 어떻게 접근하는지(Where & How)"** 지정합니다.

예를 들어, `https://shop.example.com/products/42`는 "42번 상품"이 어느 서버의 어느 경로에 있으며, HTTPS 프로토콜로 접근해야 한다는 것까지 명시합니다.

---

### 요약

**URI**는 자원을 식별하는 상위 개념이고, **URL**은 자원의 위치와 접근 방법을 구체적으로 명시하는 하위 개념입니다.

웹 개발에서는 대부분 URL을 사용하지만, API 문서나 기술 스펙에서는 더 포괄적인 의미의 URI라는 용어를 사용하는 경우가 많습니다. 두 개념의 차이를 이해하고 있으면, 이러한 문서를 읽을 때 혼란을 줄일 수 있습니다.

## 1-01. #Quiz URL vs URI

### 문제 1

다음 중 일반적인 웹 리소스 URL이 **아닌** URI는 무엇인가요?

1. `https://www.example.com/page.html`
2. `ftp://files.example.com/document.pdf`
3. `mailto:support@example.com`
4. `http://api.example.com/users/123`
- 정답 및 해설
    
    정답:
    
    3번 (`mailto:support@example.com`)
    
    해설:
    
    `mailto:`는 자원의 위치를 네트워크 상에서 직접 가리키기보다는, 이메일 주소라는 식별자 자체를 표현하는 URI입니다.
    
    일반적인 웹 리소스(http, https, ftp 등)와는 성격이 다르기 때문에 이 문제에서는 전형적인 URL로 분류하지 않습니다. 1, 2, 4번은 모두 프로토콜과 호스트, 경로를 포함하여 네트워크 상의 자원 위치를 명시하는 URL입니다.
    
    초급 단계에서는 "웹 리소스의 위치를 가리키는 것"을 URL로 이해하면 충분합니다.
    

---

### 문제 2

다음 설명 중 **올바른 것**을 모두 고르세요.

1. 모든 URL은 URI이다.
2. 모든 URI는 URL이다.
3. URI는 자원을 식별하는 역할을 한다.
4. URL은 자원의 위치와 접근 방법을 명시한다.
- 정답 및 해설
    
    정답:
    
    1, 3, 4번
    
    해설:
    
    URL은 URI의 부분집합이므로 모든 URL은 URI에 해당합니다. 반대로 URI는 URL보다 더 넓은 개념이기 때문에, 모든 URI가 URL은 아닙니다. URI는 "Uniform Resource Identifier"로 자원을 식별하는 것이 목적이며, URL은 "Uniform Resource Locator"로 자원의 위치와 접근 방법을 함께 명시합니다.
    
    쉽게 말하면 URI는 식별자를 포함한 전체 개념이고, URL은 그중에서도 실제 위치 정보가 포함된 형태입니다.
    
    ```
    URI ⊃ URL
    (URI가 더 큰 개념)
    ```

### 문제 3

다음 중 URI의 역할에 대한 설명으로 **가장 적절한 것**은?

1. 자원이 "어디에" 있는지 알려준다.
2. 자원이 "무엇인지" 식별한다.
3. 자원에 "어떻게" 접근하는지 알려준다.
4. 자원의 "프로토콜"을 정의한다.
- 정답 및 해설
    
    정답:
    
    2번
    
    해설:
    
    URI는 자원을 식별(Identify)하는 것이 주 목적입니다. 즉, 자원이 "무엇인지"를 알려주는 역할을 합니다.
    
    위치나 접근 방법을 포함하는 것은 URI의 한 종류인 URL의 역할입니다. URL은 위치와 접근 방법을 함께 담고 있지만, URI 전체의 역할은 식별에 있습니다.
    
    정리하면 URI는 자원을 식별하는 모든 문자열을 의미하고, URL은 자원의 위치와 접근 방법까지 포함한 URI입니다. 초급 단계에서는 주소 형태이면 URL, 그 외의 식별자는 URI로 이해해도 충분합니다.
    

## 1-02. URL 이름 짓는 규칙 ⭐⭐⭐

웹 애플리케이션을 만들 때 URL 구조를 잘 설계하는 것은 매우 중요합니다. 좋은 URL은 사용자가 이해하기 쉽고, 검색 엔진 최적화(SEO)에도 유리하며, 개발자가 유지보수하기도 편합니다. URL을 설계할 때는 **일관성**, **가독성**, **확장성**을 최우선으로 고려해야 합니다.

---

### 1. 소문자 사용

모든 경로는 소문자로 작성합니다.

**좋은 예:**

```
/products
/user-profile
```

**피해야 할 예:**

```
/Products
/User-Profile
```

대소문자를 섞어 쓰면 웹 서버나 라우팅 설정에 따라 서로 다른 URL로 인식될 수 있어 혼란을 만들 수 있습니다. 일관성을 위해 소문자만 사용하는 것이 좋습니다.

---

### 2. 하이픈(-)으로 단어 구분

언더스코어(`_`) 대신 하이픈(`-`)을 사용해 가독성을 높입니다.

**좋은 예:**

```
/web-development
/mobile-apps
```

**피해야 할 예:**

```
/web_development
/mobile_apps
```

하이픈은 검색 엔진이 단어를 구분하는 데 도움을 주며, URL에서도 더 자연스럽게 보입니다.

---

### 3. 명사형, 복수형 컨벤션

REST API에서는 리소스를 나타낼 때 **복수형 명사**를 사용하는 것을 권장합니다.

**좋은 예:**

```
/users
/orders
/products
```

**단수형을 사용하는 경우:**

```
/company
/me
/auth
```

복수형을 사용하면 여러 항목의 목록을 다루는지, 단일 항목을 다루는지 명확히 구분할 수 있습니다.

**참고**: 복수형 명사는 REST API에서 널리 쓰이는 권장 컨벤션이며, `/company`(회사 정보), `/me`(현재 사용자), `/auth`(인증) 같이 의미상 단일 개념인 경우에는 단수형을 사용하기도 합니다.

---

### 4. 계층 구조 반영

Next.js, Remix 같은 파일 기반 라우팅 프레임워크에서는 폴더 구조와 URL 계층을 일치시키는 것이 좋습니다.

```jsx
src/routes/
└── services/
    ├── index.tsx      → /services
    └── web-development.tsx → /services/web-development
```

많은 현대 프레임워크에서는 폴더 구조가 URL과 자연스럽게 매핑되도록 설계되어 있으며, 이를 맞추면 코드를 찾기 쉽고 프로젝트 구조를 파악하기도 쉽습니다.

---

### 5. 짧고 명확한 경로

가능한 한 경로를 짧게 유지하되, 의미는 분명해야 합니다.

**좋은 예:**

```
/auth/login
```

**너무 장황한 예:**

```
/authentication/sign-in-page
```

URL은 사용자가 기억하고 입력하기 쉬워야 하므로, 불필요하게 길게 만들지 않는 것이 좋습니다.

---

### 6. 동사 사용하지 않기

URL에는 동작이 아닌 **리소스(명사)**를 표현합니다.

**좋은 예:**

```
POST /auth/login
GET  /orders
DELETE /products/123
```

**피해야 할 예:**

```
POST /doLogin
GET  /getOrders
DELETE /deleteProduct/123
```

REST API에서는 HTTP 메서드(GET, POST, PUT, DELETE)가 동작을 나타내므로, URL에는 리소스만 표현하는 것이 원칙입니다.

---

### 7. 쿼리스트링은 필터·페이징에 사용

쿼리스트링(`?` 뒤에 오는 파라미터)은 리소스 식별에는 사용하지 않고, **검색·필터·페이지네이션** 용도로만 사용합니다.

**좋은 예:**

```
/products?search=phone&page=2
```

**피해야 할 예:**

```
/products?id=123  (대신 /products/123 사용)
```

URL 경로는 리소스를 식별하고, 쿼리스트링은 그 리소스를 어떻게 보여줄지 결정하는 옵션으로 사용하는 것이 좋습니다.

---

### 요약

좋은 URL 작명 규칙을 따르면:

- 사용자가 URL만 보고도 어떤 페이지인지 예측할 수 있습니다.
- 검색 엔진이 페이지를 더 잘 이해할 수 있습니다.
- 개발자가 코드를 찾고 수정하기 쉬워집니다.
- 프로젝트가 커져도 일관성 있게 관리할 수 있습니다.

**중요**: 이 규칙들은 문법처럼 반드시 지켜야 하는 절대 규칙이 아니라, 팀과 프로젝트에서 **일관성을 유지하기 위한 약속(컨벤션)**입니다. 웹 개발 커뮤니티에서 오랜 경험을 통해 정립된 모범 사례로, 실무에서는 프로젝트의 특성과 팀의 합의에 따라 적절히 적용하면 됩니다.

다음 중 **가장 적절한** URL 형식은 무엇인가요?

1. `/User-Profile`
2. `/user_profile`
3. `/user-profile`
4. `/UserProfile`
- 정답 및 해설
    
    정답:
    
    3번 (`/user-profile`)
    
    해설:
    
    일반적인 URL 컨벤션에 따라 경로는 소문자 사용을 권장하며, 단어 구분은 하이픈(-, kebab-case)을 사용하는 것이 가장 널리 쓰입니다.
    
    따라서 `/user-profile`이 가장 적절한 형식입니다. 이는 SEO, 가독성, 일관성 측면에서 가장 권장되는 방식입니다.

### 문제 2

다음 중 URL 설계 시 **잘못된 사용 예**는?

1. `/products?category=electronics&sort=price`
2. `/users/123/orders`
3. `/blog-posts`
4. `/products?id=456`
- 정답 및 해설
    
    정답:
    
    4번 (`/products?id=456`)
    
    해설:
    
    쿼리스트링은 주로 필터링, 정렬, 페이지네이션 같은 옵션에 사용하며, REST API 설계에서는 리소스의 고유 식별자는 경로(path)에 포함하는 것을 권장합니다.
    
    따라서 특정 상품을 표현할 때는 `/products?id=456`보다는 `/products/456` 형태가 더 적절합니다.
    
    쉽게 말해 "대상"은 경로에, "조건"은 쿼리스트링에 둡니다.
    
    1번은 카테고리 필터와 정렬 옵션으로 쿼리스트링을 올바르게 사용한 경우이고, 2번은 계층 구조를 잘 반영한 경로이며, 3번은 하이픈으로 단어를 구분한 올바른 형식입니다. 4번은 리소스 식별자를 쿼리스트링에 넣어 권장 패턴을 벗어납니다.

### 문제 3

다음 폴더 구조에 맞는 **올바른 URL 매핑**은?

```
src/routes/
└── admin/
    ├── index.tsx
    └── users.tsx
```

1. `/admin` → index.tsx, `/admin-users` → users.tsx
2. `/admin` → index.tsx, `/admin/users` → users.tsx
3. `/Admin` → index.tsx, `/Admin/Users` → users.tsx
4. `/admin_index` → index.tsx, `/admin_users` → users.tsx
- 정답 및 해설
    
    정답:
    
    2번
    
    해설:
    
    파일 기반 라우팅(Next.js, Remix 등)을 사용하는 경우, 폴더 구조와 URL 계층을 일치시키는 것이 좋습니다.
    
    `/admin`은 `admin/index.tsx`에 매핑되고, `/admin/users`는 `admin/users.tsx`에 매핑됩니다.
    
    이 방식은 코드 위치를 쉽게 찾을 수 있고, 프로젝트 구조가 직관적이며, 유지보수가 용이합니다. 또한 소문자 경로와 계층 구조를 그대로 반영하므로 URL 컨벤션에도 부합합니다.
    

## 1-03. Parameter & Query String

URL을 통해 웹 애플리케이션에 데이터를 전달하는 방법은 크게 두 가지가 있습니다. 바로 **URL Parameter(Path Parameter)**와 **Query String(Search Parameters)**입니다. 두 가지 모두 중요하지만, 사용 목적과 방법이 다릅니다.

---

### URL Parameter (Path Parameter)

URL 경로의 일부로 포함되는 동적인 값입니다.

### 특징

- URL 경로에 **직접 포함**됨
- 리소스를 **식별**하는 데 사용
- 대부분 리소스를 식별하는 데 사용되며, 생략되면 의미가 달라지는 경우가 많음

### 예시

```
[PATCH]
https://example.com/users/123/posts/456
			                    ^^^       ^^^
				                 userId    postId
```

위 URL에서:

- `123`은 사용자 ID를 나타내는 파라미터
- `456`은 게시글 ID를 나타내는 파라미터

### 코드 예시

```jsx
// React Router에서 라우트 정의
<Route
  path="/users/:userId/posts/:postId"
  element={<Post />}
/>;

// 컴포넌트에서 파라미터 추출
import { useParams } from "react-router";

function Post() {
  const { userId, postId } = useParams();
  // userId = "123", postId = "456"
}
```

`useParams()` 훅을 사용하면 URL 경로에서 파라미터 값을 쉽게 추출할 수 있습니다. 라우트 정의 시 파라미터는 `:` 기호로 시작합니다.

---

### Query String (Search Parameters)

URL 끝에 **`?`** 뒤에 붙는 **`key=value`** 형태의 선택적 데이터입니다.

### 특징

- **`?`** 뒤에 위치
- `&`로 여러 파라미터 연결
- **선택적**이며 필터링, 정렬, 페이징에 사용
- **순서가 바뀌어도 의미가 같음**: `/products?page=2&sort=price`와 `/products?sort=price&page=2`는 동일

**보안 주의**: Query String은 URL에 그대로 노출되므로 비밀번호, 토큰 같은 민감한 정보는 절대 담지 않습니다.

### 예시

```
https://example.com/products?search=laptop&page=2&sort=price
                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                   Query String
```

위 URL에서:

- `search=laptop`: 검색어 필터
- `page=2`: 2페이지 표시
- `sort=price`: 가격순 정렬

### 코드 예시

```jsx
// React Router에서 쿼리 스트링 추출
import { useSearchParams } from "react-router";

function ProductList() {
  const [searchParams] = useSearchParams();

  const search = searchParams.get("search"); // "laptop"
  const page = searchParams.get("page"); // "2"
  const sort = searchParams.get("sort"); // "price"
}
```

`useSearchParams()` 훅을 사용하면 쿼리 스트링의 값들을 추출할 수 있습니다.

---

### 주요 차이점

| **구분** | **URL Parameter** | **Query String** |
| --- | --- | --- |
| 위치 | URL 경로 내부 | URL 끝 **`?`** 뒤 |
| 목적 | 리소스 식별 | 필터링, 옵션 |
| 필수성 | 필수 | 선택적 |
| 예시 | **`/users/123`** | **`?page=2&sort=name`** |

---

### 실제 사용 예시

### 1. 상품 상세 페이지 (Parameter)

```
/products/123  ← 123이 상품 ID (Parameter)
```

특정 상품을 식별하기 위해 상품 ID를 경로에 포함합니다.

### 2. 상품 목록 + 필터링 (Query String)

```
/products?category=electronics&price_min=100&page=3
```

상품 목록 페이지에서 카테고리, 최소 가격, 페이지 번호 등의 옵션을 쿼리 스트링으로 전달합니다.

### 3. 둘 다 사용

```
/users/123/posts?status=published&limit=10
      ^^^              ^^^^^^^^^^^^^^^^^^^^^
   Parameter              Query String
```

- `123`은 특정 사용자를 식별하는 파라미터
- `status=published`와 `limit=10`은 게시글 목록을 필터링하는 쿼리 스트링

---

### 요약

- **URL Parameter(Path Parameter)**는 "무엇을" 보여줄지 결정하고, **Query String**은 "어떻게" 보여줄지 결정합니다.

**쉽게 말하면**:

- **대상은 경로(Path)에, 조건은 쿼리(Query)에**
- Path Parameter: **어떤 리소스인가?**
- Query String: **어떻게 보여줄 것인가?**

**사용 가이드**:

- 특정 리소스를 가리킬 때 → **Path Parameter** (예: `/users/123`)
- 리소스를 필터링하거나 옵션을 지정할 때 → **Query String** (예: `?sort=name&page=2`)

**참고**: Query String으로도 특정 결과를 만들 수 있지만, 리소스 자체를 식별할 때는 Path Parameter를 사용하는 것이 일반적인 관례입니다.

이 두 가지를 적절히 조합하면 명확하고 유연한 URL 구조를 설계할 수 있습니다.

## 1-03. #Quiz Parameter & Query String

### 문제 1

다음 URL에서 **Path Parameter(경로 파라미터)**에 해당하는 부분을 모두 고르세요.

```
https://example.com/users/456/posts/789?status=published&page=2
```

1. `456`
2. `789`
3. `status=published`
4. `page=2`
- 정답 및 해설
    
    정답:
    
    1번, 2번
    
    해설:
    
    `456`, `789`는 URL 경로에 포함되어 특정 리소스를 식별하는 **Path Parameter(경로 파라미터)**입니다. 반면 `status=published`, `page=2`는 `?` 뒤에 위치하여 조회 옵션을 지정하는 **Query String(Query Parameter)**입니다.
    
    Path Parameter는 URL 경로 자체에 포함되어 어떤 리소스를 조회할지 명확히 식별하는 데 사용됩니다. 예를 들어 `456`은 특정 사용자를, `789`는 특정 게시글을 가리킵니다.

### 문제 2

다음 중 **Query String을 사용하기에 가장 적절한** 상황은?

1. 특정 사용자의 프로필 페이지 표시
2. 상품 목록을 카테고리별로 필터링
3. 특정 게시글의 상세 내용 표시
4. 특정 주문의 상세 정보 조회
- 정답 및 해설
    
    정답:
    
    2번
    
    해설:
    
    Query String은 선택적인 옵션을 전달할 때 사용되며, 주로 필터링, 정렬, 페이지네이션 등에 적합합니다. 있어도 되고 없어도 되는 조건이기 때문에 같은 리소스를 다양한 방식으로 조회할 때 사용합니다.
    
    특정 사용자의 프로필, 게시글 상세, 주문 상세처럼 하나의 리소스를 정확히 식별해야 하는 경우에는 경로(Path)에 식별자를 포함하는 방식이 더 적절합니다. 예를 들어 `/users/123`, `/posts/456`, `/orders/789`과 같은 형태입니다.
    
    상품 목록을 카테고리별로 필터링하는 경우에는 목록이라는 동일한 리소스를 조건에 따라 다르게 보여주는 것이므로 Query String을 사용하는 것이 적절합니다. 예를 들면 `/products?category=electronics`와 같은 형태입니다.

### 문제 3

다음 URL 설계 중 **가장 적절한** 것은?

**상황**: 전자제품 카테고리의 상품 목록을 가격순으로 정렬하여 2페이지를 표시하고 싶습니다.

1. `/products/electronics/price/2`
2. `/products?category=electronics&sort=price&page=2`
3. `/products/electronics?price&page=2`
4. `/products?2?electronics?price`
- 정답 및 해설
    
    정답:
    
    2번 (`/products?category=electronics&sort=price&page=2`)
    
    해설:
    
    이 URL 구조는 기본 리소스 경로인 `/products`에 대해 모든 조건을 Query String으로 표현하고 있어 가장 적절합니다. `category=electronics`는 카테고리 필터, `sort=price`는 정렬 옵션, `page=2`는 페이지 번호를 의미합니다.
    
    이 조건들은 모두 선택적인 옵션이므로 Query String으로 표현하는 것이 바람직합니다. 경로에 옵션을 나열하는 방식은 조건이 늘어날수록 URL 구조가 복잡해지고 확장과 유지보수가 어려워집니다. 또한 `key=value` 형식을 지키지 않거나 `?`를 여러 번 사용하는 형태는 문법적으로 올바르지 않습니다.
    

## 1-04. Bundler ⭐

모던 웹 개발에서는 코드를 여러 파일로 나눠서 작성합니다. 하지만 브라우저는 이렇게 나눠진 파일들을 효율적으로 처리하는 데 한계가 있습니다. 이때 필요한 도구가 바로 **번들러(Bundler)**입니다.

번들러는 **여러 개의 파일과 모듈을 하나 또는 몇 개의 파일로 묶어주는 도구**입니다.

![스크린샷 2025-12-19 15.14.33.png](19%20React%20%EC%9B%B9%20%ED%8E%98%EC%9D%B4%EC%A7%80%20%EB%A7%8C%EB%93%A4%EA%B8%B0/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA_2025-12-19_15.14.33.png)

**Webpack에서 소개하는 번들러의 기본 개념**

---

### 번들러가 필요한 이유

### 1. 네트워크 워터폴(Network Waterfall) 현상

```jsx
// app.js
import { utils } from "./utils.js";
import { api } from "./api.js";
import React from "react";
```

브라우저가 ES6 모듈을 직접 로드할 때 발생하는 가장 큰 병목은 **의존성 깊이**입니다.

- **문제점:** `app.js`가 `utils.js`를 부르고, `utils.js`가 다시 `lodash`를 부르는 식으로 의존성이 꼬여있다면, 브라우저는 파일을 하나 읽을 때마다 다음 파일을 찾아야 함을 깨닫습니다.
- **결과:** 수백 개의 파일을 로드할 때 HTTP 요청이 계단식으로 발생하는 '워터폴' 현상이 생겨 초기 로딩 속도가 처참하게 느려집니다.
- **번들러의 해결:** 번들러는 이 의존성 그래프를 미리 분석해 **하나의 큰 덩어리(Bundle)**로 합쳐버립니다. 한 번의 요청으로 모든 로직을 가져오게 만드는 것이죠.

### 2. 트리 쉐이킹(Tree Shaking): "죽은 코드 제거"

최신 모듈 시스템의 가장 강력한 최적화 기능입니다.

- **내용:** 우리가 대형 라이브러리(예: `lodash`, `mui`)에서 함수 딱 하나만 꺼내 쓴다고 가정해 봅시다.
- **번들러의 역할:** 번들러는 빌드 시점에 코드를 정적 분석하여, **실제로 참조되지 않는 코드는 최종 결과물에서 아예 삭제**해 버립니다.
- **효과:** 실제 브라우저가 다운로드해야 할 자바스크립트 양을 획기적으로 줄여줍니다. 이는 브라우저 단독으로는 수행하기 어려운 고도의 최적화 작업입니다.
- ESM vs CJS

### 3. 트랜스파일링 및 폴리필(Transpilation & Polyfill)

모듈 시스템은 지원하지만, 최신 문법(Optional Chaining, Nullish Coalescing 등)은 지원하지 않는 브라우저가 있을 수 있습니다.

- **역할:** 번들러는 Babel 같은 도구와 협력하여 최신 자바스크립트 코드를 구형 브라우저도 이해할 수 있는 **ES5 수준의 코드로 변환**합니다.
- **필연성:** "모듈은 지원하는데 `??` 연산자는 모르는" 브라우저에서도 우리 앱이 터지지 않게 보장해 줍니다.

### 4. 비-자바스크립트 자원 관리 (Asset Graph)

현대 웹 개발에서 모듈 시스템은 단순히 `.js` 파일에 국한되지 않습니다.

- **통합 관리:** 번들러를 사용하면 `import "./style.css"`, `import logo from "./logo.svg"` 처럼 **CSS, 이미지, 폰트도 자바스크립트 모듈처럼 취급**할 수 있습니다.
- **최적화:** 이미지를 데이터 URI로 변환하거나, CSS를 하나로 합치고 난독화(Minify)하는 등의 작업을 한꺼번에 처리합니다.

---

### 번들러 동작 과정

번들러는 다음과 같은 순서로 동작합니다:

1. **Entry Point 분석**: `src/index.js` 같은 시작점 파일 확인
2. **의존성 그래프 생성**: import/require로 연결된 모든 파일 추적
    - ESM(ES Modules)은 import/export 구조가 정적이어서 번들러가 의존성을 분석하고 최적화(Tree Shaking)하기에 유리합니다
3. **파일 변환**: TypeScript, JSX, SCSS 등을 브라우저가 이해할 수 있는 형태로 변환
4. **번들 생성**: 하나 또는 여러 개의 최종 파일로 결합
5. **최적화**: 코드 압축(minify), 중복 제거, Tree Shaking

---

### 주요 번들러들

### 1. Webpack

가장 널리 사용되는 번들러로, 강력하고 다양한 기능을 제공합니다.

```jsx
// webpack.config.js
module.exports = {
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js",
  },
  module: {
    rules: [
      { test: /\.jsx?$/, use: "babel-loader" },
      {
        test: /\.css$/,
        use: ["style-loader", "css-loader"],
      },
    ],
  },
};
```

### 2. Vite

빠른 개발 경험을 제공하는 최신 빌드 도구입니다. 개발 중에는 번들링을 최소화하여 빠르게 실행하고, 빌드 시에는 내부적으로 Rollup을 사용해 번들링합니다.

```jsx
// vite.config.js
export default {
  build: {
    rollupOptions: {
      input: "src/main.js",
    },
  },
};
```

**개발 vs 빌드**:

- **개발 중**: ESM을 활용하여 번들링 없이 빠른 실행
- **빌드 시**: 최적화된 번들 파일 생성

### 3. Parcel (Zero-config)

설정 파일 없이도 동작하는 간편한 번들러입니다.

```bash
npx parcel src/index.html  # 설정 파일 없이도 동작
```

---

### 번들러와 빌드 도구의 역할 정리

프론트엔드 개발에서 사용하는 도구들의 역할을 명확히 구분하면 이해하기 쉽습니다:

- **번들러**: 여러 파일을 하나로 묶고 의존성을 관리 (Webpack, Rollup)
- **트랜스파일러**: 문법 변환 담당 (Babel, SWC, TypeScript Compiler)
- **빌드 도구**: 개발부터 배포까지 전체 흐름을 자동화 (Vite, Parcel)

최근의 도구들(Vite, Parcel 등)은 이러한 역할을 함께 제공하는 경우가 많습니다. 예를 들어 Vite는 개발 서버, 번들링, 최적화를 모두 담당하는 통합 빌드 도구입니다.

**개발 모드 vs 빌드 모드**:

- **개발 중**: 빠른 실행이 중요 → Vite는 번들링을 최소화
- **배포 시**: 최적화가 중요 → 번들링과 코드 압축 필수

---

### 번들링 결과 예시

### Before (개발 중):

```
src/
├── index.js
├── utils.js
├── api.js
├── components/
│   ├── Header.jsx
│   └── Footer.jsx
└── styles/
    └── main.css
```

개발할 때는 이렇게 여러 파일로 나눠서 작업합니다.

### After (빌드 후):

```
dist/
├── bundle.js     (모든 JS 파일 통합)
├── styles.css    (모든 CSS 파일 통합)
└── index.html
```

배포할 때는 번들러가 모든 파일을 묶어서 최적화된 형태로 만들어줍니다.

---

### 요약

번들러는 개발 시 모듈화된 코드를 작성할 수 있게 해주면서, 배포 시에는 브라우저가 효율적으로 로드할 수 있는 형태로 최적화해주는 필수 도구입니다.

모던 프론트엔드 개발에서 번들러는 선택이 아닌 필수입니다. React, Vue, Angular 등 대부분의 프레임워크는 내부적으로 번들러를 사용하고 있습니다.

**초급 단계에서 알아야 할 이유**: 초급 단계에서는 번들러를 직접 설정할 일은 많지 않지만, React나 Vue 프로젝트가 어떻게 동작하는지 이해하고, 빌드 과정에서 발생하는 문제를 해결하려면 번들러의 역할을 아는 것이 중요합니다.

    번들러는 여러 파일을 하나로 묶어 HTTP 요청 횟수를 줄이고, TypeScript나 JSX 같은 다양한 파일 형식을 브라우저가 이해할 수 있는 JavaScript로 변환하는 과정을 관리합니다. 또한 minify나 Tree Shaking과 같은 방식으로 코드 압축 및 최적화를 수행합니다. 반면 서버 사이드 렌더링 자체를 구현하는 것은 번들러의 역할이 아니며, 이는 Next.js 같은 프레임워크가 담당합니다. 번들러는 SSR을 위해 필요한 서버용 번들과 클라이언트용 번들을 만들어 주는 데 사용됩니다.

## 1-05. 프론트엔드 동작 방법 이해하기 ⭐⭐⭐⭐⭐

프론트엔드 애플리케이션은 사용자 경험과 성능 요구에 따라 다양한 구조와 렌더링 방식을 선택할 수 있습니다. 먼저 애플리케이션 구조를 이해하고, 그 위에서 어떤 렌더링 방식을 선택할 수 있는지 알아봅시다.

---

### 1. 싱글 페이지 애플리케이션 (SPA)

### 정의

**SPA는 렌더링 방식이 아니라**, 한 번의 HTML 로드 후 내부 라우팅으로 페이지 전환을 처리하는 **애플리케이션 구조**입니다. 대부분 CSR을 기반으로 동작하지만, SSR을 사용하는 SPA도 존재합니다.

### 동작

1. 최초 진입 시 `index.html`과 번들 파일 한 번 로드
2. 이후 `history.pushState` 기반으로 클라이언트 라우팅 수행

### 장점

- 빠른 내비게이션, 서버 요청 최소화
- 상태 관리(예: 전역 상태, 캐싱) 최적화 용이

### 단점

- 자바스크립트 오류 시 전체 앱이 영향을 받음
- 초기 로드 성능과 SEO 문제는 아래 설명할 렌더링 방식에 따라 달라짐

**참고**: SPA는 애플리케이션 구조이고, 아래에서 설명할 CSR, SSR, SSG는 렌더링 방식입니다. SPA 위에서 어떤 렌더링 방식을 선택하느냐에 따라 성능과 SEO 특성이 결정됩니다.

---

### 1-1. 클라이언트 사이드 렌더링 (CSR)

### 정의

초기 HTML은 최소한으로 로드되고, 자바스크립트가 실행되면서 데이터 요청 및 화면 구성을 수행하는 방식입니다. 대부분의 SPA가 이 방식을 사용합니다.

### 장점

- 사용자 상호작용이 부드럽고, 페이지 간 전환 시 전체 리로드 불필요
- 복잡한 UI 로직을 프론트엔드에서 자유롭게 구현 가능

### 단점

- 초기 로드 시 자바스크립트 번들 크기에 따라 로딩 속도 저하
- SEO를 적용하려면 추가 설정이나 작업이 필요함 (Google은 JavaScript 렌더링을 어느 정도 지원하지만, HTML이 바로 제공되는 방식보다는 불리함)

---

### 1-2. 서버 사이드 렌더링 (SSR)

### 정의

사용자가 요청할 때마다 서버에서 HTML을 완전히 렌더링하여 전달하는 방식입니다.

### 동작

1. 클라이언트 요청 → 서버에서 React/Vue 컴포넌트 렌더링
2. 완성된 HTML 전송 → 클라이언트가 하이드레이션(hydration) 수행

### 하이드레이션(Hydration)이란?  ⭐⭐⭐⭐⭐

SSR이나 SSG 방식에서는 서버가 먼저 HTML을 만들어서 브라우저에 전달합니다.

하지만 이 HTML은 **화면만 보일 뿐, 클릭이나 입력 같은 동작은 할 수 없습니다.**

이후 브라우저에서 자바스크립트가 로드되면, React는 기존 HTML을 다시 그리지 않고, 이 HTML에 **이벤트와 상태를 연결**합니다.

이 과정을 **하이드레이션(hydration)**이라고 하며, 간단히 말하면 **"이미 그려진 HTML에 React를 붙이는 과정"**입니다.

**CSR과 SSR + Hydration 차이**:

- **CSR**: JavaScript 실행 → 화면 렌더링 → 이벤트 바로 가능
- **SSR + Hydration**: HTML 먼저 표시 → JavaScript 로드 → 하이드레이션으로 이벤트 연결

**중요**: SSR이라고 해서 자바스크립트가 필요 없는 것은 아닙니다. React 애플리케이션에서는 이벤트와 상태 관리를 위해 반드시 자바스크립트가 필요합니다.

### 장점

- 초기 로드 시 완전한 HTML 전달로 빠른 화면 표시
- SEO 친화적

### 단점

- 서버 부하 증가(렌더링 비용)
- 최초 로드 시 서버 렌더링 비용 발생 (전통적인 SSR에서는 페이지 전환 시마다 서버 요청이 필요했으나, 현대 프레임워크에서는 다르게 동작)

### 보충 설명

전통적인 SSR(PHP, JSP 등)은 페이지를 이동할 때마다 서버에 새로운 페이지를 요청하는 방식입니다.

하지만 **Next.js**와 같은 최신 프론트엔드 프레임워크의 SSR은 다르게 동작합니다:

1. **첫 페이지 로드**: 사용자가 웹사이트에 처음 진입하면, 서버에서 완벽한 HTML을 렌더링하여 보내줍니다. (빠른 초기 로딩 + SEO 장점)
2. **이후 페이지 이동**: 그 후에는 SPA(싱글 페이지 애플리케이션)처럼 동작합니다. 즉, 페이지를 이동할 때 전체를 새로고침하는 것이 아니라, 자바스크립트를 이용해 필요한 데이터만 서버에 요청하고 변경된 부분만 클라이언트에서 렌더링합니다.

**정리**: 최신 SSR 프레임워크는 "첫 로드는 SSR, 이후 탐색은 SPA" 방식으로 동작하여 SSR의 장점(초기 로딩 속도, SEO)과 SPA의 장점(부드러운 사용자 경험)을 모두 결합한 형태에 가깝습니다.

Next.js에서 첫 페이지가 빠르게 보이는 이유는 서버가 HTML을 먼저 보내주기 때문이고, 그 이후 페이지 전환이 SPA처럼 동작하는 이유는 이미 하이드레이션이 완료된 React 앱이기 때문입니다.

**참고**: Next.js에서는 페이지 설정에 따라 SSR, SSG, CSR이 선택적으로 적용될 수 있습니다.

---

### 1-3. 정적 사이트 생성 (Static Site Generation, SSG)

### 정의

빌드 시점에 모든 페이지를 정적 HTML로 미리 생성하는 방식입니다.

### 동작

1. 빌드 단계에서 페이지별 데이터를 가져와 HTML 파일 생성
2. 클라이언트 요청 시 이미 생성된 HTML 파일 제공

### 장점

- CDN 캐싱과 결합하여 매우 빠른 초기 로드
- SEO 최적화
- 서버 부하 감소(빌드 타임 이후 서버 렌더링 불필요)

### 단점

- 빌드 시간 증가(페이지 수가 많을수록)
- 실시간 데이터 반영이 어려워, ISR(Incremental Static Regeneration) 등의 기법 필요

---

### 렌더링 방식 비교 요약

| **방식** | **초기 로드 속도** | **페이지 전환** | **SEO** | **서버 부하** |
| --- | --- | --- | --- | --- |
| CSR | 느림 | 빠름 (클라이언트) | 낮음 | 낮음 |
| SSR | 빠름 | 느림 (서버 요청) | 높음 | 높음 |
| SSG | 매우 빠름 | 매우 빠름 | 높음 | 매우 낮음 |

**참고**: 서버 부하는 캐싱 전략과 인프라 구성(CDN, Edge 서버 등)에 따라 크게 달라질 수 있습니다. SPA는 대부분 CSR 기반이므로 렌더링 특성이 유사합니다.

---

### 결론

적절한 방식을 선택하면 **성능**, **SEO**, **유지보수성** 사이에서 균형 있는 웹 애플리케이션을 구축할 수 있습니다.

- **CSR/SPA**: 복잡한 대시보드, 관리자 페이지 등 SEO가 중요하지 않은 경우
- **SSR**: 전자상거래, 뉴스 사이트 등 SEO와 초기 로딩 속도가 중요한 경우
- **SSG**: 블로그, 마케팅 페이지 등 내용이 자주 변하지 않는 경우

실제 프로젝트에서는 이러한 방식들을 **혼합**하여 사용하는 경우도 많습니다. 예를 들어, Next.js에서는 페이지별로 SSR, SSG, CSR을 선택적으로 적용할 수 있습니다.

**요즘 프레임워크의 특징**: SPA 구조 위에서 CSR, SSR, SSG를 상황에 따라 섞어 사용합니다.

**초급 단계에서는** "SPA 위에서 어떤 렌더링 방식을 쓰느냐"로 이해하면 충분합니다. 중요한 것은 **각 방식의 장단점을 이해하고, 프로젝트에 맞는 선택**을 할 수 있는 것입니다.

### 2-01. React Router 기본 라우팅

React 애플리케이션에서 여러 페이지 간 이동을 구현하려면 **React Router**가 필요합니다. React Router를 사용하면 페이지 전체를 새로고침하지 않고도 URL에 따라 다른 컴포넌트를 보여줄 수 있습니다. 이것이 바로 SPA(Single Page Application)의 핵심입니다.

---

### React Router란?

React Router는 React 애플리케이션에서 라우팅(페이지 이동)을 담당하는 라이브러리입니다.

**전통적인 웹사이트**에서는 다른 페이지로 이동할 때마다 서버에서 새로운 HTML 파일을 받아와 화면 전체가 새로고침됩니다. 하지만 **React Router**를 사용하면 URL만 바뀌고 필요한 컴포넌트만 교체되어 훨씬 빠르고 부드러운 사용자 경험을 제공할 수 있습니다.

---

### 설치


### 핵심 컴포넌트

### 1. BrowserRouter

애플리케이션의 최상위에서 라우팅 기능을 활성화합니다.

```jsx
import { BrowserRouter } from "react-router";

root.render(
  <BrowserRouter>
    <App />
  </BrowserRouter>
);
```

**역할:**

- URL이 변경될 때 **HTML5 History API**를 사용해 브라우저 주소창을 업데이트합니다.
- 페이지를 새로고침하지 않고 컴포넌트만 교체합니다.
- 라우팅이 필요한 모든 컴포넌트를 감싸야 합니다.

### 2. Routes와 Route

URL 경로에 따라 어떤 컴포넌트를 보여줄지 정의합니다.

```jsx
import { Routes, Route } from "react-router";

<Routes>
  <Route path="/" element={<HomePage />} />
  <Route path="/about" element={<AboutPage />} />
  <Route path="/contact" element={<ContactPage />} />
</Routes>;
```

- **Routes**: 여러 `Route`를 감싸는 컨테이너입니다.
- **Route**:
    - `path`: URL 경로
    - `element`: 해당 경로에서 렌더링할 컴포넌트

**동작 방식:**

1. 현재 URL을 확인합니다.
2. `Routes`는 현재 URL에 대해 **가장 잘 일치하는(가장 구체적인) Route를 선택**합니다.
3. 선택된 `Route`의 `element`를 렌더링합니다.

### 3. Link

페이지 이동을 위한 링크 컴포넌트입니다.

```jsx
import { Link } from "react-router";

<Link to="/">홈</Link>
<Link to="/about">소개</Link>
<Link to="/contact">연락처</Link>
```

**일반 `<a>` 태그와의 차이:**

| 특징 | `<a>` 태그 | `<Link>` |
| --- | --- | --- |
| 페이지 새로고침 | ✅ 발생 | ❌ 발생하지 않음 |
| 속도 | 느림 | 빠름 |
| 사용 목적 | 기본 이동(외부 링크, 파일 등) | SPA 내부 페이지 이동 |

`<Link>`는 클릭 시 페이지를 새로고침하지 않고 React Router가 `history.pushState`(History API)로 URL을 변경하고 컴포넌트를 교체합니다.

**언제 무엇을 사용할까?**

- SPA 내부 이동: `<Link to="/about">`
- 외부 사이트 이동: `<a href="https://google.com" target="_blank" rel="noopener noreferrer">`
- 파일 다운로드: `<a href="/file.pdf" download>`

- "연락처" 링크를 클릭하면 → `ContactPage` 컴포넌트가 표시됩니다.

---

### 동작 흐름

1. 사용자가 `<Link to="/about">`를 클릭합니다.
2. React Router가 `history.pushState`로 브라우저 URL을 `/about`으로 변경합니다. (페이지 새로고침 없음)
3. `<Routes>`가 현재 URL(`/about`)에 가장 잘 일치하는 `<Route>`를 찾습니다.
4. 해당 `Route`의 `element={<AboutPage />}`를 렌더링합니다.

**핵심**: React Router는 **SPA에서 URL을 '상태'처럼 다루는 도구**입니다. URL이 바뀌면 자동으로 적절한 컴포넌트가 렌더링됩니다.

---

### 요약

React Router의 기본 라우팅을 구현하려면:

1. **`react-router` 패키지 설치**
2. `BrowserRouter`로 앱 전체 감싸기 (main.jsx)
3. `Routes`와 `Route`로 경로와 컴포넌트 매핑
4. `Link`로 페이지 이동 링크 생성

이렇게 하면 페이지를 새로고침하지 않고도 부드럽게 페이지 간 이동이 가능한 SPA를 만들 수 있습니다.

### 2-02. 중첩 라우팅과 Outlet

웹 애플리케이션을 만들다 보면 특정 페이지 안에 또 다른 하위 페이지들이 필요한 경우가 많습니다. 예를 들어, **커뮤니티 페이지** 안에 **글 목록**, **글쓰기**, **내 게시글** 같은 하위 페이지들이 있을 수 있습니다. 이럴 때 사용하는 것이 **중첩 라우팅(Nested Routes)**입니다.

---

### 중첩 라우팅이란?

중첩 라우팅은 **URL 경로를 계층적으로 구성**하여 페이지 안에 페이지를 렌더링하는 방식입니다.

**참고**: 중첩 라우팅은 컴포넌트를 직접 중첩하는 것이 아니라, **Route 구조를 중첩하고 `Outlet`으로 렌더링 위치를 지정**하는 방식입니다.

**예시:**

```
/community              → 커뮤니티 메인
/community/new-post     → 커뮤니티 안의 글쓰기 페이지
/community/my-posts     → 커뮤니티 안의 내 게시글 페이지
```

이런 구조에서는:

- **공통 레이아웃**(헤더, 사이드바 등)은 유지됩니다.
- **내용 영역**만 URL에 따라 바뀝니다.

---

### 핵심 개념

### 1. 중첩된 Route 구조

부모 `Route` 안에 자식 `Route`를 중첩하여 계층을 만듭니다.

```jsx
<Routes>
  <Route path="/community" element={<CommunityPage />}>
    <Route index={true} element={<CommunityHomePage />} />
    <Route path="new-post" element={<NewPostPage />} />
    <Route path="my-posts" element={<MyPostsPage />} />
  </Route>
</Routes>
```

**URL과 렌더링:**

- `/community` → `CommunityPage` + `CommunityHomePage` (index)
- `/community/new-post` → `CommunityPage` + `NewPostPage`
- `/community/my-posts` → `CommunityPage` + `MyPostsPage`

### 2. Outlet 컴포넌트

- `Outlet`은 부모 컴포넌트에서 자식 라우트가 렌더링될 위치를 지정하는 특별한 컴포넌트입니다.

```jsx
import { Outlet } from "react-router";

export function CommunityPage() {
  return (
    <div>
      <h2>커뮤니티</h2>
      <nav>{/* 네비게이션 링크들 */}</nav>
      <hr />
      <Outlet /> {/* 자식 라우트가 여기에 렌더링됩니다 */}
    </div>
  );
}
```

**동작 방식:**

1. `/community/new-post`로 이동
2. `CommunityPage` 컴포넌트가 렌더링됨
3. `<Outlet />` 위치에 `NewPostPage` 컴포넌트가 렌더링됨

**중요**: 부모 컴포넌트에 `<Outlet />`이 없으면 자식 라우트는 화면에 렌더링되지 않습니다. `<Outlet />`은 자식 컴포넌트가 "갈 곳"을 지정해주는 역할입니다.

### 3. Index Route (기본 라우트)

`index` prop을 사용하면 부모 경로와 정확히 일치할 때 보여줄 기본 컴포넌트를 지정할 수 있습니다.

```jsx
<Route path="/community" element={<CommunityPage />}>
  {/* path 대신 index를 사용 */}
  <Route index={true} element={<CommunityHomePage />} />
</Route>
```

- `/community` 접속 시 → `CommunityHomePage`가 기본으로 표시됩니다.
- `index` 라우트는 **부모 Route의 경로와 정확히 일치할 때** 보여주는 '기본 자식 페이지'입니다.
- `index` 라우트는 **`path`가 없습니다**.

---

### NavLink의 활성 상태 스타일링

`NavLink`는 일반 `Link`와 달리 현재 경로와 일치하는지 자동으로 감지하여 **활성 상태**를 표시할 수 있습니다.

### className 함수로 동적 스타일 적용

```jsx
<NavLink
  to="/community"
  className={({ isActive }) => (isActive ? "active" : "")}
>
  글 목록
</NavLink>
```

- `className`에 함수를 전달하면 **`isActive` 매개변수**를 받을 수 있습니다.
- `isActive`는 현재 경로와 일치하면 `true`, 아니면 `false`입니다.
- 일치할 때 `'active'` 클래스를 추가하여 스타일을 다르게 표시할 수 있습니다.

**CSS 예시:**

```css
nav a {
  color: #666;
  text-decoration: none;
}

nav a.active {
  color: #000;
  font-weight: bold;
  border-bottom: 2px solid #000;
}
```

### NavLink의 end prop

```jsx
<NavLink to="/community" end={true}>
  글 목록
</NavLink>
```

- `end` prop이 **없으면**: `/community`, `/community/new-post` 모두에서 활성화되어서 CSS 클래스 `active`가 붙붙습니다.
- `end` prop이 **있으면**: **정확히** `/community`일 때만 CSS 클래스 `active` 가 붙습니다.

**언제 사용할까?**

- 부모 경로(`/community`)에 대한 링크는 `end={true}` 사용
- 자식 경로(`/community/new-post`)에 대한 링크는 `end` 생략

---

### 렌더링 흐름

1. 사용자가 `/community/new-post`로 이동
2. `CommunityPage` 컴포넌트 렌더링
3. `CommunityPage` 안의 `<Outlet />` 위치에 `NewPostPage` 렌더링
4. 결과적으로 **공통 레이아웃(헤더, 네비게이션) + 자식 페이지**가 함께 표시됨

---

### 중첩 라우팅의 장점

1. **코드 재사용**: 공통 레이아웃을 한 번만 작성
2. **명확한 구조**: URL만 봐도 페이지 계층 파악 가능
3. **유지보수 용이**: 레이아웃 변경 시 부모 컴포넌트만 수정

**참고**: 이런 공통 레이아웃을 제공하는 부모 Route를 보통 **'레이아웃 Route(Layout Route)'**라고 부릅니다.

---

### 요약

**중첩 라우팅 구현 3단계:**

1. **부모 Route 안에 자식 Route 중첩**
    
    ```jsx
    <Route path="community" element={<CommunityPage />}>
      <Route index element={<CommunityHomePage />} />
      <Route path="new-post" element={<NewPostPage />} />
    </Route>
    ```
    
2. **부모 컴포넌트에 Outlet 배치**
    
    ```jsx
    <Outlet />
    ```
    
3. **index 라우트로 기본 페이지 지정**
    
    ```jsx
    <Route index element={<CommunityHomePage />} />
    ```
    

이렇게 하면 **공통 레이아웃을 유지**하면서 **URL에 따라 내용만 바뀌는** 효율적인 구조를 만들 수 있습니다.

### 문제 1

다음 중첩 라우트 구조에서 `/community` 경로로 접속했을 때 **렌더링되는 컴포넌트**는?

```jsx

  <Route path="new-post" element={<NewPostPage />} />
</Route>
```

2. `CommunityHomePage`만 렌더링

3. 페이지를 새로고침하기 위해

- 정답 및 해설

### 2-03. 동적 라우팅과 useParams

실제 웹 애플리케이션에서는 수많은 게시글, 상품, 사용자 프로필 같은 데이터를 다룹니다. 각각에 대한 상세 페이지를 일일이 만들 수는 없습니다. 이럴 때 필요한 것이 **동적 라우팅(Dynamic Routes)**입니다.

---

### 동적 라우팅이란?

동적 라우팅은 **URL의 일부를 변수처럼 사용**하여 하나의 컴포넌트로 여러 페이지를 처리하는 방식입니다.

**예시:**

```
/posts/1   → 1번 게시글 상세 페이지
/posts/2   → 2번 게시글 상세 페이지
/posts/123 → 123번 게시글 상세 페이지
```

모두 같은 `PostDetailPage` 컴포넌트를 사용하지만, URL의 숫자에 따라 다른 데이터를 보여줍니다.

---

### 핵심 개념

### 1. 동적 세그먼트 (Dynamic Segment)

경로에 **콜론(`:`)** 을 사용하여 변수를 정의합니다.

```jsx
<Route path="/posts/:postId" element={<PostDetailPage />} />
```

- `:postId` 부분이 **동적 세그먼트**입니다.
- `postId`는 변수명으로, 원하는 이름을 사용할 수 있습니다.
- 이 라우트는 `/posts/1`, `/posts/abc`, `/posts/hello` 등과 매칭됩니다.

**참고**: 같은 위치에 정적 경로(`/posts/new` 등)가 있다면 **정적 경로가 동적 경로보다 우선 매칭**됩니다. React Router v6+는 정적 > 동적 > 와일드카드(*) 순서로 매칭합니다.

**여러 개의 동적 세그먼트:**

```jsx
<Route
  path="/users/:userId/posts/:postId"
  element={<UserPost />}
/>
```

- `/users/john/posts/5` → `{ userId: 'john', postId: '5' }`

**참고**: 중첩 라우트 안에서도 `useParams`는 부모와 자식 라우트의 모든 파라미터를 함께 반환합니다.

### 2. useParams Hook

- **`useParams`**는 URL의 동적 세그먼트 값을 가져오는 React Router Hook입니다.

**참고**: 실제로는 URL 문자열을 React Router가 파싱해서 객체 형태로 전달해줍니다.

```jsx
import { useParams } from "react-router";

function PostDetailPage() {
  const { postId } = useParams();
  // URL이 /posts/123 이면 postId는 "123"

  return <h1>게시글 {postId}</h1>;
}
```

**주의사항:**

- `useParams`가 반환하는 값은 보통 **문자열**입니다 (매칭이 안 되면 `undefined`일 수 있음).
- 숫자가 필요하면 `Number(postId)` 또는 `parseInt(postId, 10)`로 변환해야 합니다.

---

### 실습: 동적 라우팅 구현하기

### 1단계: `App.jsx`에 동적 라우트 설정하기

`App.jsx` 파일에서 게시글의 **`id`**에 따라 상세 페이지를 보여줄 수 있도록 동적 라우트를 설정합니다.

```jsx
// src/App.jsx (수정)

import { Routes, Route, Navigate } from "react-router";
import { PostPage } from "./features/post/PostPage";
import { PostDetailPage } from "./features/post/PostDetail";

export function App() {
  return (
    <Routes>
      <Route
        path="/"
        element={<Navigate to="/posts" replace />}
      />
      <Route path="/posts" element={<PostPage />} />
      <Route
        path="/posts/:postId"
        element={<PostDetailPage />}
      />
      {/* 추가 */}
    </Routes>
  );
}
```

**설명:**

- `/posts/:postId` 경로는 `/posts/1`, `/posts/2` 같은 모든 URL과 매칭됩니다.
- `:postId` 부분이 동적 세그먼트로, URL의 값이 변수처럼 전달됩니다.

### 2단계: `PostDetailPage`에서 `useParams`로 게시글 ID 가져오기

`PostDetailPage.jsx`에서 `useParams`를 사용해 URL로부터 `postId`를 가져와, 해당 ID를 가진 게시글 정보를 화면에 렌더링합니다.

```jsx
// src/features/post/PostDetail/PostDetailPage.jsx (수정)

import { useParams, Link } from "react-router";
import styles from "./PostDetail.module.css";
import { posts } from "@/data/posts";

export function PostDetailPage() {
  const { postId } = useParams();

  const post = posts.find((p) => p.id === postId);

  if (!post) {
    return <div>게시글을 찾을 수 없습니다.</div>;
  }

  return (
    <div className={styles.container}>
      <h1>{post.title}</h1>
      <main>
        <p>{post.content}</p>
        <Link to="/posts">목록으로</Link>
      </main>
    </div>
  );
}

```

**동작 흐름:**

1. URL이 `/posts/1`이면 `useParams()`는 `{ postId: "1" }`을 반환합니다.
2. `posts.find()`로 `id`가 `"1"`인 게시글을 찾습니다.
3. 찾은 게시글의 정보를 화면에 표시합니다.

### 확인하기

브라우저에서 애플리케이션을 실행하고 테스트해보세요:

1. `/posts` 페이지에서 게시글 목록을 확인합니다.
2. 특정 게시글을 클릭하면 URL이 `/posts/1` (또는 다른 ID)로 변경됩니다.
3. `PostDetailPage`에서 해당 게시글의 상세 내용이 표시됩니다.
4. 브라우저 주소창에 직접 `/posts/2`, `/posts/3` 등을 입력해도 해당 게시글이 표시됩니다.

**주의사항:**

- `useParams`로 가져온 `postId`는 **문자열**입니다.
- 만약 데이터의 `id`가 숫자라면 `Number(postId)` 또는 `parseInt(postId, 10)`로 변환해야 합니다.

---

### 동작 흐름

1. 사용자가 "React 시작하기" 링크 클릭
2. `<Link to="/posts/1">` → URL이 `/posts/1`로 변경
3. React Router가 `/posts/:postId` 라우트와 매칭
4. `PostDetailPage` 컴포넌트 렌더링
5. `useParams()`가 `{ postId: "1" }` 반환
6. `posts.find(p => p.id === "1")`로 해당 게시글 찾기
7. 게시글 내용 화면에 표시

---

### 요약

**동적 라우팅 구현 3단계:**

1. **경로에 `:변수명` 형태로 동적 세그먼트 정의**
    
    ```jsx
    <Route
      path="/posts/:postId"
      element={<PostDetailPage />}
    />
    ```
    
2. **컴포넌트에서 useParams로 값 가져오기**
    
    ```jsx
    const { postId } = useParams();
    ```
    
3. **가져온 값으로 데이터 조회 및 렌더링**
    
    ```jsx
    const post = posts.find((p) => p.id === postId);
    ```
    

이렇게 하면 **하나의 컴포넌트로 무한대의 페이지를 처리**할 수 있습니다!

### 문제 1

다음 라우트 설정에서 `/products/123` URL로 접속했을 때, `useParams()`가 반환하는 값은?

```jsx
<Route
  path="/products/:productId"
  element={<ProductDetail />}
/>
```

1. `{ productId: 123 }`
2. `{ productId: "123" }`
3. `"123"`
4. `123`
- 정답 및 해설
    
    정답:
    
    2번 (`{ productId: "123" }`)
    
    해설:
    
    `useParams()`는 항상 객체 형태로 값을 반환하며, 값은 보통 문자열(string) 타입입니다. 매칭이 안 되면 `undefined`일 수 있습니다.
    
    ```jsx
    const { productId } = useParams();
    console.log(productId); // "123" (문자열)
    console.log(typeof productId); // "string"
    ```
    
    숫자가 필요하다면 변환해야 합니다.
    
    ```jsx
    const productIdNum = Number(productId); // 123 (숫자)
    // 또는
    const productIdNum = parseInt(productId, 10); // 123 (숫자)
    
    ```
    
    `parseInt`를 사용할 때는 radix(진법) `10`을 명시하는 것이 안전합니다.

### 문제 2

다음 코드에서 **잘못된 부분**은?

```jsx
const posts = [
  { id: 1, title: "게시글 1" },
  { id: 2, title: "게시글 2" },
];

function PostDetail() {
  const { postId } = useParams(); // postId는 "1" (문자열)
  const post = posts.find((p) => p.id === postId);

  return <h1>{post.title}</h1>;
}
```

1. `useParams` 사용 방법
2. `posts.find` 사용 방법
3. 비교 연산자 (`===`)의 타입 불일치
4. 문제없음
- 정답 및 해설
    
    정답:
    
    3번 (타입 불일치)
    
    해설:
    
    `postId`는 문자열 `"1"`이고, `posts`의 `id`는 숫자 `1`이므로 `"1" === 1`은 `false`입니다. 그 결과 `post`는 `undefined`가 되고, `post.title`에서 런타임 에러가 발생합니다.
    
    해결하려면 타입을 맞추고, `post`가 없을 때를 대비한 가드 처리가 필요합니다.
    
    ```jsx
    const post = posts.find((p) => p.id === Number(postId));
    
    if (!post) {
      return <div>게시글이 없습니다.</div>;
    }
    
    return <h1>{post.title}</h1>;
    ```
    
    또는 문자열로 맞출 수도 있습니다.
    
    ```jsx
    const post = posts.find((p) => String(p.id) === postId);
    
    if (!post) {
      return <div>게시글이 없습니다.</div>;
    }
    
    return <h1>{post.title}</h1>;
    ```

다음 라우트 설정에서 URL이 `/users/john/posts/5`일 때, 컴포넌트에서 값을 가져오는 **올바른 방법**은?

```jsx
  element={<UserPost />}
```

1. `const { userId } = useParams()`
2. `const { postId } = useParams()`

    3번 (`const { userId, postId } = useParams()`)

### Not Found 페이지와 리다이렉트

웹 애플리케이션에서는 존재하지 않는 URL로 접근하거나, 과거에 사용했던 URL 구조를 변경하는 경우가 있습니다. 이때 **404 페이지**로 안내하거나 **리다이렉트**로 올바른 페이지로 자동 이동시킬 수 있습니다.

---

### 404 Not Found 페이지: `path="*"`

### Catch-all Route

React Router에서는 **`path="*"`**를 사용하여 정의되지 않은 모든 경로를 처리할 수 있습니다.

```jsx
import { Routes, Route } from "react-router";

function App() {
  return (
    <Routes>
      <Route path="/" element={<HomePage />} />
      <Route path="/about" element={<AboutPage />} />

      {/* 위의 경로와 일치하지 않으면 NotFoundPage 렌더링 */}
      <Route path="*" element={<NotFoundPage />} />
    </Routes>
  );
}
```

**동작 방식:**

- React Router는 URL과 가장 잘 일치하는 경로를 선택합니다.
- `/`, `/about` 등 정의된 경로가 있으면 해당 컴포넌트를 렌더링합니다.
- **어떤 경로와도 일치하지 않으면** `path="*"`가 선택되어 `NotFoundPage`를 렌더링합니다.
- **`path="*"`는 `Routes`의 가장 마지막에 작성**하는 것이 권장됩니다.

**참고**: React Router v6+에서는 `path="*"`의 매칭 우선순위가 가장 낮기 때문에, 중간에 있어도 다른 구체적인 경로가 먼저 매칭됩니다. 하지만 **코드 가독성을 위해 마지막에 두는 것이 관례입니다.**

---

### 리다이렉트: Navigate 컴포넌트

**리다이렉트**는 사용자가 특정 경로로 접근했을 때 자동으로 다른 경로로 이동시키는 것입니다. 예를 들어, 과거 URL 구조를 변경했을 때 이전 URL로 접근한 사용자를 새 URL로 자동 이동시킬 수 있습니다.

### Navigate 컴포넌트

```jsx
import { Routes, Route, Navigate } from "react-router";

function App() {
  return (
    <Routes>
      <Route path="/" element={<HomePage />} />
      <Route path="/profile" element={<ProfilePage />} />

      {/* /legacy-profile로 접근하면 /profile로 리다이렉트 */}
      <Route
        path="/legacy-profile"
        element={<Navigate to="/profile" replace={true} />}
      />
    </Routes>
  );
}
```

**동작:**

1. 사용자가 `/legacy-profile`로 접근
2. `<Navigate to="/profile" replace={true} />` 컴포넌트가 렌더링됨
3. 즉시 `/profile`로 자동 이동
4. `ProfilePage` 컴포넌트가 화면에 표시됨

### replace prop

`replace` prop은 브라우저 히스토리 관리를 위해 사용됩니다.

**replace 없이:**

```jsx
<Navigate to="/profile" />
```

- 히스토리: `["/", "/legacy-profile", "/profile"]`
- 뒤로 가기 → `/legacy-profile`로 이동 → 다시 `/profile`로 리다이렉트 (무한 루프!)

**replace 사용:**

```jsx
<Navigate to="/profile" replace={true} />
```

- 히스토리: `["/", "/profile"]`
- `/legacy-profile`은 히스토리에 남지 않음
- 뒤로 가기 → `/`(홈)으로 정상 이동

**대부분의 리다이렉트에서는 `replace` 사용이 권장됩니다.**

### 실습: 404 페이지와 리다이렉트 구현하기

### 1단계: 리다이렉트 구현하기

과거에 사용했던 **`/legacy-profile`** 경로를 새로운 **`/profile`** 경로로 리다이렉트 시켜봅시다.

```jsx
// src/App.jsx (수정)

import { Routes, Route, Navigate } from "react-router";
import { HomePage } from "./pages/Home";
import { ProfilePage } from "./pages/Profile";
import { NotFoundPage } from "./pages/NotFound";

export function App() {
  return (
    <Routes>
      <Route path="/" element={<HomePage />} />
      <Route path="/profile" element={<ProfilePage />} />

      {/* 1. 리다이렉트 추가 */}
      <Route
        path="/legacy-profile"
        element={<Navigate to="/profile" replace={true} />}
      />

      {/* 아직 404는 추가하지 않음 */}
    </Routes>
  );
}
```

**설명:**

- `/legacy-profile`로 접속하면 자동으로 `/profile`로 이동합니다.
- `replace={true}`를 사용하여 브라우저 히스토리에 `/legacy-profile`을 남기지 않습니다.
- 뒤로 가기를 눌렀을 때 `/legacy-profile`로 돌아가지 않고 이전 페이지로 이동합니다.

### 2단계: Not Found 페이지 구현하기

정의된 경로 외의 모든 경로로 접근 시 **`NotFoundPage`**를 보여주도록 설정합니다. **`path="*"`** 라우트는 항상 **`Routes`**의 가장 마지막에 위치해야 합니다.

```jsx
// src/App.jsx (수정)

export function App() {
  return (
    <Routes>
      {/* 정적 경로 */}
      <Route path="/" element={<HomePage />} />
      <Route path="/profile" element={<ProfilePage />} />

      {/* 리다이렉트 */}
      <Route
        path="/legacy-profile"
        element={<Navigate to="/profile" replace={true} />}
      />

      {/* 2. 404 Not Found - 가장 마지막 */}
      <Route path="*" element={<NotFoundPage />} />
    </Routes>
  );
}
```

**전체 코드:**

```jsx

import { HomePage } from "./pages/Home";
import { ProfilePage } from "./pages/Profile";

    <Routes>
      <Route path="/" element={<HomePage />} />
      <Route path="/profile" element={<ProfilePage />} />
      <Route

        element={<Navigate to="/profile" replace={true} />}
      />
      <Route path="*" element={<NotFoundPage />} />

```

브라우저에서 다음 URL로 접속하여 동작을 확인해보세요:

1. **`/`** → `HomePage` 렌더링
2. **`/profile`** → `ProfilePage` 렌더링
3. **`/legacy-profile`** → `/profile`로 리다이렉트 후 `ProfilePage` 렌더링
    - 브라우저 주소창 URL이 `/profile`로 변경됨
    - 뒤로 가기를 눌렀을 때 `/legacy-profile`로 돌아가지 않음
4. **`/asdf`** (존재하지 않는 경로) → `NotFoundPage` 렌더링 (404)

**동작 테스트:**

- 홈 페이지(`/`)에서 `/legacy-profile`로 이동 → 자동으로 `/profile`로 리다이렉트
- 뒤로 가기 버튼 클릭 → `/legacy-profile`을 건너뛰고 홈(`/`)으로 이동
- `/random-url`처럼 존재하지 않는 경로 입력 → 404 페이지 표시

### 요약

**404 페이지 (`path="*"`)**

- 정의되지 않은 모든 경로를 처리하는 특수 라우트
- `Routes`의 가장 마지막에 배치

**리다이렉트 (`Navigate`)**

- 특정 경로로 접근 시 다른 경로로 자동 이동
- `replace` prop으로 히스토리 관리 (권장)

**라우트 작성 순서**

```jsx
<Routes>
  {/* 1. 정적 경로 */}
  <Route path="/" element={<HomePage />} />

  {/* 2. 리다이렉트 */}
  <Route
    path="/old"
    element={<Navigate to="/new" replace />}
  />

  {/* 3. 404 - 가장 마지막 */}
  <Route path="*" element={<NotFoundPage />} />
</Routes>
```

### 문제 1

다음 코드에서 사용자가 `/unknown-page` URL로 접근했을 때 어떤 컴포넌트가 렌더링되나요?

```jsx
function App() {
  return (
    <Routes>
      <Route path="/" element={<HomePage />} />
      <Route path="/about" element={<AboutPage />} />
      <Route path="*" element={<NotFoundPage />} />
      <Route path="/contact" element={<ContactPage />} />
    </Routes>
  );
}
```

- 정답 및 해설
    
    정답:
    
    `NotFoundPage` 컴포넌트가 렌더링됩니다.
    
    해설:
    
    `path="*"`는 다른 어떤 경로와도 일치하지 않을 때 매칭되는 Catch-all 라우트입니다. `/unknown-page`는 `/`, `/about`, `/contact` 어느 경로에도 해당하지 않기 때문에 `path="*"`와 일치하여 `NotFoundPage`가 렌더링됩니다. 관례적으로는 가독성을 위해 `path="*"`를 `Routes`의 마지막에 두는 것이 권장되지만, React Router v6 이상에서는 Route Ranking 시스템을 사용하므로 중간에 있어도 `/contact`처럼 더 구체적인 경로가 우선 매칭됩니다.

### 문제 2

다음 코드에서 사용자가 `/old-url`로 접근한 후 **뒤로 가기 버튼**을 눌렀을 때 어떤 일이 발생하나요? (처음에 `/` 페이지에서 시작했다고 가정)

```jsx
<Routes>
  <Route path="/" element={<HomePage />} />
  <Route path="/new-url" element={<NewPage />} />
  <Route
    path="/old-url"
    element={<Navigate to="/new-url" />}
  />
</Routes>
```

- 정답 및 해설
    
    정답:
    
    `/old-url`로 돌아가면 다시 `/new-url`로 리다이렉트되어 뒤로 가기가 막힌 것처럼 보이는 현상이 발생합니다.
    
    해설:
    
    `replace` prop을 사용하지 않으면 브라우저 히스토리에 `/old-url`이 그대로 남습니다. 그 결과 히스토리는 다음과 같이 쌓입니다.
    
    ```
    ["/", "/old-url", "/new-url"]
    ```
    
    뒤로 가기를 누르면 `/old-url`로 이동하게 되는데, 이 경로는 `<Navigate to="/new-url" />`를 렌더링하므로 즉시 `/new-url`로 다시 이동합니다. 이 과정이 반복되면 사용자는 뒤로 가기가 동작하지 않는 것처럼 느끼게 됩니다. 이를 해결하려면 `replace` prop을 추가하여 히스토리 스택에 `/old-url`이 남지 않도록 해야 합니다.
    
    ```jsx
    <Route
      path="/old-url"
      element={<Navigate to="/new-url" replace />}
    />
    ```
    
    이렇게 하면 히스토리는 `["/", "/new-url"]`가 되어, 뒤로 가기를 눌렀을 때 정상적으로 홈(`/`)으로 돌아갑니다.

### 문제 3

다음 코드에서 `replace` prop의 역할은 무엇인가요?

```jsx
<Route
  path="/legacy-profile"
  element={<Navigate to="/profile" replace />}
/>
```

- 정답 및 해설
    
    정답:
    
    현재 경로(`/legacy-profile`)를 브라우저 히스토리 스택에 추가하지 않고, 새로운 경로(`/profile`)로 대체합니다.
    
    해설:
    
    `replace={true}`를 사용하면 기존 히스토리 항목을 새 경로로 교체합니다. 이를 사용하지 않으면 히스토리에 `/legacy-profile`과 `/profile`이 모두 남아 뒤로 가기 시 다시 `/legacy-profile`로 이동하게 됩니다. 반면 `replace`를 사용하면 히스토리에 `/profile`만 남아 뒤로 가기 시 이전 페이지로 자연스럽게 돌아갑니다. 대부분의 리다이렉트 상황에서는 `replace`를 사용하는 것이 권장되며, 로그인 후 원래 페이지로 복귀해야 하는 경우처럼 사용자 이동 기록을 남겨야 하는 특별한 상황에서만 `replace`를 생략합니다.

### URL 쿼리 파라미터를 이용한 검색 및 정렬

쿠팡에서 "노트북"을 검색하면 URL이 `https://www.coupang.com/np/search?q=노트북`처럼 바뀝니다. 이렇게 검색어와 정렬 옵션을 **URL의 쿼리 파라미터(Query Parameters)**에 저장하면 페이지를 새로고침해도 검색 결과가 유지되고, URL을 공유할 수도 있습니다.

### URL 쿼리 파라미터란?

```
https://example.com/search?q=React&sort=asc
                          ↑  └────────────┘
                        시작점   쿼리 파라미터
```

**쿼리 파라미터의 특징:**

- `?` 뒤에 시작
- `key=value` 형태
- 여러 개는 `&`로 연결
- 페이지의 상태를 URL에 표현

**경로 파라미터 vs 쿼리 파라미터:**

- **경로 파라미터**: 리소스 식별 → `/products/123`
- **쿼리 파라미터**: 상태/옵션 표현 → `/search?q=React&sort=asc`

### useSearchParams 훅

React Router는 쿼리 파라미터를 다루기 위해 `useSearchParams` 훅을 제공합니다.

```jsx
import { useSearchParams } from "react-router";

function SearchPage() {
  const [searchParams, setSearchParams] = useSearchParams();

  // URL에서 값 읽기
  const query = searchParams.get("q");
  const sort = searchParams.get("sort");

  // URL 업데이트
  setSearchParams({ q: "React", sort: "asc" });

  return <div>검색어: {query}</div>;
}
```

**동작:**

- `searchParams.get("q")` → URL에서 `q` 값 읽기
- `setSearchParams({ ... })` → URL 쿼리 파라미터 업데이트
- `useState`와 비슷하지만 **URL과 동기화**됨

**여러 파라미터 유지:**

정렬 변경 시 검색어를 유지하려면:

```jsx
// 기존 검색어 유지하고 정렬만 변경
setSearchParams({ q: query || "", sort: "asc" });
```

### 실습: 검색 및 정렬 기능 구현하기

### 1단계: `useSearchParams`로 URL 파라미터 읽기

`Search.jsx`에서 `useSearchParams`를 호출하고, `get` 메서드를 사용해 `q`와 `sort` 파라미터 값을 가져옵니다.

```jsx
// src/pages/Search/Search.jsx (수정)

import { useSearchParams } from "react-router";

export function Search() {
  // 1. useSearchParams 호출
  const [searchParams, setSearchParams] = useSearchParams();
  const query = searchParams.get("q");
  const sort = searchParams.get("sort");

  // ...
}
```

### 2단계: 검색 기능 구현하기

검색 form의 `onSubmit` 이벤트 핸들러인 `handleSubmit` 함수를 완성하여, 입력된 검색어로 `q` 파라미터를 업데이트합니다.

```jsx
// src/pages/Search/Search.jsx (수정)

const handleSubmit = (event) => {
  event.preventDefault();

  // 2. FormData로 검색어 가져오기
  const formData = new FormData(event.target);
  const searchQuery = formData.get("q");
  const nextParams = {};

  if (searchQuery) {
    nextParams.q = searchQuery;
  }

  setSearchParams(nextParams);
};
```

**설명:**

- `FormData`를 사용해 폼 입력값을 가져옵니다.
- 검색어가 있을 때만 `q` 파라미터를 설정합니다.
- 검색어가 없으면 빈 객체로 URL을 초기화합니다.

### 3단계: 정렬 기능 구현하기

정렬 버튼의 `onClick` 이벤트 핸들러인 `handleSort` 함수를 완성하여, `sort` 파라미터를 업데이트합니다. 이때 기존 `q` 파라미터는 유지해야 합니다.

```jsx
// src/pages/Search/Search.jsx (수정)

// 3. 정렬 핸들러 구현
const handleSort = (sort) => {
  setSearchParams({ q: query || "", sort });
};
```

**설명:**

- 정렬을 변경할 때 기존 검색어(`q`)를 유지합니다.
- `query || ""`로 검색어가 없을 때도 안전하게 처리합니다.

### 4단계: 데이터 필터링 및 정렬, 조건부 렌더링

`query`와 `sort` 값에 따라 `questions` 배열을 처리하고, 그 결과에 따라 적절한 UI를 보여주는 로직을 완성합니다.

```jsx
// src/pages/Search/Search.jsx (수정)

// 4. 필터링 및 정렬 로직
const filteredQuestions = query
  ? questions
      .filter((q) =>
        q.toLowerCase().includes(query.toLowerCase())
      )
      .sort((a, b) => {
        if (sort === "asc") return a.localeCompare(b);
        return b.localeCompare(a);
      })
  : [];

// 5. 조건부 렌더링
return (
  <div>
    {/* ... 검색 폼 및 정렬 버튼 ... */}

    <h3>검색 결과</h3>
    {!query ? (
      <p>검색어를 입력해주세요.</p>
    ) : filteredQuestions.length === 0 ? (
      <p>{query}에 대한 검색 결과가 없습니다.</p>
    ) : (
      <ul>
        {filteredQuestions.map((q) => (
          <li key={q}>{q}</li>
        ))}
      </ul>
    )}
  </div>
);
```

**설명:**

- 검색어가 있으면 필터링하고 정렬합니다.
- 검색어가 없으면 빈 배열을 반환합니다.
- 3가지 상태(검색 전, 결과 없음, 결과 있음)를 조건부 렌더링으로 처리합니다.

### 확인하기

브라우저에서 테스트해보세요:

1. **검색어 입력** → URL이 `/search?q=검색어`로 변경
2. **정렬 버튼 클릭** → URL이 `/search?q=검색어&sort=asc`로 변경
3. **페이지 새로고침** → 검색어와 정렬 상태가 유지됨
4. **URL 복사 및 새 탭에서 열기** → 동일한 검색 결과 표시

## 2-05. #Quiz Search Params

### 문제 1

다음 코드에서 URL이 `/search?q=JavaScript&sort=desc`일 때, `query`와 `sort` 변수의 값은 각각 무엇인가요?

```jsx
function SearchPage() {
  const [searchParams] = useSearchParams();
  const query = searchParams.get("q");
  const sort = searchParams.get("sort");

  console.log(query, sort);
}
```

- 정답 및 해설
    
    정답:
    
    `query`는 `"JavaScript"`, `sort`는 `"desc"`입니다.
    
    해설:
    
    `useSearchParams` 훅에서 반환된 `searchParams` 객체의 `get()` 메서드는 URL의 쿼리 파라미터 값을 가져옵니다.
    
    `searchParams.get("q")`는 URL의 `q` 파라미터 값인 `"JavaScript"`를 반환하고, `searchParams.get("sort")`는 URL의 `sort` 파라미터 값인 `"desc"`를 반환합니다.
    
    존재하지 않는 파라미터를 `get()`하면 `null`을 반환합니다. 예를 들어 URL에 `page` 파라미터가 없는 경우 `searchParams.get("page")`는 `null`입니다.
    
    쿼리 값은 URL 인코딩되어 전달되며, `get()`은 디코딩된 문자열을 반환합니다. 예를 들어 `JavaScript%20Basics`는 `"JavaScript Basics"`로 반환됩니다.

    URL이 `/search?sort=desc`로 변경됩니다. 기존 `q` 파라미터는 사라집니다.
    
    해설:
    
    `setSearchParams({ sort: sortType })`는 모든 쿼리 파라미터를 새로운 객체로 대체합니다. 따라서 기존에 있던 `q=React` 파라미터가 제거되고 `sort=desc`만 남게 됩니다.
    
    이로 인해 검색어가 사라져 사용자 경험이 나빠질 수 있습니다.
    
    이를 해결하려면 기존 파라미터를 명시적으로 포함하거나, 기존 파라미터를 기반으로 병합하는 방식을 사용해야 합니다.
    
    기존 파라미터를 명시적으로 포함하는 방법은 다음과 같습니다.
    
    ```jsx
    const handleClickButton = (sortType) => {
      setSearchParams({
        q: query || "",
        sort: sortType,
      });
    };
    ```
    
    기존 파라미터를 기반으로 병합하는 방법은 다음과 같습니다.
    
    ```jsx
    const handleClickButton = (sortType) => {
      setSearchParams((prev) => {
        const params = new URLSearchParams(prev);
        params.set("sort", sortType);
        return params;
      });
    };
    ```
    
    이 방법을 사용하면 파라미터가 많아져도 일부만 변경하기 쉽고, 실수로 파라미터를 누락하는 경우를 방지할 수 있습니다. 이렇게 수정하면 URL은 `/search?q=React&sort=desc`로 변경됩니다.

### 문제 3

다음 코드에서 `defaultValue`를 사용하는 이유는 무엇인가요?

```jsx
function SearchPage() {
  const [searchParams, setSearchParams] = useSearchParams();
  const query = searchParams.get("q");

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="q"
        type="text"
        defaultValue={query || ""}
      />
      <button type="submit">검색</button>
    </form>
  );
}
```

- 정답 및 해설
    
    정답:
    
    URL의 검색어를 input의 초기값으로 설정하여, URL 공유나 페이지 새로고침 시에도 검색어가 input에 표시되도록 하기 위함입니다.
    
    해설:
    
    `defaultValue={query || ""}`는 input의 초기값을 URL의 `q` 파라미터 값으로 설정합니다.
    
    이로 인해 URL 공유 시 `/search?q=React` 링크를 열면 input에 `"React"`가 자동으로 표시되고, 페이지를 새로고침해도 검색어가 유지됩니다.
    
    사용자는 현재 어떤 검색어로 검색했는지 명확하게 확인할 수 있습니다.
    
    `defaultValue`는 초기값만 설정하는 비제어 컴포넌트 방식이며, `value`는 항상 상태와 동기화되는 제어 컴포넌트 방식입니다.
    
    검색 input은 폼 제출 시에만 값을 읽으면 되는 경우가 많기 때문에, 굳이 제어 컴포넌트로 만들 필요 없이 `defaultValue`를 사용하는 것이 간단합니다.
    
    다만 `defaultValue`는 초기 렌더링 시에만 적용되므로, URL의 `q` 값이 변경되어도 input 값은 자동으로 업데이트되지 않습니다.
    
    URL 변경 시에도 input과 항상 동기화가 필요하다면 `value`와 `onChange`를 사용하는 방식이나, `key`를 사용해 input을 재마운트하는 패턴을 선택할 수 있습니다.
    
    `defaultValue`를 사용하지 않으면 URL에 검색어가 포함되어 있어도 input이 비어 있어 사용자가 현재 검색어를 확인하거나 수정하기 어려워집니다.

### 들어가며

지금까지 `<Link>` 컴포넌트로 페이지를 이동했습니다. 하지만 실무에서는 **폼 제출 후**, **로그인 성공 후** 같이 클릭이 아닌 특정 로직 실행 후 자동으로 페이지를 이동해야 할 때가 많습니다. 이런 경우에는 **`useNavigate`** 훅을 사용하여 **프로그래밍 방식의 네비게이션(Programmatic Navigation)**을 구현합니다.

### useNavigate 훅

React Router는 프로그래밍 방식의 페이지 이동을 위해 **`useNavigate`** 훅을 제공합니다.

### 기본 사용법

```jsx
import { useNavigate } from "react-router-dom";

function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = () => {
    // 로그인 처리 로직...
    console.log("로그인 성공!");

    // 프로그래밍 방식으로 페이지 이동
    navigate("/dashboard");
  };

  return <button onClick={handleLogin}>로그인</button>;
}
```

**동작 순서**:

1. `useNavigate()` 호출 → `navigate` 함수 반환
2. 사용자가 로그인 버튼 클릭
3. `handleLogin` 함수 실행
4. `navigate("/dashboard")` 호출
5. `/dashboard` 페이지로 자동 이동

### Link vs useNavigate

```jsx
// Link: JSX 요소로 클릭 가능한 링크
<Link to="/profile">프로필 보기</Link>;

// useNavigate: 함수 호출로 페이지 이동
const navigate = useNavigate();
navigate("/profile");
```

**차이점**:

- **Link**: 사용자가 **클릭**해야 이동 (선언적)
- **useNavigate**: **코드**에서 원하는 시점에 이동 (명령적)

**언제 무엇을 사용할까?**

- **Link**: 일반적인 네비게이션 링크 (메뉴, 내비게이션 링크)
- **useNavigate**: 특정 로직 실행 후 자동 이동 (폼 제출, 인증, 비동기 작업 후)

### navigate 함수의 옵션

`navigate` 함수는 두 번째 인자로 옵션 객체를 받을 수 있습니다.

### replace 옵션

이전 챕터에서 배운 `<Navigate>` 컴포넌트의 `replace` prop과 동일한 역할을 합니다.

```jsx
navigate("/dashboard", { replace: true });
```

**`replace: true`의 역할**:

- 현재 페이지를 히스토리 스택에 **추가하지 않고** 대체
- 뒤로 가기 버튼을 눌렀을 때 이전 페이지가 아닌 그 이전 페이지로 이동

### replace를 사용하지 않은 경우

```jsx
function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = () => {
    navigate("/dashboard"); // replace 없음
  };

  return <button onClick={handleLogin}>로그인</button>;
}
```

**히스토리 스택**:

```
1. 홈 (/)
2. 로그인 페이지 (/login)
3. 대시보드 (/dashboard)  ← 현재 위치
```

**뒤로 가기 동작**:

- 뒤로 가기 클릭 → `/login`(로그인 페이지)로 돌아감
- **문제점**: 이미 로그인했는데 로그인 페이지로 돌아가는 것이 이상함

### replace를 사용한 경우

```jsx
function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = () => {
    navigate("/dashboard", { replace: true }); // replace 사용
  };

  return <button onClick={handleLogin}>로그인</button>;
}
```

**히스토리 스택**:

```
1. 홈 (/)
2. 대시보드 (/dashboard)  ← 현재 위치 (/login이 대체됨)
```

**뒤로 가기 동작**:

- 뒤로 가기 클릭 → `/`(홈)으로 돌아감
- **더 자연스러움**: 로그인 페이지를 건너뛰고 홈으로 바로 이동

### replace를 사용하는 경우

```jsx
// ✅ 로그인/로그아웃
navigate("/dashboard", { replace: true });
navigate("/login", { replace: true });

// ✅ 폼 제출 후 이동 (재제출/중복 제출 방지 목적)
navigate("/posts", { replace: true });

// ✅ 결제 완료
navigate("/payment-success", { replace: true });
```

**판단 기준**: 사용자가 뒤로 가기를 눌렀을 때 **돌아가서는 안 되는 페이지**는 `replace: true` 사용을 고려합니다.

**참고**: 펼 제출 후에도 상황에 따라 `replace`를 사용하지 않을 수 있습니다. 예를 들어 사용자가 뒤로 가기로 폼으로 돌아가서 내용을 수정할 수 있어야 하는 편집 화면 같은 경우에는 `replace` 없이 사용하는 것이 더 자연스러울 수 있습니다.

### 숫자로 이동하기

`navigate` 함수는 숫자를 사용해 상대적으로 이동할 수도 있습니다.

```jsx
navigate(-1); // 뒤로 가기
navigate(1); // 앞으로 가기
```

### useNavigate 훅

- 프로그래밍 방식으로 페이지를 이동시키는 함수를 반환
- 클릭이 아닌 **코드 실행 흐름**에서 페이지 이동이 필요할 때 사용

### 기본 사용법

```jsx
const navigate = useNavigate();
navigate("/dashboard"); // /dashboard로 이동
```

### replace 옵션

```jsx
navigate("/dashboard", { replace: true });
// 히스토리 스택에 현재 페이지를 남기지 않고 대체
```

**사용 시점**:

- 로그인/로그아웃
- 폼 제출 완료
- 결제 완료
- 뒤로 가기로 돌아가서는 안 되는 페이지

### state 옵션

```jsx
navigate("/detail", { state: { id: 123 } });
// 다음 페이지에 데이터 전달
```

**받는 방법**:

- **Link**: 사용자 클릭으로 이동 (선언적)
- **useNavigate**: 코드 실행으로 이동 (명령적)

- 폼 제출 후 자동 이동
- 로그인/로그아웃 처리

- 조건부 리다이렉트
- 권한 확인 후 이동

### 1단계: `useNavigate`로 로그인 기능 구현하기

pages/Login/Login.jsx에서 사용자가 '로그인' 버튼을 클릭했을 때 `/dashboard` 경로로 이동하는 기능을 구현합니다.

**1. `useNavigate` 훅 호출**

`Login` 컴포넌트 최상단에서 `useNavigate`를 호출하여 `navigate` 함수를 가져옵니다.

```jsx
// src/pages/Login/Login.jsx (수정)
import { useNavigate } from "react-router-dom";

export function Login() {
  const navigate = useNavigate();
  // ...
}
```

**2. `navigate` 함수 호출**

`handleLogin` 함수 내에서, `alert` 창이 닫힌 후 `/dashboard` 경로로 이동하도록 `navigate` 함수를 호출합니다. 로그인 후에는 뒤로 가기로 로그인 페이지로 돌아가지 않도록 `replace: true` 옵션을 사용합니다.

```jsx
// src/pages/Login/Login.jsx (수정)
function handleLogin() {
  alert("로그인 성공! 대시보드로 이동합니다.");
  navigate("/dashboard", { replace: true });
}
```

이제 로그인 페이지에서 버튼을 클릭하면 경고창 후 대시보드 페이지로 자동 이동합니다.

### 요약

이번 챕터에서는 **프로그래밍 방식의 페이지 이동**을 배웠습니다.

- **`useNavigate` 훅**: 코드에서 원하는 시점에 페이지를 이동시키는 함수 제공
- **`navigate(경로)`**: 특정 경로로 이동
- **`navigate(경로, { replace: true })`**: 히스토리 스택을 대체하며 이동 (로그인/로그아웃, 폼 제출 후 등)
- **`navigate(숫자)`**: 상대적 이동 (`1`은 뒤로 가기)
- **`state` 옵션**: 페이지 이동 시 데이터 전달, `useLocation`으로 받기

**Link vs useNavigate**:

- **Link**: 클릭으로 이동하는 링크
- **useNavigate**: 로직 실행 후 자동 이동 (폼 제출, 인증, 비동기 작업 완료 등)

프로그래밍 방식의 네비게이션은 사용자가 특정 작업을 완료한 후 자연스럽게 다음 페이지로 이동시켜 더 나은 사용자 경험을 제공합니다.

### 문제 1

다음 코드에서 사용자가 "로그인" 버튼을 클릭했을 때 어떤 일이 발생하나요?

```jsx
function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = () => {
    alert("로그인 성공!");
    navigate("/dashboard");
  };

  return <button onClick={handleLogin}>로그인</button>;
}
```

- 정답 및 해설
    
    정답:
    
    "로그인 성공!" 경고창이 표시되고, 경고창을 닫으면 `/dashboard` 페이지로 자동 이동합니다.
    
    해설:
    
    `useNavigate` 훅이 반환하는 `navigate` 함수는 프로그래밍 방식으로 페이지를 이동시킵니다.
    
    사용자가 버튼을 클릭하면 `handleLogin` 함수가 실행되고, 먼저 `alert("로그인 성공!")`가 실행되어 경고창이 표시됩니다.
    
    `alert`는 동기(blocking) 함수이므로 사용자가 "확인"을 누르기 전까지 다음 코드가 실행되지 않습니다.
    
    경고창을 닫은 이후에 `navigate("/dashboard")`가 실행되어 `/dashboard`로 페이지가 이동합니다.
    
    `<Link>`는 클릭 즉시 이동하지만, `navigate`는 코드 실행 시점에 이동하므로 알림 표시, 데이터 저장 등 특정 로직이 완료된 후 페이지를 이동시킬 때 사용합니다.
    
    실무에서는 로그인, 회원가입, 폼 제출 등 특정 로직이 완료된 뒤 자동으로 페이지를 이동시킬 때 자주 사용합니다.

```jsx
  return <button onClick={handleLogin}>로그인</button>;
}

// 코드 B
function LoginPage() {
  const navigate = useNavigate();
  const handleLogin = () => {
    navigate("/dashboard", { replace: true });
  };
  return <button onClick={handleLogin}>로그인</button>;
}
```

- 정답 및 해설
    
    정답:
    
    코드 A는 로그인 페이지가 브라우저 히스토리에 남아 뒤로 가기를 누르면 로그인 페이지로 돌아가고, 코드 B는 로그인 페이지가 히스토리에서 제거되어 뒤로 가기를 누르면 로그인 이전 페이지로 돌아갑니다.
    
    해설:
    
    `replace: true` 옵션은 현재 페이지를 히스토리 스택에 추가하지 않고 기존 항목을 대체합니다.
    
    코드 A에서는 로그인 페이지가 히스토리에 남아 있기 때문에 대시보드에서 뒤로 가기를 누르면 다시 로그인 페이지로 이동합니다.
    
    코드 B에서는 로그인 페이지가 히스토리에서 대체되므로 뒤로 가기를 눌렀을 때 로그인 페이지가 아닌 이전 페이지로 이동합니다.
    
    로그인이 완료된 상태에서 다시 로그인 페이지로 돌아가는 것은 사용자 경험상 자연스럽지 않기 때문에, 로그인이나 로그아웃처럼 일회성 액션이 완료되는 경우에는 `replace: true`를 사용하는 것이 적합합니다.
    
    다만 로그인 실패 후 다시 로그인 페이지로 돌아가야 하는 경우처럼 히스토리를 남겨야 하는 요구사항이 있다면 `replace`를 사용하지 않을 수도 있습니다.

### 문제 3

다음 코드에서 `navigate(-1)`은 무엇을 의미하나요?

```jsx
function PostDetailPage() {
  const navigate = useNavigate();

  return (
    <div>
      <button onClick={() => navigate(-1)}>
        뒤로 가기
      </button>
      <h1>게시글 상세</h1>
    </div>
  );
}
```

- 정답 및 해설
    
    정답:
    
    브라우저의 뒤로 가기 버튼과 동일하게 이전 페이지로 이동합니다.
    
    해설:
    
    `navigate` 함수는 문자열 경로뿐만 아니라 숫자도 받을 수 있습니다.
    
    `navigate(-1)`은 한 단계 이전 페이지로 이동하고, `navigate(-2)`는 두 단계 이전 페이지로 이동합니다.
    
    `navigate(1)`은 뒤로 가기 이후 다시 앞으로 이동하는 동작을 의미합니다.
    
    예를 들어 게시글 목록 페이지에서 게시글 상세 페이지로 이동한 뒤 `navigate(-1)`을 호출하면 다시 목록 페이지로 돌아갑니다.
    
    특정 경로로 항상 이동해야 할 때는 문자열 경로를 사용하고, 사용자가 어디서 왔든 이전 페이지로 돌아가야 할 때는 숫자를 사용하는 상대적 이동 방식이 적합합니다.
    
    히스토리가 없는 상태에서 `navigate(-1)`을 호출하면 아무 동작이 없거나 브라우저에 따라 이전 사이트로 이동할 수 있으므로, 이런 경우에는 히스토리 길이를 확인해 홈으로 보내는 폴백 처리가 필요할 수 있습니다.

### 들어가며

웹사이트를 개발할 때 각 페이지마다 **브라우저 탭 제목**이 다르게 표시되는 것을 본 적이 있나요?

예를 들어:

- 홈 페이지: "Codeit - 프로그래밍 학습 플랫폼"
- 강의 페이지: "JavaScript 기초 - Codeit"
- 프로필 페이지: "내 프로필 - Codeit"

또한 카카오톡이나 페이스북에 링크를 공유할 때 **예쁜 미리보기 카드**가 나타나는 것도 보셨을 겁니다. 제목, 설명, 이미지가 포함된 카드 말이죠.

이런 기능들은 모두 HTML의 **`<head>` 섹션**에 있는 태그들을 통해 구현됩니다:

- `<title>`: 브라우저 탭 제목
- `<meta name="description">`: 페이지 설명 (검색 결과에 표시)
- `<meta property="og:*">`: 소셜 미디어 공유 시 미리보기 정보

이번 챕터에서는 React에서 이런 헤드 정보를 **페이지별로 동적으로 관리**하는 방법을 배웁니다.

### HTML Head 섹션이란?

HTML 문서는 크게 두 부분으로 나뉩니다:

```html
<!DOCTYPE html>
<html>
  <head>
    <!-- 메타 정보: 사용자에게 보이지 않음 -->
    <title>페이지 제목</title>
    <meta name="description" content="페이지 설명" />
    <link rel="icon" href="/favicon.ico" />
  </head>
  <body>
    <!-- 실제 콘텐츠: 사용자에게 보임 -->
    <div id="root"></div>
  </body>
</html>
```

**`<head>` 섹션의 역할**:

- **브라우저에게 정보 제공**: 페이지 제목, 문자 인코딩, 뷰포트 설정
- **검색 엔진에게 정보 제공**: 페이지 설명, 키워드, 인덱싱 제어
- **소셜 미디어에게 정보 제공**: 공유 시 표시될 제목, 설명, 이미지
- **외부 리소스 연결**: CSS, 폰트, 파비콘

### React에서의 문제

React는 SPA(Single Page Application)입니다. HTML 파일은 하나뿐이고, 페이지 전환은 JavaScript로 처리됩니다.

**기본 index.html**:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>React App</title>
    <!-- 모든 페이지가 같은 제목 -->
  </head>
  <body>
    <div id="root"></div>
  </body>
</html>
```

**문제점**:

- 홈, 프로필, 설정 페이지 모두 브라우저 탭 제목이 "React App"
- 검색 엔진이 각 페이지의 내용을 구분하기 어려움
- 링크 공유 시 항상 같은 미리보기가 표시됨

**해결책**: JavaScript로 `<head>` 태그를 동적으로 변경해야 합니다!

### React 19의 혁명: 네이티브 헤드 태그 지원

React 19부터는 **서드파티 라이브러리 없이** 컴포넌트 안에서 직접 헤드 태그를 사용할 수 있습니다!

**전제 조건**: React 19 + DOM 환경(react-dom)에서만 가능합니다.

### 기본 사용법

```jsx
function HomePage() {
  return (
    <div>
      {/* 헤드 태그를 컴포넌트 안에 바로 작성! */}
      <title>홈 - React Router 학습</title>
      <meta
        name="description"
        content="React Router를 배우는 홈 페이지입니다."
      />

      {/* 실제 콘텐츠 */}
      <h1>홈 페이지</h1>
      <p>환영합니다!</p>
    </div>
  );
}
```

**놀라운 점**:

- `<title>`과 `<meta>`를 `<div>` 안에 작성해도 React가 자동으로 `<head>`로 이동시킴
- 별도의 라이브러리 설치나 설정 불필요
- 컴포넌트가 마운트되면 자동으로 적용, 언마운트되면 제거

**중요한 한계**:

React 19의 head 관리는 **클라이언트에서 동작**하므로, 카카오톡/페이스북/검색엔진 같은 **크롤러는 서버 응답 HTML을 읽어** 미리보기를 생성합니다. CSR(Client-Side Rendering)만으로는 이런 크롤러가 동적으로 생성된 메타 태그를 읽지 못할 수 있습니다.

**공유/SEO가 중요한 서비스라면**:

- **SSR(Server-Side Rendering)**: Next.js, Remix 같은 프레임워크 사용
- **SSG(Static Site Generation)**: 빌드 시 HTML 생성
- **프리렌더링**: 크롤러용 정적 HTML 제공

React 19의 head 태그 지원은 **작성 방식을 편리하게** 만들지만, SEO/OG가 중요하다면 추가 전략이 필요합니다.

### 2. Open Graph 태그 (소셜 미디어)

**Open Graph란?**

Open Graph는 페이스북이 만든 프로토콜로, **링크를 공유할 때 표시될 정보를 정의**하는 메타 태그 규약입니다. 현재는 페이스북뿐만 아니라 카카오톡, 네이버, 트위터 등 대부분의 소셜 미디어와 메신저에서 사용됩니다.

**왜 필요할까?**

링크를 공유할 때 단순한 URL만 표시되면 사람들이 클릭하고 싶어 하지 않습니다. 하지만 **제목, 설명, 이미지**가 포함된 예쁜 카드 형태로 표시되면 클릭률이 크게 높아집니다.

**예시**: 카카오톡에 유튜브 링크를 공유하면 영상 썸네일과 제목이 자동으로 표시되는 것도 Open Graph 덕분입니다.

```jsx
function ArticlePage() {
  return (
    <div>
      <title>React 19 새 기능 - Tech Blog</title>

      {/* 일반 메타 태그 */}
      <meta
        name="description"
        content="React 19의 새로운 기능을 알아봅니다."
      />

      {/* Open Graph 태그 */}
      <meta property="og:type" content="article" />
      <meta property="og:title" content="React 19 새 기능" />
      <meta
        property="og:description"
        content="React 19의 새로운 기능을 알아봅니다."
      />
      <meta property="og:image" content="https://example.com/react19.jpg" />
      <meta property="og:url" content="https://example.com/articles/react19" />

      <h1>React 19 새 기능</h1>
      <p>React 19에서는...</p>
    </div>
  );
}

```

**Open Graph 태그 설명**:

- `og:type`: 콘텐츠 유형 (article, website, video 등)
- `og:title`: 공유 시 표시될 제목
- `og:description`: 공유 시 표시될 설명
- `og:image`: 공유 시 표시될 이미지 URL
- `og:url`: 공유할 URL (정규 URL)

**사용 예시**: 카카오톡, 페이스북, 트위터 등에서 링크 공유 시 예쁜 카드가 표시됩니다.

### 동적 데이터로 헤드 태그 설정

페이지 데이터에 따라 헤드 정보를 동적으로 생성할 수 있습니다.

```jsx

    image: "https://example.com/macbook.jpg",

  };

  return (
    <div>
      {/* 동적으로 생성된 헤드 태그 */}
      <title>{product.name} - Shop</title>
      <meta name="description" content={product.description} />

      {/* Open Graph */}
      <meta property="og:title" content={product.name} />
      <meta property="og:description" content={product.description} />
      <meta property="og:image" content={product.image} />
      <meta property="og:type" content="product" />
      <meta property="product:price:amount" content={product.price} />
      <meta property="product:price:currency" content="KRW" />

      {/* 실제 콘텐츠 */}
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <p>가격: {product.price.toLocaleString()}원</p>
      <img src={product.image} alt={product.name} />
    </div>
  );
}
```

- 링크 공유 시 MacBook Pro 이미지와 설명이 표시됨

### 핵심 정리

### React 19의 네이티브 헤드 태그

- **React 19 + DOM 환경**에서만 사용 가능
- 컴포넌트 안에서 직접 `<title>`, `<meta>` 태그 사용 가능
- React가 자동으로 `<head>`로 이동
- 별도 라이브러리 불필요

### CSR의 한계

- React 19로 작성해도 **CSR에서는 크롤러가 메타 태그를 읽지 못할 수 있음**
- 공유/SEO가 중요하면 **SSR/SSG/프리렌더링** 필요
- React Router를 사용하더라도, React 19+라면 페이지 컴포넌트에서 `<title>/<meta>`를 직접 렌더링하는 방식이 가장 단순

### 중복 title 처리

중첩 라우팅에서 부모와 자식이 모두 `<title>`을 렌더링하면, **일반적으로 마지막(가장 안쪽/현재 페이지)에 렌더링된 title이 적용**됩니다. 공통 메타 태그는 레이아웃에서, 페이지별 title/description은 각 페이지 컴포넌트에서 관리하는 것이 일반적입니다.

### 기본 사용법

```jsx
function Page() {
  return (
    <div>
      <title>페이지 제목</title>
      <meta name="description" content="페이지 설명" />
      <h1>콘텐츠</h1>
    </div>
  );
}
```

### 주요 메타 태그

- `<title>`: 브라우저 탭 제목
- `name="description"`: 검색 결과 설명
- `property="og:*"`: 소셜 미디어 공유 미리보기
- `name="twitter:*"`: 트위터 카드

### SEO 모범 사례

- 페이지마다 다른 제목/설명
- 구체적이고 명확한 제목 (50-60자)
- 설명은 150-160자 권장
- 관련 키워드 자연스럽게 포함

### 공유 최적화

- Open Graph 이미지: 1200 x 630px
- 제목, 설명, 이미지 모두 제공
- URL 정보 포함

### 요약

이번 챕터에서는 **React 19의 네이티브 헤드 태그 지원**을 배웠습니다.

- **React 19 + DOM 환경**에서 `<title>`, `<meta>` 태그를 컴포넌트에 직접 작성 가능
- React가 자동으로 `<head>`로 이동, 별도 라이브러리 불필요
- **CSR의 한계**: 크롤러가 메타 태그를 읽지 못할 수 있음 (SSR/SSG 필요)
- **SEO 모범 사례**: 페이지마다 다른 제목/설명, 구체적인 내용 (50-60자)
- **공유 최적화**: Open Graph 태그 (`og:title`, `og:description`, `og:image`)

React 19의 네이티브 헤드 태그 지원은 코드를 간결하게 만들지만, 실제 공유/SEO가 중요한 서비스라면 Next.js 같은 SSR 프레임워크와 함께 사용하는 것이 좋습니다.

## 2-07. #Quiz Dynamic Head

### 문제 1

React 19에서 다음 코드의 `<title>` 태그는 어디에 렌더링되나요?

```jsx
function HomePage() {
  return (
    <div>
      <title>홈 - My App</title>
      <meta name="description" content="홈 페이지입니다." />
      <h1>홈 페이지</h1>
    </div>
  );
}
```

- 정답 및 해설
    
    정답:
    
    HTML 문서의 `<head>` 섹션에 렌더링됩니다.
    
    해설:
    
    React 19부터는 DOM 환경에서 컴포넌트 내부에 `<title>`, `<meta>`, `<link>` 같은 태그를 작성하면 React가 이를 자동으로 HTML 문서의 `<head>` 섹션으로 이동시킵니다. 이 기능은 React 19 이상 버전에서, react-dom을 사용하는 브라우저 환경에서 동작합니다.
    
    이 동작은 클라이언트에서 실행되므로, 카카오톡이나 페이스북, 검색엔진 크롤러처럼 서버 응답 HTML을 기준으로 미리보기를 생성하는 경우에는 이 메타 정보가 반영되지 않을 수 있습니다. 공유나 SEO가 중요하다면 SSR, SSG, 프리렌더링 같은 서버 기반 렌더링 방식이 필요합니다.
    
    이전 버전의 React에서는 react-helmet-async 같은 라이브러리가 필요했지만, React 19에서는 네이티브로 지원되어 코드가 간단해지고 번들 크기도 줄어듭니다. 다만 SEO 요구사항에 따라 렌더링 전략을 함께 고려해야 합니다.


    React 19에서는 각 페이지 컴포넌트에서 데이터를 기반으로 동적으로 SEO 태그를 생성할 수 있기 때문에, 수많은 상품 페이지도 일관되게 최적화할 수 있습니다. 전자상거래나 콘텐츠 서비스에서는 이러한 SEO 최적화가 트래픽과 매출에 직결되는 중요한 요소입니다.

# 3. Zustand로 구현하는 상태 관리

## 3-01. Zustand를 왜 배우나요?

### Zustand를 왜 배우나요?

리액트로 조금씩 복잡한 앱을 만들다 보면 이런 문제를 만나게 됩니다:

```jsx
function App() {
  const [cart, setCart] = useState([]);
  const [user, setUser] = useState(null);

  return (
    <>
      <Header cart={cart} user={user} />
      <ProductList cart={cart} setCart={setCart} />
      <Cart cart={cart} setCart={setCart} />
    </>
  );
}
```

상품 목록에서 "장바구니 담기"를 누르면 헤더의 개수도 바뀌어야 하고, 장바구니 페이지에도 반영되어야 합니다. 그래서 `App` 컴포넌트에서 상태를 관리하고, props로 계속 내려줘야 합니다.

컴포넌트가 더 많아지면 어떻게 될까요?

```jsx
<App> → <Layout> → <Header> → <CartBadge cart={cart} />
```

`CartBadge`가 `cart` 데이터를 쓰려면, 중간에 있는 모든 컴포넌트가 props를 전달해야 합니다. 이런 현상을 **Prop Drilling**이라고 부릅니다.

### Context API로 해결하면 안 되나요?

Context API도 Prop Drilling을 해결할 수 있습니다. 하지만 몇 가지 문제가 있습니다:

```jsx
// Context API 방식
function App() {
  return (
    <CartProvider>
      <UserProvider>
        <ThemeProvider>
          <LanguageProvider>
            {/* Provider 지옥... */}
            <Layout />
          </LanguageProvider>
        </ThemeProvider>
      </UserProvider>
    </CartProvider>
  );
}
```

**Context API의 단점**:

1. **Provider 지옥**: 여러 상태를 관리하려면 Provider를 중첩해야 합니다
2. **보일러플레이트**: `createContext`, `Provider`, `useContext` 등 작성할 코드가 많습니다
3. **복잡한 상태 분리**: 성능을 위해 여러 개의 작은 Context로 나눠야 하는 경우가 많습니다

**참고**: React 19에서는 Context API의 성능이 많이 개선되었습니다. 하지만 여전히 Provider 구조와 보일러플레이트는 필요합니다.

### Zustand는 이 문제를 해결합니다

```jsx
// Zustand 방식 - Provider 없음!
function App() {
  return <Layout />;
}

// 필요한 곳에서 바로 사용
function CartBadge() {
  const itemCount = useCartStore(
    (state) => state.items.length
  );
  return <span>{itemCount}</span>;
}
```

**Zustand의 장점**:

- 상태를 **컴포넌트 밖**에 저장합니다
- 필요한 컴포넌트에서 **직접** 가져다 씁니다
- props 전달 없이도 모든 곳에서 같은 상태를 공유할 수 있습니다
- **Provider 불필요**: 앱을 감쌀 필요가 없습니다
- **선택적 구독**: `items.length`만 구독하면 그것만 바뀔 때만 리렌더링됩니다
- **간단한 API**: 몇 줄만으로 상태 관리를 시작할 수 있습니다

### 이번 강의에서 만들 것

작은 쇼핑몰 앱을 만들어보겠습니다:

- 상품 목록에서 장바구니에 담기
- 장바구니에서 수량 변경 / 삭제
- 헤더에 장바구니 총 개수 표시

**핵심은 이겁니다:**

"어디서든 같은 장바구니 데이터를 읽고 쓸 수 있다"

### Zustand의 장점

1. **배우기 쉽다**: Context API나 Redux보다 훨씬 간단합니다
2. **보일러플레이트가 적다**: 몇 줄만으로 상태 관리를 시작할 수 있습니다
3. **성능이 좋다**: 필요한 컴포넌트만 리렌더링됩니다
4. **TypeScript 지원**: 타입 안정성을 자연스럽게 제공합니다

실무에서도 많이 쓰이고, React 19와도 잘 동작합니다.

## 3-01. #Quiz Zustand를 왜 배우나요?

### 문제 1

다음 코드에서 발생하는 문제를 설명하세요:

```jsx
function App() {
  const [user, setUser] = useState(null);

  return (
    <>
      <Header user={user} />
      <MainContent>
        <Sidebar>
          <UserProfile user={user} />
        </Sidebar>
      </MainContent>
    </>
  );
}
```

`UserProfile` 컴포넌트가 `user` 데이터를 사용하려면 어떤 일이 일어나나요?

- 정답 및 해설
    
    정답:
    
    `UserProfile`이 `user` 데이터를 사용하려면 중간에 있는 `MainContent`와 `Sidebar` 컴포넌트가 모두 `user` props를 전달해야 합니다.
    
    해설:
    
    `UserProfile`이 `user` 데이터를 사용하려면 다음과 같이 중간 컴포넌트들이 `user` props를 전달해야 합니다.
    
    ```jsx
    <MainContent user={user}>
      <Sidebar user={user}>
        <UserProfile user={user} />
      </Sidebar>
    </MainContent>
    ```
    
    이처럼 실제로 `user` 데이터를 사용하지 않는 중간 컴포넌트들까지 props를 받아서 전달해야 하는 현상을 Prop Drilling이라고 부릅니다. Prop Drilling은 컴포넌트 구조가 깊어질수록 코드 가독성과 유지보수성을 떨어뜨립니다.
    
    Zustand 같은 상태 관리 라이브러리를 사용하면 `UserProfile`에서 직접 `user` 상태를 가져올 수 있어, 중간 컴포넌트들이 불필요하게 props를 전달하지 않아도 됩니다.

### Zustand 설치하기


- `zustand`: 상태 관리 라이브러리
- `nanoid`: 고유한 ID 생성 (장바구니 항목마다 필요)
- `react-hot-toast`: 알림 메시지 표시

### Zustand의 핵심 개념

Zustand는 **store**(저장소)를 만들어서 상태를 관리합니다:

```jsx
// stores/cartStore.js
import { create } from "zustand";

export const useCartStore = create((set, get) => ({
  // 상태
  items: [],

  // 액션 (상태를 변경하는 함수)
  addItem: (product) =>
    set((state) => {
      /* ... */
    }),
}));
```

- `create`: store를 만드는 함수
- `set`: 상태를 변경할 때 사용
- `get`: 현재 상태를 읽을 때 사용
- store는 **hook**으로 사용됩니다 (`useCartStore`)

### 장바구니 Store 만들기

### 왜 불변성이 중요한가요?

리액트는 상태가 "변했는지"를 **참조(주소)로 비교**합니다:

```jsx
// ❌ 직접 수정 (리액트가 감지 못함)
state.items.push(newItem);

// ✅ 새 배열 생성 (리액트가 감지함)
return { items: [...state.items, newItem] };
```

배열이나 객체를 직접 수정하면 주소는 그대로라서 리액트가 변화를 감지하지 못합니다. 그래서 **새로운 배열/객체를 만들어서** 반환해야 합니다.

### cartStore.js 작성

`src/stores/cartStore.js` 파일을 만듭니다:

```jsx
import { create } from "zustand";
import { nanoid } from "nanoid";

export const useCartStore = create((set, get) => ({
  items: [],

  addItem: (product) =>
    set((state) => {
      const exists = state.items.find(
        (item) => item.productId === product.id
      );

      if (exists) {
        return {
          items: state.items.map((item) =>
            item.productId === product.id
              ? { ...item, quantity: item.quantity + 1 }
              : item
          ),
        };
      }

      const newItem = {
        cartItemId: nanoid(),
        productId: product.id,
        name: product.name,
        price: product.price,
        quantity: 1,
      };

      return {
        items: [...state.items, newItem],
      };
    }),

  removeItem: (cartItemId) =>
    set((state) => ({
      items: state.items.filter(
        (item) => item.cartItemId !== cartItemId
      ),
    })),

  increaseQuantity: (cartItemId) =>
    set((state) => ({
      items: state.items.map((item) =>
        item.cartItemId === cartItemId
          ? { ...item, quantity: item.quantity + 1 }
          : item
      ),
    })),

  decreaseQuantity: (cartItemId) =>
    set((state) => {
      const target = state.items.find(
        (item) => item.cartItemId === cartItemId
      );
      
      if (!target) {
	      return { items: state.items };
      }
			
			// 개수가 하나이면 장바구니에서 제거
      if (target.quantity === 1) {
        return {
          items: state.items.filter(
            (item) => item.cartItemId !== cartItemId
          ),
        };
      }

      return {
        items: state.items.map((item) =>
          item.cartItemId === cartItemId
            ? { ...item, quantity: item.quantity - 1 }
            : item
        ),
      };
    }),

  clear: () => set({ items: [] }),

  getTotalCount: () =>
    get().items.reduce(
      (sum, item) => sum + item.quantity,
      0
    ),

  getTotalPrice: () =>
    get().items.reduce(
      (sum, item) => sum + item.price * item.quantity,
      0
    ),
}));
```

### 코드 설명

**cartItemId vs productId**

- `productId`: 상품의 고유 ID (같은 상품이면 같음)
- `cartItemId`: 장바구니 항목의 고유 ID (nanoid로 생성, 항상 다름)

**불변성 패턴**

- `map`: 특정 항목만 변경
- `filter`: 특정 항목 제거
- `[...array, item]`: 배열에 추가
- `{ ...obj, key: value }`: 객체 속성 변경

**get() 사용**

- `getTotalCount`와 `getTotalPrice`는 상태를 변경하지 않고 읽기만 하므로 `get()`을 사용합니다

## 3-03. #Quiz Zustand 기능 구현

### 문제 1

다음 코드의 문제점을 설명하세요:

```jsx
addItem: (product) =>
  set((state) => {
    state.items.push(product);
    return { items: state.items };
  }),
```

이 코드는 왜 리액트에서 제대로 동작하지 않을까요?

- 정답 및 해설
    
    정답:
    
    `state.items.push(product)`는 기존 배열을 직접 수정하기 때문입니다.
    
    해설:
    
    리액트는 상태 변경 여부를 참조(주소) 비교로 판단합니다. `push`는 기존 배열에 값을 추가하지만 배열의 주소는 바뀌지 않습니다. 주소가 같으면 리액트는 상태가 변하지 않았다고 판단하여 리렌더링을 하지 않습니다.
    
    ```jsx
    const old = [1, 2, 3];
    const same = old;
    same.push(4);
    
    console.log(old === same); // true
    ```
    
    올바른 방식은 새로운 배열을 만들어 반환하는 것입니다.
    
    ```jsx
    addItem: (product) =>
      set((state) => {
        const newItem = {
          cartItemId: nanoid(),
          productId: product.id,
          name: product.name,
          price: product.price,
          quantity: 1,
        };
    
        return { items: [...state.items, newItem] };
      }),
    ```
    
    이렇게 새 배열을 생성하면 참조가 달라지므로 리액트가 변화를 감지하고 정상적으로 리렌더링합니다. 배열과 객체는 항상 새로 만들어서 반환해야 합니다.

### 문제 2

다음 코드에서 `cartItemId`와 `productId`를 각각 언제 사용하는지 설명하세요:

```jsx
const newItem = {
  cartItemId: nanoid(), // ← 이건 언제?
  productId: product.id, // ← 이건 언제?
  name: product.name,
  price: product.price,
  quantity: 1,
};
```

- 정답 및 해설
    
    정답:
    
    `productId`는 상품 자체를 구분할 때 사용하고, `cartItemId`는 장바구니에 담긴 각 항목을 구분할 때 사용합니다.
    
    해설:
    
    `productId`는 같은 상품인지 확인할 때 사용합니다. 예를 들어 이미 장바구니에 담긴 상품인지 검사할 때 `productId`를 비교합니다.
    
    ```jsx
    const exists = state.items.find(
      (it) => it.productId === product.id
    );
    ```
    
    `cartItemId`는 장바구니에 담긴 항목 하나하나를 식별할 때 사용합니다. 삭제 버튼 처리나 React의 `key` prop에 사용됩니다.
    
    ```jsx
    <div key={item.cartItemId}>
      <button onClick={() => remove(item.cartItemId)}>
        삭제
      </button>
    </div>
    ```
    
    `productId`만 있으면 같은 상품을 여러 번 담았을 때 항목을 구분할 수 없고, `cartItemId`만 있으면 이미 담긴 상품인지 판단할 수 없습니다. 따라서 같은 상품 여부 판단에는 `productId`, 항목 삭제나 수정에는 `cartItemId`를 사용하는 것이 올바른 설계입니다.

### 문제 3

다음 코드를 보고 질문에 답하세요:

```jsx
const totalCount = useCartStore((store) =>
  store.getTotalCount()
);
```

1. `getTotalCount`가 실행되는 시점은?
2. 장바구니에 상품을 추가하면 `totalCount`는 자동으로 업데이트되나요?
- 정답 및 해설
    
    정답:
    
    `getTotalCount`는 컴포넌트가 렌더링될 때마다 실행되며, 장바구니가 변경되면 `totalCount`는 자동으로 업데이트됩니다.
    
    해설:
    
    `getTotalCount()`는 컴포넌트가 렌더링될 때마다 실행됩니다. 내부에서는 `get()`을 통해 현재 `items` 상태를 가져와 총 수량을 계산합니다.
    
    ```jsx
    getTotalCount: () =>
      get().items.reduce((sum, it) => sum + it.quantity, 0),
    ```
    
    장바구니에 상품을 추가하면 `addItem` 액션이 `items` 상태를 변경합니다. Zustand는 이 상태를 구독 중인 컴포넌트들을 다시 렌더링하고, 그 과정에서 `getTotalCount()`가 다시 실행되어 최신 값이 계산됩니다.
    
    이 방식은 `items`를 직접 가져와 컴포넌트에서 계산하는 것과 동일한 결과를 가지며, 계산 로직을 store에 모아둘 수 있어 유지보수에 유리합니다.

Redux DevTools를 사용하면 이 모든 것을 시각적으로 볼 수 있습니다.

### Store에 DevTools 추가하기

Zustand는 Redux DevTools와 호환됩니다. `devtools` 미들웨어를 추가하면 됩니다.

### import.meta.env.DEV란?

Vite 프로젝트에서는 `import.meta.env.DEV`를 사용해서 개발/프로덕션 환경을 구분할 수 있습니다:

- **개발 환경** (`npm run dev`): `import.meta.env.DEV`가 `true`
- **프로덕션 빌드** (`npm run build`): `import.meta.env.DEV`가 `false`

이를 활용하면 DevTools를 개발 중에만 활성화할 수 있습니다. 프로덕션 빌드 시에는 DevTools 관련 코드가 번들에서 제거되어 파일 크기가 줄어듭니다.

### cartStore.js 수정

`src/stores/cartStore.js`를 열고 다음과 같이 수정합니다:

```jsx
import { create } from "zustand";
import { devtools } from "zustand/middleware";
import { nanoid } from "nanoid";

const cartStore = (set, get) => ({
  items: [],

  addItem: (product) =>
    set(
      (state) => {
        const exists = state.items.find(
          (item) => item.productId === product.id
        );

        if (exists) {
          return {
            items: state.items.map((item) =>
              item.productId === product.id
                ? { ...item, quantity: item.quantity + 1 }
                : item
            ),
          };
        }

        const newItem = {
          cartItemId: nanoid(),
          productId: product.id,
          name: product.name,
          price: product.price,
          quantity: 1,
        };

        return {
          items: [...state.items, newItem],
        };
      },
      undefined,
      "cart/addItem"
    ),

  removeItem: (cartItemId) =>
    set(
      (state) => ({
        items: state.items.filter(
          (item) => item.cartItemId !== cartItemId
        ),
      }),
      undefined,
      "cart/removeItem"
    ),

  increaseQuantity: (cartItemId) =>
    set(
      (state) => ({
        items: state.items.map((item) =>
          item.cartItemId === cartItemId
            ? { ...item, quantity: item.quantity + 1 }
            : item
        ),
      }),
      undefined,
      "cart/increaseQuantity"
    ),

  decreaseQuantity: (cartItemId) =>
    set(
      (state) => {
        const target = state.items.find(
          (item) => item.cartItemId === cartItemId
        );
        if (!target) return { items: state.items };

        if (target.quantity === 1) {
          return {
            items: state.items.filter(
              (item) => item.cartItemId !== cartItemId
            ),
          };
        }

        return {
          items: state.items.map((item) =>
            item.cartItemId === cartItemId
              ? { ...item, quantity: item.quantity - 1 }
              : item
          ),
        };
      },
      undefined,
      "cart/decreaseQuantity"
    ),

  clear: () => set({ items: [] }, undefined, "cart/clear"),

  getTotalCount: () =>
    get().items.reduce(
      (sum, item) => sum + item.quantity,
      0
    ),

  getTotalPrice: () =>
    get().items.reduce(
      (sum, item) => sum + item.price * item.quantity,
      0
    ),
});

export const useCartStore = create(
  import.meta.env.DEV
    ? devtools(cartStore, { name: "cartStore" })
    : cartStore
);

```

### 변경 사항 설명

**1. devtools import**

```jsx
import { devtools } from "zustand/middleware";
```

- Redux DevTools와 연결하기 위한 미들웨어입니다.

**2. store 구현을 별도로 분리**

```jsx
const cartStore = (set, get) => ({
  // 상태와 액션들
});
```

- store 로직을 `cartStore`로 분리하면 devtools를 조건부로 적용하기 쉽습니다.

**3. 개발 환경에서만 devtools 활성화**

```jsx
export const useCartStore = create(
  import.meta.env.DEV
    ? devtools(cartStore, { name: "cartStore" })
    : cartStore
);
```

- `import.meta.env.DEV`가 `true`면 devtools 적용
- `false`면 기본 store만 사용
- 프로덕션 빌드 시 DevTools 코드가 번들에서 제거됨

**4. 액션 이름 추가**

```jsx
set(
  (state) => ({ /* 상태 변경 */ }),
  undefined, // replace 옵션
  "cart/addItem" // 액션 이름
);
```

- 첫 번째 인자: 상태 변경 함
- 두 번째 인자: `undefined` (상태를 병합, `true`면 덮어씀)
    - 상태를 병합한다는 뜻은?
        
        `set(partial, replace?, actionName?)`에서 **두 번째 인자 `replace`가 `true`**면, “부분 업데이트(병합)”가 아니라 **스토어 상태 전체를 통째로 교체(replace)** 합니다.
        
        즉, 지금 `state`에 들어있는 다른 값들이 **그대로 유지되지 않고 사라질 수 있습니다.**
        
        ---
        
        ### replace가 `false`(또는 `undefined`)일 때: 병합
        
        ```jsx
        set({ items: [] }); // replace 생략(= false)
        ```
        
        - 기존 상태가 `{ items, user, theme }` 이런 형태라면
        - `{ items: [] }`만 바뀌고 `user`, `theme`은 유지됩니다.
        
        개념적으로는 이런 느낌입니다:
        
        ```jsx
        nextState = { ...prevState, ...partialState }
        ```
        
        ---
        
        ### replace가 `true`일 때: 전체 덮어쓰기
        
        ```jsx
        set({ items: [] }, true);
        ```
        
        - 기존 상태가 `{ items, addItem, removeItem, getTotalCount, ... }`라면
        - 상태가 **그대로 `{ items: [] }`만 남는 형태로 교체**됩니다.
        
        개념적으로는:
        
        ```jsx
        nextState = partialState
        ```
        
        ---
        
        ### 실제로 어떤 문제가 생기나요?
        
        Zustand의 store는 보통 이렇게 생깁니다:
        
        ```jsx
        {
          items: [],
          addItem: fn,
          removeItem: fn,
          clear: fn,
          getTotalCount: fn,
          ...
        }
        ```
        
        그런데 `replace: true`로 이렇게 해버리면:
        
        ```jsx
        clear: () => set({ items: [] }, true, "cart/clear")
        ```
        
        이 순간 store 상태가:
        
        ```jsx
        { items: [] }
        ```
        
        만 남아버릴 수 있습니다.
        
        그러면 이후에 컴포넌트에서
        
        ```jsx
        const addItem = useCartStore((store) => store.addItem);
        addItem(product);
        ```
        
        `addItem`이 **undefined**가 되어 호출 시 **에러**가 납니다.
        
        ---
        
        ### 언제 `replace: true`를 쓰나요?
        
        - 정말로 “이 store의 전체 상태를 새로 갈아끼우고 싶을 때”
        - 예: 서버에서 통째로 복원한 스냅샷을 넣는 경우
        - 예: `persist`로 저장한 값을 “전체 state로 복원”하는 특수한 경우
            - persist란?
                
                > store 상태를 저장해두었다가 페이지를 새로 열어도 그대로 복구해주는 기능
                > 
        
        일반적인 CRUD 업데이트(추가/삭제/수정)에서는 **거의 항상 `undefined`(병합)** 를 씁니다.
        
        ---
        
        ### 정리
        
        - `undefined`(기본): **부분 업데이트** → 기존 상태 유지 + 일부만 변경
        - `true`: **전체 교체** → 기존 상태를 버리고 새 객체로 덮어씀(다른 키/함수 사라질 수 있음)
        
        원하면, 네 `cartStore` 코드에서 **`replace: true`를 실수로 넣었을 때 어떤 줄에서 어떤 에러가 터지는지**까지, 실제 흐름 예시로 더 구체적으로 써줄게요.

## 3-05. Immer로 불변성 코드 간결하게 만들기 ⭐⭐⭐

### 불변성 코드의 문제점

지금까지 우리는 불변성을 직접 구현했습니다:

```jsx
addItem: (product) =>
  set((state) => {
    const exists = state.items.find((item) => item.productId === product.id);

    if (exists) {
      // 새 배열 + 새 객체 만들기
      return {
        items: state.items.map((item) =>
          item.productId === product.id
            ? { ...item, quantity: item.quantity + 1 }
            : item
        ),
      };
    }

    return {
      items: [...state.items, newItem],
    };
  }),

```

이 코드는 올바르지만 **복잡**합니다:

- `map`으로 배열 전체를 순회
- spread 연산자로 객체 복사
- 중첩이 깊어질수록 더 복잡해짐

만약 "직접 수정하듯이" 쓸 수 있다면 훨씬 간단하지 않을까요?

```jsx
// 이렇게 쓸 수 있다면?
if (exists) {
  exists.quantity += 1; // 직접 수정
  return;
}

state.items.push(newItem); // 직접 추가
```

**Immer가 바로 이것을 가능하게 합니다.**

### Immer란?

Immer는 **"직접 수정하는 것처럼 보이지만, 내부에서는 불변성을 유지하는"** 라이브러리입니다.

```jsx
import { produce } from "immer";

const newState = produce(state, (draft) => {
  // draft를 직접 수정하듯이 작성
  draft.items.push(newItem);
});

// newState는 새로운 객체 (불변성 유지됨)

```

**핵심 개념:**

- `draft`: 원본 상태의 "임시 복사본"
- draft를 수정하면 Immer가 자동으로 새 상태 생성
- 실제로는 불변성이 유지됨

### Immer 설치

```bash
npm i immer
```

### Store를 Immer로 리팩터링하기

### 기존 코드 vs Immer 코드

**기존 (map/filter/spread):**

```jsx
addItem: (product) =>
  set((state) => {
    const exists = state.items.find((item) => item.productId === product.id);

    if (exists) {
      return {
        items: state.items.map((item) =>
          item.productId === product.id
            ? { ...item, quantity: item.quantity + 1 }
            : item
        ),
      };
    }

    const newItem = { /* ... */ };
    return { items: [...state.items, newItem] };
  }),

```

**Immer 사용:**

```jsx
import { produce } from "immer";

addItem: (product) =>
  set(
    produce((state) => {
      const exists = state.items.find((item) => item.productId === product.id);

      if (exists) {
        exists.quantity += 1; // 직접 수정!
        return;
      }

      state.items.push({
        cartItemId: nanoid(),
        productId: product.id,
        name: product.name,
        price: product.price,
        quantity: 1,
      });
    }),
    false,
    "cart/addItem"
  ),

```

훨씬 간단해졌습니다!

### 전체 cartStore.js (Immer 적용)

`src/stores/cartStore.js`를 다음과 같이 수정합니다:

```jsx
import { create } from "zustand";
import { devtools } from "zustand/middleware";
import { produce } from "immer";
import { nanoid } from "nanoid";

const cartStore = (set, get) => ({
  items: [],

  addItem: (product) =>
    set(
      produce((state) => {
        const exists = state.items.find(
          (item) => item.productId === product.id
        );

        if (exists) {
          exists.quantity += 1;
          return;
        }

        state.items.push({
          cartItemId: nanoid(),
          productId: product.id,
          name: product.name,
          price: product.price,
          quantity: 1,
        });
      }),
      undefined,
      "cart/addItem"
    ),

  removeItem: (cartItemId) =>
    set(
      produce((state) => {
        state.items = state.items.filter(
          (item) => item.cartItemId !== cartItemId
        );
      }),
      undefined,
      "cart/removeItem"
    ),

  increaseQuantity: (cartItemId) =>
    set(
      produce((state) => {
        const item = state.items.find(
          (item) => item.cartItemId === cartItemId
        );
       if (!item) {
          return;
        }
        item.quantity += 1;
      }),
      undefined,
      "cart/increaseQuantity"
    ),

  decreaseQuantity: (cartItemId) =>
    set(
      produce((state) => {
        const item = state.items.find(
          (item) => item.cartItemId === cartItemId
        );
        if (!item) return;

        if (item.quantity === 1) {
          state.items = state.items.filter(
            (item) => item.cartItemId !== cartItemId
          );
        } else {
          item.quantity -= 1;
        }
      }),
      undefined,
      "cart/decreaseQuantity"
    ),

  clear: () => set({ items: [] }, undefined, "cart/clear"),

  getTotalCount: () =>
    get().items.reduce(
      (sum, item) => sum + item.quantity,
      0
    ),

  getTotalPrice: () =>
    get().items.reduce(
      (sum, item) => sum + item.price * item.quantity,
      0
    ),
});

export const useCartStore = create(
  import.meta.env.DEV
    ? devtools(cartStore, { name: "cartStore" })
    : cartStore
);

```

### 주요 변경점

**1. addItem - 직접 수정**

```jsx
// 기존
return {
  items: state.items.map((it) =>
    it.productId === product.id
      ? { ...it, quantity: it.quantity + 1 }
      : it
  ),
};

// Immer
exists.quantity += 1;
```

**2. removeItem - filter 대입**

```jsx
// 기존
return { items: state.items.filter(...) };

// Immer
state.items = state.items.filter(...);
```

**3. increaseQuantity - 간단한 증가**

```jsx
// 기존
return {
  items: state.items.map((item) =>
    item.cartItemId === cartItemId
      ? { ...item, quantity: item.quantity + 1 }
      : item
  ),
};

// Immer
const item = state.items.find(...);
if (item) {
  item.quantity += 1;
}
```

### Immer의 동작 원리

Immer는 내부적으로 **Proxy**를 사용합니다:

```jsx
produce(state, (draft) => {
  draft.items.push(newItem);
  //    ↑
  //    Proxy 객체: 모든 변경을 추적
});

// Immer가 변경 내역을 보고 새 상태 생성:
// { items: [...원래items, newItem] }

```

1. `draft`는 원본의 Proxy 객체
2. draft를 수정하면 Immer가 변경사항을 기록
3. 함수가 끝나면 Immer가 새 상태 생성
4. 원본은 변하지 않음 (불변성 유지)

### Immer 사용 시 주의사항

### 1. return 문 주의

**두 가지 패턴:**

```jsx
// 패턴 1: draft 직접 수정 (return 없음)
produce((state) => {
  state.items.push(newItem);
  // return 없음!
});

// 패턴 2: 새 상태 반환 (권장하지 않음)
produce((state) => {
  return { items: [...state.items, newItem] };
});
```

**패턴 1을 사용하세요.** 직접 수정하는 것이 Immer의 목적입니다.

### 2. 조기 return 가능

```jsx
produce((state) => {
  const exists = state.items.find(...);

  if (exists) {
    exists.quantity += 1;
    return; // 여기서 끝!
  }

  state.items.push(newItem);
});
```

조건에 맞으면 `return`으로 일찍 종료할 수 있습니다.

### 3. 단순 값 변경은 Immer 불필요

```jsx
// Immer 불필요
clear: () => set({ items: [] }, false, "cart/clear"),
```

이미 새 객체를 만들고 있으므로 `produce`로 감쌀 필요가 없습니다.

- **produce**: draft를 수정하면 새 상태 생성
- **주의**: draft 직접 수정 (return 없이)

```jsx
const state = { items: [{ id: 1, name: "A" }] };

const newState = produce(state, (draft) => {
  draft.items[0].name = "B";
});

console.log(state.items[0].name); // ?
console.log(newState.items[0].name); // ?
console.log(state === newState); // ?
```

- 정답 및 해설
    
    정답:
    
    ```jsx
    console.log(state.items[0].name); // "A"
    console.log(newState.items[0].name); // "B"
    console.log(state === newState); // false
    ```
    
    해설:
    
    `produce`는 원본 `state`를 직접 수정하지 않고, 내부에서 `draft`(Proxy)를 만들어 변경을 기록합니다. `draft.items[0].name = "B"`가 실행되면 Immer가 변경된 경로만 복사해서 새로운 상태 객체를 생성합니다. 그 결과 원본 `state`는 그대로 `"A"`를 유지하고, `newState`는 `"B"`를 가지며, 두 객체의 참조도 서로 다릅니다.

## 3-06. useShallow로 읽기 최적화하기 ⭐⭐⭐

### 새로운 문제: 불필요한 리렌더링

Immer로 "쓰기 코드"를 깔끔하게 만들었습니다. 이제 "읽기 코드"를 최적화할 차례입니다.

현재 Cart 컴포넌트를 보면:

```jsx
export default function Cart() {
  const items = useCartStore((store) => s.items);
  const increase = useCartStore(
    (store) => s.increaseQuantity
  );
  const decrease = useCartStore(
    (store) => s.decreaseQuantity
  );
  const remove = useCartStore((store) => s.removeItem);
  const clear = useCartStore((store) => s.clear);
  const totalPrice = useCartStore((store) =>
    s.getTotalPrice()
  );

  // ...
}
```

**store를 6번이나 호출**하고 있습니다. 이걸 한 번에 가져올 수는 없을까요?

```jsx
// 이렇게 하면 어떨까?
const {
  items,
  increase,
  decrease,
  remove,
  clear,
  totalPrice,
} = useCartStore((store) => ({
  items: s.items,
  increase: s.increaseQuantity,
  decrease: s.decreaseQuantity,
  remove: s.removeItem,
  clear: s.clear,
  totalPrice: s.getTotalPrice(),
}));

```

깔끔해 보이지만 **문제가 있습니다.**

### 객체 참조 문제

selector가 객체를 반환하면 매 렌더마다 **새 객체**가 생성됩니다:

```jsx
useCartStore((store) => ({
  items: s.items,
  increase: s.increaseQuantity,
}));

// 렌더1: { items: [...], increase: fn } ← 객체 A
// 렌더2: { items: [...], increase: fn } ← 객체 B (새로 생성!)
```

리액트는 **참조 비교**로 변경을 감지합니다:

```jsx
const obj1 = { count: 1 };
const obj2 = { count: 1 };

console.log(obj1 === obj2); // false (주소가 다름)
```

값은 같아도 주소가 다르면 "변경됐다"고 판단해서 리렌더링이 발생합니다.

### 문제 시나리오

```jsx
// Header 컴포넌트
const { totalCount } = useCartStore((store) => ({
  totalCount: store.getTotalCount(),
}));
```

1. 다른 컴포넌트에서 장바구니 상품 추가
2. store의 `items` 변경
3. `totalCount` 값은 변하지 않았는데도 새 객체 생성
4. Header 불필요하게 리렌더링

### useShallow 해결책

`useShallow`는 객체의 **값(얕은 비교)**으로 변경을 감지합니다:

```jsx
import { useShallow } from "zustand/react/shallow";

const { items, increase } = useCartStore(
  useShallow((store) => ({
    items: s.items,
    increase: s.increaseQuantity,
  }))
);
```

**동작 방식:**

```jsx
// 렌더1: { items: [A], increase: fn }
// 렌더2: { items: [A], increase: fn }

// useShallow가 비교:
// items는 같은 배열? ✅
// increase는 같은 함수? ✅
// → 리렌더링 안 함!
```

### 컴포넌트에 적용하기

### Header 수정

기존:

```jsx
const totalCount = useCartStore((store) =>
  store.getTotalCount()
);
```

**참고:** Header는 단일 값만 사용하므로, 사실 useShallow가 꼭 필요하지는 않습니다. 하나의 값만 가져올 때는 일반 selector만으로도 충분합니다.

하지만 여러 값을 함께 가져온다면 useShallow가 유용합니다:

```jsx
import { useShallow } from "zustand/react/shallow";

// 만약 여러 값을 가져온다면
const { totalCount, itemsLength } = useCartStore(
  useShallow((store) => ({
    totalCount: s.getTotalCount(),
    itemsLength: s.items.length,
  }))
);
```

**핵심 원칙:**

- **하나만 가져올 때**: 일반 selector 사용 (useShallow 불필요)
- **여러 개를 객체로 묶을 때**: useShallow 사용

**장점:**

- 여러 값을 한 번에 가져올 수 있음
- 객체 생성으로 인한 불필요한 리렌더링 방지

### Cart 수정

기존:

```jsx
export default function Cart() {
  const items = useCartStore((store) => s.items);
  const increase = useCartStore(
    (store) => s.increaseQuantity
  );
  const decrease = useCartStore(
    (store) => s.decreaseQuantity
  );
  const remove = useCartStore((store) => s.removeItem);
  const clear = useCartStore((store) => s.clear);
  const totalPrice = useCartStore((store) =>
    s.getTotalPrice()
  );

  // ...
}

```

useShallow 적용:

```jsx
import { useShallow } from "zustand/react/shallow";

export default function Cart() {
  const {
    items,
    increase,
    decrease,
    remove,
    clear,
    totalPrice,
  } = useCartStore(
    useShallow((store) => ({
      items: store.items,
      increase: store.increaseQuantity,
      decrease: store.decreaseQuantity,
      remove: store.removeItem,
      clear: store.clear,
      totalPrice: store.getTotalPrice(),
    }))
  );

  // 나머지 코드는 그대로
}

```

**개선점:**

- 6번의 store 호출 → 1번으로 감소
- 코드가 더 깔끔함
- 불필요한 리렌더링 방지

### ProductList는?

ProductList는 액션 하나만 사용하므로 useShallow가 필요 없습니다:


- **하나만 가져올 때**: useShallow 불필요
- **여러 개를 객체로 묶을 때**: useShallow 사용

### useShallow 없이:

```jsx
const { items, totalPrice } = useCartStore((store) => ({
```

### useShallow 사용 시기

### 사용하면 좋을 때

**1. 여러 값을 객체로 묶을 때:**

```jsx
const { user, posts, comments } = useStore(
  useShallow((store) => ({
    user: s.user,
    posts: s.posts,
    comments: s.comments,
  }))
);
```

**2. 계산된 값들을 함께 가져올 때:**

```jsx
const { totalCount, totalPrice, isEmpty } = useCartStore(
  useShallow((store) => ({
    totalCount: s.getTotalCount(),
    totalPrice: s.getTotalPrice(),
    isEmpty: s.items.length === 0,
  }))
);
```

### 불필요한 경우

**1. 하나만 가져올 때:**

```jsx
// useShallow 불필요

```

**2. 이미 원시 값일 때:**

```jsx
// useShallow 불필요

```

### 핵심 정리

1. **문제**: 객체 반환 시 매번 새 객체 생성 → 불필요한 리렌더링
2. **해결**: useShallow로 얕은 비교 → 값이 같으면 리렌더링 안 함
3. **사용 시기**: 여러 값을 객체로 묶을 때
4. **불필요**: 하나만 가져오거나 원시 값일 때

## 3-06. #Quiz useShallow 최적화

### 문제 1

다음 코드에서 불필요한 리렌더링이 발생하는 이유를 설명하세요:

```jsx
export default function Header() {
  const { totalCount } = useCartStore((store) => ({
    totalCount: store.getTotalCount(),
  }));

  return <span>🛒 {totalCount}개</span>;
}
```

장바구니에 상품을 추가하지 않았는데도 Header가 리렌더링됩니다. 왜 그럴까요?

- 정답 및 해설
    
    정답:
    
    selector가 객체를 반환하기 때문에 매번 새로운 객체가 생성되어 참조가 달라지기 때문입니다.
    
    해설:
    
    `useCartStore((store) => ({ totalCount: store.getTotalCount() }))`는 렌더링될 때마다 `{ totalCount: ... }` 형태의 새로운 객체를 만들어 반환합니다. 값이 동일하더라도 객체의 참조(주소)는 매번 달라지므로, 변경이 없는데도 변경이 발생한 것으로 판단되어 리렌더링이 일어납니다.
    
    객체가 아닌 값 자체를 반환하면 해결됩니다.
    
    ```jsx
    const totalCount = useCartStore((store) =>
      store.getTotalCount()
    );
    ```
    
    또는 여러 값을 객체로 묶어야 한다면 `useShallow`를 사용해 얕은 비교로 최적화할 수 있습니다.
    
    ```jsx
    const { totalCount } = useCartStore(
      useShallow((store) => ({
        totalCount: store.getTotalCount(),
      }))
    );
    ```

### 문제 2

다음 두 코드의 차이점을 설명하세요:

**코드 A:**

```jsx
const items = useCartStore((store) => store.items);
const addItem = useCartStore((store) => store.addItem);
```

**코드 B:**

```jsx
const { items, addItem } = useCartStore(
  useShallow((store) => ({
    items: store.items,
    addItem: store.addItem,
  }))
);
```

어느 것이 더 좋은 방법인가요?

- 정답 및 해설
    
    정답:
    
    상황에 따라 다르지만, 여러 값을 한 번에 가져오는 목적이라면 코드 B가 더 적합합니다.
    
    해설:
    
    코드 A는 `useCartStore`를 두 번 호출해 각각 따로 구독합니다. `items`는 변경될 수 있어 리렌더링을 유발하지만, `addItem`은 보통 함수 참조가 바뀌지 않아 리렌더링에 거의 영향을 주지 않습니다. 다만 코드가 길어질 수 있습니다.
    
    코드 B는 한 번의 구독으로 `items`와 `addItem`을 함께 가져오며, `useShallow`를 통해 얕은 비교로 객체 반환의 단점을 보완합니다. 여러 값을 가져올 때 코드가 더 깔끔하고 유지보수에 유리합니다.
    
    다만 하나의 값만 가져오는 상황이라면 `useShallow`로 객체를 만드는 방식이 오히려 불필요하게 복잡해질 수 있으므로, 그때는 단일 selector로 직접 값을 가져오는 것이 더 좋습니다.

### 문제 3

다음 코드에서 useShallow의 동작을 설명하세요:

```jsx
const { user, isAdmin } = useCartStore(
  useShallow((store) => ({
    user: s.user,
    isAdmin: s.user.role === "admin",
  }))
);

// 상황 1: user.name이 "A" → "B"로 변경
// 상황 2: user.age가 20 → 21로 변경 (role은 그대로 "user")
```

각 상황에서 컴포넌트가 리렌더링되나요?

- 정답 및 해설
    
    정답:
    
    상황 1도 리렌더링되고, 상황 2도 리렌더링됩니다.
    
    해설:
    
    `useShallow`는 반환된 객체의 1단계 속성만 얕게 비교합니다. 여기서 반환 객체의 `user` 값은 `s.user` 객체 자체입니다. `user.name`이나 `user.age`가 바뀌면 보통 `user` 객체가 새로 생성되어 참조가 달라지므로, `previous.user !== current.user`가 되어 리렌더링이 발생합니다. `isAdmin`은 `role`이 바뀌지 않으면 값이 같더라도, `user` 참조가 바뀌는 순간 이미 리렌더링 조건이 충족됩니다.
    
    최적화하려면 전체 `user` 객체를 가져오기보다 필요한 값만 selector에서 뽑아오는 방식이 더 효과적입니다.
    
    ```jsx
    const { userName, isAdmin } = useCartStore(
      useShallow((store) => ({
        userName: store.user.name,
        isAdmin: store.user.role === "admin",
      }))
    );
    ```
