### 01. 토픽 소개

이번 토픽에서는 자바스크립트로 서버와 통신하는 방법을 배워보겠습니다.

### AJAX란?

자바스크립트를 공부하다 보면 **AJAX**(에이젝스)라는 단어를 들어보셨을 겁니다.

**AJAX**는 **Asynchronous JavaScript And XML**의 줄임말로, "비동기 자바스크립트와 XML"이라는 뜻입니다.

AJAX가 무엇이고 왜 중요한지 알아봅시다.

### 동적 웹 페이지의 필요성

자바스크립트는 원래 **웹 페이지를 동적으로 만드는 데** 사용되었습니다.

- 버튼을 클릭하면 화면이 움직이고
- 스크롤을 하면 새로운 콘텐츠가 로딩되고
- 사용자의 동작에 따라 웹 페이지의 모습이 바뀝니다

HTML과 CSS만으로는 이런 동적인 웹 페이지를 만들 수 없었지만, 자바스크립트를 활용하면서 가능해졌습니다.

이런 다이나믹한 기능을 구현하는 데 필요한 핵심 기술 중 하나가 바로 **자바스크립트로 서버와 데이터를 주고받는 것**입니다.

### 실제 사례: 좋아요 버튼

좋아요 버튼을 예시로 들어보겠습니다.

보통 좋아요 버튼을 누르면:

1. 버튼의 모습이 바뀌고
2. 옆에 숫자가 올라갑니다

이때 사용자가 버튼을 누르는 순간, **자바스크립트로 서버에 리퀘스트를 보내서** 좋아요 정보를 전달해야 합니다.

**왜 서버에 전달해야 할까요?**

- 사용자가 좋아요를 눌렀다는 사실을 기억해야 합니다
- 다음에 웹사이트를 방문할 때 좋아요 상태가 유지되어야 합니다
- 다른 사람들에게도 정확한 좋아요 개수가 표시되어야 합니다

### 실제 사례: 무한 스크롤

SNS 피드 같은 페이지를 생각해봅시다.

아래로 스크롤하면 추가적인 콘텐츠가 계속 로딩됩니다. 다음 페이지로 넘어가는 버튼을 클릭할 필요가 없죠.

**어떻게 동작할까요?**

1. 사용자가 특정 지점까지 스크롤합니다
2. 자바스크립트로 서버에 리퀘스트를 보내 추가 데이터를 요청합니다
3. 데이터를 받아오면 페이지 아래 부분에 추가해서 보여줍니다

### 비동기 처리의 중요성

여기서 중요한 점은, **이런 웹 통신은 비동기적으로 이루어져야 한다**는 것입니다.

**왜 비동기여야 할까요?**

자바스크립트로 리퀘스트를 보낸 후 서버의 리스폰스가 돌아오려면 비교적 오랜 시간이 걸릴 수 있습니다.

- **동기 방식**: 리스폰스를 기다리는 동안 아무 코드도 실행하지 않음 → 웹 브라우저가 멈춘 것처럼 반응 없음
- **비동기 방식**: 리스폰스를 기다리는 동안 다른 자바스크립트 코드를 실행 → 웹사이트가 계속 반응함

비동기 방식이 훨씬 효율적이고 사용자 경험도 좋습니다.

### AJAX 정리

정리하자면:

**동적 웹사이트를 만들기 위해서는:**

- 비동기적으로 자바스크립트 코드로 서버에 리퀘스트를 보내고
- 리스폰스를 받아오는 기술이 필요합니다

이 기술을 바로 **AJAX (Asynchronous JavaScript And XML)**라고 부릅니다.

**이름에 XML이 들어간 이유:**

예전에는 웹으로 데이터를 주고받을 때 **XML**이라는 형식을 많이 사용했습니다. 하지만 요즘은 XML보다는 **JSON** 형식을 훨씬 많이 사용합니다.

### AJAX 구현 방법의 발전

- 페이지 새로고침 없이 자바스크립트 코드로 리퀘스트를 보낼 수 있음
- 많은 인기를 끌었음

**단점:**

- 문법이 다소 복잡함
- 직관적이지 않은 경우가 있음

### 2. fetch와 axios (현재)

### fetch 함수

**fetch**는 2015년 ES6 표준에 소개된 문법입니다.

**특징:**

- 정말 쉽게 리퀘스트를 보낼 수 있음
- Promise 기반 문법을 사용해서 비동기 처리가 깔끔함
- 자바스크립트로 리퀘스트를 보내는 **정석**이라고 할 수 있음

### axios 라이브러리

**axios**는 fetch와 문법이 비슷하지만, 몇 가지 차이가 있습니다.

**특징:**

- 리퀘스트나 리스폰스를 다루는 방법이 조금 다름
- 실무에 유용한 편의 기능을 여러 가지 제공함
- 자바스크립트 웹 개발을 할 때 큰 도움이 됨

### 04. HTTP 기본 개념 ⭐⭐⭐

서버와 통신하는 방법을 배우기 전에, 먼저 **HTTP**가 무엇인지 이해해야 합니다.

### HTTP란?

**HTTP**(HyperText Transfer Protocol)는 웹에서 데이터를 주고받기 위한 **약속**입니다. 마치 우편 시스템처럼 정해진 규칙에 따라 정보를 주고받습니다.

예를 들어, 여러분이 웹 브라우저에서 `https://www.google.com`을 입력하면:

1. 여러분의 브라우저가 구글 서버에 "메인 페이지를 보여주세요"라고 **요청(Request)** 을 보냅니다
2. 구글 서버가 "여기 있습니다"라고 **응답(Response)** 을 보냅니다
3. 브라우저가 받은 HTML, CSS, JavaScript 파일로 화면을 그립니다

이 모든 과정이 **HTTP 프로토콜**을 따라 이루어집니다.

### 클라이언트와 서버

### 1. 클라이언트 (Client)

**요청을 보내는 쪽**입니다.

- 웹 브라우저 (Chrome, Safari 등)
- 모바일 앱
- 여러분이 작성하는 JavaScript 코드

### 2. 서버 (Server)

**요청을 받아서 처리하고 응답을 보내는 쪽**입니다.

- 데이터베이스에서 정보를 가져옴
- 파일을 저장하거나 삭제
- 계산이나 처리를 수행
- 결과를 클라이언트에게 돌려줌

**예시:**

```
클라이언트: "아이디가 42인 설문 응답 데이터를 주세요"
    ↓ (리퀘스트)
서버: 데이터베이스에서 조회
    ↓ (리스폰스)
클라이언트: { id: 42, mbti: 'ENFP', colorCode: '#CA6952' }
```

### 리퀘스트 (Request)

클라이언트가 서버에 보내는 **요청 메시지**입니다.

### 리퀘스트의 구성 요소

### 1. URL (주소)

어디로 요청을 보낼지 지정합니다.

```
https://learn.codeit.kr/api/color-surveys
```

URL의 각 부분:

- `https://`: 프로토콜 (보안 HTTP)
- `learn.codeit.kr`: 도메인 (서버 주소)
- `/api/color-surveys`: 경로 (어떤 리소스를 원하는지)

### 2. HTTP 메소드

**무슨 동작을 할지** 알려줍니다.

| 메소드 | 의미 | 예시 |
| --- | --- | --- |
| GET | 데이터 조회 | 설문 목록 가져오기 |
| POST | 새 데이터 생성 | 새 설문 응답 제출하기 |
| PATCH | 기존 데이터 수정 | 설문 응답 수정하기 |
| DELETE | 데이터 삭제 | 설문 응답 삭제하기 |
| PUT | 데이터 전체 교체 | 설문 응답 전체 업데이트 |

**GET과 POST의 차이:**

```
GET 리퀘스트:
"https://learn.codeit.kr/api/color-surveys 목록을 보여주세요"
→ 서버에서 데이터를 가져오기만 함

POST 리퀘스트:
"https://learn.codeit.kr/api/color-surveys에 새 데이터를 저장해주세요"
→ 서버에 데이터를 보내서 생성함
```

### 3. 헤더 (Headers)

요청에 대한 **부가 정보**를 담습니다.

```jsx
{
  'Content-Type': 'application/json',  // "JSON 형식으로 보냅니다"
  'Authorization': 'Bearer token123',  // "인증 토큰입니다"
  'User-Agent': 'Chrome/120.0'         // "크롬 브라우저에서 보냅니다"
}
```

자주 사용하는 헤더:

- **Content-Type**: 보내는 데이터의 형식
    - `application/json`: JSON 데이터
    - `text/html`: HTML 문서
    - `multipart/form-data`: 파일 업로드
- **Authorization**: 인증 정보
- **Accept**: 받고 싶은 응답 형식

### 4. 바디 (Body)

서버로 보내는 **실제 데이터**입니다. (GET과 DELETE는 바디가 없습니다)

```jsx
// POST 리퀘스트의 바디
{
  "mbti": "ENFP",
  "colorCode": "#CA6952",
  "password": "1234"
}
```

### 리스폰스 (Response)

서버가 클라이언트에 보내는 **응답 메시지**입니다.

### 리스폰스의 구성 요소

### 1. 상태 코드 (Status Code)

요청이 **어떻게 처리되었는지** 알려주는 숫자입니다.

| 범위 | 의미 | 예시 |
| --- | --- | --- |
| **2xx** | 성공 | 200 OK, 201 Created |
| **3xx** | 리다이렉션 | 301 Moved Permanently |
| **4xx** | 클라이언트 오류 | 400 Bad Request, 401 Unauthorized, 404 Not Found |
| **5xx** | 서버 오류 | 500 Internal Server Error, 503 Service Unavailable |

**자주 보는 상태 코드:**

```
200 OK: 성공
201 Created: 새로운 리소스 생성 성공
204 No Content: 성공했지만 보낼 데이터 없음

400 Bad Request: 잘못된 요청 (필수 필드 누락 등)
401 Unauthorized: 인증 필요
403 Forbidden: 권한 없음
404 Not Found: 리소스를 찾을 수 없음

500 Internal Server Error: 서버 내부 오류
503 Service Unavailable: 서버가 일시적으로 사용 불가
```

**예시:**

```
GET /api/color-surveys/42
→ 상태 코드: 200 OK (성공)

GET /api/color-surveys/99999
→ 상태 코드: 404 Not Found (존재하지 않는 리소스)
```

### 2. 헤더 (Headers)

응답에 대한 **부가 정보**를 담습니다.

```jsx
{
  'Content-Type': 'application/json',      // JSON 형식으로 보냅니다
  'Content-Length': '1234',                // 응답 크기
  'Date': 'Wed, 11 Dec 2024 12:00:00 GMT' // 응답 시간
}
```

### 3. 바디 (Body)

서버가 보내는 **실제 데이터**입니다.

```jsx
// GET 리퀘스트에 대한 응답 바디
{
  "id": 42,
  "mbti": "ENFP",
  "colorCode": "#CA6952",
  "createdAt": 1667969077,
  "updatedAt": 1667969077
}
```

### HTTP 통신의 흐름

전체 과정을 정리하면:

```
1. 클라이언트가 리퀘스트 생성
   - URL: https://learn.codeit.kr/api/color-surveys
   - 메소드: GET
   - 헤더: { 'Accept': 'application/json' }

2. 서버가 리퀘스트 수신 및 처리
   - 데이터베이스에서 설문 응답 목록 조회
   - JSON 형식으로 변환

3. 서버가 리스폰스 생성
   - 상태 코드: 200
   - 헤더: { 'Content-Type': 'application/json' }
   - 바디: { count: 75, results: [...] }

4. 클라이언트가 리스폰스 수신
   - 상태 코드 확인 (200 OK)
   - 바디 데이터 파싱
   - 화면에 표시
```

### HTTP 메시지 형식

### GET 리퀘스트와 리스폰스

**클라이언트가 서버에 보내는 리퀘스트:**

```
GET /api/color-surveys HTTP/1.1
Host: learn.codeit.kr
Accept: application/json
```

- **첫 줄**: 메소드(GET) + 경로 + HTTP 버전
- **헤더**: Host, Accept 등 부가 정보
- **바디**: GET 요청은 바디 없음

**서버가 클라이언트에 보내는 리스폰스:**

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 1234

{"count":75,"next":"...","previous":null,"results":[...]}
```

- **첫 줄**: HTTP 버전 + 상태 코드(200) + 상태 메시지(OK)
- **헤더**: Content-Type, Content-Length 등
- **빈 줄**: 헤더와 바디 구분
- **바디**: 실제 데이터 (JSON)

### POST 리퀘스트와 리스폰스

**클라이언트가 서버에 보내는 리퀘스트:**

```
POST /api/color-surveys HTTP/1.1
Host: learn.codeit.kr
Content-Type: application/json
Content-Length: 67

{"mbti":"ENFP","colorCode":"#CA6952","password":"1234"}
```

- **첫 줄**: 메소드(POST) + 경로 + HTTP 버전
- **헤더**: Content-Type으로 바디 형식 지정
- **바디**: 서버로 보내는 데이터 (JSON)

**서버가 클라이언트에 보내는 리스폰스:**

```
HTTP/1.1 201 Created
Content-Type: application/json

{"id":78,"mbti":"ENFP","colorCode":"#CA6952",...}
```

- **상태 코드**: 201 Created
- **바디**: 생성된 리소스 정보

### HTTPS vs HTTP

- **HTTP**: 일반 텍스트로 통신 (보안 취약)
- **HTTPS**: 암호화된 통신 (보안)

> 중요: 실제 서비스에서는 항상 HTTPS를 사용해야 합니다.
> 
> 
> 로그인 정보, 개인정보 등이 암호화되지 않으면 중간에 가로챌 수 있습니다.
> 

```
HTTP:  클라이언트 ──── 평문 데이터 ──── 서버
                       ↑
                   누구나 볼 수 있음!

HTTPS: 클라이언트 ──── 암호화된 데이터 ──── 서버
                       ↑
                   해독 불가능!
```

### REST API

**REST**(Representational State Transfer)는 HTTP를 효율적으로 사용하기 위한 **설계 원칙**입니다.

### REST API의 특징

1. **리소스 기반**: URL이 리소스를 나타냄

```
GET /api/color-surveys           → 설문 응답 목록
GET /api/color-surveys/42        → ID 42 설문 응답
POST /api/color-surveys          → 새 설문 응답 생성
```

1. **HTTP 메소드 활용**: 동사는 메소드로, 명사는 URL로

```
❌ 좋지 않은 설계:
POST /api/createSurvey
POST /api/deleteSurvey

✅ REST API 설계:
POST /api/color-surveys          → 생성
DELETE /api/color-surveys/42     → 삭제
```

1. **무상태(Stateless)**: 각 요청이 독립적

```
GET /api/color-surveys/42 HTTP/1.1
Authorization: Bearer token123

// 서버는 이전 요청을 기억하지 않음
// 매 요청마다 필요한 모든 정보(인증 토큰 등)를 포함해야 함
```

### JSON (JavaScript Object Notation)

HTTP로 데이터를 주고받을 때 가장 많이 사용하는 **데이터 형식**입니다.

```jsx
// JavaScript 객체
const survey = {
  mbti: "ENFP",
  colorCode: "#CA6952",
};

// JSON 문자열로 변환
const jsonString = JSON.stringify(survey);
console.log(jsonString);
// '{"mbti":"ENFP","colorCode":"#CA6952"}'

// JSON 문자열을 JavaScript 객체로 변환
const parsedSurvey = JSON.parse(jsonString);
console.log(parsedSurvey);
// { mbti: 'ENFP', colorCode: '#CA6952' }
```

**왜 JSON을 사용할까?**

- 사람이 읽기 쉬움
- JavaScript에서 쉽게 변환 가능
- 거의 모든 프로그래밍 언어에서 지원
- 가볍고 효율적

### 정리

HTTP 통신의 핵심 개념:

1. **HTTP**: 웹에서 데이터를 주고받는 프로토콜
2. **클라이언트-서버**: 요청을 보내는 쪽과 처리하는 쪽
3. **리퀘스트**: URL + 메소드 + 헤더 + 바디
4. **리스폰스**: 상태 코드 + 헤더 + 바디
5. **HTTP 메소드**: GET(조회), POST(생성), PATCH(수정), DELETE(삭제)
6. **상태 코드**: 2xx(성공), 4xx(클라이언트 오류), 5xx(서버 오류)
7. **JSON**: 데이터를 주고받을 때 사용하는 형식

## 2. fetch

### 01. fetch 기본 문법 ⭐⭐⭐

이번 시간에는 `fetch` 함수를 사용해서 서버에 리퀘스트를 보내는 방법을 배워보겠습니다.

### fetch 함수란?

`fetch`는 JavaScript에서 서버에 HTTP 리퀘스트를 보낼 때 사용하는 함수입니다. 이 함수를 사용하면 서버에서 데이터를 가져오거나 서버로 데이터를 보낼 수 있습니다.

### 기본 사용법

### 1. 리퀘스트 보내기

가장 기본적인 형태는 이렇습니다:

```jsx
fetch("https://learn.codeit.kr/api/color-surveys");
```

`fetch` 함수에 URL을 전달하면 됩니다. 아무런 옵션도 지정하지 않으면 기본적으로 **GET 리퀘스트**를 보냅니다.

### 2. 프로미스와 await

`fetch` 함수는 **프로미스(Promise)** 를 반환합니다. 서버로부터 응답이 도착하면 이 프로미스는 fulfilled 상태가 되고, 응답(Response)을 결과값으로 갖게 됩니다.

따라서 응답을 받아오려면 `await` 키워드를 사용해야 합니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
```

> 중요: fetch는 HTTP 에러(404, 500 등)가 발생해도 rejected되지 않습니다!
> 
> 
> 네트워크 요청만 성공하면 무조건 fulfilled됩니다.
> 
> rejected되는 경우: DNS 실패, 네트워크 끊김, CORS 차단 등
> 

### 3. 응답 확인하기

응답 객체를 콘솔에 출력해보겠습니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
console.log(response);
```

응답 객체에는 상태 코드, 헤더, 바디 등 다양한 정보가 포함되어 있습니다.

### 응답의 주요 정보

### 1. 상태 코드 (Status Code)

서버가 리퀘스트를 어떻게 처리했는지 알려주는 숫자입니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
console.log("Status:");
console.log(response.status);
```

200이 출력되면 성공적으로 데이터를 가져온 것입니다.

### response.ok로 성공 여부 확인하기

`response.ok`는 상태 코드가 200~299 범위일 때 `true`입니다.

**실무에서는 반드시 확인해야 합니다:**

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);

if (!response.ok) {
  throw new Error(`HTTP Error! Status: ${response.status}`);
}

const data = await response.json();
```

왜 필요할까요?

`fetch`는 404, 500 에러에도 `catch`로 가지 않습니다.

따라서 `response.ok`를 직접 확인해야 합니다.

```jsx
// 404 에러여도 catch로 안 감!
try {
  const response = await fetch(
    "https://learn.codeit.kr/api/wrong-endpoint"
  );
  console.log(response.status); // 404지만 여기서 출력됨
} catch (e) {
  console.log("여기로 안 옴!"); // 네트워크 에러일 때만 실행
}
```

### 2. 헤더 (Headers)

응답에 대한 부가 정보를 담고 있습니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
console.log("Headers:");
console.log(response.headers);
```

### 3. 바디 (Body)

바디는 실제 데이터 내용을 담고 있습니다. 바디 내용을 가져오려면 **메소드**를 사용해야 합니다.

> 참고: response.body는 내부적으로 ReadableStream이라는 형태입니다.
> 
> 
> `console.log(response)`를 해도 실제 데이터는 보이지 않습니다.
> 
> **반드시 `.json()` 또는 `.text()` 메소드를 호출해야** 데이터를 읽을 수 있습니다.
> 

### JSON 데이터 가져오기

대부분의 API는 JSON 형식의 데이터를 반환합니다. JSON 데이터를 가져오려면 `json()` 메소드를 사용합니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
const data = await response.json();
console.log(data);
```

`json()` 메소드도 프로미스를 반환하기 때문에 `await`를 붙여야 합니다. 이 메소드는 JSON 문자열을 JavaScript 객체로 자동 변환해줍니다.

> 참고: API에 따라 텍스트나 다른 형식을 반환하는 경우도 있습니다.
> 
> - `response.text()`: 텍스트로 반환
> - `response.blob()`: 이미지, 파일 등 바이너리 데이터로 반환
> - `response.formData()`: FormData 객체로 반환
> 
> 하지만 대부분의 RESTful API는 JSON을 사용하므로, 이 강의에서는 `json()` 메소드를 주로 사용합니다.
> 

### 데이터 활용하기

서버에서 받은 데이터는 JavaScript 객체이므로, 일반적인 객체 문법을 사용해 데이터를 다룰 수 있습니다.

예를 들어 이런 형태의 데이터를 받았다고 가정해봅시다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
const data = await response.json();
console.log(data);
// {
//   count: 75,
//   next: 'https://learn.codeit.kr/api/color-surveys/?offset=10&limit=10',
//   previous: null,
//   results: [
//     { id: 77, mbti: 'ESTJ', colorCode: '#B01CFF', createdAt: 1765437215000, updatedAt: 1765437215000 },
//     { id: 75, mbti: 'ENTJ', colorCode: '#20FC03', createdAt: 1765295699000, updatedAt: 1765295699000 },
//     { id: 74, mbti: 'ENTJ', colorCode: '#B01CFF', createdAt: 1765295460000, updatedAt: 1765295460000 },
//     ...
//   ]
// }
```

응답에는 페이지네이션 정보와 설문 응답 배열이 포함되어 있습니다:

- `count`: 전체 설문 응답 개수
- `next`: 다음 페이지 URL
- `previous`: 이전 페이지 URL (첫 페이지이므로 null)
- `results`: 현재 페이지의 설문 응답 배열 (기본 10개)

구조 분해(destructuring)를 사용해서 필요한 데이터만 추출할 수 있습니다:

```jsx
const { count, results } = data;
console.log(count); // 75
console.log(results); // 설문 응답 배열
```

배열의 첫 번째 설문 응답 정보를 더 자세히 살펴보겠습니다:

```jsx
const survey = results[0];
const { id, mbti, colorCode, createdAt } = survey;
console.log(id, mbti, colorCode, createdAt);
// 출력: 1 'ENFP' '#CA6952' 1667969077
```

### 정리

`fetch` 함수를 사용하면 서버와 통신할 수 있습니다:

1. **기본 사용법**: `fetch(url)` - URL을 전달하면 GET 리퀘스트를 보냄
2. **프로미스**: `fetch`는 프로미스를 반환하므로 `await` 사용 필요
3. **응답 정보 접근**:
    - 성공 여부: `response.ok` (200~299이면 true)
    - 상태 코드: `response.status`
    - 헤더: `response.headers`
    - JSON 바디: `await response.json()`
    - 텍스트 바디: `await response.text()`
4. **중요**: `fetch`와 `json()` 모두 프로미스를 반환하므로 각각 `await` 필요

### 실무에서 알아야 할 fetch 특성

### 1. HTTP 에러에도 rejected되지 않음

`fetch`는 404, 500 같은 HTTP 에러에도 `catch`로 가지 않습니다.

항상 `response.ok`를 확인하는 습관을 들이세요:

```jsx
const response = await fetch(url);
if (!response.ok) {
  throw new Error(`HTTP Error! ${response.status}`);
}
const data = await response.json();
```

### 2. 기본 타임아웃이 없음

`fetch`는 서버가 응답하지 않으면 **영원히 대기**할 수 있습니다.

타임아웃이 필요하면 `AbortController`를 사용합니다:

```jsx
const controller = new AbortController();
setTimeout(() => controller.abort(), 5000); // 5초 후 취소

const response = await fetch(url, {
  signal: controller.signal,
});
```

### 3. CORS 환경에서 쿠키 전송

기본적으로 `fetch`는 다른 도메인에 쿠키를 자동 전송하지 않습니다.

쿠키가 필요한 경우:

```jsx
fetch(url, { credentials: "include" });
```

        - `await response.json()`
        - `await response.text()`
    - 나머지 설명은 모두 올바릅니다:
        1. 기본적으로 GET 요청을 보냅니다.
        2. `fetch`는 프로미스를 반환하므로 `await`나 `.then()`이 필요합니다.
        3. 첫 번째 아규먼트는 URL입니다.

### 02. 다양한 GET 리퀘스트 보내기 ⭐⭐⭐

이번 시간에는 GET 리퀘스트를 보낼 때 여러 가지 옵션을 추가하는 방법을 배워보겠습니다.

### 쿼리 파라미터 사용하기

서버에 리퀘스트를 보낼 때 URL 뒤에 **쿼리 파라미터(Query Parameter)** 를 추가할 수 있습니다. 쿼리 파라미터는 `?` 뒤에 `key=value` 형태로 작성하며, 여러 개를 사용할 때는 `&`로 연결합니다.

```jsx
// 쿼리 파라미터 예시
// mbti: MBTI 유형으로 필터링
// limit: 가져올 데이터 개수
// offset: 건너뛸 데이터 개수
```

### 특정 조건으로 데이터 필터링

예를 들어, ENFP MBTI 유형의 응답만 가져오고 싶다면:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys?mbti=ENFP"
);

if (!response.ok) {
  throw new Error(`HTTP Error! Status: ${response.status}`);
}

const data = await response.json();
console.log(data);
```

`mbti=ENFP`라는 쿼리 파라미터를 추가하면, 서버는 ENFP 유형의 설문 응답만 반환합니다.

> 참고: fetch는 404, 500 같은 HTTP 에러에도 rejected되지 않습니다.
> 
> 
> 항상 `response.ok`를 확인하는 습관을 들이세요. (2-01 강의 참고)
> 

### 페이지네이션 (Pagination)

**페이지네이션**은 모든 데이터를 한 번에 보내지 않고, 페이지처럼 나눠서 보내는 방식입니다. 데이터가 많을 때 성능을 위해 사용됩니다.

### 페이지네이션 응답 구조

`offset/limit` 방식의 페이지네이션 응답은 다음과 같은 구조를 가집니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
const data = await response.json();
console.log(data);
// {
//   count: 75,  // 전체 데이터 개수
//   next: 'https://learn.codeit.kr/api/color-surveys/?offset=10&limit=10',  // 다음 페이지 URL
//   previous: null,  // 이전 페이지 URL (첫 페이지이므로 null)
//   results: [
//     { id: 77, mbti: 'ESTJ', colorCode: '#B01CFF', createdAt: 1765437215000, updatedAt: 1765437215000 },
//     { id: 75, mbti: 'ENTJ', colorCode: '#20FC03', createdAt: 1765295699000, updatedAt: 1765295699000 },
//     // ... 10개의 설문 응답 데이터
//   ]
// }
```

- `count`: 전체 데이터 개수
- `next`: 다음 페이지를 가져올 수 있는 URL
- `previous`: 이전 페이지를 가져올 수 있는 URL
- `results`: 현재 페이지의 실제 데이터 배열

### offset과 limit

페이지네이션에서 자주 사용하는 두 가지 쿼리 파라미터:

- **offset**: 건너뛸 데이터 개수
- **limit**: 가져올 데이터 개수

예를 들어, 전체 데이터가 150개이고 한 페이지에 10개씩 보여준다면:

- 1페이지: `offset=0, limit=10` (처음 10개)
- 2페이지: `offset=10, limit=10` (11~20번째 데이터)
- 3페이지: `offset=20, limit=10` (21~30번째 데이터)

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys?offset=10&limit=10"
);
const data = await response.json();
console.log(data);
// 11번째부터 20번째까지의 설문 응답 데이터를 가져옴
```

> 참고: 페이지네이션 방식은 API마다 다릅니다.
> 
> - `offset/limit` 방식: 위치 기반 (이 강의에서 설명)
> - `page/size` 방식: 페이지 번호 기반 (`?page=2&size=10`)
> - `cursor` 방식: 특정 위치 토큰 기반 (SNS API에서 많이 사용)
> 
> API 문서를 확인하여 해당 API가 어떤 방식을 사용하는지 파악하세요.
> 

### 여러 쿼리 파라미터 함께 사용하기

필터링과 페이지네이션을 동시에 사용할 수 있습니다:

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys?mbti=ENFP&offset=0&limit=10"
);
const data = await response.json();
console.log(data);
// ENFP 유형에서 처음 10개의 설문 응답을 가져옴
```

### URL 객체 사용하기

쿼리스트링을 직접 작성하는 대신, JavaScript의 `URL` 객체를 사용하면 더 깔끔하게 URL을 만들 수 있습니다.

### URL 객체 생성

```jsx
const url = new URL(
  "https://learn.codeit.kr/api/color-surveys"
);
```

### 쿼리 파라미터 추가

`searchParams`에는 두 가지 메소드가 있습니다:

### append() vs set()

```jsx
const url = new URL(
  "https://learn.codeit.kr/api/color-surveys"
);

// append: 값을 추가 (기존 키가 있어도 유지)
url.searchParams.append("mbti", "ENFP");
url.searchParams.append("mbti", "ISTJ");
// 결과: ?mbti=ENFP&mbti=ISTJ (중복 키)

// set: 값을 설정 (기존 키가 있으면 덮어쓸)
url.searchParams.set("mbti", "ENFP");
url.searchParams.set("mbti", "ISTJ");
// 결과: ?mbti=ISTJ (마지막 값만 유지)

```

> 실무 팁: 대부분의 API는 중복 키를 원하지 않으므로 set()을 사용하는 것이 더 안전합니다.
> 
> 
> 배열 파라미터가 필요한 경우에만 `append()`를 사용하세요.
> 

### 예제: URL 객체로 리퀘스트 보내기 보내기

만들어진 URL 객체를 `fetch`에 직접 전달할 수 있습니다:

```jsx
const url = new URL(
  "https://learn.codeit.kr/api/color-surveys"
);
url.searchParams.set("mbti", "ENFP");
url.searchParams.set("offset", 10);
url.searchParams.set("limit", 10);

const response = await fetch(url);

if (!response.ok) {
  throw new Error(`HTTP Error! Status: ${response.status}`);
}

const data = await response.json();
console.log(data);
// {
//   count: 9,
//   next: null,
//   previous: 'https://learn.codeit.kr/api/color-surveys/?mbti=ENFP&offset=0&limit=10',
//   results: []
// }
// ENFP 유형이 총 9개뿐이라서, offset=10으로 설정하면 가져올 데이터가 없음
```

URL 객체를 사용하면:

- 쿼리스트링을 직접 작성하지 않아도 됨
- 특수문자 인코딩을 자동으로 처리
- `set()`/`append()`로 파라미터 관리가 편리함
- 코드가 더 읽기 쉬워짐

### 특정 ID의 데이터 가져오기

대부분의 API는 특정 항목의 상세 정보를 가져오는 엔드포인트를 제공합니다. URL 경로에 ID를 포함시키면 됩니다.

### 여러 항목 가져오기

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys"
);
const data = await response.json();
console.log(data);
// { count: 75, results: [/* 여러 설문 응답 */] }
```

### 특정 항목 하나만 가져오기

```jsx
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys/42"
);
const data = await response.json();
console.log(data);
// { id: 42, mbti: 'ENTJ', colorCode: '#000000', createdAt: 1764225662000, updatedAt: 1764225662000 }
```

ID `42`에 해당하는 설문 응답의 상세 정보만 반환됩니다.

### 정리

GET 리퀘스트를 다양하게 활용하는 방법을 배웠습니다:

1. **쿼리 파라미터**: `?key=value&key2=value2` 형태로 조건 추가
2. **페이지네이션**: `offset`과 `limit`으로 데이터를 나눠서 가져오기
3. **URL 객체**: `new URL()`과 `searchParams.set()`으로 깔끔하게 URL 만들기
4. **특정 ID 조회**: URL 경로에 ID를 포함해서 하나의 데이터만 가져오기

> 기억하세요:
> 
> - `fetch`는 HTTP 에러에도 rejected되지 않으므로 `response.ok` 확인 필수!
> - `searchParams.set()`은 값을 덮어쓰고, `append()`는 추가합니다.

이제 여러분은 서버에서 원하는 데이터를 효율적으로 가져올 수 있습니다!

### 04. POST 리퀘스트 보내기 ⭐⭐⭐

### POST 리퀘스트 보내기

이번 시간에는 서버에 새로운 데이터를 생성하는 POST 리퀘스트를 보내는 방법을 배워보겠습니다.

### GET vs POST

지금까지는 GET 리퀘스트만 사용했습니다. GET은 서버에서 데이터를 **가져올 때** 사용합니다.

POST는 서버에 **새로운 데이터를 생성하거나, 서버에서 어떤 작업을 수행해 달라고 요청할 때** 사용합니다. 예를 들어:

- 회원가입 정보 전송
- 게시글 작성
- 댓글 등록
- 설문 응답 제출
- 로그인 요청 (`/auth/login`)
- 결제 처리 (`/payments/charge`)

### POST 리퀘스트 보내기

### 1. method 옵션 설정

`fetch` 함수의 두 번째 아규먼트로 옵션 객체를 전달할 수 있습니다. POST 리퀘스트를 보내려면 `method` 프로퍼티를 설정합니다:

```jsx
fetch("https://learn.codeit.kr/api/color-surveys", {
  method: "POST",
});
```

다른 HTTP 메소드도 같은 방식으로 설정할 수 있습니다:

- `'GET'`: 데이터 조회 (기본값)
- `'POST'`: 데이터 생성
- `'PATCH'` 또는 `'PUT'`: 데이터 수정
- `'DELETE'`: 데이터 삭제

### 2. 바디 데이터 준비

POST 리퀘스트는 보통 서버에 전송할 데이터를 함께 보냅니다. 예를 들어 색상 설문 응답을 등록한다면:

```jsx
const surveyData = {
  mbti: "ENFP",
  colorCode: "#CA6952",
  password: "1234",
};
```

### 3. JSON 문자열로 변환

JavaScript 객체를 서버로 보내려면 JSON 문자열로 변환해야 합니다:

```jsx
const surveyData = {
  mbti: "ENFP",
  colorCode: "#CA6952",
  password: "1234",
};

fetch("https://learn.codeit.kr/api/color-surveys", {
  method: "POST",
  body: JSON.stringify(surveyData),
});
```

`JSON.stringify()` 메소드는 JavaScript 객체를 JSON 문자열로 변환해줍니다.

### 4. Content-Type 헤더 설정

서버에 어떤 형식의 데이터를 보내는지 알려주기 위해 `Content-Type` 헤더를 설정하는 것이 좋습니다:

```jsx
const surveyData = {
  mbti: "ENFP",
  colorCode: "#CA6952",
  password: "1234",
};

fetch("https://learn.codeit.kr/api/color-surveys", {
  method: "POST",
  body: JSON.stringify(surveyData),
  headers: {
    "Content-Type": "application/json",
  },
});
```

`'Content-Type': 'application/json'`은 JSON 형식의 데이터를 보낸다는 의미입니다.

> 중요: 이 헤더가 없으면 서버가 body를 JSON으로 인식하지 못합니다.
> 
> 
> Express에서는 `req.body`가 비어 있거나, 에러가 발생할 수 있습니다.
> 
> **JSON 데이터를 보낼 때는 항상 이 헤더를 포함하세요!**
> 

필요하다면 다른 헤더들도 추가할 수 있습니다:

```jsx
headers: {
  'Content-Type': 'application/json',
  'Authorization': 'Bearer token123',
  'Custom-Header': 'value',
}
```

### 5. 응답 처리하기

POST 리퀘스트의 응답을 처리하는 방법은 GET과 동일합니다.

**단, `fetch`는 HTTP 에러(400, 404, 500 등)에도 rejected되지 않습니다!**

따라서 `response.ok`를 확인해야 합니다:

```jsx
const surveyData = {
  mbti: "ENFP",
  colorCode: "#CA6952",
  password: "1234",
};

const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys",
  {
    method: "POST",
    body: JSON.stringify(surveyData),
    headers: {
      "Content-Type": "application/json",
    },
  }
);

// HTTP 에러 확인 (필수!)
if (!response.ok) {
  throw new Error(`HTTP Error! Status: ${response.status}`);
}

const data = await response.json();
console.log(data);
```

> 왜 response.ok 확인이 필요할까요?
> 
> - `fetch`는 네트워크 에러에서만 rejected됩니다.
> - 서버가 400, 401, 500을 보내도 Promise는 fulfilled 상태입니다.
> - `response.ok`가 `false`인데 `response.json()`을 호출하면,
>     
>     서버가 JSON이 아닌 응답(HTML 에러 페이지 등)을 보내면 `SyntaxError`가 발생합니다.
>     

서버는 보통 생성된 데이터를 응답으로 반환합니다. 예를 들어:

```jsx
// 서버 응답 예시
{
  id: 157,
  mbti: 'ENFP',
  colorCode: '#CA6952',
  password: '1234',
  createdAt: 1667969077,
  updatedAt: 1667969077
}
```

생성된 리뷰에 `id`와 `createdAt` 같은 서버에서 추가한 정보가 포함되어 있습니다.

### 실제 사용 예시

웹 애플리케이션에서 사용자가 색상 설문에 응답하는 상황을 생각해봅시다:

1. 사용자가 폼에 MBTI와 좋아하는 색상을 선택
2. 제출 버튼 클릭
3. 입력된 값으로 JavaScript 객체 생성
4. POST 리퀘스트로 서버에 전송
5. 서버가 데이터베이스에 저장하고 생성된 응답 반환
6. 화면에 제출 완료 메시지 표시

```jsx
// 사용자 입력 받기 (예시)
const mbti = "ENFP";
const colorCode = "#CA6952";
const password = "1234";

// 데이터 객체 생성
const surveyData = {
  mbti: mbti,
  colorCode: colorCode,
  password: password,
};

// 서버에 전송
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys",
  {
    method: "POST",
    body: JSON.stringify(surveyData),
    headers: {
      "Content-Type": "application/json",
    },
  }
);

// 에러 처리
if (!response.ok) {
  throw new Error(`설문 제출 실패: ${response.status}`);
}

const createdSurvey = await response.json();
console.log("설문 응답이 제출되었습니다:", createdSurvey);
// 화면에 제출 완료 메시지 표시하는 코드...
```

### 정리

POST 리퀘스트로 서버에 데이터를 생성하는 방법을 배웠습니다:

1. **method 옵션**: `{ method: 'POST' }`로 설정
2. **body 옵션**: 전송할 데이터를 `JSON.stringify()`로 변환하여 설정
3. **headers 옵션**: `'Content-Type': 'application/json'` 설정 (필수!)
4. **에러 확인**: `response.ok`로 HTTP 에러 확인
5. **응답 처리**: `await response.json()`으로 응답 데이터 가져오기

> 기억하세요:
> 
> - `fetch`는 HTTP 에러(400, 500 등)에도 rejected되지 않습니다.
> - 항상 `response.ok`를 확인하는 습관을 들이세요!

이제 여러분은 서버에서 데이터를 가져오는 것뿐만 아니라, 새로운 데이터를 생성할 수도 있습니다!

```jsx
export async function getSurveys() {
```

### 파라미터를 받는 함수

쿼리 파라미터를 유연하게 설정할 수 있도록 만들어봅시다:

```jsx
export async function getSurveys(params = {}) {
  const url = new URL(
    "https://learn.codeit.kr/api/color-surveys"
  );

  // params 객체의 모든 프로퍼티를 쿼리 파라미터로 추가
  Object.keys(params).forEach((key) => {
    // undefined나 null은 건너뛰기
    if (params[key] != null) {
      url.searchParams.set(key, params[key]);
    }
  });

  const response = await fetch(url);

  if (!response.ok) {
    throw new Error(
      `HTTP Error! Status: ${response.status}`
    );
  }

  const data = await response.json();
  return data;
}
```

> set vs append:
> 
> - `set()`: 같은 키가 있으면 덮어쓰기 (대부분 이 방식 사용)
> - `append()`: 같은 키를 중복으로 추가 (`?tag=a&tag=b`)

`Object.keys(params)`는 객체의 모든 키를 배열으로 반환합니다. 예를 들어:

```jsx
const params = {
  offset: 10,
  limit: 20,
  mbti: "ENFP",
};
Object.keys(params); // ['offset', 'limit', 'mbti']
```

`forEach`로 각 키를 순회하면서 쿼리 파라미터로 추가합니다.

### 사용 예시

```jsx

});
```

```jsx
  return data;
}

// 사용 예시
const data = await getProducts({
  category: "electronics",
  limit: 10,
  offset: 0,
});
```

- 정답 및 해설
    
    정답:

    해설:


### 08. 오류 처리하기 ⭐⭐

이번 시간에는 네트워크 리퀘스트를 보낼 때 발생할 수 있는 오류를 처리하는 방법을 배워보겠습니다.

### 네트워크 오류의 종류

네트워크 리퀘스트를 할 때는 다양한 오류가 발생할 수 있습니다:

- 인터넷 연결이 끊어진 경우
- 잘못된 URL로 리퀘스트를 보낸 경우
- 서버에서 오류가 발생한 경우 (500 에러)
- 잘못된 리퀘스트를 보낸 경우 (400 에러)

이런 오류들을 제대로 처리하지 않으면 프로그램이 예상치 못하게 중단될 수 있습니다.

### fetch 오류의 두 가지 유형

### 1. 리퀘스트 자체가 실패하는 경우

잘못된 URL이나 네트워크 연결 문제로 리퀘스트 자체가 실패하면, `fetch`가 반환하는 Promise는 **rejected 상태**가 됩니다.

```jsx
// api.js
export async function getSurveys() {
  const response = await fetch(
    "https://wrong-url-krrrr.com/api/color-surveys"
  );
  const data = await response.json();
  return data;
}

// main.js
import { getSurveys } from "./api.js";

const data = await getSurveys();
console.log(data);
// 에러 발생: fetch failed
```

이 경우 해당 비동기 작업에서 에러가 발생하고, 이 에러를 처리하지 않으면 콘솔에 에러가 출력되며 이후 코드는 실행되지 않습니다.

> 참고: 브라우저 전체 앱이 "강제 종료"되는 것은 아닙니다. 해당 비동기 흐름만 실패합니다.
> 

### 2. 리퀘스트는 성공했지만 오류 응답이 온 경우

서버가 400(Bad Request)이나 500(Internal Server Error) 같은 오류 상태 코드를 반환해도, `fetch`의 Promise는 **fulfilled 상태**가 됩니다.

```jsx
// 잘못된 파라미터로 리퀘스트
const response = await fetch(
  "https://learn.codeit.kr/api/color-surveys?invalid=xyz"
);
console.log(response.status); // 400
// 하지만 Promise는 fulfilled 상태!
```

이게 중요한 포인트입니다. **상태 코드가 400이나 500이어도 Promise는 fulfilled입니다!**

### try-catch로 오류 처리하기

> 참고: 이 강의의 코드는 ES 모듈 환경을 기준으로 합니다.
> 
> - 브라우저: `<script type="module">`
> - Node.js: `package.json`에 `"type": "module"` 설정 혹은 `.mjs`  확장자 사용
> 
> 모듈이 아닌 환경에서는 `async function main() { ... }` 안에서 `await`를 사용해야 합니다.
> 

### 기본 사용법

```jsx
import { getSurveys } from "./api.js";

try {
  const data = await getSurveys();
  console.log(data);
} catch (error) {
  console.log("오류가 발생했습니다:");
  console.log(error.message);
}
```

실제 웹사이트에서는 오류 메시지를 화면에 표시할 수 있습니다:

```jsx
try {
  const data = await getSurveys();
  displaySurveys(data);
} catch (error) {
  showErrorMessage(
    "설문 데이터를 불러오는데 실패했습니다. 다시 시도해주세요."
  );
}
```

### response.ok로 오류 확인하기

### 문제 상황

현재 코드의 문제점:

- 400이나 500 상태 코드가 와도 Promise는 fulfilled
- 오류인지 정상인지 구분하기 어려움
- 데이터를 파싱하려다 예상치 못한 오류 발생 가능

### 해결 방법

`response.ok` 프로퍼티를 사용하면 응답이 성공적인지 확인할 수 있습니다:

```jsx
export async function getSurveys(params = {}) {
  const url = new URL(
    "https://learn.codeit.kr/api/color-surveys"
  );

  Object.keys(params).forEach((key) =>
    url.searchParams.append(key, params[key])
  );

  const response = await fetch(url);

  // 응답이 성공적이지 않으면 오류 발생
  if (!response.ok) {
    throw new Error("데이터를 불러오는데 실패했습니다.");
  }

  const data = await response.json();
  return data;
}
```

**`response.ok`란?**

- 상태 코드가 200~299 범위면 `true`
- 그 외의 경우 `false`

### 왜 이렇게 하나요?

이렇게 처리하면:

1. **리퀘스트 실패** → Promise rejected → catch 블록 실행
2. **오류 상태 코드(400, 500 등)** → `throw new Error()` → catch 블록 실행

두 경우 모두 `catch` 블록에서 처리할 수 있어서 코드가 일관성 있고 깔끔해집니다!

> 추가 주의: response.ok를 확인해도, 응답 바디가 JSON 형식이 아니면(HTML 에러 페이지, 204 No Content 등) response.json()에서 SyntaxError가 발생할 수 있습니다. 이 경우에도 try-catch로 잡을 수 있습니다.
>

### 정리

fetch 오류 처리의 핵심:

1. **리퀘스트 실패**: fetch의 Promise가 rejected → catch에서 처리
2. **오류 상태 코드**: Promise는 fulfilled이지만 `response.ok`가 false
3. **해결 방법**: `response.ok`를 확인하고 오류 상태면 `throw new Error()`
4. **일관성**: 모든 오류가 catch 블록에서 처리되어 코드가 깔끔해짐

> 기억하세요:
> 
> - `fetch`는 네트워크 에러에서만 rejected됩니다
> - HTTP 400, 500은 fulfilled 상태이므로 `response.ok`로 확인 필수
> - `response.json()`도 JSON이 아닌 응답에서는 에러를 던집니다

- "POST 리퀘스트가 제대로 전송되었나?"

```html
    <button id="loadBtn">상품 불러오기</button>
    <div id="products"></div>

    <script type="module" src="main.js"></script>
  </body>
</html>
```

**api.js**

```jsx
export async function getProducts() {
  const response = await fetch(
    "https://jsonplaceholder.typicode.com/posts?_limit=5"
  );

  if (!response.ok) {
    throw new Error("데이터를 불러오는데 실패했습니다.");
  }

  const data = await response.json();
  return data;
}
```

**main.js**

```jsx
import { getProducts } from "./api.js";

const loadBtn = document.getElementById("loadBtn");
const productsDiv = document.getElementById("products");

loadBtn.addEventListener("click", async () => {
```

### Network 탭에서 리퀘스트 확인하기

### 1. 기본 정보 보기

Live Server로 페이지를 열고 Network 탭을 연 상태에서 "상품 불러오기" 버튼을 클릭하면:

**리퀘스트 목록에 표시되는 정보:**

- **Name**: 리퀘스트 URL (예: `posts?_limit=5`)
- **Status**: 응답 상태 코드 (예: `200`)
- **Type**: 리소스/요청의 종류 (예: `fetch`, `xhr`, `script`, `img` 등)
- **Size**: 응답 데이터 크기 (예: `2.3 KB`)
- **Time**: 리퀘스트 소요 시간 (예: `245ms`)

### 2. 상세 정보 보기

리퀘스트를 클릭하면 우측에 상세 정보가 나타납니다:

### **Headers 탭**

요청과 응답의 헤더 정보를 확인할 수 있습니다.

**General (일반 정보)**

- Request URL: `https://jsonplaceholder.typicode.com/posts?_limit=5`
- Request Method: `GET`
- Status Code: `200 OK`

**Response Headers (응답 헤더)**

- Content-Type: `application/json; charset=utf-8`
- Content-Length: `2345`

**Request Headers (요청 헤더)**

- Accept: `/*`
- User-Agent: 브라우저 정보

### **Payload 탭** (POST/PUT/DELETE 등)

POST 리퀘스트를 보낼 때 전송한 데이터를 확인할 수 있습니다.

예를 들어 설문 응답을 생성하는 경우:

```jsx
const surveyData = {
  mbti: "ENFP",
  colorCode: "#CA6952",
  password: "1234",
};

await fetch("https://learn.codeit.kr/api/color-surveys", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(surveyData),
});
```

Payload 탭에서 전송된 데이터를 확인:

```json
{
  "mbti": "ENFP",
  "colorCode": "#CA6952",
  "password": "1234"
}
```

### **Response 탭**

서버가 보낸 응답 데이터를 확인할 수 있습니다.

```json
[
  {
    "userId": 1,
    "id": 1,
    "title": "sunt aut facere repellat",
    "body": "quia et suscipit..."
  },
  {
    "userId": 1,
    "id": 2,
    "title": "qui est esse",
    "body": "est rerum tempore..."
  }
]
```

### Network 탭의 유용한 기능

### 1. 필터링

상단의 필터 버튼으로 특정 타입만 볼 수 있습니다:

- **All**: 모든 리퀘스트
- **Fetch/XHR**: JavaScript로 보낸 리퀘스트만
- **JS**: JavaScript 파일만
- **CSS**: CSS 파일만
- **Img**: 이미지 파일만

**fetch로 보낸 리퀘스트만 보려면**: `Fetch/XHR` 클릭

### 2. 리퀘스트 검색

상단의 검색창을 사용하면 URL에 특정 단어가 포함된 리퀘스트만 볼 수 있습니다.

예: `products`를 검색하면 `/products` URL이 포함된 리퀘스트만 표시

### 3. Preserve log (로그 보존)

페이지를 새로고침하면 Network 탭의 내용이 지워집니다. **Preserve log**를 체크하면 내용이 유지됩니다.

**언제 유용한가요?**

- 페이지 새로고침 시 자동으로 호출되는 API를 확인할 때
- 페이지 이동 시의 리퀘스트를 추적할 때

### 4. Throttling (속도 제한)

상단의 "No throttling" 드롭다운을 사용하면 느린 네트워크를 시뮬레이션할 수 있습니다.

**옵션:**

- **Fast 3G**: 빠른 3G 속도 (약 1.6 Mbps)
- **Slow 3G**: 느린 3G 속도 (약 400 Kbps)
- **Offline**: 오프라인 상태

**왜 사용하나요?**

- 느린 네트워크에서 웹사이트가 어떻게 작동하는지 테스트
- 로딩 인디케이터가 제대로 표시되는지 확인
- 사용자 경험 개선

### 실전 디버깅 예시

### 예시 1: 오류 확인하기

리퀘스트가 실패한 경우, Network 탭에서 빨간색으로 표시됩니다.

**증상**: 버튼을 눌렀는데 데이터가 안 나옴

**확인 방법**:

1. Network 탭 열기
2. 버튼 클릭
3. 빨간색 리퀘스트 찾기
4. 클릭해서 Status Code 확인
    - `404`: URL이 잘못됨 → URL 수정 필요
    - `500`: 서버 오류 → 서버 로그 확인 필요
    - `400`: 잘못된 요청 → 전송 데이터 확인 필요

### 예시 2: POST 데이터 확인하기

**증상**: 설문 응답을 제출했는데 제대로 저장이 안 됨

**확인 방법**:

1. Network 탭 열기
2. 설문 응답 작성하고 제출
3. POST 리퀘스트 찾기
4. **Payload 탭** 클릭해서 전송된 데이터 확인
5. 데이터 형식이 올바른지 확인

```jsx
// 잘못된 예: colorCode에 # 누락
{
  "mbti": "ENFP",
  "colorCode": "CA6952",  // ❌ # 누락
  "password": "1234"
}

// 올바른 예: 올바른 형식
{
  "mbti": "ENFP",
  "colorCode": "#CA6952",  // ✅ # 포함
  "password": "1234"
}
```

### 예시 3: 응답 데이터 확인하기

**증상**: 데이터가 화면에 제대로 표시되지 않음

**확인 방법**:

1. Network 탭 열기
2. 리퀘스트 실행
3. 해당 리퀘스트 클릭
4. **Response 탭** 클릭
5. 서버가 보낸 데이터 구조 확인
6. JavaScript 코드에서 올바른 프로퍼티를 사용하는지 확인

```jsx
// 서버 응답:
{
  "results": [
    { "id": 1, "name": "상품1" },
    { "id": 2, "name": "상품2" }
  ]
}

// 잘못된 코드
const data = await response.json();
console.log(data); // ❌ 배열이 아니라 객체!

// 올바른 코드
const data = await response.json();
console.log(data.results); // ✅ 배열에 접근
```

### 1. Copy as fetch

리퀘스트를 우클릭하고 **Copy > Copy as fetch**를 선택하면, 해당 리퀘스트를 재현하는 fetch 코드가 복사됩니다.

```jsx
// 자동 생성된 코드
fetch(
  "https://jsonplaceholder.typicode.com/posts?_limit=5",

    },
    method: "GET",
  }

```

### 2. Clear (지우기)

상단의 🚫 아이콘을 클릭하면 리퀘스트 목록이 지워집니다. 새로 테스트할 때 유용합니다.

### 3. 리퀘스트 재전송

리퀘스트를 우클릭하고 **Replay XHR**을 선택하면 같은 리퀘스트를 다시 보냅니다.

### 정리

Network 탭은 웹 개발자의 필수 도구입니다:

1. **리퀘스트 확인**: URL, 메소드, 상태 코드 확인
2. **헤더 확인**: 요청/응답 헤더 상세 정보
3. **전송 데이터 확인**: POST/PUT 시 전송한 데이터 (Payload 탭)
4. **응답 데이터 확인**: 서버가 보낸 데이터 (Response 탭)
5. **디버깅**: 오류 원인 파악, 데이터 형식 검증
6. **성능 분석**: 리퀘스트 소요 시간, 느린 네트워크 시뮬레이션

**활용 시나리오:**

- API가 제대로 호출되는지 확인
- 전송한 데이터가 올바른지 검증
- 서버 응답 데이터 구조 파악
- 오류 상태 코드 확인 및 디버깅
- 네트워크 성능 테스트

### 문제 1

Chrome DevTools의 Network 탭에서 **fetch로 보낸 리퀘스트만** 확인하고 싶습니다. 어떤 필터를 선택해야 할까요?

1. All
2. JS
3. Fetch/XHR
4. Doc
- 정답 및 해설
    
    정답:
    
    3번 (Fetch/XHR)
    
    해설:
    
    - Network 탭에서는 리퀘스트 유형별로 필터링할 수 있습니다.
    - `fetch`나 `XMLHttpRequest`로 전송한 API 요청은 **Fetch/XHR** 범주에 속합니다.
    - API 호출 디버깅 시 가장 많이 사용하는 필터입니다.

### 문제 2

다음 코드를 실행했을 때, Network 탭의 **어느 탭**에서 전송된 데이터를 확인할 수 있을까요?

```jsx
const reviewData = {
  productId: 42,
  rating: 5,
  comment: "좋은 상품입니다!",
};

await fetch("https://api.example.com/reviews", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(reviewData),
});
```

**A**: Headers 탭

**B**: Response 탭

**C**: Payload 탭

**D**: Timing 탭

- 정답 및 해설
    
    정답:
    
    C (Payload 탭)
    
    해설:
    
    - POST 요청 시 **보낸 데이터(body)**는 Payload 탭에서 확인할 수 있습니다.
    - JSON으로 보낸 데이터가 그대로 표시됩니다.
    - Headers 탭에는 내용이 아니라 헤더 정보만 있으며, Response 탭에는 서버 응답이 표시됩니다.

### 문제 3

다음 상황에서 Network 탭을 어떻게 활용하면 좋을까요?

**상황**: 서버에서 상품 목록을 받아 화면에 표시하는데, 아무것도 출력되지 않습니다.

```jsx
async function displayProducts() {
  const response = await fetch("https://api.example.com/products");
  const data = await response.json();

  data.forEach((product) => {
    console.log(product.name);
  });
}

displayProducts();
```

**A**: Headers 탭에서 Content-Type 확인

**B**: Response 탭에서 서버가 보낸 데이터 구조 확인

**C**: Payload 탭에서 전송한 데이터 확인

**D**: Timing 탭에서 소요 시간 확인

- 정답 및 해설
    
    정답:
    
    B (Response 탭에서 서버가 보낸 데이터 구조 확인)
    
    해설:
    
    - 화면에 아무것도 표시되지 않는 이유 중 하나는 **응답 데이터 구조를 오해한 경우**입니다.
    - Response 탭을 열어 실제 받은 JSON 형식이 배열인지, 객체인지 직접 확인해야 합니다.
    
    예:
    
    ```jsx
    // 실제 응답
    {
      "results": [
        { "id": 1, "name": "상품1" },
        { "id": 2, "name": "상품2" }
      ]
    }
    
    // 잘못된 코드
    data.forEach(...) // data는 배열이 아님 → 오류 발생
    ```
    
    정정 코드:
    
    ```jsx
    data.results.forEach((product) => {
      console.log(product.name);
    });
    ```
    
    왜 다른 보기들은 틀렸나?
    
    - A: Content-Type은 JSON 여부만 파악할 수 있으나, 구조 문제 해결엔 직접적인 도움이 안 됨
    - C: GET 요청은 payload가 없음
    - D: Timing 정보는 속도 문제 분석용이지, 데이터 표시 문제 해결과 무관
    
    Response 탭은 API 디버깅에서 가장 중요한 도구입니다.
