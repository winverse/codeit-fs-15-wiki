# Day 7. 객체 속성 서술자와 동결

### 오늘의 목표

JavaScript 객체의 속성은 단순한 키-값 쌍처럼 보이지만, 실제로는 `writable`,
`enumerable`, `configurable` 같은 속성 서술자를 가집니다. 오늘은
`Object.defineProperty`, `Object.getOwnPropertyDescriptor`, `Object.freeze`를 복습하고,
런타임 불변성과 TypeScript의 `readonly`가 어떤 점에서 같고 다른지 확인합니다.

### 레퍼런스 연결

- 객체 챕터 복습
- 핵심 복습 포인트: property descriptor, enumerable, writable, `Object.freeze`
- 다시 볼 질문: `readonly`와 `Object.freeze`는 같은 수준의 안전성을 줄까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
const user = { name: "Ada" };

Object.defineProperty(user, "role", { value: "admin" });

const roleDescriptor = Object.getOwnPropertyDescriptor(user, "role");

console.log(user.role);
console.log(Object.keys(user).includes("role"));
console.log(JSON.stringify(user));
console.log(roleDescriptor.enumerable);
console.log(roleDescriptor.writable);
console.log(roleDescriptor.configurable);
```

- 정답과 해설
    
    ### 정답
    
    ```
    admin
    false
    {"name":"Ada"}
    false
    false
    false
    ```
    
    ### 해설
    
    `Object.defineProperty(user, "role", { value: "admin" })`는 `role` 속성을 추가합니다.
    따라서 `user.role`은 `"admin"`입니다.
    
    하지만 `Object.defineProperty`로 새 속성을 만들 때 `enumerable`, `writable`,
    `configurable`을 생략하면 기본값은 모두 `false`입니다.
    
    `Object.keys(user)`는 enumerable한 own property만 반환합니다. `role`은 enumerable이
    아니므로 `Object.keys(user).includes("role")`은 `false`입니다.
    
    `JSON.stringify(user)`도 enumerable한 own property를 기준으로 직렬화합니다. 그래서
    `role`은 결과에 포함되지 않고 `{"name":"Ada"}`만 출력됩니다.
    
    `Object.getOwnPropertyDescriptor(user, "role")`로 확인하면 `enumerable`, `writable`,
    `configurable`이 모두 `false`입니다.
    
    객체에 속성이 “있다”는 사실과 그 속성이 순회되거나 수정되거나 삭제될 수 있다는 사실은
    서로 다릅니다. 속성 서술자를 모르면 값이 존재하는데도 `Object.keys`나 JSON 결과에 안
    나오는 이유를 놓치기 쉽습니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `writable`과 `configurable`의 차이를
설명하세요.

```jsx
const item = { count: 1 };

Object.defineProperty(item, "count", {
  writable: false,
  configurable: false,
});

console.log(Reflect.set(item, "count", 2));
console.log(item.count);
console.log(Reflect.deleteProperty(item, "count"));
console.log("count" in item);
console.log(Object.getOwnPropertyDescriptor(item, "count").enumerable);
```

- 정답과 해설
    
    ### 정답
    
    ```
    false
    1
    false
    true
    true
    ```
    
    ### 해설
    
    처음 `item`은 객체 리터럴로 만들어졌습니다. 객체 리터럴로 만든 일반 속성은 기본적으로
    `writable: true`, `enumerable: true`, `configurable: true`입니다.
    
    그 다음 `Object.defineProperty`로 `count` 속성의 `writable`과 `configurable`을
    `false`로 바꿉니다. `enumerable`은 명시하지 않았지만, 이미 존재하던 속성을 수정하는
    경우 생략된 서술자는 기존 값을 유지합니다. 따라서 `enumerable`은 계속 `true`입니다.
    
    `Reflect.set(item, "count", 2)`는 값을 바꾸려고 시도합니다. 하지만 `writable`이
    `false`이므로 변경에 실패하고 `false`를 반환합니다. 그래서 `item.count`는 여전히
    `1`입니다.
    
    `Reflect.deleteProperty(item, "count")`는 속성을 삭제하려고 시도합니다. 하지만
    `configurable`이 `false`이므로 삭제에 실패하고 `false`를 반환합니다. 따라서
    `"count" in item`은 `true`입니다.
    
    `writable`은 값 변경 가능 여부와 관련되고, `configurable`은 삭제 가능 여부와 속성
    서술자 재구성 가능 여부와 관련됩니다. 둘은 비슷해 보이지만 서로 다른 축의 제약입니다.
    
    이 문제에서는 `Reflect.set`과 `Reflect.deleteProperty`를 사용했습니다. 직접 대입이나
    `delete` 연산자는 strict mode 여부에 따라 예외 또는 조용한 실패처럼 보일 수 있습니다.
    `Reflect` 계열 API는 성공 여부를 Boolean으로 반환하므로 학습과 검증에 더 명확합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Object.freeze`가 얕은 동결이라는 말의
의미를 설명하세요.

```jsx
const profile = Object.freeze({
  name: "Ada",
  tags: ["js"],
});

console.log(Object.isFrozen(profile));
console.log(Object.isFrozen(profile.tags));

profile.tags.push("ts");

console.log(profile.tags.join(","));
console.log(Reflect.set(profile, "name", "Lin"));
console.log(profile.name);
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    false
    js,ts
    false
    Ada
    ```
    
    ### 해설
    
    `Object.freeze(profile)`은 `profile` 객체 자체를 동결합니다. 그래서
    `Object.isFrozen(profile)`은 `true`입니다.
    
    하지만 `Object.freeze`는 기본적으로 얕은 동결입니다. 객체의 직접 속성은 바꾸지 못하게
    하지만, 속성 값이 또 다른 객체라면 그 내부 객체까지 자동으로 동결하지는 않습니다.
    `profile.tags`는 배열이고 별도로 동결하지 않았으므로 `Object.isFrozen(profile.tags)`는
    `false`입니다.
    
    따라서 `profile.tags.push("ts")`는 성공합니다. `profile`의 `tags` 속성이 다른 배열을
    가리키도록 바꾸는 것은 막혀 있지만, 이미 가리키고 있던 배열 내부를 변경하는 것은
    가능하기 때문입니다. 결과적으로 `profile.tags.join(",")`은 `"js,ts"`입니다.
    
    `Reflect.set(profile, "name", "Lin")`은 동결된 객체의 직접 속성인 `name`을 바꾸려고
    합니다. 실패하므로 `false`를 반환하고, `profile.name`은 `"Ada"`로 유지됩니다.
    
    이 문제의 핵심은 “객체를 동결했다”가 “중첩된 모든 데이터가 절대 바뀌지 않는다”는 뜻은
    아니라는 점입니다. 깊은 불변성이 필요하면 중첩 객체까지 동결하거나, 애초에 변경 대신
    새 객체를 만드는 방식으로 설계해야 합니다.
    

### 문제 4. 심화

다음 코드는 상태 객체를 동결해서 안전하게 만들려고 합니다. 실행 결과를 예측하고, 이
코드의 문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
const state = Object.freeze({
  user: { name: "Ada" },
  count: 0,
});

state.user.name = "Lin";

console.log(state.user.name);
console.log(Object.isFrozen(state));
console.log(Object.isFrozen(state.user));

function renameUser(nextName) {
  return { ...state, user: { ...state.user, name: nextName } };
}

const next = renameUser("Ken");

console.log(state.user.name);
console.log(next.user.name);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Lin
    true
    false
    Lin
    Ken
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type DeepReadonly<T> = {
      readonly [K in keyof T]: T[K] extends (...args: unknown[]) => unknown
        ? T[K]
        : T[K] extends object
          ? DeepReadonly<T[K]>
          : T[K];
    };
    
    type State = {
      user: { name: string };
      count: number;
    };
    
    const state: DeepReadonly<State> = Object.freeze({
      user: Object.freeze({ name: "Ada" }),
      count: 0,
    });
    
    function renameUser(current: DeepReadonly<State>, name: string): State {
      return {
        ...current,
        user: {
          ...current.user,
          name,
        },
      };
    }
    
    const next = renameUser(state, "Lin");
    
    console.log(state.user.name);
    console.log(next.user.name);
    ```
    
    ### 해설
    
    ### JS 관점
    
    `Object.freeze`는 `state` 객체 자체를 동결합니다. 그래서 `Object.isFrozen(state)`는
    `true`입니다. `state.count`를 바꾸거나 `state.user`에 새 객체를 대입하는 작업은 실패합니다.
    
    하지만 `state.user` 객체 자체는 동결되지 않았습니다. `Object.freeze`가 중첩 객체까지
    자동으로 재귀 동결하지 않기 때문입니다. 그래서 `Object.isFrozen(state.user)`는
    `false`입니다.
    
    `state.user.name = "Lin"`은 중첩 객체의 속성을 변경하는 코드입니다. `state.user`는
    동결되지 않았으므로 변경이 성공하고, 첫 번째 출력은 `"Lin"`입니다.
    
    그 뒤 `renameUser("Ken")`은 전개 구문으로 새 객체를 만듭니다. 이 함수는 기존 `state`를
    직접 바꾸는 대신, `user` 객체도 새로 복사해서 `name`만 `"Ken"`으로 바꿉니다. 따라서
    기존 `state.user.name`은 이미 바뀌어 있던 `"Lin"`이고, `next.user.name`은 `"Ken"`입니다.
    
    ### TS 관점
    
    TypeScript의 `readonly`는 컴파일 타임에서 대입을 막는 장치입니다. 런타임 객체를 실제로
    동결하지는 않습니다. 반대로 `Object.freeze`는 런타임에서 객체의 직접 속성을 막지만,
    타입 모델이 깊은 불변성을 자동으로 표현해 주는 것은 아닙니다.
    
    개선 예시에서는 `DeepReadonly<State>` 타입으로 중첩 속성까지 읽기 전용으로 표현하고,
    런타임에서도 `state`와 `state.user`를 각각 `Object.freeze`로 동결했습니다. 이렇게 하면
    타입 단계와 런타임 단계가 같은 방향의 제약을 갖습니다.
    
    다만 예시의 `DeepReadonly`는 학습용 단순화 버전입니다. 실제 프로젝트에서는 배열, Map,
    Set, Date, 클래스 인스턴스, 함수 속성까지 고려해야 하므로 검증된 유틸리티 타입이나
    명확한 상태 관리 규칙을 사용하는 편이 좋습니다.
    
    `renameUser`는 기존 값을 직접 변경하지 않고 새 `State`를 반환합니다. 이 방식은 React
    상태 업데이트나 Redux 스타일의 불변 업데이트와도 연결됩니다.
    
    ### 실무 관점
    
    상태 객체를 안전하게 다루려면 “수정 금지”를 어디에서 보장할지 정해야 합니다.
    
    1. 타입 단계에서는 `readonly` 또는 `DeepReadonly`로 직접 대입을 막는다.
    2. 런타임 단계에서는 필요한 범위만 `Object.freeze`로 동결한다.
    3. 중첩 객체가 있다면 얕은 동결만으로 충분한지 확인한다.
    4. 상태 변경은 직접 수정이 아니라 새 객체 반환으로 표현한다.
    
    중요한 것은 `readonly`와 `Object.freeze`를 같은 것으로 착각하지 않는 것입니다.
    `readonly`는 컴파일러가 보는 약속이고, `Object.freeze`는 런타임 객체에 적용되는 제약입니다.
    둘을 함께 이해해야 “타입은 안전해 보이는데 실제 값은 바뀌는” 상황을 줄일 수 있습니다.
