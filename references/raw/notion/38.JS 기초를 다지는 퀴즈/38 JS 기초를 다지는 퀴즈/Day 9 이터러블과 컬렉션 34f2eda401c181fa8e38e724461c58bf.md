# Day 9. 이터러블과 컬렉션

### 오늘의 목표

JavaScript의 배열, 문자열, `Map`, `Set`은 모두 반복과 컬렉션 처리에서 자주 사용됩니다.
오늘은 이터러블의 조건, 일반 객체와 `Map`의 키 처리 차이, `Set`의 동등성 비교 방식을
복습하고, TypeScript에서 `Map<K, V>`와 `ReadonlyMap<K, V>`를 어떻게 설계해야 하는지
실무 캐시 예제로 연결합니다.

### 레퍼런스 연결

- 객체와 컬렉션 관련 내용 복습
- 핵심 복습 포인트: 이터러블, iterator, `Map`, `Set`, `WeakMap`
- 다시 볼 질문: 일반 객체와 `Map` 중 무엇을 선택해야 하는 기준은 무엇일까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
const values = [10, 20];
const text = "JS";

console.log(typeof values[Symbol.iterator]);
console.log([...text].join("-"));

try {
  console.log([...{ 0: "a", length: 1 }]);
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    function
    J-S
    TypeError
    ```
    
    ### 해설
    
    이터러블은 `Symbol.iterator` 메서드를 가진 값입니다. 배열은 이터러블이므로
    `values[Symbol.iterator]`는 함수입니다. 그래서 첫 번째 출력은 `"function"`입니다.
    
    문자열도 이터러블입니다. 전개 구문 `[...text]`는 문자열 `"JS"`를 `"J"`와 `"S"`로
    순회해 배열로 만듭니다. `join("-")`을 호출하면 `"J-S"`가 됩니다.
    
    마지막 객체 `{ 0: "a", length: 1 }`은 배열처럼 보이지만 이터러블은 아닙니다. 숫자 키와
    `length` 속성이 있다고 해서 자동으로 이터러블이 되는 것은 아닙니다. `Symbol.iterator`
    메서드가 없으므로 전개 구문을 적용하면 `TypeError`가 발생합니다.
    
    배열처럼 생긴 값과 이터러블은 다릅니다. DOM의 일부 컬렉션이나 함수의 `arguments`처럼
    환경과 타입에 따라 배열처럼 보이는 값이 실제 배열인지, 이터러블인지, 단순 객체인지
    확인해야 합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 일반 객체와 `Map`이 객체 키를 다루는 방식의
차이를 설명하세요.

```jsx
const a = { id: 1 };
const b = { id: 1 };

const map = new Map();
map.set(a, "A");
map.set(b, "B");

console.log(map.size);
console.log(map.get(a));
console.log(map.get({ id: 1 }));

const obj = {};
obj[a] = "A";
obj[b] = "B";

console.log(Object.keys(obj).length);
console.log(obj[a]);
```

- 정답과 해설
    
    ### 정답
    
    ```
    2
    A
    undefined
    1
    B
    ```
    
    ### 해설
    
    `a`와 `b`는 내용이 같아 보이지만 서로 다른 객체입니다. 객체 비교에서 중요한 것은 내용이
    아니라 참조입니다.
    
    `Map`은 객체를 키로 그대로 사용할 수 있습니다. 그래서 `map.set(a, "A")`와
    `map.set(b, "B")`는 서로 다른 키를 저장합니다. `map.size`는 `2`입니다.
    
    `map.get(a)`는 같은 객체 참조 `a`로 조회하므로 `"A"`를 반환합니다. 하지만
    `map.get({ id: 1 })`은 `undefined`입니다. `{ id: 1 }`은 `a`와 내용이 같아 보여도 새로
    만든 다른 객체이기 때문입니다.
    
    일반 객체 `obj`의 속성 키는 문자열 또는 Symbol입니다. 객체를 대괄호 키로 넣으면 문자열
    키로 변환됩니다. `obj[a]`와 `obj[b]`는 둘 다 기본적으로 `"[object Object]"`라는 같은
    문자열 키로 변환됩니다.
    
    그래서 `Object.keys(obj).length`는 `1`이고, 두 번째 대입인 `obj[b] = "B"`가 첫 번째 값을
    덮어씁니다. 마지막 `obj[a]`도 `"B"`를 반환합니다.
    
    객체 자체를 키로 사용해야 한다면 일반 객체보다 `Map`이 더 적합합니다. 반대로 키가
    문자열로 정해져 있고 JSON 직렬화가 중요하다면 일반 객체나 `Record`가 더 단순할 수
    있습니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Set`이 중복을 판단하는 방식을 설명하세요.

```jsx
const set = new Set([NaN, NaN, +0, -0, "0"]);

console.log(set.size);
console.log(set.has(NaN));
console.log(set.has(-0));
console.log([...set].map((value) => `${typeof value}:${String(value)}`).join(","));
```

- 정답과 해설
    
    ### 정답
    
    ```
    3
    true
    true
    number:NaN,number:0,string:0
    ```
    
    ### 해설
    
    `Set`은 중복 없는 값의 컬렉션입니다. 중복 판단에는 `SameValueZero` 비교 방식을
    사용합니다.
    
    `SameValueZero`에서는 `NaN`과 `NaN`을 같은 값으로 봅니다. 그래서 `[NaN, NaN]`을 넣어도
    하나만 남고, `set.has(NaN)`은 `true`입니다.
    
    `+0`과 `-0`도 `Set`에서는 같은 값으로 취급됩니다. 따라서 `+0`과 `-0`을 모두 넣어도 숫자
    0은 하나만 저장됩니다. `set.has(-0)`은 `true`입니다.
    
    문자열 `"0"`은 숫자 `0`과 다릅니다. `Set`은 `==`처럼 타입 변환을 하지 않습니다. 그래서
    숫자 `0`과 문자열 `"0"`은 서로 다른 값으로 저장됩니다.
    
    최종적으로 남는 값은 `NaN`, 숫자 `0`, 문자열 `"0"` 세 개입니다. 마지막 줄은 각 값의
    타입과 문자열 표현을 함께 출력하므로 `number:NaN,number:0,string:0`이 됩니다.
    
    `Set`은 배열 중복 제거에 편리하지만, 객체의 내용이 같은지를 자동으로 비교하지는 않습니다.
    객체는 참조가 같아야 같은 값으로 취급됩니다.
    

### 문제 4. 심화

다음 코드는 사용자 객체를 키로 이름 캐시를 만들려고 합니다. 실행 결과를 예측하고, 이
코드의 문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
const user1 = { id: 1, name: "Ada" };
const user2 = { id: 2, name: "Lin" };

const cache = {};
cache[user1] = user1.name;
cache[user2] = user2.name;

console.log(JSON.stringify(Object.keys(cache)));
console.log(cache[user1]);
console.log(cache[user2]);

const safeCache = new Map();
safeCache.set(user1, user1.name);
safeCache.set(user2, user2.name);

console.log(safeCache.get(user1));
console.log(safeCache.get({ id: 1, name: "Ada" }));
```

- 정답과 해설
    
    ### 정답
    
    ```
    ["[object Object]"]
    Lin
    Lin
    Ada
    undefined
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = {
      id: number;
      name: string;
    };
    
    const user1: User = { id: 1, name: "Ada" };
    const user2: User = { id: 2, name: "Lin" };
    
    const nameByUser = new Map<User, string>();
    nameByUser.set(user1, user1.name);
    nameByUser.set(user2, user2.name);
    
    function readByUser(cache: ReadonlyMap<User, string>, user: User): string | null {
      return cache.get(user) ?? null;
    }
    
    const nameById = new Map<number, string>();
    nameById.set(user1.id, user1.name);
    nameById.set(user2.id, user2.name);
    
    function readById(cache: ReadonlyMap<number, string>, id: number): string | null {
      return cache.get(id) ?? null;
    }
    
    console.log(readByUser(nameByUser, user1));
    console.log(readByUser(nameByUser, { id: 1, name: "Ada" }));
    console.log(readById(nameById, 1));
    ```
    
    ### 해설
    
    ### JS 관점
    
    일반 객체 `cache`에 `cache[user1] = user1.name`처럼 객체를 키로 넣으면, 객체 키가
    문자열로 변환됩니다. 기본 객체의 문자열 표현은 보통 `"[object Object]"`입니다.
    
    `user1`과 `user2` 모두 같은 문자열 키로 변환되므로 `cache`에는 실제로 키 하나만 남습니다.
    `Object.keys(cache)`를 JSON 문자열로 출력하면 `["[object Object]"]`입니다.
    
    두 번째 대입인 `cache[user2] = user2.name`이 첫 번째 값을 덮어썼기 때문에 `cache[user1]`과
    `cache[user2]`는 모두 `"Lin"`입니다.
    
    `Map`은 객체 키를 문자열로 바꾸지 않습니다. `safeCache.set(user1, user1.name)`은 `user1`
    객체 참조를 키로 저장합니다. 그래서 `safeCache.get(user1)`은 `"Ada"`입니다.
    
    하지만 `safeCache.get({ id: 1, name: "Ada" })`는 `undefined`입니다. 내용이 같아도 새 객체
    참조이기 때문입니다. `Map`의 객체 키는 “같은 모양”이 아니라 “같은 객체”를 기준으로
    조회됩니다.
    
    ### TS 관점
    
    TypeScript에서 일반 객체를 캐시로 표현하려고 `Record<string, string>`을 쓰면 키가 문자열임을
    명확히 할 수 있습니다. 하지만 객체 자체를 키로 쓰고 싶다면 `Record`는 맞지 않습니다.
    객체 키가 문자열로 강제 변환되는 JavaScript 동작을 피할 수 없기 때문입니다.
    
    개선 예시에서는 `Map<User, string>`을 사용합니다. 이 타입은 “사용자 객체를 키로 하고,
    문자열 이름을 값으로 가진다”는 의도를 표현합니다.
    
    또한 함수 인자에는 `ReadonlyMap<User, string>`을 사용했습니다. 읽기만 하는 함수가 캐시를
    변경하지 않도록 타입 수준에서 의도를 드러내기 위해서입니다.
    
    다만 `Map<User, string>`은 객체 참조 기준입니다. `{ id: 1, name: "Ada" }`처럼 새로 만든
    객체로는 기존 값을 찾을 수 없습니다. 논리적으로 같은 사용자 ID를 기준으로 조회해야 한다면
    `Map<number, string>`처럼 ID를 키로 쓰는 편이 더 적합합니다. 예시의 `readById`가 그 경우입니다.
    
    ### 실무 관점
    
    컬렉션 선택은 데이터의 의미에 맞춰야 합니다.
    
    1. 키가 문자열이고 JSON으로 쉽게 저장해야 한다면 객체나 `Record<string, V>`를 고려한다.
    2. 키가 객체 참조라면 `Map<object, V>` 계열을 고려한다.
    3. 논리적 식별자가 ID라면 객체 전체보다 `Map<ID, V>`가 더 안정적이다.
    4. 중복 제거가 목적이면 `Set`을 쓰되, 객체 내용 비교는 직접 설계한다.
    
    중요한 것은 “어떤 값을 같은 키로 볼 것인가”입니다. 참조가 같아야 같은 키인지, ID가 같으면
    같은 키인지, 문자열 표현이 같으면 같은 키인지에 따라 선택해야 하는 컬렉션이 달라집니다.
