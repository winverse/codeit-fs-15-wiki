# Day 1. 원시 타입과 런타임 타입

### 오늘의 목표

JavaScript의 타입은 TypeScript의 정적 타입과 다르게 런타임 값 기준으로 결정됩니다.
오늘은 원시 타입, 객체 타입, `typeof`의 동작, `null`의 예외, 래퍼 객체, `Symbol`,
`BigInt`를 복습합니다.

### 레퍼런스 연결

- 타입 챕터 복습
- 핵심 복습 포인트: 원시 타입, 객체 타입, `typeof`, `null`의 예외
- 다시 볼 질문: TypeScript의 타입과 JavaScript 런타임 타입은 같은가?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
console.log(typeof 42);
console.log(typeof "42");
console.log(typeof true);
console.log(typeof undefined);
console.log(typeof null);
console.log(Array.isArray([]));
```

- 정답과 해설
    
    ### 정답
    
    ```
    number
    string
    boolean
    undefined
    object
    true
    ```
    
    ### 해설
    
    `typeof` 연산자는 값의 런타임 타입을 문자열로 반환합니다.
    
    `42`는 숫자 값이므로 `"number"`가 출력됩니다. `"42"`는 문자열 값이므로
    `"string"`이 출력됩니다. `true`는 Boolean 값이므로 `"boolean"`이 출력됩니다.
    `undefined`는 값이 할당되지 않았음을 나타내는 원시값이며, `typeof undefined`는
    `"undefined"`입니다.
    
    주의할 부분은 `null`입니다. `null`은 원시 타입의 값이지만 `typeof null`은
    `"object"`를 반환합니다. 이는 JavaScript 초기에 생긴 오래된 동작이 그대로 유지된
    것입니다. 따라서 `null`인지 확인할 때는 `typeof`만 쓰면 안 되고, 보통
    `value === null`처럼 직접 비교해야 합니다.
    
    배열도 객체의 한 종류이기 때문에 `typeof []`는 `"object"`입니다. 그래서 배열인지
    확인할 때는 `Array.isArray([])`를 사용합니다.
    
    헷갈리기 쉬운 오답은 `typeof null`을 `"null"`이라고 쓰는 것입니다. JavaScript에는
    `Null` 타입이 있지만, `typeof` 결과 문자열에는 `"null"`이 없습니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 예측하고, `==`와 `===`의 결과가 다른 이유를 설명하세요.

```jsx
const primitive = "hello";
const wrapped = new String("hello");

console.log(typeof primitive);
console.log(typeof wrapped);
console.log(primitive == wrapped);
console.log(primitive === wrapped);
console.log(wrapped instanceof String);
```

- 정답과 해설
    
    ### 정답
    
    ```
    string
    object
    true
    false
    true
    ```
    
    ### 해설
    
    `primitive`는 문자열 리터럴로 만든 원시값입니다. 따라서 `typeof primitive`는
    `"string"`입니다.
    
    반면 `wrapped`는 `new String("hello")`로 만든 객체입니다. 문자열처럼 보이지만
    실제로는 `String` 생성자 함수로 만들어진 객체이므로 `typeof wrapped`는
    `"object"`입니다. 또한 `wrapped instanceof String`은 `wrapped`가 `String`
    생성자의 인스턴스인지 확인하므로 `true`입니다.
    
    `primitive == wrapped`가 `true`인 이유는 동등 연산자 `==`가 비교 과정에서 암시적
    타입 변환을 수행하기 때문입니다. 객체인 `wrapped`는 비교 과정에서 원시값으로
    변환되고, 그 결과 `"hello"`와 `"hello"`를 비교하게 되어 `true`가 됩니다.
    
    하지만 `primitive === wrapped`는 `false`입니다. 일치 연산자 `===`는 타입 변환을
    하지 않고 값과 타입을 함께 비교합니다. 왼쪽은 원시 문자열이고 오른쪽은 객체이므로
    서로 다릅니다.
    
    실무에서는 `new String`, `new Number`, `new Boolean`처럼 원시값을 래퍼 객체로
    직접 감싸는 코드를 피하는 것이 좋습니다. 보기에는 문자열처럼 보여도 객체이기 때문에
    비교, 조건문, 직렬화 과정에서 예상과 다른 결과가 나올 수 있습니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Symbol.for`와 `BigInt`에서 주의할 점을
설명하세요.

```jsx
const localA = Symbol("id");
const localB = Symbol("id");
const sharedA = Symbol.for("id");
const sharedB = Symbol.for("id");

console.log(localA === localB);
console.log(sharedA === sharedB);
console.log(typeof 10n);

try {
  console.log(10n + 1);
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    false
    true
    bigint
    TypeError
    ```
    
    ### 해설
    
    `Symbol("id")`는 호출할 때마다 새로운 Symbol 값을 만듭니다. 설명 문자열이 같아도
    값 자체는 고유합니다. 그래서 `localA === localB`는 `false`입니다.
    
    `Symbol.for("id")`는 전역 Symbol 레지스트리를 사용합니다. 같은 키 `"id"`로
    호출하면 같은 Symbol 값을 다시 가져옵니다. 그래서 `sharedA === sharedB`는
    `true`입니다.
    
    `10n`은 BigInt 리터럴입니다. `typeof 10n`은 `"bigint"`입니다. BigInt는
    `Number.MAX_SAFE_INTEGER`를 넘어서는 큰 정수를 안전하게 다룰 때 유용합니다.
    
    주의할 점은 BigInt와 Number를 직접 섞어 연산할 수 없다는 것입니다. `10n + 1`에서
    `10n`은 BigInt이고 `1`은 Number입니다. JavaScript는 이 둘을 자동으로 맞춰서
    더하지 않고 `TypeError`를 발생시킵니다.
    
    실무에서 ID, 주문 번호, 큰 카운터처럼 안전한 정수 범위를 넘을 수 있는 값을 다룰 때
    BigInt를 고려할 수 있습니다. 다만 JSON 직렬화, 수학 함수, 기존 API와의 호환성에서
    제약이 있으므로 Number와 섞이지 않도록 경계를 명확히 해야 합니다.
    

### 문제 4. 심화

다음 함수는 배열과 문자열의 길이를 반환하려고 작성되었습니다. 실행 결과를 예측하고,
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function getCount(value) {
  if (typeof value === "object") {
    return value.length;
  }

  return value.length;
}

console.log(getCount(["JS", "TS"]));
console.log(getCount("type"));

try {
  console.log(getCount(null));
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    2
    4
    TypeError
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    function getCount(value: string | string[] | null): number {
      if (value === null) {
        return 0;
      }
    
      return value.length;
    }
    ```
    
    또는 배열과 문자열 처리를 더 명시적으로 나누면 다음처럼 작성할 수 있습니다.
    
    ```tsx
    function getCount(value: string | string[] | null): number {
      if (typeof value === "string") {
        return value.length;
      }
    
      if (Array.isArray(value)) {
        return value.length;
      }
    
      return 0;
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    첫 번째 호출인 `getCount(["JS", "TS"])`에서 인자는 배열입니다. 배열은 객체이므로
    `typeof value === "object"` 조건이 `true`입니다. 배열에는 `length` 속성이 있으므로
    `2`가 반환됩니다.
    
    두 번째 호출인 `getCount("type")`에서 인자는 문자열입니다. 문자열의 `typeof` 결과는
    `"string"`이므로 `if` 블록에 들어가지 않습니다. 이후 `return value.length`가
    실행되고, 문자열 `"type"`의 길이인 `4`가 반환됩니다.
    
    세 번째 호출인 `getCount(null)`이 문제입니다. `typeof null`은 `"object"`입니다.
    따라서 `if (typeof value === "object")` 조건을 통과합니다. 하지만 `null`은 객체처럼
    속성을 읽을 수 있는 값이 아닙니다. 그래서 `value.length`를 읽는 순간 `TypeError`가
    발생합니다.
    
    핵심은 `typeof value === "object"`가 `null`을 제외해주지 않는다는 점입니다.
    
    ### TS 관점
    
    TypeScript에서 이 함수를 다음처럼 선언했다고 생각해볼 수 있습니다.
    
    ```tsx
    function getCount(value: string | string[] | null): number {
      if (typeof value === "object") {
        return value.length;
      }
    
      return value.length;
    }
    ```
    
    `strictNullChecks`가 켜져 있다면 TypeScript는 `typeof value === "object"` 분기 안의
    `value`가 `string[] | null`일 수 있다고 판단합니다. 즉, TypeScript는 `null` 가능성을
    제거하지 않습니다. 이때 `value.length`에 대해 `value`가 `null`일 수 있다는 오류를
    낼 수 있습니다.
    
    이것이 TypeScript가 도움이 되는 지점입니다. JavaScript에서는 실행해야만 `TypeError`를
    만나지만, TypeScript는 타입 분석 단계에서 위험한 접근을 알려줄 수 있습니다.
    
    다만 TypeScript가 모든 런타임 문제를 자동으로 해결해주는 것은 아닙니다. 타입을
    `any`로 두거나 `strictNullChecks`를 끄면 이런 위험을 놓칠 수 있습니다. 결국
    JavaScript의 런타임 규칙을 이해한 상태에서 타입을 설계해야 합니다.
    
    ### 실무 관점
    
    실무 코드에서는 API 응답, DOM 조회 결과, 검색 결과처럼 값이 없을 수 있는 데이터를
    자주 다룹니다. 이때 `typeof value === "object"`만 믿고 속성에 접근하면 `null`에서
    런타임 오류가 발생할 수 있습니다.
    
    안전한 코드는 먼저 `null`과 `undefined` 같은 값 없음 상태를 처리하고, 그 다음 실제
    자료형을 좁혀야 합니다. 배열인지 확인할 때는 `typeof`보다 `Array.isArray`가 더
    명확합니다.
