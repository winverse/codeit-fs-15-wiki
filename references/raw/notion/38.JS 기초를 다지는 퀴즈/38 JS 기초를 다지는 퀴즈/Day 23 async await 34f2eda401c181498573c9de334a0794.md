# Day 23. async/await

### 오늘의 목표

`async/await`는 Promise 체인을 더 동기 코드처럼 읽게 해 주지만, 실제 실행은 여전히 Promise와
microtask 규칙을 따릅니다. 오늘은 async 함수의 반환값, `await` 이후 실행 순서,
`try/catch/finally`를 복습하고, TypeScript에서 async 반환 타입과 `unknown` 에러 처리를
어떻게 작성할지 정리합니다.

### 레퍼런스 연결

- 비동기 챕터 복습
- 핵심 복습 포인트: `async`, `await`, try/catch, async 함수 반환값
- 다시 볼 질문: async/await는 비동기 코드를 완전히 동기 코드처럼 만들어줄까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
async function getValue() {
  return 1;
}

const result = getValue();

console.log(result instanceof Promise);

result.then((value) => console.log(value));

console.log("sync");
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    sync
    1
    ```
    
    ### 해설
    
    `async function`은 항상 Promise를 반환합니다. 함수 안에서 `return 1`처럼 일반 값을 반환해도,
    호출 결과는 `Promise`입니다. 그래서 `result instanceof Promise`는 `true`입니다.
    
    `result.then(...)`의 콜백은 microtask로 등록됩니다.
    
    그 다음 동기 코드인 `console.log("sync")`가 먼저 실행됩니다.
    
    동기 코드가 끝난 뒤 Promise가 해결된 값 `1`이 `then` 콜백으로 전달되어 마지막에 `1`이
    출력됩니다.
    
    `async`는 반환값을 Promise로 감싸는 문법입니다. 함수 본문이 짧고 동기처럼 보여도 호출자는
    Promise를 받는다는 점을 잊으면 안 됩니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `await` 이후 코드가 언제 이어지는지 설명하세요.

```jsx
async function run() {
  console.log("A");
  const value = await Promise.resolve("B");
  console.log(value);
  return "C";
}

console.log("start");

run().then((value) => console.log(value));

console.log("end");
```

- 정답과 해설
    
    ### 정답
    
    ```
    start
    A
    end
    B
    C
    ```
    
    ### 해설
    
    먼저 동기 코드 `"start"`가 출력됩니다.
    
    `run()`을 호출하면 async 함수 본문이 시작됩니다. `await`를 만나기 전까지는 동기적으로
    실행되므로 `"A"`가 출력됩니다.
    
    `await Promise.resolve("B")`를 만나면 async 함수는 일단 멈추고, `await` 이후 코드는
    microtask로 이어집니다.
    
    그 사이 바깥 동기 코드인 `"end"`가 출력됩니다.
    
    동기 코드가 끝나면 async 함수가 재개됩니다. `await`의 결과값 `"B"`가 `value`에 들어가고,
    `console.log(value)`가 실행되어 `"B"`가 출력됩니다.
    
    그 다음 `return "C"`는 `run()`이 반환한 Promise를 `"C"`로 fulfill합니다. 바깥에서 붙여 둔
    `.then((value) => console.log(value))`가 실행되어 `"C"`가 출력됩니다.
    
    `await`는 코드를 읽기 쉽게 만들지만, 실행 순서상으로는 현재 콜 스택이 끝난 뒤 이어진다는 점을
    기억해야 합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, async 함수 안에서 던진 에러가 어떻게 처리되는지
설명하세요.

```jsx
async function fail() {
  throw new Error("boom");
}

async function handle() {
  try {
    await fail();
    console.log("skip");
  } catch (error) {
    console.log(error.message);
    return "handled";
  }
}

handle().then((value) => console.log(value));
```

- 정답과 해설
    
    ### 정답
    
    ```
    boom
    handled
    ```
    
    ### 해설
    
    `fail`은 async 함수입니다. async 함수 안에서 에러를 던지면, 호출 결과 Promise가 rejected
    상태가 됩니다.
    
    `handle` 안에서는 `await fail()`을 `try` 블록 안에서 실행합니다. `fail()`이 reject되므로
    `await` 지점에서 예외처럼 catch 블록으로 이동합니다.
    
    `console.log("skip")`은 실행되지 않습니다.
    
    `catch` 블록에서는 `error.message`를 출력하므로 `"boom"`이 나옵니다. 그 뒤 `"handled"`를
    반환합니다.
    
    `handle`도 async 함수이므로 `"handled"`는 Promise의 fulfilled 값이 됩니다. 바깥의
    `handle().then(...)`이 그 값을 받아 `"handled"`를 출력합니다.
    
    async/await의 `try/catch`는 Promise rejection을 동기 예외처럼 다룰 수 있게 해 주지만, 결과는
    여전히 Promise라는 점이 중요합니다.
    

### 문제 4. 심화

다음 코드는 사용자 이름을 async/await로 가져옵니다. 실행 결과를 예측하고, 이 코드의 의미를
JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function fetchUser(id) {
  if (id <= 0) {
    return Promise.reject(new Error("invalid id"));
  }

  return Promise.resolve({ id, name: "Ada" });
}

async function getUserName(id) {
  try {
    const user = await fetchUser(id);
    return user.name;
  } catch (error) {
    return `fallback:${error.message}`;
  } finally {
    console.log("done");
  }
}

getUserName(1).then((name) => console.log(name));
getUserName(0).then((name) => console.log(name));
```

- 정답과 해설
    
    ### 정답
    
    ```
    done
    done
    Ada
    fallback:invalid id
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = {
      id: number;
      name: string;
    };
    
    async function fetchUser(id: number): Promise<User> {
      if (!Number.isInteger(id) || id <= 0) {
        throw new Error("invalid id");
      }
    
      return { id, name: "Ada" };
    }
    
    async function getUserName(id: number): Promise<string> {
      try {
        const user = await fetchUser(id);
        return user.name;
      } catch (error: unknown) {
        if (error instanceof Error) {
          return `fallback:${error.message}`;
        }
    
        return "fallback:unknown error";
      } finally {
        console.log("done");
      }
    }
    
    getUserName(1).then((name) => console.log(name));
    getUserName(0).then((name) => console.log(name));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `getUserName(1)`은 `fetchUser(1)`을 기다립니다. `id`가 양수이므로 `{ id: 1, name: "Ada" }`를
    가진 Promise가 fulfilled됩니다. `await` 결과는 사용자 객체이고, `return user.name`은
    `"Ada"`를 반환합니다.
    
    `getUserName(0)`은 `fetchUser(0)`에서 rejected Promise를 반환합니다. `await`는 이 rejection을
    예외처럼 던지고, 흐름은 `catch` 블록으로 이동합니다. `catch`는 `"fallback:invalid id"`를
    반환합니다.
    
    두 호출 모두 `finally` 블록을 실행합니다. `finally`는 성공과 실패 여부와 관계없이 실행되며,
    여기서는 `"done"`을 출력합니다.
    
    `finally`가 먼저 두 번 출력되고, 그 다음 각 async 함수의 최종 Promise에 붙은 `then` 콜백이
    실행되어 `"Ada"`, `"fallback:invalid id"`가 출력됩니다.
    
    ### TS 관점
    
    `async function fetchUser(id: number): Promise<User>`는 성공 시 `User`를 반환하는 비동기 함수라는
    계약을 표현합니다. 함수 안에서 `return { id, name: "Ada" }`처럼 일반 객체를 반환해도 async
    함수이므로 실제 반환 타입은 `Promise<User>`입니다.
    
    `getUserName(id: number): Promise<string>`은 성공하든 fallback을 사용하든 최종적으로 문자열을
    해결하는 Promise를 반환합니다.
    
    `catch (error: unknown)`으로 에러를 받는 이유는 JavaScript에서 던져지는 값이 항상 `Error`
    객체라고 보장되지 않기 때문입니다. `error instanceof Error`로 좁힌 뒤에만 `message`를 읽는
    편이 안전합니다.
    
    TypeScript는 async 함수의 반환 타입을 명확히 표현해 주지만, API 응답이 실제로 `User` 모양인지,
    `id`가 도메인상 유효한지는 런타임 검증이 필요합니다.
    
    ### 실무 관점
    
    async/await는 Promise 체인을 읽기 쉽게 만들어 주지만, 다음 착각은 피해야 합니다.
    
    1. async 함수가 일반 값을 반환한다고 생각하는 것
    2. `await` 이후 코드가 즉시 동기적으로 이어진다고 생각하는 것
    3. `catch`의 에러가 항상 `Error`라고 가정하는 것
    4. 타입 선언만으로 외부 API 응답 검증이 끝났다고 생각하는 것
    
    실무에서는 async 함수의 반환 타입, 에러 처리 정책, fallback 정책을 명확히 정해야 합니다.
    특히 `catch`에서 에러를 삼키고 fallback을 반환하는 코드는 호출자 입장에서 실패와 성공을
    구분하기 어려워질 수 있으므로 의도적으로 사용해야 합니다.
