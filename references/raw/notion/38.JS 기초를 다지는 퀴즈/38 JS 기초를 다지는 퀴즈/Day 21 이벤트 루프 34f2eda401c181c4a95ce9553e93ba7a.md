# Day 21. 이벤트 루프

### 오늘의 목표

JavaScript는 한 번에 하나의 콜 스택을 실행하지만, 비동기 작업은 task queue와 microtask
queue를 통해 나중에 실행됩니다. 오늘은 동기 코드, Promise job, `queueMicrotask`,
`setTimeout`의 실행 순서를 복습하고, UI 상태 업데이트나 스케줄러 코드에서 실행 순서를
명확히 설명하는 방법을 TypeScript와 함께 정리합니다.

### 레퍼런스 연결

- 비동기 챕터 복습
- 핵심 복습 포인트: call stack, task queue, microtask queue, 실행 순서
- 다시 볼 질문: 비동기 코드의 실행 순서를 설명할 수 있어야 하는 이유는 무엇일까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
console.log("A");

Promise.resolve().then(() => console.log("B"));

console.log("C");

setTimeout(() => console.log("D"), 0);
```

- 정답과 해설
    
    ### 정답
    
    ```
    A
    C
    B
    D
    ```
    
    ### 해설
    
    먼저 동기 코드가 실행됩니다. 그래서 `"A"`가 출력됩니다.
    
    `Promise.resolve().then(...)`의 콜백은 바로 실행되지 않습니다. Promise의 `then` 콜백은
    microtask queue에 들어갑니다.
    
    그 다음 동기 코드인 `console.log("C")`가 실행되어 `"C"`가 출력됩니다.
    
    `setTimeout(..., 0)`의 콜백은 task queue에 들어갑니다. 지연 시간이 0이라고 해도 현재 콜
    스택이 끝나기 전에 실행되는 것은 아닙니다.
    
    현재 동기 코드가 모두 끝나면 JavaScript 런타임은 먼저 microtask queue를 비웁니다. 그래서
    Promise 콜백 `"B"`가 `setTimeout` 콜백보다 먼저 실행됩니다.
    
    그 뒤 task queue의 `setTimeout` 콜백이 실행되어 `"D"`가 출력됩니다.
    
    핵심 순서는 `동기 코드 -> microtask -> task`입니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, microtask가 추가로 등록될 때 어떤 순서로
처리되는지 설명하세요.

```jsx
console.log("start");

setTimeout(() => console.log("timeout"), 0);

queueMicrotask(() => console.log("microtask 1"));

Promise.resolve().then(() => {
  console.log("promise 1");
  queueMicrotask(() => console.log("microtask 2"));
});

console.log("end");
```

- 정답과 해설
    
    ### 정답
    
    ```
    start
    end
    microtask 1
    promise 1
    microtask 2
    timeout
    ```
    
    ### 해설
    
    먼저 동기 코드가 실행됩니다. `"start"`가 출력되고, `setTimeout` 콜백은 task queue에
    예약됩니다.
    
    `queueMicrotask(() => console.log("microtask 1"))`는 microtask를 등록합니다.
    
    `Promise.resolve().then(...)`도 microtask를 등록합니다. 이 시점의 microtask queue에는
    `microtask 1`이 먼저 있고, 그 뒤에 Promise 콜백이 있습니다.
    
    동기 코드 마지막의 `"end"`가 출력됩니다.
    
    이제 콜 스택이 비었으므로 microtask queue를 처리합니다. 먼저 `"microtask 1"`이 출력됩니다.
    그 다음 Promise 콜백이 실행되어 `"promise 1"`을 출력하고, 내부에서 또
    `queueMicrotask(... "microtask 2")`를 등록합니다.
    
    microtask 처리 중 새 microtask가 추가되면, 런타임은 task로 넘어가기 전에 그 microtask까지
    처리합니다. 그래서 `"microtask 2"`가 `"timeout"`보다 먼저 출력됩니다.
    
    모든 microtask가 끝난 뒤에야 `setTimeout` task가 실행되어 `"timeout"`이 출력됩니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `await` 이후 코드가 언제 실행되는지 설명하세요.

```jsx
async function run() {
  console.log("run start");
  await null;
  console.log("run after await");
}

console.log("script start");

run();

Promise.resolve().then(() => console.log("promise then"));

console.log("script end");
```

- 정답과 해설
    
    ### 정답
    
    ```
    script start
    run start
    script end
    run after await
    promise then
    ```
    
    ### 해설
    
    `"script start"`는 동기 코드이므로 먼저 출력됩니다.
    
    `run()`을 호출하면 async 함수 본문이 시작됩니다. `await`를 만나기 전까지는 동기적으로
    실행되므로 `"run start"`가 출력됩니다.
    
    `await null`을 만나면 async 함수는 일단 멈추고, `await` 이후 코드는 microtask로 이어서
    실행될 준비를 합니다.
    
    그 다음 `Promise.resolve().then(...)`도 microtask를 등록합니다. 이 예제에서는 `run()`이 먼저
    호출되어 `await` 이후 continuation이 먼저 등록되고, 그 다음 Promise `then` 콜백이 등록됩니다.
    
    동기 코드 마지막의 `"script end"`가 출력됩니다.
    
    이제 microtask queue가 처리됩니다. 먼저 async 함수의 `await` 이후 코드가 실행되어
    `"run after await"`가 출력되고, 그 다음 Promise `then` 콜백인 `"promise then"`이 출력됩니다.
    
    `await`는 “그 줄에서 즉시 모든 것이 동기적으로 계속된다”는 뜻이 아닙니다. `await` 뒤의 코드는
    현재 콜 스택이 끝난 뒤 microtask로 이어집니다.
    

### 문제 4. 심화

다음 코드는 상태 변경 작업을 microtask로 예약합니다. 실행 결과를 예측하고, 이 코드의 의미를
JS 관점과 TS 관점에서 설명한 뒤 더 명확한 TypeScript 스케줄러 코드로 개선하세요.

```jsx
const logs = [];

function scheduleState(value) {
  Promise.resolve().then(() => {
    logs.push(`state:${value}`);
  });
}

scheduleState(1);
logs.push("sync");
scheduleState(2);

console.log(logs.join("|"));

setTimeout(() => {
  console.log(logs.join("|"));
}, 0);
```

- 정답과 해설
    
    ### 정답
    
    ```
    sync
    sync|state:1|state:2
    ```
    
    더 명확하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type Job = () => void;
    
    function createMicrotaskScheduler() {
      const queue: Job[] = [];
      let scheduled = false;
    
      function flush(): void {
        scheduled = false;
        const jobs = queue.splice(0);
    
        for (const job of jobs) {
          job();
        }
      }
    
      return function schedule(job: Job): void {
        queue.push(job);
    
        if (!scheduled) {
          scheduled = true;
          queueMicrotask(flush);
        }
      };
    }
    
    const logs: string[] = [];
    const schedule = createMicrotaskScheduler();
    
    schedule(() => logs.push("state:1"));
    logs.push("sync");
    schedule(() => logs.push("state:2"));
    
    console.log(logs.join("|"));
    setTimeout(() => console.log(logs.join("|")), 0);
    ```
    
    ### 해설
    
    ### JS 관점
    
    `scheduleState(1)`은 `logs`에 바로 값을 넣지 않습니다. `Promise.resolve().then(...)`으로
    microtask를 등록할 뿐입니다.
    
    그 다음 `logs.push("sync")`는 동기 코드이므로 즉시 실행됩니다.
    
    `scheduleState(2)`도 두 번째 microtask를 등록합니다.
    
    첫 번째 `console.log(logs.join("|"))`가 실행되는 시점에는 아직 microtask가 처리되지
    않았습니다. 그래서 `logs`에는 `"sync"`만 있고, 첫 번째 출력은 `"sync"`입니다.
    
    동기 코드가 끝나면 microtask queue가 처리됩니다. 먼저 `state:1`이 추가되고, 그 다음
    `state:2`가 추가됩니다.
    
    그 뒤 `setTimeout` task가 실행됩니다. 이 시점에는 microtask가 모두 처리된 뒤이므로 두 번째
    출력은 `"sync|state:1|state:2"`입니다.
    
    ### TS 관점
    
    개선 예시에서는 예약할 작업을 `type Job = () => void`로 명시했습니다. 스케줄러는 임의의 값을
    받는 것이 아니라, 나중에 실행할 함수만 받습니다.
    
    `createMicrotaskScheduler`는 내부 queue와 `scheduled` 플래그를 클로저로 숨깁니다.
    여러 작업이 같은 동기 구간에서 예약되어도 `queueMicrotask(flush)`는 한 번만 등록됩니다.
    
    `flush`는 현재 queue를 `splice(0)`으로 꺼낸 뒤 순서대로 실행합니다. 이렇게 하면 어떤 작업이
    예약되었고, 언제 실행되는지 코드 구조가 더 명확해집니다.
    
    TypeScript는 작업 타입과 `logs`의 문자열 배열 타입을 보장해 주지만, microtask와 task의 실행
    순서 자체를 바꾸지는 않습니다. 실행 순서 이해는 JavaScript 이벤트 루프 규칙을 알아야 합니다.
    
    ### 실무 관점
    
    상태 업데이트, 렌더링 예약, 로깅, 배치 처리 같은 코드에서는 “지금 실행되는가, 나중에 실행되는가”가
    매우 중요합니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 동기 실행과 예약 실행을 함수 이름으로 구분한다.
    2. microtask를 쓰는지 task를 쓰는지 명확히 한다.
    3. 여러 업데이트를 배치하려면 한 번만 flush를 예약한다.
    4. 테스트에서는 즉시 결과를 기대하지 말고, microtask 또는 timer가 처리된 뒤 확인한다.
    
    비동기 코드는 문법보다 실행 순서가 핵심입니다. `Promise`, `queueMicrotask`, `setTimeout`이
    섞인 코드는 반드시 큐의 순서를 기준으로 설명할 수 있어야 합니다.
