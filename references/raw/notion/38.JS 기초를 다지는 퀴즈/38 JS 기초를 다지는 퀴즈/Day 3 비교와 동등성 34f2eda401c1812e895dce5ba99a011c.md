# Day 3. 비교와 동등성

### 오늘의 목표

JavaScript의 비교 연산은 값뿐 아니라 타입 변환 규칙과도 연결됩니다. 오늘은 `==`,
`===`, `Object.is`, `NaN`, `+0`과 `-0`, 배열 검색 메서드의 동등성 비교 차이를
복습하고, 실무 코드에서 느슨한 비교가 어떤 버그를 만들 수 있는지 확인합니다.

### 레퍼런스 연결

- 타입 챕터 복습
- 핵심 복습 포인트: `==`, `===`, `Object.is`, `NaN`, `0`
- 다시 볼 질문: `===`를 쓰면 비교 버그가 모두 사라질까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
console.log(0 == false);
console.log(0 === false);
console.log(null == undefined);
console.log(null === undefined);
console.log(NaN === NaN);
console.log(Object.is(NaN, NaN));
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    false
    true
    false
    false
    true
    ```
    
    ### 해설
    
    `==`는 동등 연산자입니다. 두 값의 타입이 다르면 JavaScript가 암시적 타입 변환을
    시도한 뒤 비교합니다. 그래서 `0 == false`는 `true`입니다. `false`가 숫자 `0`으로
    변환된 뒤 비교되기 때문입니다.
    
    `===`는 일치 연산자입니다. 타입 변환을 하지 않고 값과 타입을 함께 비교합니다.
    `0`은 Number이고 `false`는 Boolean이므로 `0 === false`는 `false`입니다.
    
    `null == undefined`는 JavaScript의 특별한 규칙 때문에 `true`입니다. 하지만
    `null === undefined`는 타입이 다르므로 `false`입니다.
    
    `NaN === NaN`은 `false`입니다. `NaN`은 자기 자신과도 같지 않은 값으로 취급됩니다.
    `NaN` 여부를 비교해야 할 때는 `Number.isNaN` 또는 `Object.is`를 사용할 수 있습니다.
    `Object.is(NaN, NaN)`은 `true`입니다.
    
    핵심은 `==`가 편해 보이지만 비교 전에 어떤 타입 변환이 일어나는지 알아야 한다는
    점입니다. 실무에서는 의도하지 않은 변환을 줄이기 위해 보통 `===`를 기본으로 사용합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `===`와 `Object.is`의 차이를 설명하세요.

```jsx
console.log(-0 === +0);
console.log(Object.is(-0, +0));
console.log(Object.is(0, -0));
console.log(Object.is(Number("JS"), NaN));
console.log(Object.is(1, "1"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    false
    false
    true
    false
    ```
    
    ### 해설
    
    - `0 === +0`은 `true`입니다. 일치 연산자 `===`는 `+0`과 `0`을 같은 값으로 봅니다.
    대부분의 코드에서는 이 차이가 중요하지 않지만, 수학 계산이나 방향성을 가진 값에서는
    차이가 의미를 가질 수 있습니다.
    
    `Object.is(-0, +0)`은 `false`입니다. `Object.is`는 `+0`과 `-0`을 구분합니다.
    `Object.is(0, -0)`도 `false`입니다.
    
    반대로 `NaN`에서는 차이가 있습니다. `Number("JS")`는 `NaN`입니다. `Object.is(NaN, NaN)`은 `true`이므로 `Object.is(Number("JS"), NaN)`도 `true`입니다.
    
    `Object.is(1, "1")`은 `false`입니다. `Object.is`는 `==`처럼 타입 변환을 하지
    않습니다. 숫자 `1`과 문자열 `"1"`은 다른 값입니다.
    
    정리하면 `Object.is`는 `===`와 거의 비슷하지만, `NaN`을 자기 자신과 같게 보고
    `+0`과 `-0`을 구분한다는 점이 다릅니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `indexOf`와 `includes`가 `NaN`을 다루는
방식의 차이를 설명하세요.

```jsx
const list = [NaN, 0, -0];

console.log(list.indexOf(NaN));
console.log(list.includes(NaN));
console.log(list.includes(+0));
console.log(Object.is(list[1], list[2]));
```

- 정답과 해설
    
    ### 정답
    
    ```
    -1
    true
    true
    false
    ```
    
    ### 해설
    
    `indexOf`는 내부적으로 `===`와 비슷한 비교 방식을 사용합니다. `NaN === NaN`은
    `false`이므로 `list.indexOf(NaN)`은 값을 찾지 못하고 `-1`을 반환합니다.
    
    `includes`는 `SameValueZero`라는 비교 방식을 사용합니다. 이 방식은 `NaN`을
    `NaN`과 같은 값으로 봅니다. 그래서 `list.includes(NaN)`은 `true`입니다.
    
    `list.includes(+0)`도 `true`입니다. `SameValueZero`는 `+0`과 `-0`을 같은 값으로
    취급합니다. 배열 안에는 `0`과 `-0`이 있고, `+0`을 찾으면 포함되어 있다고 판단합니다.
    
    마지막 줄의 `Object.is(list[1], list[2])`는 `false`입니다. `list[1]`은 `0`,
    `list[2]`는 `-0`이고, `Object.is`는 `+0`과 `-0`을 구분하기 때문입니다.
    
    실무에서 배열에 `NaN`이 들어갈 가능성이 있다면 `indexOf`보다 `includes`가 더 의도에
    맞을 수 있습니다. 반면 위치가 필요하다면 `findIndex`와 `Number.isNaN`을 조합하는
    식으로 비교 방식을 명확히 드러내는 편이 좋습니다.
    

### 문제 4. 심화

다음 코드는 사용자 ID를 입력받아 사용자를 찾으려고 작성되었습니다. 실행 결과를
예측하고, 이 코드의 문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript
코드로 개선하세요.

```jsx
const users = [
  { id: 1, name: "Ada" },
  { id: 2, name: "Lin" },
];

function findUserLoose(idInput) {
  return users.find((user) => user.id == idInput)?.name ?? "not found";
}

console.log(findUserLoose("1"));
console.log(findUserLoose("01"));
console.log(findUserLoose(true));
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    Ada
    Ada
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = { id: number; name: string };
    
    const users: User[] = [
      { id: 1, name: "Ada" },
      { id: 2, name: "Lin" },
    ];
    
    function parseUserId(input: string): number | null {
      if (!/^[1-9][0-9]*$/.test(input)) {
        return null;
      }
    
      const id = Number(input);
      return Number.isSafeInteger(id) ? id : null;
    }
    
    function findUser(idInput: string): string | null {
      const id = parseUserId(idInput);
    
      if (id === null) {
        return null;
      }
    
      return users.find((user) => user.id === id)?.name ?? null;
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    `findUserLoose("1")`은 `"1"`과 숫자 `1`을 `==`로 비교합니다. 동등 연산자는 타입이
    다르면 암시적 타입 변환을 수행하므로 문자열 `"1"`은 숫자 `1`처럼 비교됩니다. 그래서
    `Ada`가 반환됩니다.
    
    문제는 `findUserLoose("01")`도 `Ada`를 반환한다는 점입니다. 문자열 `"01"`도 숫자로
    변환하면 `1`입니다. 사용자가 입력한 ID 형식을 엄격하게 보아야 한다면 `"01"`은 `"1"`과
    다른 입력인데, 느슨한 비교 때문에 같은 사용자로 처리됩니다.
    
    `findUserLoose(true)`도 `Ada`를 반환합니다. `true`는 숫자 비교 과정에서 `1`로
    변환되므로 `user.id == true`가 `1 == 1`처럼 평가됩니다.
    
    이 코드는 “값이 비슷하면 맞다”는 식으로 동작합니다. 인증, 권한, 결제, 사용자 식별처럼
    정확한 식별이 중요한 코드에서는 매우 위험합니다.
    
    ### TS 관점
    
    TypeScript로 `idInput`을 `string`으로 제한하면 `true` 같은 Boolean 입력은 컴파일
    단계에서 막을 수 있습니다. 하지만 문자열 `"01"`이 허용되는지, `"1"`만 허용되는지는
    타입 `string`만으로는 표현되지 않습니다.
    
    그래서 개선 코드에서는 `parseUserId`로 런타임 검증을 수행합니다. 정규식
    `/^[1-9][0-9]*$/`는 첫 글자가 `1`부터 `9` 사이이고, 그 뒤에는 숫자가 이어지는 문자열만
    허용합니다. 따라서 `"1"`은 통과하지만 `"01"`은 거부됩니다.
    
    검증 후에는 숫자 `id`와 사용자 객체의 숫자 `user.id`를 `===`로 비교합니다. 이렇게 하면
    비교 시점에 암시적 타입 변환이 일어나지 않습니다.
    
    ### 실무 관점
    
    실무에서는 URL 파라미터, form 입력, localStorage 값처럼 문자열로 들어오는 ID를 자주
    다룹니다. 이 값을 DB나 상태의 숫자 ID와 비교할 때 `==`를 쓰면 의도하지 않은 값이
    같다고 처리될 수 있습니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 입력값은 문자열로 받는다.
    2. 도메인 규칙에 맞는 형식인지 검증한다.
    3. 검증된 값만 숫자로 변환한다.
    4. 내부 비교는 `===`로 수행한다.
    
    TypeScript는 입력 타입을 제한하는 데 도움을 주지만, 문자열의 형식 검증은 런타임 코드가
    담당해야 합니다.
