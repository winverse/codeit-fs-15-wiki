# Day 16. 생성자 함수와 바인딩

### 오늘의 목표

JavaScript 함수는 일반 함수로도 호출되고, `new`와 함께 생성자 함수로도 호출될 수
있습니다. 오늘은 생성자 함수, 프로토타입, `new` 없이 호출했을 때의 문제, `call`, `apply`,
`bind`를 복습하고, TypeScript에서 명시적 `this` 파라미터와 바인딩된 콜백 타입을 어떻게
표현할지 확인합니다.

### 레퍼런스 연결

- 함수와 `this` 연결 내용 복습
- 핵심 복습 포인트: 생성자 함수, `new`, `bind`, `call`, `apply`
- 다시 볼 질문: 함수 호출 방식은 API 설계에 어떤 영향을 줄까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
function User(name) {
  this.name = name;
}

User.prototype.sayName = function () {
  return this.name;
};

const ada = new User("Ada");

console.log(ada.name);
console.log(ada.sayName());
console.log(ada instanceof User);
console.log(Object.getPrototypeOf(ada) === User.prototype);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    Ada
    true
    true
    ```
    
    ### 해설
    
    `new User("Ada")`를 호출하면 새 객체가 만들어지고, 생성자 함수 안의 `this`는 그 새 객체를
    가리킵니다. 그래서 `this.name = name`은 새 객체의 `name` 속성을 `"Ada"`로 설정합니다.
    
    `ada.name`은 `"Ada"`입니다.
    
    `User.prototype.sayName`에 함수를 추가했으므로, `new User(...)`로 만들어진 객체는
    프로토타입 체인을 통해 `sayName`을 사용할 수 있습니다. `ada.sayName()`으로 호출하면
    메서드 호출 규칙에 따라 `this`는 `ada`가 되고, `"Ada"`가 반환됩니다.
    
    `ada instanceof User`는 `true`입니다. `ada`의 프로토타입 체인에 `User.prototype`이 있기
    때문입니다.
    
    `Object.getPrototypeOf(ada) === User.prototype`도 `true`입니다. `ada`의 직접 프로토타입이
    바로 `User.prototype`입니다.
    
    생성자 함수는 클래스 문법의 기반이 되는 오래된 객체 생성 방식입니다. 클래스 문법을 쓰더라도
    프로토타입을 이해하면 메서드 공유와 `instanceof` 동작을 더 정확히 설명할 수 있습니다.
    TypeScript에서 생성 가능한 함수의 타입을 별도로 표현해야 한다면
    `new (name: string) => User` 같은 constructor signature를 사용할 수 있습니다.
    

### 문제 2. 중급

다음 코드는 strict mode에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하고,
생성자 함수를 `new` 없이 호출하면 어떤 문제가 생기는지 설명하세요.

```jsx
"use strict";

function Person(name) {
  this.name = name;
}

try {
  Person("Lin");
} catch (error) {
  console.log(error.constructor.name);
}

const lin = new Person("Lin");

console.log(lin.name);
console.log(lin instanceof Person);
```

- 정답과 해설
    
    ### 정답
    
    ```
    TypeError
    Lin
    true
    ```
    
    ### 해설
    
    strict mode에서 일반 함수 호출의 `this`는 자동으로 전역 객체가 되지 않습니다. `Person("Lin")`
    처럼 `new` 없이 호출하면 함수 내부의 `this`는 `undefined`입니다.
    
    그 상태에서 `this.name = name`을 실행하려고 하면 `undefined`에 속성을 설정하는 코드가
    되므로 `TypeError`가 발생합니다. `catch` 블록에서 에러 생성자 이름을 출력해 `"TypeError"`가
    나옵니다.
    
    반면 `new Person("Lin")`은 생성자 호출입니다. 새 객체가 만들어지고 `this`가 그 객체에
    바인딩됩니다. 그래서 `lin.name`은 `"Lin"`입니다.
    
    `lin instanceof Person`도 `true`입니다. `new`로 만들어진 객체의 프로토타입 체인에
    `Person.prototype`이 연결되기 때문입니다.
    
    비엄격 모드에서는 `new` 없이 생성자 함수를 호출했을 때 전역 객체가 오염될 수 있습니다.
    strict mode에서는 이런 실수를 더 빨리 실패하게 만들어 줍니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `call`, `apply`, `bind`의 차이를 설명하세요.

```jsx
function introduce(prefix, suffix) {
  return `${prefix}${this.name}${suffix}`;
}

const user = { name: "Ada" };

console.log(introduce.call(user, "Hi", "!"));
console.log(introduce.apply(user, ["Hello", "."]));

const bound = introduce.bind(user, "Hey");

console.log(bound("?"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    Hi Ada!
    Hello Ada.
    Hey Ada?
    ```
    
    ### 해설
    
    `call`, `apply`, `bind`는 함수의 `this`를 명시적으로 지정할 때 사용합니다.
    
    `introduce.call(user, "Hi", "!")`는 `this`를 `user`로 지정하고, 나머지 인자를 하나씩
    전달합니다. 함수 안에서 `this.name`은 `"Ada"`이므로 `"Hi Ada!"`가 반환됩니다.
    
    `introduce.apply(user, ["Hello", "."])`도 `this`를 `user`로 지정합니다. 차이는 나머지
    인자를 배열로 전달한다는 점입니다. 결과는 `"Hello Ada."`입니다.
    
    `introduce.bind(user, "Hey")`는 바로 함수를 실행하지 않습니다. `this`를 `user`로 고정하고
    첫 번째 인자 `"Hey"`까지 미리 고정한 새 함수를 반환합니다.
    
    `bound("?")`를 호출하면 이미 고정된 `"Hey"`와 새로 전달한 `"?"`를 사용해 `"Hey Ada?"`를
    반환합니다.
    
    정리하면 `call`과 `apply`는 즉시 호출하고, `bind`는 나중에 호출할 새 함수를 만듭니다.
    

### 문제 4. 심화

다음 코드는 strict mode에서 메서드를 콜백으로 넘깁니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
"use strict";

const user = {
  name: "Ada",
  getName() {
    return this.name;
  },
};

function run(callback) {
  return callback();
}

try {
  console.log(run(user.getName));
} catch (error) {
  console.log(error.constructor.name);
}

const safeGetName = user.getName.bind(user);

console.log(run(safeGetName));
```

- 정답과 해설
    
    ### 정답
    
    ```
    TypeError
    Ada
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = {
      name: string;
    };
    
    function getName(this: User): string {
      return this.name;
    }
    
    function run(callback: () => string): string {
      return callback();
    }
    
    const user: User = { name: "Ada" };
    const safeGetName: OmitThisParameter<typeof getName> = getName.bind(user);
    
    console.log(run(safeGetName));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `user.getName()`처럼 메서드 형태로 호출하면 `this`는 `user`입니다. 하지만
    `run(user.getName)`은 메서드를 호출한 것이 아니라 함수 값만 꺼내서 다른 함수에 전달한
    것입니다.
    
    `run` 내부에서 `callback()`을 호출하면 앞에 객체가 붙어 있지 않은 일반 함수 호출이 됩니다.
    strict mode에서 일반 함수 호출의 `this`는 `undefined`입니다.
    
    따라서 `getName` 내부의 `return this.name`은 `undefined.name`을 읽으려는 코드가 되어
    `TypeError`가 발생합니다.
    
    `user.getName.bind(user)`는 `this`가 항상 `user`가 되도록 고정한 새 함수를 만듭니다.
    `run(safeGetName)`은 일반 함수 호출처럼 실행되지만, 바인딩된 함수 내부의 `this`는 `user`로
    고정되어 있으므로 `"Ada"`가 반환됩니다.
    
    ### TS 관점
    
    TypeScript에서는 함수의 가짜 첫 번째 파라미터처럼 `this` 타입을 명시할 수 있습니다.
    `function getName(this: User): string`은 이 함수가 `User`를 `this`로 기대한다는 뜻입니다.
    
    `bind`를 사용하면 `this`가 고정된 함수가 만들어지고, 이 함수는 더 이상 호출자에게 `this`를
    요구하지 않습니다. 예시의 `OmitThisParameter<typeof getName>`은 `this` 파라미터가 제거된
    함수 타입을 표현합니다.
    
    다만 TypeScript가 모든 런타임 바인딩 실수를 완벽하게 막아 주는 것은 아닙니다. 특히 객체
    메서드를 값으로 꺼내 전달하는 패턴은 코드 구조에 따라 타입만으로 의도가 충분히 드러나지
    않을 수 있습니다. 예를 들어 `run(callback: () => string)` 같은 콜백 타입에 명시적 `this`
    파라미터가 있는 함수를 그대로 넘겨도 TypeScript 버전과 함수 타입 호환 규칙에 따라 컴파일될
    수 있습니다. 그래서 콜백으로 넘길 함수는 처음부터 `this`를 쓰지 않게 만들거나, 넘기기 전에
    명시적으로 바인딩하는 규칙을 두는 편이 좋습니다.
    
    ### 실무 관점
    
    이 문제는 이벤트 핸들러, 타이머, 배열 콜백, 라이브러리 콜백에서 자주 발생합니다. 메서드를
    그대로 콜백으로 넘기면 원래 객체와의 연결이 끊어질 수 있습니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 콜백으로 넘길 함수가 `this`에 의존하는지 확인한다.
    2. 가능하면 `this` 대신 필요한 값을 인자로 받는 순수 함수로 만든다.
    3. 메서드를 넘겨야 한다면 `bind` 또는 화살표 함수 래퍼로 `this`를 고정한다.
    4. TypeScript에서는 명시적 `this` 파라미터와 바인딩 후 타입을 함께 사용한다.
    
    `this` 바인딩은 함수가 “어디에 정의되어 있는가”보다 “어떻게 호출되는가”에 더 크게 영향을
    받습니다.
