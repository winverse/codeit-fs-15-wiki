### 문제 1. 기본

다음 코드의 실행 결과를 작성하고, `filter(Boolean)`이 어떤 값을 제거하는지 설명하세요.

```jsx
const values = [0, 1, "", "JS", null, 2];

console.log(values.filter(Boolean).join("|"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    1|JS|2
    ```
    
    ### 해설
    
    `filter(Boolean)`은 배열의 각 값을 `Boolean(value)`로 평가한 뒤, 결과가 `true`인 값만 남깁니다.
    
    이 배열에서 falsy 값은 `0`, `""`, `null`입니다. 이 값들은 제거됩니다.
    
    truthy 값은 `1`, `"JS"`, `2`입니다. 그래서 필터링 결과는 `[1, "JS", 2]`입니다.
    
    마지막으로 `.join("|")`가 실행되어 각 값이 문자열로 이어집니다. 결과는 `"1|JS|2"`입니다.
    
    `filter(Boolean)`은 짧고 편리하지만, `0`이나 빈 문자열도 의미 있는 값일 수 있습니다. 예를 들어
    점수 `0`, 수량 `0`, 사용자가 입력한 빈 문자열을 보존해야 하는 상황이라면 명시적인 조건을
    사용해야 합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 작성하고, 왜 모든 함수가 같은 값을 반환하는지 설명하세요.

```jsx
function createLoggers() {
  const result = [];

  for (var i = 0; i < 3; i += 1) {
    result.push(() => i);
  }

  return result;
}

console.log(createLoggers().map((fn) => fn()).join(","));
```

- 정답과 해설
    
    ### 정답
    
    ```
    3,3,3
    ```
    
    ### 해설
    
    `var`는 블록 스코프가 아니라 함수 스코프를 가집니다. 따라서 `for` 반복문 안에서 매번 새로운
    `i`가 만들어지는 것이 아니라, `createLoggers` 함수 스코프에 하나의 `i`가 만들어집니다.
    
    반복문이 끝나면 `i`는 `3`이 됩니다. 배열에 들어간 세 개의 화살표 함수는 모두 같은 `i` 변수를
    참조합니다.
    
    그래서 나중에 `fn()`을 호출하면 세 함수 모두 현재의 `i` 값인 `3`을 반환합니다.
    
    `let`을 사용하면 반복마다 새로운 binding이 만들어지므로 결과가 달라집니다.
    
    ```jsx
    for (let i = 0; i < 3; i += 1) {
      result.push(() => i);
    }
    ```
    
    이 경우 각 함수는 자신이 만들어진 반복의 `i`를 기억하므로 `0,1,2`가 됩니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `catch`와 `finally`가 반환값에 어떤 영향을 주는지
설명하세요.

```jsx
async function run() {
  try {
    return await Promise.reject(new Error("fail"));
  } catch (error) {
    return error.message;
  } finally {
    console.log("cleanup");
  }
}

run().then(console.log);
```

- 정답과 해설
    
    ### 정답
    
    ```
    cleanup
    fail
    ```
    
    ### 해설
    
    `Promise.reject(new Error("fail"))`는 거부된 Promise를 만듭니다. `await`는 이 rejection을
    예외처럼 던지므로 실행 흐름은 `catch`로 이동합니다.
    
    `catch` 블록에서는 `error.message`를 반환합니다. `error.message`는 `"fail"`입니다.
    
    하지만 함수가 실제로 완료되기 전에 `finally` 블록이 먼저 실행됩니다. 그래서 `"cleanup"`이 먼저
    출력됩니다.
    
    그 후 `run()`이 `"fail"`로 resolve되고, `.then(console.log)`가 실행되어 `"fail"`이 출력됩니다.
    
    `finally`는 정리 작업에 적합합니다. 단, `finally` 안에서 값을 `return`하거나 새 에러를 던지면
    앞에서 결정된 반환값이나 에러를 덮어쓸 수 있습니다. 정리 목적이라면 보통 `return`을 넣지 않는
    편이 안전합니다.
    

### 문제 4. 심화

다음 코드는 외부에서 받은 사용자 배열을 화면에 쓰기 좋은 형태로 정규화합니다. 실행 결과를
예측하고, 이 코드의 장단점을 JS 관점과 TS 관점에서 설명한 뒤 런타임 검증을 포함한 TypeScript
코드로 개선하세요.

```jsx
const payloads = [
  { id: 1, profile: { name: " Ada " }, roles: ["admin", "mentor"] },
  { id: 2, profile: null, roles: [] },
  { id: 3, profile: { name: "" } },
];

function normalizeUsers(items) {
  return items
    .map((item) => ({
      id: String(item.id),
      name: item.profile?.name?.trim() || "Unknown",
      roleCount: item.roles?.length ?? 0,
    }))
    .filter((user) => user.roleCount > 0);
}

console.log(JSON.stringify(normalizeUsers(payloads)));
```

- 정답과 해설
    
    ### 정답
    
    ```
    [{"id":"1","name":"Ada","roleCount":2}]
    ```
    
    런타임 검증을 포함한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type RawUser = {
      id: number | string;
      profile?: { name?: string | null } | null;
      roles?: string[] | null;
    };
    
    type NormalizedUser = {
      id: string;
      name: string;
      roleCount: number;
    };
    
    function isRecord(value: unknown): value is Record<string, unknown> {
      return typeof value === "object" && value !== null;
    }
    
    function isRawUser(value: unknown): value is RawUser {
      if (!isRecord(value)) {
        return false;
      }
    
      const hasValidId = typeof value.id === "number" || typeof value.id === "string";
      const hasValidRoles =
        value.roles === undefined ||
        value.roles === null ||
        (Array.isArray(value.roles) && value.roles.every((role) => typeof role === "string"));
    
      return hasValidId && hasValidRoles;
    }
    
    function normalizeName(profile: RawUser["profile"]): string {
      const name = profile?.name?.trim();
      return name && name.length > 0 ? name : "Unknown";
    }
    
    function normalizeUsers(values: unknown[]): NormalizedUser[] {
      return values
        .filter(isRawUser)
        .map((item) => ({
          id: String(item.id),
          name: normalizeName(item.profile),
          roleCount: item.roles?.length ?? 0,
        }))
        .filter((user) => user.roleCount > 0);
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    첫 번째 payload는 다음 값으로 정규화됩니다.
    
    ```jsx
    { id: "1", name: "Ada", roleCount: 2 }
    ```
    
    `id`는 `String(1)`로 `"1"`이 됩니다. `profile.name`은 `" Ada "`이고, `trim()`을 거쳐 `"Ada"`가
    됩니다. `roles.length`는 `2`입니다. 마지막 필터 조건 `roleCount > 0`을 통과합니다.
    
    두 번째 payload는 `profile`이 `null`입니다. `item.profile?.name?.trim()`은 안전하게
    `undefined`가 되고, `|| "Unknown"`에 의해 이름은 `"Unknown"`이 됩니다. 하지만 `roles`가 빈
    배열이므로 `roleCount`는 `0`입니다. 마지막 필터에서 제거됩니다.
    
    세 번째 payload는 `profile.name`이 빈 문자열입니다. `trim()` 결과도 `""`이고, `""`는 falsy이므로
    이름은 `"Unknown"`이 됩니다. `roles`가 없으므로 `item.roles?.length`는 `undefined`이고,
    `?? 0`에 의해 `roleCount`는 `0`입니다. 이 사용자도 필터에서 제거됩니다.
    
    그래서 최종 결과에는 첫 번째 사용자만 남습니다.
    
    ### TS 관점
    
    외부 데이터는 코드가 기대하는 구조와 다를 수 있습니다. 그래서 `unknown`으로 받은 뒤 타입 가드로
    확인하는 방식이 안전합니다.
    
    `isRecord`는 값이 객체인지 확인합니다. `isRawUser`는 `id`가 문자열 또는 숫자인지, `roles`가
    없거나 `null`이거나 문자열 배열인지 확인합니다.
    
    검증을 통과한 값은 TypeScript가 `RawUser`로 좁혀서 이해합니다. 이후 `normalizeUsers` 안에서는
    `item.id`, `item.profile`, `item.roles`를 타입에 맞게 사용할 수 있습니다.
    
    `normalizeName`을 따로 분리한 이유는 이름 정규화 규칙을 명확하게 만들기 위해서입니다. 빈 문자열,
    공백 문자열, 누락된 이름을 모두 `"Unknown"`으로 처리한다는 정책이 함수 이름과 구현에 드러납니다.
    
    ### 실무 관점
    
    이 문제는 30일 동안 본 내용을 한 번에 연결합니다.
    
    1. Truthy/Falsy: `|| "Unknown"`은 빈 문자열을 기본값으로 바꿉니다.
    2. Optional chaining: `profile?.name?.trim()`은 중간 값이 없을 때 에러를 막습니다.
    3. Nullish coalescing: `roles?.length ?? 0`은 `undefined`와 `null`만 기본값으로 처리합니다.
    4. 배열 메서드: `map`으로 변환하고 `filter`로 조건에 맞는 항목만 남깁니다.
    5. TS 타입 가드: 외부 데이터를 타입으로 믿기 전에 런타임에서 확인합니다.
    
    실무에서는 “동작하는 코드”에서 한 걸음 더 나아가, 누락된 데이터와 잘못된 데이터가 들어왔을 때
    어떤 정책으로 처리할지 정해야 합니다. 조용히 제거할지, 기본값을 줄지, 에러로 보고할지에 따라
    사용자 경험과 디버깅 방식이 달라집니다.
