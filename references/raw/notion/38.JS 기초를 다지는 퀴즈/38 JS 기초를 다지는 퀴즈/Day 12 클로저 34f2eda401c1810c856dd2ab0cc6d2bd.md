# Day 12. 클로저

### 오늘의 목표

클로저는 함수가 자신이 만들어진 렉시컬 환경을 기억하는 동작입니다. 오늘은 함수가 외부
변수를 보존하는 방식, 반복문과 클로저의 관계, 클로저를 이용한 정보 은닉을 복습하고,
외부에 참조를 그대로 노출했을 때 생기는 버그를 TypeScript의 반환 타입 설계와 함께
정리합니다.

### 레퍼런스 연결

- 스코프와 클로저 챕터 복습
- 핵심 복습 포인트: 렉시컬 스코프, 상태 보존, 정보 은닉, 캡처된 변수
- 다시 볼 질문: 클로저는 언제 좋은 설계이고 언제 숨은 상태가 될까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
function makeCounter() {
  let count = 0;

  return function () {
    count += 1;
    return count;
  };
}

const c1 = makeCounter();
const c2 = makeCounter();

console.log(c1());
console.log(c1());
console.log(c2());
console.log(c1());
```

- 정답과 해설
    
    ### 정답
    
    ```
    1
    2
    1
    3
    ```
    
    ### 해설
    
    `makeCounter`가 호출되면 지역 변수 `count`가 만들어지고, 그 변수를 사용하는 내부 함수가
    반환됩니다. 바깥 함수 실행이 끝난 뒤에도 반환된 함수는 자신이 만들어질 때의 `count`를
    기억합니다. 이것이 클로저입니다.
    
    `c1`은 첫 번째 `makeCounter()` 호출로 만들어진 함수입니다. `c1()`을 처음 호출하면
    `count`가 `0`에서 `1`이 되어 `1`을 반환합니다. 두 번째 호출에서는 같은 `count`를 다시
    사용하므로 `2`가 됩니다.
    
    `c2`는 두 번째 `makeCounter()` 호출로 만들어진 별도의 함수입니다. `c1`과 `c2`는 서로
    다른 렉시컬 환경을 가집니다. 그래서 `c2()`의 첫 결과는 `1`입니다.
    
    마지막 `c1()`은 다시 `c1`이 기억하는 `count`를 증가시킵니다. 이미 `2`였으므로 `3`이
    출력됩니다.
    
    클로저는 상태를 함수 안에 숨기면서도 여러 호출 사이에 값을 유지할 수 있게 해 줍니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `var`와 `let`이 클로저 결과에 어떤 차이를
만드는지 설명하세요.

```jsx
const varFns = [];

for (var i = 0; i < 3; i += 1) {
  varFns.push(() => i);
}

console.log(varFns.map((fn) => fn()).join(","));

const letFns = [];

for (let j = 0; j < 3; j += 1) {
  letFns.push(() => j);
}

console.log(letFns.map((fn) => fn()).join(","));
```

- 정답과 해설
    
    ### 정답
    
    ```
    3,3,3
    0,1,2
    ```
    
    ### 해설
    
    `var i`는 함수 스코프를 가집니다. 반복문이 실행되는 동안 새 `i`가 매번 만들어지는 것이
    아니라 하나의 `i`를 공유합니다.
    
    `varFns`에 들어간 화살표 함수들은 모두 같은 `i`를 기억합니다. 반복문이 끝난 뒤 `i`는
    `3`입니다. 그래서 세 함수를 나중에 호출하면 모두 `3`을 반환하고, 결과는 `"3,3,3"`입니다.
    
    `let j`는 반복마다 새로운 바인딩을 만듭니다. 각 반복에서 만들어진 화살표 함수는 그 반복의
    `j`를 기억합니다. 그래서 첫 번째 함수는 `0`, 두 번째 함수는 `1`, 세 번째 함수는 `2`를
    반환합니다.
    
    이 차이는 이벤트 핸들러나 비동기 콜백을 반복문 안에서 만들 때 자주 드러납니다. 예전에는
    즉시 실행 함수로 값을 가두는 패턴을 썼지만, 현대 JavaScript에서는 반복문 카운터를 `let`으로
    두는 것이 훨씬 명확합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 클로저를 사용한 정보 은닉이 어떻게 동작하는지
설명하세요.

```jsx
function createUser(name) {
  let loginCount = 0;

  return {
    name,
    login() {
      loginCount += 1;
      return loginCount;
    },
    getLoginCount() {
      return loginCount;
    },
  };
}

const user = createUser("Ada");

console.log(user.login());
console.log(user.login());
console.log(user.loginCount);
console.log(user.getLoginCount());
```

- 정답과 해설
    
    ### 정답
    
    ```
    1
    2
    undefined
    2
    ```
    
    ### 해설
    
    `loginCount`는 `createUser` 함수의 지역 변수입니다. 반환되는 객체의 속성으로 직접
    노출되지 않습니다.
    
    `user.login()`을 호출하면 반환된 메서드가 클로저를 통해 `loginCount`에 접근합니다. 첫
    호출에서는 `1`, 두 번째 호출에서는 `2`가 반환됩니다.
    
    `user.loginCount`는 `undefined`입니다. 반환된 객체에는 `loginCount`라는 속성이 없기
    때문입니다. 지역 변수 `loginCount`는 객체 바깥에서 직접 읽거나 쓸 수 없습니다.
    
    `user.getLoginCount()`는 클로저를 통해 같은 `loginCount`를 읽습니다. 이미 두 번 로그인했으므로
    `2`를 반환합니다.
    
    이런 방식은 클래스의 private 필드와 비슷한 목적을 가질 수 있습니다. 외부에서 직접 바꾸면
    안 되는 상태를 함수 스코프 안에 숨기고, 정해진 함수로만 접근하게 만드는 패턴입니다.
    

### 문제 4. 심화

다음 코드는 장바구니 상태를 클로저 안에 숨기려고 합니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function createCart() {
  const items = [];

  return {
    add(item) {
      items.push(item);
      return items.length;
    },
    list() {
      return items;
    },
  };
}

const cart = createCart();
const leaked = cart.list();

cart.add("book");
leaked.push("pen");

console.log(cart.list().join(","));
console.log(cart.add("cup"));
console.log(cart.list().join(","));
```

- 정답과 해설
    
    ### 정답
    
    ```
    book,pen
    3
    book,pen,cup
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type CartItem = string;
    
    type Cart = {
      add(item: CartItem): number;
      list(): readonly CartItem[];
    };
    
    function createCart(): Cart {
      const items: CartItem[] = [];
    
      return {
        add(item) {
          items.push(item);
          return items.length;
        },
        list() {
          return [...items];
        },
      };
    }
    
    const cart = createCart();
    const snapshot = cart.list();
    
    cart.add("book");
    
    console.log(snapshot.length);
    console.log(cart.list().join(","));
    console.log(cart.add("cup"));
    console.log(cart.list().join(","));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `items` 배열은 `createCart`의 지역 변수입니다. 외부에서 이름으로 직접 접근할 수는 없습니다.
    이 점만 보면 클로저로 상태를 잘 숨긴 것처럼 보입니다.
    
    하지만 `list()`가 `items` 배열 자체를 반환합니다. `const leaked = cart.list()`는 내부 배열과
    같은 참조를 외부 변수에 저장합니다.
    
    그 뒤 `cart.add("book")`은 내부 배열에 `"book"`을 추가합니다. `leaked`도 같은 배열을
    가리키고 있으므로 `leaked`에서도 `"book"`이 보입니다.
    
    `leaked.push("pen")`은 더 큰 문제입니다. 외부 코드가 내부 배열을 직접 수정합니다. 그래서
    `cart.list().join(",")`은 `"book,pen"`입니다.
    
    이후 `cart.add("cup")`을 호출하면 배열에는 이미 `"book"`, `"pen"`이 있으므로 `"cup"`까지
    추가된 길이 `3`이 반환됩니다. 마지막 출력은 `"book,pen,cup"`입니다.
    
    ### TS 관점
    
    TypeScript로 `items`를 `CartItem[]`로 선언해도, 내부 배열 참조를 그대로 반환하면 외부에서
    수정할 수 있습니다. 타입만 붙였다고 정보 은닉이 완성되는 것은 아닙니다.
    
    개선 예시의 `list(): readonly CartItem[]`는 호출자가 반환된 목록을 수정하지 않는다는
    의도를 타입으로 표현합니다. 또한 구현에서 `return [...items]`를 사용해 내부 배열이 아니라
    복사본을 반환합니다.
    
    `readonly` 반환 타입과 복사본 반환은 서로 다른 안전장치입니다. `readonly`는 컴파일 단계에서
    `push` 같은 변경을 막는 데 도움을 주고, 복사본 반환은 런타임에서 내부 배열 참조가 새어
    나가지 않게 합니다.
    
    ### 실무 관점
    
    클로저로 상태를 숨길 때는 “상태 변수 이름을 외부에서 볼 수 없는가”뿐 아니라 “상태를 가리키는
    참조가 외부로 새지 않는가”도 확인해야 합니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 내부 상태는 클로저나 private 필드 안에 둔다.
    2. 외부로 반환하는 객체나 배열은 필요한 경우 복사본으로 반환한다.
    3. 반환 타입에 `readonly`를 사용해 호출자가 수정하지 못하도록 의도를 표현한다.
    4. 변경은 `add`, `remove`, `clear`처럼 통제된 메서드를 통해서만 수행한다.
    
    클로저는 강력한 정보 은닉 도구이지만, 참조 타입을 그대로 밖으로 내보내면 은닉 효과가 쉽게
    깨질 수 있습니다.
