# Day 8. 객체 전달과 복사

### 오늘의 목표

JavaScript 객체는 값 자체가 복사되는 것이 아니라 참조가 전달됩니다. 오늘은 같은 객체를
가리키는 alias, 전개 구문과 `Object.assign`의 얕은 복사, 중첩 객체 참조 공유를
복습하고, TypeScript 타입이 “복사 깊이”를 자동으로 보장하지 않는다는 점까지 실무
업데이트 코드로 연결합니다.

### 레퍼런스 연결

- 객체 챕터 복습
- 핵심 복습 포인트: 참조 전달, 얕은 복사, 깊은 복사, 불변 업데이트
- 다시 볼 질문: 복사했다고 믿었는데 원본이 바뀌는 버그를 어떻게 막을까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
const original = { name: "Ada" };
const alias = original;

alias.name = "Lin";

console.log(original.name);
console.log(original === alias);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Lin
    true
    ```
    
    ### 해설
    
    `original`에는 객체 자체가 직접 들어 있는 것이 아니라 객체를 가리키는 참조가 들어
    있다고 생각할 수 있습니다.
    
    `const alias = original`은 객체를 새로 복사하지 않습니다. `alias`와 `original`은 같은
    객체를 가리킵니다.
    
    그래서 `alias.name = "Lin"`은 `alias`만 바꾸는 것이 아니라 두 변수가 함께 가리키는
    객체의 `name` 속성을 바꿉니다. 이후 `original.name`을 읽으면 `"Lin"`이 나옵니다.
    
    `original === alias`도 `true`입니다. 두 변수가 같은 객체 참조를 가리키기 때문입니다.
    객체 비교에서 `===`는 내용이 같은지를 비교하지 않고, 같은 객체인지 비교합니다.
    
    실무에서 함수에 객체를 넘긴 뒤 그 함수가 속성을 바꾸면 호출자 쪽 객체도 바뀔 수 있습니다.
    객체를 전달할 때는 “이 함수가 원본을 변경하는가”를 명확히 해야 합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 전개 구문이 얕은 복사라는 말의 의미를
설명하세요.

```jsx
const user = {
  name: "Ada",
  profile: { city: "Seoul" },
};

const copy = { ...user };

copy.name = "Lin";
copy.profile.city = "Busan";

console.log(user.name);
console.log(user.profile.city);
console.log(user.profile === copy.profile);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    Busan
    true
    ```
    
    ### 해설
    
    `const copy = { ...user }`는 `user`의 직접 속성을 새 객체에 복사합니다. 그래서 `copy`와
    `user`는 서로 다른 최상위 객체입니다.
    
    `copy.name = "Lin"`은 `copy`의 직접 속성인 `name`만 바꿉니다. `user.name`은 그대로
    `"Ada"`입니다.
    
    하지만 `profile`은 중첩 객체입니다. 전개 구문은 `profile` 객체를 깊게 복사하지 않고,
    `profile`이 가리키는 참조를 그대로 복사합니다. 따라서 `user.profile`과 `copy.profile`은
    같은 객체입니다.
    
    `copy.profile.city = "Busan"`은 공유된 `profile` 객체 내부를 바꿉니다. 그래서
    `user.profile.city`도 `"Busan"`으로 바뀝니다.
    
    `user.profile === copy.profile`이 `true`인 이유도 같은 참조를 공유하기 때문입니다.
    
    전개 구문은 매우 유용하지만, “중첩 객체까지 새로 만들었다”는 뜻은 아닙니다. 상태
    업데이트에서 중첩 값을 바꿀 때는 변경 경로에 있는 객체를 모두 새로 만들어야 합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Object.assign`이 target 객체에 미치는
영향을 설명하세요.

```jsx
const target = { a: 1, nested: { count: 1 } };
const source = { b: 2, nested: { count: 10 } };

const result = Object.assign(target, source);

result.nested.count = 20;

console.log(target === result);
console.log(target.a);
console.log(target.b);
console.log(source.nested.count);
console.log(target.nested === source.nested);
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    1
    2
    20
    true
    ```
    
    ### 해설
    
    `Object.assign(target, source)`는 `source`의 enumerable own property를 `target`에
    복사하고, 그 `target` 객체를 반환합니다. 새 객체를 반환하는 것이 아니라 첫 번째 인자로
    받은 객체를 수정합니다.
    
    그래서 `target === result`는 `true`입니다. `result`는 수정된 `target` 그 자체입니다.
    
    `target.a`는 원래 있던 값 `1`을 유지합니다. `source`에는 `a`가 없으므로 덮어쓰지
    않습니다.
    
    `target.b`는 `source.b`가 복사되어 `2`가 됩니다.
    
    중요한 부분은 `nested`입니다. `source`에도 `nested`가 있으므로 `target.nested`는
    `source.nested`가 가리키는 객체 참조로 덮어써집니다.
    
    그 뒤 `result.nested.count = 20`을 실행하면 `target.nested`와 `source.nested`가 같은
    객체를 가리키고 있으므로 `source.nested.count`도 `20`이 됩니다.
    
    `Object.assign`도 전개 구문과 마찬가지로 얕은 복사입니다. 또한 첫 번째 인자를 직접
    수정한다는 점 때문에, 원본 보존이 필요하다면 `Object.assign({}, target, source)`처럼 빈
    객체를 target으로 두거나 전개 구문을 사용하는 방식이 더 명확합니다.
    

### 문제 4. 심화

다음 코드는 상태 객체의 도시 이름을 바꾸려고 합니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
const state = {
  user: {
    name: "Ada",
    address: { city: "Seoul" },
  },
  count: 0,
};

function renameCityBad(nextCity) {
  const next = { ...state };
  next.user.address.city = nextCity;
  return next;
}

const nextBad = renameCityBad("Busan");

console.log(state.user.address.city);
console.log(nextBad.user.address.city);
console.log(state.user === nextBad.user);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Busan
    Busan
    true
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type State = {
      user: {
        name: string;
        address: {
          city: string;
        };
      };
      count: number;
    };
    
    function renameCity(state: Readonly<State>, nextCity: string): State {
      return {
        ...state,
        user: {
          ...state.user,
          address: {
            ...state.user.address,
            city: nextCity,
          },
        },
      };
    }
    
    const state: State = {
      user: { name: "Ada", address: { city: "Seoul" } },
      count: 0,
    };
    
    const next = renameCity(state, "Busan");
    
    console.log(state.user.address.city);
    console.log(next.user.address.city);
    console.log(state.user === next.user);
    console.log(state.user.address === next.user.address);
    ```
    
    ### 해설
    
    ### JS 관점
    
    `renameCityBad` 안의 `const next = { ...state }`는 최상위 객체만 새로 만듭니다.
    `next`와 `state`는 서로 다른 객체이지만, `next.user`와 `state.user`는 같은 객체를
    가리킵니다.
    
    따라서 `next.user.address.city = nextCity`는 새 상태만 바꾸는 것이 아닙니다. 공유된
    `user` 객체 안의 `address.city`를 바꾸기 때문에 원본 `state.user.address.city`도
    `"Busan"`으로 바뀝니다.
    
    그래서 첫 번째와 두 번째 출력이 모두 `"Busan"`입니다.
    
    `state.user === nextBad.user`는 `true`입니다. 전개 구문이 최상위 속성의 참조를 그대로
    복사했기 때문입니다.
    
    이런 코드는 React 상태 업데이트에서 특히 위험합니다. 겉보기에는 새 객체를 반환하는
    것처럼 보이지만, 중첩 객체를 공유해서 이전 상태가 함께 바뀔 수 있습니다.
    
    ### TS 관점
    
    TypeScript로 `State` 타입을 정의해도 전개 구문이 깊은 복사를 수행하는지는 자동으로
    보장되지 않습니다. 타입은 객체의 구조를 설명하지만, 런타임 참조 공유 여부를 추적하지
    않습니다.
    
    개선 예시에서는 변경 경로에 있는 객체를 모두 새로 만듭니다. 최상위 `state`, 그 안의
    `user`, 그 안의 `address`까지 전개해서 새 객체를 만든 뒤 `city`만 바꿉니다.
    
    `Readonly<State>`는 함수가 받은 최상위 `state`에 직접 대입하지 않도록 돕습니다. 하지만
    `Readonly`는 기본적으로 얕은 readonly입니다. 중첩 객체까지 컴파일 타임에서 강하게
    막으려면 `DeepReadonly` 같은 타입을 추가로 고려해야 합니다.
    
    이 예시의 핵심은 타입 하나로 복사 깊이를 해결하려는 것이 아니라, 업데이트 함수의
    구조에서 어떤 경로를 새로 만들어야 하는지 명시하는 것입니다.
    
    ### 실무 관점
    
    중첩 상태를 업데이트할 때는 “어느 객체가 새로 만들어졌고, 어느 객체가 공유되는가”를
    확인해야 합니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 변경하려는 값까지의 경로를 찾는다.
    2. 그 경로에 있는 객체를 모두 새로 만든다.
    3. 변경하지 않는 형제 속성은 전개 구문으로 유지한다.
    4. 원본 상태를 직접 바꾸지 않았는지 테스트한다.
    
    상태 구조가 너무 깊어서 매번 이런 업데이트가 어렵다면, 상태를 더 평평하게 만들거나
    Immer 같은 불변 업데이트 도구를 사용하는 것도 현실적인 선택입니다.
