# Day 27. 이벤트 전파와 위임

### 오늘의 목표

브라우저 이벤트는 한 요소에서만 끝나지 않고 캡처링, 타깃, 버블링 단계를 거쳐 전달될 수
있습니다. 오늘은 이벤트 전파 순서, `stopPropagation`, 이벤트 위임, `dataset`, `closest`를
복습하고, TypeScript에서 위임 이벤트의 `target`을 안전하게 좁히는 방법을 정리합니다.

### 레퍼런스 연결

- 이벤트 챕터 복습
- 핵심 복습 포인트: capturing, bubbling, propagation, delegation, `closest`
- 다시 볼 질문: 이벤트 위임은 언제 유지보수에 도움이 되고 언제 디버깅을 어렵게 할까?

### 문제 1. 기본

다음 코드는 DOM 이벤트 전파 순서를 단순화해 배열로 표현합니다. 실행 결과를 작성하세요.

```jsx
function dispatchPath(path) {
  const logs = [];

  for (const node of path) {
    logs.push(`capture:${node}`);
  }

  for (const node of [...path].reverse()) {
    logs.push(`bubble:${node}`);
  }

  return logs;
}

console.log(dispatchPath(["root", "parent", "button"]).join("|"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    capture:root|capture:parent|capture:button|bubble:button|bubble:parent|bubble:root
    ```
    
    ### 해설
    
    브라우저 DOM 이벤트는 보통 캡처링 단계에서 바깥 요소부터 안쪽 요소로 내려오고, 버블링
    단계에서 타깃 요소부터 다시 바깥 요소로 올라갑니다.
    
    이 예제의 `path`는 `root -> parent -> button` 순서입니다. 첫 번째 반복문은 이 순서대로
    `capture:*` 로그를 추가합니다.
    
    두 번째 반복문은 `path`를 복사한 뒤 뒤집어서 `button -> parent -> root` 순서로 `bubble:*`
    로그를 추가합니다.
    
    실제 DOM 이벤트에는 타깃 단계와 캡처/버블 리스너 옵션이 더 있지만, 기본적인 방향은 이 예제와
    같습니다. 이벤트 위임을 이해하려면 이벤트가 자식에서 부모로 올라올 수 있다는 점을 알아야 합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `dataset` 기반 이벤트 위임이 어떻게 동작하는지
설명하세요.

```jsx
function delegate(event) {
  if (event.target.dataset.action === "delete") {
    return `delete:${event.target.dataset.id}`;
  }

  return "ignore";
}

console.log(delegate({ target: { dataset: { action: "delete", id: "7" } } }));
console.log(delegate({ target: { dataset: { action: "edit", id: "7" } } }));
```

- 정답과 해설
    
    ### 정답
    
    ```
    delete:7
    ignore
    ```
    
    ### 해설
    
    첫 번째 호출에서 `event.target.dataset.action`은 `"delete"`입니다. 조건을 만족하므로
    `dataset.id`인 `"7"`과 함께 `"delete:7"`을 반환합니다.
    
    두 번째 호출에서 `action`은 `"edit"`입니다. 삭제 액션이 아니므로 `"ignore"`를 반환합니다.
    
    브라우저에서는 여러 버튼에 각각 리스너를 붙이는 대신, 부모 요소 하나에 클릭 리스너를 붙이고
    `event.target.dataset.action`을 확인하는 방식으로 이벤트 위임을 구현할 수 있습니다.
    
    이 방식은 동적으로 추가되는 요소에도 대응하기 쉽지만, `event.target`이 항상 원하는 요소라고
    가정하면 위험합니다. 아이콘, span, 텍스트 노드 등 실제 클릭된 대상이 버튼 내부의 다른 노드일
    수 있기 때문입니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 작성하고, `stopPropagation`이 어떤 의미인지 설명하세요.

```jsx
function createEvent() {
  return {
    stopped: false,
    stopPropagation() {
      this.stopped = true;
    },
  };
}

function runHandlers(event) {
  const logs = [];

  logs.push("child");
  event.stopPropagation();

  if (!event.stopped) {
    logs.push("parent");
  }

  return logs;
}

console.log(runHandlers(createEvent()).join("|"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    child
    ```
    
    ### 해설
    
    `runHandlers`는 자식 요소의 핸들러가 먼저 실행되는 상황을 단순화한 예제입니다.
    
    먼저 `logs.push("child")`가 실행됩니다.
    
    그 다음 `event.stopPropagation()`이 호출되어 `event.stopped`가 `true`가 됩니다. 실제 DOM의
    `stopPropagation`은 이벤트가 다음 전파 대상의 리스너로 계속 전달되는 것을 막습니다.
    
    예제에서는 `if (!event.stopped)` 조건으로 부모 핸들러 실행 여부를 표현했습니다. 이미
    `stopped`가 `true`이므로 `"parent"`는 추가되지 않습니다.
    
    따라서 결과는 `"child"`입니다.
    
    `stopPropagation`은 문제를 빠르게 막아 주는 도구처럼 보이지만, 부모에서 이벤트 위임을 하고
    있다면 부모 핸들러가 실행되지 않아 기능이 깨질 수 있습니다. 전파를 끊기 전에 정말 필요한지
    확인해야 합니다.
    

### 문제 4. 심화

다음 코드는 테이블 클릭 이벤트에서 행 정보를 읽으려고 합니다. 실행 결과를 예측하고, 이 코드의
의미를 JS 관점과 TS 관점에서 설명한 뒤 브라우저 DOM에서 사용할 수 있는 TypeScript 코드로
개선하세요.

```jsx
const rows = [
  { dataset: { id: "1" }, textContent: "Ada" },
  { dataset: { id: "2" }, textContent: "Lin" },
];

function handleClick(event) {
  const row = event.target.closest("[data-id]");

  if (!row) {
    return "ignore";
  }

  return `${row.dataset.id}:${row.textContent}`;
}

console.log(handleClick({ target: { closest: () => rows[0] } }));
console.log(handleClick({ target: { closest: () => null } }));
```

- 정답과 해설
    
    ### 정답
    
    ```
    1:Ada
    ignore
    ```
    
    브라우저 DOM에서 사용할 수 있는 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    function isHTMLElement(target: EventTarget | null): target is HTMLElement {
      return target instanceof HTMLElement;
    }
    
    function findRow(target: EventTarget | null): HTMLElement | null {
      if (!isHTMLElement(target)) {
        return null;
      }
    
      return target.closest<HTMLElement>("[data-id]");
    }
    
    function handleTableClick(event: Event): string | null {
      const row = findRow(event.target);
    
      if (!row) {
        return null;
      }
    
      const id = row.dataset.id;
    
      if (!id) {
        return null;
      }
    
      return `${id}:${row.textContent ?? ""}`;
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    첫 번째 호출에서 `event.target.closest("[data-id]")`는 `rows[0]`을 반환합니다. `rows[0]`의
    `dataset.id`는 `"1"`이고 `textContent`는 `"Ada"`입니다. 그래서 `"1:Ada"`가 반환됩니다.
    
    두 번째 호출에서 `closest`는 `null`을 반환합니다. 이 경우 처리할 행을 찾지 못한 것이므로
    `"ignore"`가 반환됩니다.
    
    실제 DOM 이벤트 위임에서는 사용자가 클릭한 요소가 행 자체가 아닐 수 있습니다. 행 안의 버튼,
    아이콘, 텍스트 등이 `event.target`이 될 수 있으므로 `closest("[data-id]")`로 가까운 행을 찾는
    패턴이 자주 사용됩니다.
    
    ### TS 관점
    
    TypeScript에서 `event.target`은 `EventTarget | null`입니다. `EventTarget`에는 `closest`나
    `dataset` 속성이 없습니다.
    
    그래서 먼저 `target instanceof HTMLElement`로 좁혀야 합니다. `isHTMLElement`는 런타임 검사와
    타입 가드를 동시에 수행합니다.
    
    `target.closest<HTMLElement>("[data-id]")`는 조건에 맞는 가장 가까운 요소를 찾습니다. 찾지
    못할 수 있으므로 반환 타입은 `HTMLElement | null`입니다.
    
    `dataset.id`도 항상 있다고 가정하면 안 됩니다. 속성이 없으면 `undefined`일 수 있으므로, 예시처럼
    `if (!id)`로 확인한 뒤 사용합니다.
    
    ### 실무 관점
    
    이벤트 위임은 많은 자식 요소에 각각 리스너를 붙이지 않아도 되므로 성능과 유지보수 면에서
    유용합니다. 특히 동적으로 추가되는 목록, 테이블, 메뉴에 잘 맞습니다.
    
    하지만 다음을 조심해야 합니다.
    
    1. `event.target`이 원하는 요소가 아닐 수 있다.
    2. 내부 요소 클릭까지 고려해 `closest`가 필요할 수 있다.
    3. `dataset` 값은 문자열이며 없을 수도 있다.
    4. 자식에서 `stopPropagation`을 호출하면 부모 위임 핸들러가 실행되지 않을 수 있다.
    
    좋은 이벤트 위임 코드는 “어떤 요소를 처리 대상으로 볼 것인가”를 selector와 타입 가드로
    명확히 표현합니다.
