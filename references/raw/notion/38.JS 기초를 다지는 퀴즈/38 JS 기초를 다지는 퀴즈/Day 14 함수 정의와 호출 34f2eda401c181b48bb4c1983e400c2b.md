# Day 14. 함수 정의와 호출

### 오늘의 목표

JavaScript 함수는 선언된 매개변수 개수와 실제 전달된 인자 개수가 달라도 호출됩니다.
오늘은 기본 매개변수, `arguments`, rest parameter, spread 호출을 복습하고, TypeScript의
함수 시그니처가 어디까지 호출 실수를 줄여 주는지 확인합니다.

### 레퍼런스 연결

- 함수 챕터 복습
- 핵심 복습 포인트: 함수 선언문, 함수 표현식, 화살표 함수, 매개변수와 인수
- 다시 볼 질문: JS는 인자 수가 달라도 호출되는데 TS에서는 어떻게 표현해야 할까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
function greet(name, punctuation = "!") {
  return `Hello,${name}${punctuation}`;
}

console.log(greet("Ada"));
console.log(greet("Ada", "?"));
console.log(greet());
console.log(greet("Ada", undefined));
console.log(greet("Ada", null));
```

- 정답과 해설
    
    ### 정답
    
    ```
    Hello, Ada!
    Hello, Ada?
    Hello, undefined!
    Hello, Ada!
    Hello, Adanull
    ```
    
    ### 해설
    
    `greet("Ada")`는 첫 번째 인자로 `"Ada"`를 받고, 두 번째 인자 `punctuation`은 생략됩니다.
    기본값 `"!"`가 적용되어 `"Hello, Ada!"`가 출력됩니다.
    
    `greet("Ada", "?")`는 두 번째 인자가 명시적으로 전달되었으므로 기본값을 사용하지 않습니다.
    결과는 `"Hello, Ada?"`입니다.
    
    `greet()`는 `name`도 전달하지 않았습니다. JavaScript에서 전달되지 않은 매개변수의 값은
    `undefined`입니다. `punctuation`은 생략되었으므로 기본값 `"!"`가 적용됩니다. 템플릿
    문자열에서 `undefined`는 문자열 `"undefined"`로 변환되어 `"Hello, undefined!"`가 됩니다.
    
    `greet("Ada", undefined)`는 두 번째 인자를 명시적으로 `undefined`로 전달한 경우입니다.
    기본 매개변수는 값이 `undefined`일 때 적용되므로 결과는 `"Hello, Ada!"`입니다.
    
    `greet("Ada", null)`은 다릅니다. `null`은 `undefined`가 아니므로 기본값이 적용되지
    않습니다. 템플릿 문자열에서 `null`은 `"null"`로 변환되어 `"Hello, Adanull"`이 됩니다.
    
    기본 매개변수는 “인자가 없거나 `undefined`일 때” 적용됩니다. `null`, `false`, `0`, `""`는
    기본값으로 대체되지 않습니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 선언된 매개변수 개수와 실제 인자 개수의 차이를
설명하세요.

```jsx
function logArgs(a, b) {
  console.log(arguments.length);
  console.log(a);
  console.log(b);
}

logArgs(1);
logArgs(1, 2, 3);
```

- 정답과 해설
    
    ### 정답
    
    ```
    1
    1
    undefined
    3
    1
    2
    ```
    
    ### 해설
    
    첫 번째 호출 `logArgs(1)`은 실제 인자를 하나만 전달합니다. `arguments.length`는 실제로
    전달된 인자 개수이므로 `1`입니다.
    
    매개변수 `a`는 첫 번째 인자 `1`을 받습니다. 매개변수 `b`는 전달된 값이 없으므로
    `undefined`입니다.
    
    두 번째 호출 `logArgs(1, 2, 3)`은 실제 인자를 세 개 전달합니다. 함수 선언에는 매개변수
    `a`, `b`만 있지만 JavaScript는 초과 인자를 이유로 에러를 내지 않습니다.
    `arguments.length`는 `3`입니다.
    
    `a`는 `1`, `b`는 `2`를 받습니다. 세 번째 인자 `3`은 이름 붙은 매개변수로는 받지 않지만,
    `arguments[2]`로 접근할 수 있습니다.
    
    현대 코드에서는 `arguments`보다 rest parameter를 쓰는 편이 명확합니다. `arguments`는 배열이
    아닌 유사 배열 객체이고, 화살표 함수에는 자체 `arguments`가 없기 때문입니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, rest parameter와 spread 호출의 차이를
설명하세요.

```jsx
function sum(...numbers) {
  return numbers.reduce((total, value) => total + value, 0);
}

console.log(sum());
console.log(sum(1, 2, 3));

const values = [4, 5];

console.log(sum(...values));
```

- 정답과 해설
    
    ### 정답
    
    ```
    0
    6
    9
    ```
    
    ### 해설
    
    `...numbers`는 rest parameter입니다. 함수 호출 시 전달된 나머지 인자들을 배열로 모읍니다.
    
    `sum()`은 전달된 인자가 없으므로 `numbers`가 빈 배열입니다. `reduce`의 초기값이 `0`이므로
    결과는 `0`입니다.
    
    `sum(1, 2, 3)`에서는 `numbers`가 `[1, 2, 3]`이 됩니다. 합계는 `6`입니다.
    
    `sum(...values)`의 `...values`는 호출 위치에서 쓰인 spread 문법입니다. 배열 `[4, 5]`를
    개별 인자 `4`, `5`로 펼쳐서 전달합니다. 함수 내부에서는 다시 rest parameter가 이 인자들을
    `[4, 5]` 배열로 모읍니다. 합계는 `9`입니다.
    
    같은 `...` 문법이라도 함수 선언 위치에서는 rest, 호출 위치에서는 spread로 읽어야 합니다.
    

### 문제 4. 심화

다음 코드는 사용자 객체를 만들려고 작성되었습니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function createUser(name, age) {
  return { name, age, isAdult: age >= 18 };
}

console.log(JSON.stringify(createUser("Ada", 20)));
console.log(JSON.stringify(createUser("Lin")));
console.log(JSON.stringify(createUser("Ken", "20")));
```

- 정답과 해설
    
    ### 정답
    
    ```
    {"name":"Ada","age":20,"isAdult":true}
    {"name":"Lin","isAdult":false}
    {"name":"Ken","age":"20","isAdult":true}
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = {
      name: string;
      age: number;
      isAdult: boolean;
    };
    
    function createUser(name: string, age: number): User {
      if (!Number.isInteger(age) || age < 0) {
        throw new Error("age must be a non-negative integer.");
      }
    
      return {
        name,
        age,
        isAdult: age >= 18,
      };
    }
    
    console.log(JSON.stringify(createUser("Ada", 20)));
    
    try {
      console.log(JSON.stringify(createUser("Lin", Number.NaN)));
    } catch (error) {
      console.log(error instanceof Error ? error.message : "unknown error");
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    `createUser("Ada", 20)`은 자연스럽게 동작합니다. `name`은 `"Ada"`, `age`는 `20`이고,
    `20 >= 18`은 `true`이므로 `isAdult`도 `true`입니다.
    
    `createUser("Lin")`은 `age` 인자를 전달하지 않았습니다. JavaScript에서 누락된 매개변수는
    `undefined`입니다. `undefined >= 18`은 비교 과정에서 숫자 변환을 시도하고, 결과적으로
    `false`가 됩니다. 객체에는 `age: undefined`가 들어 있지만, `JSON.stringify`는 값이
    `undefined`인 객체 속성을 생략합니다. 그래서 출력에는 `age`가 없습니다.
    
    `createUser("Ken", "20")`은 문자열 `"20"`을 나이로 전달합니다. 비교 연산 `>=`는 숫자 비교를
    위해 문자열을 숫자 `20`으로 변환합니다. 그래서 `isAdult`는 `true`입니다. 하지만 반환 객체의
    `age` 값 자체는 문자열 `"20"`으로 남습니다.
    
    즉, 함수 호출은 성공했지만 데이터 모양이 일관되지 않습니다. 어떤 사용자는 `age`가 숫자이고,
    어떤 사용자는 없고, 어떤 사용자는 문자열입니다.
    
    ### TS 관점
    
    TypeScript에서 `createUser(name: string, age: number)`처럼 함수 시그니처를 선언하면
    `createUser("Lin")`처럼 필수 인자를 빠뜨리거나 `createUser("Ken", "20")`처럼 문자열을 넘기는
    호출을 컴파일 단계에서 잡을 수 있습니다.
    
    하지만 타입이 `number`라고 해서 모든 숫자가 도메인에 맞는 것은 아닙니다. `NaN`, `Infinity`,
    음수 같은 값도 타입상으로는 `number`입니다.
    
    그래서 개선 예시에서는 `Number.isInteger(age)`와 `age < 0` 검사를 추가했습니다. 필수 인자와
    기본 타입은 TypeScript가 잡고, 도메인 규칙은 런타임 검증으로 확인하는 구조입니다.
    
    ### 실무 관점
    
    함수 시그니처는 API 계약입니다. JavaScript만 사용할 때는 호출자가 인자를 빼먹거나 타입이
    다른 값을 넘겨도 런타임까지 가야 문제가 드러나는 경우가 많습니다.
    
    TypeScript를 사용하면 필수 인자, 선택 인자, 기본값, rest parameter 같은 계약을 코드에
    명확히 표현할 수 있습니다. 하지만 외부 입력, JSON 파싱 결과, URL 파라미터처럼 런타임에서
    들어오는 값은 여전히 검증해야 합니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 함수 시그니처로 필수 인자와 타입을 명확히 표현한다.
    2. 선택 가능한 값은 `?` 또는 기본값으로 의도를 드러낸다.
    3. 외부 입력은 함수 안 또는 함수 호출 전에 런타임 검증을 거친다.
    4. 반환 타입을 명시해 데이터 모양이 흔들리지 않게 한다.

### 오늘의 생각 질문

특성을 얼마나 신경 쓰지 않아도 될까요? 타입으로 막을 수 있는 호출 실수와 런타임 검증이
필요한 값 검증을 어떻게 나눌 수 있을까요?
