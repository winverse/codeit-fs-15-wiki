# Day 4. 숫자와 문자열의 함정

### 오늘의 목표

JavaScript의 `number`는 정수와 실수를 하나의 타입으로 다루지만, 내부적으로는
IEEE 754 부동소수점 표현을 사용합니다. 오늘은 소수 계산 오차, 안전한 정수 범위,
`BigInt`, 문자열 길이와 UTF-16 표현을 복습하고, 금액 계산처럼 실무에서 오차가
문제가 되는 코드를 TypeScript 관점과 함께 정리합니다.

### 레퍼런스 연결

- 타입 챕터 복습
- 핵심 복습 포인트: 부동소수점, 안전한 정수, `BigInt`, 문자열 길이
- 다시 볼 질문: 금액, 포인트, 카운트 값은 어떤 타입과 정책으로 다뤄야 할까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
console.log(0.1 + 0.2);
console.log(0.1 + 0.2 === 0.3);
console.log(Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2);
console.log(Number.MAX_SAFE_INTEGER + 1);
console.log(Number.MAX_SAFE_INTEGER + 2);
```

- 정답과 해설
    
    ### 정답
    
    ```
    0.30000000000000004
    false
    true
    9007199254740992
    9007199254740992
    ```
    
    ### 해설
    
    JavaScript의 `number`는 대부분의 일반적인 숫자 계산을 처리할 수 있지만, 모든
    소수를 정확하게 표현하지는 못합니다. `0.1`과 `0.2`는 2진 부동소수점으로 정확히
    표현되지 않습니다. 그래서 두 값을 더하면 수학적으로 기대하는 `0.3`이 아니라
    `0.30000000000000004`가 나옵니다.
    
    두 번째 줄의 `0.1 + 0.2 === 0.3`은 `false`입니다. 왼쪽 값이 정확히 `0.3`이
    아니기 때문입니다. 소수 계산 결과를 비교할 때는 보통 `===`로 직접 비교하지 않고,
    허용 가능한 오차 범위를 두거나 정수 단위로 바꿔 계산합니다.
    
    `Number.MAX_SAFE_INTEGER`는 JavaScript의 `number`가 정수로 안전하게 표현할 수
    있는 최대값입니다. 값은 `9007199254740991`입니다. 이 값을 넘어가면 모든 정수를
    정확하게 구분할 수 없습니다.
    
    그래서 `Number.MAX_SAFE_INTEGER + 1`과 `Number.MAX_SAFE_INTEGER + 2`가 둘 다
    `9007199254740992`로 표현됩니다. 서로 다른 계산인데 같은 값처럼 보입니다.
    `Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2`가 `true`가 되는
    이유가 바로 이 정밀도 한계입니다.
    
    실무에서 ID, 주문 번호, 포인트, 금액처럼 정확해야 하는 값이 안전한 정수 범위를
    넘을 가능성이 있다면 `number`만으로 처리해도 되는지 반드시 확인해야 합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `number`와 `bigint`를 섞어 계산할 수
없는 이유를 설명하세요.

```jsx
const max = Number.MAX_SAFE_INTEGER;

console.log(max);
console.log(max + 1 === max + 2);

const big = BigInt(max);

console.log(big + 1n === big + 2n);

try {
  console.log(1n + 1);
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    9007199254740991
    true
    false
    TypeError
    ```
    
    ### 해설
    
    `max`는 `Number.MAX_SAFE_INTEGER`이므로 `9007199254740991`입니다. 이 값은
    JavaScript의 `number`가 안전하게 다룰 수 있는 정수의 상한입니다.
    
    `max + 1 === max + 2`는 `true`입니다. 안전한 정수 범위를 넘어가면서 두 계산
    결과가 같은 `number` 값으로 표현되기 때문입니다. 정수처럼 보이는 값이어도
    `number`의 표현 범위를 넘어가면 정확한 비교가 깨질 수 있습니다.
    
    `BigInt(max)`는 `max`를 `bigint` 값으로 변환합니다. `bigint`는 임의 정밀도 정수를
    표현할 수 있으므로 `big + 1n`과 `big + 2n`은 서로 다른 값입니다. 따라서
    `big + 1n === big + 2n`은 `false`입니다.
    
    마지막 줄에서는 `1n + 1`을 시도합니다. `1n`은 `bigint`이고 `1`은 `number`입니다.
    JavaScript는 두 타입을 산술 연산에서 자동으로 섞어 주지 않습니다. 그래서
    `TypeError`가 발생합니다.
    
    이 제한은 불편해 보일 수 있지만, 오히려 안전장치입니다. `number`와 `bigint`를
    자동으로 섞으면 정밀도 손실이 조용히 발생할 수 있습니다. 개발자가 어느 쪽으로
    변환할지 명시해야 계산 의도가 분명해집니다.
    
    TypeScript에서도 `number`와 `bigint`는 다른 타입입니다. `1n + 1` 같은 코드는
    타입 단계에서도 오류로 잡히므로, 큰 정수를 다루는 코드는 처음부터 입력, 저장,
    계산, 출력 경로를 `bigint` 중심으로 설계하는 편이 좋습니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 문자열 길이를 계산할 때 주의할 점을
설명하세요.

```jsx
const icon = "💡";
const text = "A💡B";

console.log(icon.length);
console.log([...icon].length);
console.log(text.length);
console.log(Array.from(text).length);
```

- 정답과 해설
    
    ### 정답
    
    ```
    2
    1
    4
    3
    ```
    
    ### 해설
    
    JavaScript의 문자열은 UTF-16 코드 유닛의 시퀀스로 다뤄집니다. `length`는 사람이
    눈으로 보는 글자 수가 아니라 UTF-16 코드 유닛 수를 반환합니다.
    
    전구 이모지 `"💡"`는 화면에서는 한 글자처럼 보이지만 UTF-16 코드 유닛 두 개로
    표현됩니다. 그래서 `icon.length`는 `2`입니다.
    
    `[...icon]`은 문자열 이터레이터를 사용합니다. 문자열 이터레이터는 surrogate pair를
    하나의 코드 포인트로 묶어 순회하므로 `[...icon].length`는 `1`입니다.
    
    `"A💡B"`는 `A`가 코드 유닛 1개, `"💡"`가 코드 유닛 2개, `B`가 코드 유닛 1개입니다.
    따라서 `text.length`는 `4`입니다.
    
    `Array.from(text)`도 문자열 이터레이터를 사용하므로 `"A"`, `"💡"`, `"B"` 세 항목으로
    분리됩니다. 그래서 길이는 `3`입니다.
    
    실무에서 입력 글자 수 제한을 만들 때 단순히 `.length`를 쓰면 사용자가 보는 글자 수와
    다를 수 있습니다. 특히 이모지, 일부 기호, 조합 문자가 포함될 때 차이가 커집니다.
    정확한 사용자 인식 기준의 글자 수가 필요하다면 `Intl.Segmenter` 같은 API나 별도
    문자열 처리 정책을 검토해야 합니다.
    

### 문제 4. 심화

다음 코드는 상품 가격 목록의 합계를 계산합니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function sumPrices(prices) {
  return prices.reduce((total, price) => total + price, 0);
}

const total = sumPrices([0.1, 0.2, 0.3]);

console.log(total);
console.log(total === 0.6);
console.log(total.toFixed(2));
```

- 정답과 해설
    
    ### 정답
    
    ```
    0.6000000000000001
    false
    0.60
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type Cents = number & { readonly __brand: unique symbol };
    
    function parseCents(input: string): Cents | null {
      if (!/^[0-9]+(\.[0-9]{1,2})?$/.test(input)) {
        return null;
      }
    
      const [dollarPart, centPart = ""] = input.split(".");
      const centsText = `${dollarPart}${centPart.padEnd(2, "0")}`;
      const cents = Number(centsText);
    
      return Number.isSafeInteger(cents) ? (cents as Cents) : null;
    }
    
    function sumCents(values: Cents[]): Cents | null {
      const total = values.reduce((sum, value) => sum + value, 0);
      return Number.isSafeInteger(total) ? (total as Cents) : null;
    }
    
    function formatCents(value: Cents): string {
      return (value / 100).toFixed(2);
    }
    
    const prices = ["0.10", "0.20", "0.30"].map(parseCents);
    
    if (prices.every((price): price is Cents => price !== null)) {
      const total = sumCents(prices);
    
      if (total !== null) {
        console.log(total);
        console.log(formatCents(total));
      }
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    `sumPrices([0.1, 0.2, 0.3])`은 배열을 순회하면서 `0 + 0.1 + 0.2 + 0.3`을 계산합니다.
    하지만 `0.1`, `0.2`, `0.3` 같은 십진 소수는 2진 부동소수점으로 정확히 표현되지
    않습니다.
    
    그래서 최종 결과는 수학적으로 기대하는 `0.6`이 아니라 `0.6000000000000001`입니다.
    이 값은 `0.6`과 정확히 같지 않으므로 `total === 0.6`은 `false`입니다.
    
    `total.toFixed(2)`는 소수점 둘째 자리까지 문자열로 반올림해 `"0.60"`을 반환합니다.
    출력용으로는 그럴듯해 보이지만, 내부 계산값 자체가 정확해진 것은 아닙니다. 화면에
    보이는 값과 실제 비교에 쓰는 값이 달라질 수 있다는 점이 중요합니다.
    
    ### TS 관점
    
    TypeScript에서 `prices`를 `number[]`로 선언해도 소수 계산 오차는 막을 수 없습니다.
    `number` 타입은 “숫자”라는 사실만 표현할 뿐, 그 숫자가 금액인지, 센트 단위인지,
    부동소수점 오차가 없는 값인지는 보장하지 않습니다.
    
    개선 예시에서는 금액을 문자열로 입력받고, `parseCents`에서 소수점 이하 두 자리까지의
    형식만 허용합니다. `"0.10"`은 10센트, `"0.20"`은 20센트, `"0.30"`은 30센트로 변환되어
    정수 계산이 됩니다.
    
    `sumCents`는 합산 결과도 다시 `Number.isSafeInteger`로 확인합니다. 개별 입력이 안전한
    정수여도 여러 값을 더한 결과가 안전한 정수 범위를 넘을 수 있기 때문입니다. 합계가
    안전하지 않으면 `null`을 반환하게 해서, 호출자가 계산 실패를 명시적으로 처리하도록
    만듭니다.
    
    `type Cents = number & { readonly __brand: unique symbol }`는 브랜드 타입입니다.
    런타임에서는 그냥 `number`이지만, TypeScript 단계에서는 일반 `number`와 구분해
    사용하도록 돕습니다. 다만 브랜드 타입은 컴파일 타임 보조 장치일 뿐입니다. 실제로
    올바른 센트 값인지 확인하는 일은 `parseCents` 같은 런타임 검증 함수가 담당해야
    합니다.
    
    ### 실무 관점
    
    금액, 포인트, 수량처럼 정확성이 중요한 값은 가능하면 정수 단위로 저장하고 계산하는
    편이 안전합니다. 예를 들어 달러 대신 센트, 원화의 경우 원 단위 정수, 포인트는 최소
    단위 정수로 다루는 방식입니다.
    
    또한 계산용 값과 표시용 문자열을 분리해야 합니다. `toFixed(2)`는 화면 표시에는 유용할
    수 있지만, 계산 정확성을 보장하는 도구는 아닙니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 외부 입력은 문자열로 받는다.
    2. 허용할 숫자 형식을 명확히 검증한다.
    3. 계산 가능한 정수 단위로 변환한다.
    4. 내부 계산은 정수로 수행한다.
    5. 화면에 보여 줄 때만 원하는 형식의 문자열로 포맷한다.
    
    이렇게 하면 JavaScript의 부동소수점 표현 한계를 피하면서도 TypeScript로 값의 의미를
    조금 더 명확하게 표현할 수 있습니다.
