# Day 18. 화살표 함수와 this

### 오늘의 목표

화살표 함수는 자신만의 `this`를 만들지 않고, 정의된 위치의 `this`를 그대로 사용합니다.
오늘은 객체 메서드로서의 화살표 함수, 메서드 내부에서 반환한 화살표 함수, 콜백 안의
`this` 문제를 복습하고, 클래스 필드 화살표 함수가 콜백 전달에서 어떤 장단점을 갖는지
TypeScript와 함께 정리합니다.

### 레퍼런스 연결

- `this` 챕터 복습
- 핵심 복습 포인트: 화살표 함수의 lexical `this`, 콜백 `this`, 클래스 필드
- 다시 볼 질문: 화살표 함수는 항상 일반 함수보다 좋은 선택일까?

### 문제 1. 기본

다음 코드는 strict mode 함수 안에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하세요.

```jsx
(function () {
  "use strict";

  const user = {
    name: "Ada",
    regular() {
      return this.name;
    },
    arrow: () => this,
  };

  console.log(user.regular());
  console.log(user.arrow() === undefined);
})();
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    true
    ```
    
    ### 해설
    
    `user.regular()`는 일반 메서드 호출입니다. 호출식의 점 왼쪽 객체가 `user`이므로 `this`는
    `user`입니다. 따라서 `this.name`은 `"Ada"`입니다.
    
    `arrow: () => this`는 화살표 함수입니다. 화살표 함수는 호출될 때 새 `this`를 받지 않고,
    정의된 위치의 `this`를 캡처합니다.
    
    이 예제에서 객체 리터럴은 strict mode 일반 함수 안에서 만들어졌고, 그 일반 함수는
    `(function () { ... })()`처럼 객체 없이 호출되었습니다. strict mode에서 이 함수 내부의
    `this`는 `undefined`입니다.
    
    따라서 `user.arrow()`처럼 객체의 속성으로 호출해도 화살표 함수의 `this`는 `user`가 아니라
    이미 캡처된 `undefined`입니다. 그래서 `user.arrow() === undefined`는 `true`입니다.
    
    객체 메서드가 `this`를 사용해야 한다면 화살표 함수보다 메서드 축약 문법이나 일반 함수를
    사용하는 편이 맞습니다.
    

### 문제 2. 중급

다음 코드는 strict mode 함수 안에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하고,
메서드 안에서 만든 화살표 함수의 `this`가 어떻게 결정되는지 설명하세요.

```jsx
(function () {
  "use strict";

  const counter = {
    value: 1,
    regular() {
      return this.value;
    },
    makeArrow() {
      return () => this.value;
    },
  };

  const arrow = counter.makeArrow();

  console.log(counter.regular());
  console.log(arrow());

  const other = { value: 10 };

  console.log(counter.regular.call(other));
  console.log(arrow.call(other));
})();
```

- 정답과 해설
    
    ### 정답
    
    ```
    1
    1
    10
    1
    ```
    
    ### 해설
    
    `counter.regular()`는 메서드 호출이므로 `this`가 `counter`입니다. `counter.value`는 `1`이므로
    첫 번째 출력은 `1`입니다.
    
    `counter.makeArrow()`도 메서드 호출입니다. 이 호출 안에서 `makeArrow`의 `this`는 `counter`입니다.
    그리고 `makeArrow`는 `() => this.value`라는 화살표 함수를 반환합니다.
    
    반환된 화살표 함수는 자신이 만들어진 시점의 `this`, 즉 `counter`를 캡처합니다. 그래서
    `arrow()`를 일반 함수처럼 호출해도 `this`는 계속 `counter`이고 결과는 `1`입니다.
    
    `counter.regular.call(other)`는 일반 함수의 `this`를 `other`로 바꿔 호출합니다. 그래서
    `other.value`인 `10`이 출력됩니다.
    
    하지만 `arrow.call(other)`는 결과가 바뀌지 않습니다. 화살표 함수는 `call`, `apply`, `bind`로
    `this`를 새로 바꿀 수 없습니다. 이미 캡처된 `counter`의 `this`를 사용하므로 마지막 출력은
    `1`입니다.
    

### 문제 3. 중급

다음 코드는 strict mode 함수 안에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하고,
배열 콜백에서 화살표 함수를 쓰면 어떤 차이가 생기는지 설명하세요.

```jsx
(function () {
  "use strict";

  const user = {
    name: "Ada",
    tags: ["js", "ts"],
    printWithRegular() {
      return this.tags.map(function (tag) {
        try {
          return `${this.name}:${tag}`;
        } catch (error) {
          return error.constructor.name;
        }
      }).join("|");
    },
    printWithArrow() {
      return this.tags.map((tag) => `${this.name}:${tag}`).join("|");
    },
  };

  console.log(user.printWithRegular());
  console.log(user.printWithArrow());
})();
```

- 정답과 해설
    
    ### 정답
    
    ```
    TypeError|TypeError
    Ada:js|Ada:ts
    ```
    
    ### 해설
    
    `user.printWithRegular()`는 메서드 호출이므로 `printWithRegular` 안의 `this`는 `user`입니다.
    따라서 `this.tags`에는 접근할 수 있습니다.
    
    하지만 `map(function (tag) { ... })`에 전달한 콜백은 일반 함수입니다. `map`이 콜백을 호출할
    때 이 콜백의 `this`를 `user`로 자동 지정하지 않습니다. strict mode에서 콜백 내부의 `this`는
    `undefined`입니다.
    
    그래서 콜백 안의 `this.name`을 읽으면 `TypeError`가 발생합니다. 예제는 각 콜백 내부에서
    에러를 잡아 에러 이름을 반환하므로 `"TypeError|TypeError"`가 출력됩니다.
    
    `printWithArrow`에서는 `map((tag) => ...)`처럼 화살표 함수를 사용합니다. 화살표 함수는
    자신만의 `this`를 만들지 않고, 바깥 메서드 `printWithArrow`의 `this`를 사용합니다. 바깥
    메서드의 `this`는 `user`이므로 결과는 `"Ada:js|Ada:ts"`입니다.
    
    배열 콜백 안에서 바깥 메서드의 `this`를 그대로 사용해야 한다면 화살표 함수가 자연스럽습니다.
    다만 콜백이 자체적인 `this`를 가져야 하는 API에서는 화살표 함수가 오히려 맞지 않을 수 있습니다.
    

### 문제 4. 심화

다음 코드는 strict mode 함수 안에서 실행한다고 가정합니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
(function () {
  "use strict";

  function createTimer(label) {
    return {
      label,
      ticks: 0,
      tickRegular() {
        this.ticks += 1;
        return `${this.label}:${this.ticks}`;
      },
      tickArrow: () => {
        this.ticks += 1;
        return `${this.label}:${this.ticks}`;
      },
    };
  }

  const timer = createTimer("main");

  console.log(timer.tickRegular());

  try {
    console.log(timer.tickArrow());
  } catch (error) {
    console.log(error.constructor.name);
  }

  console.log(timer.ticks);
})();
```

- 정답과 해설
    
    ### 정답
    
    ```
    main:1
    TypeError
    1
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    class Timer {
      readonly label: string;
      ticks = 0;
    
      constructor(label: string) {
        this.label = label;
      }
    
      tickRegular(): string {
        this.ticks += 1;
        return `${this.label}:${this.ticks}`;
      }
    
      tickArrow = (): string => {
        this.ticks += 1;
        return `${this.label}:${this.ticks}`;
      };
    }
    
    function runTwice(callback: () => string): string[] {
      return [callback(), callback()];
    }
    
    const timer = new Timer("main");
    
    console.log(runTwice(timer.tickArrow).join(","));
    console.log(timer.ticks);
    ```
    
    ### 해설
    
    ### JS 관점
    
    `createTimer("main")`은 일반 함수 호출입니다. 이 예제는 strict mode 함수 안에서 실행되므로
    `createTimer` 내부의 `this`는 `undefined`입니다.
    
    `tickRegular`는 일반 메서드입니다. `timer.tickRegular()`처럼 호출하면 `this`는 `timer`가
    됩니다. 그래서 `ticks`가 `0`에서 `1`로 증가하고 `"main:1"`이 출력됩니다.
    
    반면 `tickArrow`는 객체 리터럴 안에 있지만 화살표 함수입니다. 화살표 함수는 객체를 `this`로
    받지 않습니다. 정의된 위치의 `this`를 캡처합니다. 여기서는 `createTimer` 내부의 `this`가
    `undefined`이므로, `tickArrow` 안의 `this`도 `undefined`입니다.
    
    따라서 `this.ticks += 1`에서 `TypeError`가 발생합니다. 이 실패는 `timer.ticks`를 바꾸지
    못하므로 마지막 출력은 여전히 `1`입니다.
    
    ### TS 관점
    
    클래스에서 콜백으로 넘길 메서드가 `this`를 잃지 않아야 한다면 클래스 필드 화살표 함수를
    사용할 수 있습니다. 개선 예시의 `tickArrow = (): string => { ... }`는 인스턴스가 생성될 때
    해당 인스턴스의 `this`를 캡처합니다.
    
    그래서 `runTwice(timer.tickArrow)`처럼 함수 값만 넘겨도 `this`가 유지됩니다. 실행 결과는
    `"main:1,main:2"`이고, `timer.ticks`는 `2`가 됩니다.
    
    다만 클래스 필드 화살표 함수는 인스턴스마다 새 함수가 만들어집니다. 프로토타입 메서드보다
    메모리 사용 측면에서 불리할 수 있습니다. 모든 메서드를 화살표 함수로 만드는 것이 아니라,
    콜백으로 자주 전달되어 `this` 유지가 중요한 메서드에 선택적으로 사용하는 편이 좋습니다.
    
    ### 실무 관점
    
    화살표 함수는 `this` 문제를 해결하는 도구가 될 수 있지만, 모든 상황에서 메서드 대체재는
    아닙니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 객체 리터럴의 메서드가 `this`를 써야 한다면 일반 메서드 문법을 사용한다.
    2. 메서드 내부 콜백에서 바깥 `this`가 필요하면 화살표 콜백을 사용한다.
    3. 클래스 메서드를 콜백으로 자주 넘긴다면 클래스 필드 화살표 함수 또는 명시적 `bind`를 고려한다.
    4. 성능이나 메모리 비용이 중요한 곳에서는 프로토타입 메서드와 바인딩 전략을 비교한다.
    
    화살표 함수의 핵심은 “짧은 함수 문법”이 아니라 “lexical this”입니다. 이 차이를 이해해야
    어디에 써야 할지 결정할 수 있습니다.

콜백, 클래스 필드 함수에 각각 어떤 기준으로 일반 함수와 화살표 함수를 선택하면 좋을까요?
