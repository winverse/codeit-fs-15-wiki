## 2. 백엔드 DB 모델링

- **Entity 정의**
    - 서비스에 등장하는 “명사”를 뽑아 테이블 후보를 정합니다.
    - 산출물: `Study`, `Habit`, `User` … 목록
- **Attribute 정의**
    - 각 Entity의 컬럼(필드)과 타입을 정합니다.
    - 산출물: `id`, `title`, `createdAt` 같은 필드 표 + 필수/선택 여부
- **Relations 정의**
    - Entity 간 연결을 정합니다. (예: Study ↔ User, Habit ↔ User)
    - 산출물: 어떤 FK로 연결할지, 중간 테이블이 필요한지
- **Cardinality(카디널리티) 정의**
    - 관계의 개수 규칙을 정합니다.
    - 1:1 / 1:N / N:M 중 무엇인지 확정
    - 산출물: ERD(간단 그림) + 관계 표기(예: User 1 : N Study)

## 5. 개발 규칙 정하기

- **코드 포맷/린트**
    - Prettier: 저장 시 자동 포맷(팀 공통 설정)
    - ESLint: `npm lint`로 오류 체크 (PR 전에 필수)
- **네이밍/컨벤션**
    - 컴포넌트: `PascalCase` (`StudyCard`)
    - 함수/변수: `camelCase` (`fetchStudies`), 동사가 먼저오기
    - 파일/폴더: 팀에서 하나로 통일 (예:  `camelCase`)
- **Git 규칙**
    - 브랜치: `feature/<기능명>`, `fix/<이슈>`
    - 커밋 메시지: `feat:`, `fix:`, `refactor:` 정도만 사용
        - 자세한 커밋 메시지 규칙: [링크](https://velog.io/@chojs28/Git-%EC%BB%A4%EB%B0%8B-%EB%A9%94%EC%8B%9C%EC%A7%80-%EA%B7%9C%EC%B9%99)
    - PR: 작은 단위로 올리기 + 템플릿 사용 + 리뷰 1명 이상
- **폴더 구조**
    - “페이지/라우트”, “컴포넌트”, “api/서비스”, “공통 유틸” 위치를 합의

## 6. 프론트엔드 ↔ 백엔드 API 통신 과정

- **(1) API 계약(Interface) 먼저 합의**
    - 백엔드가 엔드포인트를 정리합니다. (예: `[GET] /users`, `[POST] /posts`)
    - **Request/Response 형식(필드, 타입, 에러 케이스)까지** 간단히 확정합니다.
- **(2) 계약을 공유/문서화**
    - Notion 표 또는 가능하다면 Swagger(OpenAPI)로 “초안”을 먼저 공유합니다.
- **(3) 프론트는 Mock 데이터로 먼저 구현**
    - 계약(Response shape)에 맞춰 화면을 완성합니다.
- **(4) 백엔드는 API를 구현하며 문서를 계속 업데이트**
    - 변경이 생기면 계약 문서에 바로 반영하고 이를 동료에게 알립니다.
- **(5) 프론트는 Mock → 실제 API로 교체**
    - `fetch/axios`로 실제 응답을 붙이고, 예외처리(로딩/에러)까지 연결합니다.

## 7. 개발 준비 시작

- **개발 기간 산정 & Task 작성**
    - 정확하지 않아도 OK. **기능 단위로 티켓을 쪼개는 게 목표입니다.**
- **개발 환경 세팅**
    - FE/BE 레포 세팅, 실행 방법, `.env.example`, 포트 규칙 정하기
- **개발 진행 방식**
    - 프론트: 계약 기반 Mock으로 화면 먼저
    - 백엔드: 계약 기반으로 API 구현
    - 합치는 날: Mock → 실제 API로 교체 + 통합 테스트

## 8. 어려움이 있으면 팀장 및 팀원에게 공유

- 문제 해결은 **우선 스스로 시도**합니다.
    - 공식 문서/에러 메시지/검색을 통해 원인을 찾고, 시도한 내용을 **기록**합니다.
- 단, **3시간 이상 막히면 바로 공유합니다.**
    - 팀장 또는 해당 도메인을 잘 아는 팀원에게 **에러 메시지 + 시도한 것 + 원하는 결과**를 함께 전달합니다.

# 4. 팀원들에게

- **소통과 협업 마인드**
    - 본인의 진행 상황과 어려움을 투명하게 공유하기
    - 다른 팀원의 의견을 존중하고 건설적인 피드백 주고받기
    - 갈등이나 의견 차이가 있을 때는 열린 마음으로 대화하기
- **공통 목표 의식**
    - 프로젝트 완성도를 높이기 위해 서로 협력하기
    - 문제 발생 시 비난보다는 해결책 찾기에 집중하기
    
- **시간 관리와 책임감**
    - 정해진 데일리 스크럼 시간과 코어 타임 준수하기
    - 개인 일정 변경이나 지연 사항은 미리 팀에 공유하기
    
- **서로 돕는 문화 만들기**
    - 코드 리뷰 시 비판보다는 개선 방향 제시하기
    - 팀원의 성장과 학습을 응원하고 격려하기
    
- **팀장을 믿어주고 따르기**
    - 팀장이 내린 결정 및 팀 방향을 신뢰하고 적극적으로 실행에 옮기기
    - 팀장의 부담을 덜어주기 위해 자기 역할에 책임감 갖기
    - 합의된 사항에는 이견이 있더라도 팀 전체의 목표를 우선시하며 따르기

    - 팀원들의 의견을 충분히 듣고 합리적으로 결정하기
    - 갈등 상황에서는 중재자 역할을 하며 객관적으로 판단하기
    - 중요한 결정사항은 팀원들과 충분히 논의한 후 결정하기
        
        
- **팀원 케어와 동기부여**
    - 팀원 개개인의 강점과 관심사를 파악하고 활용하기
    - 어려워하는 팀원이 있으면 적극적으로 도움 제공하기
    - 팀 분위기를 긍정적으로 유지하고 사기 진작하기

- **팀원을 믿어주기**
    - 각 팀원의 전문성과 역할을 존중하고 맡은 일을 신뢰하기
    - 팀원이 실수를 하더라도 비난보다는 학습의 기회로 받아들이기
    - 서로가 필요한 지원과 도움을 줄 수 있다는 믿음을 갖기
