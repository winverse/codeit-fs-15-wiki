### 테스트의 역할

프로그램을 만들 때는 코드를 작성하는 일만 중요한 것이 아닙니다. 이미 만든 코드가 계속 같은 방식으로 동작하는지 확인하는 일도 함께 필요합니다. 기능을 추가하거나 기존 코드를 수정하는 과정에서, 직접 손대지 않은 부분이 함께 깨지는 경우가 있기 때문입니다. 테스트는 바로 이 확인 작업을 코드로 남기는 방법입니다.

예를 들어 로그인 API를 만든다고 생각해 보겠습니다. 정상적인 이메일과 비밀번호를 보냈을 때는 로그인에 성공해야 하고, 비밀번호가 틀리면 `401`, 입력값이 비어 있으면 `400`을 보내야 합니다. 처음에는 브라우저나 API 도구로 몇 번만 확인해도 충분해 보일 수 있습니다. 하지만 로그인 실패 조건이 하나 더 늘어나거나, 응답 본문 구조를 조금만 바꿔도 이전 동작이 조용히 깨질 수 있습니다. 이때 테스트가 있으면 `무엇이 깨졌는지`, `어느 조건에서 깨졌는지`를 더 빠르게 확인할 수 있습니다.

규모가 작은 예제에서는 프로그램을 직접 실행해 보면서 확인할 수 있습니다. 하지만 프로젝트가 조금만 커져도 이 방법만으로는 부족해집니다. 어떤 입력에서는 잘 동작해도 다른 입력에서는 다른 결과가 나올 수 있고, 예외 상황은 더 쉽게 놓치기 때문입니다. 서비스 메서드의 분기, 컨트롤러의 상태 코드, 외부 의존성이 붙은 흐름은 특히 그렇습니다.

지금처럼 AI 도구를 이용해 코드를 더 빠르게 작성하고 수정하는 시대에는 테스트의 역할이 더 중요합니다. AI는 초안을 빠르게 만들고 반복 작업을 줄여 줄 수 있지만, 생성된 코드가 현재 프로젝트의 의도와 정확히 맞는지까지 자동으로 보장해 주지는 않습니다. 겉보기에는 그럴듯하게 동작해도, 예외 조건이나 기존 규칙을 놓친 채 들어오는 경우가 있습니다. 그래서 테스트는 단순한 품질 보조 도구가 아니라, `빠르게 만든 코드가 정말로 맞는지 확인하는 검증 장치`가 됩니다.

즉, 테스트는 코드를 못 믿어서 붙이는 장치가 아닙니다. 사람이 직접 작성한 코드든 AI가 도와준 코드든, 결국 현재 프로젝트가 기대하는 동작을 계속 유지하는지 확인하기 위해 필요합니다. 테스트가 있으면 수정 속도를 높이면서도, 그 수정이 기존 동작을 망가뜨리지 않았는지 바로 확인할 수 있습니다.

### Jest의 기본 기능

Jest는 JavaScript와 TypeScript 코드를 테스트할 때 널리 사용하는 도구입니다. 테스트를 실행하고, 결과를 비교하고, 함수 호출을 검증하고, 필요한 경우 실제 의존성을 가짜 구현으로 바꾸는 기능까지 함께 제공합니다.

가장 단순한 예는 다른 파일에 있는 함수를 가져와서 테스트하는 형태입니다.

예를 들어 아주 작은 함수와 검증 문장이 아래처럼 있다고 가정해 보겠습니다. 아직 실제 파일을 만드는 단계는 아니므로, 여기서는 함수와 검증 문장의 모양만 가장 작게 봅니다.

개념 예시:

```tsx
export function add(a: number, b: number) {
  return a + b;
}
```

테스트 파일에서는 이런 검증 문장으로 결과를 확인합니다.

개념 예시:

```tsx
expect(add(1, 2)).toBe(3);
```

이 예시는 테스트 파일 안에서 어떤 검증 문장이 들어가는지 가장 작게 보여 주는 용도입니다. 실제 파일로 작성할 때는 뒤에서 보게 될 것처럼 `describe(...)`와 `test(...)` 안에 이 검증 문장을 넣고, import도 함께 적습니다. 여기서 `expect`는 실제 값이 어떤 결과여야 하는지 적는 부분입니다. Jest는 이런 테스트를 모아서 실행한 뒤, 통과했는지 실패했는지를 알려 줍니다.

Jest가 맡는 일은 크게 네 가지로 정리할 수 있습니다.

- 테스트를 모아서 실행합니다.
- 결과가 기대와 맞는지 비교합니다.
- 함수 호출과 반환값을 검증합니다.
- 실제 의존성을 가짜 구현으로 바꾸어 테스트를 단순하게 만듭니다.

### Jest 공식 문서가 강조하는 방향

Jest 공식 문서는 Jest를 `단순함에 초점을 둔 즐거운 JavaScript 테스트 프레임워크`로 소개합니다. 이 표현은 단순한 홍보 문구가 아니라, Jest가 어떤 방향으로 설계되었는지를 잘 보여 줍니다. 즉, 테스트를 시작할 때 설정과 문법 때문에 너무 빨리 지치지 않게 하고, 필요한 기능은 한 도구 안에서 비교적 자연스럽게 이어 쓰도록 만들겠다는 뜻입니다.

공식 문서의 소개 페이지를 보면 이 방향이 더 분명하게 드러납니다. Jest는 `Zero config`, `Isolated`, `Great API`, `Code Coverage`, `Easy Mocking` 같은 특징을 앞에 둡니다. 다시 말해, Jest는 테스트를 어렵게 만드는 복잡한 주변 작업을 줄이고, 테스트 작성과 실행에 더 빨리 들어갈 수 있게 하려는 쪽에 무게를 둡니다.

다만 이 표현을 그대로 받아들이면 안 됩니다. Jest는 기본적으로 시작이 쉬운 도구이지만, 지금처럼 ESM과 TypeScript를 함께 쓰는 실제 프로젝트에서는 추가 설정이 필요할 수 있습니다. 이 장도 바로 그 지점부터 다룹니다.

공식 문서를 처음 읽을 때도 이 흐름을 알고 보는 편이 좋습니다. 보통은 `Getting Started`에서 가장 작은 테스트를 한 번 실행해 보고, 그다음 `Using Matchers`, `Testing Asynchronous Code`, `Setup and Teardown`, `Mock Functions` 순서로 따라가면 자연스럽습니다. 이 순서는 단순한 값 비교에서 시작해, 비동기 처리와 준비/정리, 그리고 mock을 이용한 테스트 분리까지 점점 범위를 넓혀 가는 흐름입니다.

이 교재도 큰 방향은 비슷합니다. 다만 공식 문서가 Jest 자체를 소개하는 데 집중한다면, 이 교재는 먼저 Jest 기본기를 익히고 뒤 장에서 그 흐름을 Express 프로젝트에 적용하는 순서로 진행합니다. 그래서 1장부터 3장까지는 Jest 자체를 이해하는 데 집중하고, Express 맥락은 4장부터 다시 다룹니다.

### 정리

Jest는 테스트를 실행하고 검증하는 도구입니다. 이 장에서는 가장 작은 함수 테스트부터 시작해서, 이후 장에서 matcher, mock, 비동기 테스트 같은 기본기를 차례로 넓혀 갑니다.

## **1-03 Jest + TypeScript + ESM 설정**

### ESM 환경에서 설정이 꼬이는 이유

ESM 자체는 앞선 JavaScript, TypeScript 과정에서 이미 다뤘다고 보고 넘어가겠습니다. 여기서 새로 볼 것은 **Jest를 붙일 때 어디에서 충돌이 생기는가**입니다. 핵심은 세 가지입니다. `.js` 확장자를 붙인 import를 Jest가 테스트 시점에 어떻게 따라갈지, `ts-jest`가 TypeScript를 ESM으로 변환하도록 어떻게 고정할지, 그리고 테스트 실행 명령에서 `NODE_OPTIONS`를 어떻게 넘길지입니다.

앞선 과정에서 이미 본 것처럼, 이 프로젝트의 import는 `.js` 확장자를 붙인 형태를 사용합니다.

개념 예시:

```tsx
import { add } from "#src/add.js";

// ... 생략 ...
```

여기서 먼저 눈에 들어와야 할 것은 `#src/*` alias입니다. 이 alias는 코드 안에서는 짧고 읽기 쉽게 경로를 적게 해 주지만, Jest는 이 경로를 그대로 읽어서는 실제 파일 위치를 알지 못합니다. 게다가 테스트 시점에는 아직 빌드된 `.js` 파일이 없고 `.ts` 소스만 있으므로, `#src/add.js` 같은 경로를 그대로 따라가면 꼬일 수 있습니다.

그래서 현재 설정에서는 `moduleNameMapper`를 사용합니다. `moduleNameMapper`는 **코드에서 쓰는 import 경로를, Jest가 실제 파일 위치로 다시 해석하도록 알려 주는 설정**입니다. 이 교재의 설정에서는 `#src/add.js` 같은 alias import를 테스트 시점에 `&lt;rootDir&gt;/src/add.ts` 쪽으로 자연스럽게 따라가도록 맞춥니다. 즉, alias 자체를 없애는 것이 아니라, **Jest도 같은 alias를 이해하게 만드는 것**에 가깝습니다.

Jest는 TypeScript 파일을 그대로 읽지 못하므로, 이 교재에서는 `ts-jest`를 사용합니다. 그리고 `useESM: true`, `NODE_OPTIONS=--experimental-vm-modules` 같은 설정도 모두 같은 문제를 푸는 장치라고 보면 됩니다.

정리하면 ESM 환경에서 설정이 꼬이는 이유는 단순합니다. Node는 런타임 관점에서 모듈을 읽고, TypeScript는 타입 검사와 빌드 관점에서 모듈을 읽고, Jest는 테스트 실행 관점에서 모듈을 읽습니다. 현재 설정 파일에 들어간 여러 옵션은 설정을 복잡하게 늘어놓기 위한 것이 아니라, 서로 다른 해석 기준을 한 방향으로 맞추기 위해 필요한 장치들입니다.

### 관련 패키지 설치

이 저장소와 실습 코드는 `Jest 30`, `ts-jest 29.4`, `Node 22.x 이상`, `TypeScript 6.x 이상` 환경에서 검증합니다. 여기서 `Node 22.x`, `TypeScript 6.x`는 **이 저장소가 실제로 검증한 기본 실행 환경**을 뜻합니다. 즉, 최신 Jest의 최소 요구사항을 그대로 적은 문장은 아닙니다.

`2026-04-22` 기준으로 Jest 30 공식 업그레이드 문서는 최소 지원 범위를 `Node 18.x`, `TypeScript 5.4`로 안내합니다. 이 교재에서는 공식 최소 요구사항을 그대로 좇기보다, 저장소에서 실제로 검증한 조합을 기준 환경으로 사용합니다.

여기서 버전 숫자를 보면 `Jest 30`과 `ts-jest 29.4`가 어긋나 보일 수 있습니다. 현재 `ts-jest`는 29.x 계열이지만, 이 교재 범위에서는 저장소에서 검증한 조합을 그대로 사용합니다. 즉, 버전 숫자만 보고 조합을 바꾸기보다 먼저 저장소 기준과 lockfile 기준을 따르는 편이 안전합니다.

지금 `01-jest-ts-esm` starter의 `package.json`에는 아직 Jest와 TypeScript 실행에 필요한 패키지가 들어 있지 않습니다. 그래서 여기서는 TypeScript와 Jest 연동에 필요한 공통 패키지를 한 번에 설치합니다. 반대로 4장 이후처럼 이미 TypeScript starter가 준비된 프로젝트에서는 `typescript`, `@types/node`를 다시 설치하지 않습니다.

터미널

```bash
pnpm add -D typescript @types/node jest ts-jest @types/jest @jest/globals cross-env
```

이 목록에서 앞선 과정과 직접 이어지는 것은 `typescript`, `@types/node`이고, 이번 장에서 새로 중요한 것은 `jest`, `ts-jest`, `@types/jest`, `@jest/globals`, `cross-env`입니다. TypeScript에서 Jest API 타입을 맞추는 방식은 하나만 있는 것이 아닙니다. 이 교재는 ESM 예제에서 `jest` 객체와 일부 API를 직접 import하는 방식도 쓰고, 현재 저장소의 전역 `test`, `expect` 타입도 함께 맞추므로 `@jest/globals`와 `@types/jest`를 같이 둡니다. 즉, 이 두 패키지는 모든 Jest 프로젝트의 공통 필수라기보다, **현재 저장소가 택한 ESM + TypeScript 작성 방식에 맞춘 선택**입니다. 특히 `cross-env`는 Windows에서 `NODE_OPTIONS`를 안전하게 넘길 때 필요합니다.

현재 체크포인트 저장소는 운영체제별로 스크립트를 따로 나누지 않고, `cross-env`를 넣은 공통 스크립트 하나로 맞춥니다. 이 방식이 저장소 기준과 가장 잘 맞고, 이후 장에서 `test:watch`, `test:runInBand` 같은 스크립트를 읽을 때도 기준이 흔들리지 않습니다.

### package.json 설정

지금 `package.json`은 최소 프로젝트 상태이므로, 기존 필드를 지우지 않고 Jest 실행에 필요한 항목을 덧붙여 완성합니다. `type`과 `imports` 자체는 이미 앞선 과정에서 익힌 내용이고, 여기서는 **Jest 스크립트가 그 ESM 기준과 충돌하지 않게 붙는지**만 보면 됩니다. 이 교재의 체크포인트는 운영체제에 상관없이 같은 스크립트를 쓰도록 `cross-env`를 기본으로 둡니다. 아래 코드는 `01-jest-ts-esm` starter를 기준으로 1장 설정이 끝났을 때의 전체 `package.json` 모습입니다.

파일: `package.json`

```json
{
  "name": "@express-test/01-jest-ts-esm",
  "private": true,
  "version": "1.0.0",
  "type": "module",
  "imports": {

      "types": "./src/*.ts",
```

여기서 새로 볼 부분은 `test`, `test:cov` 스크립트입니다. 저장소 기준은 두 운영체제를 따로 나누기보다, `cross-env`를 앞에 붙여 한 번에 맞추는 쪽입니다.

`NODE_OPTIONS=--experimental-vm-modules`는 Jest가 ESM 테스트 파일을 Node의 ESM 방식에 더 가깝게 실행하도록 돕는 런타임 옵션입니다. 이 옵션이 없으면 ESM 프로젝트에서 테스트 로딩이 꼬일 수 있습니다. `2026-04-22` 기준으로 Jest 공식 문서는 ESM 지원을 여전히 `experimental`로 안내합니다. 따라서 이 교재에서는 이 설정을 `완전히 일반화된 기본 설정`이라기보다, 현재 저장소에서 실제로 검증한 ESM 실행 조합으로 이해하면 됩니다.

### tsconfig.json에서 볼 부분

이 시작점에는 `tsconfig.json`도 아직 없습니다. 다만 TypeScript 설정 전체를 다시 설명할 필요는 없습니다. 여기서는 Jest와 직접 부딪히는 값만 봅니다. `module`과 `moduleResolution`은 `NodeNext`로 맞춰 두고, 1장부터 3장까지는 테스트 파일도 `src/` 안에 함께 두므로 `include`는 `src/**/*.ts`를 읽도록 두면 충분합니다. 이때 `NodeNext` 같은 hybrid module 값을 쓰는 흐름에서는 `isolatedModules: true`도 함께 보는 편이 안전합니다. 현재 체크포인트도 이 값을 포함하고 있고, `ts-jest`의 ESM 가이드도 같은 기준을 권장합니다.

실행 가능한 최소 예시는 아래 정도면 충분합니다.

파일: `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "isolatedModules": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "types": ["node", "jest"],
    "rootDir": ".",
    "outDir": "./dist"
  },
  "include": ["src/**/*.ts"]
}
```

### Jest 설정 파일

이 예제에서는 폴더 안에서 바로 읽히는 `jest.config.mjs`를 사용합니다. 설정 예시는 아래와 같습니다.

파일: `jest.config.mjs`

```jsx
export default {
  testEnvironment: "node", // 브라우저 대신 Node 런타임에서 테스트를 실행합니다.
  testMatch: ["<rootDir>/src/**/*.spec.ts"], // `src` 아래의 `*.spec.ts` 파일만 테스트로 찾습니다.
  extensionsToTreatAsEsm: [".ts"], // `.ts` 파일도 ESM 방식으로 해석합니다.
  moduleFileExtensions: ["ts", "js", "json"], // 테스트 중 읽을 수 있는 모듈 확장자 목록입니다.
  transform: {
    // TypeScript 파일을 Jest가 실행할 수 있도록 `ts-jest`로 변환합니다.
    "^.+\\.ts$": [
      "ts-jest",
      {
        useESM: true, // 변환 결과도 ESM 기준으로 맞춥니다.
        tsconfig: "<rootDir>/tsconfig.json", // 변환할 때 현재 프로젝트의 tsconfig를 그대로 사용합니다.
      },
    ],
  },
  moduleNameMapper: {
    "^(\\.{1,2}/.*)\\.js$": "$1", // 상대 경로 import 끝의 `.js`를 제거해 TypeScript 소스와 맞춥니다.
    "^#src/(.*)\\.js$": "<rootDir>/src/$1", // `#src/...` 별칭을 실제 `src` 경로로 연결합니다.
    "^#test/(.*)\\.js$": "<rootDir>/test/$1", // `#test/...` 별칭을 실제 `test` 경로로 연결합니다.
  },
};
```

설정을 읽을 때는 “Jest도 현재 프로젝트를 ESM + TypeScript 기준으로 읽도록 맞춘다”라고 이해하면 됩니다.

### 정리

이 절에서 기억할 것은 설정 파일 이름보다 각 파일이 맡는 역할입니다.

- `package.json`은 프로젝트의 ESM 기준과 테스트 실행 명령을 정합니다.
- `tsconfig.json`은 TypeScript가 그 기준을 어떻게 해석할지 정합니다.
- `jest.config.mjs`는 테스트 실행도 같은 방향으로 맞춥니다.
- 이 장에서 새로 중요한 패키지는 `jest`, `ts-jest`, `@types/jest`, `cross-env`입니다.

## **1-04 첫 테스트 실행**

### 실행 여부 확인

설정을 마쳤다면, 이제 테스트 하나를 실제로 실행해 볼 차례입니다. 설정 파일이 있다는 사실만으로는 아직 아무것도 검증되지 않았습니다. 반대로 아주 작은 테스트 하나라도 통과하면, 이 프로젝트에서 Jest가 실제로 동작한다는 사실을 확인할 수 있습니다.

이 기준은 이후 장에서도 계속 중요합니다. 나중에 mock을 붙이거나, 비동기 테스트를 쓰거나, Express 앱에 실제 HTTP 요청을 보내 보며 흐름을 확인하는 도구를 도입하다가 문제가 생겨도 적어도 기본 실행 자체는 한 번 검증되었다는 사실을 알고 출발할 수 있기 때문입니다.

### 가장 작은 테스트 작성

이 예제는 `src/**/*.spec.ts`를 테스트 파일로 읽습니다. 즉, 제품 코드 파일이 `src/add.ts`에 있으면 그 테스트 파일도 같은 폴더 수준의 `src/add.spec.ts`에 둔다는 뜻입니다. 다시 말해 테스트 대상 코드 옆에 테스트 파일을 나란히 두는 방식입니다. 이 장에서는 설명을 단순하게 유지하기 위해 제품 코드와 테스트 파일을 모두 `src/` 안에 둡니다. 반대로 실제 Express 구조를 다루는 단계에서는 `src/`와 `test/`를 분리한 구조를 다시 다룹니다.

먼저 테스트 대상 파일을 만듭니다. `01-jest-ts-esm` starter의 `src/` 안에는 아직 `.gitkeep`만 있으므로, 이 단계에서 `add.ts`를 직접 만들어야 테스트가 이어집니다.

파일: `src/add.ts`

```tsx
export function add(a: number, b: number) {
  return a + b;
}
```

그다음 `src/add.spec.ts` 파일을 만들고 아래 내용을 적습니다.

파일: `src/add.spec.ts`

```tsx
import { add } from "#src/add.js";

describe("add", () => {
  test("두 수의 합을 반환합니다", () => {
    expect(add(1, 2)).toBe(3);
  });
});
```

이 테스트는 기능적으로 특별한 의미를 가지지는 않습니다. 그렇지만 지금 단계에서는 그 단순함이 오히려 장점입니다. 실패했을 때 비즈니스 로직보다 설정 문제를 먼저 의심할 수 있기 때문입니다.

### 실패 시 점검 항목

첫 테스트가 실패하더라도 지금은 확인 범위를 좁히기 쉽습니다. 아직 로직이 거의 없기 때문입니다. 먼저 아래를 확인하면 됩니다.

- `package.json`에 `"type": "module"`과 `test` 스크립트가 들어 있는가

- 테스트 파일이 `src/**/*.spec.ts` 위치에 있는가

이 단계에서의 실패는 테스트를 못 쓴다는 뜻이 아니라, 환경 기준이 아직 어긋나 있다는 뜻에 더 가깝습니다.

# 제 2장: matcher와 mock function 기본기

## **2-01 matcher로 결과 검증하기**

### 시작점

2장 본편은 `codeit-fs-jest-starter/02-matchers-and-mocks`를 기준으로 읽습니다.

### matcher로 결과를 읽는 기준

matcher는 테스트를 마지막 검증 문장으로 닫는 도구입니다. 값 비교, 구조 비교, 부분 비교를 어떻게 읽는지 잡혀야 이후 서비스 결과와 협력 객체 호출도 자연스럽게 검증할 수 있습니다. 이 장에서는 `jest.fn()`, `spyOn()`, `mockImplementation()`처럼 호출을 관찰하거나 대체할 때 쓰는 기본 도구까지 함께 익힙니다.

2장 체크포인트에서는 Jest 기본기 예제를 `src/` 아래에 모아 둡니다. 그래서 아래 코드 블록도 모두 이 실제 구조를 기준으로 읽겠습니다.

### matcher의 역할

Jest 테스트는 마지막에 반드시 검증 문장으로 닫힙니다. 어떤 함수를 호출했는지, 어떤 값을 반환했는지, 어떤 객체가 만들어졌는지, 예외가 발생했는지 모두 검증 문장으로 확인합니다. Jest에서는 이 검증 규칙을 matcher라고 부릅니다.

가장 단순한 matcher는 `toBe`입니다. 먼저 2장 안의 가장 작은 함수 하나를 보겠습니다.

파일: `src/orders/formatter.ts`

```tsx
export const formatter = {
  format(value: number): string {
    return `${value.toLocaleString("ko-KR")}원`;
  },
};
```

이 함수는 숫자 하나를 받아 문자열 하나를 돌려줍니다. 그래서 `"3,000원"`처럼 **값 하나**를 확인할 때는 `toBe`가 가장 자연스럽습니다.

개념 예시:

```tsx
expect(formatter.format(3_000)).toBe("3,000원");
```

반대로 주문 요약처럼 객체 전체를 비교하려면 구조 비교가 필요합니다. 이 차이를 바로 다음 예제로 이어서 보겠습니다.

### 값 비교와 구조 비교

matcher를 사용할 때 가장 먼저 구분해야 할 것은 값 자체를 비교하는 경우와 객체 구조를 비교하는 경우입니다. 이 차이는 결과 객체를 만드는 함수를 보면 더 분명하게 읽힙니다.

파일: `src/orders/create-order-summary.ts`

```tsx
export function createOrderSummary(
  items: OrderItem[],
): OrderSummary {
  // 수량이 0인 항목은 주문 요약에서 제외합니다.
  const visibleItems = items
    .filter((item) => item.quantity > 0)
    .map((item) => ({
      name: item.name,
      subtotal: item.price * item.quantity,
      quantity: item.quantity,
    }));

  // itemCount는 남은 항목의 총 수량입니다.
  const itemCount = visibleItems.reduce(
    (sum, item) => sum + item.quantity,
    0,
  );
  // subtotal은 남은 항목 가격의 합계입니다.
  const subtotal = visibleItems.reduce(
    (sum, item) => sum + item.subtotal,
    0,
  );
  // shippingFee는 무료 배송 기준을 적용한 배송비입니다.
  const shippingFee = subtotal >= 50_000 ? 0 : 3_000;

  return {
    itemCount,
    subtotal,
    shippingFee,
    total: subtotal + shippingFee,
    items: visibleItems.map(
      ({ name, subtotal: itemSubtotal }) => ({
        name,
        subtotal: itemSubtotal,
      }),
    ),
  };
}
```

이 함수의 테스트 계획은 세 갈래로 잡을 수 있습니다. 수량이 0인 항목이 빠지는지, 합계와 배송비가 기대대로 계산되는지, 최종 결과 객체가 우리가 기대한 구조인지 확인하는 일입니다. 그래서 먼저 전체 결과를 `toStrictEqual`로 단단하게 고정하고, 뒤에서 일부 값만 느슨하게 보는 비교까지 이어 갑니다.

파일: `src/orders/create-order-summary.spec.ts`

```tsx
import { createOrderSummary } from "#src/orders/create-order-summary.js";

describe("createOrderSummary", () => {
  test("itemCount와 shippingFee 같은 원시값은 toBe로 비교한다", () => {
    const summary = createOrderSummary([
      { name: "Node.js 책", price: 18_000, quantity: 2 },
      { name: "Jest 스티커", price: 6_000, quantity: 1 },
      { name: "TypeScript 카드", price: 0, quantity: 0 },
    ]);

    expect(summary.itemCount).toBe(3);
    expect(summary.shippingFee).toBe(3_000);
  });
});
```

파일 일부: `src/orders/create-order-summary.spec.ts`

```tsx
// ... 생략 ...

describe("createOrderSummary", () => {
  test("결과 객체 전체는 toStrictEqual로 비교한다", () => {
    const summary = createOrderSummary([
      { name: "Node.js 책", price: 18_000, quantity: 2 },
      { name: "Jest 스티커", price: 6_000, quantity: 1 },
      { name: "TypeScript 카드", price: 0, quantity: 0 },
    ]);

    expect(summary).toStrictEqual({
      itemCount: 3,
      subtotal: 42_000,
      shippingFee: 3_000,
      total: 45_000,
      items: [
        { name: "Node.js 책", subtotal: 36_000 },
        { name: "Jest 스티커", subtotal: 6_000 },
      ],
    });
  });
});
```

`toBe`는 숫자, 문자열, 불리언처럼 원시값을 볼 때 적합합니다. 반대로 객체와 배열은 구조를 비교해야 하므로 `toStrictEqual`이나 `toEqual`을 사용합니다. 이 장에서는 구조를 더 분명하게 고정하기 위해 `toStrictEqual`을 먼저 익힙니다. 서비스 로직이 계산한 결과 객체 전체를 한 번에 검증할 때 읽기가 분명하기 때문입니다.

여기서 `toEqual`과 `toStrictEqual`의 차이도 같이 기억해 두는 편이 좋습니다.

- `toEqual`: 값 중심으로 비교합니다.
- `toStrictEqual`: 타입, `undefined`, 배열 구조까지 더 엄격하게 비교합니다.

둘 다 객체와 배열을 깊게 비교합니다. 즉, 바깥 객체만 보는 것이 아니라 그 안쪽의 중첩 객체와 배열까지 재귀적으로 내려가며 비교합니다. 차이는 비교 깊이가 아니라 **비교 기준의 엄격함**에 있습니다.

예를 들어 아래 코드는 차이를 바로 보여 줍니다.

개념 예시:

```tsx
expect({ shippingFee: undefined }).toEqual({});
// 통과

expect({ shippingFee: undefined }).toStrictEqual({});
// 실패
```

즉, 결과 객체의 모양까지 단단하게 고정하고 싶다면 `toStrictEqual`이 더 안전합니다.

### 부분 비교와 유연한 비교

실제 테스트에서는 모든 값을 완전히 고정하기 어려운 경우가 있습니다. 계산 결과 전체를 모두 비교할 필요는 없고, 중요한 일부 값만 확인하면 충분한 경우가 있기 때문입니다. 같은 `createOrderSummary`라도 이번에는 전체 객체를 다시 복사해 비교하기보다, `shippingFee`, `total`, `items` 일부처럼 지금 확인하고 싶은 부분만 고르는 계획이 더 자연스럽습니다.

이럴 때는 일부 구조만 확인하는 matcher를 사용합니다. 강의할 때는 같은 파일을 여러 조각으로 잘라 보여 주기보다, 아래처럼 한 번에 읽히는 코드 블록으로 보여 주고 새로 추가되는 부분만 주석으로 표시하는 편이 더 자연스럽습니다.

파일: `src/orders/create-order-summary.spec.ts`

```tsx
import { createOrderSummary } from "#src/orders/create-order-summary.js";

describe("createOrderSummary", () => {
  test("중요한 일부 속성만 부분 비교로 확인한다", () => {
    const freeShippingSummary = createOrderSummary([
      { name: "Express 강의", price: 50_000, quantity: 1 },
    ]);

    expect(freeShippingSummary).toMatchObject({
      shippingFee: 0,
      total: 50_000,
    });
    expect(freeShippingSummary.items).toContainEqual({
      name: "Express 강의",
      subtotal: 50_000,
    });
  });
});
```

`toMatchObject(...)`는 객체 전체가 아니라 중요한 일부 속성만 확인합니다. `toContainEqual(...)`은 배열 안에 특정 구조를 가진 요소가 있는지 확인합니다. 일부 구조만 확인하고 싶을 때 특히 유용합니다. 예를 들어 서비스 로직 결과에서 `shippingFee`, `total`만 중요하고 나머지 필드는 덜 중요할 때 이런 matcher를 쓰면 테스트가 과하게 단단해지지 않습니다.

### 정리

matcher는 테스트 결과를 읽는 문장입니다. 값 비교, 구조 비교, 부분 비교를 구분할 수 있어야 이후 장의 컨트롤러 테스트와 서비스 테스트가 흔들리지 않습니다.

## **2-02 jest.fn, spyOn, mockImplementation**

### 테스트 더블과 `jest.fn`

테스트를 작성하다 보면 실제 의존성을 그대로 두기 어려운 순간이 많습니다. 컨트롤러가 서비스를 호출하고, 서비스가 저장소를 호출하고, 저장소가 다시 외부 시스템을 만지는 구조에서는 현재 대상만 분리해서 보고 싶은 경우가 많기 때문입니다.

이럴 때 실제 대상을 대신 넣는 가짜 객체나 가짜 함수를 `테스트 더블`이라고 부릅니다. 즉, 테스트 더블은 “진짜를 대신해서 테스트 안에 들어가는 대역”이라는 뜻입니다. 메일 발송기, 결제 모듈, 외부 API 클라이언트처럼 테스트 중에 실제로 실행하고 싶지 않은 대상을 잠깐 치워 두고, 현재 코드가 그 대상과 어떻게 상호작용하는지만 확인하고 싶을 때 사용합니다. 결국 테스트 더블은 현재 테스트가 어디까지 볼지, 어디부터 바깥 의존성으로 남겨 둘지를 통제하기 위한 도구입니다.

Jest에서 가장 기본적인 테스트 더블 도구가 `jest.fn()`입니다. 이 교재에서는 가능하면 준비 코드에 들어 있는 객체를 import한 뒤, 필요한 메서드만 가짜 함수로 바꾸는 방식으로 설명합니다. 이 장의 예제는 ESM 기준으로 작성하므로, `jest.fn()`이나 `jest.spyOn()`처럼 `jest` 객체를 직접 쓰는 코드에서는 `import { jest } from "@jest/globals";`를 함께 적어 두겠습니다.

먼저 어떤 함수를 테스트하는지부터 보겠습니다.

파일: `src/orders/send-order-mail.ts`

```tsx
export async function sendOrderMail(
  input: OrderMailInput,
  dependencies: {
    mailer: Mailer;
    formatter: CurrencyFormatter;
  },
): Promise<void> {
  const formattedShippingFee =
    dependencies.formatter.format(input.shippingFee);
  const formattedTotal = dependencies.formatter.format(
    input.total,
  );

  await dependencies.mailer.send({
    to: input.email,
    subject: "[order] 주문이 접수되었습니다",
    body: `배송비는${formattedShippingFee}이고, 총 결제 금액은${formattedTotal}입니다.`,
  });
}
```

이 함수의 테스트 계획은 세 단계입니다. 배송비와 총액을 `formatter.format(...)`으로 문자열로 바꾸는지, 그렇게 만든 값으로 메일 payload를 조합하는지, 마지막에 `mailer.send(...)`를 한 번 호출하는지 확인하면 됩니다. 실제 메일 전송은 이 절의 관심사가 아니므로, 여기서는 메일러를 가짜 함수로 바꿔 호출 기록만 검증합니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { formatter } from "#src/orders/formatter.js";
import { sendOrderMail } from "#src/orders/send-order-mail.js";

describe("sendOrderMail", () => {
  test("메일 payload를 조합해 메일러를 호출한다", async () => {
    const sendSpy = jest.fn().mockResolvedValue(undefined);
    const fakeMailer = { send: sendSpy };

    await sendOrderMail(
      {
        email: "student@example.com",
        shippingFee: 1_000,
        total: 46_000,
      },
      { mailer: fakeMailer, formatter },
    );

    expect(sendSpy).toHaveBeenCalledTimes(1);
    expect(sendSpy).toHaveBeenCalledWith({
      to: "student@example.com",
      subject: "[order] 주문이 접수되었습니다",
      body: "배송비는 1,000원이고, 총 결제 금액은 46,000원입니다.",
    });
  });
});
```

여기서 핵심은 `jest.fn()` 자체입니다. `jest.fn()`을 호출하면 Jest가 호출 기록을 남길 수 있는 가짜 함수를 하나 만들어 줍니다. 이 함수는 기본적으로 아무 일도 하지 않지만, 누가 몇 번 불렀는지, 어떤 인자를 넘겼는지, 무엇을 반환했는지를 Jest가 추적할 수 있습니다.

학습 순서도 이 기준으로 잡으면 좋습니다. 먼저 `jest.fn()`으로 호출 기록이 남는 가짜 함수를 만들고, 그다음 `mockReturnValue(...)`나 `mockResolvedValue(...)`로 결과값만 고정합니다. 입력에 따라 다른 결과를 내야 하거나, 호출 순서별 동작까지 직접 제어해야 할 때 비로소 `mockImplementation(...)` 계열로 넘어갑니다.

세 메서드의 차이도 여기서 같이 정리해 두면 읽기 쉽습니다.

- `mockReturnValue(...)`: 동기 함수가 항상 같은 값을 반환하게 고정합니다.
- `mockResolvedValue(...)`: Promise를 반환하는 비동기 함수가 성공 결과를 돌려주게 고정합니다.
- `mockImplementation(...)`: 반환값만 고정하는 것이 아니라, 인자를 받아 어떤 로직으로 동작할지 직접 정의합니다.

즉, `mockReturnValue(...)`와 `mockResolvedValue(...)`는 **결과만 고정하는 도구**이고, `mockImplementation(...)`은 **동작 자체를 적는 도구**라고 보면 됩니다.

예를 들어 호출 기록은 그대로 남기되 결과값만 잠깐 고정하고 싶다면 아래처럼 읽으면 됩니다.

```tsx
const formatter = jest.fn().mockReturnValue("1,000원");
```

이 단계에서는 계산을 새로 적지 않습니다. 그 호출이 일단 `"1,000원"`을 돌려주기만 하면 충분한 상황이기 때문입니다. 반대로 인자에 따라 다른 값을 만들어야 한다면 그때 `mockImplementation(...)`으로 넘어갑니다.

예를 들어 금액에 따라 다른 문자열을 만들어야 한다면 아래처럼 `mockImplementation(...)`을 사용할 수 있습니다.

개념 예시:

```tsx
describe("formatter", () => {
  test("입력한 금액에 따라 다른 문자열을 반환한다", () => {
    const formatter = jest
      .fn()
      .mockImplementation(
        (price: number) => `${price.toLocaleString()}원`,
      );

    expect(formatter(1_000)).toBe("1,000원");
    expect(formatter(25_000)).toBe("25,000원");
    expect(formatter).toHaveBeenNthCalledWith(1, 1_000);
    expect(formatter).toHaveBeenNthCalledWith(2, 25_000);
  });
});
```

여기서는 반환값 하나를 미리 고정한 것이 아니라, `price`라는 입력을 받아 그때그때 문자열을 만들어 돌려줍니다. 즉, `mockImplementation(...)`은 “가짜 함수가 어떤 규칙으로 동작할지”를 직접 적는다고 이해하면 됩니다.

즉, 지금 검증하는 대상은 `sendSpy`가 아니라 `sendOrderMail()`입니다. 테스트 대상 코드가 실행된 뒤, 그 코드가 메일 전송 함수를 어떤 인자로 호출했는지를 확인하는 흐름으로 읽어야 합니다. 실제 메일을 보내고 싶은 것이 아니라, 현재 코드가 메일러와 어떻게 상호작용하는지를 확인하고 싶을 때 이런 패턴을 사용합니다.

### spyOn의 용도

이미 존재하는 객체의 메서드 하나만 감시하거나 잠깐 바꾸고 싶다면 `spyOn`이 더 적합합니다.

예를 들어 기존 로거 객체는 그대로 두고, `info` 메서드가 어떤 값을 받았는지만 보고 싶은 경우가 있습니다. 이때는 새 객체를 다시 만들기보다 기존 객체 메서드에 spy를 걸 수 있습니다.

파일: `src/orders/audit-logger.ts`

```tsx
export const auditLogger = {
  info(message: string): void {
    console.log(message);
  },
};
```

이 객체는 `info` 메서드 하나만 바깥에 노출합니다. 따라서 테스트 계획도 단순합니다. 실제 `console.log` 출력을 확인하기보다, `info(...)`가 호출되었는지와 어떤 메시지를 받았는지를 검증하는 쪽이 더 자연스럽습니다.

파일: `src/orders/audit-logger.spec.ts`

```tsx
import { jest } from "@jest/globals";
import { auditLogger } from "#src/orders/audit-logger.js";

describe("auditLogger", () => {
  test("기존 메서드에 spy를 걸어 호출을 감시한다", () => {
    const loggerSpy = jest
      .spyOn(auditLogger, "info")
      .mockImplementation(() => {});

    auditLogger.info("주문 로그");

    expect(loggerSpy).toHaveBeenCalledWith("주문 로그");
  });
});
```

여기서 `jest.spyOn(auditLogger, "info")`는 두 부분으로 읽으면 됩니다.

- 첫 번째 인자 `auditLogger`는 메서드를 꺼내 올 **대상 객체**입니다.
- 두 번째 인자 `"info"`는 그 객체 안에서 감시할 **메서드 이름**입니다.

즉, `spyOn`은 `어떤 객체의 어떤 메서드를 감시할 것인가`를 적는 문법이라고 이해하면 됩니다.

`spyOn`은 기존 구조를 유지한 채 필요한 지점만 관찰할 수 있다는 장점이 있습니다. 여기서는 먼저 `info` 메서드가 호출되었는지와 어떤 인자를 받았는지만 추적합니다. 기존 메서드의 동작까지 바꾸고 싶다면 `mockImplementation()`을 함께 붙이면 됩니다.

여기서 꼭 기억해야 할 점이 하나 있습니다. `spyOn`은 기본적으로 원래 메서드를 그대로 실행하면서 호출 기록만 추적합니다. 따라서 호출 여부만 보고 싶다면 `spyOn(...)`만으로 충분하지만, 원래 동작을 막거나 다른 동작으로 바꾸고 싶다면 `mockImplementation(...)`이나 `mockReturnValue(...)`를 함께 붙여야 합니다.

### 일반 함수보다 협력 객체 메서드를 먼저 감시하는 이유

현재 교재의 ESM 설정에서는 `export function ...` 형태의 일반 함수를 `spyOn()` 대상으로 삼는 흐름을 굳이 앞에 두지 않습니다. 이 형태의 함수는 보통 **입력과 결과를 직접 비교하는 순수 함수 테스트**로 읽는 편이 더 자연스럽고, 실제 호출 기록을 보고 싶은 지점은 대개 `formatter.format(...)`, `mailer.send(...)`, `auditLogger.info(...)`처럼 협력 객체의 메서드로 드러나기 때문입니다.

즉, 이 교재에서는 기준을 이렇게 잡습니다.

- `createOrderSummary(...)` 같은 일반 함수:
    - 입력과 결과를 비교하는 테스트로 읽습니다.
- `formatter.format(...)`, `mailer.send(...)`, `auditLogger.info(...)` 같은 협력 객체 메서드:
    - `spyOn()`으로 호출 기록과 인자를 감시합니다.

일반 함수 테스트 모양은 위에서 본 `src/orders/create-order-summary.spec.ts` 예시를 다시 떠올리면 됩니다. 그 예시처럼 일반 함수는 “무엇을 몇 번 불렀는가”보다 “어떤 입력을 주었을 때 어떤 결과가 나왔는가”를 먼저 봅니다. 그래서 이 경우에는 `spyOn()`보다 `toStrictEqual(...)` 같은 결과 비교 matcher가 더 자연스럽습니다.

이렇게 나누면 `spyOn()`이 필요한 이유도 더 분명해집니다. 우리는 함수 자체를 무조건 감시하려는 것이 아니라, **테스트 대상 코드가 바깥 협력 객체와 어떻게 상호작용하는지**를 보고 싶은 것이기 때문입니다.

### 호출 기록을 검증하는 matcher

`jest.fn()`과 `spyOn()`을 준비하면, 이제 호출 기록을 보는 matcher를 자연스럽게 붙일 수 있습니다. 주문 알림 예제에서는 메일 발송 함수가 몇 번 호출되었는지, 어떤 payload를 받았는지, 포매터가 어떤 순서로 실행되었는지를 아래처럼 검증할 수 있습니다.

파일: `src/orders/send-order-mail.spec.ts`

```tsx
import { jest } from "@jest/globals";
import { formatter } from "#src/orders/formatter.js";
import { mailer } from "#src/orders/mailer.js";
import { sendOrderMail } from "#src/orders/send-order-mail.js";

describe("sendOrderMail", () => {
	// ... 생략 ...
  test("호출 횟수와 호출 인자를 검증한다", async () => {
    const formatSpy = jest
      .spyOn(formatter, "format")
      .mockReturnValueOnce("1,000원")
      .mockReturnValueOnce("46,000원");
    const sendSpy = jest
      .spyOn(mailer, "send")
      .mockResolvedValue(undefined);

    await sendOrderMail(
      {
        email: "student@example.com",
        shippingFee: 1_000,
        total: 46_000,
      },
      { mailer, formatter },
    );

    expect(sendSpy).toHaveBeenCalledTimes(1);
    expect(sendSpy).toHaveBeenCalledWith({
      to: "student@example.com",
      subject: "[order] 주문이 접수되었습니다",
      body: "배송비는 1,000원이고, 총 결제 금액은 46,000원입니다.",
    });
    expect(formatSpy).toHaveBeenNthCalledWith(1, 1_000);
    expect(formatSpy).toHaveBeenNthCalledWith(2, 46_000);

    formatSpy.mockRestore();
    sendSpy.mockRestore();
  });
});
```

여기서 사용하는 matcher는 다음과 같습니다.

- `toHaveBeenCalledWith`: 특정 인자로 호출되었는지 확인합니다.
- `toHaveBeenCalledTimes`: 몇 번 호출되었는지 확인합니다.
- `toHaveBeenNthCalledWith`: 여러 번 호출된 함수의 순서별 인자를 확인합니다.

코드 끝의 `formatSpy.mockRestore()`, `sendSpy.mockRestore()`도 같이 읽어야 합니다. `spyOn(...)`은 기존 객체 메서드를 잠깐 감시 가능한 mock 상태로 바꾸기 때문에, 테스트가 끝난 뒤에는 `mockRestore()`로 원래 구현을 되돌려 두는 편이 안전합니다. 그렇지 않으면 앞 테스트가 남긴 spy 상태가 다음 테스트까지 이어질 수 있습니다. 이 단계에서는 개념을 먼저 익히는 구간이므로 테스트 끝에서 직접 복원하고, 뒤에서는 이런 정리를 `afterEach()` 같은 공통 정리로 올리는 방식도 보게 됩니다.

이 matcher들은 컨트롤러와 서비스가 어떤 순서로 어떤 값을 넘기는지 검증할 때 자주 사용됩니다.

### mockImplementation과 호출별 동작 변경

입력에 따라 다른 결과를 내야 하거나, 호출 순서에 따라 동작을 바꾸어야 한다면 `mockImplementation` 계열이 필요합니다. 여기서 가장 먼저 잡아야 할 선택 기준은 단순합니다. `mockReturnValue`는 결과만 고정할 때 쓰고, `mockImplementation`은 로직 자체를 직접 적어야 할 때 씁니다.

앞에서 본 `sendOrderMail`은 내부에서 `formatter.format(...)`을 두 번 호출합니다. 그래서 호출마다 다른 문자열을 돌려줘야 한다고 가정하면, 테스트 계획도 “첫 번째 호출”, “두 번째 호출”, “그 이후 기본 동작”을 순서대로 덮어쓰는 방식으로 세우는 편이 읽기 좋습니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { mailer } from "#src/orders/mailer.js";

// ... 생략 ...
const sendSpy = jest
  .spyOn(mailer, "send")
  .mockImplementation(async () => {});
```

여기서 `mockImplementation(...)`은 방금 잡은 spy에 실제 동작을 덧붙이는 역할을 합니다. 즉, `spyOn(...)`으로 기존 메서드를 감시하기 시작하고, `mockImplementation()`은 그 메서드가 호출되었을 때 무엇을 하게 만들지 정합니다.

위 예시는 이미 존재하는 `mailer.send` 메서드를 비동기 함수처럼 보이게 만드는 코드입니다. 실제 메일은 보내지 않지만, `async () => {}`를 넣어 두면 호출하는 쪽에서는 Promise를 반환하는 함수처럼 다룰 수 있습니다. 그래서 현재 코드가 `await mailer.send(...)`처럼 작성되어 있어도 테스트에서 같은 인터페이스를 유지할 수 있습니다. 이미 있는 객체 메서드를 다루는 상황에서는 이렇게 `spyOn`으로 기준을 통일해 두는 편이 읽기 좋습니다.

정리하면 두 도구의 역할은 이렇게 나눠서 읽는 편이 좋습니다.

- `jest.fn()`: 호출 기록을 추적할 수 있는 가짜 함수를 만듭니다.
- `mockImplementation(...)`: 그 가짜 함수가 실제로 어떤 동작을 할지 정의합니다.
- `mockReturnValue(...)`: 계산 없이 결과값만 고정합니다.

즉, `mockImplementation`은 항상 필요한 것이 아닙니다. 호출 여부만 보면 되는 상황에서는 `jest.fn()`만으로 충분합니다. 반대로 반환값, 비동기 동작, 입력별 분기까지 흉내 내야 할 때는 `mockImplementation()`이 필요해집니다.

호출 순서에 따라 결과를 바꾸고 싶다면 `mockImplementationOnce`와 `mockReturnValueOnce`를 사용할 수 있습니다. 같은 함수가 여러 번 호출되는 흐름을 테스트할 때, 각 호출마다 다른 결과를 만들어야 해서 이 메서드들이 필요합니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { formatter } from "#src/orders/formatter.js";

describe("sendOrderMail", () => {
  test("호출 순서에 따라 다른 동작을 줄 수 있다", () => {
    const formatSpy = jest.spyOn(formatter, "format");

    formatSpy
      .mockImplementationOnce(
        (amount: number) => `첫 호출:${amount}`,
      )
      .mockReturnValueOnce("두 번째 호출")
      .mockReturnValue("기본 반환값");

    expect(formatter.format(1_000)).toBe("첫 호출:1000");
    expect(formatter.format(2_000)).toBe("두 번째 호출");
    expect(formatter.format(3_000)).toBe("기본 반환값");
    expect(formatSpy).toHaveBeenCalledTimes(3);

    formatSpy.mockRestore();
  });
});
```

여기서 각 메서드는 역할이 다릅니다.

- `mockImplementationOnce(...)`는 **다음 한 번의 호출에만 적용되는 구현**을 붙입니다.
- `mockReturnValueOnce(...)`는 **다음 한 번의 호출에만 적용되는 고정 반환값**을 붙입니다.
- `mockReturnValue(...)`는 위의 `Once` 설정이 모두 소진된 뒤부터 계속 쓰일 **기본 반환값**을 정합니다.

즉, 위 코드는 호출 순서에 따라 아래처럼 동작합니다.

1. 첫 번째 호출: `amount`를 받아 `첫 호출:${amount}`를 반환합니다.
2. 두 번째 호출: 인자와 상관없이 `"두 번째 호출"`을 반환합니다.
3. 세 번째 호출부터: `"기본 반환값"`을 계속 반환합니다.

이 차이를 이해하면 언제 어떤 메서드를 써야 하는지도 더 분명해집니다.

- 입력값을 받아 계산하거나 분기하는 **실제 동작 자체**를 한 번만 바꾸고 싶다면 `mockImplementationOnce(...)`가 더 적합합니다.
- 계산은 필요 없고, 그 호출에서 나와야 하는 **결과값만 고정**하면 된다면 `mockReturnValueOnce(...)`가 더 단순합니다.

예를 들어 첫 번째 호출에서는 인자를 이용해 문자열을 만들고, 두 번째 호출에서는 그냥 미리 정한 텍스트만 돌려주고 싶을 수 있습니다. 이럴 때처럼 호출마다 성격이 다르면 `Once` 계열을 순서대로 붙이는 방식이 읽기 좋습니다.

비동기 함수도 같은 원리입니다. Promise를 반환하는 함수라면 `mockImplementationOnce(async () =&gt; ...)`, `mockResolvedValueOnce(...)`, `mockRejectedValueOnce(...)` 같은 변형을 같은 방식으로 사용할 수 있습니다.

결국 `Once`가 붙은 메서드는 모두 **다음 호출 한 번만 덮어쓴다**는 공통 규칙을 가집니다. 이 방식은 같은 함수가 여러 번 호출되는 흐름을 순서대로 검증할 때 특히 읽기 좋습니다.

### 선택 기준

세 도구를 고를 때는 이름보다 용도를 먼저 봐야 합니다.

| 상황 | 선택 |
| --- | --- |
| 완전히 새 가짜 함수가 필요합니다 | `jest.fn` |
| 이미 있는 객체의 메서드만 감시하거나 대체합니다 | `spyOn` |
| 입력값이나 호출 순서에 따라 동작을 바꿉니다 | `mockImplementation`, `mockImplementationOnce`, `mockReturnValueOnce` |

핵심은 가장 단순한 도구로 충분하면 거기서 멈추는 것입니다. 호출 여부만 보면 되는 상황에서 복잡한 가짜 구현까지 만들면 테스트가 과해집니다. 반대로 입력별 동작이 필요한데 고정 반환값만 두면 표현력이 부족해집니다.

### 정리

테스트 더블은 외부 의존성을 끊어 현재 대상만 확인하기 위한 도구입니다. `jest.fn`, `spyOn`, `mockImplementation`을 상황에 맞게 고를 수 있어야 컨트롤러와 서비스 테스트를 안정적으로 작성할 수 있습니다.

## **3-01 비동기 함수, Promise, 에러 테스트**

### 시작점

### 비동기 테스트가 흔들리는 이유

비동기 테스트는 Promise가 끝나는 시점을 정확히 기다리지 않으면 쉽게 흔들립니다. 이 장에서는 비동기 함수, Promise, 에러 검증, 테스트 생명주기, 반복 실행 도구를 이용해 그런 흔들림을 막는 기본기를 익힙니다.

3장 체크포인트도 2장과 마찬가지로 예제 파일을 `src/` 아래에 모아 둡니다. 그래서 아래 코드 블록은 모두 실제 체크포인트 구조를 그대로 따라갑니다.

### 비동기 테스트의 기본 규칙

Express 코드에서는 비동기 함수가 자주 등장합니다. 데이터 조회, 토큰 발급, 외부 API 호출, 파일 저장 같은 작업은 대부분 Promise를 반환합니다. 따라서 테스트도 이 흐름을 정확히 따라가야 합니다.

비동기 테스트에서 가장 먼저 지켜야 할 규칙은 비동기 작업이 끝날 때까지 기다리는 것입니다. 다만 그 전에, 어떤 Promise를 검증하려는지부터 먼저 보는 편이 읽기 좋습니다.

파일: `src/users/user-lookup.service.ts`

```tsx
export class UserLookupService {
  constructor(private readonly gateway: UserGateway) {}

  async getDisplayName(userId: string): Promise<string> {
    const name = await this.gateway.findNameById(userId);

    if (!name) {
      throw new Error("USER_NOT_FOUND");
    }

    return `${name} 님`;
  }
}
```

이 함수의 테스트 계획은 두 갈래입니다. `gateway.findNameById(...)`가 이름을 돌려주면 `"이름 님"`을 반환하는지, `null`을 돌려주면 `USER_NOT_FOUND` 에러를 던지는지 나눠서 확인하면 됩니다. 두 경우 모두 Promise가 끝날 때까지 기다려야 하므로, 이 절에서는 `await expect(...).resolves`와 `await expect(...).rejects`를 중심으로 읽겠습니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { UserLookupService } from "#src/users/user-lookup.service.js";
import { userGateway } from "#src/users/user-gateway.js";

describe("UserLookupService", () => {
  // ... 생략 ...

  test("Promise를 resolve해서 결과를 검증합니다", async () => {
    const findNameByIdSpy = jest
      .spyOn(userGateway, "findNameById")
      .mockResolvedValue("win");
    const service = new UserLookupService(userGateway);

    await expect(
      service.getDisplayName("user-1"),
    ).resolves.toBe("win 님");
    expect(findNameByIdSpy).toHaveBeenCalledWith("user-1");
  });

  // ... 생략 ...
});
```

`await` 없이 함수를 호출한 뒤 바로 검증하면, 아직 끝나지 않은 작업을 검사하게 됩니다. 이런 테스트는 환경에 따라 통과하기도 하고 실패하기도 해서 가장 위험합니다. 또한 이 장에서는 비동기 코드를 항상 `test(...)` 함수 안에서 `await`하는 방식으로 통일합니다. 그래야 테스트 범위와 기다려야 할 대상이 함께 드러납니다.

여기서 `resolves`는 Promise가 **성공적으로 끝난 뒤의 값**을 검증할 때 쓰는 연결어라고 보면 됩니다. 즉, `await expect(promise).resolves.toBe(...)`는 “이 Promise가 성공하면 그 결과가 무엇이어야 하는가”를 읽기 좋게 적는 방식입니다. 비동기 성공 경로를 검증할 때 가장 자주 쓰는 형태입니다.

### 성공 경로와 실패 경로

비동기 함수는 성공만 확인해서는 부족합니다. 같은 함수라도 성공 경로와 실패 경로가 완전히 다르게 움직이기 때문입니다.

예를 들어 사용자 이름을 조회하는 서비스가 사용자를 찾지 못했을 때 예외를 던진다면, 그 실패도 같은 수준으로 검증해야 합니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { UserLookupService } from "#src/users/user-lookup.service.js";
import { userGateway } from "#src/users/user-gateway.js";

describe("UserLookupService", () => {
  // ... 생략 ...

  test("Promise를 reject하는 에러 흐름을 검증합니다", async () => {
    jest
      .spyOn(userGateway, "findNameById")
      .mockResolvedValue(null);
    const service = new UserLookupService(userGateway);

    await expect(
      service.getDisplayName("missing"),
    ).rejects.toThrow("USER_NOT_FOUND");
  });

  // ... 생략 ...
});
```

`rejects.toThrow(...)`는 Promise가 실패했는지, 그리고 그 실패가 어떤 에러인지 함께 확인합니다. 비동기 실패를 검증할 때 가장 자주 쓰는 형태입니다.

### 동기 에러와 비동기 에러의 차이

테스트에서는 동기 에러와 비동기 에러를 구분해서 읽어야 합니다. 이 차이는 `SessionService`와 `UserLookupService`를 나란히 두면 더 잘 보입니다.

파일: `src/session/session.service.ts`

```tsx
export class SessionService {
  validateTimeout(timeoutMs: number): void {
    if (timeoutMs < 1_000) {
      throw new Error("MIN_TIMEOUT");
    }
  }

  getExpiryTime(now: number, timeoutMs: number): number {
    this.validateTimeout(timeoutMs);
    return now + timeoutMs;
  }
}
```

여기서 테스트 계획은 나뉩니다. `validateTimeout(0)`은 즉시 예외를 던지는 동기 함수이므로 `toThrow`로 읽고, `getDisplayName("missing")`은 Promise가 실패하는 비동기 함수이므로 `rejects.toThrow`로 읽어야 합니다. 즉, 이번 비교의 핵심은 에러 메시지보다 “언제 예외가 발생하는가”를 구분하는 데 있습니다.

개념 예시:

```tsx
import { jest } from "@jest/globals";
import { SessionService } from "#src/session/session.service.js";
import { userGateway } from "#src/users/user-gateway.js";
import { UserLookupService } from "#src/users/user-lookup.service.js";

describe("동기 에러와 비동기 에러", () => {
  test("toThrow와 rejects.toThrow를 구분합니다", async () => {
    const sessionService = new SessionService();
    jest
      .spyOn(userGateway, "findNameById")
      .mockResolvedValue(null);
    const userLookupService = new UserLookupService(
      userGateway,
    );

    expect(() => sessionService.validateTimeout(0)).toThrow(
      "MIN_TIMEOUT",
    );
    await expect(
      userLookupService.getDisplayName("missing"),
    ).rejects.toThrow("USER_NOT_FOUND");
  });
});
```

첫 번째는 즉시 예외를 던지는 동기 함수입니다. 두 번째는 Promise를 반환한 뒤 실패하는 비동기 함수입니다. 따라서 `toThrow`는 동기 함수에, `rejects.toThrow`는 비동기 함수에 쓴다고 먼저 구분해 두는 편이 안전합니다.

이 차이를 놓치면 `toThrow`가 동작하지 않거나, `rejects`를 붙여야 할 곳에서 일반 `expect`만 쓰는 실수를 하게 됩니다.

### 정리

비동기 테스트는 기다림과 실패 검증이 핵심입니다. 성공 경로와 실패 경로를 나누어 보고, 동기 에러와 비동기 에러를 구분할 수 있어야 이후 장의 핸들러 테스트와 통합 테스트도 흔들리지 않습니다.

## **3-02 describe와 공통 준비/정리 관리**

### 테스트 묶음과 생명주기

테스트 수가 늘어나면 한 파일 안에서도 구조가 필요합니다. 어떤 클래스의 어떤 메서드를 검증하는지 한눈에 보이고, 각 테스트가 서로의 상태를 오염시키지 않아야 하기 때문입니다.

이 절의 대상 코드는 앞 절과 같은 `UserLookupService`입니다. 다만 이번에는 서비스 로직 자체보다, 그 로직을 검증하는 테스트 파일을 어떤 묶음과 생명주기로 관리할지를 먼저 계획합니다. 즉, 성공 경로와 실패 경로를 어디에 묶을지, spy를 언제 새로 만들지, 공통 정리를 어디에 둘지를 보는 절입니다.

이때 사용하는 기본 구조가 `describe`와 생명주기 훅입니다.

파일 일부: `src/users/user-lookup.service.spec.ts`

```tsx
describe("UserLookupService", () => {
  // tests
});
```

이 구조가 있으면 테스트 파일을 펼쳤을 때 대상과 기능이 바로 읽힙니다.

### beforeEach, afterEach, setupFilesAfterEnv

Jest의 생명주기 훅은 테스트 전후에 공통 작업을 끼워 넣는 문법입니다. 테스트를 여러 개 묶어 실행할 때, 어떤 준비를 한 번만 할지, 어떤 준비를 테스트마다 다시 할지, 어떤 정리를 매번 해야 할지를 이 훅으로 나눕니다.

먼저 네 가지 훅의 의미부터 정확히 잡아 두는 편이 좋습니다.

- `beforeAll`: 현재 범위의 테스트가 시작되기 전에 한 번만 실행합니다.
- `beforeEach`: 각 테스트가 시작되기 전에 매번 실행합니다.
- `afterEach`: 각 테스트가 끝난 뒤에 매번 실행합니다.
- `afterAll`: 현재 범위의 테스트가 모두 끝난 뒤에 한 번만 실행합니다.

이 네 가지는 실행 시점이 다릅니다. 따라서 무엇을 어디에 둘지도 이 시점 차이로 판단합니다. 예를 들어 비용이 큰 초기화는 `beforeAll`에, 테스트마다 새로 맞춰야 하는 상태는 `beforeEach`에, 테스트가 남긴 흔적 정리는 `afterEach`에, 마지막 연결 해제는 `afterAll`에 두는 식입니다.

가장 작은 형태로 보면 아래처럼 읽습니다.

개념 예시:

```tsx
describe("some tests", () => {
  beforeAll(() => {
    // 한 번만 필요한 준비
  });

  beforeEach(() => {
    // 테스트마다 다시 맞출 상태
  });

  afterEach(() => {
    // 테스트마다 지워야 할 흔적
  });

  afterAll(() => {
    // 마지막 한 번만 필요한 정리
  });
});
```

실무에서는 보통 비용이 큰 공용 자원은 `beforeAll`과 `afterAll`에 두고, 테스트마다 달라지는 spy, mock, 입력 상태는 `beforeEach`와 `afterEach`에서 다룹니다. 예를 들어 외부 결제 클라이언트를 쓰는 서비스를 테스트할 때는 아래처럼 둘 수 있습니다.

개념 예시:

```tsx
describe("OrderService", () => {
  const paymentClient = createTestPaymentClient();
  const orderService = new OrderService(paymentClient);
  let chargeSpy: jest.SpiedFunction<
    typeof paymentClient.charge
  >;

  beforeAll(async () => {
    await paymentClient.connect();
  });

  beforeEach(() => {
    chargeSpy = jest.spyOn(paymentClient, "charge");
  });

  afterEach(() => {
    // spyOn(...)으로 바꾼 메서드를 원래 구현으로 되돌립니다.
    jest.restoreAllMocks();
  });

  afterAll(async () => {
    await paymentClient.disconnect();
  });

  test("결제 승인에 성공하면 주문을 완료합니다", async () => {
    chargeSpy.mockResolvedValue({
      approved: true,
      transactionId: "tx-1",
    });

    // 반환 객체 전체가 아니라, 여기 적은 필드들이 포함되는지만 확인합니다.
    await expect(
      orderService.placeOrder("order-1"),
    ).resolves.toMatchObject({
      status: "PAID",
      transactionId: "tx-1",
    });
  });

  test("결제가 거절되면 주문을 실패로 처리합니다", async () => {
    chargeSpy.mockResolvedValue({
      approved: false,
      transactionId: null,
    });

    await expect(
      orderService.placeOrder("order-2"),
    ).rejects.toThrow("PAYMENT_FAILED");
  });
});
```

여기서 `let chargeSpy: jest.SpiedFunction&lt;typeof paymentClient.charge&gt;;`는 `paymentClient.charge` 메서드를 `jest.spyOn(...)`으로 감쌌을 때 돌아오는 spy의 타입을 뜻합니다. 즉, `chargeSpy`가 원래 메서드 시그니처를 유지한 채 `mockResolvedValue(...)`, `mockImplementation(...)`, `toHaveBeenCalledWith(...)` 같은 spy 전용 API를 안전하게 쓰도록 TypeScript에 알려 주는 타입입니다. 한 테스트 안에서 `const chargeSpy = jest.spyOn(...)`처럼 바로 만들면 굳이 이 타입을 적지 않아도 되지만, 지금처럼 `beforeEach`에서 만들고 바깥 `let` 변수에 담아 여러 테스트에서 함께 쓰려면 타입을 먼저 적어 두는 편이 자연스럽습니다.

이 예시에서 `beforeAll`과 `afterAll`은 테스트 전체가 공유하는 외부 클라이언트 연결을 담당합니다. 반대로 `beforeEach`는 각 테스트에서 새로 쓸 spy를 준비하고, `afterEach`는 그 테스트가 바꾼 mock 상태를 다시 원래대로 돌립니다. 실제 프로젝트에서는 테스트 DB 연결, 메시지 큐 클라이언트, 외부 API 클라이언트처럼 생성 비용이 큰 자원을 이런 식으로 다루는 경우가 많습니다.

즉, 생명주기 훅의 핵심은 문법 이름을 외우는 일이 아니라, 어떤 준비와 정리가 어느 범위에 속하는지 구분하는 일입니다.

실제 테스트에서는 네 훅을 항상 모두 쓰는 것이 아니라, 현재 문제를 가장 단순하게 풀 수 있는 것부터 선택하면 됩니다. 이 장에서는 테스트마다 spy와 mock 상태가 섞이지 않게 만드는 일이 더 중요하므로, `beforeEach`와 `afterEach`, 그리고 공통 정리를 연결하는 `setupFilesAfterEnv`에 먼저 집중하겠습니다.

파일 일부: `src/users/user-lookup.service.spec.ts`

```tsx
import { jest } from "@jest/globals";
import { userGateway } from "#src/users/user-gateway.js";

describe("UserLookupService", () => {
  let findNameByIdSpy: jest.SpiedFunction<
    typeof userGateway.findNameById
  >;

  beforeEach(() => {
    findNameByIdSpy = jest.spyOn(
      userGateway,
      "findNameById",
    );
  });

  test("사용자 이름을 조회한다", async () => {
    // ... 생략 ...
  });
});
```

앞에서 본 `jest.SpiedFunction&lt;typeof ...&gt;`와 같은 이유로, 여기서도 `findNameByIdSpy`는 `userGateway.findNameById` 메서드를 감싼 spy라고 읽으면 됩니다. 그래서 `mockResolvedValue(...)`로 결과를 바꾸거나 `toHaveBeenCalledTimes(...)`로 호출 횟수를 검증할 때 원래 메서드 시그니처를 잃지 않습니다.

여기서 `beforeEach`는 각 테스트가 시작되기 전에 `findNameByIdSpy`를 새로 만드는 역할을 합니다. 즉, 성공 경로 테스트와 실패 경로 테스트가 같은 spy 인스턴스를 지저분하게 공유하지 않도록, 테스트 출발 상태를 매번 다시 맞추는 것입니다.

개별 스펙 파일에서는 보통 이렇게 `beforeEach`로 현재 파일 안에서 필요한 spy나 mock을 새로 준비합니다. 반대로 여러 테스트 파일에서 공통으로 반복되는 정리는 setup 파일로 모아 둘 수 있습니다.

- `beforeEach`: 현재 파일 안에서 필요한 초기 상태를 각 테스트 직전에 다시 맞춥니다.
- `setup.ts`: 여러 테스트 파일에서 공통으로 필요한 정리를 모아 둡니다.

여기서 `setupFilesAfterEnv`는 생명주기 훅 자체가 아닙니다. `afterEach(...)` 같은 공통 훅을 별도 파일에 적어 두고, 그 파일을 Jest가 자동으로 읽게 연결하는 설정입니다.

공통 정리는 별도 setup 파일로 모아 둘 수 있습니다. 다만 파일을 만들어 두기만 해서는 자동 실행되지 않습니다. `jest.config.mjs`에서 `setupFilesAfterEnv`로 연결해 주어야 합니다.

파일: `jest.config.mjs`

```tsx
export default {
  setupFilesAfterEnv: ["<rootDir>/src/setup.ts"],
};
```

이 설정이 있어야 Jest가 각 테스트 파일을 실행하기 전에 `setup.ts`를 먼저 읽습니다.

파일: `src/setup.ts`

```tsx
import { jest } from "@jest/globals";

afterEach(() => {
  // mock 함수의 호출 기록만 비웁니다.
  jest.clearAllMocks();
  // spyOn(...)으로 바꾼 메서드를 원래 구현으로 되돌립니다.
  jest.restoreAllMocks();
});
```

이 예제에서는 `afterEach` 공통 정리를 `src/setup.ts`에 분리해 둡니다. `jest.fn()`과 spy는 호출 기록을 남기기 때문에, 이 기록을 지우지 않으면 다음 테스트에서 이전 테스트 흔적이 남습니다. 개별 정리 메서드의 차이는 다음 절에서 따로 비교하겠습니다.

정리하면 현재 파일 안에서만 필요한 준비는 `beforeEach`에 두고, 여러 파일에 반복되는 정리는 `setupFilesAfterEnv`로 연결한 `setup.ts`로 올린다고 이해하면 됩니다.

### 상태 오염이 만드는 실패

생명주기 정리가 빠지면 테스트는 코드보다 상태 때문에 깨집니다.

예를 들어 첫 번째 테스트는 사용자를 찾는 성공 경로이고, 두 번째 테스트는 사용자를 찾지 못하는 실패 경로라고 가정해 보겠습니다.

파일: `src/users/user-lookup.service.spec.ts`

```tsx
import { jest } from "@jest/globals";
import { userGateway } from "#src/users/user-gateway.js";
import { UserLookupService } from "#src/users/user-lookup.service.js";

describe("UserLookupService", () => {
  const service = new UserLookupService(userGateway);
  let findNameByIdSpy: jest.SpiedFunction<
    typeof userGateway.findNameById
  >;

  beforeEach(() => {
    findNameByIdSpy = jest.spyOn(
      userGateway,
      "findNameById",
    );
  });

  // ... 생략 ...

  test("Promise를 resolve해서 결과를 검증합니다", async () => {
    findNameByIdSpy.mockResolvedValue("win");
    await service.getDisplayName("user-1");
    expect(findNameByIdSpy).toHaveBeenCalledTimes(1);
  });

  test("Promise를 reject하는 에러 흐름을 검증합니다", async () => {
    findNameByIdSpy.mockResolvedValue(null);
    await expect(
      service.getDisplayName("missing"),
    ).rejects.toThrow("USER_NOT_FOUND");
    // 공통 afterEach 정리가 없다면 이전 테스트 호출 기록이 남아서 이 줄이 2회로 집계되어 실패할 수 있습니다.
    expect(findNameByIdSpy).toHaveBeenCalledTimes(1);
  });

  // ... 생략 ...
});
```

여기서 `beforeEach`와 공통 `afterEach` 정리가 빠지면 두 번째 테스트는 로직이 아니라 첫 번째 테스트의 흔적 때문에 실패할 수 있습니다. 이런 실패는 원인 파악이 어렵고, 전체 실행에서만 나타나기 때문에 더 까다롭습니다.

지금까지 본 조각을 한 파일 안에서 합치면 `user-lookup.service.spec.ts`는 아래처럼 읽으면 됩니다.

파일: `src/users/user-lookup.service.spec.ts`

```tsx
import { jest } from "@jest/globals";
import { userGateway } from "#src/users/user-gateway.js";
import { UserLookupService } from "#src/users/user-lookup.service.js";

describe("UserLookupService", () => {
  const service = new UserLookupService(userGateway);
  let findNameByIdSpy: jest.SpiedFunction<
    typeof userGateway.findNameById
  >;

  beforeEach(() => {
    findNameByIdSpy = jest.spyOn(
      userGateway,
      "findNameById",
    );
  });

  test("Promise를 resolve해서 결과를 검증합니다", async () => {
    findNameByIdSpy.mockResolvedValue("win");

    await expect(
      service.getDisplayName("user-1"),
    ).resolves.toBe("win 님");
    expect(findNameByIdSpy).toHaveBeenCalledWith("user-1");
  });

  test("Promise를 reject하는 에러 흐름을 검증합니다", async () => {
    findNameByIdSpy.mockResolvedValue(null);

    await expect(
      service.getDisplayName("missing"),
    ).rejects.toThrow("USER_NOT_FOUND");
  });

  test("상태 오염을 막기 위해 각 테스트마다 호출 기록을 비웁니다", async () => {
    findNameByIdSpy.mockResolvedValue("win");
    await service.getDisplayName("user-1");
    expect(findNameByIdSpy).toHaveBeenCalledTimes(1);
  });
});
```

## **3-03 정리 메서드 차이**

앞 절에서는 `beforeEach`, `afterEach`, `setupFilesAfterEnv`로 공통 준비와 정리를 어디에 둘지 먼저 봤습니다. 그 정리 안에서 자주 쓰는 `clearAllMocks`, `restoreAllMocks`, `mockReset`의 차이도 같이 알아 두면 좋습니다.

- `clearAllMocks()`: 호출 기록만 초기화합니다.
- `restoreAllMocks()`: `spyOn(...)`으로 바꾼 메서드를 원래 구현으로 되돌립니다.
- `mockReset()`: 호출 기록과 구현을 함께 초기화하는 별도 선택지입니다.

같은 정리 함수라도 무엇이 되돌아가는지는 다릅니다. 가장 짧게 보면 아래처럼 구분할 수 있습니다.

개념 예시:

```tsx
describe("clearAllMocks", () => {
  test("호출 기록만 지우고 구현은 남깁니다", () => {
    const sendMail = jest.fn().mockReturnValue(true);

    sendMail("a@example.com");
    sendMail("b@example.com");

    jest.clearAllMocks(); // 호출 기록 삭제

    // jest가 기록해 둔 호출 인자 목록입니다.
    expect(sendMail.mock.calls).toEqual([]);
    expect(sendMail("c@example.com")).toBe(true);
  });
});
```

위 코드는 특정 파일의 일부가 아니라 `clearAllMocks()` 동작만 따로 보여 주는 개념 예시입니다. 여기서 `sendMail.mock.calls`는 Jest가 이 mock 함수를 어떤 인자로 몇 번 호출했는지 배열 형태로 기록해 둔 값입니다. 앞에서 두 번 호출한 흔적은 `clearAllMocks()` 뒤에 사라지지만, `mockReturnValue(true)`로 넣어 둔 구현은 그대로 남아 있으므로 다음 호출도 여전히 `true`를 돌려줍니다.

개념 예시:

```tsx
describe("restoreAllMocks", () => {
  test("spyOn으로 바꾼 메서드를 원래 구현으로 되돌립니다", () => {
    const auditLogger = {
      info(message: string) {
        return `real:${message}`;
      },
    };

    jest
      .spyOn(auditLogger, "info")
      .mockReturnValue("mocked");

    expect(auditLogger.info("saved")).toBe("mocked");

    jest.restoreAllMocks();

    expect(auditLogger.info("saved")).toBe("real:saved");
  });
});
```

위 코드는 특정 파일의 일부가 아니라 `restoreAllMocks()` 동작만 따로 보여 주는 개념 예시입니다. `spyOn(...)`으로 잠깐 덮어쓴 메서드는 테스트 중에는 `"mocked"`를 돌려주지만, `jest.restoreAllMocks()` 뒤에는 다시 원래 `info(...)` 구현이 실행됩니다.

개념 예시:

```tsx
describe("mockReset", () => {
  test("호출 기록과 구현을 함께 비웁니다", () => {
    const formatPrice = jest
      .fn()
      .mockReturnValue("1,000원");

    formatPrice(1000);
    // jest.fn()으로 만든 mock 함수는 남기고, 호출 기록과 mockReturnValue(...) 구현만 비웁니다.
    formatPrice.mockReset();

    expect(formatPrice.mock.calls).toEqual([]);
    expect(formatPrice(1000)).toBeUndefined();
  });
});
```

위 코드는 특정 파일의 일부가 아니라 `mockReset()` 동작만 따로 보여 주는 개념 예시입니다. `mockReset()`은 `jest.fn()`으로 만든 mock 함수 자체를 없애는 것이 아니라, 그 mock에 쌓인 호출 기록과 `mockReturnValue("1,000원")`로 넣어 둔 구현을 함께 비웁니다. 그래서 다시 호출하면 더 이상 `"1,000원"`을 돌려주지 않고 기본값인 `undefined`가 나옵니다.

### 정리

`describe`는 테스트 대상을 범위별로 묶고, `beforeEach`, `afterEach`, `setupFilesAfterEnv`는 각 테스트를 같은 출발 상태로 되돌리는 데 씁니다. 여기에 `clearAllMocks`, `restoreAllMocks`, `mockReset`까지 어떤 정리 메서드가 무엇을 되돌리는지 구분할 수 있어야 spy 호출 기록과 mock 구현이 다음 테스트로 새지 않습니다.

## **3-04 반복 테스트와 유연한 검증**

### 반복 테스트 줄이기

비슷한 테스트가 여러 개 나오기 시작하면 입력만 바뀌고 구조는 같은 경우가 많습니다. 이럴 때 같은 코드를 계속 중복해서 작성하면 테스트 파일이 길어지고, 어떤 점만 다른지 읽기도 어려워집니다.

먼저 반복 검증 대상이 되는 코드를 보겠습니다.

파일: `src/session/session.service.ts`

```tsx
export class SessionService {
  validateTimeout(timeoutMs: number): void {
    if (timeoutMs < 1_000) {
      throw new Error("MIN_TIMEOUT");
    }
  }

  extendDeadline(
    currentDeadline: number,
    extraMs: number,
  ): number {
    return currentDeadline + extraMs;
  }
}
```

여기서 `extendDeadline(...)`는 입력만 바뀌고 계산 규칙은 같기 때문에, 테스트 계획도 “여러 입력 조합을 같은 구조로 반복 검증한다”로 세우는 편이 자연스럽습니다. 이런 경우에는 같은 테스트를 반복해서 작성하기보다 `test.each`를 쓰는 편이 더 적합합니다.

이런 상황에서는 `test.each`가 유용합니다. `test.each`는 같은 로직을 여러 입력값으로 검증할 때 쓰는 Jest 문법입니다.

파일: `src/session/session.service.spec.ts`

```tsx
import { SessionService } from "#src/session/session.service.js";

describe("SessionService", () => {
  test.each([
    {
      currentDeadline: 1_000,
      extraMs: 10_000,
      expected: 11_000,
    },
    {
      currentDeadline: 2_500,
      extraMs: 5_000,
      expected: 7_500,
    },
    {
      currentDeadline: 30_000,
      extraMs: 0,
      expected: 30_000,
    },
  ])(
    // 테스트 이름입니다. 각 행의 값이 $currentDeadline, $extraMs, $expected 자리에 들어갑니다.
    "$currentDeadline에 $extraMs를 더하면 $expected가 된다",
    ({ currentDeadline, extraMs, expected }) => {
      const sessionService = new SessionService();

      expect(
        sessionService.extendDeadline(
          currentDeadline,
          extraMs,
        ),
      ).toBe(expected);
    },
  );
});
```

이 방식은 입력값과 예상 결과만 표처럼 남기고, 공통 구조는 한 번만 쓰게 해 줍니다.

여기서는 각 행을 배열이 아니라 객체로 적고, 테스트 제목에서도 `$currentDeadline`, `$extraMs`, `$expected`처럼 속성 이름을 그대로 씁니다. 이 방식은 “이 값이 첫 번째 인자인가, 두 번째 인자인가”를 따로 세지 않아도 되므로, 교재 예시에서는 더 읽기 쉽습니다.

개념 예시:

```tsx
describe("buildLabel", () => {
  test.each([
    { kind: "free", expected: "무료" },
    { kind: "paid", expected: "유료" },
  ])(
    "$kind 상품은 $expected 라벨을 반환합니다",
    ({ kind, expected }) => {
      expect(buildLabel(kind)).toBe(expected);
    },
  );
});
```

위 코드는 특정 파일의 일부가 아니라, `test.each`에서 객체 행과 `$변수이름` 제목 템플릿을 어떻게 읽는지만 따로 보여 주는 개념 예시입니다. 즉, `test.each`를 객체 행으로 쓰면 제목도 `$변수이름` 형태로 자연스럽게 읽을 수 있고, 테스트 함수 인자도 구조 분해로 바로 꺼내 쓸 수 있습니다.

### expect.any와 유연한 검증

실행할 때마다 달라지는 값은 `expect.any`와 잘 맞습니다. 값 하나하나를 고정하지 않고도 필요한 형태만 확인하고 싶을 때 자주 사용합니다.

먼저 어떤 유틸의 형태를 검증하는지부터 보겠습니다.

파일: `src/shared/create-mock-req-res.ts`

```tsx
import { jest } from "@jest/globals";

export function createMockReqRes() {
  const req = {
    body: {},
    params: {},
    query: {},
    headers: {},
  };

  const res = {
    status: jest.fn().mockReturnThis(),
    json: jest.fn().mockReturnThis(),
    send: jest.fn().mockReturnThis(),
    set: jest.fn().mockReturnThis(),
    redirect: jest.fn().mockReturnThis(),
    cookie: jest.fn().mockReturnThis(),
  };

  return { req, res };
}
```

이 유틸의 테스트 계획은 각 함수 구현을 하나씩 다시 검증하는 것이 아닙니다. `req`와 `res`가 테스트에서 바로 쓸 수 있는 형태로 준비되는지, 특히 `body`, `params`, `status`, `json`, `cookie` 같은 핵심 속성과 메서드가 들어 있는지만 확인하면 충분합니다. 그래서 이 절에서는 값 전체를 고정하기보다 `expect.any(...)`로 형태를 검증합니다.

파일: `src/shared/create-mock-req-res.spec.ts`

```tsx
import { createMockReqRes } from "#src/shared/create-mock-req-res.js";

describe("createMockReqRes", () => {
  test("req와 res의 기본 모양을 확인한다", () => {
    const { req, res } = createMockReqRes();

    expect(req).toMatchObject({
      body: expect.any(Object),
      params: expect.any(Object),
    });
    expect(res).toMatchObject({
      status: expect.any(Function),
      json: expect.any(Function),
      cookie: expect.any(Function),
    });
  });
});
```

이 matcher는 값 전체를 고정하기보다 타입이나 형식만 확인합니다. 그래서 실행할 때마다 달라질 수 있는 객체나, 지금 당장 중요한 필드만 짚어 보고 싶은 준비 코드와 잘 맞습니다.

### 정리

입력 축만 달라질 때는 `test.each`로 구조를 한 번만 쓰고, 값 전체를 고정할 필요가 없을 때는 `expect.any`로 필요한 형태만 확인하며, 여러 파일에서 같은 정리가 반복되면 setup 파일로 모을 수 있습니다. 예를 들어 `afterEach`에서 mock 호출 기록을 지우고 spy를 원래 구현으로 되돌리는 작업은 setup 파일에 두는 편이 읽기 좋습니다. 이 세 기준을 구분하면 테스트를 줄이는 것보다 먼저 무엇을 반복하고 무엇을 유연하게 볼지 판단이 선명해집니다.

# 제 4장: Express 핸들러와 미들웨어 단위 테스트

## **4-01 첫 Express 핸들러 테스트 만들기**

### 시작점

4장 본편은 `codeit-fs-jest-starter/04-express-foundation`을 기준으로 읽습니다.

### 핸들러 단위 테스트가 확인하는 것

4장부터는 제품 코드는 `src/`, 테스트 코드는 `test/` 아래에 나누어 두는 구조로 읽겠습니다.

여기서 먼저 용어를 짧게 정리하고 가겠습니다. 이 장에서 다루는 단위 테스트(unit test)는 컨트롤러나 미들웨어처럼 계층 하나의 책임을 좁게 검증하는 테스트입니다. 지금은 요청을 받아 서비스 호출과 응답을 정리하는 흐름을 계층 하나 안에서 확인하는 데 집중합니다. 실제 Express 요청/응답 흐름 전체를 확인하는 통합 테스트(integration test)는 6장에서 따로 다룹니다. E2E 테스트(end-to-end test)는 그보다 더 넓게 브라우저나 외부 시스템까지 포함할 수 있습니다.

컨트롤러 메서드는 서버를 띄우지 않고도 테스트할 수 있습니다. 여기서 확인하는 대상은 URL 연결이나 Express 라우팅 전체가 아니라, 핸들러 메서드 하나가 요청을 받아 응답을 끝내는 방식입니다. 그래서 `app.listen()`이나 실제 HTTP 요청 도구를 쓰지 않고, 컨트롤러 인스턴스를 만든 뒤 메서드를 직접 호출합니다.

이 방식이 중요한 이유는 컨트롤러의 책임을 좁혀서 볼 수 있기 때문입니다. 컨트롤러는 보통 입력을 꺼내고, 서비스 계층을 부르고, 상태 코드와 응답 본문을 정리해 응답을 끝냅니다. 반대로 로그인 규칙 자체나 비밀번호 비교 규칙 같은 것은 서비스의 책임입니다. 컨트롤러 테스트에서 그 규칙까지 다시 검증하기 시작하면, 어느 계층이 무엇을 맡는지 금방 흐려집니다.

즉, 컨트롤러 테스트는 `입력 -&gt; 서비스 호출 -&gt; 응답` 흐름을 검증하는 테스트라고 먼저 잡아 두면 읽기가 훨씬 쉬워집니다.

### AuthController 예제로 보는 첫 테스트

`AuthController`의 `login` 메서드는 요청 본문에서 `email`, `password`를 꺼내고, `AuthService.login`을 호출한 뒤, 성공하면 `uid` 쿠키를 심고 사용자 정보를 응답으로 보냅니다. 이 흐름은 서버를 띄우지 않아도 충분히 검증할 수 있습니다.

이 장의 예시에서는 `req`, `res`를 준비할 때 `node-mocks-http`를 사용합니다. 이 라이브러리는 Express의 `req`, `res`를 테스트용 객체로 만들어 주는 도구입니다. 즉, `createRequest()`, `createResponse()`로 요청과 응답을 준비한 뒤 실제 서버를 띄우지 않고도 `req.body`, `req.cookies`, `res.status(...)`, `res.json(...)` 같은 흐름을 바로 확인할 수 있게 해 줍니다.

직접 객체를 손으로 만들 수도 있지만, 그렇게 시작하면 테스트 본문보다 준비 코드가 더 길어지기 쉽습니다. 특히 `res.status(...).json(...)`처럼 메서드 체이닝이 들어가는 응답 객체는 매번 같은 모양을 다시 흉내 내야 합니다. `node-mocks-http`는 이 반복을 줄여 주기 때문에, 4장에서는 응답 객체를 만드는 방법보다 **컨트롤러가 어떤 입력을 받고 어떤 응답으로 끝나는지**에 더 집중할 수 있습니다.

아래 설치 명령도 이 폴더에서 실행한다고 보면 됩니다.

터미널

```bash
pnpm add -D node-mocks-http
```

핸들러 첫 테스트에서 가장 중요한 협력 객체는 `AuthService` 하나입니다. 컨트롤러 테스트의 초점은 서비스 내부 규칙이 아니라, 컨트롤러가 입력을 받고 서비스 호출과 응답으로 이어지는지 확인하는 데 있기 때문입니다. 여기서는 그 역할을 **미리 정해 둔 값을 돌려주는 가짜 서비스 객체**로 바꿔 두고 읽으면 됩니다.

완성된 체크포인트에서는 이런 가짜 서비스를 `test/stub/services/auth.service.ts`의 `StubAuthService`로 분리해 둡니다. 여기서 `stub`은 테스트에서 협력 객체를 진짜 구현 대신 예측 가능한 값으로 동작하게 바꿔 둔 가짜 객체라고 읽으면 됩니다.

실제 체크포인트에서는 이 가짜 서비스를 아래처럼 클래스로 분리해 둡니다.

파일: `test/stub/services/auth.service.ts`

```tsx
import { jest } from "@jest/globals";
import { AuthService } from "#src/services/auth.service.js";
import { StubPasswordProvider } from "#test/stub/providers/password.provider.js";
import { StubUserRepository } from "#test/stub/repository/user.repository.js";

export class StubAuthService extends AuthService {
  constructor() {
    super(
      new StubUserRepository(),
      new StubPasswordProvider(),
    );
  }

  override login = jest
    .fn<AuthService["login"]>()
    .mockResolvedValue(null);

  override getUserProfile = jest
    .fn<AuthService["getUserProfile"]>()
    .mockResolvedValue(null);
}
```

여기서 `override`는 부모 클래스인 `AuthService`에 이미 있는 메서드를, 이 자식 클래스에서 다시 정의한다는 뜻입니다. 즉, `StubAuthService`는 진짜 `AuthService`를 상속하지만, 테스트에서 바로 조절하고 싶은 `login(...)`, `getUserProfile(...)`만 `jest.fn()` 기반 mock 메서드로 덮어쓴 클래스라고 읽으면 됩니다.

그래서 컨트롤러 테스트에서는 `StubAuthService`와 `AuthController`를 공통 준비로 두고, 각 테스트에서 필요한 결과만 `authService.login.mockResolvedValue(...)`로 다시 정해 주면 됩니다. 실제 spec 파일에서는 이런 공통 준비를 `beforeEach()`로 올리고, 테스트 안에는 그 테스트에서 달라지는 `req.body`, `res`, 반환값 설정만 남기는 편이 읽기 쉽습니다. `override`를 붙여 두면 TypeScript도 “부모 클래스에 같은 이름의 메서드가 실제로 있어야 한다”는 점을 함께 확인해 주므로, 오타나 잘못된 메서드명으로 덮어쓰는 실수도 줄일 수 있습니다.

아래 코드는 로그인 성공 경로를 테스트하는 핵심 부분입니다.

파일: `test/unit/controllers/auth.controller.spec.ts`

```tsx
import httpMocks from "node-mocks-http";
import { AuthController } from "#src/controllers/auth.controller.js";
import { StubAuthService } from "#test/stub/services/auth.service.js";

describe("AuthController.login", () => {
  let authService: StubAuthService;
  let controller: AuthController;
  let req = httpMocks.createRequest();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    authService = new StubAuthService();
    controller = new AuthController(authService);
    req = httpMocks.createRequest();
    res = httpMocks.createResponse();
  });

  test("로그인 성공 시 서비스 호출 뒤 사용자 응답을 반환한다", async () => {
    authService.login.mockResolvedValue({
      id: 1,
      email: "teacher@example.com",
      name: "Teacher",
    });
    req.body = {
      email: "teacher@example.com",
      password: "pw",
    };

    await controller.login(req, res);

    expect(authService.login).toHaveBeenCalledWith(
      "teacher@example.com",
      "pw",
    );
    expect(res.statusCode).toBe(200);
    expect(res._getJSONData()).toEqual({
      user: {
        id: 1,
        email: "teacher@example.com",
        name: "Teacher",
      },
    });
  });

  // ... 생략 ...
});
```

여기서 먼저 봐야 할 것은 세 가지입니다. 요청 본문에서 필요한 값을 꺼냈는지, 서비스 메서드를 기대한 인자로 불렀는지, 응답이 올바르게 끝났는지입니다. 이 세 가지가 맞으면 컨트롤러가 맡은 HTTP 책임은 일단 제대로 수행된 것입니다. 가장 흔한 실패도 이 범위 안에 있습니다. 서비스 호출이 빠지거나, `status(...)`가 빠지거나, `json(...)`이 호출되지 않거나, 필요한 쿠키가 누락되는 경우입니다.

완성된 체크포인트를 보면 테스트 자산이 `test/mock`, `test/stub`, `test/unit`으로 나뉘어 있습니다. 여기서 `mock`은 고정 데이터 같은 테스트 자산이고, `stub`은 협력 객체를 예측 가능한 기본 동작으로 바꿔 둔 테스트 자산입니다. `spy`는 이렇게 준비된 객체나 실제 메서드가 어떻게 호출됐는지를 추적하는 도구입니다. 4장에서는 이 구조를 모두 외우기보다, 컨트롤러 테스트가 `입력 -&gt; 서비스 호출 -&gt; 응답` 흐름에 집중한다는 감각부터 먼저 잡으면 됩니다.

### 서비스 전체를 다시 검증하지 않는 이유

핸들러 테스트를 처음 만들 때는 서비스의 내부 구현까지 같이 확인하고 싶은 마음이 들 수 있습니다. 하지만 이 단계에서는 서비스가 어떤 과정을 거쳐 로그인 성공 여부를 판단하는지보다, 컨트롤러가 서비스 결과를 어떻게 HTTP 응답으로 바꾸는지가 더 중요합니다.

그래서 컨트롤러 테스트에서는 서비스 결과를 미리 정해 둔 값으로 두는 편이 좋습니다. 성공 경로를 보고 싶으면 성공하는 값을 돌려주고, 실패 경로를 보고 싶으면 `null`을 돌려주게 만듭니다. 이렇게 하면 컨트롤러가 어떤 입력을 받고 어떤 응답을 만드는지에만 집중할 수 있습니다.

### 실패가 보이는 방식

핸들러 단위 테스트가 실패할 때는 보통 패턴이 분명합니다. 서비스 메서드가 호출되지 않거나, 응답 코드가 다르거나, 응답 본문이 비어 있거나, 필요한 쿠키가 빠지는 식입니다. 이런 실패는 대부분 컨트롤러의 HTTP 책임이 기대와 어긋났다는 뜻입니다.

반대로 `res.json is not a function` 같은 오류가 나오면 아직 컨트롤러 로직을 의심할 단계가 아닙니다. 그 경우는 대개 테스트용 `res` 객체가 충분히 준비되지 않았다는 뜻입니다.

### 정리

핸들러 단위 테스트는 서버 전체를 띄우는 테스트가 아닙니다. 컨트롤러 인스턴스를 만들고 메서드를 직접 호출해서, 입력 해석, 서비스 호출, 응답 종료를 확인하는 테스트입니다. 이 기준이 잡혀 있어야 `req`, `res`, `next`를 어떤 수준으로 준비해야 하는지도 더 분명해집니다.

## **4-02 req, res, next 모킹하기**

### 최소 mock 객체의 기준

핸들러를 직접 호출하려면 `req`, `res`, `next`를 테스트 안에서 준비해야 합니다. 여기서 중요한 기준은 실제 Express 객체를 전부 재현하는 것이 아니라, 현재 테스트 대상이 실제로 사용하는 부분만 준비하는 것입니다. 핸들러가 `req.body`만 읽는다면 `body`만 있으면 되고, 응답에서 `status`와 `json`만 쓴다면 그 메서드만 있으면 됩니다.

이 기준을 놓치면 테스트가 금방 무거워집니다. 아직 필요하지 않은 필드까지 한꺼번에 흉내 내려 하면, 테스트가 실제 로직보다 준비 코드에 더 많이 끌려가기 때문입니다.

### node-mocks-http로 준비하기

이 장에서는 `req`, `res`를 손으로 전부 만들지 않고 `node-mocks-http`로 요청과 응답 객체를 준비합니다. 앞 절에서 본 `test/unit/controllers/auth.controller.spec.ts` 예시도 같은 기준으로 읽으면 됩니다. 여기서는 전체 파일을 다시 보이기보다, 그 파일 안에서 `req`, `res`, `next`를 어떻게 준비하는지만 짧게 다시 짚겠습니다.

핵심 준비 부분만 다시 보면 아래 정도입니다.

파일 일부: `test/unit/controllers/auth.controller.spec.ts`

```tsx
describe("AuthController.login", () => {
  let req = httpMocks.createRequest();
  let res = httpMocks.createResponse();
  const next = jest.fn();

  beforeEach(() => {
    req = httpMocks.createRequest();
    res = httpMocks.createResponse();
    next.mockReset();
  });
});
```

이 방식의 장점은 분명합니다. `req.body`, `req.cookies`처럼 필요한 값만 테스트 안에서 바로 채울 수 있고, `res.status(...).json(...)`처럼 Express 응답 객체가 기대하는 메서드도 이미 들어 있습니다. 즉, 별도 공통 helper를 하나 더 만들지 않아도 테스트 레이어 구조가 직접 드러납니다.

### 체이닝과 응답 기록

Express 응답 객체는 `res.status(...).json(...)`처럼 체이닝을 자주 사용합니다. 그래서 `status`와 `json`은 둘 다 자기 자신을 다시 반환해야 합니다. 이 반환이 없으면 첫 메서드는 호출되더라도 뒤 메서드가 이어지지 않습니다.

`node-mocks-http`를 쓰면 체이닝을 위한 기본 응답 메서드를 직접 다시 만들 필요가 없습니다. 대신 테스트에서는 최종 상태 코드, JSON 본문, 쿠키처럼 “핸들러가 응답을 어떻게 끝냈는가”를 읽는 쪽에 집중하면 됩니다.

앞 장의 작은 유틸 예시처럼 응답 객체를 손으로 만들 때는 `status`, `json`, `send`가 자기 자신을 다시 돌려주도록 `mockReturnThis()`를 붙여 체이닝을 흉내 내야 했습니다. 4장 이후에는 그 부담을 줄이기 위해 `node-mocks-http`를 사용한다고 이해하면 됩니다.

따라서 상태 코드와 JSON 본문을 읽는 흐름도 바로 앞에서 본 로그인 성공 예시와 같습니다. `req.body`에 필요한 입력만 채우고 `await controller.login(req, res)`를 호출한 뒤, `res.statusCode`, `res._getJSONData()`를 읽으면 됩니다.

### next를 언제 준비해야 하는가

컨트롤러 테스트에서는 `next`를 아예 쓰지 않는 경우도 많습니다. 그래서 `next`는 `jest.fn()`으로 간단히 준비해 두고, 실제로 필요한 테스트에서만 검증하면 됩니다. 중요한 것은 `req`, `res`, `next`를 매 테스트마다 새로 만들고, 필요한 값만 그 테스트 안에서 채우는 일입니다.

### 정리

`req`, `res`, `next`를 준비할 때는 실제 Express 전체를 흉내 낼 필요가 없습니다. 현재 테스트 대상이 쓰는 필드와 메서드만 있으면 충분합니다. 4장 이후에는 그 준비를 `node-mocks-http`와 `jest.fn()`으로 직접 드러내는 쪽이, 테스트 레이어 구조를 이해하기에도 더 좋습니다.

## **4-03 상태 코드와 응답 종료 검증하기**

현재 테스트는 응답 객체에 최종 상태를 기록해 두기 때문에, 상태 코드와 본문을 아래처럼 읽을 수 있습니다.

### 쿠키처럼 본문 밖의 응답도 함께 봐야 한다

로그인 성공 경로는 상태 코드와 본문만으로 끝나지 않습니다. 현재 코드는 로그인한 사용자의 ID를 `uid` 쿠키로 심습니다. 이 동작은 컨트롤러의 HTTP 책임에 포함되므로, 단위 테스트에서도 같이 확인해야 합니다.

### BaseController와 에러 처리 읽기

현재 구조에서 `BaseController`는 응답 helper를 모아 두는 클래스가 아닙니다. 역할은 `router`를 들고 있고, 각 컨트롤러가 `routes()`를 구현하도록 맞추는 데 있습니다.

파일: `src/controllers/base.controller.ts`

```tsx
import { Router } from "express";

export abstract class BaseController {
  public readonly router: Router;

  constructor() {
    this.router = Router();
  }

  public abstract routes(): Router;
}
```

즉, 컨트롤러 테스트에서는 `BaseController`의 helper를 다시 검증하는 흐름이 아니라, 각 컨트롤러 메서드가 요청을 읽고 응답을 어떻게 끝내는지 직접 봅니다. 공통 에러 처리는 `BaseController`가 아니라 `App`이 등록하는 `errorHandler` 미들웨어에서 맡습니다. 그래서 이 장의 테스트도 helper 내부 구현보다 `statusCode`, `body`, `cookies` 같은 최종 응답 결과를 중심으로 읽는 편이 자연스럽습니다.

### 정리

핸들러 단위 테스트에서는 상태 코드와 응답 본문을 함께 검증해야 합니다. 로그인처럼 쿠키를 심는 흐름이 있으면 그 부분도 같이 확인해야 합니다. 현재 구조에서는 `BaseController`가 라우터 계약을 맡고, 응답과 에러 처리는 각 컨트롤러와 `errorHandler`가 나누어 맡습니다. 그래서 테스트에서는 내부 helper보다 최종 응답 결과를 중심으로 보는 편이 더 좋습니다.

## **4-04 next 호출과 인증 미들웨어 흐름 검증하기**

### 미들웨어에서 next가 의미하는 것

미들웨어 테스트는 컨트롤러 테스트와 보는 지점이 다릅니다. 컨트롤러는 응답을 어떻게 끝냈는지를 중심으로 보지만, 미들웨어는 요청을 다음 단계로 넘겼는지, 아니면 중간에서 차단했는지를 먼저 봅니다. 그래서 미들웨어 테스트에서는 `next`가 중요한 기준이 됩니다.

이 장의 예제 코드에서 `RequireAuthMiddleware`는 인증 정보가 유효하면 `next()`를 호출하고, 그렇지 않으면 `401` 응답으로 흐름을 끝냅니다.

실제 런타임 코드에서는 이 미들웨어를 라우트에 `requireAuthMiddleware.handle` 형태로 직접 등록합니다. 반면 단위 테스트에서는 라우터 전체를 거치지 않고 `middleware.handle(req, res, next)`를 직접 호출해, 현재 미들웨어의 판단만 좁게 확인합니다.

파일: `src/middlewares/require-auth.middleware.ts`

```tsx
const rawUserId = req.cookies?.uid;
const userId = Number.parseInt(rawUserId ?? "", 10);
if (!Number.isInteger(userId) || userId <= 0) {
  return res
    .status(401)
    .json({ message: "로그인이 필요합니다." });
}

const user = await this.userRepository.findById(userId);
if (!user) {
  return res
    .status(401)
    .json({ message: "사용자를 찾을 수 없습니다." });
}

req.user = {
  id: user.id,
  email: user.email,
  name: user.name,
};

return next();
```

이 구조를 보면 미들웨어 테스트에서 무엇을 확인해야 하는지도 자연스럽게 정해집니다.

여기서 `AuthenticatedRequest`는 Express의 기본 `Request`에 `user` 필드를 추가한 타입입니다. 현재 미들웨어는 인증이 끝나면 `req.user`를 채우기 때문에, 테스트에서도 요청 객체를 이 타입으로 다루어야 `req.user`를 읽고 검증할 수 있습니다. `node-mocks-http`의 `createRequest&lt;T&gt;()`는 제네릭을 받을 수 있으므로, 이 교재에서는 `httpMocks.createRequest&lt;AuthenticatedRequest&gt;()`처럼 요청을 만드는 시점에 타입을 함께 주는 쪽으로 읽겠습니다. 런타임 객체를 바꾸는 것이 아니라, TypeScript에게 “이 요청은 `user` 필드까지 가질 수 있는 요청으로 보자”라고 알려 주는 타입 정보입니다.

### 정상 흐름 검증

정상 흐름에서는 `uid` 쿠키가 있고, 그 값으로 찾은 사용자도 있어야 합니다. 이때 미들웨어는 `req.user`를 준비한 뒤 `next()`를 호출합니다.

파일 일부: `test/unit/middlewares/require-auth.middleware.spec.ts`

```tsx
import { jest } from "@jest/globals";
import httpMocks from "node-mocks-http";
import type { AuthenticatedRequest } from "#src/middlewares/require-auth.middleware.js";
import { RequireAuthMiddleware } from "#src/middlewares/require-auth.middleware.js";
import { teacherUserRecord } from "#test/mock/users.js";
import { StubUserRepository } from "#test/stub/repository/user.repository.js";

describe("RequireAuthMiddleware.handle", () => {
  let userRepository: StubUserRepository;
  let middleware: RequireAuthMiddleware;
  let next: ReturnType<typeof jest.fn>;
  let req =
    httpMocks.createRequest<AuthenticatedRequest>();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    userRepository = new StubUserRepository();
    middleware = new RequireAuthMiddleware(userRepository);
    next = jest.fn();
    // 이 미들웨어는 통과 시 req.user를 채우므로, 요청을 만들 때부터 그 타입을 함께 줍니다.
    req = httpMocks.createRequest<AuthenticatedRequest>();
    res = httpMocks.createResponse();
  });

  test("정상 흐름이면 req.user를 채우고 next를 호출한다", async () => {
    userRepository.findById.mockResolvedValue(
      teacherUserRecord,
    );
    req.cookies = { uid: "1" };

    await middleware.handle(req, res, next);

    expect(next).toHaveBeenCalled();
    expect(req.user).toEqual({
      id: 1,
      email: "teacher@example.com",
      name: "Teacher",
    });
  });

  // ... 생략 ...
});
```

여기서 중요한 것은 단순히 `next`가 호출되었다는 사실만이 아닙니다. 다음 단계로 넘기기 전에 요청 객체가 어떤 상태로 준비되었는지도 같이 봐야 합니다. 현재 코드에서는 `req.user`가 바로 그 준비 결과입니다.

### 차단 흐름 검증

실패 경로는 `next`를 호출하지 않고 응답을 끝내야 합니다. 예를 들어 쿠키가 없으면 아래처럼 확인합니다.

파일 일부: `test/unit/middlewares/require-auth.middleware.spec.ts`

```tsx
import { jest } from "@jest/globals";
import httpMocks from "node-mocks-http";
import type { AuthenticatedRequest } from "#src/middlewares/require-auth.middleware.js";
import { RequireAuthMiddleware } from "#src/middlewares/require-auth.middleware.js";
import { StubUserRepository } from "#test/stub/repository/user.repository.js";

describe("RequireAuthMiddleware.handle", () => {
  let userRepository: StubUserRepository;
  let middleware: RequireAuthMiddleware;
  let next: ReturnType<typeof jest.fn>;
  let req =
    httpMocks.createRequest<AuthenticatedRequest>();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    userRepository = new StubUserRepository();
    middleware = new RequireAuthMiddleware(userRepository);
    next = jest.fn();
    req = httpMocks.createRequest<AuthenticatedRequest>();
    res = httpMocks.createResponse();
  });

  test("uid 쿠키가 없으면 401로 차단한다", async () => {
    await middleware.handle(req, res, next);

    expect(res.statusCode).toBe(401);
    expect(res._getJSONData()).toEqual({
      message: "로그인이 필요합니다.",
    });
    expect(next).not.toHaveBeenCalled();
  });

  // ... 생략 ...
});
```

`uid` 쿠키는 있지만 사용자를 찾지 못하는 경우도 비슷합니다.

파일: `test/unit/middlewares/require-auth.middleware.spec.ts`

```tsx
import { jest } from "@jest/globals";
import httpMocks from "node-mocks-http";
import type { AuthenticatedRequest } from "#src/middlewares/require-auth.middleware.js";
import { RequireAuthMiddleware } from "#src/middlewares/require-auth.middleware.js";

    httpMocks.createRequest<AuthenticatedRequest>();
```

즉, 인증 미들웨어 테스트의 핵심은 `next()`를 호출해야 할 경로와 호출하면 안 되는 경로를 분명히 나누는 데 있습니다.

### next(error)와의 관계

이 장의 예제는 `next(error)`까지 자세히 다루지 않습니다. 다만 미들웨어에서 `next(error)`는 에러 핸들러로 흐름을 넘길 때 사용한다는 점은 같이 기억해 두는 편이 좋습니다. 지금 단계에서는 정상 흐름이면 `next()`, 차단 흐름이면 응답 종료라는 기준만 단단히 잡아 두면 충분합니다.

### 정리

미들웨어 테스트는 컨트롤러 테스트와 달리 `next`의 호출 여부가 핵심입니다. 정상 흐름에서는 `req.user` 같은 준비 결과를 남긴 뒤 `next()`를 호출해야 하고, 실패 경로에서는 `401` 응답으로 끝내며 `next()`를 호출하지 않아야 합니다. 이 기준이 잡히면 미들웨어의 역할이 훨씬 선명해집니다.

## **4-05 미들웨어 테스트 전략**

### 핸들러 테스트와 미들웨어 테스트의 차이

핸들러와 미들웨어는 모두 `req`, `res`, `next`를 다루지만, 테스트에서 확인해야 하는 대상은 다릅니다. 핸들러 테스트에서는 입력을 어떻게 읽고 어떤 응답으로 끝냈는지가 중심입니다. 반면 미들웨어 테스트에서는 요청을 통과시켰는지, 차단했는지가 중심입니다.

이 차이를 분리하지 않으면 테스트가 금방 길어집니다. 예를 들어 인증 미들웨어 테스트에서 로그인 비밀번호 검증 규칙까지 같이 보려고 하면, 이미 서비스 책임까지 끌어온 셈이 됩니다. 반대로 컨트롤러 테스트에서 인증 미들웨어가 `req.user`를 어떻게 준비하는지까지 자세히 검증하려 하면, 컨트롤러 테스트가 미들웨어 내부 구현에 기대게 됩니다.

### 핸들러 테스트와 미들웨어 테스트의 범위

현재 4장의 범위에서 핸들러 테스트는 아래 정도면 충분합니다.

- 요청에서 필요한 값을 읽는가
- 서비스 메서드를 올바르게 호출하는가
- 상태 코드, 응답 본문, 쿠키 같은 응답 결과를 올바르게 남기는가

미들웨어 테스트는 아래 정도면 충분합니다.

- 통과 조건에서 `next()`를 호출하는가
- 차단 조건에서 응답을 끝내는가
- 다음 단계가 쓰게 될 요청 정보, 예를 들어 `req.user`를 준비하는가

이 범위 안에만 머물러도 현재 `AuthController`와 `RequireAuthMiddleware`는 충분히 검증할 수 있습니다.

### 테스트 경계를 정하는 방법

핵심 기준은 단순합니다. 내가 만든 판단은 직접 테스트하고, 바깥에서 들어오는 것은 가능한 한 좁혀서 다룹니다. Express의 `req`, `res`, `next`는 현재 테스트에서 관심을 가지는 로직 자체가 아니라, 그 로직이 기대는 입력과 출력의 통로에 가깝습니다. 그래서 이 장에서는 이 객체들을 최소 mock으로 준비하고, 내가 만든 분기와 응답만 직접 확인합니다.

같은 기준은 미들웨어에도 적용됩니다. 인증 미들웨어에서 중요한 것은 “인증 쿠키 값을 내부에서 어떻게 해석하는가”보다 “현재 요청을 통과시킬 것인가, 막을 것인가”입니다. 이 기준을 지키면 테스트가 짧아지고, 나중에 구현이 바뀌어도 덜 흔들립니다.

### 정리

핸들러 테스트와 미들웨어 테스트는 같은 방식으로 보지 않습니다. 핸들러는 입력 해석과 응답 종료를 보고, 미들웨어는 통과 여부와 요청 준비 상태를 봅니다. 이 경계를 먼저 분명하게 나누어야 다음 장의 단위 테스트 확장도 자연스럽게 이어집니다.

# 제 5장: Express 단위 테스트 확장

## **5-01 코드 순서대로 테스트 만들기**

### 시작점

5장 본편은 `codeit-fs-jest-starter/05-express-unit`을 기준으로 읽습니다.

### 테스트를 넓히는 기준

단위 테스트를 늘리다 보면 가장 먼저 부딪히는 문제는 범위입니다. 어디부터 써야 하는지 정하지 않으면 테스트가 파일 여기저기로 흩어집니다. 이때 테스트를 넓히는 한 가지 방법은 코드의 책임 순서를 따라가는 것입니다. 같은 클래스 안에서도 메서드가 어떤 역할을 먼저 맡고, 어떤 메서드가 그 다음 책임을 이어받는지 순서대로 따라가면 테스트를 넓히는 기준이 분명해집니다.

`AuthService`는 이 기준을 설명하기 좋습니다. 먼저 `login()`이 있고, 그 다음에 `getUserProfile()`이 있습니다. `login()`은 사용자를 찾고, 비밀번호를 비교하고, 공개 사용자 정보로 바꾸는 흐름을 가집니다. `getUserProfile()`은 id로 사용자를 찾아 공개 사용자 정보로 바꾸는 흐름을 가집니다. 이 순서를 따라 테스트를 만들면 한 메서드 안의 분기와 다음 메서드의 책임이 섞이지 않습니다.

### 메서드 순서와 테스트 순서

`AuthService` 테스트는 실제 코드 순서와 거의 같은 흐름으로 읽힙니다.

개념 예시:

```tsx
describe("AuthService", () => {
  let userRepository: StubUserRepository;
  let passwordProvider: StubPasswordProvider;
  let service: AuthService;

  beforeEach(() => {
    userRepository = new StubUserRepository();
    passwordProvider = new StubPasswordProvider();
    service = new AuthService(
      userRepository,
      passwordProvider,
    );
  });

  test("email과 password가 모두 맞으면 공개 사용자 정보를 돌려준다", async () => {
    userRepository.findByEmail.mockResolvedValue({
      id: 1,
      email: "teacher@example.com",
      name: "Teacher",
      passwordHash: "pw",
    });
    passwordProvider.compare.mockResolvedValue(true);

    // act + assert
    await expect(
      service.login("teacher@example.com", "pw"),
    ).resolves.toEqual({
      id: 1,
      email: "teacher@example.com",
      name: "Teacher",
    });
  });

  // ... 생략 ...
});
```

여기서 `beforeEach()`는 테스트마다 똑같이 반복되는 구조를 다시 만드는 역할을 합니다. `StubUserRepository`, `StubPasswordProvider`, `AuthService`처럼 매번 같은 모양으로 준비되는 객체는 `beforeEach()`에서 만들고, 분기마다 달라지는 `mockResolvedValue(...)`와 입력값은 각 `test(...)` 안에 남겨 둡니다. 이 교재에서는 `Stub` 객체 안에 `jest` mock 함수를 넣어서 함께 사용합니다. 즉, 바깥에서는 `StubUserRepository`처럼 `stub`으로 읽고, 실제 테스트 안에서는 그 안의 `findByEmail`, `compare`를 `mockResolvedValue(...)`로 조정한다고 이해하면 됩니다.

4장에서는 첫 핸들러 테스트 감각을 잡기 위해 준비 코드를 각 `test(...)` 안에 두는 예시를 먼저 보여 주었습니다. 5장부터는 같은 구조가 여러 테스트에서 반복되기 시작하므로, 공통 준비를 `beforeEach()`로 올리고 분기 조건만 각 테스트 안에 남기는 기준을 본격적으로 사용합니다.

이 첫 테스트가 잡히면 그 다음에는 같은 메서드의 실패 분기를 붙입니다. 비밀번호가 틀린 경우, 사용자를 찾지 못한 경우를 추가하면 `login()`의 주요 흐름이 닫힙니다. 그 다음에 `getUserProfile()` 테스트를 붙이면 클래스 전체가 `코드 순서대로` 넓어집니다.

이 방식의 장점은 두 가지입니다. 첫째, 어떤 분기까지 썼는지 놓치기 어렵습니다. 둘째, 테스트를 읽는 사람이 서비스 메서드의 책임을 위에서 아래로 따라가기 쉽습니다. 단위 테스트는 실행 속도만 중요한 것이 아니라, 코드를 읽는 순서를 보존하는 것도 중요합니다.

### 성공 경로와 실패 경로의 배치

코드 순서대로 쓴다는 말은 성공 테스트만 위에서부터 나열한다는 뜻이 아닙니다. 같은 책임 안에서는 보통 성공 경로를 먼저 고정하고, 그 다음 실패 경로를 붙이는 편이 읽기 쉽습니다. `login()`에서는 다음 순서가 자연스럽습니다.

1. 사용자가 있고 비밀번호도 맞는 경우
2. 사용자는 있지만 비밀번호가 틀린 경우
3. 사용자를 찾지 못한 경우

이 순서가 좋은 이유는 정상 동작을 먼저 확정한 뒤, 어떤 조건에서 흐름이 끊기는지 내려가면서 볼 수 있기 때문입니다. 실패 경로를 먼저 붙이는 방식도 가능하지만, 구조를 설명하는 문맥에서는 정상 흐름을 먼저 고정하는 편이 더 읽기 쉽습니다.

### 정리

단위 테스트를 넓힐 때는 파일 위치보다 책임 순서를 기준으로 잡는 편이 좋습니다. `AuthService`처럼 메서드가 순서대로 읽히는 클래스는 테스트도 같은 순서로 붙이면 분기와 책임 경계가 더 또렷해집니다.

입력과 결과만 단순하게 반복되는 `test.each()` 패턴은 3장에서 이미 다루었습니다. 5장에서는 그 반복 설명을 다시 길게 하지 않고, 단위 테스트 구조를 더 쉽게 만드는 주입 가능한 설계로 바로 넘어가겠습니다.

## **5-02 주입 가능한 구조는 테스트를 쉽게 만든다**

### 주입 가능한 구조와 테스트 준비

테스트가 쉬운 코드는 대부분 필요한 협력 객체를 바깥에서 받습니다. `AuthService`는 `UserRepository`와 `PasswordProvider`를 생성자에서 받습니다. `RequireAuthMiddleware`는 `UserRepository`를 생성자에서 받습니다. 이런 구조에서는 테스트가 무엇을 가짜로 바꾸고, 무엇을 실제로 둘지 곧바로 결정할 수 있습니다.

개념 예시:

```tsx
const service = new AuthService(
  userRepository,
  passwordProvider,
);
```

개념 예시:

```tsx
const middleware = new RequireAuthMiddleware(
  userRepository,
);
```

이 구조의 장점은 테스트 준비가 명시적이라는 점입니다. 저장소를 `stub`으로 둘지, provider를 실제 객체로 둘지, spy를 붙일지 모두 테스트 코드에서 결정할 수 있습니다. 반대로 클래스 내부에서 저장소나 provider를 직접 만들면 테스트는 그 구현에 끌려 들어가게 됩니다.

### stub과 실제 객체의 선택

`AuthService` 테스트에서는 두 가지 방식이 함께 등장합니다. 대부분의 테스트에서는 저장소와 provider를 간단한 `stub`으로 준비합니다. 이렇게 하면 서비스가 어떤 입력을 받고 어떤 값을 돌려주는지에만 집중할 수 있습니다. 이 교재에서는 이런 `stub` 객체 안에 `jest.fn()`으로 만든 mock 함수를 넣어 함께 쓰는 경우가 많습니다.

한편 `compare()` 호출 여부를 보고 싶은 테스트에서는 `PasswordProvider` 실제 객체를 유지한 채 `spyOn()`을 붙입니다. `spyOn()`은 원래 구현을 관찰하는 데에도 쓸 수 있고, 아래처럼 `mockResolvedValue()`를 붙여 특정 메서드 동작만 테스트용으로 바꾸는 데에도 쓸 수 있습니다.

개념 예시:

```tsx
// ... 생략 ...

const passwordProvider = new PasswordProvider();
const compareSpy = jest
  .spyOn(passwordProvider, "compare")
  .mockResolvedValue(true);

// ... 생략 ...
```

이 방식이 가능한 이유도 결국 생성자 주입 덕분입니다. 서비스가 provider를 직접 만들지 않기 때문에, 테스트는 필요에 따라 `stub`과 실제 객체를 모두 선택할 수 있습니다. 여기서 `test/providers/password.provider.spec.ts`는 provider 자체의 동작을 `test.each(...)`로 검증하는 테스트이고, 방금 본 `spyOn()` 예시는 `AuthService` 테스트 안에서 provider를 다루는 방식이라는 점을 구분해 두는 편이 좋습니다.

### 테스트가 쉬운 구조의 기준

매개변수로 받는다는 말은 단순히 생성자 인자를 많이 받는다는 뜻이 아닙니다. 테스트에서 교체가 필요한 협력 객체를 바깥에서 받는다는 뜻입니다. 이 기준으로 보면 다음 구조가 테스트하기 쉽습니다.

- 저장소와 provider를 생성자에서 받는 서비스
- 사용자 저장소를 생성자에서 받고 `uid` 쿠키를 해석하는 미들웨어
- req와 res를 메서드 인자로 받아 처리하는 컨트롤러

반대로 테스트하기 어려운 구조는 클래스 안에서 외부 의존성을 직접 생성하거나, import 시점에 상태를 고정해 버리는 구조입니다. 이런 경우에는 테스트가 구현 세부사항과 강하게 엮입니다.

### 정리

생성자 주입과 매개변수화는 단순한 설계 취향이 아닙니다. 무엇을 가짜로 바꿀지 테스트가 직접 결정할 수 있게 만드는 구조입니다. 그래서 서비스와 미들웨어 단위 테스트가 훨씬 간단해집니다.

## **5-03 false 분기와 누락 케이스 찾기**

### 성공 경로만으로는 부족한 이유

단위 테스트를 조금만 써 보면 성공 경로는 금방 채워집니다. 문제는 그 다음입니다. 실제 장애는 대개 성공 경로가 아니라 실패 경로에서 드러납니다. 입력이 빠졌을 때, 인증 쿠키가 없거나 잘못되었을 때, 사용자를 찾지 못했을 때, 비교 함수가 호출되면 안 되는 상황이 대표적입니다.

이런 분기를 교재에서는 `false 분기`라고 부르겠습니다. 조건문에서 거짓으로 빠지는 경로뿐 아니라, 특정 호출이 일어나지 않아야 하는 누락·차단 경로까지 넓게 묶은 표현입니다.

### 컨트롤러의 false 분기

`AuthController.login()`은 `email`이나 `password`가 없으면 바로 400을 돌려줍니다. 이때 중요한 것은 상태 코드만이 아닙니다. 서비스 호출이 일어나지 않아야 한다는 점도 함께 확인해야 합니다. 준비 코드는 앞에서 본 같은 `auth.controller.spec.ts` 예시와 같고, 여기서는 false 분기에서 달라지는 검증만 보면 됩니다.

파일 일부: `test/unit/controllers/auth.controller.spec.ts`

```tsx
describe("AuthController.login", () => {
  // authService, controller 준비는 앞 예시와 같습니다.
  let req = httpMocks.createRequest();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    req = httpMocks.createRequest();
    res = httpMocks.createResponse();
  });

  // ... 생략 ...

  test("email이나 password가 없으면 서비스 호출 없이 400을 돌려준다", async () => {
    req.body = {
      email: "teacher@example.com",
    };

    await controller.login(req, res);

    expect(authService.login).not.toHaveBeenCalled();
    expect(res.statusCode).toBe(400);
    expect(res._getJSONData()).toEqual({
      message: "email과 password를 모두 전달해야 합니다.",
    });
  });

  // ... 생략 ...
});
```

이 테스트가 필요한 이유는 단순합니다. 입력이 잘못되었는데도 서비스 호출이 진행되면, 이후 분기와 부작용이 함께 엉킬 수 있기 때문입니다.

### 서비스의 false 분기

`AuthService.login()`에서도 같은 기준을 볼 수 있습니다. 사용자를 찾지 못한 경우에는 `compare()`를 호출하면 안 됩니다. 이런 검증은 단순히 반환값만 맞는지 보는 것보다 강합니다. 현재 결과가 `null`이더라도, 중간에 불필요한 호출이 끼어들지 않았는지까지 함께 확인할 수 있기 때문입니다.

파일: `test/unit/services/auth.service.spec.ts`

```tsx
describe("AuthService", () => {
  let userRepository: StubUserRepository;
  let passwordProvider: StubPasswordProvider;
  let service: AuthService;

  beforeEach(() => {
    userRepository = new StubUserRepository();
    passwordProvider = new StubPasswordProvider();
    service = new AuthService(
      userRepository,
      passwordProvider,
    );
  });

  test("사용자를 찾지 못하면 compare를 호출하지 않고 null을 돌려준다", async () => {
    userRepository.findByEmail.mockResolvedValue(null);

    await expect(
      service.login("missing@example.com", "pw"),
    ).resolves.toBeNull();
    expect(passwordProvider.compare).not.toHaveBeenCalled();
  });

  // ... 생략 ...
});
```

이 테스트는 반환값만 보는 테스트보다 더 강합니다. `null`을 돌려준다는 사실만 보면 우연히 통과할 여지가 있지만, 실제로 비교 함수를 건너뛰었는지까지 보면 분기 책임이 더 분명해집니다. 불필요한 비교 호출은 성능 낭비이자 잘못된 흐름이므로, 이런 지점은 단위 테스트에서 함께 차단해 두는 편이 좋습니다.

### 미들웨어의 false 분기

`RequireAuthMiddleware`는 `uid` 쿠키가 없거나 숫자로 해석할 수 없을 때 먼저 요청을 막습니다. 이 경우에는 사용자 조회까지 넘어가면 안 됩니다.

파일: `test/unit/middlewares/require-auth.middleware.spec.ts`

```tsx
import { jest } from "@jest/globals";
import httpMocks from "node-mocks-http";
import type { AuthenticatedRequest } from "#src/middlewares/require-auth.middleware.js";

describe("RequireAuthMiddleware.handle", () => {
  let userRepository: StubUserRepository;
  let middleware: RequireAuthMiddleware;
  let next: ReturnType<typeof jest.fn>;
  let req = httpMocks.createRequest<AuthenticatedRequest>();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    userRepository = new StubUserRepository();
    middleware = new RequireAuthMiddleware(userRepository);
    next = jest.fn();
    req = httpMocks.createRequest<AuthenticatedRequest>();
    res = httpMocks.createResponse();
  });

  test("uid 쿠키가 없으면 사용자 조회 없이 401로 차단한다", async () => {
    await middleware.handle(req, res, next);

    expect(res.statusCode).toBe(401);
    expect(res._getJSONData()).toEqual({
      message: "로그인이 필요합니다.",
    });
    expect(userRepository.findById).not.toHaveBeenCalled();
    expect(next).not.toHaveBeenCalled();
  });

  test("uid 쿠키가 숫자가 아니면 사용자 조회 없이 401로 차단한다", async () => {
    req.cookies = { uid: "not-a-number" };

    await middleware.handle(req, res, next);

    expect(res.statusCode).toBe(401);
    expect(res._getJSONData()).toEqual({
      message: "로그인이 필요합니다.",
    });
    expect(userRepository.findById).not.toHaveBeenCalled();
    expect(next).not.toHaveBeenCalled();
  });

  // ... 생략 ...
});
```

이 두 테스트를 나란히 놓고 보면 false 분기는 단순히 “실패 응답을 보냈는가”만이 아니라, “그 다음 동작을 막았는가”까지 함께 확인해야 한다는 점이 드러납니다.

### 정리

false 분기는 성공 경로의 반대편이 아닙니다. 오히려 테스트가 가장 자주 놓치는 핵심 경로입니다. 상태 코드, 반환값, 그리고 호출되지 않아야 하는 함수까지 함께 보면 누락 케이스를 더 정확하게 찾을 수 있습니다.

## **5-04 테스트를 두 번 실행해도 안전하게 만들기**

### 공유 상태가 만드는 문제

테스트는 한 번 통과했다고 끝나지 않습니다. 같은 테스트를 다시 실행해도 같은 결과가 나와야 합니다. 이때 가장 흔한 문제는 공유 상태입니다. 이전 테스트에서 만든 mock 호출 이력이나 store 상태가 다음 테스트에 남으면, 어떤 테스트는 혼자 실행할 때만 통과하고 전체 실행에서는 실패하게 됩니다.

`RequireAuthMiddleware` 테스트는 이 문제를 설명하기 좋습니다. 공통 구조인 저장소 mock, 미들웨어 인스턴스, `next`는 `beforeEach()`에서 다시 만들고, 필요한 쿠키 값과 응답 객체는 테스트 안에서 직접 준비합니다.

여기서도 런타임 라우팅 방식과 단위 테스트 방식은 구분해서 읽으면 됩니다. 실제 앱에서는 `requireAuthMiddleware.handle`가 라우트에 직접 연결되지만, 단위 테스트에서는 라우터를 거치지 않고 `middleware.handle(req, res, next)`를 직접 호출해 현재 분기만 확인합니다.

파일: `test/unit/middlewares/require-auth.middleware.spec.ts`

```tsx
import type { AuthenticatedRequest } from "#src/middlewares/require-auth.middleware.js";

describe("RequireAuthMiddleware.handle", () => {
  let userRepository: StubUserRepository;
  let middleware: RequireAuthMiddleware;
  let next: ReturnType<typeof jest.fn>;
  let req = httpMocks.createRequest<AuthenticatedRequest>();
  let res = httpMocks.createResponse();

  beforeEach(() => {
    userRepository = new StubUserRepository();
    middleware = new RequireAuthMiddleware(userRepository);
    next = jest.fn();
    req = httpMocks.createRequest<AuthenticatedRequest>();
    res = httpMocks.createResponse();
  });

  test("uid는 있지만 사용자를 못 찾으면 401을 돌려준다", async () => {
    userRepository.findById.mockResolvedValue(null);
    req.cookies = { uid: "999" };

    await middleware.handle(req, res, next);

    expect(res.statusCode).toBe(401);
    expect(res._getJSONData()).toEqual({
      message: "사용자를 찾을 수 없습니다.",
    });
    expect(next).not.toHaveBeenCalled();
  });

  // ... 생략 ...
});
```

이 방식의 장점은 테스트 사이에 상태가 섞이지 않는다는 점입니다. 공통 구조는 `beforeEach()`가 매번 새로 만들고, `cookies`처럼 테스트마다 달라지는 입력만 각 테스트 안에 남겨 둡니다. 이전 테스트가 어떤 저장소 mock을 썼는지, 어떤 쿠키를 넣었는지 다음 테스트가 알 필요가 없습니다.

### 상태를 테스트 사이에 남기지 않는 방법

이 절의 초점은 구조 설명이 아니라 상태 격리입니다. `RequireAuthMiddleware` 테스트도 같은 방향으로 정리했습니다. 저장소 mock, 미들웨어 인스턴스, `next`처럼 매번 같은 모양으로 필요한 구조는 `beforeEach()`에서 다시 만들고, `req.cookies`, `res`처럼 테스트마다 달라지는 값은 각 테스트 안에서 준비합니다. 이렇게 하면 어떤 테스트에서 `findById()`가 몇 번 호출되었는지 다음 테스트가 알 필요가 없습니다.

반대로 하나의 mock 함수를 파일 맨 위에서 공유하고 다시 만들지 않으면, 호출 횟수와 호출 인자가 테스트 사이에 남을 수 있습니다. 그래서 같은 구조가 반복될 때는 `beforeEach()`에서 새 객체를 다시 만들고, 테스트 안에서는 그때그때 필요한 mock 반환값과 입력만 바꾸는 편이 안전합니다.

### 언제 beforeEach가 필요한가

모든 것을 `beforeEach()`로 올리는 것도 좋지 않지만, 같은 구조를 여러 테스트에서 반복해서 만드는 경우에는 `beforeEach()`가 더 읽기 쉽습니다. 중요한 것은 어떤 방식을 쓰든 테스트 사이 상태가 남지 않게 만드는 것입니다.

선택 기준은 간단하게 잡아 둘 수 있습니다. 준비 구조 자체가 같고 값만 조금 바뀌면 `beforeEach()`를 먼저 검토합니다. 반대로 요청 객체나 입력 데이터처럼 테스트마다 모양이 달라지는 값은 각 테스트 안에 남겨 두는 편이 더 읽기 쉽습니다.

여기서는 공통 구조는 `beforeEach()`에 두고, 테스트별 입력 상태는 각 테스트 안에 남기는 패턴으로 정리합니다.

- `AuthService`, `AuthController`, `RequireAuthMiddleware` 테스트는 공통 `stub`과 서비스/컨트롤러/미들웨어 인스턴스를 `beforeEach()`에서 다시 만들고, 분기별 반환값과 요청·응답 상태는 각 테스트 안에서 준비합니다.

공통 준비가 반복되는데도 모두 각 테스트 안에 남겨 두면 줄 수만 늘고 핵심 분기가 흐려질 수 있습니다. 이 단계에서는 `beforeEach()`를 이미 배운 만큼, 반복되는 구조를 올리고 분기 조건만 테스트 안에 남기는 기준을 먼저 익히는 편이 더 중요합니다.

### 정리

재실행 안전성은 별도의 부가기능이 아닙니다. 테스트를 신뢰하려면 반드시 필요한 조건입니다. 공통 구조는 `beforeEach()`에서 새로 만들고, 테스트별 입력 상태만 각 테스트 안에 남겨 두면 이 기준을 더 일관되게 지킬 수 있습니다.

# 제 6장: supertest로 통합 테스트하기

## **6-01 통합 테스트의 역할**

### 시작점

6장 본편은 `codeit-fs-jest-starter/06-supertest-e2e`를 기준으로 읽습니다.

단위 테스트는 계층 하나를 비교적 좁은 범위에서 확인합니다. 컨트롤러를 테스트할 때는 `req`, `res`, `next`를 직접 만들고, 서비스 테스트에서는 외부 의존성을 가짜로 바꾸어 현재 판단만 확인합니다. 이런 방식은 빠르고, 실패 원인을 좁히기 쉽다는 장점이 있습니다.

하지만 HTTP 요청과 응답이 실제로 어떻게 연결되는지는 단위 테스트만으로 모두 확인하기 어렵습니다. 상태 코드, 헤더, 쿠키, 본문 구조는 Express 앱 전체가 한 번 이어진 상태에서 보는 편이 더 분명합니다. 그래서 이 장에서는 실제 요청과 응답 흐름을 확인하는 도구를 이용해 통합 테스트를 다룹니다.

이 장에서는 브라우저나 외부 시스템 전체를 포함하는 E2E 테스트까지 넓히지 않습니다. 여기서는 Express 앱 내부에서 요청과 응답이 오가는 흐름을 확인하는 통합 테스트에만 집중합니다.

즉, 이 장에서 확인하는 대상은 다음과 같습니다.

- `App`이 조립한 Express 인스턴스
- 라우트, 컨트롤러, 서비스, 미들웨어가 실제로 연결되는 흐름
- 상태 코드, 헤더, 쿠키, 본문으로 표현되는 HTTP 계약

정리하면 역할은 아래처럼 나눌 수 있습니다.

- 단위 테스트: 로직과 분기를 좁게 검증합니다.
- 통합 테스트: HTTP 계약이 실제 요청/응답 흐름 안에서 유지되는지 검증합니다.

이 기준을 먼저 잡아 두면 단위 테스트와 통합 테스트가 서로 경쟁하는 방식으로 보이지 않습니다. 단위 테스트는 로직을 좁게 확인하고, 통합 테스트는 그 로직이 HTTP 계약 안에서 제대로 이어지는지 확인합니다. 두 테스트는 역할이 다르기 때문에 함께 필요합니다.

## **6-02 supertest 설치와 `request(application.app)`로 첫 요청 보내기**

`supertest`는 Express 앱에 직접 요청을 보내는 테스트 도구입니다. 실제 포트를 열지 않아도 `GET`, `POST`, `PUT`, `DELETE` 같은 HTTP 요청을 보내고, 응답 객체를 받아 상태 코드와 헤더, 본문을 확인할 수 있습니다. 그래서 서버를 따로 띄우는 부담 없이 HTTP 흐름을 비교적 실제에 가깝게 검증할 수 있습니다.

`supertest`는 Node 환경에서 실행되는 HTTP 테스트 도구입니다. ESM 환경에서는 `import` 방식과 현재 Node 실행 기준이 같이 맞아야 하므로, 이 장의 예제도 `Node 22.x+`와 현재 체크포인트 설정을 그대로 따라가는 편이 안전합니다.

설치는 아래처럼 진행합니다.

터미널

```bash
pnpm add -D supertest @types/supertest
```

설치가 끝나면 통합 테스트 파일에서 `request`를 사용할 수 있습니다.

파일 일부: `test/integration/app.spec.ts`

```tsx
import request from "supertest";
import {
  createTestEnvironment,
  type TestEnvironment,
} from "#test/helper/app-environment.js";

// ... 생략 ...
```

여기서 중요한 점은 `request`가 받는 대상입니다. `supertest`는 `App` 클래스 인스턴스 전체를 받지 않습니다. 실제로는 그 안에 들어 있는 Express 인스턴스를 받아야 합니다. 이 교재의 예제에서는 그 대상이 `application.app`입니다.

`createTestEnvironment()`는 이런 통합 테스트용 앱 조립을 한곳에 모아 둔 helper입니다. 6장 단계에서는 아직 7장처럼 비동기 DB 연결과 리셋 패턴까지 다루지 않고, **테스트용 앱과 메모리 기반 의존성**을 한 번에 묶어 반환하는 정도로만 사용합니다.

먼저 helper 파일을 보면 `createTestEnvironment()`가 무엇을 조립하는지 더 분명하게 보입니다.

파일: `test/helper/app-environment.ts`

```tsx
import { App } from "#src/app.js";
import {
  AuthController,
  Controller,
  HealthController,
} from "#src/controllers/index.js";
import { RequireAuthMiddleware } from "#src/middlewares/index.js";
import { PasswordProvider } from "#src/providers/index.js";
import { UserRepository } from "#src/repositories/index.js";
import { AuthService } from "#src/services/index.js";
import { teacherUserRecord } from "#test/mock/users.js";

export interface TestEnvironment {
  application: App;
  authService: AuthService;
  userRepository: UserRepository;
}

export function createTestEnvironment(): TestEnvironment {
  const userRepository = new UserRepository([teacherUserRecord]);
  const authService = new AuthService(
    userRepository,
    new PasswordProvider(),
  );
  const requireAuthMiddleware = new RequireAuthMiddleware(userRepository);
  const controller = new Controller(
    new HealthController(),
    new AuthController(authService, requireAuthMiddleware),
  );
  const application = new App(controller);

  return {
    application,
    authService,
    userRepository,
  };
}
```

이 helper 파일의 역할은 테스트 전용 환경을 **한 번 조립해 export하는 것**입니다. 즉, 어떤 저장소를 쓰는지, 어떤 컨트롤러와 미들웨어를 묶는지, 최종적으로 어떤 `application.app`을 테스트 대상으로 꺼낼지를 여기서 먼저 고정합니다.

반면 생명주기 훅은 이 helper 안에 두지 않고, 실제 테스트 파일인 `test/integration/app.spec.ts` 안에 둡니다. 즉, 환경을 조립하는 일과 그 환경을 언제 다시 만드는지는 분리해서 봐야 합니다.

파일: `test/integration/app.spec.ts`

```tsx
describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("health 요청은 200을 반환한다", async () => {
    const response = await request(
      environment.application.app,
    ).get("/health");

    expect(response.status).toBe(200);
  });

  // ... 생략 ...
});
```

이 코드에서 테스트 대상은 여전히 `application.app`입니다. 여기서는 그 앱을 다른 곳에 한 번 더 감추지 않고, `test/helper/app-environment.ts`가 만든 테스트 전용 환경 안에서 직접 꺼내 씁니다. 이렇게 해야 통합 테스트가 어떤 환경 자원을 기대는지 구조가 더 분명하게 보입니다.

첫 요청은 가능한 한 작게 시작하는 편이 좋습니다. 이 절에서는 복잡한 로그인 흐름보다 먼저 가장 단순한 `GET /health` 같은 요청이 실제로 통과하는지 보는 것이 적절합니다.

## **6-03 app과 main 분리로 포트 충돌 없애기**

통합 테스트에서는 서버를 실제로 `listen()` 상태로 띄우지 않습니다. 그래서 앱을 조립하는 코드와 실제 실행 코드를 분리해 두어야 합니다. 이 교재에서는 `app.ts`가 Express 앱을 조립하고, `main.ts`가 실제 실행을 맡습니다.

`app.ts`는 Express 인스턴스를 만들고, 미들웨어와 라우트를 연결합니다.

파일: `src/app.ts`

```tsx
export class App {
  public readonly app: Express;

  constructor(controller: Controller) {
    this.app = express();
    this.middleware();
    this.routes(controller);
    this.errorHandling();
  }
}
```

반면 `main.ts`는 조립된 앱을 받아 실제 실행만 맡습니다. 따라서 통합 테스트에서는 `main.ts`를 실행하지 않고, `test/helper/app-environment.ts`가 준비한 `application.app`에 직접 요청을 보냅니다. 핵심은 `listen()`을 테스트 경로에서 분리하는 것입니다.

파일: `test/integration/app.spec.ts`

```tsx
describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("health 요청은 expect 체인으로도 검증할 수 있다", async () => {
    await request(environment.application.app)
      .get("/health")
      .expect(200);
  });

  // ... 생략 ...
});
```

이 방식의 장점은 분명합니다.

- 포트를 점유하지 않습니다.
- 테스트가 실행 책임까지 함께 가져가지 않습니다.
- 앱 조립 결과만 따로 검증할 수 있습니다.
- 서버 종료 시점 때문에 테스트가 흔들리지 않습니다.

즉, `App`은 조립을 담당하고, `main.ts`는 실행을 담당합니다. 이 역할 분리가 먼저 되어 있어야 `supertest`를 이용한 통합 테스트도 안정적으로 작성할 수 있습니다. 서버를 실제로 띄우는 책임과 테스트가 앱 객체를 검증하는 책임을 분리해 두어야, 포트 충돌이나 종료 타이밍 문제 없이 테스트에 집중할 수 있습니다.

## **6-04 상태 코드, 헤더, 응답 본문 검증하기**

통합 테스트에서는 응답이 왔는지만 보면 부족합니다. 진짜로 확인해야 하는 것은 어떤 응답이 왔는지입니다. 상태 코드가 맞는지, 필요한 헤더가 들어 있는지, 쿠키가 제대로 내려오는지, 응답 본문이 기대한 구조를 가지는지까지 함께 확인해야 합니다.

예를 들어 로그인 성공을 검증할 때는 `200` 하나만 확인해서는 충분하지 않습니다. 현재 예제에서는 로그인 성공 시 아래 정보가 함께 전달됩니다.

- 상태 코드 `200`
- 로그인 단계를 알리는 `x-auth-step: login`
- 인증 상태를 이어 주는 `set-cookie`
- 현재 사용자 정보를 담은 JSON 본문

실제 테스트는 아래처럼 작성할 수 있습니다.

파일: `test/integration/app.spec.ts`

```tsx
describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("로그인 성공 시 상태 코드, 헤더, 응답 본문을 함께 검증한다", async () => {
    const response = await request(
      environment.application.app,
    )
      .post("/auth/login")
      .send({
        email: "teacher@example.com",
        password: "class-based-jest",
      });

    expect(response.status).toBe(200);
    expect(response.headers["x-auth-step"]).toBe("login");
    const authCookie =
      response.headers["set-cookie"]?.[0] ?? "";

    expect(authCookie).toContain("uid=1");
    expect(response.body).toEqual({
      user: {
        id: 1,
        email: "teacher@example.com",
        name: "Teacher",
      },
    });
  });

  // ... 생략 ...
});
```

여기서 `response.headers`는 객체 형태이며, 헤더 이름은 모두 소문자 키로 접근합니다. 그래서 `X-Auth-Step`을 읽을 때도 `response.headers["x-auth-step"]`처럼 접근합니다.

`set-cookie`는 문자열 하나가 아니라 배열로 반환됩니다. 각 쿠키가 문자열 형태로 들어 있기 때문에, 첫 번째 쿠키 문자열을 꺼내 `toContain("uid=1")`처럼 핵심 조각이 포함되어 있는지 확인하는 편이 더 자연스럽습니다. 전체 문자열을 완전히 고정하면 쿠키 옵션 순서나 부가 속성 변화에 더 쉽게 흔들릴 수 있습니다.

이 테스트는 로그인 성공이라는 사실을 더 구체적으로 설명합니다. 단순히 요청이 성공했다는 수준이 아니라, 현재 API 계약이 정확히 유지되고 있는지 확인합니다. `x-auth-step` 헤더가 빠지거나, 쿠키 이름이 바뀌거나, 응답 본문 구조가 달라지면 이 테스트는 바로 실패합니다.

즉, 통합 테스트는 HTTP 계약을 검증하는 테스트이고, 이 계약은 API 문서 역할도 함께 합니다.

`GET /health`도 같은 관점으로 볼 수 있습니다. 현재 예제의 `HealthController`는 `x-test-layer: supertest` 헤더와 JSON 본문을 함께 반환합니다. 따라서 이 경로 역시 상태 코드만 보는 대신, 헤더와 본문까지 함께 확인하는 편이 더 안전합니다.

즉, 통합 테스트는 “성공했는가”보다 “어떤 HTTP 계약이 유지되고 있는가”를 확인하는 테스트입니다. 상태 코드, 헤더, 응답 본문을 함께 검증해야 실제 클라이언트가 기대하는 계약을 안정적으로 지킬 수 있습니다.

## **6-05 로그인 흐름과 `supertest.agent`**

단일 요청만 검증할 때는 `request(application.app)`으로 충분합니다. 하지만 로그인처럼 상태가 이어지는 흐름을 테스트할 때는 이것만으로 부족합니다. 로그인 후에 쿠키가 설정되고, 다음 요청에서 그 쿠키를 다시 보내야 하기 때문입니다.

이때 사용하는 것이 `supertest.agent`입니다. `agent`는 여러 요청 사이에서 쿠키와 로그인 상태를 유지합니다. 즉, 같은 사용자가 연속해서 요청을 보내는 흐름을 흉내 낼 수 있습니다.

현재 예제에서는 로그인 후 `/auth/me`를 호출하는 흐름을 아래처럼 확인합니다.

파일: `test/integration/app.spec.ts`

```tsx
describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("supertest.agent는 로그인 이후 같은 세션으로 me 요청을 보낼 수 있다", async () => {
    const agent = request.agent(
      environment.application.app,
    );

    await agent.post("/auth/login").send({
      email: "teacher@example.com",
      password: "class-based-jest",
    });

    const response = await agent.get("/auth/me");

    expect(response.status).toBe(200);
    expect(response.body).toEqual({
      user: {
        id: 1,
        email: "teacher@example.com",
        name: "Teacher",
      },
    });
  });

  // ... 생략 ...
});
```

이 테스트는 단순히 로그인 응답만 보는 것이 아닙니다. 로그인으로 심은 쿠키가 다음 요청에서도 실제로 유지되는지 확인합니다. 즉, 쿠키 기반 인증 흐름 전체를 한 번에 검증합니다.

반대로 `request(application.app)`를 요청마다 새로 호출하면 쿠키가 이어지지 않습니다. 그러면 로그인 직후 `/auth/me`를 요청해도 인증 쿠키가 없는 새 클라이언트처럼 동작하게 됩니다. 따라서 로그인 상태를 이어 보는 흐름에서는 `agent`가 더 적합합니다.

다만 `agent`를 너무 넓은 범위에서 공유하면 테스트 간 상태가 섞일 수 있습니다. 한 테스트의 로그인 상태가 다른 테스트에 남아 있으면, 실패해야 할 테스트가 통과하거나 원인을 읽기 어려운 실패가 생길 수 있습니다. 그래서 `agent`는 상태가 이어지는 테스트에서만 사용하고, 각 테스트 안에서 새로 만드는 편이 안전합니다.

즉, 선택 기준은 아래처럼 정리할 수 있습니다.

- `request`: 단일 요청 테스트에 사용합니다.
- `agent`: 로그인처럼 상태가 이어지는 테스트에만 사용합니다.

이 기준을 먼저 고정해 두면 읽기 쉬워집니다.

## **6-06 `done`, `return`, `async/await`와 비동기 종료 처리**

`supertest` 테스트는 대부분 비동기입니다. 요청을 보내고 응답을 기다린 뒤에 검증이 이루어지기 때문입니다. 그래서 테스트가 언제 끝났는지 Jest가 정확히 알아야 합니다. 종료 시점이 흐리면 테스트가 통과한 것처럼 보이면서 실제 요청은 아직 끝나지 않았거나, 반대로 테스트가 끝난 뒤에도 핸들이 남아 있을 수 있습니다.

현재 예제는 세 가지 종료 방식을 모두 보여 줍니다.

첫 번째는 `done`입니다. 콜백 기반 흐름에서 테스트 종료를 직접 알려 주는 방식입니다.

파일: `test/integration/app.spec.ts`

```tsx
// ... 생략 ...

describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("health 요청은 done 방식으로도 검증할 수 있다", (done) => {
    request(environment.application.app)
      .get("/health")
      .expect(200)
      .end((error, response) => {
        if (error) {
          done(error);
          return;
        }

        expect(response.headers["x-test-layer"]).toBe(
          "supertest",
        );
        done();
      });
  });

  // ... 생략 ...
});
```

두 번째는 Promise를 그대로 `return`하는 방식입니다.

파일: `test/integration/app.spec.ts`

```tsx
// ... 생략 ...

describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("잘못된 로그인은 return 기반 검증으로도 읽을 수 있다", () => {
    return request(environment.application.app)
      .post("/auth/login")
      .send({
        email: "teacher@example.com",
        password: "wrong",
      })
      .expect(401);
  });

  // ... 생략 ...
});
```

세 번째는 `async/await`입니다. 세 방식 가운데서는 이 방식이 가장 읽기 쉽고, `supertest`의 Promise 흐름과도 잘 맞습니다.

파일: `test/integration/app.spec.ts`

```tsx
// ... 생략 ...

describe("integration/app", () => {
  let environment: TestEnvironment;

  beforeEach(() => {
    environment = createTestEnvironment();
  });

  test("로그인 성공 시 상태 코드, 헤더, 응답 본문을 함께 검증한다", async () => {
    const response = await request(
      environment.application.app,
    )
      .post("/auth/login")
      .send({
        email: "teacher@example.com",
        password: "class-based-jest",
      });

    expect(response.status).toBe(200);
  });

  // ... 생략 ...
});
```

세 방식 모두 사용할 수 있지만, 한 테스트 안에서는 종료 시점을 한 가지 방식으로만 잡는 편이 좋습니다. 예를 들어 `async` 함수 안에서 다시 `done`을 받으면 Promise 종료와 callback 종료를 동시에 섞는 셈이 됩니다. 이 경우 한쪽 실패가 다른 쪽에 가려지거나, 테스트가 중복 종료되는 등 읽기 어려운 문제가 생길 수 있습니다.

특히 `async` 함수 안에서 `done`을 함께 사용하는 패턴은 피해야 합니다. 이 장에서는 `async/await`, `return`, `done` 중 한 가지 방식만 고르는 것을 기준으로 잡습니다.

또한 `return`을 써야 하는 테스트에서 `return`을 빼먹거나, `async/await`에서 `await`를 빼먹는 실수도 자주 나옵니다. 이런 경우 Jest는 아직 끝나지 않은 비동기 요청을 기다리지 못하고 테스트를 먼저 끝낸 것으로 처리할 수 있습니다.

통합 테스트가 실패하면 가장 먼저 `response.body`를 출력해 보는 편이 좋습니다. 상태 코드만 보고 원인을 추측하기보다, 실제로 어떤 본문이 내려왔는지부터 확인해야 문제를 더 빨리 좁힐 수 있습니다.

정리하면 선택 기준은 단순합니다.

- 기본은 `async/await`를 사용합니다.
- 짧은 Promise 체인은 `return`으로도 충분합니다.
- 콜백 기반 흐름을 직접 다뤄야 할 때만 `done`을 사용합니다.

핵심은 문법을 많이 아는 것이 아니라, 테스트 종료 시점을 하나의 방식으로만 분명하게 고정하는 것입니다.

# 제 7장: Prisma Client와 Jest 테스트 셋업

## **7-01 시작점과 generated Client 위치 고정하기**

### 시작점

7장 본편은 `codeit-fs-jest-starter/07-prisma-client-testing`을 기준으로 읽습니다. 7장은 별도 실습 없이 이 폴더 안에서 Prisma Client singleton과 Jest 테스트 셋업을 점검합니다.

이 장에서는 Prisma 전체 입문보다, 이미 준비된 `starter/07-prisma-client-testing`에서 Jest 테스트 seam을 어떻게 잡는지에 집중합니다. 따라서 `schema.prisma`, `prisma.config.ts`, `.env`, `package.json`은 미리 준비된 상태를 기준으로 읽습니다.

### schema.prisma와 prisma.config.ts

먼저 Prisma Client 출력 위치를 고정해야 합니다. 이 장에서는 generated client를 `generated/prisma` 아래로 뽑고, 런타임 코드는 그 결과를 `#generated/prisma/client.js` alias로 가져옵니다.

파일: `prisma/schema.prisma`

```
generator client {
  provider = "prisma-client"
  output   = "../generated/prisma"
}

datasource db {
  provider = "sqlite"
}

model User {
  id           Int    @id @default(autoincrement())
  email        String @unique
  name         String
  passwordHash String
}
```

파일: `prisma.config.ts`

```tsx
import { defineConfig, env } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
  },
  datasource: {
    url: env("DATABASE_URL"),
  },
});
```

그리고 `package.json`에서는 generated client를 이렇게 읽습니다.

파일 일부: `package.json`

```json
{
  "imports": {
    "#generated/prisma/client.js": {
      "types": "./generated/prisma/client.ts",
      "default": "./dist/generated/prisma/client.js"
    }
  }
}
```

이 기준을 먼저 고정해야 `src/db/prisma.ts`, `test/mock/prisma.ts`, 서비스 코드가 모두 같은 Client 타입을 보게 됩니다.

참고로 `file:./dev.db`는 학생이 미리 손으로 만들어 두는 파일이 아닙니다. `prisma generate`는 Client 타입만 만들고 DB 파일은 만들지 않습니다. SQLite 파일은 실제로 데이터베이스를 동기화할 때, 예를 들어 `prisma db push`나 `prisma migrate dev`를 실행하면 없을 경우 생성됩니다. 반대로 7장 테스트는 singleton Prisma Client를 mock 해서 돌리므로, 이 장의 Jest 테스트만 실행할 때는 `dev.db`가 생기지 않아도 괜찮습니다.

### 정리

7장의 첫 기준은 단순합니다. Prisma 설정의 핵심은 “실제 DB를 연결한다”가 아니라, **generated client가 어디에 생기고 그 타입을 앱과 테스트가 같은 경로로 읽는가**입니다.

## **7-02 src/db/prisma.ts에 singleton을 모으기**

### Prisma 진입점 고정하기

이 장에서는 Prisma 진입점을 `src/db/prisma.ts` 한 파일로 모읍니다.

그리고 Jest unit testing을 보여 주는 장이므로, `PrismaService` 이름은 유지하되 실제 사용은 singleton `prisma` instance로 단순화합니다. 여기서 `PrismaService`는 `AuthService` 같은 비즈니스 서비스가 아니라, Prisma Client 생성 책임을 한곳에 모으는 인프라용 wrapper입니다.

파일: `src/db/prisma.ts`

```tsx
import "dotenv/config";
import { PrismaBetterSqlite3 } from "@prisma/adapter-better-sqlite3";
import { PrismaClient } from "#generated/prisma/client.js";

export class PrismaService extends PrismaClient {
  constructor() {
    const adapter = new PrismaBetterSqlite3({
      url: process.env.DATABASE_URL ?? "file:./dev.db",
    });

    super({ adapter });
  }
}

const prisma = new PrismaService();

export default prisma;
```

여기서 먼저 볼 것은 세 가지입니다.

- `PrismaService`
    - Prisma Client를 감싸는 진입 클래스입니다.
- `const prisma = new PrismaService()`
    - 앱과 테스트가 같이 바라보는 singleton instance입니다.
- `default export`
    - 서비스 코드에서는 이 singleton만 import해서 사용합니다.

이 장의 예제는 SQLite를 쓰므로 adapter는 `@prisma/adapter-better-sqlite3`를 사용합니다. 여기서 중요한 것은 adapter 종류보다, Prisma 진입점을 `src/db/prisma.ts` 한 곳에 모은다는 점입니다.

### 정리

Prisma가 들어간 코드를 테스트하려면, 먼저 “Prisma Client를 어디서 만들 것인가”가 한 곳으로 모여 있어야 합니다. 7장에서는 그 기준점을 `src/db/prisma.ts`로 고정합니다.

## **7-03 Jest에서 singleton Prisma Client를 mock 하기**

### 공식 singleton 패턴을 현재 ESM 구조에 맞추기

앞 장들에서는 생성자 인자로 주입할 수 있는 의존성을 `stub`으로 바꿨습니다. 반면 7장은 Prisma Client가 `src/db/prisma.ts` singleton entry point로 묶여 있으므로, seam을 생성자보다 모듈 경계에서 잡습니다. Prisma 공식 문서의 unit testing 페이지도 singleton 패턴에서 `singleton.ts`와 `setupFilesAfterEnv`를 함께 두고, 7장은 그 구조를 현재 ESM 저장소에 맞게 `jest.unstable_mockModule(...)`로 옮깁니다.

파일: `test/mock/prisma.ts`

```tsx
import type { PrismaClient } from "#generated/prisma/client.js";
import { beforeEach, jest } from "@jest/globals";
import {
  mockDeep,
  mockReset,
  type DeepMockProxy,
} from "jest-mock-extended";

export const prismaMock =
  mockDeep<PrismaClient>() as DeepMockProxy<PrismaClient>;

await jest.unstable_mockModule("#src/db/prisma.js", () => ({
  __esModule: true,
  default: prismaMock,
}));

beforeEach(() => {
  mockReset(prismaMock);
});
```

이 파일이 하는 일은 두 가지입니다.

- `#src/db/prisma.js`를 실제 singleton 대신 deep mock으로 바꾼다
- `beforeEach()`마다 이전 테스트에서 남은 Prisma 호출 기록과 mock 구현을 비운다

그리고 이 파일은 Jest 설정에서 먼저 읽히게 연결합니다.

파일 일부: `jest.config.mjs`

```jsx
export default {
  preset: "ts-jest/presets/default-esm",
  testEnvironment: "node",
  clearMocks: true, // 기존 mock 함수 호출 기록은 Jest가 자동으로 비웁니다.
  setupFilesAfterEnv: ["<rootDir>/test/mock/prisma.ts"], // 7장에서 추가: Prisma singleton mock 파일을 각 테스트 전에 먼저 읽습니다.
  moduleNameMapper: {
    "^(\\.{1,2}/.*)\\.js$": "$1", // ESM 상대 경로의 .js import를 ts-jest가 읽을 수 있게 맞춥니다.
    "^#generated/prisma/client\\.js$":
      "<rootDir>/generated/prisma/client.ts", // 7장에서 추가: generate된 Prisma Client alias를 테스트에서도 해석합니다.
    "^#src/(.*)\\.js$": "<rootDir>/src/$1", // 기존 #src alias를 Jest에서도 같은 뜻으로 읽습니다.
    "^#test/(.*)\\.js$": "<rootDir>/test/$1", // 기존 #test alias를 Jest에서도 같은 뜻으로 읽습니다.
  },
};
```

즉, 이 장의 핵심은 “Prisma를 테스트할 수 있나?”가 아니라, **singleton으로 만든 Prisma Client를 테스트 전에 mock으로 바꿔 놓을 수 있나**입니다.

### 정리

7장의 테스트 셋업 핵심은 `test/mock/prisma.ts`입니다. 공식 문서의 singleton 아이디어를 그대로 가져오되, 현재 ESM 프로젝트에 맞게 mock 등록 방식만 맞춰 주면 됩니다.

## **7-04 서비스 코드는 class로 두고, 테스트는 prismaMock을 직접 조절하기**

### 서비스 코드의 기준

이제 서비스 코드는 별도 context를 받지 않고, `src/db/prisma.ts`의 singleton을 사용하는 class로 둡니다. 앞 장들이 생성자 주입 seam을 테스트했다면, 7장은 같은 원칙을 singleton 모듈 경계에 적용하는 장이라고 보면 됩니다. 그래야 다른 장의 서비스 구조와도 맞고, 실제 런타임과 테스트 기준도 같은 entry point를 보게 됩니다.

파일: `src/services/user.service.ts`

```tsx
import prisma from "#src/db/prisma.js";

export interface CreateUserInput {
  email: string;
  name: string;
  passwordHash: string;
  acceptTermsAndConditions: boolean;
}

export interface RenameUserInput {
  id: number;
  name: string;
}

export interface PublicUser {
  id: number;
  email: string;
  name: string;
}

export class UserService {
  constructor(private readonly prismaService = prisma) {}

  async createUser(input: CreateUserInput): Promise<PublicUser> {
    if (!input.acceptTermsAndConditions) {
      throw new Error("TERMS_NOT_ACCEPTED");
    }

    const user = await this.prismaService.user.create({
      data: {
        email: input.email,
        name: input.name,
        passwordHash: input.passwordHash,
      },
    });

    return this.toPublicUser(user);
  }

  async renameUser(input: RenameUserInput): Promise<PublicUser> {
    const user = await this.prismaService.user.update({
      where: {
        id: input.id,
      },
      data: {
        name: input.name,
      },
    });

    return this.toPublicUser(user);
  }

  async getPublicUserByEmail(email: string): Promise<PublicUser | null> {
    const user = await this.prismaService.user.findUnique({
      where: {
        email,
      },
    });

    if (!user) {
      return null;
    }

    return this.toPublicUser(user);
  }

  private toPublicUser(user: {
    id: number;
    email: string;
    name: string;
  }) {
    return {
      id: user.id,
      email: user.email,
      name: user.name,
    };
  }
}
```

### 테스트 파일의 기준

테스트는 `prismaMock.user.*`를 직접 조절합니다. 앞 장들처럼 `beforeEach()`에서 서비스 인스턴스를 다시 만든다는 리듬은 같지만, 여기서 바꾸는 대상은 생성자 stub이 아니라 singleton Prisma mock 입니다. 각 테스트는 필요한 Prisma 결과만 다시 정합니다.

파일 일부: `test/services/user.service.spec.ts`

```tsx
import {
  beforeEach,
  describe,
  expect,
  test,
} from "@jest/globals";
import { UserService } from "#src/services/user.service.js";
import {
  assistantUserRecord,
  teacherUserRecord,
} from "#test/mock/users.js";
import { prismaMock } from "#test/mock/prisma.js";

describe("UserService.createUser", () => {
  let service: UserService;

  beforeEach(() => {
    service = new UserService();
  });

  test("약관에 동의하면 Prisma Client로 사용자를 생성한다", async () => {
    prismaMock.user.create.mockResolvedValue(teacherUserRecord);

    await expect(
      service.createUser({
        email: "teacher@example.com",
        name: "Teacher",
        passwordHash: "hashed:pw",
        acceptTermsAndConditions: true,
      }),
    ).resolves.toEqual({
      id: 1,
      email: "teacher@example.com",
      name: "Teacher",
    });

    expect(prismaMock.user.create).toHaveBeenCalledWith({
      data: {
        email: "teacher@example.com",
        name: "Teacher",
        passwordHash: "hashed:pw",
      },
    });
  });

  test("약관에 동의하지 않으면 Prisma Client를 호출하지 않는다", async () => {
    await expect(
      service.createUser({
        email: "teacher@example.com",
        name: "Teacher",
        passwordHash: "hashed:pw",
        acceptTermsAndConditions: false,
      }),
    ).rejects.toThrow("TERMS_NOT_ACCEPTED");

    expect(prismaMock.user.create).not.toHaveBeenCalled();
  });
});

// ... 생략 ...
```

`renameUser(...)`, `getPublicUserByEmail(...)` 테스트도 같은 패턴입니다. `beforeEach()`에서 `UserService`를 다시 만들고, 각 테스트는 `prismaMock.user.update`, `prismaMock.user.findUnique`의 결과와 호출 인자만 직접 정합니다.

여기서 먼저 읽을 기준은 세 가지입니다.

- 서비스 코드는 singleton `prisma`를 쓰는 `UserService` class다
- 테스트는 `beforeEach()`에서 `new UserService()`를 만든다
- 테스트는 `prismaMock.user.*`로 결과를 직접 정한다
- 결과뿐 아니라 Prisma 호출 인자도 같이 검증한다

이 구조가 잡히면 Prisma 테스트는 “DB를 어떻게 가짜로 만들까?”가 아니라 “singleton Prisma Client를 어떤 mock 값으로 바꿀까?”로 단순해집니다.

### 정리

7장 테스트의 핵심은 별도 context가 아니라 `prismaMock`입니다. singleton entry point를 고정하고, 테스트에서는 그 entry point를 mock 한 뒤 필요한 Prisma 결과만 다시 정하면 됩니다.

# 제 8장: Express 테스트 품질과 마무리

## **8-01 커버리지 수치와 테스트 품질 해석하기**

### 시작점

8장 본편은 `codeit-fs-jest-starter/08-final`을 기준으로 읽습니다.

### 커버리지 보고서를 읽는 기준

이 장에서는 먼저 커버리지를 점수표가 아니라 해석 도구로 읽는 기준부터 잡습니다. 커버리지는 테스트가 아직 닿지 않은 경로를 보여 주는 지도에 가깝기 때문입니다. 즉, 여기서 중요한 것은 분기(branch)와 함수(function) 수치가 왜 아직 낮은지, 그 낮은 수치가 정말 품질 문제를 뜻하는지 구분하는 일입니다.

### 수치가 가리키는 항목

커버리지 보고서에는 보통 네 가지 항목이 나옵니다.

| 항목 | 의미 | 낮게 나왔을 때 먼저 볼 것 |
| --- | --- | --- |
| statements | 실행된 문장 비율 | 아예 지나가지 않은 코드 블록 |
| branches | 실행된 분기 비율 | `if/else`, 조기 반환, 예외 경로 |
| functions | 호출된 함수 비율 | 아직 한 번도 실행되지 않은 함수 |
| lines | 실행된 줄 비율 | 빠진 줄, 조건에 가려진 줄 |

이 표를 다시 보는 이유도 숫자 정의를 외우기 위해서가 아니라, 프로젝트 전체 보고서에서 어떤 항목이 실제로 더 의미 있는 신호인지 구분하기 위해서입니다. 특히 Express 백엔드에서는 `branches`가 가장 늦게 올라가는 경우가 많습니다.

### 이 체크포인트에서 눈여겨볼 수치

`codeit-fs-jest-completed` 저장소가 준비되어 있다면 `08-test-quality-and-wrap-up`에서 `test:cov`를 실행해 볼 수 있습니다. statements와 lines는 비교적 높게 나오지만, branches는 그보다 더 낮게 나오는 경우가 많습니다. 이런 패턴은 보통 주요 흐름은 대체로 지나갔지만, 조건 분기와 실패 경로는 아직 덜 확인되었다는 뜻으로 읽습니다.

이 체크포인트의 테스트를 떠올려 보면 그 이유를 이해하기 쉽습니다.

- `/health`의 정상 응답을 검증하는 테스트를 넣었습니다.
- 로그인 성공과 로그아웃 이후 흐름을 점검하는 테스트도 넣었습니다.
- 잘못된 비밀번호와 본문 누락처럼 대표적인 실패 경로도 함께 넣었습니다.

그런데도 branches 수치가 더 낮다는 것은, 아직 조건 분기 전체를 다 채우지 못했다는 뜻입니다. 예를 들어 서비스 안쪽의 더 세부적인 분기, 저장소 예외, 외부 의존성 실패 같은 경로는 별도 단위 테스트가 더 필요할 수 있습니다. 즉, 수치가 낮다고 해서 테스트가 쓸모없다는 뜻은 아니고, 어떤 종류의 경로가 비어 있는지 알려 주는 신호로 읽는 것이 맞습니다.

### 100% 목표를 읽는 방식

100%는 목표라기보다 질문을 끝까지 밀어 보는 장치에 가깝습니다. 다만 여기서도 숫자를 품질과 같은 뜻으로 읽으면 안 됩니다. 어떤 줄을 한 번 지나갔다고 해서 그 줄의 의미를 제대로 검증했다는 뜻은 아니기 때문입니다.

### 테스트 품질과 커버리지의 차이

커버리지는 어디를 지나갔는가를 보여 줍니다. 테스트 품질은 왜 이 경로를 검증했는가, 실패했을 때 무엇을 알 수 있는가를 더 중요하게 봅니다. 둘은 겹치지만 같은 개념은 아닙니다.

| 항목 | 커버리지 | 테스트 품질 |
| --- | --- | --- |
| 중심 질문 | 어떤 코드가 실행되었는가 | 이 테스트가 실제 가치를 가지는가 |
| 강점 | 빠진 줄과 분기를 찾기 쉽습니다 | 읽기 쉬움, 유지보수성, 실패 해석을 봅니다 |
| 한계 | 숫자만으로 의미를 판단하지 못합니다 | 정량화가 쉽지 않습니다 |

이 구분은 class 기반 Express 구조에서 더 중요합니다. 계층마다 확인해야 할 내용이 다른데, 커버리지 숫자만으로는 그 차이를 충분히 설명할 수 없기 때문입니다.

좋은 테스트를 조금 더 실전적으로 표현하면 아래 기준에 가깝습니다.

- 실패했을 때 원인을 바로 알 수 있습니다.
- 테스트 이름만 읽어도 의도가 보입니다.
- 구현이 조금 바뀌어도 계약이 유지되면 쉽게 깨지지 않습니다.

### 커버리지를 해석할 때의 기준

커버리지 보고서를 볼 때는 아래 순서로 읽는 편이 좋습니다.

1. 어떤 파일의 branches가 특히 낮은지 봅니다.
2. 그 파일의 조건문, 조기 반환, 예외 흐름 중 무엇이 비어 있는지 확인합니다.
3. 그 경로가 내 코드인지, 라이브러리나 프레임워크 바깥 경계인지 구분합니다.
4. 실제로 중요한 계약을 놓친 것인지, 단순한 수치 차이인지 판단합니다.

여기서 세 번째가 중요합니다. 테스트 경계 바깥의 것은 전부 다 채우려 하기보다 적절히 모킹하고, 내 코드가 책임지는 분기와 판단을 먼저 확인해야 합니다. 즉, 커버리지도 결국 테스트 경계를 따라 읽어야 합니다.

### 정리

커버리지 보고서는 테스트 품질 자체를 판정하는 도구가 아닙니다. 대신 아직 보지 못한 줄과 분기를 알려 주는 도구입니다. 따라서 커버리지는 높고 낮음만 볼 것이 아니라, 어떤 경로가 비어 있는지와 그 경로가 현재 구조에서 왜 중요한지를 함께 해석해야 합니다.

## **8-02 유지보수 가능한 Express 테스트 규칙**

### 테스트 경계와 유지보수

테스트는 한 번 쓰고 끝나는 코드가 아닙니다. 실제 비용은 그다음부터 생깁니다. 응답 형식이 바뀌고, 서비스 규칙이 수정되고, 의존성이 늘어날 때 기존 테스트가 얼마나 덜 흔들리는지가 중요합니다. 그래서 좋은 테스트는 많이 쌓는 것보다, 경계를 잘 지킨 채 오래 유지되는 쪽이 더 가치가 있습니다.

class 기반 Express에서는 이 기준이 더 분명합니다. Controller는 HTTP 입출력을, Service는 규칙과 판단을, Repository와 Provider는 외부 의존성 경계를 담당합니다. 테스트도 이 구분을 그대로 따라가야 합니다. 한 테스트가 여러 계층의 책임을 동시에 붙잡기 시작하면 작은 수정에도 테스트가 크게 흔들립니다.

### 내 코드는 테스트하고, 바깥 의존성은 대역으로 다루기

실무에서 자주 쓰는 기준은 단순합니다. 내 코드는 직접 검증하고, 바깥 의존성은 테스트 대역으로 좁혀 다루는 것입니다. 여기서 말하는 바깥 의존성은 Express의 `req`, `res`, `next`, 데이터베이스 드라이버, 외부 라이브러리, 네트워크 API처럼 현재 테스트의 관심사 바깥에 있는 것들입니다.

여기서 용어도 마지막으로 한 번 정리해 두는 편이 좋습니다.

- `stub`: 동작을 고정한 테스트용 객체
- `spy`: 호출 여부와 인자를 추적하는 도구
- `mock function`: `jest.fn()`, `spyOn()`처럼 호출을 관찰하거나 대체하는 함수
- `test/mock` 데이터: 이 저장소에서 고정 테스트 데이터를 두는 폴더

이 기준을 실제 계층에 대입하면 더 분명해집니다.

- 내 코드: `Controller`, `Service`, 비즈니스 규칙
- 경계 바깥 코드: Express, DB 드라이버, 외부 API, 암호화 라이브러리

이 기준이 중요한 이유는 테스트 경계를 분명하게 잡아 주기 때문입니다. Controller 테스트에서 정말 보고 싶은 것은 `res.status(401)`이 호출되었는지, `next(error)`가 넘어갔는지, 응답 본문이 맞는지입니다. Express 내부가 `req`를 어떻게 구성하는지까지 이 테스트에서 전부 검증하려 들면, 테스트는 금방 무거워집니다.

반대로 Service 테스트에서는 HTTP 상태 코드보다 규칙과 판단이 더 중요합니다. 입력이 다르면 결과가 어떻게 달라지는지, 예외일 때 어떤 값을 던지는지, 외부 의존성이 실패했을 때 어떤 흐름으로 끝나는지가 중심입니다. 이 구분이 잡히면 테스트는 읽기 쉽고, 수정할 때도 어디가 깨졌는지 더 빨리 좁힐 수 있습니다.

### 좋은 테스트 이름의 기준

유지보수 가능한 테스트는 이름부터 다릅니다. 좋은 테스트 이름은 구현 세부보다 행동과 결과를 먼저 드러냅니다. 테스트 이름을 읽었을 때 무엇을 확인하는지 바로 보여야 이후 유지보수도 더 쉬워집니다.

예를 들어 아래 둘을 비교해 보면 차이가 분명합니다.

- `status와 send를 호출한다`
- `로그인 정보가 없으면 401과 안내 메시지를 응답한다`

첫 번째 이름은 구현 세부를 알고 있어야 의미가 보입니다. 두 번째 이름은 이 테스트가 어떤 상황과 결과를 다루는지 바로 드러납니다. 이후 구현이 `send`에서 `json`으로 바뀌어도 행동과 결과가 같다면 테스트 이름은 유지할 수 있습니다. 그래서 테스트 이름은 메서드 이름보다 계약을 먼저 적는 편이 좋습니다.

조건, 행동, 결과 순서로 이름을 붙이면 더 읽기 쉬워집니다. 흔히 Given / When / Then이라고 부르는 구조를 그대로 한국어 문장으로 풀어 적는다고 생각하면 됩니다.

### 한 테스트에 한 의도 두기

테스트가 읽기 어려워지는 가장 흔한 이유 중 하나는, 한 테스트 안에 여러 의도를 한꺼번에 넣는 것입니다. 로그인 성공, 인증 쿠키 설정, 리다이렉트, 로그 기록, 후속 작업 호출까지 모두 한 테스트에 넣으면 실패했을 때 어디가 깨졌는지 바로 알기 어렵습니다.

예를 들면 아래처럼 너무 많은 것을 한 번에 확인하는 테스트는 읽기도 어렵고 실패 원인도 늦게 보입니다. 아래 두 블록은 같은 `describe` 안에서 `environment`와 `loginPayload`가 이미 준비되어 있다고 가정한 개념 예시입니다.

```tsx
// ... 생략 ...
test("로그인 전체 흐름", async () => {
  const response = await request(
    environment.application.app,
  )
    .post("/auth/login")
    .send(loginPayload);

  expect(response.status).toBe(200);
  expect(response.headers["set-cookie"]).toBeDefined();
  expect(response.body.user.email).toBe(
    "teacher@example.com",
  );
});
```

같은 흐름이라도 의도를 나누면 테스트가 더 읽기 쉬워집니다. 아래도 실제 파일 조각을 그대로 옮긴 것이 아니라, 이름과 의도를 어떻게 나눌지 보여 주는 개념 예시입니다.

```tsx
// ... 생략 ...
const login = () =>
  request(environment.application.app)
    .post("/auth/login")
    .send(loginPayload);

test("올바른 로그인 정보면 200을 반환합니다", async () => {
  const response = await login();

  expect(response.status).toBe(200);
});

test("로그인에 성공하면 인증 쿠키를 설정합니다", async () => {
  const response = await login();

  expect(response.headers["set-cookie"]).toBeDefined();
});
```

이럴 때는 검증 대상을 더 잘게 나누는 편이 좋습니다.

- 로그인 성공 자체를 확인하는 테스트
- 로그인 후 인증 쿠키 상태를 확인하는 테스트
- 실패 시 401을 반환하는 테스트

이렇게 나누면 테스트 개수는 늘어날 수 있지만, 각 테스트의 의도는 더 선명해집니다. 다만 너무 잘게 쪼개면 테스트 수가 빠르게 늘어날 수 있으므로, 서로 강하게 연결된 검증은 한 테스트 안에 함께 두는 판단도 필요합니다. 핵심은 개수가 아니라 실패 원인을 바로 읽을 수 있느냐입니다.

### 복잡한 로직은 테스트보다 코드 분리 먼저

테스트가 지나치게 복잡해질 때는 억지로 assertion만 늘리기보다 로직을 함수로 분리하는 쪽이 더 낫습니다. 이 기준은 유지보수 관점에서 매우 중요합니다.

예를 들어 로그인 핸들러 안에 입력 검증, 해시 비교, 토큰 생성, 저장까지 한꺼번에 들어 있으면 테스트도 자연스럽게 복잡해집니다. 아래 코드는 특정 실제 파일 조각이 아니라, 책임이 한곳에 너무 많이 모인 상태를 보여 주는 개념 예시입니다.

```tsx
// 좋지 않은 예
login() {
  // 입력 검증
  // 비밀번호 비교
  // 토큰 생성
  // 저장
}
```

이때는 핸들러가 모든 책임을 품은 상태라서 테스트도 길어질 수밖에 없습니다. 더 좋은 방향은 검증과 토큰 생성 같은 일부 책임을 바깥 함수나 서비스로 빼고, 그 단위를 따로 테스트하는 것입니다.

즉, 복잡한 테스트는 종종 테스트가 나쁘다기보다 코드 경계가 아직 덜 정리되었다는 신호일 수 있습니다. 유지보수 가능한 테스트를 만들려면, 필요할 때는 테스트가 아니라 코드 구조를 먼저 고쳐야 합니다.

### 실전 규칙 정리

유지보수 가능한 Express 테스트를 위해서는 아래 규칙을 계속 확인하면 좋습니다.

- 테스트 이름은 행동과 결과를 먼저 드러냅니다.
- 한 테스트에는 한 의도만 둡니다.
- Controller는 HTTP 계약만, Service는 규칙만, Repository와 Provider는 의존성 경계만 봅니다.
- 내 코드는 테스트하고, 경계 바깥은 적절히 모킹합니다.
- 테스트가 지나치게 복잡하면 코드 구조를 먼저 나눕니다.
- 시간, 데이터, 리소스 상태를 반복 가능한 환경으로 고정합니다.

읽기 쉬운 테스트도 외부 의존성 mock setup이 흔들리면 결국 유지보수하기 어렵다는 점은 7장에서 본 Prisma Client 테스트 셋업과 바로 연결됩니다.

반대로 아래 같은 테스트는 피하는 편이 좋습니다.

- `console.log`로만 결과를 확인하고 assertion이 없는 테스트
- mock을 너무 많이 붙여 실제 계약보다 구현 세부만 검증하는 테스트
- 한 테스트에서 여러 계층을 한꺼번에 끌어와 책임이 섞인 테스트

### 정리

유지보수 가능한 테스트의 핵심은 이름, 의도, 경계를 한 번에 흐리지 않는 것입니다. Controller는 HTTP 계약, Service는 규칙, 바깥 의존성은 경계로 남겨 두면 실패했을 때 어느 층이 깨졌는지 더 빨리 좁힐 수 있습니다.

## **8-03 최종 점검 체크리스트**

### 구조 점검

마지막 단계에서 가장 먼저 볼 것은 테스트 개수나 커버리지 숫자가 아닙니다. 먼저 구조가 아직 같은 방향을 유지하고 있는지 점검해야 합니다. 이 교재의 구현 기준은 class 기반 Express입니다. 따라서 아래 항목이 흔들리지 않아야 합니다.

- `main.ts`가 실행 책임을 맡고 있는가
- `App`이 서버 실행보다 조립 책임을 맡고 있는가
- `BaseController`가 `router`와 `routes()` 계약만 맡고 있는가
- `Controller`가 하위 컨트롤러를 묶는 root controller 역할만 맡고 있는가
- `App`이 `errorHandler`를 별도로 등록해 공통 에러 흐름을 맡고 있는가
- `Controller`, `Service`, `Repository`, `Provider`의 역할이 섞이지 않았는가

구조가 흔들리면 테스트도 흔들립니다. 예를 들어 Controller가 서비스 규칙까지 직접 품기 시작하면, 단위 테스트와 통합 테스트의 경계도 흐려집니다. 따라서 최종 점검의 첫 항목은 항상 구조입니다.

### 테스트 점검

구조가 맞다면 다음은 테스트 자체를 봅니다. 여기서는 개수보다 범위와 의도를 더 중요하게 봅니다.

- 단위 테스트가 규칙과 판단을 중심으로 구성되어 있는가
- 통합 테스트가 HTTP 흐름과 조립 상태를 중심으로 구성되어 있는가
- 테스트 이름만 읽어도 상황과 결과가 보이는가
- 실패했을 때 원인을 한 단계 안에서 좁힐 수 있는가
- 커버리지 수치를 올리기 위해 경계 밖까지 억지로 검증하고 있지 않은가

이 체크리스트는 4장부터 8장까지 쌓아 온 테스트 전부에 적용됩니다. 결국 중요한 것은 좋은 테스트가 많다는 사실보다, 무엇을 보려고 만든 테스트인지가 선명한지입니다.

### 실행 점검

테스트가 아무리 잘 써져 있어도 실행 환경이 흔들리면 마지막 단계에서 문제가 생깁니다. 따라서 구조와 테스트를 본 뒤에는 반드시 실행 상태를 다시 확인해야 합니다.

다만 최종 실습 시작 직후에는 아직 spec 파일이 없을 수 있으므로, `pnpm test`가 실패해도 이상하지 않습니다. 최소 한 개 이상의 테스트 파일을 만든 뒤 아래 명령으로 점검합니다.

이 장의 최종 실습에서는 아래 명령이 기본 점검 항목입니다.

터미널

```bash
pnpm run typecheck
pnpm test
pnpm run test:cov
```

필요하면 아래 명령도 함께 봅니다.

터미널

```bash
pnpm run test:runInBand
pnpm run test:watch
```

문제가 생기면 아래 순서로 좁혀 가는 편이 좋습니다.

1. `pnpm test`로 기본 실패를 먼저 확인합니다.
2. `pnpm run test:runInBand`로 순차 실행했을 때도 같은 문제가 나는지 봅니다.
3. 종료가 되지 않거나 환경이 흔들리면 열린 리소스 정리를 다시 확인합니다.

이 명령을 통해 다음을 확인할 수 있습니다.

- TypeScript, ESM, Jest 설정이 서로 충돌하지 않는가
- 테스트가 반복 실행될 때 같은 결과를 유지하는가
- 커버리지 리포트가 실제로 생성되는가
- 열린 핸들이 남아 실행을 붙잡지 않는가

최종 실습을 실제로 옮길 때는 작성 순서도 같이 고정해 두는 편이 좋습니다.

1. 먼저 `codeit-fs-jest-practice` 저장소의 `08-final-practice/src/` 현재 런타임 구조를 읽고, `AuthService.login()`과 `AuthController.login()`처럼 지금 있는 책임부터 단위 테스트로 검증합니다. 필요하면 5장의 테스트 패턴을 참고합니다.
2. 그다음 `App`, `Controller`, `HealthController`, `AuthController`가 실제로 연결되는 `/health`, `/auth/login` 흐름을 통합 테스트로 붙입니다. 이 단계에서는 6장에서 본 기준대로 `test/integration/app.spec.ts`를 만들고, 필요하면 같은 장에서 본 방식대로 `test/helper/app-environment.ts` 같은 환경 helper를 추가합니다.
3. 마지막에 `pnpm run typecheck`, `pnpm test`, `pnpm run test:cov`, 필요하면 `pnpm run test:runInBand` 순서로 점검합니다. 최종 비교가 필요하고 `codeit-fs-jest-completed` 저장소가 준비되어 있다면 `08-test-quality-and-wrap-up`를 참고합니다.

### 최종 확인 항목

마지막으로는 아래 질문에 모두 예라고 답할 수 있어야 합니다.

- Jest가 현재 Express 프로젝트에서 어떤 역할을 하는지 설명할 수 있는가
- 단위 테스트와 통합 테스트의 경계를 계층 기준으로 설명할 수 있는가
- 커버리지 수치를 품질과 구분해서 해석할 수 있는가
- 반복 실행 가능한 테스트 환경을 직접 준비할 수 있는가

이 기준을 만족하면 실무 프로젝트에서도 테스트를 더 안정적으로 유지할 수 있습니다.

### 정리

마지막 점검에서는 구조, 테스트 범위, 실행 환경을 따로 확인해야 합니다. 이 세 가지가 같은 방향을 가리킬 때 Express 테스트는 단순한 실행 확인이 아니라 구조와 계약을 오래 지키는 도구로 남습니다.
