# 3. Git Flow: 시나리오로 익히기 (초보자용)

Git Flow는 여러 개발자가 협업할 때 사용하는 규칙 중 하나입니다. 이번 시간에는 컴퓨터에 폴더를 만드는 것부터 시작해서, 실제 프로젝트처럼 Git Flow를 단계별로 연습해 보겠습니다.

**시나리오: 간단한 계산기 프로그램 만들기**

---

### 1. 프로젝트 시작하기 (바탕화면에서)

가장 먼저, 우리 프로젝트를 위한 폴더를 만들고 Git을 시작하겠습니다.

1.  **바탕화면에 `git-flow-practice` 폴더를 만드세요.**

2.  **터미널(Git Bash, PowerShell 등)을 열고 다음 명령어를 입력하여 폴더로 이동합니다.**
    ```bash
    cd Desktop/git-flow-practice
    ```

3.  **Git을 시작하고, 첫 파일을 만듭니다.**
    ```bash
    # 이 폴더를 Git으로 관리하겠다고 선언합니다.
    git init
    ```
    > 이제 VS Code를 열고, 이 `git-flow-practice` 폴더를 열어주세요.
    >
    > 폴더가 열리면, VS Code의 파일 탐색기에서 `calculator.js`라는 새 파일을 만듭니다.

    ```bash
    # calculator.js 파일을 Git에 추가하고 첫 커밋(버전)을 만듭니다.
    git add calculator.js
    git commit -m "Initial commit: Add calculator.js"
    ```
    이제 우리는 `main`이라는 기본 브랜치에 첫 번째 코드 버전을 저장했습니다.

---

### 2. `main`과 `develop` 브랜치 설정

Git Flow에서는 항상 두 개의 중심 브랜치를 사용합니다.

-   **`main`**: 출시된, 안정적인 버전의 코드만 보관하는 브랜치입니다. (사용자가 실제로 사용하는 코드)
-   **`develop`**: 앞으로 개발할 기능들이 합쳐지는 브랜치입니다. (개발의 중심)

```bash
# 현재 main 브랜치에서 develop 브랜치를 만듭니다.
git branch develop
```

이제 우리는 `main`과 `develop`이라는 두 개의 브랜치를 갖게 되었습니다. `main` 브랜치의 현재 상태를 첫 공식 버전인 **`v1.0.0`** 으로 표시해줍시다. **태그(Tag)**는 특정 버전을 기념하는 꼬리표와 같습니다.

> **💡 버전은 어떻게 표시하나요? (Semantic Versioning)**

> `v1.0.0`과 같은 버전 표시는 **유의적 버전(Semantic Versioning)**이라는 규칙을 따르는 것이 일반적입니다. 보통 **SemVer(셈버)**라고 줄여 읽습니다. 버전은 `MAJOR.MINOR.PATCH` 형식으로 구성됩니다.>
> -   **MAJOR (1.x.x)**: 기존 버전과 호환되지 않는 큰 변경이 있을 때 올립니다. (예: 프로그램의 구조가 완전히 바뀔 때)
> -   **MINOR (x.1.x)**: 새로운 기능이 추가되었지만, 기존 버전과 호환될 때 올립니다. (예: '더하기' 기능 추가)
> -   **PATCH (x.x.1)**: 기존 기능의 버그를 수정했을 때 올립니다. (예: '더하기'가 잘못 계산되는 버그 수정)
>
> 앞에 붙는 `v`는 'version'을 의미하는 관례적인 표시입니다.

```bash
# v1.0.0 태그를 만듭니다.
git tag -a "v1.0.0" -m "Initial release version 1.0.0"
```

> **💡 Git 그래프로 변화 확인하기**
>
> VS Code에 [Git Graph](https://marketplace.visualstudio.com/items?itemName=mhutchie.git-graph)와 같은 확장을 설치하면, 지금까지의 작업 흐름을 시각적으로 멋지게 확인할 수 있습니다.
>
> 현재 그래프는 아래와 같은 모습일 것입니다. `main`과 `develop` 브랜치가 같은 커밋을 가리키고 있습니다. (자세한 내용은 VS Code의 Git Graph로 확인해 보세요.)
>
> ```text
> * (HEAD -> main, tag: v1.0.0, develop) Initial commit: Add calculator.js
> ```

### 3. 새로운 기능 개발: `feature` 브랜치

'더하기' 기능을 만들어 보겠습니다. Git Flow에서는 새로운 기능은 `feature`라는 이름의 브랜치에서 따로 작업합니다.

1.  **개발의 중심인 `develop` 브랜치로 이동합니다.**
    ```bash
    git switch develop
    ```

2.  **'더하기 기능'을 위한 `feature/add-function` 브랜치를 만듭니다.**
    ```bash
    git switch -c feature/add-function
    ```

3.  **`calculator.js` 파일에 더하기 코드를 추가하고 커밋합니다.**
    ```javascript
    // calculator.js 파일에 아래 내용 추가
    export const add = (a, b) => a + b;
    ```
    ```bash
    git add calculator.js
    git commit -m "feat: Add 'add' function"
    ```

    > **💡 커밋 메시지는 어떻게 쓰는 게 좋을까요? (Conventional Commits)**
    >
    > 커밋 메시지 앞에 `feat:`, `fix:` 같은 꼬리표를 붙이는 것을 **Conventional Commits** 규칙이라고 합니다. 이 규칙을 따르면 커밋 기록만 봐도 어떤 변경사항인지 쉽게 알 수 있습니다.
    >
    > -   **`feat`**: 새로운 기능 추가
    > -   **`fix`**: 버그 수정
    > -   **`docs`**: 문서 수정
    > -   **`style`**: 코드 스타일 변경 (포맷팅, 세미콜론 등)
    > -   **`refactor`**: 코드 리팩토링 (기능 변경 없이 코드 구조 개선)
    > -   **`chore`**: 빌드, 패키지 매니저 설정 등 기타 잡다한 작업
    >
    > 앞으로 이 규칙에 따라 커밋 메시지를 작성해 봅시다.

    > **💡 Git 그래프로 변화 확인하기**
    >
    > `feature/add-function` 브랜치가 `develop`에서 뻗어 나와 새로운 커밋이 생긴 것을 볼 수 있습니다. (자세한 내용은 VS Code의 Git Graph로 확인해 보세요.)
    >
    > ```text
    > * (HEAD -> feature/add-function) feat: Add 'add' function
    > |
    > * (develop, tag: v1.0.0, main) Initial commit: Add calculator.js
    > ```

4.  **기능 개발이 끝났으니, `develop` 브랜치에 합칩니다.**
    ```bash
    # develop 브랜치로 다시 돌아가서
    git switch develop

    # feature/add-function 브랜치의 내용을 합칩니다.
    git merge --no-ff feature/add-function

    # 작업이 끝난 기능 브랜치는 삭제합니다.
    git branch -d feature/add-function
    ```
    이제 `develop` 브랜치에는 '더하기' 기능이 추가되었습니다.

    > **💡 Git 그래프로 변화 확인하기**
    >
    > `feature` 브랜치가 `develop` 브랜치로 병합되었습니다. `--no-ff` 옵션 덕분에 기능 개발의 흐름이 그래프에 명확하게 기록으로 남습니다. (자세한 내용은 VS Code의 Git Graph로 확인해 보세요.)
    >
    > ```text
    > *   (HEAD -> develop) Merge branch 'feature/add-function' into develop
    > |\
    > | * feat: Add 'add' function
    > |/
    > * (tag: v1.0.0, main) Initial commit: Add calculator.js
    > ```

### 4. 버전 출시 준비: `release` 브랜치

`develop` 브랜치에 기능들이 모였으니, `v1.1.0`으로 정식 출시를 준비해 봅시다. 이때 사용하는 것이 `release` 브랜치입니다.

> **🤔 `release` 브랜치는 왜 필요한가요? 그냥 `tag`를 쓰면 안 되나요?**
> 좋은 질문입니다! **`tag`는 출시가 완료된 버전을 기념하는 꼬리표**입니다. 반면, **`release` 브랜치는 출시를 '준비'하는 공간**입니다.
> 이 브랜치에서는 버전 번호를 올리거나, 마지막 테스트에서 발견된 사소한 버그를 고치는 등 출시 직전의 마무리 작업을 합니다. 이 공간 덕분에 `develop` 브랜치는 멈추지 않고 다음 버전에 추가될 새로운 기능 개발을 계속할 수 있습니다.

1.  **`develop` 브랜치에서 `release/v1.1.0` 브랜치를 만듭니다.**
    ```bash
    git switch -c release/v1.1.0 develop
    ```

2.  **출시 준비를 합니다. (예: 버전 정보 파일 업데이트)**
    ```bash
    # version.txt 파일을 만들고 버전 정보를 기록
    echo "Version: 1.1.0" > version.txt
    git add version.txt
    git commit -m "chore: Bump version to 1.1.0"
    ```

3.  **출시 준비가 완료되면, `main`과 `develop`에 합칩니다.**
    -   `main` 브랜치에 합쳐서 **정식 출시**합니다.
    -   `develop` 브랜치에도 합쳐서, 버전 번호를 올린 변경사항을 개발 브랜치에도 반영합니다.

    ```bash
    # 1. main 브랜치에 합치기
    git switch main
    git merge --no-ff release/v1.1.0

    # 2. 출시된 main 브랜치에 v1.1.0 태그 달기!
    git tag -a "v1.1.0" -m "Release version 1.1.0"

    # 3. develop 브랜치에 합치기
    git switch develop
    git merge --no-ff release/v1.1.0

    # 4. 이제 필요 없어진 release 브랜치 삭제
    git branch -d release/v1.1.0
    ```

    > **💡 Git 그래프로 변화 확인하기**
    >
    > `release` 브랜치에서 준비한 내용이 `main`과 `develop`에 모두 합쳐졌습니다. `main` 브랜치에는 `v1.1.0` 태그가 추가되었습니다. (자세한 내용은 VS Code의 Git Graph로 확인해 보세요.)
    >
    > ```text
    > *   (HEAD -> develop) Merge branch 'release/v1.1.0' into develop
    > |\
    > * |   (tag: v1.1.0, main) Merge branch 'release/v1.1.0'
    > |\ \
    > | |/
    > | * chore: Bump version to 1.1.0
    > |/
    > *   Merge branch 'feature/add-function' into develop
    > |\
    > | * feat: Add 'add' function
    > |/
    > * (tag: v1.0.0) Initial commit: Add calculator.js
    > ```

### 5. 긴급 버그 수정: `hotfix` 브랜치

출시한 `v1.1.0` 버전에 치명적인 버그가 발견되었습니다! 당장 고쳐야 합니다. 이럴 때 `hotfix` 브랜치를 사용합니다.

1.  **'출시된 버전'인 `main` 브랜치에서 `hotfix/critical-bug` 브랜치를 만듭니다.**
    ```bash
    git switch -c hotfix/critical-bug main
    ```

2.  **버그를 수정하고 커밋합니다.** (예: 더하기가 아니라 빼기로 잘못 동작하는 버그 수정)
    ```bash
    # 버그 수정...
    git add calculator.js
    git commit -m "fix: Fix critical bug in add function"
    ```

3.  **수정이 끝나면 `main`과 `develop`에 모두 합칩니다.**
    -   `main`에 합쳐서 **빨리 재배포**해야 합니다.
    -   `develop`에도 합쳐서, 개발 중인 코드에도 버그 수정이 반영되도록 합니다.

    ```bash
    # 1. main 브랜치에 합치기
    git switch main
    git merge --no-ff hotfix/critical-bug

    # 2. 버그를 수정한 새 버전에 태그 달기 (v1.1.1)
    git tag -a "v1.1.1" -m "Hotfix for critical bug"

    # 3. develop 브랜치에 합치기
    git switch develop
    git merge --no-ff hotfix/critical-bug

    # 4. 작업이 끝난 hotfix 브랜치 삭제
    git branch -d hotfix/critical-bug
    ```

    > **💡 Git 그래프로 변화 확인하기**
    >
    > `main` 브랜치에서 시작된 `hotfix`가 `main`과 `develop`에 모두 적용되었습니다. `main`에는 새로운 `v1.1.1` 태그가 생겼습니다. (자세한 내용은 VS Code의 Git Graph로 확인해 보세요.)
    >
    > ```text
    > *   (HEAD -> develop) Merge branch 'hotfix/critical-bug' into develop
    > |\
    > * |   (tag: v1.1.1, main) Merge branch 'hotfix/critical-bug'
    > |\ \
    > | |/
    > | * | fix: Fix critical bug in add function
    > | |/
    > |/|
    > * |   (tag: v1.1.0) Merge branch 'release/v1.1.0'
    > |\|
    > ... (이전 기록)
    > ```

---

### 정리

-   **`main`**: 항상 안정적인 최신 버전
-   **`develop`**: 다음 버전을 위한 개발이 진행되는 곳
-   **`feature`**: 새로운 기능 개발
-   **`release`**: 버전 출시 준비
-   **`hotfix`**: 긴급 버그 수정
-   **`tag`**: 출시된 버전을 기념하는 꼬리표

이 흐름을 따르면 여러 사람이 함께 일해도 코드 버전이 꼬이지 않고, 안정적으로 프로젝트를 관리할 수 있습니다.
