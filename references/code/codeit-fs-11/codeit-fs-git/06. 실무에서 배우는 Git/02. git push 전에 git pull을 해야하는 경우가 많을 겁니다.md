### 02. git push 전에 git pull을 해야하는 경우가 많을 겁니다

이전에는 `git push`로 로컬의 변경사항을 원격 저장소에 올리고, `git pull`로 원격 저장소의 변경사항을 로컬로 가져온다고 배웠습니다.

이번 시간에는 여러 개발자가 함께 작업할 때 `git push`가 실패하는 매우 흔한 상황과, 이를 해결하기 위해 왜 `git pull`을 먼저 해야 하는지 알아보겠습니다.

---

#### **1. 상황 설정: 로컬과 원격의 내용이 달라지는 경우**

먼저, 로컬 저장소와 원격 저장소의 내용이 달라지는 상황을 시뮬레이션 해보겠습니다. 한 명은 로컬에서, 다른 한 명(또는 나 자신)은 원격(GitHub)에서 같은 파일을 다르게 수정하는 상황입니다.

**1-1. 로컬 저장소에서 작업**

-   `premium` 브랜치에서 `LICENSE` 파일에 '기업용 요금' 정보를 추가하고 커밋합니다.

    ```bash
    # premium 브랜치로 이동
    git switch premium

    # LICENSE 파일에 내용 추가 (예: "Premium Plan for Enterprise: $100/month")
    # ... 파일 수정 후 저장 ...

    # 변경 내용 커밋
    git add LICENSE
    git commit -m "Add enterprise pricing"
    ```

**1-2. 원격 저장소(GitHub)에서 작업 (동료의 작업 시뮬레이션)**

-   GitHub 웹사이트에서 직접 `premium` 브랜치의 `LICENSE` 파일을 수정합니다. '교육용 요금' 정보를 추가하고 커밋합니다.
    -   GitHub 저장소 > `premium` 브랜치 선택 > `LICENSE` 파일 클릭 > 연필 모양(Edit) 아이콘 클릭
    -   파일 내용 수정 (예: "Premium Plan for Education: $10/month")
    -   "Commit changes" 버튼을 눌러 원격 저장소에 직접 커밋

-   **상황 분석**: 이제 로컬 저장소의 `premium` 브랜치와 원격 저장소의 `premium` 브랜치는 **같은 부모 커밋에서 갈라져 나와(Diverged)** 각자 다른 커밋을 가지게 되었습니다.

---

#### **2. Push 실패와 그 원인**

이 상태에서, 로컬의 '기업용 요금' 커밋을 원격 저장소로 `push` 해봅시다.

```bash
git push
```

**명령 결과 (터미널 출력 예) - 💥 Push 실패!**

```
To github.com:user/repo.git
 ! [rejected]        premium -> premium (non-fast-forward)
error: failed to push some refs to 'github.com:user/repo.git'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. This is usually caused by another repository pushing
hint: to the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
```

-   **실패 원인**: `non-fast-forward` 에러는 "원격 저장소에 당신이 가지고 있지 않은 새로운 변경사항(커밋)이 있으니, `push`를 할 수 없습니다" 라는 의미입니다. 만약 이 `push`가 성공한다면 원격 저장소에 있던 동료의 '교육용 요금' 커밋이 사라져버릴 수 있기 때문에, Git이 안전장치로서 이를 막는 것입니다.

---

#### **3. Git Pull 시도와 Pull 전략 선택**

Git의 제안대로, 원격 저장소의 변경 내용을 먼저 가져와 통합하기 위해 `git pull`을 실행합니다.

```bash
git pull
```

**명령 결과 (터미널 출력 예) - 💥 Pull 설정 필요!**

```
...
You have divergent branches and need to specify how to reconcile them.
You can do so by running one of the following commands sometime before
your next pull:

  git config pull.rebase false  # merge (the default strategy)
  git config pull.rebase true   # rebase
  git config pull.ff only       # fast-forward only

You can replace "git config" with "git config --global" to set a default
preference for all repositories. You can also pass --rebase, --no-rebase,
or --ff-only on the command line to override the configured default per
invocation.
fatal: Need to specify how to reconcile divergent branches.
```

-   **실패 원인**: Git은 서로 갈라진 브랜치를 합쳐야 하는데, 어떤 방식(`merge`, `rebase`, `fast-forward only`)으로 합칠지 개발자에게 정해달라고 요청하는 것입니다.
    -   `--rebase`: 커밋 히스토리를 한 줄로 깔끔하게 만들지만, 흐름이 복잡해져 초보자에게는 혼란스러울 수 있습니다.
    -   `--no-rebase` (또는 `--merge`): 두 브랜치의 내역을 그대로 유지하며, 별도의 병합 커밋(Merge Commit)을 만듭니다. 히스토리 추적이 용이하여 협업 시 권장됩니다.
    -   `--ff-only`: 현재 상황처럼 로컬과 원격 모두에 새 커밋이 있으면 사용할 수 없습니다.

---

#### **4. 충돌 해결 및 병합 완료**

우리는 히스토리를 명확하게 남기기 위해 `merge` 방식을 사용할 것입니다. `--no-rebase` 옵션을 주어 `pull`을 다시 실행합니다.

```bash
git pull --no-rebase
```

**명령 결과 (터미널 출력 예) - 💥 또 다른 충돌!**

```
Auto-merging LICENSE
CONFLICT (content): Merge conflict in LICENSE
Automatic merge failed; fix conflicts and then commit the result.
```

-   **상황 분석**: `git pull`은 `git fetch` (원격 내용 가져오기)와 `git merge` (병합하기)가 합쳐진 명령어입니다. 원격의 '교육용 요금' 커밋과 로컬의 '기업용 요금' 커밋이 같은 파일의 같은 부분을 수정했기 때문에, 병합 과정에서 **Merge Conflict**가 발생했습니다.
-   이제 이전 시나리오에서 배운 대로, VS Code에서 충돌을 해결합니다.
    1.  `LICENSE` 파일을 열어 충돌 표시자(`<<<`, `===`, `>>>`)를 확인합니다.
    2.  두 요금 정보를 모두 남기기로 결정하고, 원하는 최종 모습으로 코드를 직접 수정 후 저장합니다.
    3.  `git add LICENSE` 와 `git commit`을 차례로 실행하여 병합을 완료합니다.

---

#### **5. 최종 Push 성공**

이제 로컬 `premium` 브랜치는 원격의 변경 내용과 로컬의 변경 내용을 모두 포함한 최신 상태가 되었습니다. 마지막으로 `push`를 실행합니다.

```bash
git push
```

**명령 결과 (터미널 출력 예)**

```
...
To github.com:user/repo.git
   a1b2c3d..e4f5g6h  premium -> premium
```

-   이번에는 아무런 에러 없이 `push`가 성공합니다. GitHub 저장소에서 `LICENSE` 파일과 커밋 히스토리를 확인하면, 모든 변경 내용이 안전하게 통합된 것을 볼 수 있습니다.

---

### **요약: 협업의 기본 흐름**

1.  내 작업을 `commit`한다.
2.  `push`를 시도한다.
3.  `non-fast-forward` 에러로 `push`가 실패하면, 다른 팀원이 새로운 내용을 올렸다는 신호이다.
4.  `git pull --no-rebase` (또는 `--merge`) 명령어로 원격 저장소의 변경 내용을 가져와 내 로컬 브랜치와 병합한다.
5.  만약 `pull` 과정에서 충돌(Conflict)이 발생하면, 파일을 직접 수정하여 해결하고 커밋한다.
6.  모든 내용이 통합된 로컬 브랜치를 다시 `push`한다.

이 흐름은 실무 협업에서 매우 빈번하게 발생하므로, 꼭 익숙해지는 것이 좋습니다.
