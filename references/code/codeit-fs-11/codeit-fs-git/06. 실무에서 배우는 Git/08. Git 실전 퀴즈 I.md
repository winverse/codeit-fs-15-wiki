### 08. Git 실전 퀴즈 I

지금까지 배운 실무 Git 지식을 퀴즈를 통해 점검해 보세요.

---

### 질문 1

로컬에서 `git push`를 실행했을 때, `non-fast-forward` 에러가 발생하며 `push`가 거부되었습니다. 이 에러가 발생하는 가장 일반적인 이유는 무엇일까요?

<details>
<summary>정답 및 해설 보기</summary>

원격 저장소에 내가 로컬에 가지고 있지 않은 새로운 커밋(즉, 다른 팀원의 작업물)이 있기 때문입니다. 내 변경사항을 강제로 올리면 다른 사람의 작업이 유실될 수 있어 Git이 이를 막는 것입니다.

</details>

---

### 질문 2

`non-fast-forward` 에러를 마주했을 때, `push`를 성공시키기 위해 가장 먼저 해야 할 올바른 행동은 무엇인가요?

1.  `git push --force`를 실행하여 강제로 올린다.
2.  `git pull`을 실행하여 원격 저장소의 최신 변경 내용을 가져와 내 로컬 브랜치와 통합한다.
3.  `git reset`으로 내 최신 커밋을 취소한다.

<details>
<summary>정답 및 해설 보기</summary>

2번. `git pull`을 통해 원격 저장소의 변경 내용을 먼저 내 로컬에 반영하고, 그 후에 내 작업을 다시 `push`해야 합니다.

</details>

---

### 질문 3

특정 파일의 코드 한 줄 한 줄을 마지막으로 수정한 사람과 커밋이 무엇인지 확인하고 싶을 때 사용하는 명령어는 무엇인가요?

<details>
<summary>정답 및 해설 보기</summary>

`git blame [파일 이름]`

</details>

---

### 질문 4

원격 저장소(GitHub)에 이미 `push`하여 모든 팀원에게 공유된 커밋을 안전하게 취소하고 싶습니다. 다음 중 어떤 명령어를 사용해야 할까요?

1.  `git reset --hard [커밋 아이디]`
2.  `git revert [커밋 아이디]`
3.  `git remove [커밋 아이디]`

<details>
<summary>정답 및 해설 보기</summary>

2번. `git revert`는 기존 커밋을 삭제하는 대신, 변경 내용을 되돌리는 **새로운 커밋**을 생성하므로 공유된 히스토리를 안전하게 수정할 수 있습니다.

</details>

---

### 질문 5

협업 시, 원격 저장소에 이미 올라간 커밋에 대해 `git reset`을 사용하면 안 되는 이유는 무엇인가요?

<details>
<summary>정답 및 해설 보기</summary>

`git reset`은 기존 커밋을 히스토리에서 지워버리기 때문에, 다른 팀원들이 이미 받아 간 커밋 히스토리와 내 히스토리가 달라지게 됩니다. 이는 팀 전체의 저장소를 꼬이게 만드는 매우 위험한 상황을 초래할 수 있습니다.

</details>

---

### 질문 6

`git pull`을 실행했더니, 로컬과 원격 브랜치가 나뉘어(diverged) 어떤 방식으로 합칠지 정하라는 에러가 발생했습니다. 커밋 히스토리를 한 줄로 만들지 않고, 병합 내역을 명확히 남기는 `merge` 방식으로 `pull`을 실행하고 싶을 때 추가해야 할 옵션은 무엇인가요?

<details>
<summary>정답 및 해설 보기</summary>

`--no-rebase` (또는 `--merge`)

</details>

---

### 질문 7

`git blame`으로 특정 코드 라인을 수정한 커밋 아이디를 찾아냈습니다. 이 커밋에서 정확히 어떤 변경 작업이 있었는지 그 상세 내용을 확인하기 위해 이어서 사용하면 좋은 명령어는 무엇인가요?

<details>
<summary>정답 및 해설 보기</summary>

`git show [커밋 아이디]`

</details>
