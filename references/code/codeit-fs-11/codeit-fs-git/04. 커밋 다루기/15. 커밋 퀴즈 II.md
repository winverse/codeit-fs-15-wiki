### 15. 커밋 퀴즈 II

`HEAD`, `reset`, `reflog`, `switch` 등 심화 개념에 대해 잘 이해했는지 퀴즈를 통해 확인해보세요.

---

### 질문 1

`HEAD`는 현재 작업 위치를 알려주는 중요한 포인터입니다. `HEAD`가 가리키는 커밋은 현재 프로젝트의 어떤 상태를 직접적으로 결정하나요?

<details>
<summary>정답 확인하기</summary>

워킹 디렉토리(Working Directory)의 상태. 즉, 현재 내가 보고 수정하는 파일들의 버전입니다.

</details>

---

### 질문 2

로컬 `main` 브랜치에서 `git reset --hard`로 과거의 커밋으로 돌아갔습니다. 이때 `git graph`로 확인하면 원격 저장소의 상태를 나타내는 `origin/main`은 어디에 위치해 있을까요?

1. 로컬 `main`과 함께 과거 커밋으로 이동한다.
2. `reset`과 상관없이, 원래 있던 최신 커밋 위치에 그대로 남아있다.
3. 그래프에서 사라진다.

<details>
<summary>정답 확인하기</summary>

2번. `reset`은 로컬 저장소에만 영향을 주므로 `origin/main`의 위치는 변하지 않습니다. 이로 인해 로컬과 원격의 히스토리가 달라지게(Diverged) 됩니다.

</details>

---

### 질문 3

`git reset --hard`나 `git push --force` 같은 명령어로 인해 `git log` 상에서는 보이지 않게 된 커밋들을 복구해야 하는 상황입니다. `HEAD`가 이동했던 모든 기록을 확인하여 '사라진' 커밋을 찾을 수 있게 해주는 명령어는 무엇일까요?

<details>
<summary>정답 확인하기</summary>

`git reflog`

</details>

---

### 질문 4

브랜치 히스토리를 변경하는 위험한 `reset` 대신, 단순히 과거 특정 커밋 시점의 코드를 **안전하게 확인만 하고 싶을 때** 사용하는 명령어는 무엇인가요? (최신 Git 버전에서 권장하는 명령어로 답변해주세요)

<details>
<summary>정답 확인하기</summary>

`git switch --detach [커밋 아이디]`

</details>

---

### 질문 5

`git switch --detach` 명령어를 사용했을 때 진입하게 되는, 특정 브랜치에 속하지 않고 커밋을 직접 가리키는 상태를 무엇이라고 부르나요?

<details>
<summary>정답 확인하기</summary>

분리된 헤드 (Detached HEAD) 상태

</details>
