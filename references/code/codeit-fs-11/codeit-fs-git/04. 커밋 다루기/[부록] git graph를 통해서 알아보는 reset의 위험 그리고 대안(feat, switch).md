### [부록] Git Graph를 통해 알아보는 reset의 위험과 대안

`git reset`은 강력하지만, 히스토리를 변경하기 때문에 협업 시 위험할 수 있습니다. VS Code의 `Git Graph` 확장 프로그램을 통해 `reset`의 동작과 위험성, 그리고 안전한 대안을 시각적으로 알아보겠습니다.

---

### 1. Git Graph로 히스토리 시각화하기

- **Git Graph**: VS Code의 확장 프로그램으로, 브랜치와 커밋 히스토리를 그래프 형태로 보여주어 복잡한 내역을 한눈에 파악하기 쉽게 도와줌
- **그래프 읽기**:
    - **`main`**: 내 컴퓨터(로컬 저장소)의 브랜치 위치
    - **`origin/main`**: 원격 저장소(GitHub 등)의 브랜치 위치
    - **⚪ (테두리만 있는 원)**: 현재 나의 작업 위치를 나타내는 `HEAD` 포인터

---

### 2. `git reset --hard`의 위험성 시뮬레이션

1.  **과거로 `reset` 실행**: `git reset --hard [과거 커밋 아이디]`를 실행.
    - **결과**: `Git Graph`에서 로컬 `main`과 `HEAD`는 과거 커밋으로 이동하지만, 원격 `origin/main`은 그대로 남아있게 됨. **(히스토리 분기 발생)**

2.  **새로운 커밋 생성**: `reset`한 상태에서 코드를 수정하고 새로운 커밋을 만듦.
    - **결과**: `Git Graph`에서 로컬 `main`은 새로운 커밋을 가리키게 되어, `origin/main`과의 경로가 완전히 갈라짐.

3.  **`push` 시도 및 실패**: 이 상태에서 `git push`를 실행하면 `non-fast-forward` 에러가 발생하며 거부됨.
    - **이유**: 원격 저장소의 커밋이 유실되는 것을 방지하기 위한 Git의 안전장치.

4.  **`push --force` 실행**: `git push --force`로 강제 푸시.
    - **결과**: `Git Graph`에서 원격 `origin/main`이 강제로 로컬 `main`의 위치로 옮겨짐. **원격 저장소에 있던 기존 커밋 내역이 그래프에서 사라짐.** 이는 팀 전체의 작업을 유실시킬 수 있는 매우 위험한 상황.

---

### 3. `git reflog`: 실수 복구를 위한 안전망

만약 `force push`로 히스토리를 잘못 변경했더라도, `git reflog` 명령어로 대부분 복구할 수 있습니다.

- **`reflog`란?**: `HEAD`가 이동했던 모든 기록(commit, reset, checkout 등)을 가지고 있는 로그.
- **복구 방법**:
    1. `git reflog`를 실행하여 돌아가고 싶은 시점의 커밋 아이디를 찾음.
    2. `git reset --hard [reflog에서 찾은 커밋 아이디]`를 실행하여 로컬 저장소를 원하는 상태로 되돌림.
    3. `git push --force`를 다시 실행하여 복구된 상태를 원격 저장소에 덮어씀.

---

### 4. 안전한 대안: `switch`와 `checkout`

히스토리를 파괴하지 않고 **단순히 과거 커밋의 내용을 확인만 하고 싶을 때**는 `reset` 대신 `switch`나 `checkout`을 사용해야 합니다.

- **`git switch --detach [커밋 아이디]`**: **(권장)** 브랜치는 그대로 둔 채 `HEAD`만 특정 커밋으로 이동하여 과거 시점의 코드를 안전하게 둘러볼 수 있음.
- **`git checkout [커밋 아이디]`**: `switch`와 동일하게 동작하지만, 구버전 명령어이며 다른 기능이 많아 혼동의 여지가 있음.

이 명령어들은 "Detached HEAD" 상태가 되는데, 이는 "현재 특정 브랜치에 속하지 않고 임시로 커밋만 둘러보는 중"이라는 의미입니다. 원래 작업하던 브랜치로 돌아가려면 `git switch main`을 실행하면 됩니다.

---

### 용어설명

- **Diverged History**: 로컬과 원격 저장소의 커밋 내역이 서로 다른 방향으로 갈라진 상태.
- **reflog**: `HEAD`가 변경된 모든 기록을 저장하는 로그. 실수로 커밋을 삭제했을 때 복구하는 데 유용.
- **Detached HEAD**: 특정 브랜치에 속하지 않고, 커밋을 직접 가리키고 있는 상태. 과거 버전을 확인하거나 실험적인 작업을 할 때 사용.
