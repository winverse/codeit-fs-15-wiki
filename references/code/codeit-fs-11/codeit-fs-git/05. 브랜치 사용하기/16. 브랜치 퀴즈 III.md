### 16. 브랜치 퀴즈 III

`HEAD`와 브랜치의 관계, 그리고 `git reset`의 동작 원리에 대해 정확히 이해했는지 퀴즈를 통해 확인해보세요.

---

### 질문 1

로컬에서 `feature/login`이라는 새로운 브랜치를 만들어 작업을 완료했습니다. 이 브랜치를 `origin` 원격 저장소에 처음으로 `push`하면서, 앞으로의 `push` 작업을 위해 원격 브랜치와 로컬 브랜치를 연결하려고 합니다. 다음 중 올바른 명령어는 무엇일까요?

1. `git push`
2. `git push origin feature/login`
3. `git push --set-upstream origin feature/login`
4. `git push --u origin`

<details>
<summary>정답 확인하기</summary>

3번. 로컬에서 새로 만든 브랜치를 원격 저장소에 처음 올릴 때는 `--set-upstream` (또는 단축형 `-u`) 옵션을 사용하여 로컬 브랜치가 추적할 원격 브랜치를 명시적으로 지정해주어야 합니다.

</details>

---

### 질문 2

`git log`를 확인했을 때 보이는 `(HEAD -> main)`이 의미하는 바를 가장 정확하게 설명한 것은 무엇일까요?

1. `HEAD`와 `main` 브랜치는 같은 커밋을 가리킨다.
2. `HEAD`는 `main` 브랜치를 가리키고, `main` 브랜치가 특정 커밋을 가리킨다.
3. `main` 브랜치가 `HEAD`를 가리킨다.
4. `HEAD`는 포인터이고, `main`은 그냥 이름이다.

<details>
<summary>정답 확인하기</summary>

2번. `HEAD`는 현재 작업중인 '브랜치'를 가리키는 포인터입니다. 따라서 `HEAD`는 `main` 브랜치를 가리키고, 그 `main` 브랜치가 최신 커밋을 가리킴으로써, `HEAD`는 간접적으로 현재 커밋을 가리키게 됩니다.

</details>

---

### 질문 3

`main` 브랜치에서 작업 중일 때, `git reset --hard abc1234` 명령어를 실행했습니다. 이 명령의 결과로 발생하는 일을 가장 정확하게 설명한 것은 무엇일까요?

1. `HEAD` 포인터가 `abc1234` 커밋을 직접 가리키도록 변경된다.
2. `main` 브랜치가 삭제되고, `HEAD`가 `abc1234` 커밋을 가리킨다.
3. `HEAD`는 계속 `main` 브랜치를 가리키고, `main` 브랜치가 가리키는 대상이 `abc1234` 커밋으로 변경된다.
4. `abc1234` 커밋 이후에 있던 모든 커밋들이 삭제된다.

<details>
<summary>정답 확인하기</summary>

3번. `git reset`은 `HEAD`가 가리키는 대상을 바꾸는 것이 아니라, `HEAD`가 가리키고 있는 '브랜치'가 가리키는 커밋을 변경하는 명령어입니다. 그 결과 `HEAD`가 간접적으로 가리키는 커밋이 바뀌게 됩니다.

</details>

### 질문 4

(O/X 퀴즈) `git reset`을 사용하여 과거의 커밋으로 돌아가면, 그 이후에 있던 커밋들은 모두 영구적으로 삭제된다.

<details>
<summary>정답 확인하기</summary>

X (거짓). `git reset`은 브랜치 포인터의 위치만 옮길 뿐, 기존의 커밋을 삭제하지 않습니다. 커밋들은 여전히 Git 데이터베이스에 남아있으며, 해당 커밋의 ID를 알고 있다면 다시 `reset`하여 돌아갈 수도 있습니다.

</details>

### 질문 5

로컬 브랜치가 추적하도록 설정된 원격 브랜치를 무엇이라고 부르나요? `push`할 때 변경사항이 올라가는 기준점이 되는 이 브랜치를 지칭하는 용어는 무엇일까요?

<details>
<summary>정답 확인하기</summary>

업스트림 브랜치(Upstream Branch)입니다.

</details>
