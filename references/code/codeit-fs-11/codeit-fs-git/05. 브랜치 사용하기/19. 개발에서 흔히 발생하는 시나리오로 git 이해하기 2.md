### 19. 개발에서 흔히 발생하는 시나리오로 git 이해하기 2

이전 시나리오에서는 새로운 기능을 만들고 합치는 순탄한 과정을 경험했습니다. 하지만 실제 협업은 항상 순탄하지만은 않습니다. 이번에는 팀 동료와 나의 작업이 겹쳤을 때 발생하는 **충돌(Conflict)**을 직접 해결해보는, 조금 더 현실적인 시나리오를 진행하겠습니다.

---

### **시나리오 2: 팀원과 협업하며 기능 개발하기 (충돌 해결)**

#### **1. 사전 준비: 동료의 작업 시뮬레이션**

먼저, 내가 다른 작업을 하는 동안 동료가 `main` 브랜치의 `login.js` 파일을 수정했다고 가정하는 상황을 만들어보겠습니다.

```bash
# 0. 이전 시나리오를 진행했던 git-scenario 폴더에서 시작합니다.
#    현재 브랜치가 main인지 확인합니다.
git switch main

# 1. VS Code로 login.js 파일을 열어 아래와 같이 내용을 변경하고 저장합니다.
#    변경 전: console.log('Login logic is complete');
#    변경 후: console.log('Main branch login logic');
```

```bash
# 2. 동료의 작업 내용을 커밋합니다.
git add login.js
git commit -m "Refactor: Improve login logic on main"

# 3. 커밋 결과를 확인합니다.
git log --oneline -n 1
```

> **💡 새로운 옵션을 알아볼까요?**
> *   `--oneline`: `--pretty=oneline`과 비슷하게 커밋을 한 줄로 보여주는 간결한 옵션입니다.
> *   `-n 1`: `n`은 number를 의미하며, `1`은 가장 최신 커밋 1개만 보겠다는 뜻입니다.

**명령 결과 (터미널 출력 예)**
```
a1b2c3d (HEAD -> main) Refactor: Improve login logic on main
```
- **상황 분석**: 이제 `main` 브랜치의 최신 버전은 동료가 수정한 `a1b2c3d` 커밋입니다.

---

#### **2. 나의 새로운 기능 개발 시작**

이제 당신의 차례입니다. 당신은 동료가 작업하기 **전**의 코드를 기준으로 새로운 '고급 로그인 기능' 개발을 시작합니다. `main` 브랜치의 가장 최신 커밋(`HEAD`) 바로 이전(`~1`) 커밋에서 새로운 브랜치를 만들어보겠습니다.

```bash
# 1. HEAD~1을 사용하여 바로 이전 커밋에서 새 브랜치를 생성하고 이동합니다.
git switch -c feature/advanced-login HEAD~1
```

> **💡 `~` (물결)과 `^` (캐럿) 기호란?**
> `~`와 `^` 기호는 모두 현재 `HEAD`를 기준으로 이전 커밋(부모 커밋)을 가리킬 때 사용합니다.
> *   `HEAD~1`: `HEAD`의 바로 이전(1번째 부모) 커밋을 의미합니다. `HEAD~2`는 2단계 이전 커밋을 가리킵니다.
> *   `HEAD^`: `HEAD`의 첫 번째 부모 커밋을 의미합니다. `~1`과 동일한 대상을 가리키는 경우가 많습니다.
>
> **차이점**: `^`는 merge로 만들어진 커밋처럼 부모가 여러 개일 때, 몇 번째 부모인지를 선택할 때 주로 사용합니다. (예: `HEAD^2`는 두 번째 부모를 의미). 지금은 `~`와 거의 같다고 이해하셔도 좋습니다.

이렇게 하면, 동료가 `main`에 커밋하기 전의 상태에서 내 작업을 시작하는 것과 동일한 상황이 만들어집니다.

```bash
# 2. VS Code로 login.js 파일을 다시 열어, 당신의 기능에 맞게 내용을 수정합니다.
#    동료와 당신은 같은 줄을 다르게 수정한 셈이 됩니다.
#    변경 전: console.log('Login logic is complete');
#    변경 후: console.log('Advanced login feature added');
```

```bash
# 3. 당신의 작업 내용을 커밋합니다.
git add login.js
git commit -m "Feat: Add advanced login feature"

# 4. 현재 브랜치의 로그를 확인합니다.
git log --oneline -n 1
```
**명령 결과 (터미널 출력 예)**
```
e4f5g6h (HEAD -> feature/advanced-login) Feat: Add advanced login feature
```
- **상황 분석**: 이제 `main` 브랜치와 `feature/advanced-login` 브랜치는 같은 부모 커밋에서 갈라져 나와, 각자 다른 내용으로 `login.js` 파일을 변경한 상태입니다.

---

#### **3. 충돌 발생! 그리고 해결하기**

좋은 개발자는 자신의 작업을 시작하기 전이나 통합하기 전에 항상 `main` 브랜치의 최신 변경사항을 가져와 반영합니다. 한 번 시도해봅시다.

```bash
# feature/advanced-login 브랜치에서 main 브랜치를 merge 합니다.
git merge main
```

**명령 결과 (터미널 출력 예) - 💥 충돌 발생!**
```
Auto-merging login.js
CONFLICT (content): Merge conflict in login.js
Automatic merge failed; fix conflicts and then commit the result.
```
- **실패 원인**: Git이 `login.js` 파일을 자동으로 합치려 했지만, 당신과 동료가 **같은 줄을 다르게 수정**했기 때문에 어떤 것을 선택해야 할지 몰라 충돌을 일으키고 병합을 멈췄습니다.

#### **충돌 해결 과정**

1.  **충돌 확인**: VS Code에서 `login.js` 파일을 열면, 아래와 같이 충돌 표시가 나타납니다.

    ```javascript
    <<<<<<< HEAD
    console.log('Advanced login feature added');
    =======
    console.log('Main branch login logic');
    >>>>>>> main
    ```
    - `<<<<<<< HEAD`: 현재 내 브랜치(`feature/advanced-login`)의 변경 내용
    - `=======`: 변경 내용의 구분선
    - `>>>>>>> main`: 병합하려는 브랜치(`main`)의 변경 내용

2.  **코드 수정**: 이제 개발자가 직접 어떤 코드를 남길지 결정해야 합니다. VS Code에서는 `Accept Current Change`(현재 변경 사항 수락), `Accept Incoming Change`(수신 변경 사항 수락) 등의 버튼을 클릭하여 한쪽 변경사항만 선택할 수도 있지만, 여기서는 **두 기능을 모두 살리기 위해** 직접 코드를 편집하겠습니다.

    에디터에서 `<<<<<<< HEAD`, `=======`, `>>>>>>> main` 와 같은 충돌 표시자들을 **모두 삭제**하고, 아래와 같이 두 `console.log`가 모두 남도록 최종 코드를 작성한 뒤 저장해주세요.

    ```javascript
    // 최종 코드 모습
    console.log('Advanced login feature added');
    console.log('Main branch login logic');
    ```

3.  **충돌 해결 완료**: 수정한 파일은 일반적인 커밋 과정과 동일하게, `add`와 `commit`을 통해 병합을 마무리합니다.

    ```bash
    # 1. 충돌을 해결한 파일을 스테이징합니다.
    git add login.js

    # 2. 'commit'을 실행하면, Git이 자동으로 merge commit 메시지를 작성해줍니다.
    #    (편집기 창이 뜨면 내용 수정 없이 그대로 저장 및 종료)
    git commit
    ```

**명령 결과 (터미널 출력 예)**
```
[feature/advanced-login 8f9g0h1] Merge branch 'main' into feature/advanced-login
```
- **학습 포인트**: 충돌은 에러가 아니며, Git이 개발자에게 "상의해서 결정해줘!"라고 보내는 신호임을 이해했습니다. 충돌 지점을 직접 수정하고, `add`와 `commit`으로 병합을 완료하는 방법을 배웠습니다.

---

#### **4. 최종 통합**

이제 당신의 브랜치는 `main`의 최신 내용까지 모두 포함한 완전한 상태가 되었습니다. `main` 브랜치에 안심하고 합칠 수 있습니다.

```bash
# 1. main 브랜치로 이동
git switch main

# 2. --no-ff 옵션을 사용하여, 기능 브랜치의 작업이었다는 히스토리를 명확히 남기며 병합합니다.
git merge --no-ff feature/advanced-login
```
- 이 명령을 실행하면, 병합 커밋 메시지를 작성하기 위한 텍스트 편집기가 열립니다. 기본 메시지를 그대로 사용하고 저장 및 종료합니다.

**명령 결과 (터미널 출력 예)**
```
Merge made by the 'recursive' strategy.
 login.js | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
```
- **학습 포인트**: `Fast-forward`가 가능한 상황이었지만, `feature/advanced-login` 브랜치에서 진행된 일련의 작업들(기능 추가, 충돌 해결)을 하나의 묶음으로 명확히 히스토리에 남기기 위해 `--no-ff` 옵션을 사용했습니다. 이렇게 하면 나중에 `git log --graph`로 히스토리를 볼 때 작업 흐름을 파악하기 훨씬 수월해집니다.

---

#### **5. 원격 저장소에 최종 결과 반영하기**

이제 로컬 `main` 브랜치에 모든 작업(동료의 작업, 내 작업, 충돌 해결 내역)이 통합되었습니다. 마지막으로 이 결과를 원격 저장소에 `push`하여 모든 팀원과 공유합니다.

```bash
# main 브랜치의 최신 상태를 원격 저장소에 보냅니다.
git push
```

이제 GitHub 저장소 페이지를 새로고침하여 `login.js` 파일의 내용과 커밋 히스토리가 어떻게 변경되었는지 직접 확인해보세요. 당신과 동료의 작업이 모두 합쳐진 것을 볼 수 있습니다.

- **학습 포인트**: 로컬에서 완료된 모든 작업을 `push`하여 원격 저장소를 최신 상태로 동기화함으로써, 팀 전체가 동일한 코드 기반 위에서 작업을 이어갈 수 있게 됩니다.

---

축하합니다! 여러분은 이제 팀원과의 작업이 겹쳐 충돌이 발생했을 때, 당황하지 않고 침착하게 문제를 해결하고 프로젝트에 기여하는 전체 협업 과정을 마스터했습니다.
