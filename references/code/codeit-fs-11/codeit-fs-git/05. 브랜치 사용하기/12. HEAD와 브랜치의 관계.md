### 12. HEAD와 브랜치의 관계

이전에는 `HEAD`가 '현재 작업중인 커밋'을, `브랜치`는 '코드 관리 흐름'이라고 배웠습니다. 이번 시간에는 이 둘의 진짜 정체와 관계를 더 깊이 알아보겠습니다. `git log`에서 보이는 `(HEAD -> main)`의 의미를 완벽히 이해하는 것이 목표입니다.

---

### 1. 브랜치와 HEAD의 진짜 정체: 포인터(Pointer)

- **브랜치(Branch)란?**: 단순히 '작업 흐름'이라는 추상적인 개념이 아니라, **특정 커밋을 가리키는 포인터**입니다. 브랜치는 항상 자신의 라인에서 가장 마지막(최신) 커밋을 가리킵니다. 새로운 커밋이 추가되면, 해당 브랜치 포인터는 새로 추가된 커밋을 가리키도록 자동으로 이동합니다.

- **HEAD란?**: `HEAD` 역시 특정 대상을 가리키는 **포인터**입니다.

---

### 2. 둘의 관계: `HEAD` → `브랜치` → `커밋`

- **핵심**: `HEAD`는 대부분의 경우 커밋을 직접 가리키지 않습니다. 대신, **`HEAD`는 현재 작업 중인 '브랜치'를 가리킵니다.**
- **`log` 메시지 해석**: `(HEAD -> main)`이 보이는 이유는 바로 이 관계 때문입니다.
    - "현재 작업 위치인 `HEAD`는 `main` 브랜치를 가리키고 있고, `main` 브랜치는 `f1a2b3c` 커밋을 가리키고 있다"는 의미입니다.
    - 즉, `HEAD`는 `main` 브랜치를 통해 간접적으로 최신 커밋을 가리키게 됩니다.

---

### 3. 이 모델로 이해하는 Git 동작 원리

#### 커밋(Commit)
- 새로운 커밋을 생성하면, `HEAD`가 직접 움직이는 것이 아니라 **`HEAD`가 가리키고 있던 브랜치(예: `main`) 포인터가 새로운 커밋으로 이동**합니다.
- `HEAD`는 여전히 `main` 브랜치를 가리키고 있으므로, 결과적으로 `HEAD`도 새로운 커밋을 간접적으로 가리키게 됩니다.

#### 브랜치 전환(Switch)
- `git switch premium` 명령어의 본질은 **`HEAD` 포인터가 가리키는 대상을 `main` 브랜치에서 `premium` 브랜치로 바꾸는 것**입니다.
- `HEAD`가 `premium`을 가리키게 되면, 워킹 디렉토리의 내용도 `premium` 브랜치가 가리키는 커밋의 상태로 변경됩니다.

#### 분기(Diverge)
1.  `HEAD`가 `premium`을 가리키는 상태에서 새로운 커밋을 생성합니다.
2.  `premium` 브랜치 포인터만 새로 생긴 커밋으로 이동합니다. `main` 브랜치 포인터는 원래 위치에 그대로 있습니다.
3.  이것이 바로 브랜치가 나뭇가지처럼 갈라지는, 즉 **분기(Diverge)**하는 원리입니다.

---

### 용어설명

- **포인터(Pointer)**: 특정 커밋의 위치(주소)를 기억하고 가리키는 존재. Git에서는 `HEAD`, `브랜치`, `태그` 등이 모두 포인터의 일종입니다.
