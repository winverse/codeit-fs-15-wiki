### 05. 브랜치 merge하기

`merge`는 '병합하다'라는 뜻으로, Git에서는 한 브랜치의 변경 내용을 다른 브랜치로 합치는 작업을 의미합니다. 이를 통해 각기 다른 브랜치에서 작업한 내용을 하나의 브랜치로 모을 수 있습니다.

---

### Merge 기본 사용법

- **규칙**: `A` 브랜치의 내용을 `B` 브랜치로 합치고 싶다면, 먼저 `B` 브랜치로 이동한 후 `merge` 명령어를 실행해야 합니다.

```bash
# 1. 내용을 받을 브랜치(B)로 이동
git switch B

# 2. 내용을 가져올 브랜치(A)를 지정하여 merge 실행
git merge A
```

---

### Merge의 두 가지 방식

`merge`는 브랜치들의 상태에 따라 두 가지 방식으로 동작합니다.

#### 1. 3-Way Merge (Merge Commit 생성)

- **상황**: 두 브랜치가 서로 다른 커밋을 가지며 분기되었을 때 (Diverged History) 발생합니다.
- **동작**: Git은 각 브랜치의 변경 내용을 합친 후, 병합 내역을 기록하기 위한 새로운 커밋, 즉 **Merge Commit**을 생성합니다. 이때 커밋 메시지를 작성하는 편집기 창이 열립니다.
- **결과**: `git graph`에서 두 브랜치가 하나의 Merge Commit으로 합쳐지는, 가지가 합쳐지는 모양의 히스토리가 남습니다.

#### 2. Fast-Forward Merge

- **상황**: 내용을 받을 브랜치(`main`)에 새로운 커밋이 없고, 내용을 가져올 브랜치(`premium`)가 단지 앞으로만 진행되었을 때 발생합니다.
- **동작**: Git은 별도의 Merge Commit을 만들지 않고, 단순히 `main` 브랜치 포인터를 `premium` 브랜치의 최신 커밋 위치로 '빨리 감기(Fast-forward)'처럼 이동시킵니다.
- **문제점**: 히스토리가 한 줄로 합쳐져서, `premium` 브랜치의 작업 흐름이 별도로 존재했다는 사실을 파악하기 어렵습니다.

---

### Merge Commit 강제로 만들기 (`--no-ff`)

`Fast-forward` 방식의 문제점을 해결하고, 브랜치의 작업 내역을 명확히 남기기 위해 `merge` 시 `--no-ff` (No Fast-Forward) 옵션을 사용하는 것이 좋습니다.

- **명령어**: `git merge --no-ff [가져올 브랜치 이름]`
- **동작**: `Fast-forward`가 가능한 상황에서도 이를 방지하고, 항상 **Merge Commit**을 생성하도록 강제합니다.
- **장점**: `git graph` 상에서 기능 브랜치가 언제, 어떻게 병합되었는지 명확하게 히스토리에 남길 수 있어, 협업 시 코드의 흐름을 추적하는 데 매우 유용합니다.

```bash
# main 브랜치로 이동
git switch main

# --no-ff 옵션을 사용하여 premium 브랜치를 병합
git merge --no-ff premium
```
