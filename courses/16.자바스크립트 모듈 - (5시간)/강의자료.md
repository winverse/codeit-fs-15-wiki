### 03. import와 export ⭐⭐⭐

### 모듈의 핵심 기능

모듈의 핵심은 **코드를 파일 단위로 나눠놓고, 필요한 것을 불러와서 사용하는 것**입니다.

```
product.js     →  export (내보내기)  →  main.js
(상품 기능)                              (메인 프로그램)
                ←  import (가져오기)  ←

```

### .mjs 파일 확장자

ES 모듈을 사용하는 방법은 환경에 따라 다릅니다.

### 브라우저에서

브라우저에서는 **`<script type="module">`**을 사용하면 `.js` 파일에서도 import/export를 쓸 수 있습니다:

```html
<script type="module" src="./main.js"></script>
```

### Node.js에서

Node.js에서는 ES 모듈을 사용하기 위해 **두 가지 방법**이 있습니다:

1. `package.json`에 `"type": "module"`을 설정하고 `.js`를 그대로 쓰기
2. `type` 설정 없이 **`.mjs` 확장자**를 사용하기

```
일반 파일:    calculator.js   (CommonJS - require/module.exports)
모듈 파일:    calculator.mjs  (ES 모듈 - import/export)
```

이 강의에서는 설명을 단순하게 하기 위해 **`.mjs` 확장자**를 사용합니다.

이렇게 하면 "이 파일은 ES 모듈이다"라는 것을 파일 이름만 보고 알 수 있습니다.

실무에서는 `package.json`에 `"type": "module"`을 설정하고 `.js`를 사용하는 경우가 더 많습니다.

### export - 내보내기

다른 파일에서 사용할 수 있도록 **변수나 함수를 내보내는** 키워드입니다.

### 기본 사용법

```jsx
// math.mjs
export const PI = 3.14;

export function add(a, b) {
  return a + b;
}

export function subtract(a, b) {
  return a - b;
}
```

`export` 키워드를 변수나 함수 선언 앞에 붙이면 됩니다.

### export 없이는 사용할 수 없음

```jsx
// math.mjs
const E = 2.718; // export 없음

export const PI = 3.14; // export 있음

function multiply(a, b) {
  // export 없음
  return a * b;
}

export function divide(a, b) {
  // export 있음
  return a / b;
}
```

**다른 파일에서 사용 가능:**

- `PI`
- `divide`

**다른 파일에서 사용 불가능:**

- `E`
- `multiply`

### import - 가져오기

다른 파일에서 **export된 것을 가져오는** 키워드입니다.

### 기본 사용법

```jsx
// main.mjs
import { PI, add } from "./math.mjs";

console.log(PI); // 3.14
console.log(add(5, 3)); // 8
```

**구조:**

1. `import` 키워드
2. `{ }` 안에 가져올 변수/함수 이름 (쉼표로 구분)
3. `from` 키워드
4. 파일 경로 (문자열)

### import 문은 파일 맨 위에

```jsx
// main.mjs
import { PI, add } from "./math.mjs";
import { subtract, divide } from "./math.mjs";

// 이제 아래부터 사용 가능
console.log(add(10, 5));
console.log(subtract(10, 5));
```

import 문은 보통 파일의 **가장 위쪽**에 작성합니다.

### 파일 경로 작성하기

### 현재 디렉토리 - `./`

같은 디렉토리(폴더)에 있는 파일을 가져올 때:

```
project/
  ├── main.mjs
  └── math.mjs
```

```jsx
// main.mjs
import { PI, add } from "./math.mjs";
```

`./`는 **현재 디렉토리**를 의미합니다.

### 하위 디렉토리 - `./폴더명/`

하위 디렉토리 안의 파일을 가져올 때:

```
project/
  ├── main.mjs
  └── utils/
      └── math.mjs
```

```jsx
// main.mjs
import { PI, add } from "./utils/math.mjs";
```

`./utils/`는 현재 디렉토리의 `utils` 폴더를 의미합니다.

### 더 깊은 하위 디렉토리

```
project/
  ├── main.mjs
  └── lib/
      └── utils/
          └── math.mjs
```

```jsx
// main.mjs
import { PI, add } from "./lib/utils/math.mjs";
```

슬래시(`/`)로 디렉토리를 계속 연결할 수 있습니다.

### 상위 디렉토리 - `../`

상위 디렉토리의 파일을 가져올 때:

```
project/
  ├── math.mjs
  └── src/
      └── main.mjs
```

```jsx
// src/main.mjs
import { PI, add } from "../math.mjs";
```

`../`는 **상위 디렉토리**를 의미합니다.

### 경로 정리

```jsx
// 현재 디렉토리
import { PI } from "./math.mjs";

// 하위 디렉토리
import { PI } from "./utils/math.mjs";

// 상위 디렉토리
import { PI } from "../math.mjs";

// 상위의 상위 디렉토리
import { PI } from "../../math.mjs";

// 상위 디렉토리의 다른 폴더
import { PI } from "../lib/math.mjs";
```

**기억하세요:**

- `./` = 현재 디렉토리
- `../` = 상위 디렉토리
- `/` = 디렉토리 안으로 들어가기

용어 정리: 디렉토리(directory)는 폴더(folder)를 지칭하는 다른 단어입니다. Node.js와 웹 개발에서는 디렉토리라는 용어를 더 많이 사용합니다.

### 실전 예제

### 예제 1: 간단한 계산기 모듈

```jsx
// calculator.mjs
export function add(a, b) {
  return a + b;
}

export function subtract(a, b) {
  return a - b;
}

export function multiply(a, b) {
  return a * b;
}

export function divide(a, b) {
  if (b === 0) {
    return "0으로 나눌 수 없습니다";
  }
  return a / b;
}
```

```jsx
// main.mjs
import {
  add,
  subtract,
  multiply,
  divide,
} from "./calculator.mjs";

console.log("10 + 5 =", add(10, 5)); // 15
console.log("10 - 5 =", subtract(10, 5)); // 5
console.log("10 * 5 =", multiply(10, 5)); // 50
console.log("10 / 5 =", divide(10, 5)); // 2
console.log("10 / 0 =", divide(10, 0)); // 0으로 나눌 수 없습니다
```

### 예제 2: 사용자 관리 모듈

```jsx
// user.mjs
export function createUser(name, email) {
  return {
    name,

  };

  return email.includes("@");

  return `이름: ${user.name}, 이메일: ${user.email}`;

```

```jsx
// main.mjs
import {

const email = "user@example.com";

  console.log("유효하지 않은 이메일입니다");
}
```

### 정리

**export (내보내기):**

- 다른 파일에서 사용할 수 있도록 변수나 함수를 내보냄
- 선언문 앞에 `export` 키워드 붙이기

```jsx
export const PI = 3.14;
export function add(a, b) {
  return a + b;
}
```

**import (가져오기):**

- 다른 파일에서 export된 것을 가져옴
- 파일 맨 위에 작성

```jsx
import { PI, add } from "./math.mjs";
```

**파일 경로:**

- `./` : 현재 디렉토리
- `../` : 상위 디렉토리
- `/` : 디렉토리 안으로 들어가기

**필요한 것만 가져오기:**

```jsx
// 전부 export되어 있어도
export const A = 1;
export const B = 2;
export const C = 3;

// 필요한 것만 import
import { A, C } from "./module.mjs";
```

**모듈 파일 확장자:**

- Node.js에서 `.mjs` 확장자를 사용하면 ES 모듈로 동작
- 또는 `package.json`에 `"type": "module"` 설정 후 `.js` 사용
- 브라우저에서는 `<script type="module">`이면 `.js`도 가능

다음 코드에서 빈칸을 채워서 `calculator.mjs` 모듈의 `add` 함수를 `main.mjs`에서 사용할 수 있도록 하세요.

**calculator.mjs**

```jsx
_______ const PI = 3.14;

_______ function add(a, b) {
  return a + b;
}
```

**main.mjs**

```jsx
_______ { PI, add } _______ './calculator.mjs';

console.log(PI);
console.log(add(5, 3));
```

- 정답 및 해설
    
    정답:
    
    **calculator.mjs**
    
    ```jsx
    export const PI = 3.14;
    
    export function add(a, b) {
      return a + b;
    }
    ```
    
    **main.mjs**
    
    ```jsx
    import { PI, add } from "./calculator.mjs";
    
    console.log(PI);
    console.log(add(5, 3));
    ```

    해설:
    
    - `export` 키워드를 붙여야 다른 파일에서 가져올 수 있습니다.
    - `import { 이름 } from '경로'` 문법을 사용하면 export된 값들을 가져올 수 있습니다.
    - 중괄호 안의 이름은 export된 이름과 동일해야 합니다.

### 문제 3

다음 `product.mjs` 모듈이 있습니다. `main.mjs`에서 이 모듈을 사용하려고 할 때, **에러가 발생하는** 코드를 모두 고르세요. (복수 정답 가능)

**product.mjs**

```jsx
export const TAX_RATE = 0.1;

function calculateTax(price) {
  return price * TAX_RATE;
}

export function getPrice(price) {
  return price + calculateTax(price);
}
```

A.

```jsx
import { TAX_RATE, getPrice } from "./product.mjs";

console.log(TAX_RATE);
console.log(getPrice(10000));
```

B.

```jsx
import {
  TAX_RATE,
  getPrice,
  calculateTax,
} from "./product.mjs";

console.log(calculateTax(10000));
```

C.

```jsx
import { getPrice } from "./product.mjs";

console.log(getPrice(10000));
```

D.

```jsx
import { TAX_RATE } from "./product.mjs";

console.log(TAX_RATE);
console.log(getPrice(10000));
```

- 정답 및 해설
    
    정답:
    
    B, D
    
    해설:
    
    **export된 항목**
    
    - `TAX_RATE` (O)
    - `getPrice` (O)
    
    **export되지 않은 항목**
    
    - `calculateTax` (X)
    
    각 선택지 분석:
    
    - **A (정상)**
        
        export된 두 개만 import한 정상 코드.
        
    - **B (에러)**
        
        `calculateTax`는 export되지 않았기 때문에 import하면 에러 발생.
        
        ❌ `The requested module does not provide an export named 'calculateTax'`
        
    - **C (정상)**
        
        필요한 함수인 `getPrice`만 import → 정상 실행 가능.
        
    - **D (에러)**
        
        `getPrice`를 import하지 않고 사용하려 해서 ReferenceError 발생.
        
        ❌ `getPrice is not defined`
        
    
    핵심 요약:
    
    - **export되지 않은 것은 import할 수 없다.**
    - **import하지 않은 것은 사용할 수 없다.**

### 05. default export ⭐⭐

### named export 복습

지난 시간에 배운 방식은 **named export**입니다.

```jsx
// calculator.mjs
export const PI = 3.14;

export function add(a, b) {
  return a + b;
}
```

```jsx
// main.mjs
import { PI, add } from "./calculator.mjs";

console.log(PI); // 3.14
console.log(add(1, 2)); // 3
```

- export할 때 사용한 **이름 그대로** import해야 함
- 중괄호 `{ }` 필수

### default export란?

**default export**는 모듈에서 **기본적으로 내보낼 것**을 지정하는 방식입니다.

```jsx
// calculator.mjs
export default function add(a, b) {
  return a + b;
}
```

`export` 뒤에 `default` 키워드를 추가합니다.

### default export의 장점

### 1. 중괄호 없이 import 가능

```jsx
// main.mjs
import add from "./calculator.mjs";

console.log(add(1, 2)); // 3
```

- **중괄호 `{ }` 없이** import
- 더 간단하고 편리함

### 2. 원하는 이름으로 import 가능

- 함수를 만든 사람의 의도를 무시할 수 있음 (주의 필요)

```jsx
// main.mjs
import myAdd from "./calculator.mjs";

console.log(myAdd(5, 3)); // 8
```

```jsx
// main.mjs
import plus from "./calculator.mjs";

console.log(plus(10, 20)); // 30
```

export할 때 사용한 이름(`add`)이 아닌, **아무 이름**으로 import 가능합니다!

### named export와 default export 함께 사용하기

한 파일에서 두 방식을 동시에 사용할 수 있습니다.

```jsx
// calculator.mjs
export const PI = 3.14; // named export

export default function add(a, b) {
  // default export
  return a + b;
}
```

```jsx
// main.mjs
import add, { PI } from "./calculator.mjs";

console.log(add(1, 2)); // 3
console.log(PI); // 3.14
```

**구조:**

- default import: 중괄호 없이, 쉼표 앞에 작성
- named import: 중괄호 안에 작성

### default export 문법 규칙

### 규칙 1: 함수는 바로 export 가능

```jsx
// 가능 ✅
export default function add(a, b) {
  return a + b;
}
```

### 규칙 2: const, let은 바로 export 불가능

```jsx
// 불가능 ❌
export default const PI = 3.14;
```

**올바른 방법:**

```jsx
// 방법 1: 먼저 선언하고 나중에 export (권장)
const PI = 3.14;
export default PI;

// 방법 2: 값 자체를 export (비권장)
export default 3.14;
```

### 규칙 3: 파일당 한 번만 사용 가능

```jsx
// 오류 발생! ❌
const PI = 3.14;
export default PI;

export default function add(a, b) {  // 에러!
  return a + b;
}
```

**에러 메시지:** `A module cannot have multiple default exports`

**올바른 방법:**

```jsx
// 하나만 default export
const PI = 3.14;
export default PI;

export function add(a, b) {
  // named export로 변경
  return a + b;
}
```


```jsx
// main.mjs
import random from "./random.mjs";
```

### 정리

**default export:**

- `export default`로 내보내기
- 중괄호 없이 import
- 원하는 이름으로 import 가능
- 파일당 한 번만 사용

```jsx
// 내보내기
export default function add(a, b) {
  return a + b;
}

// 가져오기
import add from "./calculator.mjs";
import myAdd from "./calculator.mjs"; // 다른 이름 가능
```

**named export:**

- `export`로 내보내기
- 중괄호 `{ }`로 import
- export할 때 사용한 이름 그대로 import
- 여러 번 사용 가능

```jsx
// 내보내기
export const PI = 3.14;
export function add(a, b) {
  return a + b;
}

// 가져오기
import { PI, add } from "./calculator.mjs";
```

**함께 사용하기:**

```jsx
// 내보내기
export const PI = 3.14; // named
export default function add(a, b) {
  // default
  return a + b;
}

// 가져오기
import add, { PI } from "./calculator.mjs";
```

**핵심 규칙:**

1. 함수는 `export default function` 가능
2. `const`, `let`은 먼저 선언 후 `export default`
3. 파일당 default export는 **한 번만**

### 객체로 묶어서 default export 하기

여러 기능이 있을 때 **객체로 묶어서** default export 할 수도 있습니다.

```jsx
// math.mjs
const PI = 3.14;
const E = 2.718;

function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

function divide(a, b) {
  return a / b;
}

// 객체로 묶어서 export
const math = {
  PI,
  E,
  add,
  subtract,
  multiply,
  divide,
};

```

```jsx
// main.mjs
import math from "./math.mjs";

console.log("원주율:", math.PI);
console.log("덧셈:", math.add(10, 5));

```

**장점:**

- 하나의 객체로 모든 기능 접근
- 네임스페이스 역할 (이름 충돌 방지)
- 간단한 import

**단점:**

- 점 표기법으로 접근해야 함
- 필요한 기능만 선택적으로 import 불가

### 두 방식을 함께 사용하기

**주요 기능**은 default export, **보조 기능**은 named export로 제공할 수 있습니다.

### 예제 1: 객체 + 개별 함수

```jsx
// math.mjs
export const PI = 3.14;

export function add(a, b) {
  return a + b;
}

export function subtract(a, b) {
  return a - b;
}

export function multiply(a, b) {
  return a * b;
}

export function divide(a, b) {
  return a / b;
}

// 모든 기능을 담은 객체도 제공
const math = {
  PI,
  add,
  subtract,
  multiply,
  divide,
};

export default math;
```

**사용 방법 1: 객체로 사용**

```jsx
// main.mjs
import math from "./math.mjs";

console.log(math.PI);
console.log(math.add(10, 5));
console.log(math.multiply(10, 5));
```

**사용 방법 2: 개별 함수만 import**

```jsx
// main.mjs
import { PI, add, multiply } from "./math.mjs";

console.log(PI);
console.log(add(10, 5));
console.log(multiply(10, 5));
```

```jsx
// main.mjs
import math, { PI, add } from "./math.mjs";

```

**장점:**

- 사용하는 쪽에서 **원하는 방식 선택** 가능
- 유연성이 높음
- 다양한 사용 사례 지원

### 실전 가이드

### named export를 선택하세요

✅ 여러 개의 독립적인 함수/변수가 있을 때

✅ 필요한 것만 선택적으로 import하고 싶을 때

✅ 유틸리티 함수 모음일 때

✅ 상수 모음일 때

```jsx
// 이런 경우는 named export
export const CONFIG_A = 1;
export const CONFIG_B = 2;
export function helperA() {}
export function helperB() {}
```

### default export를 선택하세요

✅ 파일의 주요 목적이 하나일 때

✅ 클래스를 제공할 때

✅ 하나의 큰 설정 객체를 제공할 때

✅ 하나의 핵심 함수만 있을 때

```jsx
// 이런 경우는 default export
export default class User {}
export default function authenticate() {}
export default { theme: "dark" };
```

### 두 방식을 함께 사용하세요

✅ 주요 기능 + 보조 기능이 있을 때

✅ 다양한 사용 방식을 지원하고 싶을 때

✅ 클래스 + 관련 상수가 있을 때

```jsx
// 이런 경우는 함께 사용
export const CONSTANT_A = 1;
export default class MainFeature {}
```

### 비교 정리

| 기준 | named export | default export |
| --- | --- | --- |
| **개수** | 여러 개 가능 | 파일당 1개 |
| **import 방법** | `import { add } from ...` | `import add from ...` |
| **이름** | 정확히 일치해야 함 | 원하는 이름 사용 가능 |
| **사용 사례** | 여러 함수/상수 모음 | 주요 기능 하나 |
| **선택성** | 필요한 것만 선택 가능 | 하나의 값을 가져옴 |

### 정리

**named export:**

- 여러 개의 기능을 개별적으로 제공
- 필요한 것만 선택해서 import
- 이름이 명확하고 의도가 분명함

```jsx
export function add() {}
export function subtract() {}

import { add, subtract } from "./math.mjs";
```

**default export:**

- 파일의 주요 기능 하나를 제공
- 간단하고 직관적인 import
- 원하는 이름으로 사용 가능 **(강사의 의견: 이것이 문제!)**

```jsx
export default class User {}

import User from "./User.mjs";
```

**함께 사용:**

- 주요 기능과 보조 기능을 모두 제공
- 유연하고 다양한 사용 방식 지원

```jsx
export const ROLE_ADMIN = "admin";
export default class User {}

import User, { ROLE_ADMIN } from "./User.mjs";
```

**핵심 원칙:**

- 파일의 목적과 사용 방식을 고려해서 선택
- 사용하는 쪽의 편의성을 생각
- 일관성 있게 사용

### 진짜 알아야하는 것!

1. **CommonJS(CJS)는 “현실적인 편법”이었고, 표준이 아니었다.**
2. **ES Module(ESM)은 브라우저+Node 모두를 위한 “표준 모듈 시스템”이다.**

---

### 1. CommonJS 모듈이 왜 생겼나

원래 JavaScript는 **브라우저 안에서만 돌아가는 스크립트 언어**였고, 모듈 시스템이 없었어.

- 전역 변수 범람
- 파일 간 의존성 관리 지옥
- 코드 재사용 어려움

Node.js가 등장하면서 “서버에서도 JS 쓰자”가 되니까,

**파일 단위로 코드를 나누고 가져오는 문법이 필요해졌고**, 그때 커뮤니티에서 만든 **사실상의 표준**이 **CommonJS 모듈 시스템**이야.

```jsx
// math.js (CommonJS)
const PI = 3.14;

function add(a, b) {
  return a + b;
}

module.exports = {
  PI,
  add,
};
```

```jsx
// main.js
const { PI, add } = require("./math");

console.log(add(PI, 2));
```

- `require(...)`
- `module.exports` / `exports`

이게 **표준은 아니지만**, Node.js 진영에서 너무 많이 쓰이다 보니 “JS 모듈 = CJS”처럼 되어버렸던 거지.

---

### 2. CommonJS 모듈의 문제점

실무에서 쓰기엔 괜찮았지만, **언어 차원 / 브라우저 / 번들러 / 최적화 관점**에서 한계가 많았어.

### (1) 동기(synchronous) 로딩 모델

`require()`는 **실행 시점에, 동기적으로** 모듈을 읽어와.

```jsx
const a = require("./a"); // 여기서 멈춰서 파일 읽고, 실행하고, 결과 반환
```

- Node.js에서는 로컬 파일 읽는 거라 그럭저럭 괜찮았지만
- **브라우저 환경에서는 “네트워크 요청”이 동기로 일어나는 것과 같음 → 말도 안 됨!!**

그래서 CJS는 **브라우저에서 직접 쓰기 어렵고** 반드시 **번들러(webpack, Browserify 등)** 가 필요했어.

### (2) “정적 분석”이 어렵다 (트리 셰이킹, 최적화 문제)

CommonJS는 **동적(require가 코드 흐름에 따라 바뀔 수 있음)**이라서, 번들러 입장에서 분석이 힘들다.

```jsx
// 이런 것도 허용됨
const moduleName = condition ? "./a" : "./b";
const mod = require(moduleName);
```

- 어떤 모듈을 가져오는지 **빌드 타임에 100% 확신하기 어려움**
- 사용하지 않는 코드만 쏙쏙 제거하는 **tree-shaking**이 까다로움
- 정적 분석 기반 최적화(코드 분할, dead code 제거 등)가 제한적

### (3) 언어 표준이 아니라 “Node 전용”이었다

- CJS는 **ECMAScript 표준이 아니라**, Node + 일부 도구 생태계의 관습
- 브라우저 JS에는 `require`, `module.exports`가 애초에 존재하지 않음
- 그래서 브라우저에서 쓰려면 항상 **번들링 + 변환 과정**이 필요했어

결국,

“JavaScript 모듈을 **언어 차원에서 표준화**하고, **브라우저/Node 어디서나 같은 방식으로** 쓰고 싶다”는 요구가 커졌고, 그 결과가 **ES Modules**야.

### (4) CJS ↔ 다른 환경과의 호환성 문제

- 브라우저는 `<script>`만 알고, `require` 모른다.
- Node는 CJS를 기본으로 쓰고 있었고, 나중에 ESM을 지원하기 시작함.
- 패키지들이 `module.exports`로 내보내기도 하고, `exports.default` 같은 꼼수도 쓰고,
    
    ESM에서 `import`할 때 `default`가 꼬이는 등 **interop 문제가 계속 발생**했음.
    

---

### 3. ES Module(ESM)의 등장 배경

이런 배경에서 ECMAScript(자바스크립트 표준 위원회)에서:

- **브라우저와 서버 환경 모두에서 작동하는**
- **언어 차원의 공식 모듈 시스템**을 만들자

해서 나온 것이 **ES6(ES2015)의 ES Modules**야.

브라우저에서는 이렇게 바로 사용 가능:

```html
<script type="module" src="main.js"></script>
```

그리고 `main.js` 같은 파일 안에서:

```jsx
// math.js
export const PI = 3.14;
export function add(a, b) {
  return a + b;
}

// main.js
import { PI, add } from "./math.js";
```

Node.js도 이제는:

- `package.json`에 `"type": "module"`
    
    또는
    
- `.mjs` 확장자

같은 방식으로 **ESM을 공식 지원**하고 있지.

---

### 4. ES Module의 장점 (CommonJS와 비교)

### (1) 정적 구조 → 정적 분석 쉬움

ESM의 핵심 특징은 **import / export 구조가 “정적(static)”이라는 것**이야.

- `import` 문은 **파일 최상단에**, **문자 그대로** 적혀야 함
- 조건문 안에서 `import` 못 함
- 동적으로 모듈 경로를 바꾸는 것 불가능 (동적 import는 `import()` 별도 문법)

```jsx
// ESM
import { add } from "./math.js"; // 반드시 코드 최상단, 고정 문자열

// CJS
const name = "./" + getModuleName();
const mod = require(name); // 실행 중에야 어떤 모듈인지 알 수 있음
```

이 정적 구조 덕분에:

- 번들러가 **빌드 타임에 의존성 그래프를 완전히 파악**할 수 있고
- 사용 안 하는 부분은 과감하게 제거(tree-shaking)
- 코드 스플리팅, 최적화, 분석이 훨씬 쉬워짐

### (2) 브라우저에서 직접 사용할 수 있음

CJS는 브라우저에서 `require`가 없어서 **무조건 번들러 필요**였는데,

ESM은 브라우저 표준이라서:

```html
<script type="module" src="main.js"></script>
```

만 해도 **브라우저가 직접 import를 해석**하고, HTTP/2 같은 프로토콜과 잘 어울리는 방식으로 가져올 수 있어.

(실무에서는 그래도 번들러를 쓰지만, “언어/플랫폼 레벨에서 지원된다”는 게 크다.)

### (3) 비동기 로딩 모델

브라우저 환경에서 ESM은 **네트워크를 고려한 비동기 로딩**을 기본 전제로 설계되어 있어.

- CJS: `require`는 **동기** (파일 다 읽을 때까지 멈춤)
- ESM: 모듈 로딩 자체가 **비동기**, 브라우저가 의존성 그래프를 보고 병렬 요청 등 최적화 가능

### (4) 표준화된 import/export 문법

- `default export` / `named export` 를 언어 차원에서 지원
- Node, 브라우저, 번들러 모두 같은 문법을 기준으로 돌아감

```jsx
// math.js
export default function add(a, b) {
  return a + b;
}

export const PI = 3.14;

// main.js
import add, { PI } from "./math.js";
```

이제 “Node에서는 require, 브라우저에서는 ES Module” 같은 이중 구조에서

점점 **표준으로 수렴** 중이라고 보면 돼.

---

### 5. 요약: CommonJS의 문제 vs ES Module의 목표

**CommonJS의 문제점**

1. **동기 로딩 모델** → 브라우저와 맞지 않음
2. **동적 require** → 정적 분석, tree-shaking, 최적화가 어려움
3. **언어 표준이 아님** → 브라우저/도구/런타임 간 일관성 부족
4. **interop 지옥** → CJS ↔ ESM 섞여 있을 때 타입/디폴트/이름 헷갈림

**ES Module의 목표와 장점**

1. **ECMAScript 표준 모듈 시스템**
2. 브라우저와 Node를 아우르는 **공통 문법**
3. **정적 구조** 덕분에 빌드/최적화/분석 쉬움
4. **비동기 로딩**을 전제로 설계 → 네트워크 환경에 적합
5. default/named export, import 문법으로 코드 의도 명확

### 문제 1

다음 CommonJS 모듈 코드를 ES 모듈 형식으로 변환하세요.

```jsx
// utils.js (CommonJS)
function formatDate(date) {
  return date.toISOString().split("T")[0];
}

function formatTime(date) {
  return date.toTimeString().split(" ")[0];
}

module.exports = {
  formatDate,
  formatTime,
};
```

```jsx
// main.js (CommonJS)
const utils = require("./utils.js");

const now = new Date();
console.log(utils.formatDate(now));
console.log(utils.formatTime(now));
```

- 정답 및 해설
    
    정답:
    
    ```jsx
    // utils.mjs (ES Module)
    export function formatDate(date) {
      return date.toISOString().split("T")[0];
    }
    
    export function formatTime(date) {
      return date.toTimeString().split(" ")[0];
    }
    ```
    
    ```jsx
    // main.mjs (ES Module)
    import { formatDate, formatTime } from "./utils.mjs";
    
    const now = new Date();
    console.log(formatDate(now));
    console.log(formatTime(now));
    ```
    
    해설:
    
    CommonJS의 `module.exports` → ES 모듈의 `export`로 변경하고, `require` → `import` 문법으로 변환한다. ES 모듈에서는 함수 자체를 직접 import하여 사용할 수 있다.

### 문제 3

다음 코드를 보고 각 상황에서 무엇이 출력될지 예측하세요.

**코드 1: CommonJS**

```jsx
// calculator.js

function add(a, b) {
  return a + b;
}

```

```jsx
// main.js
const calculator = require("./calculator.js");

console.log(calculator.PI);
```

```jsx
// calculator.mjs
export const PI = 3.14;

export default function add(a, b) {

```

```jsx
// main.mjs
import calculator, { PI } from "./calculator.mjs";

console.log(calculator(10, 5));

```

    ```
    15
    undefined
    ```

    ```
    3.14
    ```
