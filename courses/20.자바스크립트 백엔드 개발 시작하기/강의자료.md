# 20. 자바스크립트 백엔드 개발 시작하기

# 1. 웹 개발 기초 개념

## 1-1. REST API란? ⭐⭐⭐

### REST(Representational State Transfer)

**REST**는 웹 기반 시스템에서 리소스를 HTTP 메서드로 조작하는 아키텍처 스타일입니다.

쉽게 말하면, 웹에서 데이터를 주고받을 때 **어떻게 구조화하고 통신할 것인가**에 대한 설계 방법입니다.

### 핵심 구성 요소

### 1. 리소스 (Resource)

리소스는 URI(Uniform Resource Identifier)로 식별됩니다:

```
/users              → 사용자 목록
/users/123          → 특정 사용자 (ID: 123)
/posts              → 게시글 목록
/posts/456          → 특정 게시글 (ID: 456)
```

리소스는 **명사**로 표현하며, URI를 통해 명확하게 구분할 수 있어야 합니다.

### 2. HTTP 메서드 (HTTP Methods)

리소스에 대한 작업은 HTTP 메서드로 표현합니다:

| HTTP 메서드 | 역할 | 예시 |
| --- | --- | --- |
| `GET` | 조회 | `GET /users` → 사용자 목록 가져오기 |
| `POST` | 생성 | `POST /users` → 새 사용자 추가 |
| `PUT` | 전체 수정 | `PUT /users/123` → 사용자 123 전체 수정 |
| `PATCH` | 부분 수정 | `PATCH /users/123` → 사용자 123 일부 수정 |
| `DELETE` | 삭제 | `DELETE /users/123` → 사용자 123 삭제 |

### REST의 주요 특징

### 1. 무상태 (Stateless)

각 요청은 **독립적**입니다. 서버는 이전 요청의 정보를 기억하지 않습니다.

```jsx
// 모든 요청에 필요한 정보를 포함해야 함
GET / users / 123;
Headers: {
  Authorization: "Bearer token123";
}
```

### 2. 계층화 (Layered System)

클라이언트는 중간에 어떤 서버가 있는지 의식하지 않습니다:

```
클라이언트 → 로드밸런서 → API 서버 → 데이터베이스
```

클라이언트 입장에서는 그냥 API 서버와 통신하는 것처럼 보입니다.

### 3. 캐시 가능 (Cacheable)

응답에 캐시 제어 헤더를 사용하여 성능을 개선할 수 있습니다:

```jsx
// 서버 응답
Headers: {
  "Cache-Control": "max-age=3600"  // 1시간 동안 캐시 유효
}
```

### REST API vs RESTful API

### REST API

REST 아키텍처 스타일을 **따르는** API

### RESTful API

REST의 설계 원칙을 **완전히 준수**하는 API

**RESTful**이 되려면 다음 조건들을 모두 만족해야 합니다:

### 1. 자원 식별이 명확해야 함

```jsx
// ✅ RESTful
GET /users/123

// ❌ Not RESTful
GET /getUser?id=123
```

### 2. HTTP 메서드를 적절히 사용해야 함

```jsx
// ✅ RESTful - 메서드로 의도를 명확히 표현
GET /users              // 사용자 목록 조회
POST /users             // 새 사용자 생성
PUT /users/123          // 사용자 123 전체 수정
DELETE /users/123       // 사용자 123 삭제

// ❌ Not RESTful - URL에 동작을 포함
GET /users/create
GET /users/delete/123
POST /users/update/123
```

### 실제 예시

온라인 쇼핑몰 API를 RESTful하게 설계한다면:

```jsx
// 상품 관련
GET / products; // 모든 상품 조회
GET / products / 789; // 특정 상품 조회
POST / products; // 새 상품 등록
PUT / products / 789; // 상품 전체 정보 수정
DELETE / products / 789; // 상품 삭제

// 주문 관련
GET / orders; // 모든 주문 조회
GET / orders / 456; // 특정 주문 조회
POST / orders; // 새 주문 생성
PATCH / orders / 456; // 주문 상태 변경 (부분 수정)

// 상품의 리뷰 (계층적 리소스)
GET / products / 789 / reviews; // 상품 789의 모든 리뷰
POST / products / 789 / reviews; // 상품 789에 리뷰 작성
```

### 정리

- **REST**는 웹에서 리소스를 다루는 아키텍처 스타일
- **리소스**는 URI로 명확히 식별
- **HTTP 메서드**로 작업(CRUD)을 표현
- **무상태**, **계층화**, **캐시** 가능한 특징
- **RESTful API**는 REST 설계 원칙을 완전히 준수하는 API
- URI는 **명사**(리소스), HTTP 메서드는 **동사**(행동)로 역할 분담

## 1-1. #Quiz REST API란?

### 문제 1

다음 중 RESTful API 설계 원칙에 가장 부합하는 것은?

```jsx
A. GET /createUser?name=kim
B. POST /user/create
C. POST /users
D. GET /users/add
```

- 정답 및 해설
    
    정답:
    
    C. POST /users
    
    해설:
    
    RESTful API는 리소스를 명사로 표현하고, 행위는 HTTP 메서드로 표현해야 합니다.
    
    A와 D는 URL에 동작(createUser, add)이 포함되어 있고, A는 GET 메서드로 생성 행위를 시도하고 있습니다.
    
    B는 POST를 사용했지만 URL에 create라는 동작이 포함되어 있어 REST 원칙에 어긋납니다.
    
    C는 `/users`라는 리소스를 명사로 표현하고, 생성 행위를 POST 메서드로 표현해 RESTful 설계에 가장 부합합니다.

### 문제 2

다음 코드에서 REST의 어떤 특징을 위반하고 있나요?

```jsx
// 첫 번째 요청
GET / users;
Server: "사용자를 조회했으니 세션에 저장해둘게요";

// 두 번째 요청 (Authorization 없이)
GET / users / 123;
Server: "아까 조회했으니 인증 정보가 있다고 가정할게요";
```

- 정답 및 해설
    
    정답:
    
    무상태(Stateless) 특징 위반
    
    해설:
    
    REST의 무상태(Stateless) 원칙은 각 요청이 독립적이어야 하며, 서버가 이전 요청의 상태나 정보를 기억하지 않아야 한다는 의미입니다.
    
    위 코드에서는 서버가 첫 번째 요청의 정보를 기억하고, 두 번째 요청에서 이를 전제로 동작하고 있습니다. 이는 REST 원칙을 위반한 설계입니다.
    
    올바른 방식에서는 각 요청이 처리에 필요한 모든 정보를 포함해야 합니다.
    
    ```jsx
    GET / users / 123;
    Headers: {
      Authorization: "Bearer token123";
    }
    ```
    
    이처럼 인증 정보나 필요한 데이터는 매 요청마다 함께 전달해야 합니다.

### 문제 3

블로그 API를 설계할 때, "게시글 ID가 5번인 글의 모든 댓글을 조회"하려면 어떤 URI와 HTTP 메서드를 사용해야 할까요?

- 정답 및 해설
    
    정답:
    
    GET /posts/5/comments
    
    해설:
    
    RESTful API에서는 리소스를 명사로 표현하고, 리소스 간의 계층 관계를 URI로 나타냅니다.
    
    게시글과 댓글은 계층적 관계이므로 `/posts/{postId}/comments` 형태가 적절합니다.
    
    ```jsx
    GET / posts / 5 / comments; // 게시글 5번의 모든 댓글 조회
    POST / posts / 5 / comments; // 게시글 5번에 댓글 작성
    GET / posts / 5 / comments / 10; // 게시글 5번의 10번 댓글 조회
    DELETE / posts / 5 / comments / 10; // 게시글 5번의 10번 댓글 삭제
    ```
    
    이 설계는 리소스를 명확히 표현하고, 작업의 의미를 HTTP 메서드로 구분하는 REST 원칙을 잘 따르고 있습니다.
    

## 1-2. HTTP Status Code 간단 이해하기 (암기하자!) ⭐⭐⭐

### HTTP 상태 코드란?

HTTP 상태 코드는 서버가 클라이언트의 요청을 어떻게 처리했는지 알려주는 **3자리 숫자**입니다.

```jsx
// 예시: 사용자 정보 조회 요청
GET /users/123

// 서버 응답
Status: 200 OK
{
  "id": 123,
  "name": "김코딩"
}
```

상태 코드는 **첫 번째 숫자**로 카테고리가 구분됩니다:

- **1xx**: 정보 (요청 수신, 처리 진행 중)
- **2xx**: 성공
- **3xx**: 리다이렉션 (다른 위치로 이동)
- **4xx**: 클라이언트 오류
- **5xx**: 서버 오류

---

### 1xx: 정보 (Informational)

요청을 받았으며, 처리가 진행 중임을 나타냅니다.

실무에서는 거의 볼 일이 없으며, 주로 프로토콜 레벨에서 사용됩니다.

---

### 2xx: 성공 (Success)

요청이 성공적으로 처리되었습니다.

### 200 OK

가장 일반적인 성공 응답입니다.

```jsx
// GET 요청 성공
GET /products/456
Status: 200 OK
{
  "id": 456,
  "name": "노트북",
  "price": 1500000
}
```

### 201 Created

새로운 리소스가 생성되었습니다. 주로 `POST` 요청 후 반환됩니다.

```jsx
// 새 게시글 작성
POST /posts
Body: {
  "title": "첫 게시글",
  "content": "안녕하세요"
}

// 응답
Status: 201 Created
{
  "id": 789,
  "title": "첫 게시글",
  "content": "안녕하세요",
  "createdAt": "2025-01-15"
}
```

### 204 No Content

요청은 성공했지만, 응답 바디가 없습니다. 주로 `DELETE` 요청 후 반환됩니다.

```jsx
// 게시글 삭제
DELETE /posts/789
Status: 204 No Content
// (응답 바디 없음)
```

---

### 3xx: 리다이렉션 (Redirection)

요청을 완료하려면 추가 조치가 필요합니다. 서버가 클라이언트에게 "다른 위치로 가세요"라고 알려주는 것입니다.

### 301 Moved Permanently

리소스가 **영구적으로** 다른 위치로 이동했습니다.

```jsx
// 예: 도메인 변경 또는 URL 구조 변경
GET /old-page
Status: 301 Moved Permanently
Location: /new-page
```

실무 활용:

```jsx
// 도메인 변경
GET http://old-domain.com/products
Status: 301 Moved Permanently
Location: https://new-domain.com/products

// URL 구조 변경
GET /blog/2024/01/post-title
Status: 301 Moved Permanently
Location: /posts/post-title
```

특징:

- 브라우저와 검색엔진이 새 주소를 **저장**합니다
- SEO(검색엔진 최적화)에 반영됩니다
- 다음부터는 새 주소로 바로 접근합니다

언제 사용하나요?

- 사이트 도메인을 변경했을 때
- URL 구조를 영구적으로 변경했을 때
- HTTP를 HTTPS로 영구 전환했을 때

### 302 Found

리소스가 **임시로** 다른 위치에 있습니다.

```jsx
// 예: 로그인 후 대시보드로 이동
POST /login
Body: { "email": "user@example.com", "password": "1234" }

Status: 302 Found
Location: /dashboard
```

실무 활용:

```jsx
// 로그인 후 리다이렉트
POST /login
Status: 302 Found
Location: /dashboard

// 로그인 필요한 페이지 접근 시
GET /my-profile
Status: 302 Found
Location: /login?redirect=/my-profile

// 임시 점검 페이지로 이동
GET /
Status: 302 Found
Location: /maintenance
```

특징:

- 브라우저가 새 주소를 **저장하지 않습니다**
- 다음 요청 시 원래 주소로 다시 시도합니다
- SEO에 영향을 주지 않습니다

301 vs 302 비교:

| 상태 코드 | 의미 | 저장 여부 | SEO 반영 | 주요 사용 사례 |
| --- | --- | --- | --- | --- |
| **301** | 영구 이동 | 저장 | 반영 | 도메인 변경, URL 구조 변경 |
| **302** | 임시 이동 | 저장 안 함 | 반영 안 함 | 로그인 후 이동, 임시 점검 페이지 |

실무 팁

로그인 후 리다이렉트는 거의 항상 **302**(또는 303)를 사용합니다. 영구적인 이동이 아니기 때문입니다.

### 304 Not Modified

캐시된 버전을 사용하라는 의미입니다. 리소스가 **변경되지 않았습니다**.

```jsx
// 브라우저가 이전에 받은 데이터를 저장하고 있음 (캐시)
GET /products/123
Headers: {
  "If-Modified-Since": "2025-01-01 10:00:00"
}

// 서버: 그 이후로 변경된 게 없어요
Status: 304 Not Modified
// (응답 바디 없음, 브라우저는 캐시된 데이터를 사용)
```

캐싱 작동 원리:

```jsx
// 1️⃣ 최초 요청: 전체 데이터를 받음
GET /api/products
Status: 200 OK
Headers: {
  "Last-Modified": "2025-01-15 10:00:00",
  "Cache-Control": "max-age=3600" // 1시간, 초 단위
}
Body: { "id": 1, "name": "노트북", "price": 1500000 }

// 2️⃣ 두 번째 요청: 브라우저가 캐시 유효성 확인
GET /api/products
Headers: {
  "If-Modified-Since": "2025-01-15 10:00:00"
}

// 3️⃣ 서버 응답: 변경 없음
Status: 304 Not Modified
// 응답 바디 없음 → 네트워크 트래픽 절약
```

실무 효과:

- 네트워크 비용 절감
- 속도 향상
- 서버 부하 감소

언제 사용하나요?

- 이미지, CSS, JavaScript 파일 등 자주 변하지 않는 리소스
- API 응답 중 자주 조회되지만 변경이 드문 데이터
- 대용량 파일 다운로드

예시: 프로필 이미지 로딩

```jsx
// 첫 방문
GET /users/123/profile.jpg
Status: 200 OK
Last-Modified: 2025-01-10
// (100KB 이미지 전송)

// 다시 방문 (페이지 새로고침)
GET /users/123/profile.jpg
If-Modified-Since: 2025-01-10
Status: 304 Not Modified
// (데이터 전송 없음, 브라우저 캐시 사용)
```

---

### 4xx: 클라이언트 오류 (Client Error)

클라이언트의 요청에 문제가 있습니다.

### 400 Bad Request

요청이 잘못되었습니다. 필수 필드 누락, 형식 오류 등.

```jsx
// 이메일 형식 오류
POST /users
Body: {
  "name": "김코딩",
  "email": "invalid-email"  // @ 없음
}

Status: 400 Bad Request
{
  "error": "이메일 형식이 올바르지 않습니다."
}
```

### 401 Unauthorized

인증 정보가 없거나 유효하지 않습니다. 로그인이 필요하거나 토큰이 잘못되었습니다.

```jsx
// 토큰 없이 요청
GET /my-profile

Status: 401 Unauthorized
{
  "error": "인증이 필요합니다."
}
```

### 403 Forbidden

인증은 되었지만, 권한이 없습니다.

```jsx
// 일반 사용자가 관리자 페이지 접근
GET /admin/users

Status: 403 Forbidden
{
  "error": "접근 권한이 없습니다."
}
```

401 vs 403 차이:

| 상태 코드 | 의미 | 예시 |
| --- | --- | --- |
| **401** | 인증 안 됨 (로그인 필요) | 토큰 없음, 토큰 만료 |
| **403** | 인증은 됐지만 권한 없음 | 일반 사용자의 관리자 접근 |

핵심

401은 로그인이 안 된 상태, 403은 로그인은 했지만 권한이 없는 상태입니다.

### 404 Not Found

요청한 리소스를 찾을 수 없습니다.

```jsx
// 존재하지 않는 사용자 조회
GET /users/99999

Status: 404 Not Found
{
  "error": "사용자를 찾을 수 없습니다."
}

```

### 409 Conflict

요청이 현재 서버 상태와 충돌합니다. 주로 리소스 중복 시 사용됩니다.

```jsx
// 이미 존재하는 이메일로 회원가입 시도
POST /users
Body: {
  "name": "김코딩",
  "email": "existing@example.com",
  "password": "password123"
}

Status: 409 Conflict
{
  "error": "이미 사용 중인 이메일입니다."
}

```

### 422 Unprocessable Entity

요청 형식은 올바르지만, 의미상 처리할 수 없습니다.

```jsx
// 형식은 맞지만 비즈니스 로직상 문제
POST /orders
Body: {
  "productId": 123,
  "quantity": -5  // 음수 수량
}

Status: 422 Unprocessable Entity
{
  "error": "수량은 0보다 커야 합니다."
}
```

400 vs 422 차이:

- `400`: JSON 형식 오류, 필수 필드 누락 등 형식 오류
- `422`: 형식은 맞지만 비즈니스 규칙 위반

### 429 Too Many Requests

요청 횟수 제한을 초과했습니다. (Rate Limiting)

```jsx
// 1분에 100번 이상 요청
GET /api/products

Status: 429 Too Many Requests
{
  "error": "요청 횟수를 초과했습니다. 1분 후 다시 시도해주세요.",
  "retryAfter": 60
}
```

---

### 5xx: 서버 오류 (Server Error)

서버에서 요청을 처리하는 중 오류가 발생했습니다.

### 500 Internal Server Error

서버 내부에서 예상치 못한 오류가 발생했습니다.

```jsx
GET /products

Status: 500 Internal Server Error
{
  "error": "서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요."
}
```

### 502 Bad Gateway

게이트웨이나 프록시 서버가 잘못된 응답을 받았습니다.

```jsx
// API 서버가 응답하지 않을 때
GET /users

Status: 502 Bad Gateway
```

### 503 Service Unavailable

서버가 일시적으로 요청을 처리할 수 없습니다. (유지보수, 과부하 등)

```jsx
GET /products

Status: 503 Service Unavailable
{
  "error": "서비스 점검 중입니다. 오후 2시에 다시 이용해주세요."
}
```

---

### 실전 예시

쇼핑몰 API의 다양한 상태 코드:

```jsx
// 1. 상품 목록 조회 성공
GET /products
Status: 200 OK

// 2. 새 상품 등록 성공
POST /products
Status: 201 Created

// 3. 필수 필드 누락
POST /products
Body: { "name": "노트북" }  // price 누락
Status: 400 Bad Request

// 4. 로그인 없이 장바구니 접근
GET /cart
Status: 401 Unauthorized

// 5. 존재하지 않는 상품 조회
GET /products/99999
Status: 404 Not Found

// 6. 상품 삭제 성공
DELETE /products/123
Status: 204 No Content

// 7. 데이터베이스 연결 오류
GET /products
Status: 500 Internal Server Error

```

---

### 정리

| 카테고리 | 의미 | 주요 코드 |
| --- | --- | --- |
| **1xx** | 정보 | 거의 사용 안 함 |
| **2xx** | 성공 | 200, 201, 204 |
| **3xx** | 리다이렉션 | 301, 302, 304 |
| **4xx** | 클라이언트 오류 | 400, 401, 403, 404, 409, 422, 429 |
| **5xx** | 서버 오류 | 500, 502, 503 |

### 핵심 원칙

- **2xx**: 모든 게 잘 됨
- **4xx**: 클라이언트가 뭔가 잘못함
- **5xx**: 서버가 뭔가 잘못함

적절한 상태 코드를 사용하면 API를 사용하는 개발자가 문제를 빠르게 파악하고 대응할 수 있습니다.

## 1-2. #Quiz HTTP Status Code 간단 이해하기

### 문제 1

다음 상황에서 서버가 반환해야 하는 적절한 HTTP 상태 코드는?

```jsx
// 사용자가 회원가입 요청
POST /users
Body: {
  "name": "이코딩",
  "email": "coding@example.com",
  "password": "password123"
}

// 서버가 새 사용자를 데이터베이스에 성공적으로 저장함
```

**A. 200 OK**

**B. 201 Created**

**C. 204 No Content**

**D. 202 Accepted**

- 정답 및 해설
    
    정답:
    
    B. 201 Created
    
    해설:
    
    `201 Created`는 새로운 리소스가 생성되었을 때 사용하는 상태 코드입니다. `POST` 요청으로 회원가입을 하면 새 사용자가 생성되므로 `201`이 가장 적절합니다.
    
    `200 OK`는 일반적인 성공 응답이지만, 리소스 생성 상황에서는 `201`이 더 명확합니다.
    
    `204 No Content`는 성공은 했지만 응답 바디가 없을 때 사용되며, 보통 DELETE에 적합합니다.
    
    `202 Accepted`는 요청을 받았지만 아직 처리되지 않았을 때 사용합니다.

### 문제 2

다음 코드의 출력 결과로 예상되는 상태 코드들을 순서대로 나열하세요:

```jsx
// 1) 로그인하지 않은 사용자가 프로필 페이지 접근
GET /my-profile

// 2) 존재하지 않는 상품 조회
GET /products/99999

// 3) 일반 사용자가 관리자 전용 페이지 접근 (로그인은 되어 있음)
GET /admin/settings
Headers: {
  Authorization: "Bearer valid-token"
}

// 4) 잘못된 이메일 형식으로 회원가입 시도
POST /users
Body: {
  "email": "invalid-email",  // @ 없음
  "password": "123456"
}
```

- 정답 및 해설
    
    정답:
    
    1. `401 Unauthorized`
    2. `404 Not Found`
    3. `403 Forbidden`
    4. `400 Bad Request`
    
    해설:
    
    1. `401 Unauthorized`는 인증이 필요한 페이지에 로그인 없이 접근했을 때 사용됩니다.
    2. `404 Not Found`는 요청한 리소스가 존재하지 않을 때 사용됩니다.
    3. `403 Forbidden`은 인증은 되었지만 해당 리소스에 접근할 권한이 없을 때 반환됩니다.
    4. `400 Bad Request`는 클라이언트가 잘못된 형식의 데이터를 보냈을 때 사용됩니다.
    
    `401`은 “로그인이 안 됨”, `403`은 “로그인은 됐지만 권한 없음”이라는 차이가 있습니다.

### 문제 3

다음 시나리오에서 발생할 수 있는 상태 코드를 모두 고르세요:

```jsx
// 게시글 삭제 요청
DELETE / posts / 123;
Headers: {
  Authorization: "Bearer token";
}
```

**가능한 시나리오:**

**A. 200 OK**

**B. 204 No Content**

**C. 401 Unauthorized**

**D. 403 Forbidden**

**E. 404 Not Found**

**F. 409 Conflict**

**G. 500 Internal Server Error**

- 정답 및 해설
    
    정답:
    
    A, B, C, D, E, F, G (모두 가능)
    
    해설:
    
    게시글 삭제 API는 상황에 따라 다양한 상태 코드를 반환할 수 있습니다.
    
    삭제가 성공하고 데이터를 함께 반환하면 `200 OK`, 바디 없이 성공만 알리면 `204 No Content`를 사용할 수 있습니다.
    
    인증 문제가 있으면 `401 Unauthorized`, 권한 문제가 있으면 `403 Forbidden`이 반환됩니다.

    동시 삭제 충돌이나 비즈니스 충돌이 발생하면 `409 Conflict`가 사용될 수 있습니다.
    
    서버 내부 오류나 데이터베이스 장애 같은 예외 상황에서는 `500 Internal Server Error`가 반환됩니다.
    
    적절한 상태 코드를 사용하면 클라이언트가 상황을 정확히 이해하고 올바르게 대응할 수 있습니다.
    

## 1-3. 요즘 Frontend와 요즘 Backend의 통신

### 개요

과거에는 서버가 HTML을 만들어 전달했지만, 요즘에는 **Frontend와 Backend가 완전히 분리**되어 있습니다.

```
과거: 서버 → HTML 전체 생성 → 브라우저
요즘: 서버 → JSON 데이터 → Frontend(React, Vue) → 화면 렌더링
```

Frontend와 Backend는 주로 **HTTP 요청**으로 JSON 데이터를 주고받으며 통신합니다.

---

### 주요 통신 방식

### 1. Fetch API / Axios

브라우저에서 HTTP 요청을 보내는 가장 일반적인 방법입니다.

Fetch API (기본 제공)

```jsx
// GET 요청
fetch("https://api.example.com/users")
  .then((response) => response.json())
  .then((data) => console.log(data))
  .catch((error) => console.error("오류:", error));

// POST 요청
fetch("https://api.example.com/users", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    name: "김코딩",
    email: "coding@example.com",
  }),
})
  .then((response) => response.json())
  .then((data) => console.log("생성됨:", data));
```

Axios (라이브러리)

```jsx
// GET 요청
axios
  .get("https://api.example.com/users")
  .then((response) => console.log(response.data))
  .catch((error) => console.error("오류:", error));

// POST 요청
axios
  .post("https://api.example.com/users", {
    name: "김코딩",
    email: "coding@example.com",
  })
  .then((response) =>
    console.log("생성됨:", response.data)
  );
```

Axios의 장점:

- JSON 자동 변환
- 요청/응답 인터셉터 (중간에 가로채서 처리)
- 에러 처리가 더 간단
- 타임아웃 설정 가능

---

### 2. GraphQL

REST API의 문제점:

```jsx
// 사용자와 게시글 정보가 필요하면 2번 요청
GET /users/123
GET /posts?userId=123

// 또는 과도한 데이터 반환
GET /users/123
// → 필요 없는 프로필 이미지, 주소 등도 함께 옴
```

GraphQL은 필요한 데이터만 한 번에 요청:

```jsx
// 하나의 요청으로 원하는 데이터만
query {
  user(id: "123") {
    name
    email
    posts {
      title
      createdAt
    }
  }
}

// 응답
{
  "user": {
    "name": "김코딩",
    "email": "coding@example.com",
    "posts": [
      {
        "title": "첫 게시글",
        "createdAt": "2025-01-15"
      }
    ]
  }
}
```

GraphQL 특징:

- 단일 엔드포인트: `/graphql` 하나만 사용
- 필요한 데이터만 요청: Over-fetching, Under-fetching 방지
- 타입 시스템: 스키마로 API 구조 명확히 정의

---

### 3. REST + JWT (토큰 기반 인증)

요즘 웹 앱은 **상태를 저장하지 않는(Stateless)** JWT를 사용합니다.

기존 세션 방식:

```jsx
// 1. 로그인
POST /login
// 서버가 세션 생성 → 서버 메모리에 저장

// 2. 인증 필요한 요청
GET /my-profile
Cookie: session_id=abc123
// 서버가 세션 확인
```

JWT 방식:

```jsx
// 로그인
POST /login

// 응답 (쿠키 저장 지시)
HTTP/1.1 200 OK
Set-Cookie: accessToken=...; HttpOnly

// 이후 요청 (브라우저가 자동 전송)
GET /my-profile
Cookie: accessToken=...
```

JWT 장점:

- 서버가 상태를 저장하지 않음 (확장성 좋음)
- 토큰에 사용자 정보 포함 가능
- 여러 서버에서 쉽게 검증 가능

---

### 현대적 아키텍처

### 1. CORS (Cross-Origin Resource Sharing)

Frontend와 Backend가 다른 도메인에 있을 때 필요합니다.

```jsx
// Frontend: http://localhost:3000
// Backend: http://localhost:4000

// Frontend에서 요청
fetch("http://localhost:4000/api/users").then((response) =>
  response.json()
);

// Backend 설정 (Express)
app.use(
  cors({
    origin: "http://localhost:3000", // CORS 허용: 이 프론트엔드에서 오는 요청만 허용
    credentials: true, // 인증 정보 전달 허용: 쿠키 기반 인증을 위해 필요
  })
);
```

CORS가 없으면:

```
Access to fetch at 'http://localhost:4000/api/users'
from origin 'http://localhost:3000' has been blocked by CORS policy
```

---

### 2. API Gateway

여러 마이크로서비스를 하나의 진입점으로 통합합니다.

```
                    ┌─────────────────┐
                    │   API Gateway   │
                    │   (단일 진입점)    │
                    └────────┬────────┘
                             │
            ┌────────────────┼────────────────┐
            │                │                │
      ┌─────▼──────┐    ┌────▼─────┐    ┌─────▼─────┐
      │  User      │    │  Product │    │  Order    │
      │  Service   │    │  Service │    │  Service  │
      └────────────┘    └──────────┘    └───────────┘

```

API Gateway의 역할:

- 라우팅: 요청을 적절한 서비스로 전달
- 인증: 토큰 검증을 한 곳에서 처리
- 로드밸런싱: 트래픽 분산
- 캐싱: 자주 요청되는 데이터 캐시
- 로깅: 모든 요청 기록

---

### 3. 마이크로서비스 아키텍처

기능별로 서비스를 분리하여 개발·배포합니다.

```jsx
// 모놀리식
┌───────────────────────────┐
│  하나의 거대한 애플리케이션      │
│  - 사용자 관리 (User)        │            
│  - 상품 관리   (Product)    │
│  - 주문 처리   (Order)      │
│  - 결제 처리   (Payment)    │
└───────────────────────────┘

// 마이크로서비스
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  User    │  │ Product  │  │  Order   │  │ Payment  │
│ Service  │  │ Service  │  │ Service  │  │ Service  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

마이크로서비스 장점:

- 각 서비스를 독립적으로 배포
- 기술 스택을 서비스별로 다르게 선택 가능
- 장애가 격리됨 (한 서비스 다운 ≠ 전체 다운)

---

### 4. Monorepo 아키텍처

Frontend와 Backend 코드를 하나의 저장소에서 관리합니다.

```
my-project/
├── apps/
│   ├── frontend/
│   │   ├── src/
│   │   └── package.json
│   └── backend/
│       ├── src/
│       └── package.json
├── packages/
│   └── shared/
│       ├── types/
│       └── utils/
└── package.json
```

Monorepo 장점:

- 코드 공유가 쉬움 (타입, 유틸리티 등)
- 일관된 버전 관리
- 하나의 PR로 Frontend-Backend 동시 수정 가능

---

### 정리

### 주요 통신 방식

- Fetch API / Axios: HTTP 요청의 표준
- GraphQL: 필요한 데이터만 효율적으로 요청
- REST + JWT: 토큰 기반 인증으로 무상태 유지

### 현대적 아키텍처

- CORS: 도메인 간 요청 제어
- API Gateway: 단일 진입점으로 보안·라우팅 관리
- 마이크로서비스: 기능별 서비스 분리로 확장성 확보
- Monorepo: Frontend-Backend 코드를 하나의 저장소에서 관리

## 1-3. #Quiz 요즘 Frontend와 요즘 Backend의 통신

# 요즘 Frontend와 Backend 통신 퀴즈

### 문제 1

다음 중 Fetch API와 Axios의 차이점으로 **틀린** 것은?

**A. Axios는 JSON 변환을 자동으로 해주지만, Fetch는 `.json()` 메서드를 명시적으로 호출해야 한다**

**B. Axios는 별도로 설치해야 하는 라이브러리이지만, Fetch API는 브라우저에 기본 제공된다**

**C. Axios는 요청/응답 인터셉터를 제공하지만, Fetch API는 제공하지 않는다**

**D. Axios는 POST 요청을 보낼 수 없지만, Fetch API는 가능하다**

- 정답 및 해설
    
    정답:
    
    D. Axios는 POST 요청을 보낼 수 없지만, Fetch API는 가능하다
    
    해설:
    
    Axios도 당연히 POST 요청을 보낼 수 있습니다. 오히려 Axios가 더 간단합니다.
    
    ```jsx
    // Fetch API - POST 요청
    fetch("https://api.example.com/users", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify({ name: "김코딩" }),
    }).then((response) => response.json());
    
    // Axios - POST 요청 (더 간단)
    axios
      .post("https://api.example.com/users", {
        name: "김코딩",
      })
      .then((response) => response.data);
    ```
    
    나머지 선택지들은 모두 맞는 설명입니다. Axios는 `response.data`로 바로 접근할 수 있고, Fetch는 `.json()` 메서드를 호출해야 합니다. Fetch API는 브라우저에 기본 제공되지만 Axios는 별도 설치가 필요합니다. 또한 Axios는 요청과 응답 인터셉터를 제공하여 중간 처리 로직을 쉽게 추가할 수 있습니다.

### 문제 2

다음 REST API와 GraphQL 요청을 비교하고, GraphQL의 장점을 설명하세요.

```jsx
// REST API
GET /users/123
// 응답: 사용자의 모든 정보 (이름, 이메일, 주소, 전화번호, 프로필 이미지 등)

GET /posts?userId=123
// 응답: 해당 사용자의 게시글 목록

// GraphQL
query {
  user(id: "123") {
    name
    posts {
      title
    }
  }
}
// 응답: 이름과 게시글 제목만

```

- 정답 및 해설
    
    정답:
    
    GraphQL의 주요 장점은 필요한 데이터만 요청할 수 있고, 하나의 요청으로 여러 리소스를 조회할 수 있으며, 단일 엔드포인트를 사용한다는 점입니다.
    
    해설:
    
    REST API에서는 특정 리소스를 조회하면 서버가 정의한 모든 필드를 응답으로 받게 되기 때문에, 실제로 필요하지 않은 데이터까지 함께 전달되는 경우가 많습니다. 이를 Over-fetching이라고 합니다. GraphQL은 클라이언트가 필요한 필드를 직접 명시하기 때문에 이런 문제를 줄일 수 있습니다.
    
    또한 REST API에서는 사용자 정보와 게시글 정보를 가져오기 위해 각각 다른 엔드포인트로 여러 번 요청해야 하지만, GraphQL에서는 하나의 쿼리로 관련된 여러 데이터를 한 번에 가져올 수 있습니다. 마지막으로 REST는 여러 엔드포인트를 사용하는 반면, GraphQL은 일반적으로 `/graphql` 하나의 엔드포인트만 사용하여 데이터 요청을 처리합니다. 이로 인해 Over-fetching과 Under-fetching 문제를 효과적으로 해결할 수 있습니다.

### 문제 3

JWT 토큰 기반 인증과 세션 기반 인증의 차이를 설명하고, 다음 코드의 빈칸을 채우세요.

```jsx
// 로그인 성공 후 토큰 저장
const login = async (email, password) => {
  const response = await axios.post(
    "https://api.example.com/auth/login",
    {
      email,
      password,
    }
  );

  // 빈칸 1: 토큰을 어디에 저장할까요?
  __________.setItem("token", response.data.token);
};

// 인증이 필요한 API 요청
const getMyProfile = async () => {
  const token = __________.getItem("token"); // 빈칸 2

  const response = await axios.get(
    "https://api.example.com/users/me",
    {
      headers: {
        // 빈칸 3: 어떤 헤더에 토큰을 넣을까요?
        __________: `Bearer ${token}`,
      },
    }
  );

  return response.data;
};
```

- 정답 및 해설
    
    정답:
    
    빈칸 1은 `localStorage`이고, 빈칸 2 역시 `localStorage`이며, 빈칸 3은 `Authorization`입니다.
    
    해설:
    
    JWT 방식에서는 로그인 성공 시 서버가 토큰을 생성해 클라이언트에게 전달하고, 클라이언트는 이 토큰을 `localStorage`나 `sessionStorage`에 저장하는 경우가 많습니다. 이후 인증이 필요한 요청마다 HTTP 헤더의 `Authorization` 필드에 `Bearer 토큰값` 형태로 포함시켜 서버에 전달합니다.
    
    세션 기반 인증은 서버가 로그인 시 세션을 생성하고, 세션 ID를 쿠키로 클라이언트에게 전달합니다. 서버는 이후 요청에서 전달된 세션 ID를 기반으로 서버 메모리나 데이터베이스에서 사용자 정보를 조회합니다. 이 방식은 서버가 상태를 저장하는 구조입니다.
    
    반면 JWT 기반 인증은 서버가 상태를 저장하지 않습니다. 토큰 자체에 사용자 정보와 서명이 포함되어 있고, 서버는 요청이 올 때마다 토큰의 서명을 검증해 유효성을 판단합니다. 이로 인해 서버 확장이 쉽고, 여러 서버 환경에서도 인증 처리가 수월합니다. 다만 토큰이 탈취될 경우 만료 전까지 사용 가능하다는 단점이 있어 짧은 만료 시간과 Refresh Token 전략으로 보완합니다.
    

## 1-4. Postman ⭐⭐

### Postman이란?

**Postman**은 API 개발과 테스트를 위한 강력한 도구입니다. 브라우저 없이 HTTP 요청을 보내고, 응답을 확인하며, API를 체계적으로 관리할 수 있습니다.

왜 필요할까?

```jsx
// 브라우저에서는 GET 요청만 쉽게 테스트 가능
주소창에 입력: https://api.example.com/users

// POST, PUT, DELETE는?
// → JavaScript 코드를 작성해야 함 (번거로움)
// → Postman을 사용하면 GUI로 간편하게 테스트!
```

# 2. 백엔드 서버 아키텍처

## 2-1. 서버 아키텍처 개요 ⭐⭐⭐

### 1. 기능별 구조 (Feature-based Architecture, PART.2)

기능별 구조는 **“서버를 구성하는 요소들을 성격별로 정리하는 방식”**입니다.

서버 프로젝트를 처음 만들 때 가장 먼저 접하게 되는 형태이기도 합니다.

```jsx
src/
├── config/          # 환경설정 (환경변수 검증, 데이터베이스 연결)
├── models/          # 데이터 모델 정의 (MongoDB 스키마)
├── repository/      # 데이터 접근 계층 (DB CRUD 작업)
├── middlewares/     # 공통 미들웨어 (인증, CORS, 로깅, 에러처리)
├── routes/          # API 라우터 정의
├── errors/          # 커스텀 에러 클래스
└── server.js        # 서버 진입점
```

이 구조는 “아키텍처 이론”이라기보다,

**서버에 필요한 구성 요소들을 정리하는 관점**이라고 보면 됩니다.

---

### config/

- 환경변수 검증
- 데이터베이스 연결
- 서버 설정 로딩

서버 실행에 필요한 설정들을 한 곳에 모아둡니다.

---

### models/

- 데이터베이스 스키마 정의
- MongoDB Mongoose Schema
- ORM을 쓰면 Entity 역할

데이터 구조를 정의하는 곳입니다.

---

### repository/

- 데이터베이스 CRUD 작업
- 쿼리 로직 모음

DB 접근 코드를 직접 서비스나 컨트롤러에 쓰지 않기 위해 분리합니다.

---

### middlewares/

- 인증
- CORS
- 로깅
- 에러 핸들링

요청 흐름의 **앞/뒤에서 공통으로 처리되는 로직**입니다.

---

### routes/

- API 엔드포인트 정의
- URL과 컨트롤러 연결

라우터는 로직이 아니라 **경로 정의 역할**만 담당합니다.

---

### errors/

- HTTP 에러 클래스 정의
- BadRequest, NotFound, Unauthorized 등

에러를 의미 있게 다루기 위한 계층입니다.

---

### server.js

- Express 앱 초기화
- 미들웨어 등록
- 라우터 등록
- 서버 실행

서버의 시작점입니다.

---

### 기능별 구조의 한계

프로젝트가 커질수록 이런 문제가 생깁니다.

- 비즈니스 로직이 여기저기 흩어짐
- 컨트롤러가 점점 비대해짐
- DB 접근과 로직이 섞이기 시작함

이 문제를 해결하기 위해 등장하는 개념이 **계층형 아키텍처**입니다.

---

### 2. 계층형 아키텍처 (Layered Architecture,  PART.3)

계층형 아키텍처는 소프트웨어를 **책임(역할)별로 나누는 설계 방식**입니다.

핵심 특징은 다음과 같습니다.

- 각 계층은 명확한 책임을 가진다
- 데이터 흐름은 단방향이다
- 유지보수, 테스트, 확장이 쉬워진다

```jsx
┌─────────────────┐
│   Controllers   │  ← HTTP 요청/응답 처리
├─────────────────┤
│    Services     │  ← 비즈니스 로직 처리
├─────────────────┤
│   Repository    │  ← 데이터 접근 계층
├─────────────────┤
│    Database     │  ← 데이터 저장소
└─────────────────┘
```

---

### Controllers 계층

- HTTP 요청을 받는다
- 요청 데이터를 파싱한다
- 서비스 계층에 위임한다
- 결과를 HTTP 응답으로 반환한다

컨트롤러는 **비즈니스 로직을 가지지 않는다**는 점이 중요합니다.

---

### Services 계층

- 실제 비즈니스 로직이 위치
- 데이터 가공
- 검증, 중복 체크
- 여러 repository 조합

프로그램의 핵심 로직은 모두 이 계층에 존재합니다.

---

### Repository 계층

- 데이터베이스 접근 전담
- CRUD 쿼리 처리

서비스는 “무엇을 가져와라”만 요청하고,

“어떻게 DB에 접근하는지”는 repository가 책임집니다.

---

### Database 계층

- 실제 데이터 저장소
- MongoDB, PostgreSQL, MySQL 등

코드가 아니라 인프라 자원입니다.

---

### 단방향 흐름의 중요성

기본 흐름은 항상 다음과 같아야 합니다.

```
Controller → Service → Repository → Database
```

이 규칙이 깨지면 구조는 금방 무너집니다.

---

### 3. 최종 폴더 구조와 역할

실무에서는 **기능별 구조 + 계층형 아키텍처**를 함께 사용합니다.

```jsx
src/
├── controllers/     # API 엔드포인트 처리
├── services/        # 비즈니스 로직
├── repository/      # 데이터 접근 계층
├── moodels/         # 데이터베이스 스키마 (# 또는 entities/)
├── providers/       # DI로 주입되는 공통 의존성
├── middlewares/     # 횡단 관심사(AOP)
└── common/          # 공통 유틸리티 (DTO, Exception)
```

이 구조는 **계층형 아키텍처를 폴더 구조로 구현한 결과**입니다.

---

### 4. providers와 common의 차이

### providers/

**providers는 “DI 생성 시 주입해주는 대상들”입니다.**

즉,

- 클래스(service, repository)를 만들 때
- 생성자에서 주입해주는 공통 의존성 객체

예를 들면:

- db (DB 커넥션, client)
- config (환경설정 객체)
- jwt (토큰 생성/검증 객체)
- redis, email, logger 등

공통 특징은 이렇습니다.

- 외부 의존성을 가진다
- 상태나 설정을 가진다
- 주입하지 않으면 테스트/교체가 어렵다
- 객체 형태로 사용된다

---

### common/

**common은 “주입할 필요 없는 재사용 코드”입니다.**

예를 들면:

- 순수 유틸리티 함수
- DTO 정의 (공통으로 사용하는)
- 공통 에러 클래스
- 공통 타입 정의

공통 특징은 다음과 같습니다.

- 순수 함수 또는 단순 구조
- 상태를 가지지 않는다
- new 해서 만들 필요가 없다
- DI 대상이 아니다

---

### providers vs common 핵심 정리

- **providers**
    
    → “주입해야 하는 객체(의존성)”
    
- **common**
    
    → “주입할 필요 없는 재사용 코드”
    

---

### 정리

- 기능별 구조는 **서버 구성 요소를 정리하는 관점**
- 계층형 아키텍처는 **책임과 흐름을 통제하는 설계**
- 최종 구조는 **두 개념을 합친 실무형 구조**
- providers와 common의 차이는 **DI 가능 여부**

이 구조를 이해하면, 서버 코드가 커져도 “왜 여기에 있어야 하는 코드인지” 명확하게 판단할 수 있습니다.

```jsx
        process: 'readonly',
      },
    },
    rules: {
      'no-unused-vars': 'warn',
      'no-console': 'off',
      'prefer-const': 'error',
      'no-var': 'error',
      semi: ['error', 'always'],
      quotes: ['error', 'single'],
    },
  },
];
```

`"dev": "node src/server.js",`

`npm run dev`를 실행하면 `node src/server.js`가 실행됩니다. 개발 단계에서 서버를 켤 때 쓰는 명령이에요.

`"format": "prettier --write src/**/*.js",`

Prettier로 `src` 폴더 아래의 모든 `.js` 파일을 자동으로 정리합니다. `--write`라서 파일이 실제로 수정됩니다.

`"format:check": "prettier --check src/**/*.js"`

코드를 수정하지 않고, 포맷이 맞는지만 검사합니다. 팀 프로젝트에서 CI로 “포맷 안 맞으면 실패”시키려고 자주 씁니다.

`"keywords": ["express"]`

프로젝트를 설명하는 키워드입니다. 실행에는 영향 없고, 주로 패키지 배포/분류용 정보예요.

`"author": { ... }`

작성자 정보입니다. 실행에는 영향 없고, 프로젝트/패키지 메타데이터로 쓰입니다.

`"engines": { ... }`

이 프로젝트가 기대하는 Node/npm 버전을 명시합니다. 팀에서 버전 달라서 생기는 문제를 줄이려고 넣습니다.

`"license": "MIT",`

이 코드의 사용 허가(라이선스) 정보입니다. 실행에는 영향 없고, 공개할 때 명시하는 용도입니다.

## 02-express-server-start

### 1. server.js 파일 생성

```jsx
import express from 'express';

const app = express();
const PORT = 5001;

// JSON 파싱 미들웨어
app.use(express.json());

// 기본 라우트
app.get('/', (req, res) => {
  res.json({
    message: 'Hello Express!!!!',
    timestamp: new Date().toISOString()
  });
});

// 서버 시작
app.listen(PORT, () => {
  console.log(`🚀 Server running on http://localhost:${PORT}`);
});
```

### 2. 미들웨어(Middleware)란? ⭐⭐⭐

Express의 핵심 개념 중 하나로, **요청(request)이 들어와 응답(response)이 나가기 전 중간 과정에 끼어들어 특정 작업을 처리하는 함수**입니다.

Express에서는 **`app.use()`**를 사용해 미들웨어를 등록합니다.

여기서 중요한 건 “미들웨어는 항상 모든 요청에 대해 먼저 실행될 수도 있다”는 점입니다.

라우트(`/`)로 들어오든, 다른 주소로 들어오든, `app.use()`로 등록된 미들웨어는 기본적으로 먼저 거칩니다.

또 한 가지 감각은 이거예요.

- 라우트 핸들러(`app.get(...)`)는 “특정 주소/메서드에만” 동작하는 로직
- 미들웨어(`app.use(...)`)는 “요청 전체 흐름에서 공통으로 적용”되는 로직

그래서 JSON 파싱, 로그 기록, 인증 같은 것들은 보통 미들웨어로 빼서 처리합니다.

### 3. `express.json()`의 역할

- 클라이언트(웹 브라우저, 모바일 앱 등)가 서버로 데이터를 전송할 때, 데이터를 **요청 본문(Request Body)**에 담아 보냅니다.
- 보통 요청 데이터 형식이 JSON(**`{ "key": "value" }`**)일 때가 많습니다.
- Express는 기본적으로 이 JSON 데이터를 바로 사용할 수 없습니다.
- **`express.json()`** 미들웨어는 **요청의 Content-Type 헤더가 application/json**인 경우,
    
    요청 본문의 JSON 문자열을 **자바스크립트 객체로 변환**하여 **`req.body`** 프로퍼티에 저장해줍니다.
    
- 따라서 라우트 핸들러 안에서 **`req.body.key`*처럼 쉽게 데이터를 쓸 수 있습니다.

초보자들이 여기서 많이 착각하는 부분이 하나 있는데요.

`express.json()`은 “응답을 JSON으로 보내주는 기능”이 아니라, **요청(body)을 JSON으로 읽게 해주는 기능**입니다.

그리고 이 코드에서는 `app.get('/')`라서 body를 보내는 요청은 아니지만,

다음 단계에서 `POST /users` 같은 걸 만들 때 `req.body`를 쓰려면 미리 등록해두는 게 자연스러운 흐름입니다.

### 1. 기본 라우팅

`/src/server.js`

```jsx
// JSON 파싱 미들웨어
app.use(express.json());

// ----- 추가 -----
// 기본 HTTP 메서드
app.get('/users', (req, res) => {
  res.json({ users: [] });
});

// postman에서 변경 테스트 하려면 Header -> Content-Type을 application/json 확인하기
app.post('/users', (req, res) => {
  const { name, email } = req.body;
  res.json({ message: '사용자 생성됨', name, email });
});

app.put('/users/:id', (req, res) => {
  res.json({ message: `사용자 ${req.params.id} 업데이트` });
});

app.delete('/users/:id', (req, res) => {
  res.json({ message: `사용자 ${req.params.id} 삭제` });
});
```

여기서는 Express에서 가장 기본이 되는 “라우팅”을 한 번에 보는 파트입니다.

라우팅은 쉽게 말해서 **“어떤 요청이 들어오면 어떤 응답을 줄지”를 정하는 규칙**이에요.

핵심은 이 구조입니다.

- `app.get(...)` → GET 요청이 들어오면 실행
- `app.post(...)` → POST 요청이 들어오면 실행
- `app.put(...)` → PUT 요청이 들어오면 실행
- `app.delete(...)` → DELETE 요청이 들어오면 실행

즉, 같은 `/users` 주소여도 **“요청 메서드”가 다르면 서로 다른 코드가 실행**됩니다.

- `GET /users` → 유저 목록 조회
- `POST /users` → 유저 생성
- `PUT /users/:id` → 특정 유저 수정
- `DELETE /users/:id` → 특정 유저 삭제

그리고 POST 요청에서 `req.body`를 쓰는 이유는,

사용자 생성 정보(name, email)를 보통 요청 본문에 담아서 보내기 때문입니다.

### 2. 매개변수와 쿼리

`/src/server.js`

```jsx
app.delete('/users/:id', (req, res) => {
  res.json({ message: `사용자 ${req.params.id} 삭제` });
});

// ----- 추가 -----
// URL 매개변수
app.get('/users/:id', (req, res) => {
  const { id } = req.params;
  res.json({ userId: id });
});

// 쿼리 문자열 - /search?q=express&limit=10
app.get('/search', (req, res) => {
  const { q, limit = 10 } = req.query;
  res.json({ query: q, limit: Number(limit) });
});

// 여러 매개변수
app.get('/users/:userId/posts/:postId', (req, res) => {
  const { userId, postId } = req.params;
  res.json({ userId, postId });
});
```

여기서는 URL에서 값을 받는 두 가지 방식을 배우는 파트입니다.

### 1) URL 매개변수(req.params)

`/users/:id` 처럼 콜론(`:`)이 붙은 부분은 **“변수 자리”**입니다.

그래서 실제 요청이 `/users/123`으로 들어오면,

- `req.params.id`는 `"123"`이 됩니다. (문자열로 들어옴)

이 방식은 보통 “특정 하나를 조회/수정/삭제”할 때 씁니다.

- `GET /users/1` → 1번 유저 조회
- `PUT /users/1` → 1번 유저 수정
- `DELETE /users/1` → 1번 유저 삭제

### 2) 쿼리(req.query)

`/search?q=express&limit=10`처럼 `?key=value` 형태로 붙는 값들은 `req.query`로 들어옵니다.

이 방식은 보통 “필터/검색/정렬/페이지네이션” 같은 옵션에 쓰입니다.

- 검색어 `q`
- 몇 개 가져올지 `limit`
- 정렬 기준 `sort`

여기서 `limit`을 `Number(limit)`로 바꾼 이유는,

`req.query` 값이 기본적으로 문자열이기 때문입니다.

즉, `limit`이 `"10"`으로 들어오니까 숫자로 쓰려면 변환이 필요합니다.

### 3) 여러 매개변수

`/users/:userId/posts/:postId`는 중첩된 리소스를 표현할 때 자주 씁니다.

- “어떤 유저의”
- “어떤 게시글”

이런 구조로 URL을 설계하면 의미가 명확해집니다.

## 04-route-separation

### 라우팅(Routing) 분리 또는 라우트(Route) 모듈화란?

Express의 **`Router`** 객체를 사용해서 기능별(예: 사용자, 게시글, 상품 등)로 라우트 파일을 분리하는 작업입니다.

라우트(route) 코드를 여러 파일로 나누어 모듈로 관리합니다.

여기서 중요한 포인트는 이거예요.

- `app`은 “서버 전체 설정(미들웨어, 포트, 서버 실행)”에 집중
- `router`는 “URL별 처리 로직(라우트)”에 집중

즉, 서버가 커질수록 `server.js`(또는 app.js)는 얇아지고, 기능별 라우트 파일이 늘어나는 구조가 됩니다.

---

### 왜 라우팅을 분리해야 할까요?

- **유지보수성**
    
    애플리케이션이 커지면 app.js 파일 하나에 수십, 수백 개의 라우트가 생길 수 있습니다.
    
    기능별로 파일을 분리하면 코드를 찾고 수정하기가 훨씬 쉬워집니다.
    
- **모듈성 및 재사용성**
    
    특정 기능에 관련된 API 전체를 하나의 모듈처럼 다룰 수 있습니다.
    
    각 라우트 파일을 독립적으로 관리하고, 필요에 따라 재사용할 수 있습니다.
    
- **가독성**
    
    메인 파일(app.js)은 전반적인 설정(미들웨어, 서버 연결 등)에만 집중하고,
    
    세부적인 API 경로는 각 라우트 파일에 위임하여 코드가 훨씬 깔끔해집니다
    

추가로 초보자 입장에서 한 가지 더 이유가 있습니다.

파일이 분리되면 “내가 지금 어디를 수정해야 하는지”가 명확해져요.

- 사용자 API 수정 → `routes/users.js`
- 검색 API 수정 → `routes/search.js`
- 전체 라우트 묶기/기본 라우트 → `routes/index.js`
- 서버 포트/미들웨어/실행 → `server.js`

---

### 코드 작성하기

### 1. routes/users.js

여기서는 “사용자 관련 URL”을 전부 한 파일에 모읍니다.

중요한 감각은 `userRouter.get('/')`의 `/`가 “진짜 /가 아니라” **이 라우터의 기준 경로**라는 점입니다.

즉, `server.js`나 `index.js`에서 `/users`로 붙여주면, 최종 경로는:

- `GET /users`
- `GET /users/:id`

이렇게 완성됩니다.

```jsx
import express from 'express';

export const userRouter = express.Router();

// GET /users - 모든 사용자 조회
userRouter.get('/', (req, res) => {
  res.json({ users: [] });
});

// GET /users/:id - 특정 사용자 조회
userRouter.get('/:id', (req, res) => {
  const { id } = req.params;
  res.json({ userId: id });
});

// POST /users - 새 사용자 생성
userRouter.post('/', (req, res) => {
  const { name, email } = req.body;
  res.json({ message: '사용자 생성됨', name, email });
});

// PUT /users/:id - 사용자 정보 업데이트
userRouter.put('/:id', (req, res) => {
  res.json({ message: `사용자 ${req.params.id} 업데이트` });
});

// DELETE /users/:id - 사용자 삭제
userRouter.delete('/:id', (req, res) => {
  res.json({ message: `사용자 ${req.params.id} 삭제` });
});

// GET /users/:userId/posts/:postId - 중첩 리소스
userRouter.get('/:userId/posts/:postId', (req, res) => {
  const { userId, postId } = req.params;
  res.json({ userId, postId });
});

```

---

### 2. routes/search.js

검색 기능처럼 단일 기능은 이렇게 파일 하나로 분리합니다.

`req.query`는 문자열로 들어오니까 `limit`을 `Number(limit)`로 바꿔주는 흐름이 자연스럽습니다.

```jsx
import express from 'express';

export const searchRouter = express.Router();

// GET /search - 검색 기능
searchRouter.get('/', (req, res) => {
  const { q, limit = 10 } = req.query;
  res.json({ query: q, limit: Number(limit) });
});

```

---

### 3. routes/index.js

이 파일은 “라우트들을 한 번에 모아서 내보내는 역할”입니다.

여기에서 중요한 포인트는 `router.use('/users', userRouter);` 같은 코드가 **기준 경로를 붙여주는 작업**이라는 점입니다.

```jsx
import express from 'express';
import { userRouter } from './users.js';
import { searchRouter } from './search.js';

export const router = express.Router();

// 기본 라우트
router.get('/', (req, res) => {
  res.json({
    message: 'Hello Express!',
    timestamp: new Date().toISOString(),
  });
});

router.use('/users', userRouter);
router.use('/search', searchRouter);

```

즉, 여기서 라우트 경로가 합쳐져서 최종 경로가 이렇게 되는 구조입니다.

- `router.use('/users', userRouter)` + `userRouter.get('/')` → `GET /users`
- `router.use('/search', searchRouter)` + `searchRouter.get('/')` → `GET /search`

---

### 4. server.js

마지막으로 server.js는 “서버 설정 + 라우트 등록 + 서버 실행”만 담당합니다.

1. `app.use(express.json())`은 **라우트보다 먼저 등록**되어야 합니다.
    
    그래야 POST 요청에서 `req.body`를 읽을 수 있습니다.
    
2. `app.use('/', router)`는 “router를 실제 서버에 연결”하는 단계입니다.
    
    이걸 해야 routes/index.js에서 만든 router가 진짜로 동작합니다.
    

```jsx
import express from 'express';
import { router } from './routes/index.js';

const app = express();
const PORT = 5001;

// JSON 파싱 미들웨어
app.use(express.json());

// 모든 라우트 등록
app.use('/', router);

// 서버 시작
app.listen(PORT, () => {
  console.log(`🚀 Server running on http://localhost:${PORT}`);
});

```

### 2. CRUD 라우트 구현

CRUD는 서버에서 가장 기본이 되는 작업 4가지입니다.

- **Create**(생성) → POST
- **Read**(조회) → GET
- **Update**(수정) → PATCH(또는 PUT)
- **Delete**(삭제) → DELETE

그리고 지금 예제는 응답을 통일하려고 이런 형태를 쓰고 있어요.

- 성공 여부: `success`
- 데이터: `data`
- 개수: `count`
- 메시지: `message`

---

```jsx
// 모든 사용자 조회
userRouter.get('/users', (req, res) => {
  res.json({
    success: true,
    data: users,
    count: users.length
  });
});
```

여기서 중요한 건 `count: users.length`입니다.

프론트엔드가 “총 몇 명인지”를 바로 알 수 있어서, 목록 UI나 페이지네이션 만들 때 유용해요.

---

```jsx
// 특정 사용자 조회
userRouter.get('/users/:id', (req, res) => {
  const id = parseInt(req.params.id);
  const user = users.find(u => u.id === id);

  if (!user) {
    return res.status(404).json({
      success: false,
      message: '사용자를 찾을 수 없습니다'
    });
  }

  res.json({
    success: true,
    data: user
  });
});

```

여기 핵심은 3가지입니다.

- `req.params.id`는 문자열이라서 `parseInt`로 숫자로 바꿉니다.
    
    (users의 id가 숫자이기 때문에 타입 맞추는 게 중요)
    
- `find`는 “조건에 맞는 첫 번째 요소”를 찾아줍니다.
- 사용자가 없으면 `404 Not Found`로 응답하는 게 REST API에서 자연스러운 흐름입니다.

또 중요한 습관 하나: `return res.status(...).json(...)`

이렇게 `return`을 붙여야 아래 코드가 더 실행되지 않아서 “중복 응답”을 막을 수 있습니다.

```jsx
// 사용자 생성
userRouter.post('/users', (req, res) => {
  const { name, email } = req.body;

  if (!name || !email) {
    return res.status(400).json({
      success: false,
      message: '이름과 이메일은 필수입니다'
    });
  }

  const newUser = {
    id: nextId++,
    name,
    email
  };

  users.push(newUser);

  res.status(201).json({
    success: true,
    data: newUser,
    message: '사용자가 생성되었습니다'
  });
});
```

여기 핵심은 4가지입니다.

- `req.body`에서 `name`, `email`을 꺼내려면 `express.json()` 미들웨어가 먼저 등록되어 있어야 합니다.
- `name`이나 `email`이 없으면 `400 Bad Request`로 처리합니다.
    
    “요청 자체가 잘못됐다”는 뜻이에요.
    
- `id: nextId++`로 “새로운 id를 만들고 다음 id로 넘어갑니다.”
- 생성은 보통 `201 Created`가 표준이라서 `res.status(201)`을 사용합니다.

```jsx
// 사용자 수정
userRouter.patch('/users/:id', (req, res) => {
  const id = parseInt(req.params.id);
  const { name, email } = req.body;

    return res.status(404).json({

	users[userIndex] = {
	  ...users[userIndex],
	  ...(name !== undefined ? { name } : {}),

  });

```

여기 핵심은 3가지입니다.

- `PATCH`는 “일부 수정”에 자주 씁니다. (name만 바꿀 수도, email만 바꿀 수도)
- `findIndex`는 “인덱스”를 찾기 때문에 배열에서 직접 수정할 때 편합니다.
- `{ ...users[userIndex], ... }`은 기존 값을 복사한 뒤 새 값을 덮어쓰는 방식입니다.

```jsx
// 사용자 삭제
userRouter.delete('/users/:id', (req, res) => {
```

삭제 응답은 보통 두 가지 방식이 있습니다.

- 지금처럼 JSON 메시지를 보내거나
- `204 No Content`로 본문 없이 보내는 방식

## 06-express-middleware

### 미들웨어 기본 개념

미들웨어는 **요청과 응답 사이에서 실행되는 함수**입니다.

요청이 들어오면 라우트 핸들러로 바로 가는 게 아니라, 중간에 등록된 미들웨어들을 순서대로 거칠 수 있습니다.

```jsx
// 기본 구조
const myMiddleware = (req, res, next) => {
  console.log('미들웨어 실행됨');
  req.customData = 'Hello'; // 요청 객체에 데이터 추가
  next(); // 다음 미들웨어로 이동
};

app.use(myMiddleware);
```

- `req`, `res`는 Express가 매 요청마다 만들어주는 요청/응답 객체입니다.
- `next()`를 호출해야 다음 미들웨어(또는 라우트)로 넘어갑니다.
- `next()`를 호출하지 않으면 요청이 거기서 멈춰서 “로딩만 도는 상태”가 될 수 있습니다.

---

### 주요 미들웨어 종류

### 1. 내장 미들웨어

```jsx
// JSON 파싱
app.use(express.json());

// URL 인코딩 파싱
app.use(express.urlencoded({ extended: true })); // qs 사용

// 정적 파일 제공
app.use(express.static('public')); // 최상위에 public 폴더 생성
```

- `express.json()`
    
    `Content-Type: application/json` 요청의 body를 파싱해서 `req.body`에 넣어줍니다.
    
- `express.urlencoded(...)`
    
    HTML form 전송처럼 `application/x-www-form-urlencoded` 형태의 body를 파싱합니다.
    
    `extended: true`면 내부적으로 더 풍부한 파싱(중첩 객체 등)을 지원하는 방식(보통 qs 기반)으로 동작합니다.
    
- `express.static('public')`
    
    `public` 폴더 안 파일을 URL로 그대로 제공하는 기능입니다.
    
    예를 들어 `public/logo.png`가 있으면 `/logo.png`로 접근됩니다.
    

---

### 2. 커스텀 미들웨어

```jsx
// 로깅 미들웨어
const logger = (req, res, next) => {
  console.log(`${req.method} ${req.url} - ${new Date().toISOString()}`);
  next();
};

// 인증 미들웨어
const auth = (req, res, next) => {
  const token = req.headers.authorization;

  if (!token) {
    return res.status(401).json({ error: '인증 필요' });
  }

  req.user = { id: 1, name: 'User' };
  next();
};

// 사용
app.use(logger);
app.get('/protected', auth, (req, res) => {
  res.json({ user: req.user });
});

```

- 로깅 미들웨어는 “모든 요청”에 공통으로 적용하기 좋습니다. (`app.use(logger)`)
- 인증 미들웨어는 “특정 라우트만 보호”하고 싶을 때 라우트에 끼워 넣습니다. (`app.get('/protected', auth, ...)`)
- `return res.status(...).json(...)`처럼 `return`을 붙여서 **응답 후 흐름을 종료**하는 습관이 중요합니다.

```jsx
// middlewares/requestTimer.js
// 요청 처리 시간 측정
const requestTimer = (req, res, next) => {
```

### 단계 4: CORS 미들웨어 ⭐⭐⭐

```jsx
// middlewares/cors.js
// CORS 미들웨어
export const cors = (req, res, next) => {
	const origin = req.headers.origin;
  const isProduction = process.env.NODE_ENV === "production";
  
	const whiteList = [
    'https://your-production-site.com',
    'https://admin.your-site.com'
  ];

  const isAllowed = !isProduction || (origin && whiteList.includes(origin));

	if (isAllowed && origin) {
    res.header("Access-Control-Allow-Origin", origin);
    res.header("Access-Control-Allow-Credentials", "true");
  } else if (!isProduction) {
    // 개발 환경인데 Origin 헤더가 없는 경우(Postman 등)를 위해 최소한의 허용
    res.header("Access-Control-Allow-Origin", "*");
  }
  
  // 공통 헤더 설정
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, PATCH, DELETE, OPTIONS');
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  
  // Preflight(사전 요청) 처리
  if (req.method === 'OPTIONS') {
    return res.sendStatus(200);
  }

  next();
};

// server.js
app.use(cors);
```

### 설명

- **CORS(Cross-Origin Resource Sharing)**는 브라우저가 가진 보안 규칙인 **SOP(Same-Origin Policy, 동일 출처 정책)** 때문에 생기는 개념입니다.

브라우저는 기본적으로 **출처(Origin)**가 다른 곳으로 보내는 요청을 막습니다.

여기서 출처(Origin)는 보통 아래 3개 조합이에요.

- 프로토콜: `http` / `https`
- 도메인: `example.com`
- 포트: `3000`, `5001` 등

예를 들어,

- 프론트: `http://localhost:5173`
- 백엔드: `http://localhost:5001`

이 둘은 포트가 다르니까 **출처가 다르고**, 브라우저는 기본적으로 이런 요청을 “위험할 수 있다”고 보고 막습니다.

그래서 서버가 응답 헤더로

- “이 출처는 우리 API를 호출해도 된다”

라는 허용 정보를 내려주면, 브라우저가 그제서야 요청을 통과시켜 줍니다.

그 “허용 정보를 주고받는 규칙”이 바로 CORS입니다.

즉, CORS는 “서버 보안 기능”이라기보다는

- **브라우저가 요청을 허용/차단할 때 참고하는 약속(응답 헤더 규칙)**

이라고 이해하면 됩니다.

그리고 이 미들웨어는 “다른 출처(Origin)에서 우리 서버 API를 호출할 수 있게 허용할지”를 결정합니다.

브라우저는 기본적으로 보안 때문에 **다른 Origin 요청을 막는데**, 서버가 CORS 헤더로 “허용”을 선언하면 브라우저가 요청을 통과시켜 줍니다.

---

### 1) `origin`을 가져오는 이유

```jsx
const origin = req.headers.origin;
```

브라우저에서 요청이 오면 보통 `Origin` 헤더가 같이 옵니다.

- 예: `https://admin.your-site.com`에서 API 요청 → `Origin: https://admin.your-site.com`

서버는 이 값을 보고 “허용할 사이트인지”를 판단합니다.

> 참고: Postman이나 서버-서버 요청은 Origin 헤더가 없을 수도 있습니다.
> 

---

### 2) 운영/개발 환경을 나누는 이유

```jsx
const isProduction = process.env.NODE_ENV ==="production";
```

개발 환경에서는 로컬 테스트가 많아서 보통 CORS를 널널하게 열어둡니다.

반대로 운영 환경에서는 **허용된 도메인만** 정확히 열어야 안전합니다.

---

### 3) 화이트리스트(허용 도메인 목록)

```jsx
const whiteList = [
	'https://your-production-site.com',
	'https://admin.your-site.com'
]
```

운영 환경에서 허용할 Origin 목록입니다.

- 서비스 도메인(유저용 프론트)
- 어드민 도메인(관리자 프론트)

이런 식으로 “내가 직접 운영하는 프론트만” 허용하는 구조가 일반적입니다.

---

### 4) 허용 여부 판단 로직

```jsx
const isAllowed = !isProduction || (origin && whiteList.includes(origin));
```

- 운영이 아니라면(`!isProduction`) → 일단 허용(개발 편의)
- 운영이라면 → `origin`이 존재하고 `whiteList`에 포함된 경우만 허용

이렇게 하면 운영 환경에서 외부 사이트가 마음대로 API 호출하는 걸 줄일 수 있습니다.

---

### 5) `Access-Control-Allow-Origin`을 이렇게 설정하는 이유

```jsx
if (isAllowed && origin) {
  res.header("Access-Control-Allow-Origin", origin);
  res.header("Access-Control-Allow-Credentials","true");
} else if (!isProduction) {
  res.header("Access-Control-Allow-Origin","*");
}
```

여기서 핵심은 두 가지입니다.

### (1) Credentials를 쓰려면 를 못 쓴다

`Access-Control-Allow-Credentials: true`는 쿠키/인증정보를 허용하겠다는 뜻입니다.

이때 `Access-Control-Allow-Origin: *`를 같이 쓰면 브라우저가 거부합니다.

그래서 credentials를 허용할 때는 반드시:

- `Access-Control-Allow-Origin: https://정확한-도메인`

처럼 **정확히 한 Origin을 내려줘야** 합니다.

### (2) 개발 환경에서 Origin이 없을 수도 있다

Postman처럼 Origin이 없는 요청도 편하게 테스트하려고 개발 환경에서는 `*`로 최소 허용을 해 둔 겁니다.

---

### 6) 공통 헤더 설정

```jsx
res.header('Access-Control-Allow-Methods','GET, POST, PUT, PATCH, DELETE, OPTIONS');
res.header('Access-Control-Allow-Headers','Content-Type, Authorization');
```

- 어떤 HTTP 메서드를 허용할지
- 어떤 헤더를 허용할지

를 브라우저에게 알려줍니다.

특히 `Authorization` 헤더는 토큰 인증에서 자주 쓰기 때문에 미리 허용해두는 게 일반적입니다.

---

### 7) OPTIONS(Preflight) 요청을 빠르게 처리하는 이유

```jsx
if (req.method ==='OPTIONS') {
	return res.sendStatus(200);
}
```

브라우저는 실제 요청을 보내기 전에 “이 요청 보내도 돼?”를 묻는 사전 요청(Preflight)을 보낼 수 있습니다.

그게 OPTIONS 요청입니다.

이 요청은 “바디가 필요한 요청”이 아니라 “허용 여부 확인”이라서, 빠르게 200으로 끝내는 게 좋습니다.

## 07-handle-error

### 기본 에러 처리

```jsx
// 동기 에러 처리
app.get('/users/:id', (req, res) => {
  const users = [{ id: '1', name: 'John' }];
  const user = users.find((u) => u.id === req.params.id);

  if (!user) {
    return res.status(404).json({
      error: '사용자를 찾을 수 없습니다',
    });
  }

  res.json(user);
});

// 비동기 에러 처리 (async/await)
app.get('/data', async (req, res) => {
  try {
    const data = await fetchData();
    res.json(data);
  } catch (error) {
    res.status(500).json({
      error: '서버 오류가 발생했습니다',
      message: error.message,
    });
  }
});
```

여기서는 “에러 처리를 왜 하냐”부터 감 잡는 게 중요합니다.

- 서버는 항상 정상 데이터만 주는 게 아니라, **실패 상황도 응답으로 설명**해야 합니다.
- 그래서 성공이면 200번대, 요청이 잘못되면 400번대, 서버가 터지면 500번대를 내려줍니다.

동기 코드에서는 조건문으로 바로 `return res.status(...).json(...)` 처리하면 끝나는데,

비동기(async/await)에서는 중간에 터질 수 있으니 `try/catch`로 감싸는 패턴이 기본입니다.

---

### 커스텀 에러 클래스

```jsx
export class HttpException extends Error {
  statusCode;
  constructor(description, statusCode) {
    super(description);
    this.name = this.constructor.name;
    this.statusCode = statusCode;
  }
}

// 사용 예
app.get('/users/:id', async (req, res) => {
  try {
    const user = await findUser(req.params.id);

    if (!user) {
      throw new HttpException('사용자를 찾을 수 없습니다', 404);
    }

    res.json(user);
  } catch (error) {
    if (error instanceof HttpException) {
      return res.status(error.statusCode).json({
        error: error.message,
      });
    }

    res.status(500).json({
      error: '서버 오류',
    });
  }
});
```

이 방식의 핵심은 “에러를 그냥 문자열로 처리하지 말고, 상태코드까지 같이 들고 다니게 만들자”입니다.

- `throw new HttpException(..., 404)`처럼 던지면
- catch에서 “이건 내가 의도한 에러인지” 판단해서
- 상태코드를 통일된 방식으로 내려줄 수 있습니다.

다만 라우트마다 이런 `try/catch`가 반복되면 코드가 길어집니다.

그래서 다음 단계가 “에러 처리 미들웨어로 몰아넣기”입니다.

### 실습해보자!

### 1. errors/httpException.js

이 코드는 “에러도 하나의 객체로 만들고, **HTTP 상태코드까지 같이 들고 다니게** 하자”는 목적의 **커스텀 에러 클래스**예요.

```jsx
export class HttpException extends Error {
  statusCode;
  constructor(description, statusCode) {
    super(description);
    this.name = this.constructor.name;
    this.statusCode = statusCode;
  }
}
```

### 1) 왜 굳이 클래스로 만들까?

원래 JS에서 에러는 보통 이렇게 던지죠.

```jsx
throw new Error('사용자를 찾을 수 없습니다');
```

근데 이건 **메시지밖에 없습니다.**

즉, 이 에러가 “404인지 400인지 401인지”를 에러 자체만 보고는 알기 어려워요.

그래서 `HttpException`은 에러 객체 안에 `statusCode`를 추가해서,

- 에러가 발생한 곳에서는 `throw new HttpException('...', 404)`로 던지고
- 에러 핸들러에서는 `error.statusCode`로 상태코드를 꺼내서 응답하는

일관된 흐름을 만들 수 있습니다.

---

### 2) `extends Error`가 의미하는 것

`extends Error`는 “**기본 Error를 상속받아 확장한다**”는 뜻입니다.

그래서 이 클래스의 인스턴스는:

- `instanceof Error`도 true
- 기본 Error가 제공하는 `message`, `stack`도 그대로 가짐

즉, “에러로서의 성질”은 유지하면서 기능만 추가하는 거예요.

---

### 3) `statusCode;` 이 줄은 뭐야?

이건 클래스 필드 선언입니다.

“이 클래스에는 statusCode라는 속성이 있을 거야”라는 선언이고, 실제 값은 constructor에서 넣습니다

---

### 4) constructor에서 하는 일

```jsx
constructor(description, statusCode) {
	super(description);
	this.name =this.constructor.name;
	this.statusCode = statusCode;
}
```

### `super(description)`

부모 클래스인 `Error`의 생성자를 호출합니다.

여기서 `description`이 Error의 `message`로 들어가요.

즉,

```jsx
throw new HttpException('사용자를 찾을 수 없습니다', 404);
```

하면,

- `error.message`는 `'사용자를 찾을 수 없습니다'`가 됩니다.

### `this.name = this.constructor.name`

에러 이름을 클래스 이름으로 맞춰주는 겁니다.

- 기본 Error면 `name`이 `'Error'`
- HttpException이면 `name`이 `'HttpException'`

이렇게 되면 로그 찍을 때 “어떤 에러 타입인지”가 더 선명해져요.

### `this.statusCode = statusCode`

이게 핵심입니다.

상태 코드를 에러 객체에 붙여서, 에러 핸들러가 그대로 응답에 쓸 수 있게 합니다.

### 7. middlewares/errorHandler.js

```jsx
import { isDevelopment } from '../config/config.js';
import { HttpException } from '../errors/httpException.js';

// next를 지우면 안됨!
export const errorHandler = (error, req, res, _next) => {
  console.error('error', error);
  if (error instanceof HttpException) {
    return res.status(error.statusCode).json({
      success: false,
      message: error.message,
    });
  }

	// 추가됨
  let result = {
    success: false,
    message: 'Internal Server Error',
  };
	
	// 개발 환경에서는 메시지를 상세하게 안내 for 디버깅
  if (isDevelopment) {
    result = { ...result, error };
  }

  return res.status(500).json(result);
};

```

여기서 핵심은 “에러 응답을 한 곳에서 통일하는 것”입니다.

- `HttpException`이면: 우리가 의도한 에러 → 상태코드 그대로 내려주기
- 그 외 에러면: 진짜 서버 에러 → 500으로 내려주기

그리고 개발 환경에서만 `error`를 같이 내려주는 이유는,

운영 환경에서 에러 객체를 그대로 내려주면 내부 정보가 노출될 수 있기 때문입니다.

### Express 에러 처리 미들웨어의 인자 규칙과 에러 발생 원리

- Express에서 에러를 처리하는 미들웨어는 반드시 4개의 인자 `(error, req, res, next)`
    
    를 가져야만 합니다. 이것은 Express가 "이 미들웨어는 에러 처리용이구나!"라고 인식하는 약속(규칙)입니다.
    

여기서 `next`를 실제로 쓰지 않더라도, 인자 자리가 있어야 Express가 에러 미들웨어로 인식합니다.

그래서 `_next`처럼 언더스코어로 받아서 “안 쓰지만 필요하다”를 표현합니다.

---

### ESlint 규칙을 수정하여 에러를 고쳐보자

```jsx
'no-unused-vars': ['warn', { argsIgnorePattern: '^_' }],
```

이 설정은 `_next`처럼 언더스코어로 시작하는 인자는 “안 쓰는 인자여도 경고하지 말아라”라는 의미입니다.

### validateUser

```jsx
export const validateUser = (req, res, next) => {
  const { name, email } = req.body;
  try {
    if (!name || name.trim().length < 2) {
      throw new BadRequestException('이름은 2글자 이상이어야 합니다');
    }

    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    if (!email || !emailRegex.test(email)) {
      throw new BadRequestException('유효한 이메일 형식이어야 합니다');
    }
  } catch (error) {
    next(error);
    return;
  }
  next();
};
```

### src/config/config.js

```jsx
import { flattenError, z } from 'zod';

const envSchema = z.object({
  NODE_ENV: z
    .enum(['development', 'production', 'test'])
    .default('development'),
  PORT: z.coerce.number().min(1000).max(65535).default(5001),
});

const parseEnvironment = () => {
  try {
    return envSchema.parse({
      NODE_ENV: process.env.NODE_ENV,
      PORT: process.env.PORT,
    });
  } catch (error) {
    if (error instanceof z.ZodError) {
      const flatten = flattenError(error);
      console.error('환경 변수 검증 실패:', flatten);
    }
    process.exit(1);
  }
};

export const config = parseEnvironment();

// 환경별 헬퍼 함수들
export const isDevelopment = config.NODE_ENV === 'development'; // boolean
export const isProduction = config.NODE_ENV === 'production'; // boolean
export const isTest = config.NODE_ENV === 'test'; // boolean

```

### 핵심 동작 흐름

- `envSchema`에서 **필요한 환경변수 키 + 타입 + 제약조건**을 정의합니다.
- `parseEnvironment()`에서 실제 `process.env` 값을 가져와 **검증/변환**합니다.
    - `PORT`는 `z.coerce.number()`라서 `"5001"` 같은 문자열도 숫자로 변환해서 검사합니다.
    - 검증 실패하면 에러 로그 찍고 `process.exit(1)`로 **서버를 바로 종료**합니다. (환경변수 깨진 상태로 서버 켜지지 않게)

## 09-mongodb

### 1. 등장 배경 및 이유 (2000년대 후반)

MongoDB가 등장하기 전, 데이터베이스 시장은 관계형 데이터베이스(RDBMS, Oracle, MySQL, PostgreSQL 등)가 절대적이었습니다. 하지만 **웹 2.0** 이후 IT 환경이 급변하면서 기존 RDBMS의 한계가 드러났습니다.

**당시 RDBMS의 한계점**

1. **엄격하고 경직된 스키마 (Rigid Schema)**
    - RDBMS는 테이블/컬럼/데이터 타입을 미리 정의해야 하는 **고정된 스키마**를 가짐
    - 문제점: 빠르게 변화하는 웹 서비스(소셜 미디어, 블로그 등)에서 데이터 구조 변경 시 **복잡하고 비용 큰 마이그레이션** 발생 → 개발 속도 저하
2. **객체-관계 불일치 (Object-Relational Impedance Mismatch)**
    - 애플리케이션은 객체 지향 언어(Java, Python, JS 등)를 사용하지만, RDBMS는 데이터를 **행(Row)** 기반으로 저장
    - 복잡한 중첩 객체(예: 사용자 안에 게시물 배열)를 RDBMS에 저장하려면 **여러 테이블 분해 + JOIN 연산** 필요
    - 이로 인해 **ORM 기술 복잡성** 증가
3. **수평적 확장의 어려움 (Scalability)**
    - 웹 서비스 확장으로 쓰기 트래픽 및 데이터 폭증
    - RDBMS는 **수직적 확장(Scale-up)**에는 유리하지만, **수평적 확장(Sharding)**은 구조적으로 매우 어려움
    
    ---
    
    ### MongoDB의 해결책: NoSQL의 대표주자
    
    1. **유연한 문서(Document) 모델**
        - JSON과 유사한 BSON(Binary JSON) 문서 저장 방식
        - 애플리케이션 객체 구조와 동일 → 객체-관계 불일치 문제 자연스럽게 해결
    2. **동적 스키마 (Dynamic Schema)**
        - 스키마 사전 정의 불필요
        - 필요할 때 필드를 자유롭게 추가/삭제 가능 → 애자일 개발과 빠른 프로토타이핑에 최적
    3. **수평적 확장을 위한 설계**
        - 샤딩(Sharding)을 염두에 두고 설계
        - 대용량 데이터 자동 분산 저장 및 처리 지원 → 빅데이터 시대에 적합

### 4. 코드에 MongoDB 적용하기

### 0단계. mongoose install

- 데이터의 안정성, 코드의 간결함, 개발 생산성 향상

```jsx
npm install mongoose
```

- `mongoose`는 MongoDB를 JS 코드로 다루기 쉽게 해주는 ODM입니다.
- 지금 단계의 목표는 “메모리 배열(users)” 대신 “MongoDB 컬렉션(User)”로 CRUD를 바꾸는 거예요.

### 1단계. 환경 변수 설정 및 구성 관리 (.env, config)

- `.env.development` 파일에 MongoDB 연결 정보 등 환경 변수 정의 (.env.example도 수정)

```json
# .env.development
NODE_ENV=development
PORT=5001
MONGO_URI=mongodb+srv://<username>:<password>@<cluster>/<database>?<options>
```

여기 체크할 부분이 있습니다.

### 2단계. 데이터베이스 연결 모듈화 및 안전성 강화

- 연결 상태 모니터링과 안전한 종료(Graceful Shutdown) 처리

`db/index.js`

```jsx
import mongoose from 'mongoose';
import { config } from '../config/config.js';

export const connectDB = async () => {
  try {
    await mongoose.connect(config.MONGO_URI);
    console.log('✅ MongoDB connected successfully.');
  } catch (err) {
    console.error('❌ MongoDB connection error:', err);
    process.exit(1);
  }
};

// 애플리케이션 종료 시 DB 연결 해제
export const disconnectDB = async () => {
  await mongoose.connection.close();
}
```

`src/server.js`

```jsx
...

const app = express();
await connectDB(); // DB 연결
...
```

---

`src/server.js`

```jsx
const server = app.listen(config.PORT, () => {
  console.log(`🚀 Server running on http://localhost:${config.PORT}`);
});

// 2. 안전한 종료(Graceful Shutdown) 로직을 추가합니다.
const shutdown = (signal) => {
  console.log(`\n${signal} received. Shutting down gracefully...`);
  // a. 새로운 요청을 더 이상 받지 않도록 서버를 닫습니다.
  server.close(async () => {
    console.log('HTTP server closed.');
    // b. 서버가 닫히면, DB 연결을 해제합니다.
    await disconnectDB();
  });
};

// 3. SIGINT, SIGTERM 신호를 감지하여 shutdown 함수를 실행합니다.
process.on('SIGINT', () => shutdown('SIGINT'));
process.on('SIGTERM', () => shutdown('SIGTERM'));
```

### 3단계: User 스키마 및 모델 정의

`id`, `name`, `email` 구조에 맞춘 Mongoose 스키마 정의. MongoDB `_id`를 `id`처럼 활용하는 가상 필드 적용.

```jsx
import mongoose from 'mongoose';

const userSchema = new mongoose.Schema(
  {
    name: { type: String, required: true },
    email: { type: String, required: true, unique: true }
  },
  {
    timestamps: true,
    toJSON: { virtuals: true },
    toObject: { virtuals: true },
    versionKey: false, // __v 필드를 제거
  }
);

// 옛날 mongoose 버전, 최신은 자동 지원
```

그리고 “id 가상 필드” 관련해서는 이렇게 이해하면 됩니다.

- 프론트/클라이언트는 `id`를 더 선호하는 경우가 많아서 `id`를 만들어주는 패턴을 씁니다.

### 4단계: Route에 MongoDB 로직 직접 적용하기

`users.js` 라우트 파일에서 Mongoose `User` 모델로 바로 DB CRUD를 수행하도록 전체 수정.

- 기존 users, nextId는 삭제

`src/routes/users.js`

```jsx
import express from 'express';
import { validateUser } from '../middlewares/validateUser.js';
import { NotFoundException } from '../errors/notFoundException.js';
import { User } from '../models/user.model.js';
import { ConflictException } from '../errors/conflictException.js';

export const userRouter = express.Router();

// GET /users - 모든 사용자 조회
userRouter.get('/', async (req, res, next) => {
  try {
    const users = await User.find();
    res.json({ success: true, data: users, count: users.length });
  } catch (error) {
    next(error);
    return;
  }
});

// GET /users/:id - 특정 사용자 조회
userRouter.get('/:id', async (req, res, next) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) throw new NotFoundException('사용자를 찾을 수 없습니다.');
    res.json({ success: true, data: user });
  } catch (error) {
    next(error);
    return;
  }
});

// POST /users - 새 사용자 생성
userRouter.post('/', validateUser, async (req, res, next) => {
  try {
    const { name, email } = req.body;

    const existingUser = await User.findOne({ email });
    const isDuplicatedEmail = !!existingUser;

    if (isDuplicatedEmail) {
      throw new ConflictException('중복된 이메일 입니다.');
    }

    const newUser = new User({ name, email });
    await newUser.save();
    res.status(201).json({
      success: true,
      data: newUser,
      message: '사용자가 생성되었습니다',
    });
  } catch (error) {
    next(error);
    return;
  }
});

// PUT /users/:id - 사용자 정보 업데이트
userRouter.patch('/:id', validateUser, async (req, res, next) => {
  try {
    const { name, email } = req.body;
    const { id: userId } = req.params;

    const existingUser = await User.findOne({ email, _id: { $ne: userId } });
    const isDuplicatedEmail = !!existingUser;

    if (isDuplicatedEmail) {
      throw new ConflictException('중복된 이메일 입니다.');
    }

    const updatedUser = await User.findByIdAndUpdate(
      req.params.id,
      { name, email },
      { new: true },
    );

    if (!updatedUser) {
      throw new NotFoundException('사용자를 찾을 수 없습니다.');
    }

    res.json({
      success: true,
      data: updatedUser,
      message: '사용자가 수정되었습니다',
    });
  } catch (error) {
    next(error);
    return;
  }
});

// DELETE /users/:id - 사용자 삭제
userRouter.delete('/:id', async (req, res, next) => {
  try {
    const deletedUser = await User.findByIdAndDelete(req.params.id);

    if (!deletedUser) {
      throw new NotFoundException('사용자를 찾을 수 없습니다.');
    }

    res.json({ success: true, message: '사용자가 삭제되었습니다' });
  } catch (error) {
    next(error);
    return;
  }
});

// GET /users/:userId/posts/:postId - 중첩 리소스
userRouter.get('/:userId/posts/:postId', (req, res) => {
  const { userId, postId } = req.params;
  res.json({ userId, postId });
});
```

여기서 “중요한 흐름”만 짚으면 이렇습니다.

- 조회: `User.find()`, `User.findById()`
- 생성: 중복 이메일 검사 → `new User(...)` → `save()`
- 수정: 중복 이메일 검사(자기 자신 제외 `$ne`) → `findByIdAndUpdate(..., { new: true })`
- 삭제: `findByIdAndDelete()`
- 에러는 전부 `throw` → `catch`에서 `next(error)`로 넘겨서 errorHandler가 응답 통일
