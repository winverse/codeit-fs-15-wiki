# Day 19. 클래스와 프로토타입

### 오늘의 목표

JavaScript의 `class`는 완전히 새로운 객체 모델이 아니라 프로토타입 기반 객체 모델 위에
올라간 문법입니다. 오늘은 클래스 생성자, 프로토타입 메서드, 프로토타입 체인, static 멤버,
`#private` 필드를 복습하고, TypeScript class 문법이 런타임 동작과 어떻게 연결되는지
확인합니다.

### 레퍼런스 연결

- 클래스와 프로토타입 챕터 복습
- 핵심 복습 포인트: prototype chain, class 문법, method, static, private 필드
- 다시 볼 질문: TS class 문법은 JS prototype 동작을 얼마나 가려줄까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
class User {
  constructor(name) {
    this.name = name;
  }

  sayName() {
    return this.name;
  }
}

const ada = new User("Ada");

console.log(typeof User);
console.log(ada.sayName());
console.log(Object.prototype.hasOwnProperty.call(ada, "sayName"));
console.log(Object.prototype.hasOwnProperty.call(User.prototype, "sayName"));
console.log(ada instanceof User);
```

- 정답과 해설
    
    ### 정답
    
    ```
    function
    Ada
    false
    true
    true
    ```
    
    ### 해설
    
    JavaScript의 `class`는 문법적으로는 클래스처럼 보이지만 런타임에서 `typeof User`는
    `"function"`입니다. 클래스도 생성자 함수의 형태로 동작한다는 점을 보여줍니다.
    
    `new User("Ada")`는 새 인스턴스를 만들고, `constructor` 내부의 `this.name = name`으로
    인스턴스에 `name` 속성을 추가합니다. 그래서 `ada.sayName()`은 `"Ada"`입니다.
    
    `sayName`은 인스턴스 자신의 속성이 아닙니다. 클래스 본문에 작성한 메서드는
    `User.prototype`에 들어갑니다. 따라서 `Object.prototype.hasOwnProperty.call(ada, "sayName")`은
    `false`입니다.
    
    반대로 `Object.prototype.hasOwnProperty.call(User.prototype, "sayName")`은 `true`입니다.
    메서드가 프로토타입에 있기 때문입니다.
    
    `ada instanceof User`는 `true`입니다. `ada`의 프로토타입 체인에 `User.prototype`이 연결되어
    있기 때문입니다.
    
    정리하면 클래스 메서드는 인스턴스마다 새로 복사되는 것이 아니라 프로토타입을 통해 공유됩니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 프로토타입 체인을 통해 속성을 찾는 과정을
설명하세요.

```jsx
const base = {
  kind: "base",
  describe() {
    return this.kind;
  },
};

const child = Object.create(base);
child.kind = "child";

console.log(child.describe());
console.log(Object.prototype.hasOwnProperty.call(child, "describe"));
console.log("describe" in child);

delete child.kind;

console.log(child.describe());
```

- 정답과 해설
    
    ### 정답
    
    ```
    child
    false
    true
    base
    ```
    
    ### 해설
    
    `Object.create(base)`는 `base`를 프로토타입으로 갖는 새 객체를 만듭니다. `child`에는
    처음에는 자신의 속성이 거의 없고, 속성을 찾을 때 `base`까지 올라갈 수 있습니다.
    
    `child.kind = "child"`는 `child` 자신의 속성을 만듭니다. `child.describe()`를 호출하면
    `describe` 메서드는 `child` 자신에게 없으므로 프로토타입인 `base`에서 찾습니다.
    
    하지만 메서드를 찾은 위치와 `this`는 별개입니다. `child.describe()`처럼 호출했으므로
    `describe` 내부의 `this`는 `child`입니다. 따라서 `this.kind`는 먼저 `child.kind`를 찾고
    `"child"`를 반환합니다.
    
    `Object.prototype.hasOwnProperty.call(child, "describe")`는 `false`입니다. `describe`는
    `child` 자신의 속성이 아니라 `base`에 있는 속성입니다.
    
    `"describe" in child`는 `true`입니다. `in` 연산자는 자신의 속성뿐 아니라 프로토타입 체인의
    속성도 확인합니다.
    
    `delete child.kind` 이후에는 `child` 자신의 `kind`가 사라집니다. 다시 `child.describe()`를
    호출하면 `this`는 여전히 `child`이지만, `this.kind`를 찾을 때 `child`에 없으므로 프로토타입
    `base.kind`를 찾습니다. 그래서 `"base"`가 출력됩니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, static 멤버와 instance 멤버의 차이를 설명하세요.

```jsx
class Counter {
  static created = 0;

  constructor() {
    Counter.created += 1;
    this.value = 0;
  }

  increase() {
    this.value += 1;
    return this.value;
  }

  static reset() {
    Counter.created = 0;
  }
}

const c1 = new Counter();
const c2 = new Counter();

console.log(Counter.created);
console.log(c1.increase());
console.log(typeof c1.reset);
console.log(typeof Counter.reset);
```

- 정답과 해설
    
    ### 정답
    
    ```
    2
    1
    undefined
    function
    ```
    
    ### 해설
    
    `static created = 0`은 클래스 자체의 속성입니다. 인스턴스가 아니라 `Counter.created`로
    접근합니다.
    
    `new Counter()`가 호출될 때마다 생성자에서 `Counter.created += 1`이 실행됩니다. `c1`과
    `c2`를 만들었으므로 `Counter.created`는 `2`입니다.
    
    `increase`는 인스턴스 메서드입니다. `c1.increase()`는 `c1.value`를 `0`에서 `1`로 증가시키고
    `1`을 반환합니다.
    
    `reset`은 static 메서드입니다. static 메서드는 인스턴스가 아니라 클래스 생성자에 붙습니다.
    그래서 `typeof c1.reset`은 `"undefined"`입니다.
    
    반면 `typeof Counter.reset`은 `"function"`입니다.
    
    static 멤버는 인스턴스 개별 상태보다 클래스 전체와 관련된 동작에 어울립니다. 예를 들어
    팩토리 메서드, 클래스 수준 설정, 인스턴스 생성 통계 같은 경우입니다.
    

### 문제 4. 심화

다음 코드는 계좌의 잔액을 클래스 내부에 숨기려고 합니다. 실행 결과를 예측하고, 이 코드의
의미를 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
class Account {
  #balance = 0;

  deposit(amount) {
    this.#balance += amount;
    return this.#balance;
  }

  getBalance() {
    return this.#balance;
  }
}

const account = new Account();

console.log(account.deposit(100));
console.log(account.getBalance());
console.log(Object.keys(account).length);
console.log("balance" in account);

try {
  console.log(eval("account.#balance"));
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    100
    100
    0
    false
    SyntaxError
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    class Account {
      #balance = 0;
    
      deposit(amount: number): number {
        if (!Number.isFinite(amount) || amount <= 0) {
          throw new Error("amount must be a positive finite number.");
        }
    
        this.#balance += amount;
        return this.#balance;
      }
    
      getBalance(): number {
        return this.#balance;
      }
    }
    
    const account = new Account();
    
    console.log(account.deposit(100));
    console.log(account.getBalance());
    console.log("balance" in account);
    ```
    
    ### 해설
    
    ### JS 관점
    
    `#balance`는 JavaScript private field입니다. 이름이 `balance`인 일반 속성이 아니라, 클래스
    본문 안에서만 접근 가능한 private 슬롯입니다.
    
    `account.deposit(100)`은 클래스 내부 메서드에서 `this.#balance`에 접근합니다. 처음 값은
    `0`이고 `100`을 더하므로 `100`이 반환됩니다.
    
    `account.getBalance()`도 클래스 내부 메서드이므로 private field를 읽을 수 있습니다. 결과는
    `100`입니다.
    
    `Object.keys(account).length`는 `0`입니다. `#balance`는 enumerable한 일반 객체 속성이
    아니고, 예제에는 다른 public own property도 없습니다.
    
    `"balance" in account`는 `false`입니다. `#balance`는 문자열 `"balance"` 키의 속성이
    아닙니다.
    
    `account.#balance`는 클래스 본문 밖에서 문법적으로 접근할 수 없습니다. 직접 작성하면 코드
    파싱 단계에서 실패하므로, 예제에서는 `eval("account.#balance")`로 에러를 잡았습니다. 결과는
    `SyntaxError`입니다.
    
    ### TS 관점
    
    TypeScript의 `private` 키워드는 오래전부터 있었지만, 기본적으로는 TypeScript 타입 검사
    단계의 접근 제한입니다. 반면 JavaScript의 `#private` 필드는 런타임 문법 자체가 외부 접근을
    막습니다.
    
    개선 예시에서는 `#balance`를 사용해 런타임에서도 잔액을 외부에서 직접 읽거나 쓸 수 없게
    했습니다. 또한 `deposit(amount: number)`로 입력 타입을 명시했습니다.
    
    하지만 `number` 타입만으로는 `NaN`, `Infinity`, 음수 같은 도메인에 맞지 않는 값을 막을 수
    없습니다. 그래서 `Number.isFinite(amount)`와 `amount <= 0` 검사를 추가했습니다.
    
    이 예시를 TypeScript로 컴파일할 때는 private field를 지원하는 target이 필요합니다. 검증은
    `--target es2022 --lib es2022,dom` 기준으로 수행합니다.
    
    ### 실무 관점
    
    클래스는 상태와 동작을 묶는 데 유용하지만, 모든 상태가 자동으로 안전해지는 것은 아닙니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 외부에서 직접 수정하면 안 되는 상태는 `#private` 필드로 숨긴다.
    2. 상태 변경은 public 메서드에서 검증 후 수행한다.
    3. TypeScript 접근 제한과 JavaScript 런타임 private field의 차이를 이해한다.
    4. 클래스가 필요한 문제인지, 단순 함수와 객체 조합이 더 나은 문제인지 판단한다.
    
    클래스 문법은 프로토타입을 감추는 문법처럼 보이지만, 런타임 객체 모델은 여전히 프로토타입과
    속성 접근 규칙 위에서 동작합니다.
