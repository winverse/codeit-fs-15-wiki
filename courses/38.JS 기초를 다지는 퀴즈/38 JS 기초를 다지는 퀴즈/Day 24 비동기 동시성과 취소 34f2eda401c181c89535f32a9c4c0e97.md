# Day 24. 비동기 동시성과 취소

### 오늘의 목표

여러 비동기 작업을 동시에 다룰 때는 “모두 성공해야 하는가”, “가장 빠른 결과만 필요한가”,
“실패를 어떻게 처리할 것인가”, “취소를 지원해야 하는가”를 구분해야 합니다. 오늘은
`Promise.all`, `Promise.race`, `Promise.any`, `AbortController`를 복습하고, TypeScript에서
제네릭 Promise와 `AbortSignal`을 어떻게 연결할지 확인합니다.

### 레퍼런스 연결

- 비동기 챕터 복습
- 핵심 복습 포인트: `Promise.all`, `race`, `any`, `AbortController`, 취소 설계
- 다시 볼 질문: 빠른 응답과 정확한 에러 처리는 어떻게 균형을 잡아야 할까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
const p1 = Promise.resolve("A");
const p2 = Promise.resolve("B");

Promise.all([p1, p2]).then((values) => console.log(values.join(",")));

console.log("sync");
```

- 정답과 해설
    
    ### 정답
    
    ```
    sync
    A,B
    ```
    
    ### 해설
    
    `Promise.all([p1, p2])`는 배열 안의 Promise가 모두 fulfilled될 때 fulfilled됩니다. 결과값은
    입력 배열 순서를 유지한 배열입니다.
    
    `p1`과 `p2`는 이미 fulfilled된 Promise지만, `then` 콜백은 동기적으로 즉시 실행되지 않고
    microtask로 실행됩니다.
    
    그래서 동기 코드인 `console.log("sync")`가 먼저 실행됩니다.
    
    그 뒤 `Promise.all`의 `then` 콜백이 실행되고, `values`는 `["A", "B"]`입니다. `join(",")`으로
    `"A,B"`가 출력됩니다.
    
    `Promise.all`의 결과 순서는 완료 순서가 아니라 입력 순서라는 점이 중요합니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Promise.race`와 `Promise.all`의 차이를
설명하세요.

```jsx
const slow = new Promise((resolve) => setTimeout(() => resolve("slow"), 20));
const fast = new Promise((resolve) => setTimeout(() => resolve("fast"), 0));

Promise.race([slow, fast]).then(console.log);
Promise.all([slow, fast]).then((values) => console.log(values.join(",")));
```

- 정답과 해설
    
    ### 정답
    
    ```
    fast
    slow,fast
    ```
    
    ### 해설
    
    `fast`는 0ms timer로 해결되고, `slow`는 20ms timer로 해결됩니다. 실제 시간은 환경에 따라
    약간 달라질 수 있지만, 이 예제에서는 `fast`가 먼저 해결됩니다.
    
    `Promise.race([slow, fast])`는 입력된 Promise 중 가장 먼저 settled되는 Promise의 결과를
    따릅니다. 여기서는 `fast`가 먼저 fulfilled되므로 `"fast"`가 출력됩니다.
    
    `Promise.all([slow, fast])`는 모든 Promise가 fulfilled될 때까지 기다립니다. 결과 배열은 완료
    순서가 아니라 입력 배열 순서입니다.
    
    따라서 `slow`가 나중에 끝나더라도 결과 배열은 `[slow의 값, fast의 값]`, 즉
    `["slow", "fast"]`입니다. 출력은 `"slow,fast"`입니다.
    
    `race`는 빠른 응답이나 timeout 구현에 유용하고, `all`은 여러 결과가 모두 필요할 때 적합합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `Promise.all`과 `Promise.any`가 실패를 다루는
방식을 설명하세요.

```jsx
const failFast = Promise.reject(new Error("fail fast"));
const success = Promise.resolve("success");

Promise.all([failFast, success])
  .then(() => console.log("all ok"))
  .catch((error) => console.log(`all:${error.message}`));

Promise.any([failFast, success])
  .then((value) => console.log(`any:${value}`))
  .catch((error) => console.log(error.constructor.name));
```

- 정답과 해설
    
    ### 정답
    
    ```
    any:success
    all:fail fast
    ```
    
    ### 해설
    
    `Promise.all`은 입력 중 하나라도 reject되면 전체가 reject됩니다. `failFast`가 rejected
    Promise이므로 `Promise.all([failFast, success])`는 실패합니다. `catch`에서
    `"all:fail fast"`를 출력합니다.
    
    `Promise.any`는 입력 중 하나라도 fulfilled되면 그 값을 사용해 fulfilled됩니다. 실패한 Promise가
    있어도, 성공한 Promise가 하나라도 있으면 전체는 성공합니다.
    
    여기서는 `success`가 `"success"`로 fulfilled되므로 `Promise.any`의 `then`이 실행되어
    `"any:success"`가 출력됩니다.
    
    출력 순서가 `any:success` 다음 `all:fail fast`인 이유는 두 체인이 모두 microtask로 처리되지만,
    각 Promise 조합이 내부적으로 결과를 확정하고 이어지는 handler를 예약하는 순서가 이렇게
    나오기 때문입니다. 핵심은 두 결과가 모두 같은 동기 구간이 끝난 뒤 비동기적으로 출력된다는
    점입니다.
    
    `Promise.any`의 모든 입력이 실패하면 `AggregateError`로 reject됩니다. 반면 `Promise.all`은
    첫 실패 이유로 reject됩니다.
    

### 문제 4. 심화

다음 코드는 `AbortController`로 지연 작업을 취소합니다. 실행 결과를 예측하고, 이 코드의
의미를 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
function delay(ms, value, signal) {
  return new Promise((resolve, reject) => {
    if (signal.aborted) {
      reject(new Error("aborted before start"));
      return;
    }

    const id = setTimeout(() => resolve(value), ms);

    signal.addEventListener("abort", () => {
      clearTimeout(id);
      reject(new Error("aborted"));
    }, { once: true });
  });
}

const controller = new AbortController();

delay(20, "done", controller.signal)
  .then((value) => console.log(value))
  .catch((error) => console.log(error.message));

controller.abort();

setTimeout(() => console.log(controller.signal.aborted), 0);
```

- 정답과 해설
    
    ### 정답
    
    ```
    aborted
    true
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    function delay<T>(ms: number, value: T, signal: AbortSignal): Promise<T> {
      return new Promise((resolve, reject) => {
        if (signal.aborted) {
          reject(new Error("aborted before start"));
          return;
        }
    
        const id = setTimeout(() => resolve(value), ms);
    
        signal.addEventListener(
          "abort",
          () => {
            clearTimeout(id);
            reject(new Error("aborted"));
          },
          { once: true },
        );
      });
    }
    
    async function loadWithTimeout<T>(task: Promise<T>, timeoutMs: number): Promise<T> {
      const controller = new AbortController();
      const timeout = delay(timeoutMs, "timeout", controller.signal).then(() => {
        throw new Error("timeout");
      });
    
      try {
        return await Promise.race([task, timeout]);
      } finally {
        controller.abort();
      }
    }
    
    loadWithTimeout(Promise.resolve("ok"), 10)
      .then((value) => console.log(value))
      .catch((error: unknown) => {
        console.log(error instanceof Error ? error.message : "unknown error");
      });
    ```
    
    ### 해설
    
    ### JS 관점
    
    `delay(20, "done", controller.signal)`은 20ms 뒤 `"done"`으로 resolve될 Promise를 만듭니다.
    동시에 abort 이벤트를 들을 수 있도록 `signal.addEventListener("abort", ...)`를 등록합니다.
    
    그 직후 `controller.abort()`를 호출합니다. abort 이벤트 리스너가 실행되어 timer를
    `clearTimeout`으로 취소하고, Promise를 `Error("aborted")`로 reject합니다.
    
    따라서 `then`은 실행되지 않고, `catch`가 실행되어 `"aborted"`가 출력됩니다.
    
    마지막 `setTimeout` 콜백은 다음 task에서 `controller.signal.aborted`를 확인합니다.
    이미 abort가 호출되었으므로 `true`가 출력됩니다.
    
    취소는 “Promise 자체를 마법처럼 없애는 것”이 아니라, 비동기 작업이 `AbortSignal`을 확인하고
    정리 작업을 수행하도록 약속하는 구조입니다.
    
    ### TS 관점
    
    `delay<T>(ms: number, value: T, signal: AbortSignal): Promise<T>`는 어떤 타입의 값이든 지연 후
    같은 타입으로 반환할 수 있음을 제네릭으로 표현합니다.
    
    `AbortSignal`을 명시적으로 인자로 받으면 이 함수가 취소 가능한 작업이라는 사실이 타입
    시그니처에 드러납니다.
    
    `loadWithTimeout<T>`는 실제 작업 `task`와 timeout Promise를 `Promise.race`로 경쟁시킵니다.
    작업이 먼저 끝나면 그 값을 반환하고, timeout이 먼저 끝나면 `Error("timeout")`을 던집니다.
    
    `finally`에서 `controller.abort()`를 호출하는 이유는 timeout 쪽 timer와 이벤트 리스너를 정리하기
    위해서입니다. 이미 작업이 성공했더라도 timeout 예약을 계속 둘 필요가 없습니다.
    
    다만 `Promise.race`에 timeout을 붙인다고 해서 원본 `task`가 자동으로 취소되는 것은 아닙니다.
    원본 작업까지 중단하려면 그 작업도 `AbortSignal`을 인자로 받고, 내부에서 signal을 확인해
    협조적으로 멈추도록 설계해야 합니다.
    
    ### 실무 관점
    
    비동기 동시성을 설계할 때는 목적이 다릅니다.
    
    1. 모든 결과가 필요하면 `Promise.all`
    2. 가장 빠른 settled 결과가 필요하면 `Promise.race`
    3. 하나라도 성공하면 되는 경우에는 `Promise.any`
    4. 오래 걸리는 작업을 중단해야 하면 `AbortController`
    
    또한 timeout을 만들 때는 “빠른 결과를 반환하는 것”뿐 아니라, 늦게 남아 있는 작업을 어떻게
    정리할지도 생각해야 합니다. 취소와 정리 정책이 없는 동시성 코드는 화면 깜빡임, 중복 요청,
    메모리 누수, 오래된 응답 반영 같은 문제로 이어질 수 있습니다.
