# Day 20. 상속과 오버라이딩

### 오늘의 목표

JavaScript 클래스 상속은 프로토타입 체인을 연결하고, 하위 클래스가 상위 클래스의 동작을
재사용하거나 덮어쓰게 합니다. 오늘은 `extends`, 메서드 오버라이딩, `super`, 정적 속성 상속을
복습하고, TypeScript에서 상속과 `implements` 기반 조합을 어떻게 구분할지 확인합니다.

### 레퍼런스 연결

- 클래스와 프로토타입 챕터 복습
- 핵심 복습 포인트: 상속, 메서드 오버라이딩, `super`, 정적 속성 상속
- 다시 볼 질문: 상속보다 조합이 더 나은 상황은 언제일까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
class Animal {
  speak() {
    return "sound";
  }
}

class Dog extends Animal {
  speak() {
    return "bark";
  }
}

const dog = new Dog();

console.log(dog.speak());
console.log(dog instanceof Dog);
console.log(dog instanceof Animal);
console.log(Object.getPrototypeOf(Dog.prototype) === Animal.prototype);
```

- 정답과 해설
    
    ### 정답
    
    ```
    bark
    true
    true
    true
    ```
    
    ### 해설
    
    `Dog`는 `Animal`을 `extends`합니다. 하위 클래스 `Dog`는 상위 클래스 `Animal`의 프로토타입
    체인을 이어받습니다.
    
    `Dog`는 `speak` 메서드를 다시 정의했습니다. 이를 오버라이딩이라고 합니다. `dog.speak()`를
    호출하면 프로토타입 체인에서 가장 가까운 `Dog.prototype.speak`가 사용되므로 `"bark"`가
    출력됩니다.
    
    `dog instanceof Dog`는 `true`입니다. `dog`는 `new Dog()`로 만들어졌습니다.
    
    `dog instanceof Animal`도 `true`입니다. `Dog`가 `Animal`을 상속하므로 `dog`의 프로토타입
    체인에는 `Dog.prototype`뿐 아니라 `Animal.prototype`도 포함됩니다.
    
    `Object.getPrototypeOf(Dog.prototype) === Animal.prototype`도 `true`입니다. `extends`가
    하위 클래스의 프로토타입을 상위 클래스의 프로토타입에 연결하기 때문입니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 작성하고, `super`가 어떤 역할을 하는지 설명하세요.

```jsx
class Base {
  constructor(name) {
    this.name = name;
  }

  label() {
    return `Base:${this.name}`;
  }
}

class Child extends Base {
  label() {
    return `Child>${super.label()}`;
  }
}

const child = new Child("Ada");

console.log(child.label());
```

- 정답과 해설
    
    ### 정답
    
    ```
    Child>Base:Ada
    ```
    
    ### 해설
    
    `Child`는 `Base`를 상속합니다. `Child`에는 별도 생성자가 없으므로, `new Child("Ada")`를
    호출하면 상위 클래스 `Base`의 생성자가 사용되어 `this.name`이 `"Ada"`가 됩니다.
    
    `Child`는 `label` 메서드를 오버라이딩했습니다. `child.label()`을 호출하면
    `Child.prototype.label`이 실행됩니다.
    
    메서드 안의 `super.label()`은 상위 클래스의 `label` 메서드를 호출합니다. 이때 `this`는
    여전히 현재 인스턴스인 `child`입니다. 따라서 `Base`의 `label`은 `this.name`을 읽어
    `"Base:Ada"`를 반환합니다.
    
    그 결과를 `Child>`와 합치므로 최종 출력은 `"Child>Base:Ada"`입니다.
    
    `super`는 상위 클래스 구현을 재사용하면서 하위 클래스에서 앞뒤 동작을 추가할 때 유용합니다.
    하지만 상위 클래스 구현에 강하게 의존하게 되므로, 상속 계층이 깊어지면 동작 추적이 어려워질
    수 있습니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 정적 속성과 정적 메서드가 상속되는 방식을
설명하세요.

```jsx
class Parent {
  static role = "parent";

  static describe() {
    return this.role;
  }
}

class ChildClass extends Parent {}

console.log(Parent.describe());
console.log(ChildClass.describe());

ChildClass.role = "child";

console.log(Parent.describe());
console.log(ChildClass.describe());
```

- 정답과 해설
    
    ### 정답
    
    ```
    parent
    parent
    parent
    child
    ```
    
    ### 해설
    
    정적 속성과 정적 메서드는 인스턴스가 아니라 클래스 생성자 객체에 붙습니다.
    
    `Parent.describe()`는 `this`가 `Parent`인 상태로 실행됩니다. `Parent.role`은 `"parent"`이므로
    첫 번째 출력은 `"parent"`입니다.
    
    `ChildClass`는 `Parent`를 상속합니다. JavaScript 클래스 상속은 인스턴스 프로토타입뿐 아니라
    생성자 함수 쪽의 상속도 연결합니다. 그래서 `ChildClass.describe()`를 호출할 수 있습니다.
    이때 `this`는 `ChildClass`입니다.
    
    처음에는 `ChildClass` 자신의 `role`이 없으므로 상위 생성자 쪽의 `Parent.role`을 찾아
    `"parent"`가 출력됩니다.
    
    `ChildClass.role = "child"`를 실행하면 `ChildClass` 자신의 정적 속성 `role`이 생깁니다.
    `Parent.role`을 바꾸는 것이 아닙니다.
    
    그래서 이후 `Parent.describe()`는 여전히 `"parent"`이고, `ChildClass.describe()`는
    `ChildClass.role`인 `"child"`를 반환합니다.
    

### 문제 4. 심화

다음 코드는 알림 전송 클래스를 상속으로 확장합니다. 실행 결과를 예측하고, 이 코드의
장단점을 JS 관점과 TS 관점에서 설명한 뒤 조합 중심의 TypeScript 코드로 개선하세요.

```jsx
class EmailNotifier {
  send(message) {
    return `email:${message}`;
  }
}

class SmsNotifier extends EmailNotifier {
  send(message) {
    if (message.length > 5) {
      return super.send(message);
    }

    return `sms:${message}`;
  }
}

function notify(notifier, message) {
  return notifier.send(message);
}

console.log(notify(new EmailNotifier(), "hello"));
console.log(notify(new SmsNotifier(), "hi"));
console.log(notify(new SmsNotifier(), "welcome"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    email:hello
    sms:hi
    email:welcome
    ```
    
    조합 중심으로 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type Notifier = {
      send(message: string): string;
    };
    
    class EmailNotifier implements Notifier {
      send(message: string): string {
        return `email:${message}`;
      }
    }
    
    class SmsNotifier implements Notifier {
      send(message: string): string {
        return `sms:${message}`;
      }
    }
    
    class LengthBasedNotifier implements Notifier {
      private readonly shortMessageNotifier: Notifier;
      private readonly fallbackNotifier: Notifier;
      private readonly maxSmsLength: number;
    
      constructor(
        shortMessageNotifier: Notifier,
        fallbackNotifier: Notifier,
        maxSmsLength: number,
      ) {
        this.shortMessageNotifier = shortMessageNotifier;
        this.fallbackNotifier = fallbackNotifier;
        this.maxSmsLength = maxSmsLength;
      }
    
      send(message: string): string {
        if (message.length > this.maxSmsLength) {
          return this.fallbackNotifier.send(message);
        }
    
        return this.shortMessageNotifier.send(message);
      }
    }
    
    function notify(notifier: Notifier, message: string): string {
      return notifier.send(message);
    }
    
    const notifier = new LengthBasedNotifier(
      new SmsNotifier(),
      new EmailNotifier(),
      5,
    );
    
    console.log(notify(new EmailNotifier(), "hello"));
    console.log(notify(notifier, "hi"));
    console.log(notify(notifier, "welcome"));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `EmailNotifier`의 `send`는 항상 `"email:..."` 형식의 문자열을 반환합니다.
    
    `SmsNotifier`는 `EmailNotifier`를 상속하고 `send`를 오버라이딩합니다. 메시지 길이가 5보다
    크면 `super.send(message)`를 호출해 상위 클래스의 이메일 전송 동작을 재사용하고, 그렇지
    않으면 `"sms:..."`를 반환합니다.
    
    `notify(new EmailNotifier(), "hello")`는 이메일 알림 객체를 사용하므로 `"email:hello"`입니다.
    
    `notify(new SmsNotifier(), "hi")`는 `SmsNotifier.send`를 호출합니다. `"hi"`의 길이는 2이므로
    `"sms:hi"`입니다.
    
    `notify(new SmsNotifier(), "welcome")`에서 `"welcome"`의 길이는 7입니다. 조건을 만족해
    `super.send("welcome")`를 호출하므로 `"email:welcome"`이 됩니다.
    
    이 코드는 동작은 하지만, `SmsNotifier`가 정말 `EmailNotifier`의 한 종류인지 의문이 생깁니다.
    짧은 메시지는 SMS로 보내고 긴 메시지는 이메일로 보내는 정책은 “상속 관계”라기보다 “전송 전략”
    또는 “정책 조합”에 가깝습니다.
    
    ### TS 관점
    
    TypeScript에서는 `implements`를 사용해 클래스가 특정 동작 계약을 만족한다고 표현할 수
    있습니다. 개선 예시의 `Notifier`는 `send(message: string): string` 메서드를 가진 객체라는
    계약입니다.
    
    `EmailNotifier`와 `SmsNotifier`는 서로 상속하지 않습니다. 둘 다 `Notifier`를 구현할 뿐입니다.
    `LengthBasedNotifier`는 두 구현체를 조합해 짧은 메시지는 SMS로 보내고, 기준 길이를 넘는
    메시지는 fallback인 이메일로 보냅니다. 이렇게 하면 원래 예제의 `"welcome" -> "email:welcome"`
    정책을 상속 없이 보존할 수 있습니다.
    
    `notify`는 구체 클래스가 아니라 `Notifier` 계약에 의존합니다.
    
    이렇게 하면 알림 전송 방식이 늘어나도 상속 계층을 깊게 만들 필요가 없습니다. `PushNotifier`,
    `SlackNotifier`, `MockNotifier` 같은 클래스를 같은 계약으로 추가할 수 있습니다.
    
    상속이 필요한 경우도 있습니다. 상위 클래스의 상태와 알고리즘을 하위 클래스가 자연스럽게
    공유하고, “하위 클래스는 상위 클래스의 한 종류다”라는 관계가 명확할 때입니다. 하지만 단순히
    같은 메서드 이름을 공유하기 위해 상속을 쓰는 것은 과한 결합이 될 수 있습니다.
    
    ### 실무 관점
    
    상속과 조합을 고를 때는 다음 질문을 던질 수 있습니다.
    
    1. 하위 클래스가 상위 클래스의 진짜 한 종류인가
    2. 상위 클래스의 내부 구현 변경이 하위 클래스에 영향을 줘도 괜찮은가
    3. 동작을 런타임에 바꿔 끼워야 하는가
    4. 테스트에서 대체 구현을 쉽게 넣어야 하는가
    
    상속은 공통 구조와 기본 동작을 공유할 때 유용합니다. 조합은 정책을 갈아 끼우거나 의존성을
    느슨하게 유지할 때 유리합니다. 좋은 설계는 상속을 금지하는 것이 아니라, 상속이 실제 관계를
    정확히 표현하는지 확인하는 데서 시작합니다.
    

### 오늘의 생각 질문

`extends`를 선택해도 되는 관계와, interface 또는 객체 조합으로 푸는 편이 나은 관계를 어떻게
구분할 수 있을까요?
