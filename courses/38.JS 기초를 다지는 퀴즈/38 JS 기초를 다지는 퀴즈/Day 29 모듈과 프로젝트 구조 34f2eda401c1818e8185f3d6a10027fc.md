- 모듈 챕터 복습

### 문제 1. 기본

다음 코드는 ESM 모듈을 동적으로 만들어 불러옵니다. 실행 결과를 작성하세요.

```jsx
const source = `
  export const language = "JS";
  export default "quiz";
`;

const mod = await import(`data:text/javascript,${encodeURIComponent(source)}`);

console.log(mod.language);
console.log(mod.default);
console.log(Object.keys(mod).sort().join(","));
```

- 정답과 해설
    
    ### 정답
    
    ```
    JS
    quiz
    default,language
    ```
    
    ### 해설
    
    이 코드는 ESM 환경에서 실행되는 예제입니다. `source` 문자열 안에는 두 가지 export가 있습니다.
    
    첫 번째는 named export입니다.
    
    ```jsx
    export const language = "JS";
    ```
    
    이 값은 모듈을 불러온 뒤 `mod.language`로 접근할 수 있습니다.
    
    두 번째는 default export입니다.
    
    ```jsx
    export default "quiz";
    ```
    
    default export도 모듈 namespace 객체 안에서는 `default`라는 이름의 속성으로 들어갑니다. 그래서
    `mod.default`는 `"quiz"`입니다.
    
    마지막 줄은 모듈 namespace 객체의 key를 정렬해서 출력합니다. key는 `"default"`와 `"language"`가
    있으므로 `"default,language"`가 출력됩니다.
    
    ESM에서는 named export와 default export를 섞어 쓸 수 있지만, 프로젝트에서는 일관성이 중요합니다.
    특히 리팩터링과 자동 import를 고려하면 named export가 추적하기 쉬운 경우가 많습니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 작성하고, ESM의 live binding이 무엇인지 설명하세요.

```jsx
const source = `
  export let count = 0;
  export function increase() {
    count += 1;
  }
`;

const mod = await import(`data:text/javascript,${encodeURIComponent(source)}`);

console.log(mod.count);
mod.increase();
console.log(mod.count);
```

- 정답과 해설
    
    ### 정답
    
    ```
    0
    1
    ```
    
    ### 해설
    
    처음 모듈이 평가될 때 `count`는 `0`입니다. 따라서 첫 번째 출력은 `0`입니다.
    
    그 다음 `mod.increase()`가 호출됩니다. `increase` 함수는 모듈 내부의 `count` 변수 값을
    `1` 증가시킵니다.
    
    마지막 `console.log(mod.count)`는 증가한 값을 읽으므로 `1`이 출력됩니다.
    
    ESM의 export는 값 복사가 아니라 live binding입니다. 즉, export된 변수가 모듈 내부에서 바뀌면
    그 변화를 import한 쪽에서도 볼 수 있습니다.
    
    다만 이 말이 import한 쪽에서 마음대로 값을 바꿀 수 있다는 뜻은 아닙니다. import binding은 읽기
    전용입니다. 상태를 바꾸고 싶다면 예시처럼 모듈이 제공하는 함수를 통해 바꾸는 편이 안전합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 동적 `import()`가 어떤 값을 반환하는지 설명하세요.

```jsx
const source = `export const value = "loaded";`;
const url = `data:text/javascript,${encodeURIComponent(source)}`;

console.log("A");

import(url).then((mod) => {
  console.log(mod.value);
});

console.log("B");
```

- 정답과 해설
    
    ### 정답
    
    ```
    A
    B
    loaded
    ```
    
    ### 해설
    
    `console.log("A")`는 즉시 실행되므로 첫 번째 출력은 `"A"`입니다.
    
    `import(url)`은 모듈 namespace 객체로 resolve되는 `Promise`를 반환합니다. 따라서 `.then(...)`
    안의 코드는 모듈 로딩과 평가가 끝난 뒤 비동기적으로 실행됩니다.
    
    그 사이 다음 줄의 `console.log("B")`가 먼저 실행됩니다. 그래서 두 번째 출력은 `"B"`입니다.
    
    이후 동적 import가 완료되면 `mod.value`가 출력됩니다. 모듈에는 `export const value = "loaded"`가
    있으므로 마지막 출력은 `"loaded"`입니다.
    
    동적 `import()`는 조건부 로딩, 라우트 단위 코드 분할, 무거운 기능의 지연 로딩에 자주 사용됩니다.
    하지만 반환값이 `Promise`라는 점 때문에 에러 처리와 로딩 상태 설계를 함께 해야 합니다.
    

### 문제 4. 심화

다음 세 파일을 같은 폴더에 두고 `node main.mjs`로 실행한다고 가정합니다. 실행 결과를 예측하고,
순환 의존성이 왜 문제가 되는지 JS 관점과 TS 관점에서 설명한 뒤 개선 방향을 제시하세요.

```jsx
// a.mjs
import { bValue } from "./b.mjs";

export const aValue = "A";

export function readB() {
  return bValue;
}
```

```jsx
// b.mjs
import { aValue } from "./a.mjs";

export const bValue = `B sees${aValue}`;
```

```jsx
// main.mjs
import { aValue, readB } from "./a.mjs";

console.log(aValue);
console.log(readB());
```

- 정답과 해설
    
    ### 정답
    
    정상 출력이 나오지 않고 다음 종류의 에러가 발생합니다.
    
    ```
    ReferenceError: Cannot access 'aValue' before initialization
    ```
    
    순환 의존성을 줄인 TypeScript 구조 예시는 다음과 같습니다.
    
    ```tsx
    // shared-types.ts
    export type ModuleName = "a" | "b";
    ```
    
    ```tsx
    // a.ts
    import type { ModuleName } from "./shared-types";
    
    export const aName: ModuleName = "a";
    
    export function describeA(readB: () => string): string {
      return `${aName} reads${readB()}`;
    }
    ```
    
    ```tsx
    // b.ts
    import type { ModuleName } from "./shared-types";
    
    export const bName: ModuleName = "b";
    
    export function readB(): string {
      return bName;
    }
    ```
    
    ```tsx
    // main.ts
    import { describeA } from "./a";
    import { readB } from "./b";
    
    console.log(describeA(readB));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `main.mjs`는 `a.mjs`를 import합니다. 그런데 `a.mjs`는 다시 `b.mjs`를 import합니다. `b.mjs`는
    다시 `a.mjs`의 `aValue`를 import합니다. 이 구조가 순환 의존성입니다.
    
    문제는 `b.mjs`가 평가되는 시점입니다.
    
    ```jsx
    export const bValue = `B sees${aValue}`;
    ```
    
    이 줄은 `aValue`를 즉시 읽습니다. 하지만 `a.mjs`의 본문은 아직 끝까지 실행되지 않았고,
    `export const aValue = "A"`도 초기화되기 전입니다.
    
    ESM의 `const` export도 TDZ의 영향을 받습니다. 초기화 전의 binding을 읽으려고 하므로
    `ReferenceError`가 발생합니다. 그래서 `console.log(aValue)`와 `console.log(readB())`까지
    도달하지 못합니다.
    
    ### TS 관점
    
    TypeScript는 타입을 검사해 주지만, 런타임 모듈 평가 순서를 자동으로 안전하게 바꿔 주지는
    않습니다. 타입이 맞아도 순환 의존성 때문에 런타임에서 터질 수 있습니다.
    
    그래서 타입만 필요한 의존성은 `import type`으로 분리하는 편이 좋습니다. `import type`은 타입
    검사용이며 런타임 import를 만들지 않습니다.
    
    또한 한 모듈이 다른 모듈의 초기화 중인 값을 직접 읽는 구조를 피해야 합니다. 예시 개선 코드처럼
    공유 타입은 `shared-types.ts`로 분리하고, 런타임 값은 함수 인자로 전달하면 순환 참조를 줄일 수
    있습니다.
    
    ### 실무 관점
    
    순환 의존성은 처음에는 잘 동작하는 것처럼 보이다가 파일 분리, bundler 설정, 테스트 실행 순서가
    바뀔 때 문제를 만들 수 있습니다.
    
    다음 기준으로 구조를 점검하면 좋습니다.
    
    1. 하위 모듈이 상위 모듈을 import하지 않게 한다.
    2. 공통 타입과 상수는 별도 파일로 분리한다.
    3. 초기화 시점에 다른 모듈의 값을 바로 읽는 코드를 줄인다.
    4. 모듈 사이의 실행 의존성은 함수 인자나 생성자 주입으로 표현한다.
    5. `import type`으로 타입 의존성과 런타임 의존성을 구분한다.
