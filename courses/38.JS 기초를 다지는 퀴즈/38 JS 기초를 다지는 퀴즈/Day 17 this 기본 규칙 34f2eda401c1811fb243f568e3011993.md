# Day 17. this 기본 규칙

### 오늘의 목표

JavaScript의 `this`는 함수가 어디에 작성되었는지가 아니라 어떻게 호출되었는지에 따라
결정됩니다. 오늘은 strict mode의 일반 함수 호출, 메서드 호출, `call`/`apply`/`bind`, 콜백
전달 시 `this`가 사라지는 문제를 복습하고, TypeScript의 명시적 `this` 파라미터로 의도를
드러내는 방법을 정리합니다.

### 레퍼런스 연결

- `this` 챕터 복습
- 핵심 복습 포인트: 전역 `this`, 함수 호출, 메서드 호출, 암시적 바인딩
- 다시 볼 질문: `this`를 쓰는 코드와 명시적 인자를 받는 코드 중 무엇이 더 명확할까?

### 문제 1. 기본

다음 코드는 strict mode에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하세요.

```jsx
"use strict";

function showThis() {
  return this;
}

console.log(showThis() === undefined);
console.log(showThis.call({ name: "Ada" }).name);
```

- 정답과 해설
    
    ### 정답
    
    ```
    true
    Ada
    ```
    
    ### 해설
    
    strict mode에서 일반 함수 호출의 `this`는 `undefined`입니다. `showThis()`는 앞에 객체가
    붙어 있지 않은 일반 함수 호출이므로 `showThis() === undefined`는 `true`입니다.
    
    `showThis.call({ name: "Ada" })`는 `call`을 사용해 함수 내부의 `this`를 직접 지정합니다.
    따라서 `showThis` 안의 `this`는 `{ name: "Ada" }`이고, `.name`을 읽으면 `"Ada"`가 나옵니다.
    
    `this`는 함수의 선언 위치보다 호출 형태에 의해 결정됩니다. 이 규칙을 놓치면 같은 함수도
    호출 방식에 따라 완전히 다른 결과를 낼 수 있습니다.
    

### 문제 2. 중급

다음 코드는 strict mode에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하고, 메서드를
다른 객체에 할당하거나 변수에 담았을 때 `this`가 어떻게 달라지는지 설명하세요.

```jsx
"use strict";

const user = {
  name: "Ada",
  getName() {
    return this.name;
  },
};

const other = { name: "Lin", getName: user.getName };

console.log(user.getName());
console.log(other.getName());

const fn = user.getName;

try {
  console.log(fn());
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    Lin
    TypeError
    ```
    
    ### 해설
    
    `user.getName()`은 메서드 호출입니다. 호출식의 점 왼쪽 객체가 `this`가 되므로, `this`는
    `user`입니다. 결과는 `"Ada"`입니다.
    
    `other.getName()`도 메서드 호출입니다. 함수 자체는 `user.getName`과 같은 함수지만, 호출식의
    점 왼쪽 객체는 `other`입니다. 따라서 `this`는 `other`이고 결과는 `"Lin"`입니다.
    
    `const fn = user.getName`은 함수 값만 변수에 담습니다. 이 시점에 `this`가 함께 저장되는
    것은 아닙니다.
    
    `fn()`은 일반 함수 호출입니다. strict mode에서 일반 함수 호출의 `this`는 `undefined`입니다.
    함수 내부에서 `this.name`을 읽으려고 하므로 `TypeError`가 발생합니다.
    
    핵심은 “어떤 객체의 속성으로 저장되어 있는가”보다 “어떤 형태로 호출되는가”입니다.
    

### 문제 3. 중급

다음 코드는 strict mode에서 실행한다고 가정합니다. 실행 결과를 순서대로 작성하고,
`call`, `apply`, `bind`가 `this`를 다루는 방식을 설명하세요.

```jsx
"use strict";

function getName() {
  return this.name;
}

console.log(getName.call({ name: "Lin" }));
console.log(getName.apply({ name: "Ken" }));

const bound = getName.bind({ name: "Mia" });

console.log(bound.call({ name: "Noah" }));
```

- 정답과 해설
    
    ### 정답
    
    ```
    Lin
    Ken
    Mia
    ```
    
    ### 해설
    
    `getName.call({ name: "Lin" })`은 `this`를 `{ name: "Lin" }`으로 지정해 즉시 호출합니다.
    결과는 `"Lin"`입니다.
    
    `getName.apply({ name: "Ken" })`도 `this`를 `{ name: "Ken" }`으로 지정해 즉시 호출합니다.
    이 예제에는 추가 인자가 없으므로 `call`과 결과 차이는 없습니다. 일반적으로 `apply`는 인자
    목록을 배열로 넘길 때 사용합니다.
    
    `getName.bind({ name: "Mia" })`는 `this`가 고정된 새 함수를 만듭니다. 이 함수는 이미
    `this`가 `"Mia"` 객체로 묶여 있습니다.
    
    `bound.call({ name: "Noah" })`처럼 나중에 `call`로 다른 `this`를 지정하려 해도, 이미
    바인딩된 함수의 `this`는 바뀌지 않습니다. 결과는 `"Mia"`입니다.
    
    `bind`는 한 번 바인딩된 함수의 `this`를 강하게 고정합니다. 콜백으로 넘길 때 유용하지만,
    나중에 다른 컨텍스트로 재사용하기 어렵게 만들 수도 있습니다.
    

### 문제 4. 심화

다음 코드는 strict mode에서 포맷터 메서드를 콜백으로 넘깁니다. 실행 결과를 예측하고, 이
코드의 문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
"use strict";

const formatter = {
  prefix: "Hi",
  format(name) {
    return `${this.prefix},${name}`;
  },
};

function render(names, format) {
  return names.map(format).join("|");
}

try {
  console.log(render(["Ada", "Lin"], formatter.format));
} catch (error) {
  console.log(error.constructor.name);
}

console.log(render(["Ada", "Lin"], formatter.format.bind(formatter)));
```

- 정답과 해설
    
    ### 정답
    
    ```
    TypeError
    Hi, Ada|Hi, Lin
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type Formatter = {
      prefix: string;
    };
    
    function format(this: Formatter, name: string): string {
      return `${this.prefix},${name}`;
    }
    
    function render(
      names: readonly string[],
      formatter: (this: void, name: string) => string,
    ): string {
      return names.map(formatter).join("|");
    }
    
    const context: Formatter = { prefix: "Hi" };
    const boundFormat: OmitThisParameter<typeof format> = format.bind(context);
    
    console.log(render(["Ada", "Lin"], boundFormat));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `formatter.format`은 `this.prefix`를 읽는 메서드입니다. `formatter.format("Ada")`처럼
    호출하면 `this`는 `formatter`가 됩니다.
    
    하지만 `render(["Ada", "Lin"], formatter.format)`은 메서드를 호출한 것이 아니라 함수 값만
    콜백으로 넘긴 것입니다. `render` 내부의 `names.map(format)`에서 콜백은 `formatter` 객체와
    분리되어 호출됩니다.
    
    strict mode에서 분리된 일반 함수 호출의 `this`는 `undefined`입니다. 따라서 `this.prefix`를
    읽는 순간 `TypeError`가 발생합니다.
    
    `formatter.format.bind(formatter)`는 `this`를 `formatter`로 고정한 새 함수를 만듭니다. 이
    바인딩된 함수를 `render`에 넘기면 콜백으로 호출되어도 `this.prefix`가 `"Hi"`로 유지됩니다.
    
    그래서 두 번째 출력은 `"Hi, Ada|Hi, Lin"`입니다.
    
    ### TS 관점
    
    TypeScript에서는 `function format(this: Formatter, name: string): string`처럼 명시적
    `this` 파라미터를 사용할 수 있습니다. 이 파라미터는 실제 런타임 인자가 아니라, 함수가 어떤
    `this`를 기대하는지 타입으로 표현하는 장치입니다.
    
    `render`는 `(this: void, name: string) => string` 형태의 콜백만 받습니다. 즉, 호출자가
    따로 `this`를 제공하지 않아도 실행 가능한 함수여야 합니다. 여기서 `this: void`를 명시하는
    이유가 중요합니다. 단순히 `(name: string) => string`으로만 쓰면 명시적 `this` 파라미터가
    있는 함수를 직접 넘기는 실수를 충분히 막지 못할 수 있습니다.
    
    `format.bind(context)`로 `this`를 고정하면 더 이상 호출 시점의 `this`가 필요하지 않습니다.
    `OmitThisParameter<typeof format>`은 이처럼 `this` 파라미터가 제거된 함수 타입을 표현합니다.
    
    이 구조는 `this`에 의존하는 함수와 일반 콜백을 타입 수준에서 구분하게 해 줍니다.
    예를 들어 `render(["Ada", "Lin"], format)`처럼 바인딩하지 않은 `format`을 넘기면
    `this` 타입이 맞지 않아 컴파일 오류가 됩니다.
    
    ### 실무 관점
    
    `this` 문제는 이벤트 핸들러, 배열 메서드 콜백, 타이머, 외부 라이브러리 콜백에서 반복해서
    나옵니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 콜백으로 넘길 함수가 `this`를 사용하는지 확인한다.
    2. 콜백 API는 가능하면 `this`가 필요 없는 함수 타입으로 설계한다.
    3. 기존 메서드를 콜백으로 넘겨야 한다면 `bind`나 화살표 함수 래퍼로 고정한다.
    4. TypeScript에서는 명시적 `this` 파라미터와 `OmitThisParameter`로 의도를 드러낸다.
    
    `this`는 함수의 “소속”이 아니라 호출 방식의 결과입니다. 그래서 메서드를 값으로 꺼내는 순간
    원래 객체와의 연결이 끊길 수 있다는 점을 항상 확인해야 합니다.
