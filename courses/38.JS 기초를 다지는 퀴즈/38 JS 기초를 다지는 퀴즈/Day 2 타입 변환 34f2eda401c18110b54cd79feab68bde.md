# Day 2. 타입 변환

### 오늘의 목표

JavaScript는 표현식을 평가하는 과정에서 값을 다른 타입으로 바꿀 수 있습니다. 오늘은
명시적 타입 변환과 암시적 타입 변환의 차이를 확인하고, `+` 연산자와 다른 산술
연산자의 변환 규칙, Boolean 변환, 객체의 원시값 변환, 사용자 입력 처리에서 생기는
실무 버그를 복습합니다.

### 레퍼런스 연결

- 타입 챕터 복습
- 핵심 복습 포인트: 명시적 변환, 암시적 변환, 산술 연산자와 문자열 변환
- 다시 볼 질문: 실무 코드에서 암시적 타입 변환을 어디까지 허용할 수 있을까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
console.log(Number(""));
console.log(Number("  "));
console.log(Number("JS"));
console.log(String(null));
console.log(Boolean("false"));
console.log(Boolean(0));
```

- 정답과 해설
    
    ### 정답
    
    ```
    0
    0
    NaN
    null
    true
    false
    ```
    
    ### 해설
    
    `Number("")`는 `0`입니다. 빈 문자열은 숫자로 변환될 때 `0`으로 처리됩니다.
    `Number("  ")`도 `0`입니다. 공백 문자열은 먼저 공백이 제거된 뒤 빈 문자열처럼
    처리됩니다.
    
    `Number("JS")`는 숫자로 해석할 수 없는 문자열이므로 `NaN`입니다. `NaN`은
    `Not-a-Number`라는 뜻이지만, JavaScript에서 `typeof NaN`은 `"number"`입니다.
    
    `String(null)`은 문자열 `"null"`을 반환합니다. `null` 자체가 출력되는 것처럼 보일 수
    있지만, 실제 값은 문자열입니다.
    
    `Boolean("false")`는 `true`입니다. 문자열의 내용이 `"false"`라고 해서 Boolean
    `false`가 되는 것이 아닙니다. 빈 문자열 `""`만 Falsy이고, 비어 있지 않은 문자열은
    Truthy입니다.
    
    `Boolean(0)`은 `false`입니다. 숫자 `0`, `-0`, `NaN`, `0n`, 빈 문자열, `null`,
    `undefined`, `false`는 Falsy 값입니다.
    
    헷갈리기 쉬운 지점은 `"false"`, `"0"`, `" "` 같은 문자열입니다. 문자열 안의 의미가
    거짓처럼 보여도, 비어 있지 않은 문자열이면 Boolean 변환 결과는 `true`입니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `+` 연산자와 다른 산술 연산자의 차이를
설명하세요.

```jsx
const a = "5" + 2;
const b = "5" - 2;
const c = true + false;
const d = null + 3;
const e = undefined + 3;
const f = "10" * "2";

console.log(a, typeof a);
console.log(b, typeof b);
console.log(c, typeof c);
console.log(d, typeof d);
console.log(e, typeof e);
console.log(f, typeof f);
```

- 정답과 해설
    
    ### 정답
    
    ```
    52 string
    3 number
    1 number
    3 number
    NaN number
    20 number
    ```
    
    ### 해설
    
    `"5" + 2`는 문자열 `"52"`가 됩니다. `+` 연산자는 덧셈에도 쓰이고 문자열 연결에도
    쓰입니다. 피연산자 중 하나가 문자열이면 다른 피연산자도 문자열로 변환한 뒤 이어
    붙입니다. 그래서 결과는 문자열 `"52"`이고 `typeof a`는 `"string"`입니다.
    
    `"5" - 2`는 숫자 `3`입니다. `-` 연산자는 문자열 연결 기능이 없으므로 피연산자를
    숫자로 변환한 뒤 산술 연산을 합니다. 문자열 `"5"`는 숫자 `5`가 되고, `5 - 2`는
    `3`입니다.
    
    `true + false`는 `1`입니다. Boolean 값은 숫자 연산에서 `true -> 1`, `false -> 0`으로
    변환됩니다. 따라서 `1 + 0`이 되어 결과는 `1`입니다.
    
    `null + 3`은 `3`입니다. 숫자 변환에서 `null`은 `0`이 됩니다. 따라서 `0 + 3`입니다.
    
    `undefined + 3`은 `NaN`입니다. 숫자 변환에서 `undefined`는 `NaN`이 되고, `NaN`이
    포함된 산술 연산 결과는 `NaN`입니다. 이때 `NaN`의 타입은 Number이므로 `typeof e`는
    `"number"`입니다.
    
    `"10" * "2"`는 `20`입니다. `*` 연산자는 문자열 연결을 하지 않고 두 피연산자를 숫자로
    변환합니다. `"10"`은 `10`, `"2"`는 `2`가 되어 결과는 `20`입니다.
    
    핵심은 `+`만 문자열 연결과 숫자 덧셈을 모두 담당한다는 점입니다. 그래서 `+`가 들어간
    표현식은 다른 산술 연산자보다 암시적 타입 변환을 더 신중하게 읽어야 합니다.
    

### 문제 3. 중급

다음 객체는 원시값으로 변환될 때 상황에 따라 다른 값을 반환합니다. 실행 결과를
순서대로 작성하고, 어떤 변환 힌트가 사용되는지 설명하세요.

```jsx
const label = {
  [Symbol.toPrimitive](hint) {
    if (hint === "number") return 80;
    if (hint === "string") return "가격";
    return 90;
  },
  valueOf() {
    return 100;
  },
  toString() {
    return "상품";
  },
};

console.log(+label);
console.log(`${label}`);
console.log("총액:" + label);
console.log(label + 10);
```

- 정답과 해설
    
    ### 정답
    
    ```
    80
    가격
    총액:90
    100
    ```
    
    ### 해설
    
    객체가 숫자나 문자열처럼 사용되면 JavaScript는 객체를 원시값으로 변환하려고 합니다.
    이때 객체에 `Symbol.toPrimitive` 메서드가 있으면 이 메서드가 우선 호출됩니다.
    
    `+label`은 단항 `+` 연산자입니다. 단항 `+`는 값을 숫자로 변환하려고 하므로
    `Symbol.toPrimitive`에 `"number"` 힌트가 전달됩니다. 따라서 `80`이 반환되고 출력도
    `80`입니다.
    
    ``${label}``은 템플릿 리터럴 안에서 문자열을 만들고 있습니다. 이 경우 문자열
    변환이 필요하므로 `"string"` 힌트가 전달됩니다. 따라서 `"가격"`이 반환됩니다.
    
    `"총액:" + label`은 이항 `+` 연산입니다. 이항 `+`는 숫자 덧셈이 될 수도 있고 문자열
    연결이 될 수도 있으므로 객체를 원시값으로 바꿀 때 `"default"` 힌트를 사용합니다.
    위 코드에서는 `"default"`일 때 `90`을 반환합니다. 이후 왼쪽 피연산자가 문자열이므로
    문자열 연결이 일어나 `"총액:90"`이 출력됩니다.
    
    `label + 10`도 이항 `+` 연산이므로 `"default"` 힌트가 사용됩니다. `label`은 원시값
    `90`이 되고, 오른쪽 피연산자 `10`도 숫자이므로 숫자 덧셈이 일어나 `100`이 출력됩니다.
    
    `valueOf`와 `toString`이 정의되어 있어도, `Symbol.toPrimitive`가 있으면 이 메서드가
    우선합니다. 그래서 이 문제에서는 `valueOf()`의 `100`이나 `toString()`의 `"상품"`이
    직접 사용되지 않습니다.
    
    실무에서는 객체의 암시적 변환에 의존하는 코드는 읽기 어렵습니다. 금액, 날짜, 도메인
    객체처럼 문자열과 숫자 표현이 모두 필요한 값은 명시적인 메서드 이름을 두는 편이 더
    안전합니다.
    

### 문제 4. 심화

다음 코드는 상품 가격과 수량을 입력받아 배송비 3000원을 더한 총액을 계산하려고
작성되었습니다. 실행 결과를 예측하고, 이 코드의 문제점을 JS 관점과 TS 관점에서
설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

개선 코드에서는 가격과 수량을 `1` 이상의 정수 문자열로만 허용한다고 가정합니다.

```jsx
function calculateTotal(priceInput, quantityInput) {
  const price = Number(priceInput);
  const quantity = Number(quantityInput);
  return price * quantity + 3000;
}

console.log(calculateTotal("12000", "2"));
console.log(calculateTotal("", "2"));
console.log(calculateTotal("12,000", "2"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    27000
    3000
    NaN
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    function parsePositiveInteger(input: string): number | null {
      const normalized = input.trim();
    
      if (!/^[0-9]+$/.test(normalized)) {
        return null;
      }
    
      const value = Number(normalized);
      return Number.isSafeInteger(value) && value > 0 ? value : null;
    }
    
    function calculateTotal(priceInput: string, quantityInput: string): number | null {
      const price = parsePositiveInteger(priceInput);
      const quantity = parsePositiveInteger(quantityInput);
    
      if (price === null || quantity === null) {
        return null;
      }
    
      return price * quantity + 3000;
    }
    ```
    
    ### 해설
    
    ### JS 관점
    
    첫 번째 호출은 `calculateTotal("12000", "2")`입니다. `Number("12000")`은 `12000`,
    `Number("2")`는 `2`입니다. 따라서 `12000 * 2 + 3000`이 되어 `27000`이 출력됩니다.
    
    두 번째 호출은 `calculateTotal("", "2")`입니다. 여기서 핵심은 `Number("")`가
    `0`이라는 점입니다. 빈 입력을 숫자로 바꾸면 실패가 아니라 `0`이 됩니다. 그래서
    `0 * 2 + 3000`이 되어 `3000`이 출력됩니다. 사용자가 가격을 입력하지 않았는데도
    배송비만 붙은 총액이 계산되는 이상한 결과입니다.
    
    세 번째 호출은 `calculateTotal("12,000", "2")`입니다. 쉼표가 포함된 문자열
    `"12,000"`은 `Number`로 바로 변환할 수 없습니다. 따라서 `Number("12,000")`은
    `NaN`입니다. `NaN * 2 + 3000`의 결과도 `NaN`입니다.
    
    이 코드는 명시적으로 `Number(...)`를 사용하고 있지만 여전히 안전하지 않습니다.
    명시적 변환을 사용했다고 해서 입력값 검증이 끝나는 것은 아닙니다. 변환 결과가
    `0`, `NaN`, 안전하지 않은 숫자일 수 있는지 별도로 확인해야 합니다.
    
    ### TS 관점
    
    TypeScript에서 `priceInput`과 `quantityInput`을 `string`으로 선언하면, 입력이
    문자열이라는 사실은 표현할 수 있습니다. 하지만 `Number(priceInput)`의 결과가
    유효한 가격인지까지 타입만으로 보장되지는 않습니다.
    
    `Number(...)`의 반환 타입은 항상 `number`입니다. 이 `number` 안에는 정상 숫자뿐
    아니라 `NaN`, `Infinity`, 안전하지 않은 정수도 포함될 수 있습니다. 따라서
    TypeScript가 `number`라고 판단했다고 해서 계산해도 안전한 값이라는 뜻은 아닙니다.
    
    개선 예시에서는 `parsePositiveInteger`가 문자열을 먼저 검증합니다. 이 예시는 가격과
    수량을 `1` 이상의 정수 문자열로만 허용한다는 정책을 가집니다. 먼저 숫자로만 이루어진
    문자열인지 정규식으로 확인하고, `Number.isSafeInteger`와 `value > 0`으로 안전한
    양의 정수인지 확인합니다. 검증에 실패하면 `null`을 반환합니다.
    
    그 결과 `calculateTotal`의 반환 타입은 `number | null`입니다. 호출하는 쪽은 결과가
    `null`일 수 있음을 타입으로 알 수 있고, 실패 처리를 빠뜨리기 어려워집니다.
    
    ### 실무 관점
    
    폼 입력값, URL query, localStorage 값, API 응답은 자주 문자열로 들어옵니다. 이 값을
    바로 `Number`로 바꾸고 계산하면 빈 문자열, 쉼표가 들어간 숫자, 공백, 잘못된 단위가
    예상치 못한 결과를 만들 수 있습니다.
    
    실무에서는 타입 변환과 입력 검증을 분리해서 생각해야 합니다.
    
    - 타입 변환: 문자열을 숫자로 바꾼다.
    - 입력 검증: 이 값이 우리 도메인에서 허용되는 값인지 확인한다.
    
    금액처럼 중요한 값은 “숫자로 바뀌었다”만으로 충분하지 않습니다. 빈 값은 허용할지,
    쉼표는 허용할지, 음수는 허용할지, 소수는 허용할지 같은 정책을 먼저 정하고 그 정책을
    코드에 반영해야 합니다.
