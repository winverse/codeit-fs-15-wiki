# Day 6. 객체 생성과 속성 접근

### 오늘의 목표

JavaScript 객체는 키와 값의 모음이지만, 속성 키는 문자열 또는 Symbol로 다뤄지고
프로토타입 체인의 영향도 받습니다. 오늘은 객체 리터럴, 점 표기법, 대괄호 표기법,
계산된 속성명, 동적 키 접근을 복습하고, TypeScript의 `keyof`와 타입 가드로 안전한
속성 접근을 설계하는 방법까지 연결합니다.

### 레퍼런스 연결

- 객체 챕터 복습
- 핵심 복습 포인트: 객체 리터럴, 속성 접근, 계산된 속성명, 동적 키
- 다시 볼 질문: 동적 속성을 허용하는 객체는 어디까지 타입으로 열어야 할까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
const user = { name: "Ada", "user-id": 7, 1: "one" };

console.log(user.name);
console.log(user["user-id"]);
console.log(user[1]);
console.log(user["1"]);
```

- 정답과 해설
    
    ### 정답
    
    ```
    Ada
    7
    one
    one
    ```
    
    ### 해설
    
    `user.name`은 점 표기법입니다. 속성 이름이 유효한 식별자라면 점 표기법으로 접근할 수
    있습니다. `name` 속성의 값은 `"Ada"`입니다.
    
    `"user-id"`는 하이픈을 포함합니다. 하이픈은 JavaScript 식별자 이름에 그대로 사용할 수
    없으므로 `user.user-id`처럼 접근할 수 없습니다. 이 경우에는 `user["user-id"]`처럼
    대괄호 표기법을 사용해야 합니다.
    
    객체 리터럴에서 `1: "one"`처럼 숫자처럼 보이는 키를 작성할 수 있지만, 일반 객체의
    속성 키는 내부적으로 문자열 키로 취급됩니다. 따라서 `user[1]`은 실제로 `user["1"]`에
    접근하는 것과 같습니다.
    
    마지막 두 줄이 모두 `"one"`을 출력하는 이유가 여기에 있습니다. `1`이라는 숫자와
    `"1"`이라는 문자열은 객체 속성 접근 과정에서 같은 속성 키 `"1"`로 처리됩니다.
    
    실무에서 API 응답이나 폼 데이터처럼 키 이름이 식별자 규칙을 따르지 않는 경우에는
    대괄호 표기법이 필요합니다. 반대로 고정된 속성은 점 표기법을 쓰는 편이 읽기 쉽습니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `key`와 `[key]`의 차이를 설명하세요.

```jsx
const key = "status";
const item = { key: "literal", [key]: "ready" };

console.log(item.key);
console.log(item.status);
console.log(item[key]);
```

- 정답과 해설
    
    ### 정답
    
    ```
    literal
    ready
    ready
    ```
    
    ### 해설
    
    객체 리터럴 안에서 `key: "literal"`은 변수 `key`의 값을 사용하는 것이 아닙니다.
    속성 이름이 문자 그대로 `"key"`가 됩니다. 그래서 `item.key`는 `"literal"`입니다.
    
    반면 `[key]: "ready"`는 계산된 속성명입니다. 대괄호 안의 표현식 `key`를 먼저 평가하고,
    그 결과인 `"status"`를 속성 이름으로 사용합니다. 그래서 객체에는 `status` 속성이
    생깁니다.
    
    `item.status`는 `"ready"`입니다. `item[key]`도 `key` 변수의 값이 `"status"`이므로
    `item["status"]`와 같고, 결과는 `"ready"`입니다.
    
    동적 키를 만들 때 이 차이를 놓치면 `{ key: value }`처럼 작성해 놓고 실제로는 변수
    이름이 아니라 `"key"`라는 고정 속성이 생기는 버그가 납니다. 상태 객체, 폼 필드 값,
    쿼리 파라미터 객체를 만들 때 자주 나오는 실수입니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, `in`과 `hasOwnProperty` 계열 검사의 차이를
설명하세요.

```jsx
const config = { theme: "dark" };

console.log(config.mode);
console.log("mode" in config);
console.log("toString" in config);
console.log(Object.prototype.hasOwnProperty.call(config, "toString"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    undefined
    false
    true
    false
    ```
    
    ### 해설
    
    `config.mode`는 존재하지 않는 속성에 접근합니다. JavaScript 객체에서 없는 속성을
    읽으면 에러가 아니라 `undefined`가 반환됩니다.
    
    `"mode" in config`는 `false`입니다. `mode`는 `config` 자신의 속성에도 없고, 프로토타입
    체인에도 없습니다.
    
    `"toString" in config`는 `true`입니다. `config` 객체에 직접 `toString`을 정의하지
    않았더라도, 일반 객체는 `Object.prototype`을 프로토타입으로 가집니다. `toString`은 그
    프로토타입 쪽에 존재하기 때문에 `in` 연산자는 `true`를 반환합니다.
    
    `Object.prototype.hasOwnProperty.call(config, "toString")`은 `false`입니다. 이 검사는
    프로토타입 체인을 보지 않고 객체 자신의 속성인지 확인합니다.
    
    정리하면 `in`은 “객체 또는 프로토타입 체인 어딘가에 있는가”를 확인하고,
    `hasOwnProperty` 계열 검사는 “이 객체가 직접 가진 속성인가”를 확인합니다.
    
    동적 키 입력을 받을 때 `in`만 사용하면 `toString`, `constructor` 같은 상속된 속성까지
    허용될 수 있습니다. 실제 데이터 필드만 허용해야 한다면 자신의 속성인지 확인해야 합니다.
    

### 문제 4. 심화

다음 코드는 설정 객체에서 키 이름으로 값을 읽습니다. 실행 결과를 예측하고, 이 코드의
문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로 개선하세요.

```jsx
const settings = { theme: "dark", pageSize: 20 };

function readSetting(key) {
  return settings[key];
}

console.log(readSetting("theme"));
console.log(readSetting("missing"));
console.log(typeof readSetting("toString"));
```

- 정답과 해설
    
    ### 정답
    
    ```
    dark
    undefined
    function
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    const settings = {
      theme: "dark",
      pageSize: 20,
    } as const;
    
    type SettingKey = keyof typeof settings;
    type SettingValue = (typeof settings)[SettingKey];
    
    function isSettingKey(key: string): key is SettingKey {
      return Object.prototype.hasOwnProperty.call(settings, key);
    }
    
    function readSetting(key: SettingKey): SettingValue {
      return settings[key];
    }
    
    function readSettingFromInput(key: string): SettingValue | null {
      if (!isSettingKey(key)) {
        return null;
      }
    
      return readSetting(key);
    }
    
    console.log(readSettingFromInput("theme"));
    console.log(readSettingFromInput("missing"));
    console.log(readSettingFromInput("toString"));
    ```
    
    ### 해설
    
    ### JS 관점
    
    `readSetting("theme")`은 `settings["theme"]`을 읽으므로 `"dark"`를 반환합니다.
    
    `readSetting("missing")`은 존재하지 않는 속성을 읽습니다. JavaScript는 이 상황에서
    에러를 던지지 않고 `undefined`를 반환합니다.
    
    세 번째 줄이 중요한 지점입니다. `settings`는 일반 객체이므로 `Object.prototype`을
    상속합니다. `settings["toString"]`은 객체 자신의 데이터 속성이 아니라 프로토타입에서
    상속된 `toString` 함수를 찾습니다. 그래서 `typeof readSetting("toString")`은
    `"function"`입니다.
    
    즉, 문자열 키를 그대로 객체 접근에 사용하면 우리가 의도한 데이터 필드뿐 아니라 상속된
    속성까지 읽을 수 있습니다. 설정, 권한, feature flag처럼 허용된 키만 읽어야 하는 코드에서는
    위험합니다.
    
    ### TS 관점
    
    TypeScript에서 `key`를 단순히 `string`으로 두면 너무 넓습니다. 문자열이라면 어떤 값이든
    들어올 수 있기 때문에 `"theme"`뿐 아니라 `"missing"`, `"toString"`도 타입상으로는 같은
    입력입니다.
    
    개선 예시에서는 `settings`에 `as const`를 붙이고, `keyof typeof settings`로 허용 가능한
    키의 union 타입을 만듭니다. `SettingKey`는 `"theme" | "pageSize"`가 됩니다.
    
    `readSetting(key: SettingKey)`는 허용된 키만 받습니다. 그래서 코드 내부에서는
    `settings[key]`가 안전합니다.
    
    하지만 외부 입력은 여전히 `string`으로 들어옵니다. URL 파라미터나 폼 입력은 처음부터
    `SettingKey`가 아닙니다. 그래서 `isSettingKey`가 필요합니다. 이 함수는
    `hasOwnProperty` 방식으로 객체 자신의 속성인지 검사하고, TypeScript에는
    `key is SettingKey`라는 타입 가드로 알려 줍니다.
    
    ### 실무 관점
    
    동적 키 접근은 편리하지만 허용 범위를 분명히 하지 않으면 유지보수 비용이 커집니다.
    특히 다음 상황에서는 문자열 키를 그대로 객체 접근에 넣지 않는 편이 좋습니다.
    
    1. 사용자가 직접 입력한 키를 처리할 때
    2. API 쿼리 파라미터로 정렬 기준이나 필터 키를 받을 때
    3. 권한, 설정, feature flag처럼 허용된 목록이 중요한 값을 읽을 때
    4. 상속된 속성과 실제 데이터 속성을 구분해야 할 때
    
    좋은 흐름은 “외부 입력은 문자열로 받고, 허용 목록 검사로 좁힌 뒤, 좁혀진 키만 내부
    객체 접근에 사용한다”입니다. TypeScript의 `keyof`는 이 허용 목록을 타입으로 표현하는 데
    도움이 되고, 런타임의 `hasOwnProperty` 검사는 실제 입력을 안전하게 걸러 줍니다.

그대로 객체 접근에 사용해도 되는 경우와, `keyof`나 허용 목록 검증이 꼭 필요한 경우를
