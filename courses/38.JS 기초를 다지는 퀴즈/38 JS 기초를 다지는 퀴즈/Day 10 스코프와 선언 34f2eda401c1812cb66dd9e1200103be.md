# Day 10. 스코프와 선언

### 오늘의 목표

JavaScript에서 식별자를 찾는 방식은 스코프에 의해 결정됩니다. 오늘은 함수 스코프,
블록 스코프, `var`, `let`, `const`, shadowing을 복습하고, 전역 또는 모듈 수준 상태가
실무 코드의 예측 가능성을 어떻게 낮추는지 TypeScript 설계와 함께 확인합니다.

### 레퍼런스 연결

- 스코프와 클로저 챕터 복습
- 핵심 복습 포인트: 전역 스코프, 함수 스코프, 블록 스코프, `var`, `let`, `const`
- 다시 볼 질문: 전역 상태는 언제 편리하고 언제 위험해질까?

### 문제 1. 기본

다음 코드의 실행 결과를 순서대로 작성하세요.

```jsx
function scopeTest() {
  var a = "function";
  let b = "block outer";

  if (true) {
    var a = "changed";
    let b = "block inner";

    console.log(a);
    console.log(b);
  }

  console.log(a);
  console.log(b);
}

scopeTest();
```

- 정답과 해설
    
    ### 정답
    
    ```
    changed
    block inner
    changed
    block outer
    ```
    
    ### 해설
    
    `var`는 함수 스코프를 가집니다. `scopeTest` 안에서 `var a = "function"`을 선언했고,
    `if` 블록 안의 `var a = "changed"`는 새로운 블록 스코프 변수를 만드는 것이 아니라 같은
    함수 스코프의 `a`를 다시 선언하고 값을 바꿉니다.
    
    그래서 `if` 블록 안에서 `console.log(a)`는 `"changed"`를 출력합니다. 블록이 끝난 뒤에도
    같은 함수 스코프의 `a`가 바뀐 상태이므로 다시 `"changed"`가 출력됩니다.
    
    `let`은 블록 스코프를 가집니다. 바깥의 `let b = "block outer"`와 `if` 블록 안의
    `let b = "block inner"`는 서로 다른 변수입니다.
    
    블록 안에서 `console.log(b)`는 가장 가까운 블록 스코프의 `b`를 찾으므로 `"block inner"`가
    출력됩니다. 블록 밖에서는 안쪽 `b`에 접근할 수 없고, 바깥 `b`가 보이므로 `"block outer"`가
    출력됩니다.
    
    핵심은 `var`는 블록을 기준으로 새 스코프를 만들지 않고, `let`과 `const`는 블록을 기준으로
    스코프를 만든다는 점입니다.
    

### 문제 2. 중급

다음 코드의 실행 결과를 순서대로 작성하고, 반복문에서 `var`와 `let`의 스코프 차이를
설명하세요.

```jsx
for (var i = 0; i < 3; i += 1) {}

console.log(i);

try {
  for (let j = 0; j < 3; j += 1) {}

  console.log(j);
} catch (error) {
  console.log(error.constructor.name);
}
```

- 정답과 해설
    
    ### 정답
    
    ```
    3
    ReferenceError
    ```
    
    ### 해설
    
    `for (var i = 0; i < 3; i += 1) {}`에서 `i`는 블록 스코프가 아니라 현재 함수 또는 전역
    스코프에 속합니다. 이 코드를 최상위에서 실행하면 반복문이 끝난 뒤에도 `i`에 접근할 수
    있습니다.
    
    반복은 `i`가 `0`, `1`, `2`일 때 실행되고, 마지막 증가 후 `i`는 `3`이 됩니다. 조건
    `i < 3`이 `false`가 되어 반복이 끝나므로 `console.log(i)`는 `3`을 출력합니다.
    
    반면 `for (let j = 0; j < 3; j += 1) {}`의 `j`는 반복문 블록에 묶입니다. 반복문 밖에서는
    `j`를 참조할 수 없습니다.
    
    그래서 `console.log(j)`를 실행하면 `ReferenceError`가 발생하고, `catch` 블록에서
    `error.constructor.name`을 출력해 `"ReferenceError"`가 나옵니다.
    
    반복문 바깥으로 카운터가 새어 나가는 것은 예전 JavaScript 코드에서 흔한 혼란의 원인입니다.
    새 코드에서는 특별한 이유가 없다면 `let` 또는 `const`를 기본으로 사용하는 편이 안전합니다.
    

### 문제 3. 중급

다음 코드의 실행 결과를 순서대로 작성하고, shadowing이 무엇인지 설명하세요.

```jsx
const label = "global";

function printLabel() {
  const label = "function";

  {
    const label = "block";
    console.log(label);
  }

  console.log(label);
}

printLabel();
console.log(label);
```

- 정답과 해설
    
    ### 정답
    
    ```
    block
    function
    global
    ```
    
    ### 해설
    
    JavaScript는 식별자를 찾을 때 현재 스코프에서 시작해 바깥 스코프로 이동합니다. 가장
    가까운 스코프에 같은 이름의 변수가 있으면 그 변수를 사용합니다.
    
    전역 스코프에는 `label = "global"`이 있습니다. `printLabel` 함수 안에는 또 다른
    `label = "function"`이 있습니다. 그리고 블록 안에는 다시 `label = "block"`이 있습니다.
    
    블록 안의 `console.log(label)`은 가장 가까운 블록 스코프의 `label`을 사용하므로
    `"block"`을 출력합니다.
    
    블록이 끝난 뒤 함수 안의 `console.log(label)`은 함수 스코프의 `label`을 사용하므로
    `"function"`을 출력합니다.
    
    함수 호출이 끝난 뒤 전역의 `console.log(label)`은 전역 스코프의 `label`을 사용하므로
    `"global"`을 출력합니다.
    
    이처럼 안쪽 스코프의 식별자가 바깥 스코프의 같은 이름 식별자를 가리는 현상을 shadowing
    이라고 부릅니다. shadowing 자체가 항상 나쁜 것은 아니지만, 이름이 너무 넓거나 의미가
    비슷하면 어떤 값을 읽는지 헷갈리기 쉽습니다.
    

### 문제 4. 심화

다음 코드는 현재 사용자를 모듈 수준 변수에 저장하고 인사말을 만듭니다. 실행 결과를
예측하고, 이 코드의 문제점을 JS 관점과 TS 관점에서 설명한 뒤 더 안전한 TypeScript 코드로
개선하세요.

```jsx
let currentUser = "Ada";

function setCurrentUser(name) {
  currentUser = name;
}

function greet() {
  return `Hello,${currentUser}`;
}

console.log(greet());
setCurrentUser("Lin");
console.log(greet());
```

- 정답과 해설
    
    ### 정답
    
    ```
    Hello, Ada
    Hello, Lin
    ```
    
    안전하게 개선한 TypeScript 예시는 다음과 같습니다.
    
    ```tsx
    type User = {
      name: string;
    };
    
    type GreetingContext = Readonly<{
      currentUser: User;
    }>;
    
    function createGreetingService(context: GreetingContext) {
      return {
        greet(): string {
          return `Hello,${context.currentUser.name}`;
        },
      };
    }
    
    const adaService = createGreetingService({ currentUser: { name: "Ada" } });
    const linService = createGreetingService({ currentUser: { name: "Lin" } });
    
    console.log(adaService.greet());
    console.log(linService.greet());
    ```
    
    ### 해설
    
    ### JS 관점
    
    처음 `currentUser`는 `"Ada"`입니다. `greet()`는 이 모듈 수준 변수를 읽어
    `"Hello, Ada"`를 반환합니다.
    
    그 다음 `setCurrentUser("Lin")`이 같은 변수 `currentUser`를 `"Lin"`으로 바꿉니다.
    두 번째 `greet()` 호출은 바뀐 값을 읽으므로 `"Hello, Lin"`을 반환합니다.
    
    작은 예제에서는 자연스럽게 보이지만, 모듈 수준의 변경 가능한 상태는 코드가 커질수록
    예측하기 어려워집니다. 어느 함수가 언제 값을 바꿨는지 추적해야 하고, 테스트도 실행
    순서에 영향을 받을 수 있습니다.
    
    특히 서버 코드에서 요청별 사용자 정보를 전역 또는 모듈 수준 변수에 저장하면 서로 다른
    요청의 상태가 섞이는 심각한 문제가 생길 수 있습니다.
    
    ### TS 관점
    
    TypeScript는 `currentUser`가 문자열인지, `setCurrentUser`가 문자열을 받는지 같은 타입
    오류는 잡아줄 수 있습니다. 하지만 “이 모듈 수준 상태가 여러 호출 사이에서 공유되어도
    괜찮은가”라는 설계 문제를 자동으로 해결하지는 않습니다.
    
    개선 예시에서는 현재 사용자를 전역처럼 공유하지 않고 `GreetingContext`로 주입합니다.
    `createGreetingService`는 컨텍스트를 받아 해당 컨텍스트에 묶인 서비스를 만듭니다.
    
    `GreetingContext`에 `Readonly`를 붙여 서비스 내부에서 컨텍스트 자체를 바꾸지 않는다는
    의도를 타입으로 표현했습니다. 이 예시에서는 `adaService`와 `linService`가 서로 다른
    컨텍스트를 가지므로, 한쪽 사용자가 다른 쪽 결과에 영향을 주지 않습니다.
    
    ### 실무 관점
    
    스코프는 단순히 변수가 어디서 보이는지의 문제가 아닙니다. 상태가 어느 범위에서 공유되는지,
    누가 변경할 수 있는지를 결정합니다.
    
    좋은 흐름은 다음과 같습니다.
    
    1. 변경 가능한 상태의 범위를 가능한 좁힌다.
    2. 함수가 필요한 값은 전역에서 읽기보다 인자로 받거나 컨텍스트로 주입한다.
    3. 테스트마다 새 컨텍스트나 새 인스턴스를 만들어 상태 공유를 줄인다.
    4. 모듈 수준 상태가 꼭 필요하다면 초기화와 변경 규칙을 명확히 문서화한다.
    
    TypeScript는 상태의 모양을 표현하는 데 도움을 주지만, 상태의 공유 범위와 생명주기는
    JavaScript 스코프와 애플리케이션 구조를 이해하고 설계해야 합니다.
