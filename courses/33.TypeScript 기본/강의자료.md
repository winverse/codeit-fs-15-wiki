## **1-02. TypeScript 기본기 ⭐️⭐️⭐️**

### 타입이 왜 필요할까?

USB-A 포트에 USB-C 케이블을 꽂으려 하면 물리적으로 안 들어갑니다. 형태가 다르니까요. 억지로 넣으려 해도 꽂히지 않는다는 걸 **그 자리에서 바로** 알 수 있습니다.

자판기 동전 투입구도 마찬가지입니다. 동전이 들어가야 할 자리에 지폐를 넣으려 하면, 지폐는 들어가지 않습니다. 음료가 안 나오는 걸 경험하기 전에, **넣는 순간** 이미 안 된다는 걸 알 수 있죠.

두 경우의 공통점이 있습니다. “결과를 보기 전에, 맞지 않는다는 사실을 즉시 알 수 있다”는 것입니다.

이 상황을 프로그래밍에 그대로 대입하면, 자바스크립트는 포트 모양 구분이 없는 컴퓨터에 해당합니다. 어떤 케이블이든 일단 꽂힙니다. 맞는지 틀린지는 전원을 켜봐야 알 수 있죠. 코드도 마찬가지로, 어떤 값이든 어떤 변수에든 일단 들어갑니다. 오류가 있는지는 **코드를 실행해봐야** 알 수 있습니다.

실제 개발에서 이 문제가 어떤 식으로 나타나는지 구체적인 사례로 확인하면 이렇습니다. 쇼핑몰 프로젝트에서 상품 가격을 계산하는 코드를 작성했다고 가정합니다. 처음엔 잘 동작합니다. 몇 주 후, 다른 팀원이 가격 값을 `"129000원"` 같은 문자열로 변경했고, 해당 변경이 연산에 영향을 미친다는 사실을 인지하지 못한 채로 커밋했습니다.

그 순간부터 할인 계산은 `NaN`을 반환하고, 총 금액은 `0원`이 됩니다. 하지만 이 버그는 바로 발견되지 않습니다. 며칠이 지나고, 테스터나 실제 사용자가 “결제 금액이 이상해요”라고 신고하고서야 알게 됩니다. 그때부터 원인을 찾기 위해 수십 줄의 코드를 역추적해야 합니다.

오류를 나중에 발견할수록 수정 비용이 커집니다. 작성 시점에 발견하면 그 자리에서 바로 고치면 됩니다. 배포 후 사용자가 신고하면, 원인을 찾고 수정하고 다시 배포하는 전 과정을 반복해야 합니다. TypeScript가 주는 가장 큰 가치는 바로 이겁니다. **실수를 없애는 게 아니라, 실수가 가장 저렴한 순간에 발견하게 해주는 것**입니다.

### 타입이 없으면 생기는 문제

자바스크립트로 상품 할인 가격을 계산하는 코드를 작성한 상황입니다.

```jsx
// JavaScript
const product = {
  name: "코드잇 후드",
  price: 129_000,
};

// 어딘가에서 실수로 price에 문자열을 넣었습니다
product.price = "129000원";

// 나중에 할인 계산을 시도합니다
const salePrice = product.price * 0.9;
console.log(salePrice); // NaN
```

결과: `NaN`

왜 이런 일이 생겼을까요? `'129000원'`은 문자열이라 곱셈이 안 됩니다. 하지만 **자바스크립트는 이 실수를 코드를 실행하기 전까지 절대 알려주지 않습니다.** 실제 서비스라면 사용자가 잘못된 가격을 보게 되고, 그제서야 버그를 발견하게 됩니다.

이처럼 실행 중에 발생하는 오류를 **런타임 오류**라고 합니다. 자바스크립트에서 런타임 오류가 유독 많이 발생하는 이유는, 변수에 어떤 타입의 값이든 마음대로 넣을 수 있는 **동적 타이핑** 언어이기 때문입니다.

오류가 발견되는 시점을 비교해보면 차이가 확연합니다.

```
 JavaScript       코드 작성 → 빌드 → 배포 → 테스트 → 사용자 신고 → 그제서야 발견
                                                               ↑
                                                         이 시점까지 모릅니다

 TypeScript       코드 작성 → ❌ 바로 발견
                     ↑
            이 시점에 바로 알 수 있습니다
```

### TypeScript가 해결하는 방법

같은 코드를 TypeScript로 작성하면 동작 방식이 달라집니다.

```tsx
// TypeScript
const product = {
  name: "코드잇 후드",
  price: 129_000, // TypeScript가 price를 number 타입으로 기억합니다
};

product.price = "129000원";
//              ^^^^^^^^^^
// ❌ 에디터에서 빨간 줄: Type 'string' is not assignable to type 'number'
```

코드를 실행하기도 전에, **에디터나 TypeScript 컴파일러가 코드를 검사하는 순간** 오류가 표시됩니다. 문자열을 숫자 자리에 넣으려 했다는 사실을 즉시 알 수 있죠.

이처럼 실행 전에 타입을 검사하는 방식을 **정적 타이핑(Static Typing)**이라고 합니다. 정적 타이핑 언어에서는 변수에 어떤 종류의 값이 들어와야 하는지를 미리 정해 두고, 그 약속이 지켜지는지 코드를 실행하기 전에 확인합니다. 반면 JavaScript는 **동적 타이핑(Dynamic Typing)** 언어로, 타입 검사가 코드를 실제로 실행하는 시점에 이루어집니다. TypeScript는 JavaScript 위에 정적 타이핑 규칙을 더해, 실행 중에 드러나야 했던 타입 관련 오류를 작성 단계에서 먼저 발견할 수 있게 만든 언어입니다.

아래 예시를 보면 두 방식의 차이가 분명해집니다. 실수로 인자 순서를 바꿔 넣었을 때입니다.

```jsx
// JavaScript (동적 타이핑) — 실행해봐야 문제를 알 수 있습니다
function repeat(str, count) {
  return str.repeat(count);
}

repeat(3, "hello");
// 실행 결과: TypeError: str.repeat is not a function
// 숫자 3에는 repeat() 메서드가 없어서 실행 중에 충돌합니다
```

```tsx
// TypeScript (정적 타이핑) — 에디터나 TypeScript 컴파일러가 코드를 검사하는 순간 오류가 표시됩니다
function repeat(str: string, count: number) {
  return str.repeat(count);
}

repeat(3, "hello");
// ❌ Argument of type 'number' is not assignable to parameter of type 'string'
// ❌ Argument of type 'string' is not assignable to parameter of type 'number'
```

두 코드의 로직은 동일합니다. 차이는 오류를 **언제 발견하느냐**에 있습니다. JavaScript는 실행 후, TypeScript는 에디터나 컴파일러가 코드를 검사하는 단계에서 발견합니다.

### 정적 타이핑의 이점

정적 타이핑이 주는 이점은 오류 조기 발견에서 끝나지 않습니다. 실무에서 자주 체감하는 다른 이점이 세 가지 더 있습니다.

**코드 자체가 문서가 됩니다.** 여기서 **함수 시그니처(Function Signature)**란 함수 이름, 매개변수의 타입, 반환 타입을 합친 선언부를 말합니다. 실제 구현 내용(`{ ... }` 안)은 포함되지 않습니다. 이 시그니처만 봐도 함수가 무엇을 받고 무엇을 돌려주는지 알 수 있어, 주석 없이도 함수의 계약이 명확해집니다.

```tsx
// 함수 시그니처만 봐도 무엇을 주고 무엇을 받는지 명확합니다
function sendEmail(to: string, subject: string, body: string): boolean { ... }

// JavaScript였다면 내부를 읽어봐야 알 수 있습니다
function sendEmail(to, subject, body) { ... }
```

**타입을 바꾸면 모든 연관 오류가 한 번에 표시됩니다.** 프로젝트 규모가 커지면 하나의 타입이 수십 군데 쓰입니다. TypeScript는 타입 정의를 수정하는 순간, 그 타입을 잘못 쓰고 있는 모든 위치를 즉시 표시합니다.

아래 예시는 개념 설명을 위한 축약 예시입니다.

> [!NOTE]
축약 예시이지만, `types.ts`에 `stock` 프로퍼티를 추가하면 연관 파일 전체에 오류가 표시된다고 보면 됩니다.
> 

**`types.ts`**

```tsx
// types.ts
interface Product {
  name: string;
  price: number;
  stock: number; // 새로 추가
}
```

에디터나 TypeScript 컴파일러가 코드를 검사하는 순간, `Product`를 사용하는 모든 파일에 일제히 오류가 표시됩니다.

**`a.ts`**

```tsx
// a.ts
const hoodie: Product = { name: "후드", price: 49_000 };
//                       ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
// ❌ Property 'stock' is missing in type '{ name: string; price: number; }'
```

**`b.ts`**

```tsx
// b.ts
const tshirt: Product = { name: "티셔츠", price: 29_000 };
//                        ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
// ❌ Property 'stock' is missing in type '{ name: string; price: number; }'
```

`types.ts` 하나를 수정했을 뿐인데, `a.ts`와 `b.ts`가 각각 어느 위치에서 `stock`을 빠뜨렸는지 에디터가 파일을 넘나들며 즉시 표시합니다. 파일이 수십 개로 늘어나도 마찬가지입니다.

반면에 JavaScript로 같은 작업을 하려면 `stock`을 어디서 썼는지 직접 전체 검색해야 합니다.

**에디터가 훨씬 똑똑해집니다.** 타입 정보가 있으면 점(`.`)을 찍는 순간 에디터가 사용 가능한 프로퍼티를 자동으로 나열합니다.

```tsx
const product = {
  name: '코드잇 후드',
  price: 129_000,
  membersOnly: true,
};

product.  // ← 점을 찍는 순간
          // 에디터가 name, price, membersOnly를 자동완성으로 보여줍니다
```

프로퍼티 이름의 오타도 바로 잡아줍니다. 자바스크립트였다면 그냥 `undefined`를 반환하고 끝나지만, TypeScript는 실행 전에 알려줍니다.

```tsx
console.log(product.naem);
//                   ^^^^
// ❌ Property 'naem' does not exist on type '{ name: string; price: number; ... }'
```

이점을 정리하면 아래와 같습니다.

| 이점 | JavaScript | TypeScript |
| --- | --- | --- |
| 잘못된 타입 대입 | 실행 후에야 발견 | 코드 검사 단계에서 표시 |
| 없는 프로퍼티/오타 접근 | `undefined` 반환 | 코드 검사 단계에서 표시 |
| 함수 시그니처 파악 | 내부 코드를 직접 읽어야 | 타입 선언만 봐도 즉시 파악 |
| 타입 변경 후 누락 위치 | 전체 검색으로 직접 확인 | 오류로 즉시 표시 |
| 프로퍼티 자동완성 | 지원 없음 | 타입 기반으로 정확하게 제공 |

### TypeScript는 어떻게 타입을 검사하는 건가요?

자동차 정기검사에서 검사관은 차를 위아래로 살펴보며 각 부품 자리에 규격에 맞는 부품이 장착됐는지 확인합니다. 타이어 자리에는 타이어, 휠에는 정해진 볼트 규격대로 체결됐는지 순서대로 점검합니다. 규격이 맞지 않는 부품이 발견되면 도로에 나가기 전에 그 자리에서 바로 지적합니다.

TypeScript의 **타입 검사기(Type Checker)**가 정확히 이 방식으로 동작합니다. 코드를 위에서 아래로 읽으며, 각 변수에 어떤 타입이 들어가야 하는지 기록해 둡니다. 이후 그 자리에 맞지 않는 타입의 값이 들어오면, 실행 전에 즉시 오류를 표시합니다.

```
 에디터나 TypeScript 컴파일러가 코드를 검사하는 순간 타입 검사기가 동작합니다

 let price = 129_000;          ← ① price는 number 타입으로 기억
 price = '129000원';           ← ② number 자리에 string?
                                    → ❌ 빨간 줄 표시
 const sale = price * 0.9;    ← ③ (②가 막혔으므로 여기까지 오지 않음)
```

검사기가 하는 일을 단계별로 보면 이렇습니다.

| 단계 | 하는 일 | 예시 |
| --- | --- | --- |
| ① | 각 변수의 타입을 기록합니다 | `129_000` → `number`로 기억 |
| ② | 대입·연산이 일어날 때 타입이 맞는지 확인 | `number` 자리에 `string` → 불일치 |
| ③ | 맞지 않으면 에디터에 오류 표시 | 빨간 줄 + 오류 메시지 |

중요한 것은, 이 모든 검사가 **코드를 실행하기 전**에 일어난다는 점입니다. 자바스크립트에서는 실행해봐야 알았던 오류를, TypeScript는 에디터나 컴파일러가 코드를 검사하는 단계에서 먼저 잡아냅니다.

### 그런데 TypeScript는 JavaScript와 별개의 언어인가요?

아닙니다. TypeScript는 JavaScript의 **슈퍼셋(Superset)**입니다. 슈퍼셋이란 “한쪽이 다른 쪽을 완전히 포함한다”는 의미입니다. 구체적으로 말하면, **모든 JavaScript 코드는 TypeScript 코드이기도 합니다.** `.ts` 파일에 기존 JavaScript 코드를 그대로 붙여 넣어도 수정 없이 동작합니다. 반대로 TypeScript 코드가 항상 JavaScript인 것은 아닙니다. 타입 선언처럼 TypeScript 전용 문법은 JavaScript에 없기 때문입니다.

```
╔══════════════════════════════════════╗
║  TypeScript                          ║
║                                      ║
║  ╔══════════════════════════╗        ║
║  ║  JavaScript              ║        ║
║  ║  (JS 코드 = TS 코드)       ║        ║
║  ╚══════════════════════════╝        ║
║                                      ║
║  + 타입 문법 (TS 전용, JS에는 없음)        ║
╚══════════════════════════════════════╝

  JS → TS  ✅ 항상 성립 (JS는 TS의 부분집합)
  TS → JS  ❌ 항상 성립하지 않음 (타입 문법은 JS에 없음)
```

이 때문에 TypeScript 코드는 실행 전에 타입 선언을 제거하고 순수한 JavaScript로 변환하는 컴파일 과정을 거칩니다. 타입 문법은 개발 단계에서 오류를 검사하는 용도로만 존재하며, 실행 시점에는 흔적 없이 제거됩니다.

```
TypeScript 코드 작성  →  타입 검사  →  JavaScript로 변환  →  실행
      (.ts)                               (.js)
```

이 과정을 글로만 보면 다소 추상적으로 느껴질 수 있습니다. 그래서 아주 짧은 예시를 보면, 타입 문법이 컴파일 뒤에 어떻게 사라지는지 바로 확인할 수 있습니다.

예를 들어 `example.ts` 파일에 다음과 같이 코드를 작성했다고 가정합니다.

```tsx
let price: number = 129_000;
let productName: string = "후드 집업";

console.log(price, productName);
```

실제로는 TypeScript 설치가 끝난 뒤 터미널에서 아래 명령으로 이 파일을 JavaScript로 변환합니다.

```bash
pnpm exec tsc example.ts
```

그러면 같은 위치에 `example.js` 파일이 생성되고, 실행 환경에는 아래와 같은 JavaScript만 남습니다.

```jsx
let price = 129_000;
let productName = "후드 집업";

console.log(price, productName);
```

여기서 중요한 점은 `: number`, `: string` 같은 타입 선언이 모두 제거되었다는 사실입니다. 즉, 타입은 개발 중 오류를 잡기 위해 존재하고, 실제 실행은 변환된 JavaScript가 담당합니다.

이후 2장에서는 직접 TypeScript 프로젝트를 만들고, 방금 살펴본 타입 검사와 컴파일 과정이 실제 개발 흐름에서 어떻게 이어지는지 확인합니다.

## **2-02. TypeScript가 실행되는 과정**

레시피 책을 보고 요리를 시작하려면, 먼저 재료가 다 있는지·순서는 맞는지 확인합니다. 그래야 중간에 “소금이 없다!”는 상황을 막을 수 있기 때문입니다.

TypeScript 컴파일러가 하는 역할이 바로 그겁니다. 코드를 실행하기 전에 먼저 **타입 검사**를 통해 오류가 있는지 확인하고, 통과하면 실행 가능한 JavaScript로 변환합니다.

```
 ┌─────────────────────────────────────────────────┐
 │                                                 │
 │  TypeScript 코드 (.ts)                           │
 │         │                                       │
 │         ▼                                       │
 │  ┌─────────────────┐                            │
 │  │  1단계: 타입 검사   │ ← 오류 있으면 여기서 멈춤       │
 │  └─────────────────┘                            │
 │         │  통과                                  │
 │         ▼                                       │
 │  ┌─────────────────┐                            │
 │  │  2단계: 트랜스파일  │ ← 타입 코드 제거              │
 │  └─────────────────┘                            │
 │         │                                       │ 
 │         ▼                                       │
 │  JavaScript 코드 (.js)                           │
 │         │                                       │
 │         ▼                                       │
 │  3단계: 실행 (Node.js / 브라우저)                   │
 │                                                 │
 └─────────────────────────────────────────────────┘
```

**중요한 점**: 타입 선언 코드는 JavaScript 파일에 남지 않습니다.

```tsx
// TypeScript 코드 (main.ts)
let price: number = 129_000;
```

```jsx
// 변환된 JavaScript 코드 (main.js)
let price = 129_000; // ← ': number' 가 사라집니다
```

타입은 검사 단계에서만 사용되고, 실행에는 영향을 주지 않습니다.

## **2-03. 타입을 정하는 법**

물건에 이름표를 붙일 때, 어떤 물건은 모양만 봐도 “이건 가위구나” 하고 바로 알 수 있지만, 어떤 물건은 모양만으로는 알 수 없어 이름표를 직접 붙여야 합니다.

TypeScript에서 타입을 정하는 방식도 두 가지입니다.

```
 ┌──────────────────────────────────────────────┐
 │   타입을 정하는 두 가지 방법                       │
 │                                              │
 │  자동 추론             │   직접 선언             │
 │  ─────────────      │   ─────────────       │
 │  let size = 100     │   let size: number    │
 │       ↑             │          ↑            │
 │  값을 보고 TypeScript │  내가 직접 타입 지정       │
 │  가 알아서 파악        │                        │
 └──────────────────────────────────────────────┘
```

**방법 1: 타입 추론** — 값을 바로 할당하면 TypeScript가 자동으로 타입을 추론합니다.

파일: `example1.ts`

```tsx
let size = 100;
```

`size` 위에 마우스를 올려보면 `number` 타입으로 잡혀 있습니다. TypeScript가 `100`이라는 값을 보고 숫자형이라고 판단한 것입니다. 이후 다른 타입을 넣으면 오류가 납니다.

```tsx
let size = 100;
size = "크다"; // ❌ 오류: number 타입에 string을 넣을 수 없습니다
```

**방법 2: 명시적 선언** — 변수 이름 뒤에 `: 타입`을 직접 적습니다. 값을 바로 넣지 않을 때 특히 유용합니다.

```tsx
let username: string; // 타입만 먼저 선언
let age: number = 20; // 타입 선언 + 값 할당
let isLoggedIn: boolean;

username = "codeit"; // OK
username = 123; // ❌ 오류
```

| 타입을 정하는 방식 | 예시 | 적합한 상황 |
| --- | --- | --- |
| 타입 추론 | `let size = 100;` | 값을 즉시 할당하고 타입이 분명할 때 |
| 명시적 선언 | `let size: number = 100;` | 의도를 더 명확하게 남기고 싶을 때 |
| 선언 후 나중에 할당 | `let size: number;` | 초기값 없이 타입부터 정해야 할 때 |

> **참고**: 타입 선언 코드는 JavaScript로 변환될 때 사라집니다. 타입은 개발 단계에서 오류를 잡아내는 용도이고, 실제 실행에는 영향을 주지 않습니다.
>

## **2-04. 타입 오류 이해하기 ⭐️⭐️⭐️**

건물 안내도를 예로 들면, 처음에는 “이 건물에 뭔가 문제가 있습니다”라는 큰 표지판만 보이지만, 층별 안내도를 따라가면 “3층 304호의 소화기가 없습니다”처럼 정확한 위치를 특정할 수 있습니다.

TypeScript의 오류 메시지도 마찬가지입니다. **바깥에서 안쪽으로 좁혀 읽으면** 정확한 원인에 도달할 수 있습니다.

**예시 1: 프로퍼티 타입 불일치**

파일: `example1.ts`

```tsx
const product = {
  id: "c001",
  name: "라이트 윈드 브레이커",
  price: 129_000,
};

product.price = "139000원"; // ❌ 오류

const salePrice = product.price * 0.9;
console.log(`할인 가격:${salePrice}`);
export {};
```

`price`가 처음에 `number`로 추론됐기 때문에, `string`을 넣으면 오류가 납니다. 이후 `* 0.9` 같은 숫자 연산이 깨지기 때문입니다.

**예시 2: 객체 대입 실패 — 오류를 단계별로 읽기**

파일: `example2.ts`

```tsx
let product = {
  id: "c001",
  name: "라이트 윈드 브레이커",
  price: 129_000,
  sizes: ["M", "L", "XL"], // string[]
};

const newProduct = {
  id: "c002",
  name: "다크 윈드 브레이커",
  price: 139_000,
  sizes: [90, 95, 100], // number[]
};

product = newProduct; // ❌ 오류
export {};
```

오류 메시지가 길 때는 다음 순서로 읽으면 됩니다.

| 오류 읽기 단계 | 확인할 질문 | 이 예시에서의 의미 |
| --- | --- | --- |
| 1. 전체 대입 | 어떤 문장이 실패했는가 | `product = newProduct` 대입이 실패했습니다 |
| 2. 프로퍼티 | 어디서 구조가 달라졌는가 | `sizes` 프로퍼티가 문제입니다 |
| 3. 타입 수준 | 어떤 타입끼리 충돌하는가 | `number[]`를 `string[]` 자리에 넣을 수 없습니다 |

오류 메시지가 길수록 단서가 많습니다. 바깥에서 안쪽으로 좁혀 읽으면 됩니다.

> [!NOTE]
**왜 파일이 다른데 이름이 충돌하나요?**
TypeScript 파일에 `import`나 `export`가 하나도 없으면 전역 스코프(Script)로 취급되어 다른 파일과 변수 이름이 겹칠 수 있습니다. 각 파일을 독립된 모듈로 분리하려면 파일 끝에 `export {};`를 추가하면 됩니다.
> 

## **2-05. 기본형 ⭐️⭐️⭐️**

JavaScript에서 값은 크게 두 그룹으로 나뉩니다. 숫자·문자열처럼 변수에 값 자체가 담기는 **원시타입(primitive type)** 과, 배열·객체처럼 변수에 실제 데이터의 위치(참조)가 담기는 **참조타입(reference type)** 입니다. TypeScript는 이 두 그룹에 각각 이름을 붙여 타입으로 선언합니다.

JavaScript의 원시타입에는 `symbol`, `bigint`도 포함되지만, 이 절에서는 현재 실습에서 바로 쓰게 되는 다섯 가지를 먼저 다룹니다. 참조타입(배열, 튜플, 객체)은 2-07과 2-09에서 이어서 다룹니다.

```
┌────────────────────────────────────────────────────────────────────┐
│                 이번 절에서 먼저 다루는 원시타입                           │
│                                                                    │
│  값의 종류        TypeScript 표기      예시                            │
│  ───────────     ───────────────   ────────────────────────────    │
│  문자열            string           'hello', "codeit", `템플릿`       │
│  숫자             number           42, 12.5, NaN, Infinity          │
│  불린(참/거짓)      boolean          true, false                      │
│  초기화 안 됨       undefined        변수 선언 후 값 미할당 상태            │
│  의도적 빈 값       null             명시적으로 "없음"을 나타냄             │
└────────────────────────────────────────────────────────────────────┘
```

> **주의**: 타입 이름은 반드시 **소문자**로 씁니다. `String`이 아니라 `string`, `Number`가 아니라 `number`입니다. 대문자로 시작하는 `String`, `Number`, `Boolean`은 TypeScript에서 다른 의미(래퍼 객체 타입)를 가지므로 원시타입에는 사용하지 않습니다.
> 

### string

상품 이름, 사용자 입력, 안내 문구처럼 글자로 이루어진 값은 대부분 `string`으로 다룹니다. 작은따옴표(`'`), 큰따옴표(`"`), 백틱(```) 중 어떤 표기법을 써도 TypeScript는 모두 문자열로 처리합니다.

```tsx
let itemName: string = "코드잇 블랙 후드";
let greet: string = `안녕하세요,${itemName}`;

// itemName = 42;  // ❌ 오류: number를 string 자리에 넣을 수 없습니다
```

### number

가격, 할인율, 재고 수량처럼 계산에 참여하는 값은 `number`로 다룹니다. JavaScript는 정수와 소수를 따로 나누지 않기 때문에 TypeScript도 하나의 `number` 타입으로 묶습니다. `NaN`(Not a Number)과 `Infinity`도 `number` 타입에 속합니다.

```tsx
let price: number = 129_000;
let rate: number = 0.1;
let stock: number = NaN;
```

문제는 `NaN`도 `number`라는 점입니다. 타입만 맞는다고 해서 곧바로 정상적인 숫자라고 단정할 수 없으므로, 계산 전에 유효성 검사가 필요할 때가 있습니다.

```tsx
function applyDiscount(price: number): number {
  if (Number.isNaN(price)) {
    return 0; // 비정상 입력 처리
  }
  return price * 0.9;
}
```

### boolean

로그인 여부, 회원 전용 여부처럼 켜짐/꺼짐이나 참/거짓 상태만 표현하면 되는 값은 `boolean`으로 다룹니다. `boolean`은 `true` 또는 `false` 두 값만 가지며, 조건 분기나 상태 플래그에 주로 쓰입니다.

```tsx
let isLoggedIn: boolean = false;
let membersOnly: boolean = true;
```

### undefined와 null

`undefined`와 `null`은 모두 “값이 없다”는 뜻이지만, 비어 있는 이유가 다릅니다. 이 차이를 구분해 두면 API 응답, DB 조회 결과, 폼 입력 상태를 더 정확하게 표현할 수 있습니다.

| 타입 | 의미 | 대표 상황 |
| --- | --- | --- |
| `undefined` | 아직 값이 할당되지 않음 | 변수를 선언했지만 초기화하지 않은 상태 |
| `null` | 의도적으로 “없음”을 명시한 상태 | DB 조회 결과가 없을 때, 삭제된 데이터 |

```tsx
let lastSeen: undefined;
let deletedAt: null = null;
```

실무에서 `null`은 “값이 존재하지 않는다”는 사실을 명시적으로 전달해야 할 때 자주 사용합니다. 반면 `undefined`는 변수에 아직 값이 배정되지 않은 상태를 뜻하며, 나중에 5장에서 배우는 유니온 타입과 함께 다시 다루게 됩니다.

원시타입은 값 자체가 변수에 저장됩니다. 반면 배열(`string[]`)이나 객체(`{ id: string }`)처럼 여러 값을 묶어 다루는 타입은 실제 데이터를 메모리의 다른 위치에 저장하고 변수에는 그 위치(참조)만 담습니다. 이 차이 때문에 TypeScript에서 배열·객체 타입의 표기법이 원시타입과 별도로 정의됩니다.

## **2-07. 배열과 튜플 ⭐️⭐️⭐️**

장바구니 상품 ID 목록, 사이즈 목록처럼 같은 종류의 값을 여러 개 묶어야 할 때가 있습니다. 이때는 값이 몇 개 들어올지 미리 알 수 없고, 나중에 요소를 추가하거나 제거할 수도 있습니다.

반면 좌표 `[x, y]`, 신체 치수 `[키, 허리]`, 재고 정보 `['M', 50]`처럼 각 위치가 이미 정해진 의미를 가질 때도 있습니다. 이런 값은 단순한 목록이 아니라 “첫 번째 자리에는 무엇이 오고, 두 번째 자리에는 무엇이 온다”는 약속 자체가 구조의 일부가 됩니다.

TypeScript는 이 차이를 **배열(array)** 과 **튜플(tuple)** 로 구분합니다. 배열은 같은 타입의 값을 반복해서 담는 구조이고, 튜플은 개수와 각 위치의 타입이 고정된 구조입니다.

| 구분 | 배열 | 튜플 |
| --- | --- | --- |
| 핵심 성격 | 같은 타입의 값 목록 | 위치가 의미를 갖는 고정 구조 |
| 길이 | 자유 | 고정 |
| 각 위치의 타입 | 보통 동일 | 위치마다 다를 수 있음 |
| 대표 용도 | 장바구니, 태그 목록, 사이즈 목록 | 좌표, 크기 정보, 라벨+수량 묶음 |

먼저 배열부터 보면, 중요한 점은 “같은 타입의 값을 여러 개 담는다”는 것입니다. 요소 개수가 늘어나도 타입 자체는 바뀌지 않으므로, 목록처럼 다뤄야 하는 데이터에 적합합니다. 반대로 다른 타입의 값을 섞거나, 각 위치가 고정된 의미를 갖게 만들고 싶다면 배열보다 다른 구조가 더 적절합니다.

파일: `main.ts`

```tsx
// string[] : 문자열 배열
const cart: string[] = ["c001", "c002"];
cart.push("c003"); // OK
// cart.push(123); // ❌ 오류: number는 넣을 수 없습니다

// string[][] : 배열의 배열
const carts: string[][] = [["c001", "c002"], ["c003"]];

// number[] : 신발 사이즈 목록 (길이 제한 없음)
const shoeSizes: number[] = [230, 250, 280];
shoeSizes.push(300); // OK, 언제든 추가 가능
```

이 예제에서 `cart`와 `shoeSizes`는 길이가 늘어나도 문제가 없습니다. 핵심 제약은 길이가 아니라 요소 타입이기 때문입니다. 그래서 `string[]`에는 문자열만, `number[]`에는 숫자만 들어갈 수 있습니다.

튜플은 배열과 비슷하게 대괄호를 사용하지만, 의미는 다릅니다. 튜플은 “몇 개가 들어오는가”뿐 아니라 “각 자리에 어떤 타입이 와야 하는가”까지 함께 정의합니다. 즉, 튜플은 목록이라기보다 작고 고정된 구조를 인덱스로 표현하는 방식에 가깝습니다.

```tsx
// [number, number] : 키와 허리 사이즈 (순서와 개수 고정)
const measurements: [number, number] = [175, 80];

// [string, number] : 사이즈 라벨과 재고 수량
const sizeStock: [string, number] = ["M", 50];

// measurements = [175, 80, 90]; // ❌ 오류: 개수가 맞지 않습니다
// sizeStock = [50, 'M']; // ❌ 오류: 순서가 맞지 않습니다
```

여기서는 `measurements[0]`이 키, `measurements[1]`이 허리라는 뜻을 가집니다. `sizeStock`도 첫 번째 자리는 사이즈 라벨, 두 번째 자리는 수량으로 고정되므로, 순서를 바꾸거나 개수를 늘리면 구조 자체가 달라졌다고 판단해 오류가 납니다.

튜플은 자리가 적고 의미가 분명할 때 유용합니다. 반대로 각 값에 이름을 붙여야 하거나, 구조가 길어져서 `0번`, `1번`, `2번` 위치만으로 의미를 파악하기 어려워진다면 튜플보다 객체 타입이 더 적합합니다. 그래서 TypeScript에서는 배열, 튜플, 객체를 서로 경쟁 관계로 보기보다 데이터 성격에 따라 선택하는 도구로 이해합니다.

## **2-09. 객체 타입 ⭐️⭐️⭐️**

튜플은 각 위치의 순서가 의미를 가질 때 적합하지만, 상품처럼 `id`, `name`, `price`가 서로 다른 역할을 가지는 데이터는 위치보다 이름으로 읽는 편이 자연스럽습니다. `0번 값`, `1번 값`으로 설명하기 시작하는 순간 데이터의 의미가 흐려지기 때문입니다.

이때 사용하는 것이 객체 타입(object type)입니다. 객체 타입은 프로퍼티 이름과 값의 타입을 함께 정의해서 “이 객체가 어떤 모양(shape)을 가져야 하는가”를 고정합니다. 즉, 튜플이 위치를 기준으로 구조를 정한다면, 객체 타입은 이름을 기준으로 구조를 정합니다.

| 문법 | 의미 | 대표 상황 |
| --- | --- | --- |
| `name: string` | `name` 프로퍼티는 반드시 있어야 하고 값은 문자열입니다 | 이름, ID, 가격처럼 필수 정보 |
| `membersOnly?: boolean` | 프로퍼티가 없어도 되지만, 있다면 `boolean`이어야 합니다 | 회원 전용 여부처럼 선택적 정보 |
| `[id: string]: number` | 키 이름은 정해지지 않았지만 모든 값은 `number`여야 합니다 | 상품별 재고처럼 키 개수를 모를 때 |

먼저 기본 객체 타입부터 보면, 핵심은 프로퍼티의 이름과 타입을 나란히 적는다는 점입니다. 이렇게 정의하면 필수 프로퍼티가 빠졌는지, 선택 프로퍼티가 어떤 타입이어야 하는지, 배열 같은 중첩 데이터가 어떤 구조를 가져야 하는지까지 한 번에 검사할 수 있습니다. 특히 `?`를 붙인 옵셔널 프로퍼티(optional property)는 “없어도 되지만, 있다면 이 타입이어야 한다”는 뜻을 정확하게 표현합니다.

파일: `main.ts`

```tsx
const memberProduct: {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
  sizes: string[];
} = {
  id: "c001",
  name: "코드잇 블랙 후디",
  price: 129_000,
  membersOnly: true,
  sizes: ["M", "L", "XL"],
};

const guestProduct: {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
  sizes: string[];
} = {
  id: "c002",
  name: "코드잇 화이트 후디",
  price: 119_000,
  sizes: ["S", "M"],
};

if (memberProduct.membersOnly) {
  console.log("회원 전용 상품");
}

console.log(guestProduct.membersOnly); // undefined
```

여기서 `memberProduct`는 `membersOnly`를 가지고 있고, `guestProduct`는 그 프로퍼티가 아예 없습니다. 그래도 오류가 나지 않는 이유는 `membersOnly?`가 선택 프로퍼티이기 때문입니다. 반대로 `membersOnly`가 있다면 반드시 `boolean`이어야 하므로, 문자열이나 숫자를 넣으면 구조가 맞지 않는다고 판단합니다.

하지만 모든 객체의 프로퍼티 이름을 미리 확정할 수 있는 것은 아닙니다. 재고표처럼 `c001`, `c002`, `c003`처럼 키가 계속 늘어나는 데이터는 “이름이 정확히 무엇인가”보다 “값이 어떤 타입으로 통일되어야 하는가”가 더 중요합니다. 이런 경우에는 인덱스 시그니처(index signature)를 사용합니다.

```tsx
const stock: { [id: string]: number } = {
  c001: 3,
  c002: 0,
  c003: 1,
};

stock["c004"] = 5; // OK
// stock['c005'] = '없음'; // ❌ 오류: 값은 number여야 합니다

console.log(stock["c001"], stock["c004"]);
```

`[id: string]: number`는 “키는 어떤 문자열이든 될 수 있지만, 그 키에 대응하는 값은 모두 숫자여야 한다”는 뜻입니다. 여기서 `id`라는 이름은 설명을 위한 자리표시자일 뿐이며, 실제 제약은 키 타입이 `string`, 값 타입이 `number`라는 점에 있습니다. 따라서 인덱스 시그니처는 이름을 미리 열거하기 어려운 사전(dictionary) 형태의 데이터를 다룰 때 적합합니다.

정리하면 튜플은 자리의 순서가 의미를 가질 때, 객체 타입은 프로퍼티 이름이 의미를 가질 때 적합합니다. 그래서 상품, 사용자, 주문처럼 각 값의 역할을 이름으로 드러내야 하는 데이터는 객체 타입으로 표현하면 구조를 더 정확하게 보여 줍니다.

## **2-11. any ⭐️⭐️⭐️**

외부에서 들어오는 데이터는 TypeScript 컴파일러가 구조를 미리 알 수 없습니다. `JSON.parse`, API 응답, 사용자 입력은 실행되기 전까지 실제 값이 숫자인지 문자열인지조차 확정되지 않기 때문입니다. 문제는 이 지점에서 타입 검사를 꺼 버리면, 잘못된 값이 들어와도 컴파일 단계에서는 걸러지지 않는다는 점입니다.

`any`는 이 문제를 가장 극단적으로 악화시키는 타입입니다. `any`를 붙이는 순간 컴파일러는 그 값에 대해 거의 아무 제약도 걸지 않습니다. 잘못된 계산을 하거나 존재하지 않는 프로퍼티에 접근해도, 컴파일 단계에서는 막지 못합니다.

그래서 외부 데이터를 다룰 때는 `any` 대신 `unknown`을 기준으로 삼습니다. `unknown`은 “이 값의 정체를 아직 모른다”는 사실을 타입에 남겨 두고, 검사를 통과하기 전까지는 사용을 막습니다. 즉, `any`가 제약을 없애는 방향이라면 `unknown`은 확인 전 사용을 막는 방향입니다.

아래 예제는 같은 외부 데이터를 두 가지 방식으로 다룹니다. 첫 번째는 `any`, 두 번째는 `unknown`입니다. 어떤 방식이 왜 위험하고, 어떤 방식이 확인 후에만 사용되도록 강제하는지 비교해 보면 차이가 분명해집니다.

파일: `main.ts`

```tsx
// any는 타입 검사를 꺼 버립니다.
const priceJson = '"129000원"';

const unsafePrice: any = JSON.parse(priceJson);
console.log(unsafePrice * 0.9); // 컴파일은 통과하지만 결과는 의도와 다를 수 있습니다

const parsedPrice: unknown = JSON.parse(priceJson);

// console.log(parsedPrice * 0.9); // ❌ unknown은 바로 계산할 수 없습니다

if (typeof parsedPrice === "number") {
  console.log(parsedPrice * 0.9);
} else {
  console.log("숫자 데이터가 아닙니다.");
}
```

이 코드에서 `unsafePrice`는 실제로는 문자열이 들어 있어도 컴파일 단계에서 아무 제약을 받지 않습니다. 그래서 숫자 할인 계산처럼 보이는 코드도 그대로 통과하고, 실행해 보면 `NaN` 같은 원치 않는 결과가 나올 수 있습니다.

반면 `parsedPrice: unknown`은 검사 전 사용을 막습니다. 그래서 바로 계산에 사용할 수 없고, 먼저 `typeof parsedPrice === "number"` 같은 확인 과정을 거쳐야 합니다. `unknown`이 안전한 이유는 더 많은 것을 허용해서가 아니라, 확인 전에는 덜 허용하기 때문입니다.

여기서 중요한 점은 `unknown`이 불편한 타입이 아니라, 확인을 강제하는 타입이라는 사실입니다. 외부 입력을 받는 순간에는 값의 정체를 모른다고 인정하고, 필요한 검사 이후에만 좁혀 쓰는 쪽이 TypeScript의 의도와 맞습니다.

| 방식 | 컴파일러의 태도 | 런타임 검증 | 적합한 상황 |
| --- | --- | --- | --- |
| `any` | 거의 아무 검사도 하지 않습니다 | 없음 | 되도록 피해야 합니다 |
| `unknown` + 기본 검사 | 검사 전에는 사용을 막습니다 | 있음 | `JSON.parse`, API 응답, 사용자 입력처럼 타입이 불확실한 값 |

실무에서는 외부 입력을 만나는 경계에서 `any`보다 `unknown`을 기본값으로 둡니다. 값의 종류를 확인한 뒤에만 사용하는 습관이 있어야, TypeScript의 타입 검사가 실제 안전장치로 작동합니다.

여기서는 `unknown`과 기본 검사만 먼저 익혀 두면 충분합니다. 이 흐름을 더 일반화한 **타입 좁히기(type narrowing)**와, 검사 이후 제한적으로 사용하는 **타입 어썰션(type assertion)**은 5장에서 다시 정리합니다.

## **2-12. 함수에 타입 정하기 ⭐️⭐️⭐️**

변수 타입이 “이 값이 어떤 종류인가”를 설명한다면, 함수 타입은 “이 함수가 어떤 값을 받고 어떤 결과를 내놓는가”를 설명합니다. 함수는 값이 드나드는 경계이기 때문에, 파라미터와 반환값의 규칙이 흐려지면 호출하는 쪽과 구현하는 쪽이 쉽게 어긋납니다.

TypeScript는 이 규칙을 함수 시그니처(function signature)에 직접 적도록 합니다. 파라미터 뒤에는 받는 값의 타입을, 괄호 뒤에는 반환값의 타입을 적습니다. 이 선언이 있으면 잘못된 인자 전달을 막을 수 있을 뿐 아니라, 함수가 어떤 역할을 가지는지도 선언부만 보고 파악할 수 있습니다.

```
 함수 시그니처
 ─────────────────────────────────────────────────────
 function addToCart(id: string, quantity: number = 1): boolean
                    ↑           ↑                     ↑
               파라미터 타입   기본값 파라미터         리턴 타입
```

장바구니에 상품을 넣는 함수는 이런 계약을 보여 주기에 적합합니다. 상품 ID는 문자열이어야 하고, 수량은 숫자여야 하며, 재고가 부족하면 성공 여부를 불린형으로 돌려줄 수 있습니다. 이 규칙을 함수 선언에 적어 두면 잘못된 호출을 막는 동시에, 함수가 실패 가능성을 가진다는 사실도 타입으로 드러납니다.

파일: `main.ts`

```tsx
const stock: { [id: string]: number } = {
  c001: 3,
  c002: 1,
};
const cart: string[] = [];

// id: string  → 첫 번째 파라미터는 문자열
// quantity: number = 1  → 두 번째는 숫자, 기본값은 1
// : boolean  → 리턴값은 불린형
function addToCart(
  id: string,
  quantity: number = 1,
): boolean {
  if (stock[id] < quantity) {
    return false;
  }

  stock[id] -= quantity;
  for (let i = 0; i < quantity; i++) {
    cart.push(id);
  }

  return true;
}

addToCart("c001", 2);
addToCart("c001");
// addToCart(123, 2);
```

여기서 `id: string`은 첫 번째 인자가 문자열이어야 한다는 뜻이고, `quantity: number = 1`은 숫자를 받되 인자가 생략되면 기본값 `1`을 사용하겠다는 뜻입니다. 그래서 `addToCart('c001')`처럼 호출해도 함수 안에서는 `quantity`를 그대로 숫자로 다룰 수 있습니다. `: boolean`은 이 함수가 성공 여부를 `true` 또는 `false`로 돌려준다는 약속이므로, 호출 결과를 조건문에 바로 사용할 수 있습니다.

함수는 항상 새로운 값을 반환할 필요가 없습니다. 어떤 함수는 상태를 출력하거나 여러 입력을 한 번에 처리한 뒤 끝나면 충분합니다. 이때는 반환값이 없음을 `void`로, 인자를 생략할 수 있음을 옵셔널 파라미터(optional parameter)로, 개수가 가변적인 인자를 받을 수 있음을 레스트 파라미터(rest parameter)로 표현합니다.

```tsx
function printCart(title?: string): void {
  if (title) {
    console.log(title);
  }

  console.log(cart);
}

printCart();
printCart("현재 장바구니");

function addManyToCart(...ids: string[]): void {
  for (const id of ids) {
    addToCart(id);
  }
}

addManyToCart("c001", "c002");
```

`title?: string`은 인자를 넘기지 않아도 된다는 뜻입니다. 따라서 `printCart()`처럼 호출할 수도 있고, 값이 들어온 경우에만 제목을 출력하도록 분기할 수도 있습니다. `...ids: string[]`는 여러 개의 문자열 인자를 하나의 배열처럼 받는 방식이며, `: void`는 반환값이 없다는 사실을 함수 선언에 분명히 드러냅니다.

| 함수 타입 요소 | 문법 | 의미 | 현재 예시 |
| --- | --- | --- | --- |
| 파라미터 타입 | `id: string` | 받는 값의 종류를 고정합니다 | `addToCart` |
| 리턴 타입 | `): boolean` | 반환하는 값의 종류를 고정합니다 | `addToCart` |
| 기본값 파라미터 | `quantity: number = 1` | 인자를 생략하면 기본값을 사용합니다 | `addToCart` |
| 옵셔널 파라미터 | `title?: string` | 인자를 아예 넘기지 않아도 됩니다 | `printCart` |
| 레스트 파라미터 | `...ids: string[]` | 여러 인자를 배열처럼 받습니다 | `addManyToCart` |
| `void` | `): void` | 반환값이 없음을 뜻합니다 | `printCart`, `addManyToCart` |

정리하면 함수 타입은 구현 세부사항보다 앞서 입력과 출력의 계약을 고정합니다. 변수 타입이 값 하나의 안전장치를 제공한다면, 함수 타입은 값이 오가는 지점 전체를 보호합니다. 그래서 함수 선언은 가능한 한 초기에 타입을 명확히 적어 두면 구조를 읽기 쉽고 오류도 먼저 드러납니다.

# 03. Enum

## **3-01. Enum ⭐️**

프로그램 안에는 값의 자료형은 문자열이지만, 실제로는 아무 문자열이나 허용하면 안 되는 데이터가 자주 등장합니다. 상품 사이즈는 `S`, `M`, `L`, `XL` 중 하나여야 하고, 캐릭터 이동 방향도 `UP`, `DOWN`, `LEFT`, `RIGHT`처럼 정해진 후보 안에서만 선택되어야 합니다. 이런 값을 일반 문자열로 두면 오타 하나만으로도 잘못된 값이 섞일 수 있습니다.

Enum은 이 문제를 해결하기 위한 TypeScript의 열거형 문법입니다. 여러 상수 값을 하나의 이름 아래 묶고, 그 목록 안에 있는 값만 선택하도록 제한합니다. 즉, 값의 후보가 고정되어 있을 때 “허용 가능한 선택지 목록 자체”를 타입으로 만드는 방식입니다.

```
 일반 문자열                       Enum
 ────────────────────────         ─────────────────────────
 size = 'M'                  →    size = Size.M
 size = 'Medium'             →    size = Size.Medium  ❌ (선언되지 않은 멤버)
 size = 'm'                  →    정해진 멤버만 선택 가능
```

상품 사이즈는 Enum의 역할을 가장 단순하게 보여 주는 예시입니다. 문자열 `'M'`은 값 하나만 보면 이해할 수 있지만, `'m'`, `'MIDIUM'`, `'Medium'` 같은 값이 섞여 들어오는 순간 프로그램 안에서 같은 의미를 여러 표현이 나눠 갖게 됩니다. Enum으로 이름을 붙여 두면, 호출하는 쪽은 `Size.M`처럼 정해진 멤버만 사용하게 됩니다.

파일: `main.ts`

```tsx
enum Size {
  S = "S",
  M = "M",
  L = "L",
  XL = "XL",
}

const productSize: Size = Size.M;
console.log(productSize); // 출력: M

function printSize(size: Size): void {
  console.log(`사이즈:${size}`);
}

printSize(Size.L); // 출력: 사이즈: L
// printSize('L');
```

이 예제를 읽을 때 중요한 점은 `Size`가 타입과 값의 목록 역할을 함께 가진다는 사실입니다. 변수 선언에서는 `: Size`처럼 “이 변수는 `Size` Enum의 멤버만 담을 수 있다”는 뜻으로 읽고, 실제 대입에서는 `Size.M`처럼 점 표기법으로 멤버를 선택합니다. 따라서 `'L'`이라는 문자열을 직접 넘기는 방식과 `Size.L`을 넘기는 방식은 겉보기에는 비슷해도, 타입 시스템이 검사할 수 있는 정보의 양이 다릅니다.

Enum은 상품 정보처럼 외부에 보여 주는 값뿐 아니라, 프로그램 내부 상태를 표현할 때도 유용합니다. 이동 방향을 문자열로 직접 넘기면 `'LEFT'`, `'Lift'`, `'left'`처럼 비슷한 값이 뒤섞일 수 있지만, Enum으로 묶으면 이동 가능한 방향의 집합이 코드 수준에서 고정됩니다. 이렇게 목록을 먼저 정의해 두면 함수 시그니처도 더 분명해집니다.

```tsx
enum Direction {
  Up = "UP",
  Down = "DOWN",
  Left = "LEFT",
  Right = "RIGHT",
}

function move(direction: Direction): void {
  console.log(`이동 방향:${direction}`);
}

move(Direction.Up); // 출력: 이동 방향: UP
move(Direction.Left); // 출력: 이동 방향: LEFT
// move('UP');
```

Enum은 숫자 기반과 문자열 기반으로 선언할 수 있습니다. 숫자 기반 Enum은 값을 생략하면 `0`, `1`, `2`처럼 자동으로 할당되기 때문에 선언은 짧아지지만, 실행 결과만 봐서는 의미를 바로 읽기 어렵습니다. 반면 문자열 기반 Enum은 각 멤버의 의미가 그대로 남기 때문에 로그를 보거나 디버깅할 때 해석이 쉬워집니다. 이 교재에서는 입문 단계에서 혼동을 줄이기 위해 문자열 기반 Enum을 기준으로 설명합니다.

이 차이는 코드로 직접 보면 더 분명합니다. 숫자 기반 Enum은 선언은 짧지만, 로그에 찍힌 숫자만 보고는 어떤 상태였는지 다시 추적해야 합니다. 반면 문자열 기반 Enum은 출력된 값만 보아도 어떤 멤버였는지 바로 해석할 수 있습니다.

```tsx
enum NumericDirection {
  Up, // 0
  Down, // 1
  Left, // 2
  Right, // 3
}

console.log(NumericDirection.Left); // 출력: 2

enum StringDirection {
  Up = "UP",
  Down = "DOWN",
  Left = "LEFT",
  Right = "RIGHT",
}

console.log(StringDirection.Left); // 출력: LEFT
```

| 방식 | 예시 | 장점 | 주의할 점 |
| --- | --- | --- | --- |
| 숫자 기반 Enum | `enum Direction { Up, Down }` | 값을 따로 적지 않아도 됩니다 | 출력된 숫자만으로는 의미를 읽기 어렵습니다 |
| 문자열 기반 Enum | `Up = 'UP'` | 값의 의미가 그대로 남습니다 | 각 멤버 값을 직접 적어야 합니다 |

한 가지 더 알아 둘 점은 일반 `enum`이 `type`이나 `interface`와 다르게 동작한다는 사실입니다. `type`과 `interface`는 TypeScript 컴파일 이후 JavaScript 코드에서 사라집니다. 반면 일반 `enum`은 컴파일 이후에도 JavaScript 객체 형태로 남아 있어, `Size.M`처럼 선언한 멤버를 런타임에서도 그대로 참조할 수 있습니다.

정리하면 Enum은 “미리 정해진 선택지 목록”을 코드에 이름 붙여 두는 방식입니다. 일반 문자열은 값의 범위를 넓게 열어 두지만, Enum은 허용 가능한 후보를 미리 닫아 둡니다. 상태값, 분류값, 방향값처럼 선택지가 고정된 경우에 쓰면 코드의 의도가 더 분명하게 드러나고, 오타로 인한 오류를 사전에 막을 수 있습니다.

# 04. Interface

## **4-01. Interface ⭐️⭐️⭐️**

같은 형태의 객체를 여러 군데에서 다루기 시작하면, 가장 먼저 늘어나는 것은 로직보다 타입 중복입니다. 상품 객체를 변수에도 쓰고 함수 파라미터에도 쓰고 배열 원소에도 쓰기 시작하면, `id`, `name`, `price` 같은 프로퍼티 목록을 계속 반복해서 적게 됩니다. 이 상태에서 구조가 바뀌면, 타입을 적어 둔 모든 위치를 다시 찾아 수정해야 합니다.

Interface는 이 문제를 해결하기 위한 객체 타입 전용 설계도입니다. 객체가 어떤 프로퍼티를 가져야 하는지 한 번 이름 붙여 정의하고, 이후에는 그 이름을 재사용합니다. 따라서 여러 위치가 같은 구조를 공유할 때 타입 정의를 한곳으로 모을 수 있고, 구조가 바뀌어도 수정 지점이 분산되지 않습니다.

```
 Interface 없을 때                Interface 사용 시
 ──────────────────────────────    ─────────────────────────
 const p1: {                       interface Product {
   id: string;                       id: string;
   name: string;                     name: string;
   price: number;                    price: number;
 } = { ... }                       }
                                                   ↙     ↘
 const p2: {              const p1: Product     const p2: Product
   id: string;
   name: string;        → 타입이 바뀌면 Interface 하나만 수정
   price: number;
 } = { ... }
```

상품 정보를 예로 들면, 변수 선언과 함수 파라미터에서 같은 구조를 반복해서 쓰는 상황이 금방 만들어집니다. 이때 Interface를 도입하면 “상품은 이런 모양을 가진다”는 약속을 한 줄 이름으로 치환할 수 있습니다. 아래 예제는 `Product`라는 Interface를 만들고, 두 개의 상품 객체와 출력 함수가 모두 그 설계도를 공유하도록 정리한 모습입니다.

파일: `main.ts`

```tsx
interface Product {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
  
}

const product1: Product = {
  id: "c001",
  name: "코드잇 블랙 후드 집업",
  price: 129_000,
  membersOnly: true,
};

const product2: Product = {
  id: "d001",
  name: "코드잇 텀블러",
  price: 25_000,
};

function printProduct(product: Product): void {
  console.log(
    `${product.name}의 가격은${product.price}원입니다.`,
  );
}

printProduct(product1);
printProduct(product2);
```

이 코드에서 핵심은 `product1`, `product2`, `printProduct`가 모두 `Product`라는 같은 이름을 공유한다는 점입니다. `membersOnly?: boolean`처럼 `?`를 붙인 프로퍼티는 선택적으로만 있어도 되므로, `product2`처럼 해당 속성이 없는 객체도 같은 Interface를 만족할 수 있습니다. 결국 Interface는 객체의 실제 데이터를 담는 도구가 아니라, 객체 구조에 대한 약속을 재사용하는 도구입니다.

Interface의 재사용은 단순 복붙 제거에서 끝나지 않습니다. 이미 정의한 객체 구조를 바탕으로 더 구체적인 타입을 만들 수도 있습니다. 상품 공통 정보는 그대로 유지하면서 의류 상품에만 필요한 `sizes`, `color`를 추가하고 싶다면, `extends`로 기존 Interface를 확장하면 됩니다.

```tsx
interface ClothingProduct extends Product {
  sizes: string[];
  color: string;
}

const jacket: ClothingProduct = {
  id: "c002",
  name: "코드잇 자켓",
  price: 149_000,
  sizes: ["M", "L", "XL"],
  color: "navy",
};

printProduct(jacket);
```

`ClothingProduct`는 `Product`가 요구하는 속성을 모두 포함하면서, 의류 전용 속성을 추가한 타입입니다. 그래서 `jacket`은 더 구체적인 타입이지만, 공통 계약인 `Product`를 만족하므로 `printProduct`처럼 `Product`를 받는 함수에도 그대로 전달할 수 있습니다. 이 흐름은 Interface가 단순한 별칭이 아니라, 객체 구조 사이의 관계를 단계적으로 설계하는 도구라는 점을 보여 줍니다.

| 방식 | 장점 | 한계 |
| --- | --- | --- |
| 인라인 객체 타입 | 선언 위치에서 바로 구조를 볼 수 있습니다 | 같은 구조가 반복되면 중복이 빠르게 늘어납니다 |
| Interface | 이름을 붙여 여러 곳에서 재사용할 수 있습니다 | 객체 타입에만 적용되며, 유니온처럼 여러 타입을 조합하는 형태에는 사용할 수 없습니다 |

# 05. 그 밖의 타입들

## **5-01. 리터럴 타입 ⭐️⭐️**

지금까지는 `string`, `number`처럼 값의 종류를 넓게 묶는 타입을 주로 다뤘습니다. 그런데 실제 프로그램에서는 “문자열이면 다 된다”보다 `"S"`, `"M"`, `"L"`처럼 허용 가능한 값이 정확히 정해져 있는 경우가 더 자주 등장합니다. 이때 필요한 것이 **리터럴 타입(literal type)**입니다.

리터럴 타입은 값의 종류가 아니라 **값 그 자체**를 타입으로 사용하는 방식입니다. TypeScript는 특히 `const`와 함께 이 개념을 자연스럽게 추론합니다. 값이 바뀌지 않는다고 판단되면, 그 값 하나만 허용하는 더 좁은 타입으로 읽는 것입니다.

먼저 같은 값을 선언해도 `let`과 `const`가 타입 추론을 어떻게 다르게 만드는지 살펴봅니다. 이 차이를 파악해 두면 뒤에서 나오는 더 정밀한 타입 표현으로 자연스럽게 이어집니다.

파일: `main.ts`

```tsx
let productName1 = "코드잇 블랙 후드";
const productName2 = "코드잇 텀블러";

let discountRate1 = 10;
const discountRate2 = 15;
```

이 코드에서 `productName1`은 나중에 다른 문자열로 바뀔 수 있으므로 `string`으로 추론됩니다. 반면 `productName2`는 다시 대입할 수 없기 때문에 `'코드잇 텀블러'`라는 값 하나만 허용하는 리터럴 타입으로 추론됩니다. 숫자도 마찬가지로 `discountRate1`은 `number`, `discountRate2`는 `15`라는 리터럴 타입이 됩니다.

| 선언 방식 | 추론 결과 | 의미 |
| --- | --- | --- |
| `let` | 넓은 타입 | 이후 다른 값으로 바뀔 가능성을 열어 둡니다 |
| `const` | 리터럴 타입 | 현재 값 하나를 그대로 타입 정보로 보존합니다 |

리터럴 타입이 중요한 이유는 “허용 가능한 값의 범위”를 더 정확하게 표현할 수 있기 때문입니다. 값 하나만 허용해야 하는 상황이라면 `string`보다 `'M'`처럼 값을 직접 타입으로 적는 편이 훨씬 정확합니다. 이때 `const`로 선언된 값은 좁은 타입으로 유지되므로 이런 함수에 바로 전달할 수 있습니다.

파일: `main.ts`

```tsx
function printSelectedSize(size: "M") {
  console.log(`선택한 사이즈:${size}`);
}

const selectedSize1 = "M";
let selectedSize2 = "M";

printSelectedSize(selectedSize1);
// printSelectedSize(selectedSize2); // 오류: string은 'M'에 바로 넣을 수 없습니다.
```

`selectedSize1`은 `'M'`이라는 리터럴 타입으로 추론되므로 함수에 바로 전달할 수 있습니다. 반면 `selectedSize2`는 값이 지금 `'M'`이더라도 타입이 이미 `string`으로 넓어졌기 때문에, TypeScript는 이 값이 항상 `'M'`이라고 보장하지 않습니다. 즉, 리터럴 타입은 “현재 값이 무엇인가”를 타입 수준까지 끌어올려 더 정밀한 제약을 설정하는 도구입니다.

여러 리터럴 타입을 묶어 허용 가능한 선택지 집합을 만드는 방식은 뒤에서 Union 타입과 타입 별칭을 다룰 때 다시 정리합니다.

## **5-02. 타입 별칭 ⭐️⭐️⭐️⭐️⭐️**

프로그램이 커질수록 같은 타입 표현식이 여러 곳에서 반복됩니다. 처음에는 `string[]`, `(result: boolean) => void`처럼 짧아 보여도, 그 표현이 무엇을 뜻하는지 매번 다시 해석해야 하면 코드의 의미가 흐려집니다. 구조는 같아도 역할이 다른 데이터를 구분하기 어렵다는 점도 문제입니다.

이럴 때 사용하는 것이 **타입 별칭(Type Alias)**입니다. 타입 별칭은 `type` 키워드로 기존 타입 표현식에 이름을 붙이는 방식입니다. 새로운 런타임 값을 만드는 것이 아니라, 컴파일러가 읽는 타입 표현식에 별명을 부여하는 것에 해당합니다.

먼저 같은 타입 표현식이 반복될 때 어떤 문제가 생기는지 살펴봅니다. 장바구니 상품 ID 목록을 여러 곳에서 사용할 때 `string[]`만으로는 이 배열이 어떤 의미를 갖는지 바로 드러나지 않습니다. 타입 구조는 보이지만, 도메인 의미는 이름이 없어서 코드 밖에서 추론해야 합니다.

파일: `main.ts`

```tsx
// 똑같은 타입이 여러 곳에 반복되는 상황
const cart1: string[] = ["c001", "c001", "c002"];

interface User1 {
  username: string;
  email: string;
  cart: string[];
}

const user1: User1 = {
  username: "codeit",
  email: "typescript@codeit.kr",
  cart: cart1,
};
```

이 코드에서 `string[]`는 “문자열 배열”이라는 구조만 말해 줍니다. 하지만 실제 의도는 “장바구니에 담긴 상품 ID 목록”입니다. 즉, 타입 별칭은 긴 타입을 줄이기 위한 도구이기도 하지만, 그보다 먼저 **타입에 의미를 붙이는 도구**라고 보는 편이 더 정확합니다.

이제 같은 타입에 `Cart`라는 이름을 붙입니다. 그러면 배열의 구조와 함께 이 값이 어떤 역할을 맡는지도 코드에 드러납니다. 같은 방식은 함수 타입에도 적용할 수 있어서, 콜백 시그니처가 반복될 때도 읽기 쉬운 이름으로 정리할 수 있습니다.

파일: `main.ts`

```tsx
// 타입 별칭으로 의미를 붙여 재사용
type Cart = string[];

const cart2: Cart = ["c001", "c001", "c002"];

interface User2 {
  username: string;
  email: string;
  cart: Cart;
}

const user2: User2 = {
  username: "codeit",
  email: "typescript@codeit.kr",
  cart: cart2,
};

type AddToCartCallback = (result: boolean) => void;

function addToCartWithCallback(
  id: string,
  callback: AddToCartCallback,
): void {
  const result = true;
  callback(result);
}

addToCartWithCallback("c001", (result) => {
  console.log(result ? "장바구니 담기 성공" : "재고 없음");
});
```

여기서 `Cart`나 `AddToCartCallback`은 새로운 자료형을 런타임에 만들어 내는 것이 아닙니다. `string[]`, `(result: boolean) => void`라는 기존 타입 표현식을 더 읽기 쉬운 이름으로 감싼 것뿐입니다. 따라서 타입 별칭은 “의미 부여”와 “재사용”이 동시에 필요한 지점에서 가장 효과적입니다.

| 타입 별칭이 잘 맞는 경우 | 예시 |
| --- | --- |
| 배열, 튜플 | `type Cart = string[]` |
| 함수 시그니처, 콜백 | `type Callback = (result: boolean) => void` |
| 리터럴 유니온 | `type Size = 'S' \| 'M' \| 'L' \| 'XL'` |

반면 객체 구조 자체를 설계하고 확장 관계를 드러내는 데에는 `Interface`가 더 잘 맞는 경우가 많습니다. 그래서 이 교재에서는 배열, 함수, 리터럴 유니온처럼 `Interface`로 표현하기 어색한 타입에는 타입 별칭을 사용하고, 객체 형태의 설계도는 `Interface`로 설명하는 흐름을 유지합니다.

## **5-03. type과 interface, 무엇을 써야 할까**

5-01과 5-02에서 리터럴 타입과 타입 별칭을 익혔고, 4장에서는 Interface로 객체 구조를 설계하는 방법을 다뤘습니다. 두 도구를 모두 접하면 자연스럽게 한 가지 질문이 생깁니다. `type`과 `interface`, 무엇을 써야 할까요?

둘 다 타입에 이름을 붙이는 방법이지만, 각자 잘 맞는 영역이 다릅니다. `interface`는 객체 구조를 설계하고 `extends`로 계층을 만드는 데 초점을 맞춥니다. 반면 `type`은 배열, 튜플, 함수 시그니처, 리터럴 유니온처럼 `interface`로 표현하기 어색한 모든 타입 표현식에 이름을 붙이는 데 적합합니다.

|  | `type` | `interface` |
| --- | --- | --- |
| 잘 맞는 대상 | 배열, 튜플, 함수 시그니처, 리터럴 유니온 | 객체 구조 |
| 확장 방식 | 유니온·인터섹션으로 조합 | `extends`로 계층 설계 |

이 기준을 실제 코드에 적용합니다. 좌표처럼 객체보다 튜플이 더 자연스러운 구조는 `type`으로, 여러 엔티티가 공유하는 공통 속성은 `interface`로 분리합니다.

파일: `main.ts`

```tsx
type Point = [number, number]; // 튜플 → type

interface Entity {
  // 객체 공통 구조 → interface
  id: string;
  createdAt: Date;
  updatedAt: Date;
}

interface Monster extends Entity {
  name: string;
  level: number;
  hasGold?: boolean;
  skills: string[];
  move: (fromPoint: Point, toPoint: Point) => void;
}

interface Npc extends Entity {
  name: string;
  dialog: string;
}

function getDiff(fromPoint: Point, toPoint: Point): Point {
  const dx = toPoint[0] - fromPoint[0];
  const dy = toPoint[1] - fromPoint[1];
  return [dx, dy];
}

const monster: Monster = {
  id: "g001",
  name: "고블린",
  level: 22,
  skills: ["태권도", "특공무술"],
  move(fromPoint, toPoint) {
    const [dx, dy] = getDiff(fromPoint, toPoint);
    console.log(
      `오른쪽으로${dx} 위쪽으로${dy} 만큼 이동!`,
    );
  },
  createdAt: new Date(),
  updatedAt: new Date(),
};

const npc: Npc = {
  id: "n001",
  name: "상인",
  dialog: "희귀 아이템을 보고 가세요.",
  createdAt: new Date(),
  updatedAt: new Date(),
};

const current: Point = [0, 0];
const target: Point = [4, 5];

monster.move(current, target);
console.log(npc.dialog);
```

`Point`는 `[number, number]`라는 튜플 표현에 이름을 붙인 것입니다. `interface`로 표현하면 어색하고, 좌표라는 의미를 담기에는 `type`이 더 적합합니다. 반면 `Entity`는 여러 객체 타입이 `extends`로 확장할 공통 계약이므로 `interface`가 더 적합합니다.

`type`과 `interface` 모두 타입에 이름을 붙이는 도구입니다. 객체 구조의 설계도가 필요하다면 `interface`를, 그 외 모든 타입 표현식에 이름을 붙일 때는 `type`을 선택하면 대부분의 상황에서 판단이 명확해집니다.

## **5-04. Union 타입**

서로 다른 타입을 하나의 흐름에서 함께 처리해야 할 때가 있습니다. 예를 들어 상품 목록을 출력하는 함수가 의류 상품도 받고 신발 상품도 받아야 한다면, 매개변수는 한 가지 타입으로 고정하기 어렵습니다. 그렇다고 너무 넓은 상위 타입만 사용하면 실제로 필요한 속성에 접근하지 못하게 됩니다.

이때 사용하는 것이 **Union 타입**입니다. Union 타입은 “`이 타입이거나 저 타입`”이라는 가능성을 `|`로 표현합니다. 여기서 중요한 점은 값이 두 타입을 동시에 모두 가진다는 뜻이 아니라, 실행 시점에 여러 후보 중 하나임을 의미합니다.

```
 Union 타입
 ──────────────────────────────────────────────
 ClothingProduct | ShoeProduct
        ↑                ↑
   의류 상품이거나   신발 상품이거나 (둘 중 하나)

 공통 프로퍼티 (name, price, sizes)  →  바로 접근 가능
 개별 프로퍼티 (color, handmade)     →  분기 후 접근
```

먼저 의류 상품과 신발 상품을 하나의 함수에서 함께 처리하는 예제를 살펴봅니다. 이 예제의 핵심은 Union 타입을 쓰면 두 타입에 공통으로 존재하는 정보는 바로 사용할 수 있고, 한쪽에만 있는 정보는 조건 분기 뒤에만 사용할 수 있다는 점입니다.

파일: `main.ts`

```tsx
enum ClothingSize {
  S = "S",
  M = "M",
  L = "L",
  XL = "XL",
}

interface Product {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
}

interface ClothingProduct extends Product {
  sizes: ClothingSize[];
  color: string;
}

interface ShoeProduct extends Product {
  sizes: number[];
  handmade: boolean;
}

function printProduct(
  product: ClothingProduct | ShoeProduct,
) {
  console.log(
    `${product.name}:${product.price.toLocaleString()}원`,
  );
  console.log(
    `구매 가능한 사이즈:${product.sizes.join(", ")}`,
  );

  if ("color" in product) {
    console.log(`색상:${product.color}`);
  } else {
    console.log(`수제화 여부:${product.handmade}`);
  }
}

const product1: ClothingProduct = {
  id: "c001",
  name: "코드잇 블랙 후드 집업",
  price: 129_000,
  membersOnly: true,
  sizes: [ClothingSize.M, ClothingSize.L],
  color: "black",
};

const product2: ShoeProduct = {
  id: "s001",
  name: "코드잇 스니커즈",
  price: 69_000,
  membersOnly: false,
  sizes: [220, 230, 240, 260, 280],
  handmade: false,
};

printProduct(product1);
printProduct(product2);
```

`name`, `price`, `sizes`는 두 타입 모두에 있으므로 바로 접근할 수 있습니다. 반면 `color`는 의류 상품에만 있고 `handmade`는 신발 상품에만 있으므로, `in` 연산자로 현재 값이 어느 타입인지 먼저 좁힌 뒤에 접근해야 합니다. 이런 과정을 **타입 좁히기(narrowing)** 라고 하며, 뒤의 `5-08. 타입 좁히기` 절에서 `typeof`, `in`, `Array.isArray`를 기준으로 다시 자세히 다룹니다.

Union 타입은 객체 타입끼리만 묶는 데 쓰이지 않습니다. 리터럴 타입 여러 개를 `|`로 연결하면 허용 가능한 값의 목록 자체를 타입으로 만들 수도 있습니다. 이 방식은 “올 수 있는 값이 무엇인가”를 코드에 직접 남긴다는 점에서 입력 제한을 명확하게 드러냅니다.

파일: `main.ts`

```tsx
type ClothingSizeOption = "S" | "M" | "L" | "XL";
type ShoeSizeOption =
  | 220
  | 230
  | 240
  | 250
  | 260
  | 270
  | 280;

function selectClothingSize(size: ClothingSizeOption) {
  console.log(`선택한 의류 사이즈:${size}`);
}

const selectedClothingSize: ClothingSizeOption = "M";

selectClothingSize(selectedClothingSize);
// selectClothingSize('XXL'); // 오류: 허용된 값이 아닙니다
```

리터럴 타입과 Union을 조합하면 “문자열 또는 숫자”처럼 넓은 범위가 아니라, 그중에서도 실제로 허용할 값만 골라서 타입으로 표현할 수 있습니다. 따라서 Union 타입은 여러 객체 타입을 한 흐름에서 처리할 때도 쓰이고, 선택 가능한 값의 집합을 정확하게 제한할 때도 쓰입니다.

| Union에서 확인할 기준 | 예시 | 해석 |
| --- | --- | --- |
| 공통 정보 | `name`, `price`, `sizes` | 모든 후보 타입에 있으므로 바로 접근할 수 있습니다 |
| 개별 정보 | `color`, `handmade` | 특정 타입에만 있으므로 분기 후에 접근해야 합니다 |
| 허용값 제한 | `'S' \| 'M' \| 'L'`, `220 \| 230 \| 240` | 가능한 값의 목록 자체를 타입으로 만들 수 있습니다 |

## **5-06. Intersection 타입**

여러 타입 조각을 조합해 하나의 값을 더 정확하게 표현해야 할 때가 있습니다. 예를 들어 모든 엔티티가 `id`를 가지지만, 어떤 타입은 시간 정보도 함께 가지고 있고, 또 어떤 타입은 별도의 도메인 속성을 추가로 가져야 할 수 있습니다. 이런 경우 매번 같은 속성을 반복해서 쓰면 구조는 비슷한데 선언만 길어지고, 공통 규칙이 바뀌었을 때 수정 지점도 흩어집니다.

이때 사용하는 것이 **Intersection 타입**입니다. Intersection 타입은 “`이 타입이면서 동시에 저 타입`”이라는 뜻을 `&`로 표현합니다. Union 타입이 여러 후보 중 하나를 표현했다면, Intersection 타입은 여러 타입의 조건을 한 번에 모두 만족하는 구조를 표현합니다.

```
 Union (|)               Intersection (&)
 ─────────────           ─────────────────────
 A | B                   A & B
 = A이거나 B             = A이면서 동시에 B

 예) 의류 OR 신발         예) 공격자 AND 방어자
     (둘 중 하나)              (둘 다)
```

Intersection 타입은 보통 두 가지 상황에서 자주 등장합니다. 첫째는 반복되는 속성 조각을 합칠 때이고, 둘째는 서로 다른 역할을 동시에 수행하는 타입을 표현할 때입니다. 이 두 경우를 함께 보면 `&`가 단순한 문법이 아니라, 작은 타입 조각을 조합해 더 구체적인 타입을 만드는 도구라는 점이 잘 드러납니다.

먼저 공통 속성을 작은 조각으로 나눈 뒤 합치는 예제를 살펴봅니다. `id`와 시간 정보는 여러 엔티티가 반복해서 사용하므로 별도 타입으로 분리하고, 각 도메인 타입은 필요한 조각만 `&`로 합칩니다. 이렇게 하면 공통 구조를 한곳에 모아 두면서도, 최종 타입은 각 엔티티의 의미에 맞게 구체화할 수 있습니다.

파일: `main.ts`

```tsx
type Id = {
  id: string;
};

type Timestamp = {
  createdAt: Date;
  updatedAt: Date;
};

type Product = Id & {
  name: string;
  price: number;
  membersOnly?: boolean;
};

type User = Id &
  Timestamp & {
    username: string;
    email: string;
  };

type Review = Id &
  Timestamp & {
    productId: string;
    userId: string;
    content: string;
  };

interface Attacker {
  attackPower: number;
  attack(): void;
}

interface Defender {
  defensePower: number;
  defend(): void;
}

type BattleCharacter = Attacker &
  Defender & {
    name: string;
  };

const product: Product = {
  id: "c001",
  name: "코드잇 블랙 후드티",
  price: 129_000,
};

const user: User = {
  id: "user0001",
  username: "codeit",
  email: "typescript@codeit.kr",
  createdAt: new Date(),
  updatedAt: new Date(),
};

const review: Review = {
  id: "review001",
  userId: user.id,
  productId: product.id,
  content: "아주 좋음",
  createdAt: new Date(),
  updatedAt: new Date(),
};

const character: BattleCharacter = {
  name: "팔라딘",
  attackPower: 80,
  defensePower: 60,
  attack() {
    console.log("공격!");
  },
  defend() {
    console.log("방어!");
  },
};

console.log(product.name);
console.log(review.content);
character.attack();
character.defend();
```

`Product`는 `Id`와 상품 전용 속성을 합친 타입입니다. `User`와 `Review`는 `Id`뿐 아니라 `Timestamp`도 함께 사용하므로, 공통 속성 조각을 여러 번 재사용할 수 있습니다. 따라서 공통 규칙이 바뀌면 조각 타입을 수정하는 것만으로 여러 최종 타입에 같은 변경을 반영할 수 있습니다.

아래쪽의 `BattleCharacter`는 Intersection 타입의 두 번째 용도를 보여 줍니다. 이 타입은 공격자이면서 동시에 방어자여야 하므로, `Attacker`와 `Defender`의 조건을 모두 만족해야 합니다. 즉, `&`는 상속 계층을 만드는 도구라기보다, 서로 다른 역할이나 속성 조각을 합쳐 하나의 완성된 타입을 만드는 도구입니다.

| 비교 기준 | Intersection (`&`) | `interface extends` |
| --- | --- | --- |
| 핵심 질문 | 어떤 타입 조각을 함께 만족해야 하는가 | 어떤 상위 구조를 확장하는가 |
| 읽는 방식 | 조합 | 계층 |
| 잘 맞는 장면 | 공통 속성 조각 재사용, 역할 합성 | 상위-하위 구조를 분명히 드러낼 때 |

## **5-08. 타입 좁히기 ⭐️⭐️⭐️**

Union 타입이나 `unknown`은 일부러 넓게 잡아 둔 타입입니다. 이렇게 넓은 타입은 여러 가능성을 한 번에 표현할 수 있다는 장점이 있지만, 그 대신 지금 손에 들어온 값이 정확히 무엇인지는 코드 안에서 다시 확인해야 합니다. 그렇지 않으면 문자열 전용 메서드나 특정 프로퍼티처럼 일부 경우에만 안전한 동작을 곧바로 사용할 수 없습니다.

이처럼 실행 중에 확인한 정보를 바탕으로 컴파일러가 타입의 범위를 더 구체적으로 줄여 이해하는 과정을 **타입 좁히기(type narrowing)**라고 합니다. 값 자체가 바뀌는 것은 아니고, 같은 변수라도 분기 안에서는 더 구체적인 타입으로 해석된다는 점이 핵심입니다. 앞에서 `unknown`을 다룰 때 `typeof`로 숫자인지 확인했던 흐름이나, Union 타입에서 `in` 연산자로 객체 종류를 나눴던 흐름도 모두 타입 좁히기의 예시였습니다.

| 검사 방식 | 좁혀지는 기준 | 자주 쓰는 상황 |
| --- | --- | --- |
| `typeof` | 원시값 종류 | `string \| number`처럼 기본형 Union을 다룰 때 |
| `in` | 특정 프로퍼티 존재 여부 | 서로 다른 객체 타입 Union을 구분할 때 |
| `Array.isArray` | 배열 여부 | 단일 값과 배열이 함께 올 수 있을 때 |

세 가지 방식은 서로 경쟁하는 문법이 아니라, 값의 모양에 따라 선택하는 도구입니다. 문자열과 숫자처럼 원시값을 나눌 때는 `typeof`가 가장 직접적입니다. 반면 객체 타입끼리 묶인 Union에서는 공통으로 없는 프로퍼티를 기준으로 `in` 검사를 해야 하고, 배열일 수도 있는 입력은 `Array.isArray`로 먼저 분기해야 이후 처리가 안전해집니다.

파일: `main.ts`

```tsx
interface ClothingProduct {
  id: string;
  name: string;
  color: string;
  sizes: string[];
}

interface ShoeProduct {
  id: string;
  name: string;
  handmade: boolean;
  sizes: number[];
}

function printSize(size: string | number) {
  if (typeof size === "string") {
    console.log(`의류 사이즈:${size}`);
  } else {
    console.log(`신발 사이즈:${size}mm`);
  }
}

function printProductDetail(
  product: ClothingProduct | ShoeProduct,
) {
  console.log(product.name);

  if ("color" in product) {
    console.log(`색상:${product.color}`);
    console.log(`사이즈:${product.sizes.join(", ")}`);
  } else {
    console.log(`수제화:${product.handmade}`);
    console.log(`사이즈:${product.sizes.join(", ")}mm`);
  }
}

function printSelectedSizes(sizes: string | string[]) {
  if (Array.isArray(sizes)) {
    console.log(`선택 가능한 사이즈:${sizes.join(", ")}`);
  } else {
    console.log(`대표 사이즈:${sizes}`);
  }
}

printSize("M");
printSize(260);

const hoodie: ClothingProduct = {
  id: "c001",
  name: "코드잇 블랙 후드 집업",
  color: "black",
  sizes: ["M", "L"],
};

const sneakers: ShoeProduct = {
  id: "s001",
  name: "코드잇 스니커즈",
  handmade: false,
  sizes: [260, 270],
};

printProductDetail(hoodie);
printProductDetail(sneakers);

printSelectedSizes("M");
printSelectedSizes(["S", "M", "L"]);
```

이 코드에서 `printSize`는 `typeof` 검사 뒤에 문자열과 숫자를 서로 다른 방식으로 처리합니다. `printProductDetail`은 `color` 프로퍼티의 존재 여부를 기준으로 의류 상품과 신발 상품을 나누고, `printSelectedSizes`는 `Array.isArray`로 단일 값과 배열을 구분합니다.

즉 타입 좁히기는 “값을 먼저 확인하고, 그 확인 결과를 근거로 가능한 연산을 선택하는 과정”입니다. 넓은 타입을 쓸수록 이런 분기 로직이 함께 따라온다는 점은, 뒤에서 다룰 타입 어썰션과의 차이를 구분하는 데도 연결됩니다.

## **5-09. 타입 어썰션 ⭐️⭐️⭐️**

타입 좁히기는 코드가 직접 검사한 결과를 근거로 타입을 더 구체적으로 만드는 방식입니다. 그런데 실제 코드에서는 개발자가 충분히 확인했다고 생각하지만, 컴파일러가 그 사실을 끝까지 따라가지 못하는 경우가 있습니다. 배열인지 확인했더라도 원소 타입까지 자동으로 확정하지 못하거나, 외부 데이터의 구조를 일부 검사했지만 원하는 구체 타입까지는 연결하지 못하는 식입니다.

이때 사용할 수 있는 문법이 **타입 어썰션(type assertion)**입니다. `as` 키워드로 “이 값을 이 타입으로 취급해 달라”고 컴파일러에게 요청합니다. 다만 이것은 값을 다른 형태로 변환하는 기능이 아니라, 개발자의 판단을 타입 검사기에 전달하는 기능이므로 검사 없이 남용하면 안전장치를 쉽게 무너뜨립니다.

| 구분 | 타입 좁히기 | 타입 어썰션 |
| --- | --- | --- |
| 근거 | 코드가 직접 검사한 결과 | 개발자의 판단 |
| 런타임 검증 | 있음 | 없음 |
| 기본 원칙 | 먼저 고려 | 정말 필요할 때만 제한적으로 사용 |

따라서 타입 어썰션은 보통 마지막 단계에서만 사용합니다. 먼저 `Array.isArray`, `typeof`, `in` 같은 검사로 범위를 최대한 줄이고, 그다음에도 컴파일러가 원하는 타입까지 확정하지 못할 때만 `as`를 붙입니다. 아래 예제는 검사 없이 곧바로 단정하는 방식이 왜 위험한지와, 검사 뒤에 제한적으로 어썰션을 붙이는 방식이 어떻게 다른지를 함께 보여 줍니다.

파일: `main.ts`

```tsx
const unsafeParsedData: unknown =
  JSON.parse('{"size":"M"}');

// const unsafeSizes = unsafeParsedData as string[];
// console.log(unsafeSizes.join(', '));
// ↑ 컴파일은 통과하지만, 실제 값은 배열이 아니라서 실행하면 오류가 납니다.

const parsedData: unknown = JSON.parse('["S", "M", "L"]');

if (
  Array.isArray(parsedData) &&
  parsedData.every((size) => typeof size === "string")
) {
  const sizes = parsedData as string[];
  console.log(sizes.join(", "));
}
```

여기서 `unsafeParsedData as string[]`는 값 자체를 배열로 바꾸지 않습니다. 실제 값이 객체여도, 개발자가 배열이라고 단정하면 컴파일러는 그 선언을 그대로 받아들입니다. 그래서 타입 어썰션은 안전장치를 더하는 문법이 아니라, 경우에 따라 안전장치를 우회하는 문법이 될 수 있습니다.

반면 아래쪽 `parsedData as string[]`는 이미 `Array.isArray`와 원소 검사로 근거를 만든 뒤에만 사용됩니다. 즉 타입 어썰션은 타입 좁히기를 대체하는 도구가 아니라, 좁히기 이후에 남은 마지막 간격을 메우는 보조 수단입니다.

## **5-10. keyof와 typeof 연산자 ⭐️⭐️⭐️**

객체 타입을 다루다 보면 두 가지 중복이 자주 생깁니다. 하나는 프로퍼티 이름을 문자열 Union 타입으로 다시 적는 중복이고, 다른 하나는 이미 만들어 둔 값과 같은 구조를 타입으로 다시 적는 중복입니다. 이런 중복은 구조가 바뀌었을 때 한쪽만 수정되고 다른 쪽이 남아 타입 정보가 어긋나는 원인이 됩니다.

`keyof`와 `typeof`는 이런 문제를 줄이는 데 함께 쓰이는 연산자입니다. `keyof`는 객체 타입에서 프로퍼티 이름만 꺼내 Union 타입으로 만들고, `typeof`는 이미 존재하는 값에서 타입을 가져옵니다. 즉 `keyof`는 타입에서 키를 꺼내는 연산자이고, `typeof`는 값에서 타입을 꺼내는 연산자입니다.

| 연산자 | 입력 | 결과 | 주로 해결하는 문제 |
| --- | --- | --- | --- |
| `keyof` | 객체 타입 | 프로퍼티 이름의 Union 타입 | 키 이름을 수동으로 다시 적는 중복 |
| `typeof` | 값 | 그 값의 TypeScript 타입 | 같은 구조의 타입을 다시 적는 중복 |

두 연산자는 각각 따로 써도 되지만, 실제 코드에서는 `keyof typeof value` 형태로 이어서 쓰는 경우도 많습니다. 먼저 `typeof`로 값의 타입을 얻고, 그 결과에 `keyof`를 적용해 키 이름을 자동으로 추출하는 방식입니다. 이렇게 하면 객체 구조를 한곳에서만 유지하면서, 키 목록과 함수 파라미터 타입도 함께 맞출 수 있습니다.

여기서는 두 연산자의 관계가 한 번에 보이도록 `keyof typeof product` 형태를 사용합니다. 이미 `Product` 같은 타입 이름이 있다면 `keyof Product`라고 적어도 같은 결과를 얻을 수 있습니다.

파일: `main.ts`

```tsx
interface Product {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
}

const product: Product = {
  id: "c001",
  name: "코드잇 블랙 후드 집업",
  price: 129_000,
  membersOnly: true,
};

type ProductFromValue = typeof product;
type ProductProperty = keyof typeof product;

const productTableKeys: ProductProperty[] = [
  "name",
  "price",
  "membersOnly",
];

function printProductValue(
  item: ProductFromValue,
  key: ProductProperty,
) {
  console.log(`${key} |${item[key]}`);
}

for (const key of productTableKeys) {
  printProductValue(product, key);
}

const product2: typeof product = {
  id: "c002",
  name: "코드잇 텀블러",
  price: 25_000,
};

const product3: ProductFromValue = {
  id: "c003",
  name: "코드잇 머그컵",
  price: 19_000,
  membersOnly: false,
};

console.log(product2, product3);
```

이 코드에서 `ProductFromValue`는 `product` 값의 구조를 그대로 타입으로 가져옵니다. 그래서 `product2`, `product3`는 같은 타입 표현식을 다시 쓰지 않고도 `product`와 같은 구조를 유지합니다.

`ProductProperty`는 `product`의 타입에서 키 이름만 뽑아 만든 Union 타입입니다. 따라서 `printProductValue()`는 실제 상품 객체에 있는 프로퍼티 이름만 `key`로 받을 수 있고, `product` 구조가 바뀌면 이 타입도 함께 바뀝니다.

> **참고**: JavaScript의 `typeof`는 `"string"`, `"object"` 같은 문자열을 반환하지만, TypeScript 타입 위치의 `typeof`는 값의 타입 자체를 가져옵니다.

## **5-12. 타입을 집합으로 이해하기**

앞에서 리터럴 타입, Union 타입, Intersection 타입을 각각 따로 배웠지만, 세 개념은 하나의 관점으로 묶어 읽을 수 있습니다. TypeScript에서 타입을 허용 가능한 값들의 집합으로 보면, 어떤 타입이 더 좁고 어떤 타입이 더 넓은지 훨씬 선명해집니다. 이 관점은 뒤에서 제네릭을 읽을 때도 타입 사이의 포함 관계를 판단하는 기준이 됩니다.

가장 먼저 보기 쉬운 장면은 기본 타입과 리터럴 타입의 관계입니다. `number`는 숫자 값을 허용하는 큰 집합이고, `129_000`은 그 안의 값 하나만 가리키는 작은 집합입니다. 같은 방식으로 `string`과 `'M'`도 큰 집합과 작은 집합의 관계로 읽을 수 있습니다.

![5-12-primitive-literal-sets.svg](33%20TypeScript%20%EA%B8%B0%EB%B3%B8/5-12-primitive-literal-sets.svg)

기본 타입과 리터럴 타입

이제 Union 타입을 보면 집합 관점이 더 직접적으로 드러납니다. `'S' | 'M' | 'L'`는 세 값을 허용하는 집합이고, 여기에 `'FREE'`를 더한 `'S' | 'M' | 'L' | 'FREE'`는 그보다 넓은 집합입니다. 따라서 Union 타입은 가능한 값의 후보를 늘려 타입의 범위를 넓히는 방식이라고 이해할 수 있습니다.

![5-12-types-as-sets.svg](33%20TypeScript%20%EA%B8%B0%EB%B3%B8/5-12-types-as-sets.svg)

Union으로 허용값 집합이 넓어지는 과정

반대로 Intersection 타입은 조건을 겹쳐 더 구체적인 값을 남깁니다. `HasName`이 이름을 가진 객체들의 집합이고, `HasPrice`가 가격을 가진 객체들의 집합이라면, `HasName & HasPrice`는 두 조건을 동시에 만족하는 객체들만 허용합니다. 그래서 Intersection 타입은 여러 조건을 합쳐 더 좁고 구체적인 타입을 만드는 방식입니다.

![5-12-object-union-intersection.svg](33%20TypeScript%20%EA%B8%B0%EB%B3%B8/5-12-object-union-intersection.svg)

Intersection으로 객체 조건이 합쳐지는 과정

집합 관점의 바깥 경계도 함께 정리할 필요가 있습니다. `unknown`은 어떤 값이 와도 담을 수 있을 만큼 넓은 타입이고, `never`는 정상적으로 만들 수 있는 값이 하나도 없는 타입입니다. 그래서 `unknown`은 가장 넓은 경계, `never`는 공집합이라는 설명과 연결됩니다.

![5-12-special-types-map.svg](33%20TypeScript%20%EA%B8%B0%EB%B3%B8/5-12-special-types-map.svg)

unknown, never, any의 관계

여기까지의 내용을 표로 정리하면 다음과 같습니다.

| 타입 표기 | 집합 관점에서의 의미 |
| --- | --- |
| `129_000` | 값 하나만 포함하는 작은 집합 |
| `number` | 숫자 값을 허용하는 큰 집합 |
| `'S' \| 'M' \| 'L'` | 세 리터럴 값으로 이루어진 유한 집합 |
| `'S' \| 'M' \| 'L' \| 'FREE'` | 기존 집합에 값을 더해 넓힌 Union 집합 |
| `HasName & HasPrice` | 두 조건을 동시에 만족하는 값만 남는 교집합 |
| `unknown` | 어떤 값이든 들어올 수 있는 가장 넓은 경계 |
| `never` | 값이 하나도 없는 공집합 |
| `any` | 집합의 포함 관계보다 타입 검사를 우회하는 예외 |

아래 코드는 지금까지 다룬 집합 개념이 실제 TypeScript 코드에서 어떻게 나타나는지 확인하는 예시입니다. `'M'`이 더 넓은 `ClothingSize`에 포함되는 장면, Union으로 허용값이 늘어나는 장면, Intersection으로 객체 조건이 합쳐지는 장면, `never`가 정상 반환값이 없는 함수를 표현하는 장면을 한 파일 안에서 함께 확인합니다.

파일: `main.ts`

```tsx
type MediumSize = 'M';
type ClothingSize = 'S' | 'M' | 'L';
type ExtendedSize = ClothingSize | 'FREE';

type HasName = {
  name: string;
};

type HasPrice = {
  price: number;
};

type ProductPreview = HasName & HasPrice;

const selectedSize: MediumSize = 'M';
const sizeOption: ClothingSize = selectedSize;
const eventSize: ExtendedSize = 'FREE';

const preview: ProductPreview = {
  name: '코드잇 블랙 후드 집업',
  price: 129_000,
};

function fail(message: string): never {
  throw new Error(message);
}

console.log(sizeOption, eventSize, preview);

try {
  fail('never 예시');
} catch (error) {
  console.log(error);
}
```

이 코드에서 `selectedSize`를 `sizeOption`에 대입할 수 있는 이유는 `'M'` 집합이 `ClothingSize` 집합 안에 포함되기 때문입니다. `eventSize`는 `ClothingSize`에 `'FREE'`를 추가한 더 넓은 집합이므로, 허용 가능한 후보 값이 하나 더 많습니다.

`preview`는 `name`과 `price`를 모두 갖춰야 하므로 Intersection 타입이 여러 조건을 한 객체에 모으는 방식도 함께 드러납니다. 마지막의 `never`는 값을 하나 더 포함시키는 개념이 아니라, 정상 반환값이 존재하지 않는 특수한 경우라는 점에서 공집합과 연결됩니다.

여기서 `void`와 `never`는 모두 “반환값이 보이지 않는다”는 인상 때문에 함께 묶어 이해하기 쉽지만, 함수가 끝나는 방식은 전혀 다릅니다. `void`는 함수 실행이 정상적으로 끝난 뒤 특별히 사용할 반환값이 없다는 뜻이고, `never`는 `throw`나 무한 루프처럼 함수가 정상 종료 지점에 도달하지 못한다는 뜻입니다.

| 반환 타입 | 함수 실행 결과 | 집합 관점 | 대표 예시 |
| --- | --- | --- | --- |
| `void` | 함수가 정상적으로 끝남 | 반환값을 따로 사용하지 않는 종료 | 로그를 출력하고 끝나는 함수 |
| `never` | 함수가 정상적으로 끝나지 않음 | 가능한 반환값이 전혀 없는 공집합 | `throw`, 무한 루프 |

따라서 `fail()`은 오류를 던지는 순간 함수 흐름이 끊기므로 `void`가 아니라 `never`입니다. 반대로 함수가 끝까지 실행된 뒤 단지 반환값을 따로 쓰지 않는 경우라면 `void`를 사용합니다. 집합 비유를 그대로 적용하면 공집합과 직접 연결되는 쪽은 `never`이고, `void`는 정상 종료 여부를 설명하는 반환 타입이라는 점에서 성격이 다릅니다.

마지막으로 `any`는 `unknown`처럼 가장 넓은 집합이라고 단순화하면 정확하지 않습니다. 어떤 값이든 담을 수 있다는 점만 보면 넓은 타입처럼 보이지만, 실제로는 컴파일러가 더 이상 타입을 확인하지 않도록 만드는 예외 장치에 가깝습니다. 그래서 집합의 포함 관계를 설명할 때는 `unknown`과 `never`를 경계로 이해하고, `any`는 그 바깥에서 타입 시스템을 우회하는 특수한 타입으로 구분합니다.

# 06. 제네릭

## **6-01. 제네릭 ⭐️⭐️⭐️⭐️**

같은 로직을 값의 종류만 바꿔 여러 번 쓰는 상황은 자주 생깁니다. 배열을 출력하는 함수는 숫자 배열이든 문자열 배열이든 구조가 거의 같고, API 응답 객체도 실제 데이터 타입만 달라질 뿐 `status`, `message` 같은 바깥 구조는 반복됩니다. 이럴 때 타입까지 매번 새로 작성하면 함수 수와 타입 정의가 불필요하게 늘어납니다.

제네릭(Generic)은 이런 중복을 줄이기 위한 장치입니다. 함수나 interface의 구조는 그대로 두고, 그 안에 들어갈 타입만 나중에 채워 넣도록 만드는 방식입니다. 즉 제네릭은 “타입을 매개변수처럼 받는 구조”입니다.

| 문제 상황 | 제네릭 없이 처리할 때 | 제네릭으로 처리할 때 |
| --- | --- | --- |
| 배열 처리 함수 | `printNumberArray`, `printStringArray`처럼 타입마다 함수가 늘어남 | `printArray<T>` 하나로 처리 |
| 첫 번째 값 반환 | `getFirstNumber`, `getFirstString`처럼 중복 발생 | `getFirstItem<T>` 하나로 처리 |
| API 응답 구조 | `ProductResponse`, `UserResponse`처럼 wrapper 타입이 반복 | `ApiResponse<T>` 하나로 처리 |

아래 예제는 제네릭 함수와 제네릭 interface를 한 흐름으로 묶은 예제입니다. `printArray<T>`는 배열 원소 타입이 무엇이든 같은 방식으로 출력하고, `getFirstItem<T>`는 입력 배열 타입에 맞는 첫 값을 돌려줍니다. 이어서 `ApiResponse<T>`는 바깥 응답 구조는 유지하면서, 실제 `data` 타입만 바꿀 수 있게 만듭니다.

파일: `main.ts`

```tsx
function printArray<T>(items: T[]): void {
  for (const item of items) {
    console.log(item);
  }
}

function getFirstItem<T>(items: T[]): T | undefined {
  return items[0];
}

interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

interface Product {
  id: string;
  name: string;
}

printArray([230, 250, 280]);
printArray(["M", "L", "XL"]);
printArray<boolean>([true, false]);

const firstShoeSize = getFirstItem([230, 250, 280]);
const firstClothingSize = getFirstItem(["M", "L", "XL"]);

const response: ApiResponse<Product> = {
  data: { id: "c001", name: "코드잇 후드" },
  status: 200,
  message: "OK",
};

const listResponse: ApiResponse<Product[]> = {
  data: [{ id: "c001", name: "코드잇 후드" }],
  status: 200,
  message: "OK",
};

console.log(firstShoeSize, firstClothingSize);
console.log(response, listResponse);
```

여기서 `printArray([230, 250, 280])`를 호출하면 `T`는 `number`로 추론되고, 문자열 배열을 넘기면 `T`는 `string`으로 추론됩니다. 즉 함수 본문은 하나인데, 호출 시점에 타입만 달라져 같은 구조를 여러 데이터에 재사용할 수 있습니다.

`ApiResponse<T>`도 같은 원리입니다. `status`와 `message`는 항상 같고, `data` 자리만 `Product`, `Product[]`처럼 바뀝니다. 이런 식으로 구조는 고정하고 데이터 타입만 바꾸는 패턴이 제네릭의 가장 기본적인 사용법입니다.

| 표기 | 의미 |
| --- | --- |
| `<T>` | 호출 시점에 채워질 임의의 타입 파라미터 |
| `T[]` | 타입 `T`의 배열 |
| `ApiResponse<T>` | `data`의 타입만 바뀌는 공통 응답 구조 |
| `T \| undefined` | 값이 없을 수도 있는 제네릭 반환 타입 |

타입 파라미터 이름은 보통 대문자 한 글자(`T`, `U`, `V`)를 사용합니다. 의미를 조금 더 드러내고 싶다면 `TItem`, `TData`처럼 `T` 접두사를 붙여도 됩니다. 중요한 점은 이름보다도, “구조는 같고 타입만 나중에 채운다”는 원리가 코드에서 살아 있는가입니다.

# 07. 그 외 타입들

## 7-01. 인덱스드 엑세스 타입 (Indexed Access Types)

- 객체 타입에서 특정 속성의 타입을 추출
- Type['propertyName'] 형태

```tsx
interface User {
  id: number;
  name: string;
  email: string;
}

// 인덱스드 엑세스 타입으로 id 타입 추출
function findUserById(users: User[], id: User["id"]) {
  return users.find(user => user.id === id);
}

function deleteUser(users: User[], id: User["id"]) {
  return users.filter(user => user.id !== id);
}

// 사용
const users = [
  { id: 1, name: "김철수", email: "kim@email.com" },
  { id: 2, name: "이영희", email: "lee@email.com" }
];

findUserById(users, 1);      // ✅ 타입: User | undefined
deleteUser(users, 2);        // ✅ 타입: User[]
// findUserById(users, "1"); // ❌ 에러! id는 number여야 함
```

## 7-02. keyof 연산자

- 객체 타입의 **모든 키(속성명)들을 유니온 타입으로** 가져오는 연산자

```tsx
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// type User = "id" | "name" | "email" | "age"
type UserKeys = keyof User;

// 사용 예시
function getProperty(user: User, key: UserKeys) {
  return user[key];
}

const user = { id: 1, name: "김철수", email: "kim@email.com", age: 25 };

getProperty(user, "name");// ✅ 가능
getProperty(user, "email");// ✅ 가능
getProperty(user, "height");// ❌ 에러! User에 height 속성 없음
```

## 7-03. 맵드 타입 (Mapped Types)

- 기존 객체 타입을 기반으로, 각 속성을 변환하여 새로운 객체 타입을 만드는 기능
- 객체의 모든 속성을 순회하면서 변형
- { [K in keyof T]: 새로운 타입 } 형태

```tsx
interface User {
  id: number;
  name: string;
  email: string;
}

// 모든 속성을 선택적(optional)으로 만들기
type PartialUser = {
  [K in keyof User]?: User[K];
};
// 결과: { id?: number; name?: string; email?: string; }

// 모든 속성을 문자열로 변경
type StringifiedUser = {
  [K in keyof User]: string;
};
// 결과: { id: string; name: string; email: string; }
```

### 수식어 사용 (+ / -) - 매우 중요!

맵드 타입에서 `+`와 `-` 수식어로 `?`(선택적)과 `readonly`를 추가하거나 제거할 수 있습니다.

```tsx
interface User {
  readonly id: number;
  name?: string;
  email: string;
}

// 1. 모든 속성을 필수로 만들기 (-? 수식어)
type RequiredUser = {
  [K in keyof User]-?: User[K];
};
// 결과: { readonly id: number; name: string; email: string; }

// 2. 모든 속성을 선택적으로 만들기 (+? 수식어, +는 생략 가능)
type PartialUser = {
  [K in keyof User]+?: User[K];
};
// 결과: { readonly id?: number; name?: string; email?: string; }

// 3. readonly 제거 (-readonly 수식어)
type MutableUser = {
  -readonly [K in keyof User]: User[K];
};
// 결과: { id: number; name?: string; email: string; }

// 4. readonly 추가 (+readonly 수식어, +는 생략 가능)
type ReadonlyUser = {
  +readonly [K in keyof User]: User[K];
};
// 결과: { readonly id: number; readonly name?: string; readonly email: string; }

// 5. 조합: readonly 제거 + 모든 속성 필수
type MutableRequiredUser = {
  -readonly [K in keyof User]-?: User[K];
};
// 결과: { id: number; name: string; email: string; }
```

## 7-04. 유틸리티 타입들 (Partial, Required, Readonly, 등등)

### 1. Partial<T> - 모든 속성을 선택적으로

```tsx
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// 모든 속성이 선택적(optional)이 됨
type PartialUser = Partial<User>;
// 결과: { id?: number; name?: string; email?: string; age?: number; }// 실무 활용: 사용자 정보 업데이트
function updateUser(id: number, updates: Partial<User>) {
// id만 필수, 나머지는 선택적으로 업데이트
}

updateUser(1, { name: "새이름" });// ✅ 가능
updateUser(1, { email: "new@email.com" });// ✅ 가능
updateUser(1, { name: "새이름", age: 30 });// ✅ 가능
updateUser(1, {});// ✅ 빈 객체도 가능
```

### 2. Required<T> - 모든 속성을 필수로

```tsx
interface Config {
  host?: string;
  port?: number;
  ssl?: boolean;
  timeout?: number;
}

// 모든 속성이 필수가 됨
type RequiredConfig = Required<Config>;
// 결과: { host: string; port: number; ssl: boolean; timeout: number; }

// 실무 활용: 기본값 적용 후 완전한 설정 객체
function createServer(config: RequiredConfig) {
// 모든 설정이 보장됨
  console.log(`서버 시작: ${config.host}:${config.port}`);
}

function initializeConfig(userConfig: Config): RequiredConfig {
  return {
    host: userConfig.host || "localhost",
    port: userConfig.port || 3000,
    ssl: userConfig.ssl || false,
    timeout: userConfig.timeout || 5000
  };
}
```

### 3. Readonly<T> - 모든 속성을 읽기 전용으로


## 7-05. 조건부 타입

**삼항 연산자 같은 문법으로 타입을 조건부로 결정**하는 기능

```tsx
// JavaScript 삼항 연산자
const result = condition ? valueA : valueB;

// TypeScript 조건부 타입
type Result<T> = T extends SomeType ? TypeA : TypeB;
```

### 1. 간단한 조건부 타입

```tsx
// 숫자인지 확인하는 타입
type IsNumber<T> = T extends number ? true : false;

type Test1 = IsNumber<42>;// true
type Test2 = IsNumber<string>;// false
type Test3 = IsNumber<boolean>;// false
```

### 2. 배열인지 확인

```tsx
// 조건부 타입: 배열인지 확인
type IsArray<T> = T extends any[] ? true : false;

type Test1 = IsArray<string[]>; // true
type Test2 = IsArray<number>; // false
type Test3 = IsArray<boolean[]>; // true

// 조건부 타입: 배열이면 요소 타입, 아니면 원래 타입 반환
type Unwrap<T> = T extends (infer U)[] ? U : T;

type UnwrapTest1 = Unwrap<string[]>; // string
type UnwrapTest2 = Unwrap<number>; // number
type UnwrapTest3 = Unwrap<boolean[]>; // boolean

// 조건부 타입: 배열이면 "array", 아니면 "single" 반환
type DataType<T> = T extends any[] ? "array" : "single";

type DataType1 = DataType<number[]>; // "array"
type DataType2 = DataType<string>; // "single"
```

## 분산적 조건부 타입 (Distributive Conditional Types)

유니온 타입이 조건부 타입을 만나면 **각각 따로 적용**되는 특별한 동작

### 1. 기본 분산 동작

```tsx
type ToArray<T> = T extends any ? T[] : never;

// 일반적인 경우
type StringArray = ToArray<string>;// string[]
type NumberArray = ToArray<number>;// number[]// 유니온 타입의 경우 - 분산적으로 동작!
type UnionArray = ToArray<string | number>;

// 결과: string[] | number[] (not (string | number)[])
// 풀어서 설명하면:
// ToArray<string | number>// = ToArray<string> | ToArray<number>  
// 분산 적용
// = string[] | number[]
```

### 2. 실무 활용: 타입 필터링

```tsx
// null과 undefined를 제거하는 유틸리티
type NonNullable<T> = T extends null | undefined ? never : T;

type CleanedType = NonNullable<string | number | null | undefined>;
// 결과: string | number
// 분산 적용 과정:
// NonNullable<string | number | null | undefined>
// = NonNullable<string> | NonNullable<number> | NonNullable<null> | NonNullable<undefined>
// = string | number | never | never
// = string | number  (never는 유니온에서 제거됨)
```

## infer 키워드 - 타입 추론

- 키워드 extends 키워드가 있는 조건부 타입에서 **타입을 추출(추론)** 할 때 사용
- 타입을 추론하고 그 추론된 타입을 새로운 타입 변수로 저장
- 주로 함수의 매개변수나 반환 타입을 추출할 때 사용

### 1. 함수 반환 타입 추출

```tsx
// 함수의 반환 타입을 추출하는 유틸리티 타입
type GetReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

// 간단한 함수들
function add(a: number, b: number): number {
    return a + b;
}

function sayHello(): string {
    return "안녕하세요!";
}

// GetReturnType을 사용하여 함수의 반환 타입 추출
type AddReturnType = GetReturnType<typeof add>;    // number
type SayHelloReturnType = GetReturnType<typeof sayHello>;    // string

// 실제 사용 예시
const result1: AddReturnType = 42;        // ✅
const result2: SayHelloReturnType = "반갑습니다";    // ✅

// 타입 에러 예시
const error1: AddReturnType = "42";       // ❌ number 타입에 string 할당 불가
const error2: SayHelloReturnType = 42;    // ❌ string 타입에 number 할당 불가
```

### 2. 배열 요소 타입 추출

```tsx
// 배열에서 요소 타입을 추출
type ArrayElement<T> = T extends (infer U)[] ? U : never;

const fruits = ["apple", "banana", "orange"];
type Fruit = ArrayElement<typeof fruits>; // string

const numbers = [1, 2, 3, 4, 5];
type NumberType = ArrayElement<typeof numbers>; // number
```

### 3. 객체 속성 타입 추출

```tsx
// 특정 키의 값 타입을 추출
type GetPropertyType<T, K extends keyof T> = T extends { [key in K]: infer V } ? V : never;
 
interface User {
  id: number; 
  name: string;
  email: string;
}

type IdType = GetPropertyType<User, "id">;// number
type NameType = GetPropertyType<User, "name">;// string
```

## **8-01. 모듈 사용하기 ⭐️⭐️**

파일이 하나일 때는 타입, Enum, 실행 로직이 모두 가까이 있어 읽기 쉽습니다. 하지만 코드가 늘어나면 선언과 사용이 뒤섞이고, 특정 타입을 수정할 때 어떤 로직이 영향을 받는지 한눈에 보기가 어려워집니다. 이 시점부터는 코드를 역할별로 나누고, 필요한 것만 가져와 연결하는 방식이 필요합니다.

TypeScript의 모듈은 파일 단위로 코드를 분리하고 `export`와 `import`로 연결하는 구조입니다. 어떤 파일이 바깥으로 공개할 대상을 `export`하면, 다른 파일은 필요한 대상만 `import`해서 사용합니다. 이렇게 나누면 타입 정의, Enum, 실제 실행 로직의 경계가 분명해집니다.

| 파일 | 맡는 역할 | 바깥으로 공개하는 대상 |
| --- | --- | --- |
| `Product.ts` | 상품 구조 정의 | `Product` interface |
| `Size.ts` | 사이즈 값 목록 정의 | `Size` Enum |
| `main.ts` | 실제 데이터 생성과 실행 | 다른 파일에서 가져온 타입과 Enum 사용 |

먼저 `Product.ts`는 상품 데이터의 구조만 정의합니다. 이 파일은 실제 값을 만들지 않고, 다른 파일이 참조할 기준 타입만 바깥으로 내보냅니다.

파일: `Product.ts`

```tsx
export default interface Product {
  id: string;
  name: string;
  price: number;
  membersOnly?: boolean;
  sizes?: string[];
}
```

다음 `Size.ts`는 문자열 리터럴을 직접 반복해서 적지 않도록, 사이즈 값 목록을 Enum으로 분리한 파일입니다. 이 파일도 마찬가지로 `main.ts`에서 사용할 값을 `export default`로 공개합니다.

파일: `Size.ts`

```tsx
enum Size {
  S = "S",
  M = "M",
  L = "L",
  XL = "XL",
}

export default Size;
```

이제 `main.ts`에서는 두 파일에서 필요한 것만 가져와 실제 상품 데이터를 만듭니다. 타입은 `import Product`로, 값으로 쓰는 Enum은 `import Size`로 불러온다는 점이 핵심입니다.

파일: `main.ts`

```tsx
import Product from "./Product";
import Size from "./Size";

const product1: Product = {
  id: "c001",
  name: "코드잇 블랙 후드 집업",
  price: 129_000,
  membersOnly: true,
  sizes: [Size.S, Size.M],
};

const product2: Product = {
  id: "d001",
  name: "코드잇 텀블러",
  price: 25_000,
};

function printProduct(product: Product) {
  console.log(product);
}

printProduct(product1);
printProduct(product2);
```

이 구조의 장점은 수정 지점이 분명하다는 데 있습니다. 상품 구조가 바뀌면 `Product.ts`를 보면 되고, 허용 가능한 사이즈 값이 바뀌면 `Size.ts`를 보면 됩니다. `main.ts`는 그런 기준을 가져와 실제 로직에 연결하는 역할만 담당합니다.

즉 모듈 사용하기의 핵심은 파일을 여러 개로 나누는 행위 자체가 아니라, “무엇을 정의하는 파일인지”와 “무엇을 실행하는 파일인지”의 책임을 분리하는 것입니다. 이 분리가 되어야 프로젝트가 커져도 타입과 로직을 안정적으로 추적할 수 있습니다.

## **8-02. 자주 사용하는 옵션**

`tsconfig.json`은 TypeScript 컴파일러가 프로젝트를 어떤 기준으로 읽고, 어떤 방식으로 JavaScript로 바꿀지를 정하는 파일입니다. 설정이 없으면 명령어를 실행할 때마다 옵션을 직접 붙여야 하고, 팀이나 강의에서 같은 프로젝트를 다뤄도 사람마다 다른 방식으로 빌드될 수 있습니다. 따라서 `tsconfig.json`은 단순한 부가 설정이 아니라, 프로젝트의 타입 검사 기준과 출력 기준을 고정하는 역할을 합니다.

처음부터 모든 옵션을 외울 필요는 없습니다. 입문 단계에서는 “출력 환경을 정하는 옵션”, “타입 검사를 엄격하게 만드는 옵션”, “소스와 빌드 결과의 경로를 정하는 옵션” 정도로 나누어 이해하면 충분합니다. 아래 예제는 그 세 범주를 한 파일에 모아 둔 형태입니다.

파일: `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "rootDir": "./src",
    "outDir": "./dist"
  },
  "include": ["src/**/*"],
  "exclude": ["src/**/*.test.ts"]
}
```

이 설정 파일은 세 가지 층위로 읽으면 정리가 쉽습니다. `target`, `module`, `moduleResolution`은 어떤 실행 환경과 모듈 해석 방식을 기준으로 출력할지를 정하고, `strict`는 타입 검사를 얼마나 엄격하게 할지를 정합니다. 그다음 `rootDir`, `outDir`, `include`, `exclude`는 어떤 파일을 읽고 어디에 결과를 둘지를 결정합니다.

| 옵션 | 역할 | 처음 볼 때의 해석 |
| --- | --- | --- |
| `target` | 출력 JavaScript 문법 수준을 정함 | 어느 정도 최신 문법까지 결과물에 남길지 결정 |
| `module` | 모듈 출력 형식을 정함 | `import`/`export`를 어떤 방식으로 다룰지 결정 |
| `moduleResolution` | 모듈 경로를 찾는 방식을 정함 | import 경로를 어떤 규칙으로 해석할지 결정 |
| `strict` | 엄격한 타입 검사 묶음을 켬 | 가능하면 느슨한 타입을 줄이고 오류를 빨리 잡게 함 |
| `rootDir` | 소스 루트 경로를 정함 | 어떤 폴더를 원본 코드 기준으로 볼지 결정 |
| `outDir` | 빌드 결과 경로를 정함 | 생성된 `.js` 파일을 어디에 둘지 결정 |
| `include` | 컴파일 대상 범위를 정함 | 어떤 파일들을 읽을지 결정 |
| `exclude` | 컴파일 대상에서 제외할 범위를 정함 | 읽지 않을 파일들을 뺌 |

여기서 특히 먼저 파악할 값은 `strict`, `rootDir`, `outDir`입니다. `strict`는 “이 프로젝트가 느슨한 타입 검사를 허용할 것인가”를, `rootDir`과 `outDir`는 “원본과 결과물을 어떻게 분리할 것인가”를 결정하기 때문입니다. 나머지 옵션은 프로젝트가 커지거나 번들러, 테스트 환경이 추가될 때 조금씩 의미가 더 선명해집니다.

## **8-03. tsconfig.json 파일 불러오기**

프로젝트가 작을 때는 `tsconfig.json` 하나로도 충분해 보입니다. 그런데 빌드가 추가되면 공통 기준과 빌드 전용 기준이 조금씩 나뉘기 시작합니다. 예를 들어 `strict`, `target`, `module`처럼 프로젝트 전체에서 유지할 값도 있지만, `outDir`이나 빌드에서 제외할 파일 범위처럼 “출력할 때만” 필요한 값도 있습니다.

이때 설정 파일을 복사해서 여러 벌 관리하면 공통 값이 바뀔 때마다 모든 파일을 함께 수정해야 합니다. `extends`는 이런 중복을 줄이는 기능입니다. 먼저 여러 설정 파일이 함께 쓸 기준을 base 파일에 모아 두고, 실제 프로젝트 설정과 빌드 설정은 그 파일을 불러온 뒤 필요한 값만 덧붙이는 방식입니다.

먼저 공통으로 유지할 기준을 `tsconfig.base.json`에 모아 둡니다. 여기에는 여러 설정 파일이 함께 공유할 값만 넣는 것이 핵심입니다.

파일: `tsconfig.base.json`

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "strict": true,
    "module": "ESNext",
    "moduleResolution": "Bundler"
  }
}
```

이제 기본 프로젝트 설정인 `tsconfig.json`은 위 파일을 `extends`로 불러오고, 현재 프로젝트에서 읽을 소스 범위만 정의합니다. 에디터가 기본으로 읽는 설정 파일이 이 `tsconfig.json`이므로, 먼저 “이 프로젝트는 어떤 파일들을 타입 검사 대상으로 볼 것인가”를 여기서 정해 두는 편이 자연스럽습니다.

파일: `tsconfig.json`

```json
{
  "extends": "./tsconfig.base.json",
  "include": ["src/**/*"]
}
```

이제 빌드할 때만 필요한 설정은 별도의 `tsconfig.build.json`으로 분리합니다. 이 파일은 `tsconfig.json`을 다시 불러온 뒤, 실제 출력 폴더와 빌드에서 제외할 파일 범위를 추가합니다. 이렇게 하면 기본 프로젝트 설정과 빌드 설정이 분리되어, “평소 타입 검사”와 “실제 출력”의 역할이 더 또렷해집니다.

파일: `tsconfig.build.json`

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./dist"
  },
  "exclude": ["src/**/*.test.ts"]
}
```

이 구조에서는 `target`, `strict`, `module`, `moduleResolution`이 `tsconfig.base.json`에서 상속되고, `include`는 `tsconfig.json`에서 상속됩니다. 그다음 `tsconfig.build.json`은 `rootDir`, `outDir`, `exclude`만 추가해 빌드 전용 기준을 완성합니다. 즉 공통 기준은 위로 올리고, 현재 목적에만 필요한 값은 아래 파일에 남기는 방식입니다.

빌드를 실행할 때는 기본 `tsconfig.json`이 아니라, 빌드 전용 설정 파일을 직접 지정하면 됩니다. 예를 들어 `pnpm exec tsc -p tsconfig.build.json`처럼 실행하면 TypeScript 컴파일러는 공통 설정과 프로젝트 설정을 모두 불러온 뒤, 마지막으로 빌드 전용 설정까지 적용합니다.

| 파일 | 담는 내용 | 바뀌는 빈도 |
| --- | --- | --- |
| `tsconfig.base.json` | 여러 설정 파일이 함께 쓰는 공통 기준 | 비교적 적음 |
| `tsconfig.json` | 기본 프로젝트 범위와 공통 진입점 | 비교적 적음 |
| `tsconfig.build.json` | 출력 경로와 빌드 제외 범위 같은 빌드 전용 기준 | 비교적 자주 바뀜 |

`extends`를 쓰기 시작하면 `tsconfig.json`을 하나의 거대한 설정 파일로 키우지 않고, 공통 설정과 목적별 설정으로 나눠 관리할 수 있습니다. 이 방식은 모노레포처럼 프로젝트가 여러 개일 때 특히 중요하지만, 작은 프로젝트에서도 “공통 기준은 base에 두고, build 같은 개별 목적은 별도 파일로 분리한다”는 감각이 있으면 이후 설정 파일을 읽는 부담이 줄어듭니다.
